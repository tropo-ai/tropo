# Refresh a View

*Atomic action: update the memberships inside an existing view to reflect the current state of the Vault.*
*The counterpart to `generate-view`: generate creates a view from scratch; refresh updates an existing one.*

---

## 1. Intent

This action walks an existing view (a folder hierarchy of collections under `collections/<view-name>/`) and updates each collection's memberships to reflect the current state of the Vault. The folder structure does NOT change. The view's shape is preserved. Only the contents within each collection are refreshed.

**Refresh, not regenerate.** The distinction matters: a refresh preserves the human's chosen structure and updates only what's inside it. A regenerate would rebuild the structure itself, which is a different operation (use `generate-view` for that).

**When to invoke this action:**

The user says something like:
- "Refresh the keystone view"
- "Update Mike's view with the latest entries"
- "Pull in the new research files I just added to the Vault"
- "Refresh all my views"

**When to use `generate-view` instead:** when the user wants to create a new view or fundamentally restructure an existing one.

**Reference:** [`collection-ref.capsule.md`](../capsules/collection-ref.capsule.md), [`generate-view.action.md`](generate-view.action.md), Phase 1 spec §7.

---

## 2. Prerequisites

1. **The Vault exists.**
2. **The view exists** at `collections/<view-name>/`. If missing, stop and suggest `generate-view` instead.
3. **The view's root collection file exists** at `collections/<view-name>/<view-name>.collection.md`. This is the entry point for walking the view.
4. **The collection files in the view have valid frontmatter** including `purpose` statements (and optionally `query` fields) that define what belongs in each collection. If a collection file is missing its purpose, flag it but continue.

---

## 3. Inputs

### Required inputs

| Input | Type | Notes |
|-------|------|-------|
| `view_name` | string | The name of the view to refresh. Example: `keystone-view`. |

### Optional inputs

| Input | Type | Notes |
|-------|------|-------|
| `preserve_manual_additions` | boolean | Default: `true`. If true, UIDs that were hand-added to a collection's member list (not generated by a previous refresh) are preserved even if they don't match the current purpose or query. Set to `false` to force a strict regeneration of memberships. |
| `dry_run` | boolean | Default: `false`. If true, the action reports what WOULD change without actually writing any files. Useful for previewing refreshes before committing. |
| `scope` | string | If specified, only refresh collections matching this sub-path. Example: `scope: overview` only refreshes `collections/<view-name>/overview/*.collection.md`. Default: refresh everything. |

---

## 4. Process

### Step 1 — Locate the view

Verify `collections/<view-name>/` exists. Read the root collection file to understand the view's structure. If the view is missing, stop and report — suggest `generate-view`.

### Step 2 — Walk the view's collection files

Traverse the folder hierarchy under `collections/<view-name>/`. For every `.collection.md` file found, add it to a work list. This includes nested collections at any depth.

### Step 3 — For each collection, re-evaluate membership

For every collection in the work list, do the following:

1. **Read the collection's current state:** frontmatter (purpose, query if present, collection_type), current members list in the body, manually-added markers if any.

2. **Determine the new candidate members.** Use the Pattern A + Pattern B layering:
 - If the collection has a `query` field (Pattern B), run the query against the vault index and produce a list of matching UIDs. This is fast and deterministic.
 - If the collection has no query (Pattern A only), interpret the `purpose` field in natural language. Walk the vault index and decide which entries match the purpose through reasoning. This is slower and less deterministic but more flexible.
 - If both are present, the query is the primary source of truth. The purpose is used for validation and for entries that the query misses but match the intent.

3. **Merge with preserved manual additions.** If `preserve_manual_additions: true` (default), UIDs that were hand-added and are not captured by the query or purpose are preserved in the member list. If `false`, they are removed.

4. **Update the collection file.** Write a new members list to the body, preserving the existing structure (section headers, manual grouping, render mode). Update `last_modified` and `member_count` in the frontmatter. Update `last_synced` if present.

5. **Update the matching collection-ref entry in the ledger.** The `member_count` and `last_synced` fields in `vault/files/<uid>.md` should reflect the refresh. Update the index record.

### Step 4 — Detect and report drift

During the refresh, track:
- **New members added** — entries that now match a collection's purpose/query but weren't previously listed
- **Members removed** — entries that previously matched but no longer do (archived, deleted, or reclassified)
- **Dangling references** — UIDs in existing member lists that no longer exist in the Vault
- **Empty collections** — collections whose membership is now empty (either because they were always empty or because all members were removed)
- **Collections with missing purpose/query** — collections that can't be refreshed because they lack a definition of what belongs

Report each of these to the user after the refresh completes.

### Step 5 — If `dry_run: true`, do not write

If the user requested a dry run, produce the same report showing what WOULD change without writing any files. The user can review and then invoke the action again with `dry_run: false` to commit.

### Step 6 — Confirm to the user

Report:
- Total collections walked
- Number refreshed (with changes)
- Number unchanged
- Number failed (missing purpose, unreadable, etc.)
- Summary of drift (new members, removed members, dangling references)
- Any collections flagged for human attention

---

## 5. Template

This action does not use a template for new file creation — it updates existing files in place. The target collection files retain their existing templates (flat or hierarchical) and only the membership sections are rewritten.

The rewrite preserves:
- Frontmatter fields other than `last_modified`, `last_synced`, `member_count`
- Section headers in the body (for hierarchical collections)
- The collection's render mode
- Any manually-added notes or commentary in the body

The rewrite updates:
- The `members` list (within the existing section structure for hierarchical collections)
- The `last_modified` timestamp
- The `last_synced` timestamp (if present)
- The `member_count` in the frontmatter

---

## 6. Verification

1. Every collection in the view was visited
2. Every refreshed collection file is still valid (frontmatter parses, body renders)
3. Every member UID in any refreshed collection exists in the Vault
4. The matching collection-ref entries in the Vault have updated `member_count` and `last_synced`
5. No file was left in a half-written state

**Boot test (thought experiment):** Could the user navigate the refreshed view in their editor and get the same navigation experience as before, but with updated contents? The structure should feel identical — only the contents should differ.

---

## Failure Modes

| Failure | Action |
|---------|--------|
| View does not exist | Stop and suggest `generate-view` |
| Root collection file missing | Stop and report — the view is corrupted; manual repair or regeneration needed |
| Some collections refreshed, others failed | Complete the ones that work; report the failures individually; do not roll back successful refreshes |
| Dangling reference detected | Report it; do not automatically remove the UID (that's a human decision) |
| Query field present but query is malformed | Fall back to interpreting the purpose; report the query failure for human review |
| Purpose field missing AND no query field | Skip that collection; report it as unrefreshable; suggest the user add a purpose |

---

## Relationship to Manual Curation

If a human has hand-added UIDs to a collection (either by editing the file directly or by working with an agent in a session), those additions should be preserved across refreshes by default. The `preserve_manual_additions` flag controls this behavior.

The way this works operationally:
- When the refresh computes the new candidate members from query/purpose, it marks them as "generated"
- It reads the existing member list and identifies UIDs that are NOT in the generated set — those are manual additions
- If `preserve_manual_additions: true`, the manual additions are added to the refreshed member list
- If `preserve_manual_additions: false`, the member list is replaced entirely with the generated set

This gives the human a stable collaborative relationship with the refresh action: they can hand-curate, and the refresh respects their edits while still pulling in new entries that match the collection's purpose.

---

## A Note on Frequency

At L1, refresh is invoked explicitly by an agent in a session. A user who adds entries to the Vault must eventually trigger a refresh to see the view update.

At L2 (when the brain exists), refresh happens continuously in the background. The brain watches the Vault and re-evaluates view memberships in real time. Views are always fresh without explicit invocation.

Between these two extremes, a middle-ground pattern works at L1: the vault steward or a scheduled standing operation runs `refresh-view` periodically (nightly, hourly, per session close) for all views the user cares about. This keeps views approximately fresh without requiring the user to remember to ask. Implementation of this standing operation is follow-on work for Phase 1.B.
