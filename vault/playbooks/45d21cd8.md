# Setup New Pipeline

*Comprehensive one-time-per-pipeline setup process. Authors a v3.0-shape pipeline definition with per-step rich schema (step_owner_role / step_verifier_role / exit_criteria / depends_on_steps / trust_level / retry_policy / timeout_hours). Output: a root pipeline WorkflowNode + N child step WorkflowNode entries in `vault/files/`, ready for pipeline-activate.py + pipeline-runtime.py to operate on.*

*Mike-A75 lock 2026-05-20 verbatim: "the pipeline.capsule definition will simply require the agent setting up the new pipeline to ask more questions of the user to make sure the pipeline is constructed properly. That process creates the specific pipeline definition for web-pipeline (for instance). This is one-time setup per pipeline, It's okay if it's comprehensive."*

---

## Intent

A pipeline is a declarative workflow template. Setting one up properly — with machine-checkable exit criteria, explicit DAG dependencies, named executor + verifier roles, calibrated trust gradient — is the load-bearing one-time work that pays off at every activation. Skipping the comprehensive setup means activating against a memory-resident contract; that's the cardboard-muffin failure mode v1.46.0 ships to fix.

This playbook governs:

- **Create-new mode** — Author a brand-new pipeline from scratch (e.g., a new domain-specific workflow Mike or another principal wants to govern).
- **Retrofit mode** — Convert an existing v2.6-shape pipeline (no per-step rich schema) to v3.0-shape. The pipeline structure preserved; per-step rich fields added to each leaf step WorkflowNode.

In both modes, the output is a v3.0-shape pipeline definition that `pipeline-runtime.py` can read + lock into an activation contract.

---

## Rules

**Comprehensive by design.** This is not a fast playbook. Each step in the resulting pipeline gets walked individually with the principal; abbreviated walks produce under-specified contracts that fail at activation. Estimated wall-clock: 30-60 minutes per pipeline of ~5-9 steps.

**One-time-per-pipeline.** This playbook runs ONCE per pipeline definition. Subsequent activations of the same pipeline don't re-run setup; they use the locked contract baked into the WorkflowNode entries.

**The principal authorizes the contract.** The setup walk produces a contract proposal; the human (Mike or other principal) walks each step + locks it. The authoring agent does NOT lock the contract unilaterally.

**Exit-criteria must be machine-checkable.** Every step's `exit_criteria:` list MUST use the DSL shapes defined in pipeline.capsule v3.0 §6 (`<uid-ref>.exists`, `<uid-ref>.<field> == <value>`, `<uid-ref>.<field> contains <value>`, `<uid-ref>.body matches <pattern>`, `file:<path>.exists`). Free-text exit-criteria like "the brief looks good" are rejected at validation time. If a step's verification logic can't be expressed in the DSL today, mark the step `verification_class: true` and rely on the step's natural output (e.g., build exit code) as the verdict.

**DAG invariant.** `depends_on_steps:` edges form a directed acyclic graph. Cycles are rejected by `check_step_depends_on_acyclic` (pipeline.capsule v3.0 §Check 18).

**Default verifier is same-as-executor.** Per Mike-A75 lock 2026-05-20: the executor agent writes the structured verification_receipt; the structured-receipt format is the forcing function. Override (`step_verifier_role: <other-class>` with separate-context sub-agent dispatch) only when the step is high-stakes enough to warrant the extra coordination cost.

**Pipelines start at `status: draft`.** Per pipeline.capsule v3.0 §State Machine Rule 1: every new pipeline file is authored at `status: draft`. Direct creation at `active` or `locked` is a validation error. The principal explicitly flips to `active` after the setup walk completes + the playbook's Group 6 verification passes.

## Modes

### Create-new mode

Authoring a new pipeline from scratch. Group 0 confirms create-new + collects pipeline-level metadata (name, domain, purpose). Groups 1-3 walk pipeline-level then per-step. Group 4 validates DAG. Group 5 authors substrate. Group 6 verifies + hands off.

### Retrofit mode

Converting an existing v2.6-shape pipeline to v3.0 schema. Group 0 confirms retrofit + identifies the target pipeline UID. Group 1 reads existing structure; Group 2 confirms the step graph (typically no shape change). Group 3 adds per-step rich schema to each leaf step. Group 4 validates the now-v3.0-shape graph. Group 5 edits existing vault entries in place (UIDs preserved). Group 6 verifies.

---

## Groups

```
Group 0 — Mode + Scope
 ↓
Group 1 — Pipeline-Level Definition
 ↓
Group 2 — Step Graph Design
 ↓
Group 3 — Per-Step Rich Schema Walk (loops N times — once per step)
 ↓
Group 4 — DAG Validation
 ↓
Group 5 — Substrate Authoring
 ↓
Group 6 — Verification + Handoff
```

### Group 0 — Mode + Scope

**Owner:** Authoring agent + principal
**Parallel:** no

#### Step 0.1 — Confirm mode

Ask the principal: *"Are we creating a new pipeline (create-new) or retrofitting an existing one to v3.0 schema (retrofit)?"*

**If create-new:** continue to Step 0.2 (collect pipeline name + domain + purpose).
**If retrofit:** continue to Step 0.3 (identify target pipeline UID + read existing structure).

#### Step 0.2 — Create-new metadata

Ask the principal:

1. *"What's the pipeline name?"* (≤120 chars; human-readable; e.g., "Release Pipeline", "Customer-Event Master Pipeline")
2. *"What's the domain?"* (lowercase-hyphen; ONE root pipeline per domain per active version per pipeline.capsule Rule 1)
3. *"What workflow does this govern?"* (one-paragraph purpose statement)
4. *"What workflows does this NOT govern?"* (disambiguation against adjacent pipelines)

Record responses. Continue to Group 1.

#### Step 0.3 — Retrofit target identification

Ask the principal:

1. *"What's the UID of the existing pipeline to retrofit?"* (8-hex)
2. *"Is this the root WorkflowNode or a stage/step?"* (must be root for retrofit; stages/steps retrofitted via root)

Read the existing pipeline entry + walk its children recursively to build the full WorkflowNode tree. List every leaf step + present to principal:

> *"Existing pipeline `<name>` (v<version>) has N stages and M leaf steps. We'll add v3.0 per-step rich schema to each leaf step. Confirm shape unchanged?"*

If principal confirms, continue to Group 1 (mostly no-ops in retrofit mode). If principal wants to also reshape the graph, escalate: that's a supersession cycle, not a retrofit; author a new pipeline UID via create-new mode and set bidirectional `supersedes:` / `superseded_by:` per pipeline.capsule Rule 8.

---

### Group 1 — Pipeline-Level Definition

**Owner:** Authoring agent
**Parallel:** no

#### Step 1.1 — Default trust gradient

Ask the principal:

> *"What's the default trust_level for steps in this pipeline?"*
>
> *Options: `approval-required` (every step pauses for human confirmation), `auto-with-verification` (default — proceed once structured receipt passes), `auto-after-track-record` (proceed after ≥ 3 prior pass events; v1.46.0 ships shape, calibration starts v1.47.0+), `human-only` (only humans can mark steps complete).*

Record as `default_trust_gradient:` on the root pipeline WorkflowNode.

#### Step 1.2 — Default retry policy

Ask:

> *"What's the default retry policy on step failure?"*
>
> *Options: `{ max_retries: 0, backoff: linear }` (no retries; default), or specify `{ max_retries: <N>, backoff: linear|exponential }`.*

Record as `default_retry_policy:` on the root pipeline.

#### Step 1.3 — Default timeout

Ask:

> *"What's the default timeout per step, in hours?"*
>
> *Default 24. Step-level overrides via `timeout_hours:` on each step.*

Record as `default_timeout_hours:`.

#### Step 1.4 — Cascade behavior (optional)

Ask:

> *"Does this pipeline spawn workstream sub-pipelines at activation time (cascade pattern per pipeline.capsule §6)? If yes, declare cascade_spec."*

If cascade declared, walk through the `cascade_spec:` sub-fields per pipeline.capsule §6 (generates_project_plan; spawns_workstreams: array with pipeline_uid + name + owner_agent_class per entry; spawns_tasks if deeper cascade). Otherwise skip.

---

### Group 2 — Step Graph Design

**Owner:** Authoring agent + principal
**Parallel:** no

#### Step 2.1 — Identify stages

Ask:

> *"What stages does this workflow have? (linear list of stage names; we'll add depends_on_steps DAG edges in Group 3 if branching exists.)"*

For each stage name, mint a UID (or note that stage WorkflowNodes get authored in Group 5).

#### Step 2.2 — Identify steps per stage

For each stage, ask:

> *"What steps are in this stage? (linear list of step names — each step is a leaf WorkflowNode that does discrete work.)"*

For each step name, mint a UID. Steps are the leaves; their rich schema gets walked in Group 3.

#### Step 2.3 — Identify cross-stage dependencies

Ask:

> *"Which steps depend on steps from earlier stages? (e.g., 'Step X cannot start until Step Y in stage Z completes.')"*

Map these to `depends_on_steps:` edges on the dependent step. Forward-only by default; depends_on_steps captures the explicit edges beyond simple linear flow.

#### Step 2.4 — Identify branching / conditional flow

Ask:

> *"Does any step have multiple next_steps (branching)? If so, what condition decides which branch fires?"*

Branching uses `next_steps:` (length > 1) + `condition:` expression on each branch target. v2.0 schema-supports branching; v3.0 step schema (depends_on_steps DAG) coexists. Note: branching behavior beyond "first condition true" defers to capsule v3.1.

---

### Group 3 — Per-Step Rich Schema Walk

**Owner:** Authoring agent + principal
**Parallel:** no (each step walked individually)

Loop over every leaf step from Group 2. For each step:

#### Step 3.1 — Owner role

Ask: *"Which agent class executes step `<step-name>`?"*

Values: `argus` / `talos` / `vela` / `metis` / `sa.<slug>` / `director` / `human-only` / domain-specific.

Record as `step_owner_role:`.

#### Step 3.2 — Verifier role

Ask: *"Does this step need separate-context verification (a different class than the executor)?"*

**Default (recommended for most steps):** `same-as-executor`. The executor writes the structured verification_receipt; the structured format itself is the forcing function.

**Override (high-stakes steps only):** specify a different class. This triggers separate-context sub-agent dispatch at verify-step time per pipeline-runtime.py §8.

Record as `step_verifier_role:`.

#### Step 3.3 — Verification class (bypass for inherently-verifying steps)

Ask: *"Is this step's natural output ITSELF the verification (build exit code, HTTP response, validator script pass-fail)?"*

If yes: set `verification_class: true`. No structured verification_receipt expected; the step's natural output is captured in `step_completed.data.natural_verdict:`.

If no: leave default `false`. Step requires explicit verification_receipt event.

#### Step 3.4 — Exit criteria

Ask: *"What machine-checkable assertions must hold true for this step to be verified pass?"*

Walk the principal through the DSL shapes:

- `<uid-ref>.exists` — a vault entry exists on disk
- `<uid-ref>.<field> == <value>` — frontmatter field equals value
- `<uid-ref>.<field> contains <value>` — frontmatter array contains element
- `<uid-ref>.body matches <pattern>` — markdown body matches pattern
- `file:<path>.exists` — file exists at relative path

For each assertion the principal names, render it as a single DSL line + collect into `exit_criteria:` array. If the principal can't express a needed assertion in the DSL, ask whether the step is verification-class instead, or note the gap for v3.1 DSL extension.

If `verification_class: true`: `exit_criteria:` can be empty (step's natural verdict is the receipt).

#### Step 3.5 — Depends-on edges

Confirm `depends_on_steps:` array from Group 2.3. If this is the first/root step in the pipeline, leave empty `[]`.

#### Step 3.6 — Trust level (override default)

Ask: *"Override the pipeline's default trust gradient for this step?"*

If override: specify one of `approval-required` / `auto-with-verification` / `auto-after-track-record` / `human-only`. Record as `trust_level:`. Otherwise leave field absent (pipeline default applies).

#### Step 3.7 — Retry policy (override default)

Ask: *"Override the pipeline's default retry policy for this step?"*

If override: specify `{ max_retries: <N>, backoff: linear|exponential }`. Record as `retry_policy:`. Otherwise leave absent.

#### Step 3.8 — Timeout hours (override default)

Ask: *"Override the pipeline's default timeout for this step?"*

If override: specify integer. Record as `timeout_hours:`. Otherwise leave absent.

#### Step 3.9 — Compensation step (saga pattern; optional)

Ask: *"If this step needs to be undone (saga rollback), what step runs as the compensation?"*

If declared: specify the UID of the compensation step. Record as `compensation_step_id:`. v3.0 ships single-step compensation only.

#### Step 3.10 — Instructions reference (optional)

Ask: *"Is there a capsule, playbook, or document that explains HOW the step's work is done?"*

If yes: record path or UID in `instructions_ref:`. Centralizes prose; doesn't duplicate.

---

### Group 4 — DAG Validation

**Owner:** Authoring agent

#### Step 4.1 — Walk depends_on_steps graph

Build the DAG: nodes are step UIDs; edges are `depends_on_steps:` entries. Run cycle detection (depth-first; flag any back-edge).

If a cycle is detected: halt; report the cycle path to the principal; ask which dependency to remove. Loop back to Group 2.3.

#### Step 4.2 — Walk exit_criteria DSL

For each step, parse each `exit_criteria:` entry through the DSL parser (pipeline-runtime.py §10 parse_exit_criterion logic). Reject malformed entries.

#### Step 4.3 — Confirm verifier-override discipline

For each step where `step_verifier_role` ≠ `same-as-executor`: confirm the value differs from `step_owner_role:` (else `check_step_verifier_distinct_from_owner_when_overridden` will fail at validator time).

---

### Group 5 — Substrate Authoring

**Owner:** Authoring agent

#### Step 5.1 — Author root pipeline WorkflowNode

Write `vault/files/<root-pipeline-uid>.md` with frontmatter:

```yaml
uid: <root-uid>
type: pipeline
subtype: workflow-node
name: "<pipeline-name>"
version: "0.1"
author: <authoring-agent-uid>
status: draft
state: active
domain: <domain>
children: [<stage-uids>]  # OR [<step-uids>] if no stages
default_trust_gradient: <value>
default_retry_policy: <value>
default_timeout_hours: <value>
cascade_spec: <object-or-omit>
member_of: [<governing-subsystem-uids>]
tags: [pipeline, v3-0-shape]
```

Body: 6 required sections per pipeline.capsule v3.0 §2 (Purpose, Structure, Nodes, Flow Rules, Cold-Boot Walk-Through, Known Enforcement Gaps).

#### Step 5.2 — Author stage WorkflowNodes (if any)

For each stage from Group 2: write `vault/files/<stage-uid>.md` with `type: pipeline, subtype: workflow-node, role: "stage", children: [<step-uids>]`.

#### Step 5.3 — Author leaf step WorkflowNodes

For each step: write `vault/files/<step-uid>.md` with all v3.0 per-step rich schema fields collected in Group 3 (step_owner_role, step_verifier_role, verification_class, depends_on_steps, exit_criteria, trust_level, retry_policy, timeout_hours, compensation_step_id, instructions_ref).

Step body explains what the step does in human-readable prose; the structured fields drive the runtime.

#### Step 5.4 — Rebuild vault index

Run `python3 .tropo/scripts/rebuild-vault.py --apply` to incorporate the new entries into `vault/00-index.jsonl`.

---

### Group 6 — Verification + Handoff

**Owner:** Authoring agent + principal

#### Step 6.1 — Run validator

Run `python3 .tropo/scripts/tropo-validate.py 2>&1 | grep -A 2 "Step Verifier\|Step Depends\|Cascade Spec"` to confirm:

- `check_step_verifier_distinct_from_owner_when_overridden` — passes (no override defects)
- `check_step_depends_on_acyclic` — passes (DAG is acyclic)
- `check_cascade_spec_validity` — passes (if cascade declared)

If any failure: report to principal + loop back to Group 3 or Group 2 to fix.

#### Step 6.2 — Render preview

Render a human-readable preview of the pipeline shape to the principal:

```
Pipeline: <name> (UID <root-uid>; status draft)
  Stage 1: <stage-name>
    Step A — owner=<role>  verifier=<role>  trust=<level>  exit_criteria=<N>
    Step B — owner=<role>  verifier=<role>  trust=<level>  exit_criteria=<N>
  Stage 2: <stage-name>
    Step C — owner=<role>  verifier=<role>  trust=<level>  exit_criteria=<N>
DAG edges:
  Step C depends_on Step A, Step B
```

Ask: *"Pipeline shape looks right? Ready to flip status: draft → active?"*

If principal confirms: edit the root pipeline frontmatter `status: draft → active`. The pipeline is now live; activations can pin this version + run under pipeline-runtime.py.

If principal wants changes: loop back to the relevant Group.

#### Step 6.3 — Handoff

Tell the principal:

> *"Pipeline `<name>` is live at `vault/files/<root-uid>.md`, v0.1, status active. To activate a run, invoke:*
>
> *`python3 .tropo/scripts/pipeline-activate.py --pipeline-uid <root-uid> --activated-by <UID> --cycle-context "<context>"`*
>
> *Then bootstrap the runtime:*
>
> *`python3 .tropo/scripts/pipeline-runtime.py --activation-uid <returned-uid> bootstrap`*
>
> *Pipeline-runtime.py will read the activation entry, mint a v2.0-shape pipeline-run entry, prompt you through the four contract questions, lock the contract, and seed the run.jsonl event log."*

Pipeline-runtime.py is now responsible for the run-time concerns; this playbook's work is complete.

## Failure Handling

| Failure | Response |
|---------|----------|
| Principal cancels mid-walk | Save WIP state (which steps were walked; what was collected); allow resume at the unwalked step. Mid-walk pipeline files are NOT written until Group 5; cancellation before Group 5 leaves no substrate. |
| DAG cycle detected at Group 4 | Halt; report cycle path; loop back to Group 2.3 for dependency revision. |
| Exit-criteria DSL parse failure | Halt; show the failing line; ask principal to rephrase or mark step verification-class. |
| Verifier-override discipline violation | Halt; report which step has step_verifier_role == step_owner_role with non-same-as-executor value; ask principal to either drop the override or pick a different verifier class. |
| Validator FAIL after Group 5 authoring | Surface to principal; do NOT flip status: draft → active until validator clean. Most likely cause is an unresolved UID reference (governance_by, member_of, depends_on_steps). |
| UID collision at vault write | Mint a fresh UID; retry. |
| Filesystem write failure | Halt at the failed write; preserve partial state; report to principal. Authoring is per-file; partial authoring is recoverable by re-running Step 5.X for the failed entries. |
