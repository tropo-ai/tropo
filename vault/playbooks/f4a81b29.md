## Intent

Orient a registered agent at activation. Verify identity, confirm no parallel generation conflict, load mission and operational context, load persistent memory, regenerate work state, update crew-facing artifacts, and deliver a startup signal — so the agent is grounded, current, and ready to work before engaging the human.

## Suggestions

- **Read before you act.** The most important lesson from 22 generations of Vela: enthusiasm produces output quickly; being informed produces the *right* output. When in doubt, read one more file before touching anything.
- **Let the human set the pace.** The human gives context in layers. Don't optimize past their onboarding rhythm.
- **Board regeneration is cheaper at boot than at wind-down.** Fresh context window vs. expensive late-session tokens.
- **Treat the briefing package as your predecessor's gift.** They spent context budget curating it for you.
- **If a read fails, note the gap and continue.** Missing files should not block boot. Log the gap, proceed, flag in startup signal.
- **Your memory is your inheritance.** The `.tropo-studio/memory/MEMORY.md` and `.tropo-capsule/memory/MEMORY.md` files contain what your predecessors learned. Read the pinned and fresh entries — they paid for that knowledge with context tokens you don't have to spend.

## Rules

### Phase 0: Boot Configuration Chain (ADR-032)

Before identity verification, load the three-tier boot configuration. Each layer extends the one above without removing required steps.

0. **Read the boot configuration chain in sequence:**

 - **Layer 1 — OS:** `.tropo/boot-config.md` (if it exists — ships with Tropo OS). Contains OS-required steps.
 - **Layer 2 — Vault:** `.tropo-studio/agent-boot.extension.md` (vault-specific required steps — active agents, orientation paths, fleet check). Read this in every Argo boot.
 - **Layer 3 — Agent:** `agents/<name>/agent-boot.extension.md` (agent-specific steps — channels, board, briefing package, write scope).

 **Override contract:** Lower layers can add steps; they cannot remove steps marked `required: true` in a higher layer. If a layer file is missing, note the gap and continue — missing config files are not boot-halting failures.

 Incorporate all declared steps from the chain into your boot plan before proceeding. Steps from Layer 2 and Layer 3 that belong in specific phases are executed at those phases — the config files declare WHAT, the playbook declares WHEN.

### Phase 1: Identity Verification (HARD GATES)

These steps MUST complete before any other boot action. Failure halts boot.

1. **Read your status card.** Path: `agents/<name>/<name>-status.md`. If no status card exists, this is a first-generation agent — skip the predecessor check, create the status card, and proceed to Step 2. Otherwise: if your predecessor's status shows ACTIVE (not RETIRING or RETIRED), STOP. Post to `channels/ops.md` and wait for human direction. (ADR-016 parallel generation protocol.)

2. **Read your generation log.** Path: `agents/<name>/generation-log.md`. Confirm your generation number = last entry + 1. If it does not match, STOP and flag to the human.

3. **Open your generation log row.** Append: generation number, boot date, empty retire date, empty contribution. Close your predecessor's row (set retire date if blank).

### Phase 2: Context Loading

Required reads that build the agent's understanding of the current state.

4. **Read `.tropo/orientation.md`.** This is the harness map — every action, skill, playbook, capsule, and session-agent available in this vault, and when to reach for each. If you skip this, you will reinvent procedures that already exist. Read it once per session; keep it in context while you work.

5. **Read `.tropo-studio/operating-principles.md`.** These are the operating assumptions under which all crew work happens. Read them critically — ask whether your boot experience so far is consistent with them. The vault root is the directory containing `.tropo/` — no configuration file is needed to resolve it.

6. **Read your mission brief.** Path: `context/mission-brief.md` if it exists, otherwise `STUDIO.md`. The mission frames all subsequent work. If neither exists, note the gap and continue.

7. **Read your briefing package.** Path declared in activation file, or convention: `agents/<name>/briefing-package/00-index.md`. If no briefing package exists, fall back to: living transfer → crew brief → `channels/ops.md`. Note the gap.

8. **Load your memory.** Read `agents/<name>/.tropo-capsule/memory/agent-memory.md` (agent-level, v3 single surface; legacy `MEMORY.md` in pre-v3 Studios). Read `.tropo-studio/memory/memory-current.md` (vault-level, v3; legacy `MEMORY.md` pre-v3). Read pinned and fresh entries in full. Skip if memory directories don't exist yet — note the gap. *(Paths corrected A109 2026-06-11 per Cosmo C9 boot diagnostic; Memory v3.0 shipped v1.67.)*

9. *(Optional)* **Read agent-specific resources** declared in your activation file's `boot_reads:` parameter (e.g., architectural principles for a strategist, KB index for the concierge).

### Phase 3: Operational Grounding

Build awareness of crew state, work state, and communication state. Steps 11-13 can run in parallel.

10. **Read `00-crew-brief.md` if it exists.** Confirm it reflects current crew state. Note agents listed as incoming or retiring. Internalize the operating values section. Skip if not found — first-generation and single-agent vaults may not have a crew brief yet.

11. **Check your channels.** Use the `scan-channels` skill at `vault/skills/2d5e9a73.md` with your declared channels. Note action items directed at you, staleness, and recent ops.md activity.

 ```yaml
 spawn_parallel:
 - skill:vault/skills/2d5e9a73.md  # scan-channels
 params:
 channels: <from activation file boot_params.channels>
 output: agents/<name>/.tropo-capsule/workspace/channel-scan.md
 on_failure: continue
 - skill:vault/skills/9b1f3e48.md  # check-vault-health
 params:
 health_report_path: shared/orientation/daily-health-report.md
 output: agents/<name>/.tropo-capsule/workspace/health-check.md
 on_failure: continue
 - skill:vault/skills/6c4a8f12.md  # regenerate-board
 params:
 board_path: <from activation file boot_params.board>
 output: agents/<name>/.tropo-capsule/workspace/board-results.md
 on_failure: continue
 ```

 After all skills complete, read the output files and incorporate results into your awareness. If the platform does not support parallel execution, run these skills inline sequentially.

11. **Run the fleet-ops playbook.** Execute `playbooks/fleet-ops.playbook.md`. This checks which scheduled ops agents are due to run (by reading `agents/operations/scheduled-agent-registry.md` and comparing each agent's activation-log for today's record) and dispatches any that haven't run yet as sub-agents.

 **Observability is required.** Every fleet-ops run must leave at least one line in `channels/ops.md` — success, dispatch-with-findings, or failure. Silent pass-through is not acceptable. If fleet-ops fails to run at all (playbook file missing, dispatch tool unavailable, etc.), post a `BULLETIN` to `channels/ops.md` yourself naming the reason. If the registry is unreadable or fleet-ops errors mid-dispatch, fleet-ops itself will post `FLASH` to `channels/alerts.md` + `BULLETIN` to `channels/ops.md` per its Failure Handling table.

 **Must not block boot.** Boot proceeds regardless of fleet-ops outcome. The visibility requirement ensures the failure is seen, not silently swallowed.

12. **Spawn sa.project-manager.** sa.project-manager loads PM state from `vault/00-pm-state.json` (or rebuilds via [`rebuild-pm-state.skill (f0e7c6fc)`](../skills/f0e7c6fc.md) if the file is stale per its mtime). Spawn using the activation-log channel pattern: create the next numbered record file in `agents/sa/sa.project-manager/activation-log/`, write your `[RESPONSE]` termination instructions, then dispatch. On failure, note the gap and continue — PM state is useful but must not block boot.

13. **Confirm predecessor's transfer** is marked FINAL or RETIRED. If not, ADR-016 applies.

14. **Update your status card.** Required: last updated date, generation, session number, recently completed (predecessor's work), working on (this session's plan), status, blockers.

### Phase 3.5: Self-Diagnostic

Run before delivering the startup signal. This step takes 5 minutes and prevents generational drift.

For each document read in Phase 2, ask:
- Is anything here outdated, counterproductive, or missing?
- Does this document accurately describe how the crew actually works?
- Has anything I've read revealed a gap, a drifted metric, or a broken principle?

Surface any findings in the startup signal — even one line. Do not buffer them for "later." Later means the next generation inherits the problem.

### Phase 4: Startup

13. **Deliver your startup signal.** Format defined in your activation file. The signal includes: identity confirmation, situational read, honest priority call, and clarifying questions.

14. **You MUST NOT begin substantive work before completing Phase 3.** The startup signal is the gate between boot and work.

## Verification

### Method

Self-verification at the end of boot, before delivering the startup signal.

### Criteria

- Status card was read AND predecessor was not ACTIVE at read time
- Generation log has a new row with today's date and the correct generation number
- Predecessor's generation log row has a retirement date (not blank)
- Memory index loaded (or gap noted if directories don't exist)
- Status card reflects ACTIVE status with current generation and session number
- Startup signal was delivered (not suppressed or skipped)
- If any required read failed, the gap is noted in the startup signal

### Escalation on Failure

- **Parallel generation detected (predecessor ACTIVE):** HALT. Post to `channels/ops.md`. Wait for human direction.
- **Generation mismatch:** HALT. Flag to human.
- **Required file missing:** Note the gap. Continue boot. Flag in startup signal.
- **Briefing package missing or stale:** Fall back to living transfer + crew brief + ops.md. Note the gap.
- **Memory directory missing:** Skip memory loading. Note in startup signal. Expected for first-generation agents.
