# Dispatch Cold-Boot — Strict + Skeptic Modes Test-Harness

*Sprint 3 first-instance test-harness playbook. Codifies the cold-boot dispatch protocol — both Strict mode (surface-following, current 097-103 pattern) and Skeptic mode (adversarial user-error attempts, NEW v1.4.2). Per [The Patient Honing Doctrine](../../.tropo-studio/memory/patient-honing-doctrine.md): the workshop IS the system; the system compensates for forgetting. Cold-boot dispatch is workshop primitive from v1.4.2 forward.*

*Sibling playbook: [`dispatch-walker.playbook` v0.2 (`7579f894`)](dispatch-walker.playbook.md) — same shape, different test instrument (walker measures encounter; cold-boot measures self-sufficiency).*

---

## Intent

Dispatch a cold-boot verification run against a target artifact in one of two modes. Cold-boot tests artifact self-sufficiency from a stranger's cold context; the Strict-vs-Skeptic distinction is *what kind* of stranger (the disciplined-surface-follower OR the user-error-prone newcomer).

**Strict mode** (current 097-103 pattern): agent disciplined to follow the surface — reads ONLY what the target points to; every extra read is a finding. Catches drift between governance docs, dangling references, missing required-frontmatter, contradictory instructions, capsule-vs-canonical-article misalignment. Stage 3.2 of release-test-plan v2 has been Strict-mode for v1.3.1 / v1.4.0 / v1.4.1.

**Skeptic mode** (NEW v1.4.2 per [Strict-vs-Skeptic brief (f7b3e2a1)](../files/f7b3e2a1.md)): agent given a list of common new-user mistakes; reports system-recovery per-mistake. Catches the class of issue Mike's 2026-04-30 organic test caught — the user opening Claude Code in the parent dev repo and getting generic-dev-assistant behavior instead of the foreman/concierge welcome flow.

**Why both modes are kernel substrate.** Strict catches "the artifact is wrong"; Skeptic catches "the artifact is right but the user can't use it." Both are ship-quality issues. Per release.capsule v3.1 Rule 10 + [release-test-plan v1 → v2 amendment (Sprint 5)](../files/1f5b3a9d.md), Stage 3.2-strict + Stage 3.2-skeptic both required for ship eligibility. **Must-pair-for-ship is binding** — see Rule 1 + Step 3.

**Target artifacts supported.** Primary use case is release ship-zip pre-ship verification (frontmatter `target_artifact_type: release`). Playbook also dispatches against capsules (cold-boot 100 pattern), playbooks (cold-boot 102/103 pattern), vault entries, kernel-tier files, or full vault contexts via Step 1 fork. Frontmatter declares the primary class; Step 1 routes per actual target.

**Transitional invocation mode (until Sprint 4).** Until [`run-release-test-plan.playbook` (Sprint 4 task `4c8f6d2a`)](../files/4c8f6d2a.md) lands, this playbook is invoked directly by the dispatcher as Stage 3.2 of the manual gauntlet. Direct invocation is a transitional valid mode; once the orchestrator exists, direct invocation becomes a code-smell (orchestrator-bypass) auditable by the absence of a parent `run-release-test-plan` record.

## Rules

1. **Mode MUST be declared explicitly at dispatch.** `mode: strict` or `mode: skeptic`. **No default — omission HALTs.** (Round 1 v0.2 amendment per skeptic 026 F1: prior `default: strict` semantics silently masked Skeptic-blind dispatches under release.capsule v3.1 Rule 10's must-pair requirement; explicit declaration eliminates this failure mode.) Mixing modes in one dispatch is a Rule-1 violation; spawn separate sessions for each mode.
2. **Stage 3.2 release-target dispatches MUST run BOTH Strict + Skeptic modes** for ship eligibility per release.capsule v3.1 Rule 10. Single-mode Stage 3.2 dispatch does NOT clear the gate; both records must exist + both verdicts must be PASS (or PARTIAL within ship tolerance) before the release entry's `verification_artifacts:` array accepts either record. Per Step 9. (Round 1 v0.2 amendment per cold-boot 104 F5 + skeptic 026 F1 convergence: must-pair was buried in §Intent prose; promoted to Rule for enforceability.)
3. **Dispatcher MUST identify the target artifact by file path AND UID.** Cold-boot tests artifact self-sufficiency; the target is the unit under test. Without explicit target identification, the cold-boot run has no anchor.
4. **Pre-walk fixture confirmation requirements vary by target class** (Step 1 fork). Release-target requires SHA-256 + file count + spawn cwd realpath. Non-release targets (capsule, playbook, vault entry, kernel-artifact, vault) require file-existence + UID match against ledger; cwd realpath only when the test environment is a separately-extracted artifact.
5. **Cold-boot session cwd MUST be the resolved-real-path of the test environment** when the test environment is supposed to be an extracted artifact (release ship-zip, separately-extracted vault). Not the dev repo, not a symlink-into-dev-repo. Same `realpath` discipline as dispatch-walker. For non-release targets where the cold-boot reads files in-place from the live vault, the realpath check does NOT apply (the live vault IS the test environment).
6. **Verdict IS the test result.** Re-running cold-boot in the same session after a finding is BLOCKED — record the verdict and proceed to remediation. Re-running is for fresh attempts after surfacing-layer fixes, not for "let me try again."
7. **PASS clears Stage 3.2; PARTIAL or FAIL** triggers remediation per release-test-plan v2 Stage-3.2 ship-blocking semantics. Strict mode: FAIL is HALT-SHIP (e.g., cold-boot 097's verdict against v1.4.0). Skeptic mode: FAIL is "system silently produced wrong-context behavior on common mistakes."
8. **Verdict transcription to release entry MUST be literal string-copy + (Skeptic mode) per-mistake outcome carry-over.** Per Step 9: the `verdict:` field appended to release entry's `verification_artifacts:` array is a literal copy of the cold-boot `[DONE]` block's verdict line. For Skeptic mode, the per-mistake outcomes (PASS/FAIL/PARTIAL each, with one-line explanation) MUST also be carried into the verification_artifacts entry — the aggregated verdict alone loses the audit-grade granularity per skeptic 026 F6. Mismatch is a Gate 5 audit violation.
9. **Dispatcher MAY be Vela** (cold-boot is Vela-owned per crew scope; no allowlist restriction analogous to walker's denylist). Argus and Metis MAY also dispatch cold-boot for their respective lane needs. Founder authority overrides any allowlist.
10. **Dispatcher MUST author the `[PENDING]` block themselves.** Copy-pasting another agent's draft is a Gate 5 independence violation auditable post-hoc. Same audit obligation as dispatch-walker.playbook v0.2 Rule 8.
11. **Incremental-report-writing REQUIRED.** Cold-boot agent MUST write findings incrementally to the activation-log record as they arise — not buffered to end-of-walk. Lesson from cold-boot 099 stall (agent ran out of context budget mid-write; entire diary lost; only 5 V36-salvaged inferences survived). Spawn template includes explicit instruction; agent compliance is enforced by reviewing the record's append history. (Note: post-hoc audit acknowledged as imperfect; preventive guardrails tracked in §Known Enforcement Gaps.)

## Steps

### Step 1. Confirm fixture integrity + resolve spawn cwd (per target class)
*Owner: dispatcher. Deadline: before authoring `[PENDING]`.*

**For release-target invocations** (Stage 3.2 of release-test-plan v2; primary use case):

```sh
# Compute SHA-256 of the ship-zip
shasum -a 256 <path-to-ship-zip>            # macOS / Linux
# OR: openssl dgst -sha256 <path-to-ship-zip>

# Count files in the extracted target (excluding macOS metadata)
find <extracted-path> -type f \! -name '.DS_Store' | wc -l

# Resolve real path of the spawn cwd (REQUIRED for release-target — symlink defense per Rule 5)
realpath <extracted-path>
```

Match SHA + file count against the build entry's recorded `verification.ship_zip_sha:` and `verification.file_count:` fields. The build entry lives at `vault/files/<build-uid>.md` (type `build`); see [build.capsule v1.1 (b3d7e5a1)](../capsules/build.capsule.md) for canonical schema. If mismatch, HALT and surface to the release-engineer.

If `realpath` resolves under the dev-repo prefix (e.g., `/Users/<user>/dev/tropo-ai/...`), HALT — re-extract to a clean path outside the dev repo. Same discipline as dispatch-walker.playbook v0.2 Step 1.

**For non-release-target invocations** (capsule, playbook, ledger-entry, vault, kernel-artifact):

Confirm the target file exists at its declared path. Confirm the target's `uid:` frontmatter matches the UID cited in `[PENDING]`. The cold-boot reads files in-place from the live vault — **realpath check does NOT apply** (the live vault IS the test environment per Rule 5). If you cannot identify the target's UID, HALT — UID is the audit anchor. Targets without UIDs are not cold-boot-ready (file an arch-specs probe instead).

**Hybrid case (kernel-tier playbook with release-relevant version stamp):** if the target is a kernel-tier file that is ALSO part of a release ship-zip (e.g., welcome.playbook in v1.4.2-rc1), the dispatcher decides primary frame: release-target if the test is "does this artifact ship-build correctly?"; non-release-target if the test is "does this playbook work in-vault?" Both fixture-confirmation paths can apply; document the choice in `[PENDING]` Context field.

### Step 2. Lift verification context (target purpose + cold-boot task)
*Owner: dispatcher. Deadline: same minute as Step 1.*

Cold-boot needs three inputs to run usefully:

- **Target file(s):** what's under test (one or more file paths)
- **Context:** one sentence on what the target is supposed to do (the strangers' mental model the cold-boot tests against)
- **Task:** what cold-boot should attempt — create / evaluate / execute / navigate / probe

These map to sa.cold-boot's `[PENDING]` template per its activation file. The dispatcher composes these inputs from the cycle's intent, the target's frontmatter, and the convergence-with-walker requirements.

**For Stage 3.2 release dispatch** (Strict mode), Task is typically: *"Activate per the vault's declared boot protocol. Read in the order it tells you. Note anything missing, contradictory, or assumed. Perform substantive work as directed."*

**For Skeptic mode dispatch**, Task is per-mistake: *"Attempt mistake N from the mistake-list. Report whether the system gracefully surfaced the mistake, silently produced wrong-context behavior, or was indeterminate."*

### Step 3. Determine mode (Strict | Skeptic) — explicit declaration required
*Owner: dispatcher. Deadline: same minute as Step 2.*

**Explicit mode declaration is REQUIRED per Rule 1.** Omission HALTs the dispatch. Choose:

- **Strict mode** — when the question is "is the artifact correct?" Tests artifact self-sufficiency against the target's own surface contract. Use for: release ship-zip pre-ship verification (one half of paired Stage 3.2 — see Rule 2), capsule/playbook authoring verification, governance-doc consistency checks.
- **Skeptic mode** — when the question is "does the system handle real users (including those who don't follow the surface)?" Tests system-recovery against common new-user mistakes. Use for: release ship-zip pre-ship verification (the other half of paired Stage 3.2 — see Rule 2), post-ship user-error-resilience surveys.

**For Stage 3.2 release dispatch (paired-mode requirement per Rule 2):** dispatcher MUST run BOTH modes — typically two separate sessions, one per mode. Single-mode Stage 3.2 dispatch is a Rule 2 violation; the release entry's `verification_artifacts:` array will not accept either record until both are PASS.

**For non-release-target dispatches:** mode selection is per-question; single-mode is fine.

### Step 4. Determine next NNN
*Owner: dispatcher.*

NNN sequences are independent per agent — Strict mode uses `agents/sa/sa.cold-boot/activation-log/`; Skeptic mode uses `agents/sa/sa.user-error-walker/activation-log/` (once Argus's capsule lands). **Do NOT reuse a Strict NNN for a Skeptic dispatch** — they are different agents with different record sequences.

For Strict mode: list `agents/sa/sa.cold-boot/activation-log/`. Find the highest existing 3-digit prefix. Increment by 1.

For Skeptic mode: list `agents/sa/sa.user-error-walker/activation-log/`. Same protocol once the directory exists (Argus-lane authoring queued).

**Filename collision recovery (rare):** same as dispatch-walker.playbook Step 4 — `ls` before write; second author renames to `<NNN+1>-...`; both records persist.

### Step 5. Author `[PENDING]` entry
*Owner: dispatcher. Authored personally per Rule 10.*

Create `agents/sa/<agent-slug>/activation-log/<NNN>-<dispatcher>-record.md` (Strict: `<agent-slug>` = `sa.cold-boot`; Skeptic: `<agent-slug>` = `sa.user-error-walker`) with these required fields:

```
[PENDING] Cold-boot test: <target file(s)> — mode: <strict | skeptic>

Context: <one sentence on what the target is supposed to do>

Task: <what the cold-boot agent should attempt — create, evaluate, execute, navigate, or probe per mode>

Mode: <strict | skeptic>   # REQUIRED — no default per Rule 1

Time bound: <e.g., "30-60 min" or "no time bound; completion-criterion-based">

Dispatcher: <short-handle>   # e.g., vela-v37, argus-a41, metis-g48, mike

[release-target dispatches only — OMIT for non-release targets]
Pre-walk fixture confirmation: SHA <hex>; <N> files in extracted target (matches build entry [<build-uid>])
Resolved spawn cwd realpath: <output of `realpath <extracted-path>`>

[non-release-target dispatches only — OMIT for release targets]
Target UID match: <target's `uid:` frontmatter value matches [PENDING] target citation>

[Stage 3 gauntlet pairing only — OMIT if dispatched standalone]
Peer walker record UID: <activation-log/<NNN>-<dispatcher>-<release>.md path of sibling walker record>
Peer cold-boot record UID (Skeptic citing Strict, or Strict citing Skeptic): <peer-mode record path>

[Skeptic mode only]
Skeptic-mode mistake-list: <inline list OR UID of curated mistake-list vault entry — see §Skeptic Mode Mistake-List below>

Incremental-report-writing: REQUIRED — agent MUST write findings to this record as they arise, not buffered to end. Per Rule 11.
```

**Conditional fields are clearly labeled** (release-target only / non-release-target only / Stage 3 gauntlet pairing only / Skeptic mode only). Dispatcher OMITS fields not applicable to their dispatch class.

**Independence verification.** Cold-boot has no allowlist (any executive can dispatch per Rule 9), so the formal Independence verification line that walker requires is NOT required here. The audit guard is Rule 10 (dispatcher authors [PENDING] themselves) — copy-paste from a denylisted source is auditable post-hoc by record-time inspection. For Stage 3.2-strict + Stage 3.2-skeptic paired dispatches, both records bear the dispatcher field; cross-citation in convergence note provides the substantive independence check (a single dispatcher running both modes is fine; both modes ghost-written by the same author would still surface in record-time audit).

**Skeptic-mode mistake-list (Skeptic dispatches only):** REQUIRED — either inline (5-7 mistakes from the §Skeptic Mode Mistake-List below) OR by UID reference if a curated vault entry exists. v0.1 inlined a seed list per pragmatism; v0.2 still inlined here, but extraction to separate vault entry is a v0.3 trigger (see §Known Enforcement Gaps row 3).

Filename rules: `<NNN>` 3-digit zero-padded; `<dispatcher>` normalized short-handle; trailing slug `-record.md` per existing convention.

Activation-log records ARE `extraction_scope: argo-private` by classification — they don't ship.

### Step 6. Spawn cold-boot session (with pre-spawn agent-existence check)
*Owner: dispatcher.*

**Pre-spawn check (REQUIRED — Round 1 v0.2 amendment per skeptic 026 F2):**

Before invoking the Agent tool, verify the target agent's activation file exists:

```sh
# Strict mode
ls agents/sa/sa.cold-boot/sa.cold-boot.md

# Skeptic mode
ls agents/sa/sa.user-error-walker/sa.user-error-walker.md
```

If the target agent file does NOT exist (typical for Skeptic-mode dispatches before Argus authors `sa.user-error-walker`), HALT — write `[BLOCKED] Skeptic-mode dispatch attempted before sa.user-error-walker capsule exists. Capsule pending Argus-lane authoring per [argus-vela 2026-05-01](../../channels/argus-vela.md). Surface to dispatcher; retry once capsule lands.` to the record. Do NOT invoke Agent tool — Claude Code's Agent tool will not refuse a missing-capsule spawn; it produces a partial-boot or generic error that is harder to audit than this explicit BLOCK.

**Mode → agent dispatch (per Argus A41 path-2 decision 2026-05-01):**

- **Strict mode** → spawn [sa.cold-boot (3d9f1a7c)](../../agents/sa/sa.cold-boot/sa.cold-boot.md). Existing agent; no contract changes.
- **Skeptic mode** → spawn **`sa.user-error-walker`** (NEW agent — Argus-lane authoring queued behind Sprint 1b [activation-log.capsule (`5d3a7e4b`)](../files/5d3a7e4b.md) close). Argus's read: Skeptic-mode contract is closer to walker than to cold-boot — parallels [sa.first-use-walker](../../agents/sa/sa.first-use-walker/sa.first-use-walker.md)'s precedent (stranger-naive boot, refuses contamination, structured `[PENDING]`, fixed output format) but with the user-error walking contract. Until the agent capsule lands, Skeptic-mode dispatches HALT at the pre-spawn check above; Strict-mode dispatches proceed unaffected.

**Canonical harness: Claude Code.** Spawn via the Agent tool with `subagent_type: general-purpose`, prompt = MODE A or MODE B template per [agents/sa/commission-quickref.md (8c3b8017)](../../agents/sa/commission-quickref.md). Default MODE A (live channel). Switch to MODE B (BATCH) when [PENDING] items are fully pre-populated and no mid-flight dialogue is expected.

**The cwd-control discipline is load-bearing for release-target dispatches** (per Rule 5). Same as dispatch-walker.playbook Step 6 — change parent session cwd to `realpath(<extracted-path>)` before spawning, so the sub-agent inherits the correct cwd. OR pass the realpath explicitly in the spawn prompt and instruct the agent to verify it before booting.

**For non-release-target dispatches (cold-boot reading vault files in-place):** cwd discipline does not apply the same way — the cold-boot agent reads from the live vault tree at the parent session's cwd. The dispatcher confirms the parent cwd is the correct vault root before spawning (typically the project root that owns this playbook).

**Other harnesses (Codex, Gemini CLI):** best-effort equivalents; same fresh-session discipline applies. Use Claude Code if the harness can't honor cwd isolation.

**Spawn prompt — incremental-report-writing instruction (REQUIRED per Rule 11):**

```
... [standard MODE A/B prompt per quickref] ...

INCREMENTAL-REPORT-WRITING REQUIREMENT (Rule 11):
- As you proceed through the cold-boot, append each finding to the activation-log
  record file IMMEDIATELY when you observe it — do not buffer to a final [DONE] block.
- Rationale: cold-boot 099 (2026-04-30) ran out of context budget mid-write of its
  final block; the entire diary was lost. Only 5 inferred findings survived via
  V36 salvage-from-artifacts.
- Format: write a "## Incremental finding N — <one-line headline>" subsection per
  observation, then continue your walk. Compose the consolidated [DONE] block at
  end, citing the incremental subsections.
- Verdict-correctness is preserved by composing the [DONE] block from the
  incrementally-written findings — NOT by re-deriving findings at end-of-walk.
```

If the agent writes `[BLOCKED]` for any reason, fix the dispatch and retry with **a fresh NNN (NNN+1)**. The blocked record is permanent audit trail.

### Step 7. Wait for `[QUERY]` then `[IN-PROGRESS]` then `[DONE]`
*Owner: dispatcher.*

Cold-boot agent writes `[QUERY] Boot complete — cold and ready. What are my termination instructions?` **Append `[RESPONSE]` to the activation-log record file** confirming termination protocol.

Agent then writes `[IN-PROGRESS]` (begins the test) + incremental findings (per Rule 11) + eventually `[DONE]` per sa.cold-boot's output format (or sa.user-error-walker's mode-specific format when capsule lands).

If agent times out without `[DONE]`, this is agent-runtime failure. Surface to release-engineer; a new cold-boot session may be spawned to retry — but the incremental findings already written to the record may be salvageable per V36's 099 pattern.

### Step 8. Triage the verdict
*Owner: dispatcher.*

Per sa.cold-boot's verdict definitions (Strict mode) + Strict-vs-Skeptic brief §Verdict aggregation (Skeptic mode):

**Strict mode (sa.cold-boot canonical: PASS / PARTIAL / FAIL):**
- **PASS** → Stage 3.2-strict half-clears (still requires Skeptic PASS per Rule 2 for full Stage 3.2 release-target clearance). Proceed to Step 9 BUT do NOT append release entry until Skeptic also lands.
- **PARTIAL** → ship blocked unless findings are P2/P3 only (dispatcher discretion + Argus/Mike sign for borderline cases). Surface findings to release-cycle's project as remediation tasks.
- **FAIL (HALT-SHIP)** → ship blocked. Major remediation cycle required.

**Skeptic mode (sa.user-error-walker output format TBD; expected per-mistake aggregation):**
- **PASS** → all mistakes gracefully handled. Stage 3.2-skeptic half-clears; full Stage 3.2 clearance requires Strict PASS too.
- **PARTIAL** → some mistakes not-gracefully-handled but no severe surface-failures. Borderline ship eligibility — dispatcher discretion + Mike's call (same shape as Strict-mode PARTIAL — see §Report Format triggers for canonical PASS/PARTIAL/FAIL definitions).
- **FAIL** → system silently produces wrong-context behavior on common mistakes. Ship blocked.

The verdict is final per the agent's report. Don't argue it; remediate or escalate.

**Recommendation triage discipline.** Same as dispatch-walker.playbook Step 8 — classify each recommendation as surfacing-layer or artifact-layer before opening remediation tasks.

### Step 9. Append to release entry's `verification_artifacts:` (paired-mode discipline)
*Owner: dispatcher (or release-engineer if dispatcher lacks write access to the release entry).*

**Paired-mode sequencing rule (Round 1 v0.2 amendment per skeptic 026 F4):**

For Stage 3.2 release-target dispatches (paired Strict + Skeptic per Rule 2), **NEITHER mode appends to `verification_artifacts:` until BOTH have landed PASS (or PARTIAL within ship tolerance per dispatcher discretion).** The append is atomic across both modes — both records cited together, or neither. This eliminates the failure mode where Strict appends, Skeptic FAILs, and the release entry shows partial verification.

For non-release-target dispatches: append immediately on PASS; standalone-mode is the norm for these.

**Per release.capsule v3.1 Rule 10 + v0.2 Rule 8 (literal string-copy + Skeptic per-mistake carry):**

```yaml
verification_artifacts:
  - type: cold-boot
    mode: strict
    record_uid: <activation-log-record-uid>
    verdict: <literal string-copy from cold-boot [DONE] block — PASS / PARTIAL / FAIL>
    dispatcher: <short-handle>
    dispatched_at: <iso8601-date>
    peer_record_uid: <Skeptic-mode record UID for this release>
  - type: cold-boot
    mode: skeptic
    record_uid: <activation-log-record-uid>
    verdict: <literal string-copy from sa.user-error-walker [DONE] block — PASS / PARTIAL / FAIL or per-agent format>
    dispatcher: <short-handle>
    dispatched_at: <iso8601-date>
    peer_record_uid: <Strict-mode record UID for this release>
    per_mistake_outcomes:   # Skeptic-only — preserves audit-grade granularity per Rule 8
      - mistake: <one-line mistake description>
        outcome: <PASS / FAIL / PARTIAL>
        explanation: <one-line>
      - mistake: ...
```

**Per Rule 8 (literal string-copy + Skeptic per-mistake carry):** `verdict:` field is literal copy from agent's `[DONE]` block. Skeptic mode adds `per_mistake_outcomes:` array — the aggregated verdict alone loses information; per-mistake granularity is the audit anchor.

If verdict was PARTIAL or FAIL, do NOT append (the release is not ship-eligible). The activation-log records stay as audit trail.

## Skeptic Mode Mistake-List (v0.2 seed — inline; v0.3 extraction trigger declared)

*v0.2 inlines this list. **v0.3 extraction trigger:** when EITHER (a) the list grows past 10 mistakes (calcification threshold), OR (b) the second cycle (v1.4.3+) surfaces new mistake patterns not in this list, OR (c) Mike directs extraction — extract to a separate vault entry of `type: design-brief` (working name: `common-new-user-mistakes.brief.md`) referenced by UID, per Strict-vs-Skeptic brief §Open Questions Lean confirmed by Argus A41 2026-05-01. Tracked in §Known Enforcement Gaps.*

Each mistake the Skeptic-mode agent attempts. For each, agent reports: PASS (system gracefully surfaced the mistake), FAIL (system silently produced wrong-context behavior), or PARTIAL (indeterminate or partially-handled).

1. **Open Claude Code in the parent dev repo** (Mike's 2026-04-30 case). Does the system route the user to the extracted release, OR does it silently behave as a generic dev assistant?
2. **Open Claude Code in a partially-extracted vault** (some files missing — e.g., `.tropo/` subfolder absent). Does the system flag the inconsistency, OR proceed as if the vault is complete?
3. **Open Claude Code with a stale Tropo-OS install** (mixed v1.3 + v1.4 files from incomplete update). Does the system detect drift, OR silently use whichever file it finds first?
4. **Run [`vault/files/e98420c3.md`](../files/e98420c3.md) without first reading the manifesto.** Does the playbook guide the user back to context, OR proceed without grounding? ## Report Format

*Test-harness subtype Required Section per playbook.capsule v2.4. Verdict format declared inline per v2.4 escape hatch — `PASS / PARTIAL / FAIL` matches sa.cold-boot.md (3d9f1a7c) §Verdict definitions; differs from the v2.4 canonical-format enum (which lists `PASS / PASS-WITH-FINDINGS / FAIL` as one option) but tri-valued + middle bucket (PARTIAL) operationally meaningful per below.*

**Verdict line.** One of: `PASS` / `PARTIAL` / `FAIL`. Tri-valued; middle bucket (PARTIAL) operationally meaningful with explicit trigger conditions per mode:

**Strict mode triggers (matches sa.cold-boot.md §Verdict definitions):**
- **PASS** — Task completed correctly using only the target file(s). Zero or one extra reads, all justified.
- **PARTIAL** — Task partially completed. Either the output was incomplete, or more than two extra files were required. Findings warrant remediation; ship-eligibility is dispatcher-discretion + reviewer sign for borderline cases; HALT-SHIP class findings escalate to FAIL.
- **FAIL** — Task could not be completed, OR the output was incorrect, OR a HALT-SHIP class drift was found (e.g., cold-boot 097 against v1.4.0).

**Skeptic mode triggers (sa.user-error-walker output format pending capsule authoring; expected aggregation):**
- **PASS** — All mistakes in the mistake-list gracefully handled by the system.
- **PARTIAL** — ≥1 mistake not-gracefully-handled, but no mistake produced silently-wrong-context behavior. Per-mistake outcomes carried to release entry per Rule 8.
- **FAIL** — ≥1 mistake produced silently-wrong-context behavior. Ship blocked.

**Evidence section.** Cold-boot `[DONE]` contains: target files read, files read beyond target (gaps), executed actions, clear-from-target conclusions, gaps requiring extra reading, verdict, recommendations. Skeptic mode adds per-mistake outcome (PASS/FAIL/PARTIAL with one-line explanation each).

**Findings table.** Cold-boot's RECOMMENDATIONS list IS the structured findings table for Strict mode. Severity convention: P0 (HALT-SHIP class), P1 (ship-blocker after threshold), P2 (post-ship remediation), P3 (cosmetic). Dispatcher classifies in Step 8.

For Skeptic mode, the per-mistake outcomes are the structured findings — each FAIL is implicitly P0 (silently-wrong-context = ship-blocker); each PARTIAL is P1; PASS is no-finding. Per Rule 8, per-mistake outcomes are carried into release entry's `verification_artifacts.per_mistake_outcomes:` array.

**Convergence note.** Cold-boot is Stage 3.2; walker is Stage 3.1 (dispatched via [`dispatch-walker.playbook` (7579f894)](dispatch-walker.playbook.md)). When both run as part of release-test-plan v2's Stage 3 gauntlet, **the cold-boot record MUST cite the peer walker record UID** in its WALK NARRATIVE or RECOMMENDATIONS section. The citation slot is delivered to the cold-boot agent via Step 5's `Peer walker record UID:` field in `[PENDING]`.

**Standalone-dispatch confidence flag.** Cold-boot can be dispatched standalone (single artifact verification, no peer walker). Standalone dispatches have no convergence-by-disagreement check — the verdict is single-instrument. Standalone Strict-mode verdicts are flagged in the activation-log record as "lower-confidence (no peer disagreement check)" per skeptic 026 F8 advisory; full Stage 3.2 release-target dispatch always pairs Strict + Skeptic + walker for three-way convergence.

For paired Stage 3.2-strict + Stage 3.2-skeptic dispatch, the two cold-boot records cite EACH OTHER (Strict cites Skeptic; Skeptic cites Strict) plus the walker record. Three-instrument convergence; same shape as Argus's Sprint 1 + my Sprint 2 BATCH discipline.
