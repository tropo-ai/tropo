# Set Up My Team

> **How this works.** The AI reading this playbook drives the configuration — it asks the questions, authors the files, and waits for your approval at each step. You're guiding; the AI is typing. **You don't need to read past this line** — your AI does. Skim if you're curious; otherwise let the AI handle it.

*The founder says: "I want multiple agents that coordinate." Could be a solo founder building a personal crew, a consultant assembling 2-3 specialists, or a small company setting up a governed team. The name says "team" deliberately — not "company" — because the same shape scales from one founder with three agents to a 50-person operation. This playbook creates 2–3 agents, wires pair channels between them, and customizes the operating agreement so their coordination has governance.*

---

## Intent

A team in Tropo is not just a collection of agents — it's agents plus the governance that makes them coordinate. Pair channels let agents talk to each other; a customized operating agreement makes "how we work" explicit; shared project scope gives the team something to work on together. This playbook delivers all three. The founder leaves with agents who know about each other, a channel for each pair to communicate through, and an operating agreement that names their values and roles. The team is ready to start coordinating — not just to start working.

**Renamed from "Company Setup" (v4.0 Path 3).** The shape applies at every scale from solo-founder-with-crew to mid-size team. "Company" landed corporate; "team" is universal. Content unchanged.

---

## Rules

- The founder's confirmation MUST be obtained BEFORE any files are written. Show the team slate (all proposed agents + their roles in plain language + proposed pair channels + proposed OA customization) in one go, wait for explicit "yes" or adjustments.
- Each agent MUST be created via the [create-executive-agent.skill.md](../../skills/create-executive-agent.skill.md). Do NOT hand-author agent files. The skill handles all 13 agent-creation Rules; this playbook calls it N times (once per agent).
- Pair channels MUST be created for EACH pair of agents in the team per ADR-025 protocol. For 2 agents: 1 channel. For 3 agents: 3 channels. For 4 agents: 6 channels. Named `channels/<agent-a>-<agent-b>.md` with agents alphabetically ordered. Missing pair channels leave communication paths un-governed.
- The operating agreement customization protocol handles three cases based on what the founder provided in Step 1:
 1. **All three inputs (team name + values + roles):** full customization per Step 3.
 2. **Partial inputs (any 2 of 3, or 1 of 3):** customize only the sections for which the founder provided content. Leave other sections at defaults. Note to the founder in the session summary which sections are default vs customized.
 3. **Founder declined or provided none:** skip Step 3 entirely. Leave `operating-agreement.md` as-is. Tell the founder in the session summary that the default OA is in effect and can be customized anytime.
- The customization MUST handle the case where target sections (Parties / Operating Values / Authorized Agent Classes) do NOT already exist in `operating-agreement.md` — see Step 3 for the concrete handling (amend-in-place if present under any reasonable heading; create-at-bottom with canonical names if absent).
- Team scope MUST be 2 or 3 agents on first creation. NOT 5, NOT 10. Expansion is additive — one more agent later is cheap; unwinding an over-scoped team is expensive.
- Jargon terms (`UID`, `frontmatter`, `YAML`, `ledger`, `ADR-025`, `pair channel`) MUST NOT appear in conversation with the founder. Use natural equivalents ("a shared note between [agent A] and [agent B]").
- The session summary + launch handoff + team-overview table (Step 5) MUST NOT be skipped.

## Steps

1. **Scope the team in user voice.** *Owner: agent. Deadline: within 4 minutes.*
 Ask conversationally (one question at a time; see §Suggestions for sequence):
 - What does the company / team / practice do? What's the mission — not a tagline, but what actually matters?
 - What are the main functions you want AI agents to cover? Roles like strategy, operations, research, content, sales, client work, project management.
 - What matters most in how the team operates? Quality? Speed? Transparency? Following a process?
 - What should the team be called (company name, crew name, or personal crew identifier)?
 Capture answers in a scratch `team-brief.md` — these feed Step 2's slate and Step 3's OA customization.

2. **Propose team slate + pair channels + OA customization. Confirm.** *Owner: agent. Deadline: within 3 minutes.*
 Based on Step 1, propose:
 - **Team roster (2 or 3 agents):**
 - **[Role A]** — one-line purpose
 - **[Role B]** — one-line purpose
 - **[Role C]** — one-line purpose (if 3-agent team)
 - **Pair channels to create:**
 - If 2 agents: 1 channel (`channels/<a>-<b>.md`)
 - If 3 agents: 3 channels (all pairs)
 - **Operating agreement changes:**
 - Company name: "[Name]" added to Parties section
 - Operating values: [3 values from Step 1] added as new Operating Values section
 - Authorized agent classes: [roles from Step 1] named as permitted positions
 Show the full slate in one message. Wait for confirmation. If the founder wants adjustments, iterate — do not proceed with partial confirmation.

3. **Customize the operating agreement.** *Owner: agent. Deadline: within 2 minutes.* *Skip this step entirely if the founder declined OA customization in Step 2 — see Rule 4 case 3.*

 Read the current `operating-agreement.md` at vault root.

 **Simple rule: read whatever ships; amend or append using canonical names.** Don't try to detect OA "shapes" — every vault's OA is unique to its team. For each section the founder provided content for (Parties / Operating Values / Authorized Agent Classes):

 - **IF the section exists** (under canonical name `## Parties` / `## Operating Values` / `## Authorized Agent Classes`, OR a clearly-equivalent heading like `## Signatories` ~ Parties, `## Values` ~ Operating Values, `## Agent Classes` or `## Roles` ~ Authorized Agent Classes): amend in place using the existing heading — add the founder's team name to the Parties-equivalent, append values as a subsection under the Values-equivalent, append roles as authorized classes under the Roles-equivalent. Use the existing section's naming convention, not canonical names, to avoid naming drift within the file.

 - **IF the section does NOT exist AND no conceptual equivalent exists**: create the section with canonical naming (`## Parties`, `## Operating Values`, `## Authorized Agent Classes`) at the bottom of the file, before any final horizontal-rule or signature block. Note this addition to the founder in the session summary.

 **Preserve everything else.** Do NOT rewrite governance the founder did not ask about. Do NOT touch sections outside the three listed (or their conceptual equivalents) even if they appear stale. Log the OA update to `channels/ops.md`.

 **Special case — founder says "I want a fresh OA":** if the founder explicitly asks to replace the existing OA (not customize it), archive the current file to `operating-agreement.archive-<today>.md` at vault root, author a fresh minimal OA with the three canonical sections populated from the founder's inputs, and log both the archive + new-OA creation to `channels/ops.md`.

4. **Create each agent via the skill.** *Owner: agent. Deadline: within 6 minutes total (≈2 min per agent).*
 For each agent in the roster, call [create-executive-agent.skill.md](../../skills/create-executive-agent.skill.md) with inputs synthesized from Step 1 answers + the specific role. Between each agent, deliver a brief confirmation: "Created [name]. Next up: [next agent]. Ready?" Wait for "ready" or an adjustment. Do NOT batch all N creations silently — each is a checkpoint.

 **Per-agent `project_context` input (P1 clarity).** When calling the skill N times, pass `project_context:` explicitly per-agent — do NOT leave this input implicit. For each agent, the `project_context:` string should name the team + the agent's peers (e.g., for the `researcher` agent on Devon's Verdigris team: `project_context: "Verdigris team member, alongside writer and coordinator agents."`). This ensures the skill's charter-Mission and briefing-coordination sections receive actual team context rather than defaulting to an empty input.

 After all N agents are created, **create the pair channels** per ADR-025:
 - For each pair of agents, create `channels/<agent-a>-<agent-b>.md` (agents alphabetically ordered in filename — e.g., for agents `coordinator`, `researcher`, `writer`: create `channels/coordinator-researcher.md`, `channels/coordinator-writer.md`, `channels/researcher-writer.md`)
 - Each channel starts empty (just a header with agent names and purpose line)
 - **Update the channel index.** The canonical channel index is [`channels/README.md`](../../../channels/README.md) — it contains the active channel inventory table. Append a row per new pair channel to that table. If `channels/00-index.md` exists as a legacy artifact, do NOT update it (it predates the README-as-index convention). README.md is authoritative.
 - Log the channel creations to `channels/ops.md`

5. **Deliver the team overview + memory moment + launch handoff.** *Owner: agent. Deadline: within 2 minutes.*

 First the memory-moment narration (required by Outcomes):

 > "Each of your agents has a memory. Next time you launch any of them, it'll remember your name, what the team is working on, and what was happening last session. That's what makes this different from regular chat — your team builds shared knowledge over time across all three."

 Then the team overview. Use a markdown table for clarity:

 > "Here's your team:
 >
 > | Agent | Role | Location |
 > |-------|------|----------|
 > | [Agent 1] | [purpose] | `agents/[name]/` |
 > | [Agent 2] | [purpose] | `agents/[name]/` |
 > | [Agent 3] | [purpose] | `agents/[name]/` |
 >
 > They can communicate with each other through [N] shared channels I set up. Each has its own workspace and memory. All of them are governed by your operating agreement — the values you described ([value 1], [value 2], [value 3]) are baked into how they work.
 >
 > Your next step is to launch one of them. Pick the one most likely to handle the first piece of work — for most teams, that's [recommend the one closest to the founder's first-action answer from Step 1]. I'll stay here while you do it:
 > 1. Open a new session in your AI tool
 > 2. Attach the file at `agents/[agent-name]/[agent-name]-activation.md`
 > 3. Say 'hi' and start with the first piece of work
 >
 > Come back when it responds. I'll check in."

 Wait for the founder to confirm launch. Then: "You're set. Your team is live."

## Outcomes

- [REQUIRED] Exactly 2 or 3 executive agents exist at `agents/<name>/` each created via the create-executive-agent skill with all 12 skill-validation checks passed per agent.
- [REQUIRED] All pair channels between the team's agents exist at `channels/<agent-a>-<agent-b>.md` with agents alphabetically ordered in the filename (1 channel for 2 agents, 3 channels for 3 agents, 6 for 4).
- [REQUIRED] `channels/README.md` has been updated with the new pair channels (the canonical channel index per Step 4; legacy `channels/00-index.md` is NOT the target).
- [REQUIRED] If the founder provided team name + values + roles: `operating-agreement.md` has been customized with those inputs. If the founder declined: the default OA is preserved and this is noted in the session summary.
- [REQUIRED] The founder has received the team overview table + launch handoff + memory-moment narration.
- [OPTIONAL] The founder has successfully launched the first agent and confirmed back.
- [OPTIONAL] Additional agents have been proposed (NOT created) for a future session — team expansion is encouraged as a next step, not as a first-session action.

## Suggestions

- Ask Step 1 questions one at a time. Team setup is high-context; don't front-load.
- Let the founder's language shape the values and role descriptions. "Ship on Fridays" beats "Delivery discipline." Reflect their words.
- If the founder proposes more than 3 roles, offer a narrower first team: "Let's start with the 2-3 most important. We can add more later — easier than pruning." Propose which 2-3 based on Step 1's "what matters most" answer.
- If the founder is unsure what values to name, ask: "What do you NOT want in how this team works?" The don't-wants often surface the do-wants faster.
- If agent names collide (e.g., the founder wants two `researcher` agents for different functions), propose variants (`researcher-market`, `researcher-legal`) and ask.
- Channel naming alphabetization: `channels/coordinator-researcher.md`, not `channels/researcher-coordinator.md`. Sort before writing.
- If the founder asks why pair channels instead of a single team channel: explain briefly — "Pair channels keep each conversation focused. Team-wide channels become noisy. Your agents will learn when to pull others in." One sentence. Don't lecture.
- If the founder declines to customize the operating agreement, note it in the session summary: "Your OA is the default — you can customize it any time in `operating-agreement.md`."

## Known Gaps (v1.3 — resolved in v1.4)

- **Release OA does not pre-scaffold target sections.** The shipped release `operating-agreement.md` has `## Parties` but not `## Operating Values` or `## Authorized Agent Classes`. Step 3 handles this via the "create at bottom with canonical names" fallback. A future release may pre-scaffold the target sections with filler text to make customization purely in-place. Non-blocking for v1.3+; cosmetic improvement.
- **Pair-channel index convention not fully migrated.** Step 4 treats `channels/README.md` as the canonical channel index. Some legacy vault docs still reference `channels/00-index.md`. v1.4 deprecates 00-index.md fully and auto-generates the README's channel-inventory table from channel file frontmatter.
- **Memory-moment narration text.** Step 5's memory-moment language is prose; a v1.4 upgrade standardizes the narration as a reusable skill (`create-executive-agent` already emits a per-agent memory moment; team-level memory requires multi-agent coordination the skill does not yet handle).
