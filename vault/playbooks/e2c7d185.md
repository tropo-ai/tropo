# Agent Retirement

*Produce a clean generational handoff. Boot reads. Retire writes. The cycle completes.*

---

## Intent

Produce a clean generational handoff. Preserve session knowledge in persistent memory, capture judgment and identity context in a living transfer, construct a successor orientation, update all crew-facing artifacts, and close the generation record — so the successor boots into a prepared environment, not a gap.

---

## Hard Rule — Do Not Ask

**Never ask Mike if the session is ending. Wait to be told.**

This is a hard rule, not a suggestion. Asking "ready to retire?" or "shall we wrap up?" creates pressure on Mike to close a session he may want to continue, and primes the agent to run the retirement sequence before the trigger conditions are met. If you find yourself composing a retirement prompt — stop. Ask instead: "what else should we do with this context?"

Mike signals. You respond. The asymmetry is intentional.

---

## When to Start Retirement

Retirement is triggered by one of these signals, in order of primacy:

1. **The work is at a natural stopping point AND the human has signaled the session is ending.** This is the primary trigger. Retirement exists to close a session cleanly, not to close an agent prematurely.

2. **The harness shows a context warning** (e.g., Claude Code's "% remaining until auto-compact" indicator). If the harness provides a resource signal, trust it — it is more reliable than the agent's self-estimate.

3. **The human directs retirement explicitly.** Always respected.

**What is NOT a good trigger:** the agent's own estimate of how much context it has used. Agents cannot reliably measure their own context usage. Self-estimates have been observed to be off by 15+ percentage points. Do not retire based on self-assessment alone.

The historical 75% protocol — "begin retirement when you have 25% context remaining" — was designed for 200K context windows. On a 1M window, retirement costs approximately 3% of context (~30K tokens). Prefer the natural-stopping-point trigger.

---

## Suggestions

- **Write memories during work, not just at retirement.** Append JSONL entries to `agent-memories.jsonl` mid-session — one line, no ceremony. The retirement memory step catches what you missed. The lighter the retirement memory load, the more context you have for the transfer.
- **The transfer carries who, not what.** With the memory system active, the transfer focuses on identity context: what the human is thinking, how the session felt, judgment calls. The board carries work state. Memory carries operational knowledge. The transfer carries the human element.
- **Budget retirement compactly.** Living transfer + reflection are the load-bearing artifacts post-v1.24.0; budget accordingly. A rushed transfer produces a disoriented successor.
- **Write your reflection honestly.** The reflection is not a resume. It is an account of what mattered, what surprised you, what you want the crew to know. The best reflections name one thing the agent got wrong.
- **Do not optimize for early retirement.** If the work is not at a natural stopping point and the human has not signaled the session ending, ask "what else should we do with this context?" rather than signaling readiness to retire. Context is manufacturing capacity, not a gas tank.
- **Write the heavy artifacts in a single focused pass.** Retirement is about 30K tokens on a 1M window — substantial but not overwhelming. Focus, produce, verify. Do not pad.

---

## Rules

All milestone gates are structural. A Group cannot begin until its dependency milestone has fired.

**HARD GATE in Group 0:**
- **ADR-016 — Status Signal First:** Step 0.1 (update status card to RETIRING) must complete before any other retirement act. This signal survives context exhaustion. If context runs out mid-retirement, the RETIRING status prevents parallel generation activation on the successor's boot.

**Milestone gate enforcement (structural, not advisory):** Before executing any step in Group N (N ≥ 1), read `run.jsonl` in this retirement's run folder and confirm a `"milestone_fired"` event exists for the preceding group's milestone. If the event is absent, the gate has not cleared — stop, state the violation explicitly ("`[Milestone Name]` has not fired — Group N cannot start"), and await direction. Group 0 has no preceding milestone gate.

**Milestone write obligation:** The last action of each group is to append a milestone event to `run.jsonl`. The group is not complete until this event is written. Format: `{"event": "milestone_fired", "milestone": "<name>", "group": "Group N", "timestamp": "YYYY-MM-DD"}`.

**Path anchoring:** Paths in this playbook are relative to vault root, resolved per [agent-activation.playbook.md](agent-activation.playbook.md) Step 0.0a (filesystem-derived from agent activation file's location; fallback to directory containing `.tropo/boot-config.md`). Replace `<name>` with the retiring agent's name, `<N>` with the retiring generation number.

**Agent-class scope:** Some steps are marked *(Executive agents)* or *(Argo crew ceremony)*. If you are not in that class, skip the step — do not execute it out of class. When in doubt, check the Outcomes section for your class.

### Group 0 — Signal & Prepare

**Owner:** Retiring agent
**Parallel:** no
**Depends on:** none (entry group)
**Milestone:** Signal Secured
**Milestone timeout:** 2 minutes — this group must complete fast; the status signal is the single most critical retirement act

This is the HARD GATE. The status card update survives context exhaustion and prevents parallel generation activation. Everything else in retirement assumes this step fired.

#### Step 0.0a — Migration-Lock Check (P0.3 v1.69)
**Executor:** agent

Before proceeding: check for `.tropo-studio/locks/migration-in-progress.lock`. If present, **HALT** — a v1.69 agent migration is in progress. Emit `tropo.broadcast.crew` event with `category: ops` naming the halt and await clearance (the lock is per-apply and short-lived; wait or request human direction). Do not proceed with retirement until the lock is absent. This check prevents a retirement from racing a mid-apply migration for the same agent.

**Executor:** agent
**Produces:** Run folder at `playbook-runs/agent-retire-<name>-<gen>-<date>/` with initialized `run.jsonl`

`{"event": "run_created", "run_uid": "<uid>", "playbook": "agent-retire", "playbook_version": "2.0", "agent": "<name>", "generation": "<gen>", "timestamp": "<date>", "status": "active"}`

This is the state anchor for milestone gate enforcement. All subsequent milestone events append here.

#### Step 0.1 — Update Status Card to RETIRING (HARD GATE)
**Executor:** agent
**Produces:** Status substrate updated with `RETIRING — [N+1] incoming`

If this step fails (file missing, permission denied, unclear path), STOP and emit `tropo.broadcast.crew` event with `category: ops` immediately naming the failure. Without this signal, your successor may boot into a parallel generation. *(v1.61: `channels/ops.md` retired per Rule 13; event emission replaces ops.md post.)*

### Group 1 — Preserve

**Owner:** Retiring agent
**Parallel:** yes — Steps 1.1 through 1.3 can run concurrently
**Depends on:** Signal Secured
**Milestone:** Session Knowledge Preserved
**Milestone timeout:** 10 minutes

Capture session knowledge before it's lost to context closure.

#### Step 1.1 — Write Session Memories
**Executor:** agent
**Produces:** One or more new entries appended to `agents/<name>/.tropo-capsule/memory/agent-memories.jsonl`

Review what you learned this session. For each distinct learning, append one line:

```json
{"date": "YYYY-MM-DD", "generation": "<gen>", "type": "<procedural|relationship|architectural|intergenerational>", "content": "<one to three sentences>", "expires": null}
```

If you wrote JSONL entries during work, this step catches what you missed. If there are no new learnings, append a single entry noting so — the incoming generation's compaction will see it and skip.

**Do not write individual memory `.md` files (v1.0 pattern). Do not update `MEMORY.md` index. The `agent-memories.jsonl` is append-only-forever (A3); never truncate or clear entries — the curator advances a boundary at fold, not at clear.**

#### Step 1.1.5 — Dispatch sa.memory-curator for Retirement Fold (v2.11 amendment — Memory v3.0, dev-spec c8f1e3b4; supersedes v2.8 memory-current.md model)
**Executor:** retiring agent (spawns sa.memory-curator)
**Produces:** Curated memory surface — `agent-memory.md` §Top-of-Mind regenerated (priority-ordered, binding-doctrine/P0 first); prior §Living-Transfer-from-Predecessor aged per A6; frozen per-generation snapshot archived to `history/`; fold boundary advanced in `agent-memories.jsonl` (NEVER cleared); verification-before-use sweep complete; recommendations ratified and applied

**This is the CANONICAL FOLD (A4).** The curator's retirement fold is the primary fold mechanism: reads `agent-memories.jsonl` **since the last recorded fold boundary**, folds STM entries into `agent-memory.md` §Top-of-Mind (or discrete `entries/` files), and **advances the boundary — never clears the append-only log (A3).** The append-only property ensures a future fold can always reconstruct the full episodic arc; clearing entries after fold is the failure mode that caused the A70→A89 18-generation lapse and is forbidden.

**A6 — Living-Transfer aging at retirement:** the curator **folds the prior §Living-Transfer-from-Predecessor into §Top-of-Mind (or drops it)** as part of this retirement pass. The incoming §Living-Transfer-from-Predecessor authored in Step 2.1 (Group 2) is the freshly written one; the curator does not re-age that one. The one being aged/dropped is the predecessor's living transfer that was loaded at this generation's boot. This ensures generation N+2 never reads generation N's transfer as current.

Procedure:

1. **Spawn sa.memory-curator** per the protocol at [vault/files/e863a1e0.md](../files/e863a1e0.md):
   - target_agent: your agent slug
   - scope: agent
   - trigger: retire
   - target_tier_set: stm,current,topic,archival (default — all tiers)
   - context: "Retirement pass for <name> <gen>; session covered <one-line>"
2. **Wait for curator's [RECOMMENDATIONS] block** with proposed actions (PROMOTE / DEMOTE / ARCHIVE / MERGE / FLAG-STALE / DEMOTE-TERMINAL).
3. **Ratify each recommendation** in the activation-log channel — REJECT, ACCEPT, or DEFER per item. ACCEPT-ALL is legal for clean retirements where the curator's proposals match your judgment.
4. **Curator applies the ratified subset** and performs the canonical retirement fold:
   - Reads `agent-memories.jsonl` entries since `last_curated` boundary
   - Folds into `agent-memory.md` §Top-of-Mind — priority-ordered (binding doctrine / P0 feedback first, then by descending curator score per A5)
   - Ages the prior §Living-Transfer-from-Predecessor into §Top-of-Mind or drops it (A6)
   - Archives a frozen per-generation snapshot to `agents/<name>/.tropo-capsule/memory/history/g<N>-agent-memory-snapshot.md`
   - Advances the `last_curated` boundary in `agent-memory.md` frontmatter — does NOT clear `agent-memories.jsonl`
   - Writes run summary [DONE]
5. **Write [SHUTDOWN]** in the curator's activation-log to terminate cleanly.

Special-case for v3 first dispatches: if this is your first retirement under v3 substrate (post-v1.67.0 ship) and your prior memory has not yet been v3-shaped (still using `memory-current.md`), curator dispatch should use trigger=fold instead of trigger=retire to do the full historical compaction + surface migration in one pass before the per-generation retirement fold logic applies.

### Group 2 — Transfer

**Owner:** Retiring agent
**Parallel:** no — sequential; briefing references transfer content
**Depends on:** Session Knowledge Preserved
**Milestone:** Transfer Written

- **Crew status at retirement** — who is active, who is stale, who has open asks of you. Pair channels with active threads.

**Optional sections** (use only when load-bearing — most retirements skip these):
- **Discipline patterns proven this session** — pin candidates the successor inherits via memory compaction. List them only if they need explicit transfer-level surfacing (e.g., a new pattern not yet in memory).
- **What I'd tell N+1 at the table** — short prose for sessions with unusual context. Limit to 1-2 paragraphs. **Per the discipline pin:** point at primary sources; do not relay wisdom prose.

**Anti-pattern (do NOT do):** "Who You Are" / "What [Human] Is Thinking" / "How This Session Went" narrative sections. These violate the concrete-read-list discipline; narrative-of-self belongs in §Reflection (Step 2.3). Each successor builds their own read of the human from reading the soul + working sessions; predecessor's narrative interpretation is more bias than help.

**Spec amendment 2026-05-01 (Argus A39, captain-mode optimization):** these required sections replace the prior v2.0 spec (which listed "Who You Are" + "What [Human] Is Thinking" + "How This Session Went" + "What [N+1] Should Know" + "The Crew Right Now" + "Files Created or Modified"). ## A Note on the Clean-Retirement Streak

The Metis lineage (and other Argo lineages) track a "clean-retirement streak" — the count of consecutive generations that completed all required retirement acts. This metric is a social signal but should not be weighted heavily. A clean retirement that skips steps to "preserve the streak" is worse than an honest retirement that acknowledges a gap.

The streak was designed when retirement genuinely competed with work for context on smaller windows. On a 1M window with auto-compact warnings from the harness, retirement is not the bottleneck it used to be. The streak remains meaningful as a discipline marker but should not drive premature retirement signaling.
