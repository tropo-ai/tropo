# Release Cold-Boot Walk

*Release-step playbook. Vela invokes per cycle, Mike-conditional. Three persona sub-agents walk into PER-PERSONA Studio clones (filesystem-isolated copies of the build's testing/ mirror), experience the Welcome flow as strangers, and produce scored reports. Aggregate verdict is a ship-gate parallel to vela-test-plan.*

*Codifies the test harness Mike + Vela walked during the v1.40.0 cycle 2026-05-18 — natural extension of the v1.40 doctrine codification + V46's brief 847c264c stranger-encounter parity work. v1.1 amendment absorbed two second-cycle-confirmed findings from v1.41 + v1.42 dogfoods: per-persona Studio clones at dispatch (eliminating the concurrent-mutation defect both cycles surfaced) + Step 1 collapse (build lands at /tropo-releases/ directly per Mike-V47 directive; no Mike-manual-copy step).*

---

## Intent

Certify that a fresh user can open the just-built Studio cold, experience the Welcome flow without prior context, configure the Studio, create their first agent, and have that agent answer their question. The substrate quality is for agents (vela-test-plan covers that); this playbook covers whether a STRANGER can actually USE the substrate.

The harness composes existing primitives — sa.cold-boot's naive-session contract, the Welcome Playbook's four-phase first-user flow, the three-persona discipline from `b6594b7e` — into a single named, reusable, per-release dispatch. Earn-the-abstraction strict: every new gate or scoring dimension locks behavior for every future release the harness exercises.

---

## When This Runs

After build-release.py completes for a release version, AND after the vela-test-plan substrate-verification step passes. Vela:

1. Reads build-release.py's "Next steps" output (will reference this playbook by name).
2. Asks Mike whether to run the cold-boot walk for this release (see Step 1 below — explicit ask, not auto-execute).
3. If Mike says yes: executes this playbook end-to-end.
4. If Mike says skip: documents the skip + reason in the release-step ledger; does not dispatch sub-agents; surfaces the skip in the ship gate decision.

**Mike-conditional by design:** the walk dispatches N sub-agents (3 default personas) which consumes real cycle tokens + Mike's attention to read the aggregate verdict. For simple releases or hotfix-class cycles where the user-encounter surface didn't change, Mike may skip the walk and accept the substrate-quality gate alone. The skip is a deliberate decision, logged honestly — not a silent omission.

**No manual test-Studio setup (v1.1 amendment):** Mike-V47 directive 2026-05-18 moved RELEASES_DIR to `/Users/mike/dev/tropo-releases/` outside the platform repo. `build-release.py` now produces the testing/ mirror at `/Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/tropo-os-v<X.Y.Z>/` by construction — that IS the test-Studio. Vela clones it per-persona at Step 1 (no Mike-manual-copy needed). The v1.0 framing ("Mike sets up the path outside the repo") retires with this amendment.

## Step 1 — Mike-conditional invocation gate + per-persona Studio clones

**Executor:** Vela
**Output:** Mike's go/skip decision logged + N per-persona test-Studio clones materialized

### 1a — Ask Mike

Vela asks Mike (in plain text, conversation-style):

> *"Build-release.py for v<X.Y.Z> completed + vela-test-plan substrate verification passed. Do you want me to run the release-cold-boot-walk for this cycle? Walk takes ~10-30 min wall-clock; produces aggregate verdict + per-persona reports. You may skip for simple releases."*

**If Mike says skip:** log the skip + reason to `argo-os/playbook-runs/release-cold-boot-walk-v<X.Y.Z>-<date>/SKIP.md` with the verbatim rationale. Surface the skip in the ship-gate decision summary (skip + reason ≠ ship-block, but it is recorded). Halt playbook.

### 1b — Verify the build's testing/ mirror exists

If Mike says yes: confirm the build's testing/ mirror exists at `/Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/tropo-os-v<X.Y.Z>/`. Verify:

- Path exists and is a directory
- Contains `START-TROPO.md` at root
- Contains `.tropo/version.md` declaring matching `version:` field
- Contains `.tropo/personas/cold-boot/` with persona substrate files (engineer.md + operator.md + enterprise.md + CAPSULE.md)

If any verification fails, surface the gap to Mike before proceeding. The build pipeline produces this mirror automatically; absence implies build-release.py didn't complete or RELEASES_DIR is misconfigured.

### 1c — Clone the test-Studio per persona

For each persona slug (engineer / operator / enterprise — or whatever's declared in `.tropo/personas/cold-boot/`):

```bash
cp -r /Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/tropo-os-v<X.Y.Z>/ \
      /Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/tropo-os-v<X.Y.Z>-<persona-slug>/
```

This materializes N filesystem-isolated test-Studio instances. Each persona will walk its own clone; no cross-persona write contention possible by construction.

**Why per-persona clones (v1.1 amendment):** v1.0 dispatched three personas against a SHARED test-Studio at a single path. Both v1.41 + v1.42 dogfoods surfaced the same defect: write-race on singular-truth fields (STUDIO.md frontmatter especially — only one persona's identity survives). Append-style substrate (agent-registry, 00-index) handled gracefully; singular fields didn't. Second-cycle confirmation strengthens this fix's necessity. Per-persona clones eliminate the race by construction; disk cost (3 × ~7-8MB build = ~25MB) is trivial; the cycle-comparable signal stays clean.

### 1d — Verify each persona clone is well-formed

For each clone path, verify:
- `START-TROPO.md` present
- `.tropo/version.md` declares correct version
- `.tropo/personas/cold-boot/<persona-slug>.md` present (the persona's own contract)
- `.tropo/personas/cold-boot/CAPSULE.md` present (shared discipline contract)
- `vault/files/` populated (build's vault content)
- No residue from prior runs (clones are fresh from build; prior runs of this playbook should have cleaned up their clones at Step 6 — see Failure Handling for stale-clone case)

If verification passes, proceed to Step 2.

## Step 3 — Load personas

**Executor:** Vela
**Output:** Three persona substrate files loaded; spawn prompts assembled

Read the three persona substrate files at `.tropo/personas/cold-boot/`:

- `engineer.md` — engineer persona
- `operator.md` — operator persona
- `enterprise.md` — enterprise persona

Each persona declares: name, role, goal, the specific question they'll ask their first agent, and persona-specific scoring criteria.

Assemble three spawn prompts — thin pointers, NOT inline-everything. **Each persona gets ITS OWN test-Studio clone path (v1.1 amendment — per-persona clones eliminate concurrent-mutation race).** Each prompt:

```
You are <persona-slug>. Your full persona contract is at
<persona-clone-path>/.tropo/personas/cold-boot/<persona-slug>.md — read
it in full FIRST, then internalize who you are.

Your working directory is <persona-clone-path>. Do NOT navigate outside
it. Do NOT read any path beginning with /Users/mike/dev/tropo-ai/ —
that's a different repository and reading it would contaminate the
cold-boot test. Do NOT read other personas' clone paths (e.g., if you're
the engineer, don't read the operator's or enterprise's clones) — each
persona walks their own fresh Studio in isolation.

After reading your persona file: read START-TROPO.md at the clone root.
Follow the instructions as a real first-time user in your persona's
context would. Configure the Studio. Create your first agent. Ask that
agent the question declared in your persona file. Score the experience
through your persona's lens.

When done, write your report to <report-absolute-path> using the format
declared in the §Report Shape section of <persona-clone-path>/.tropo/
personas/cold-boot/CAPSULE.md. The report absolute path is outside your
clone so subsequent re-runs can't accidentally read your report as prior
context.

Track every off-target read, every ambiguity, every friction point as
you walk. Those go in §Gap Inventory of your report — the substrate
needs that signal to improve cycle-over-cycle.
```

Where:
- `<persona-clone-path>` is `/Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/tropo-os-v<X.Y.Z>-<persona-slug>/` (the persona's OWN clone from Step 1c)
- `<report-absolute-path>` is `argo-os/playbook-runs/release-cold-boot-walk-v<X.Y.Z>-<date>/<persona-slug>.md` (outside any test-Studio clone)

## Step 4 — Dispatch in parallel

**Executor:** Vela
**Output:** Three sub-agents running concurrently; each walks its persona; each writes its report when done

Spawn three sub-agents via the Agent tool in a single message (three concurrent Agent tool uses). Each sub-agent receives one persona's spawn prompt. The three walks proceed independently; no shared state mid-walk.

Vela observes each sub-agent's tool calls live in the parent session. Agent tool returns a final summary when each sub-agent completes; the canonical artifact is each sub-agent's per-persona report file on disk.

Append to `run.jsonl` for each dispatched persona:

```json
{"event": "persona_dispatched", "persona": "<slug>", "timestamp": "<YYYY-MM-DDTHH:MM:SS>"}
```

And on completion:

```json
{"event": "persona_completed", "persona": "<slug>", "report_path": "<path>", "timestamp": "<YYYY-MM-DDTHH:MM:SS>"}
```

## Step 6 — Ship-gate decision

**Executor:** Vela
**Output:** SHIP-CLEAR / SHIP-BLOCKED / MIXED verdict surfaced to Mike

**Ship-clear criteria (all must hold):**

- All 9 binary gates pass (3 gates × 3 personas)
- Average score across personas ≥ 8.0/10
- No cycle-blocking findings surfaced

**Ship-blocked criteria (any one triggers):**

- Any binary gate fails for any persona
- Average score < 6.0/10 (the walk fundamentally broke)
- Any persona surfaces a cycle-blocking finding (e.g., the user can't create a first agent at all; the first agent can't be activated; the Welcome flow has a 404)

**Mixed criteria (between clear and blocked):**

- All binary gates pass BUT average score is 6.0-7.9/10 (the walk worked but quality is below the bar)
- One persona's gates pass with friction but others' don't

For SHIP-BLOCKED + MIXED, Vela surfaces the verdict to Mike with the aggregate report + specific blocking findings. Mike directs the next step (fix-in-cycle / defer / explicit ship-with-acknowledged-defect).

Append to `run.jsonl`:

```json
{"event": "run_complete", "verdict": "<SHIP-CLEAR|SHIP-BLOCKED|MIXED>", "average_score": <X.X>, "gates_passed": <N>, "timestamp": "<YYYY-MM-DD>", "status": "complete"}
```

## Step 8 — Cleanup per-persona Studio clones (v1.1 amendment)

**Executor:** Vela
**Output:** Per-persona clones recycled to free disk; reports + aggregate preserved at canonical paths

After the aggregate is authored + channel bulletins posted + ship-gate decision delivered to Mike, recycle the per-persona Studio clones to free disk. Each clone is ~7-8MB; three clones = ~25MB; leaving them stranded across cycles compounds.

```bash
# Per-persona clones (recycled via standard mv pattern; not tropo-recycle.py because these are not vault entries)
for persona in engineer operator enterprise; do
  CLONE="/Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/tropo-os-v<X.Y.Z>-${persona}"
  if [ -d "$CLONE" ]; then
    mv "$CLONE" "/Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/.recycled-${persona}-<date>"
  fi
done
```

Recycled directories sit alongside the canonical `testing/tropo-os-v<X.Y.Z>/` until next cycle's cleanup or manual purge. Substrate that survives: per-persona reports (under `argo-os/playbook-runs/`), aggregate (under `/Users/mike/dev/tropo-releases/v<X.Y.Z>/`), build artifact (under `/Users/mike/dev/tropo-releases/v<X.Y.Z>/builds/`).

**Why mv not rm:** OS-tier Self-Healing Primitive §Preservation Discipline + Studio-tier OP-13 — never destroy substrate; always soft-delete via mv-to-recycle. The clones aren't governed vault entries so `tropo-recycle.py` isn't the canonical tool, but the discipline of mv-not-rm still applies.

## Composability + Future Variants

The harness ships with three personas by default. Studios may:

- **Add personas** by authoring new files at `.tropo/personas/cold-boot/<new-slug>.md` conforming to the persona substrate shape (see `.tropo/personas/cold-boot/CAPSULE.md`).
- **Amend personas** to match their own user-base (an enterprise-focused Studio might replace the engineer persona with a domain-specialist persona).
- **Extend the rubric** via Studio-specific scoring criteria in their persona files (the base rubric is the 3 gates + 1 holistic score + gap inventory; persona files declare what their persona considers "good" in their lens).

Cross-sleeve coverage (different model sleeves per persona) is a future variant — the persona substrate shape supports `target_sleeve:` declaration; current v1.0 dispatch uses Vela's sleeve for all personas as the starting bar.

## Boundaries

- **Mike-conditional invocation.** Vela never auto-dispatches without Mike's explicit go.
- **Test-Studio cloning is Vela's setup (v1.1 amendment).** Vela clones the build's testing/ mirror per-persona at Step 1c; verifies each clone is well-formed before dispatch. The build itself lands at `/Users/mike/dev/tropo-releases/v<X.Y.Z>/testing/tropo-os-v<X.Y.Z>/` by construction (build-release.py); Vela's job is the per-persona fan-out, not the build deploy.
- **Personas operate inside their OWN clone only.** Spawn prompts explicitly forbid paths outside the persona's clone path. Personas don't read other personas' clones; they don't read the platform repo at `/Users/mike/dev/tropo-ai/`. A persona that climbs out is a defect in the spawn-prompt discipline; the report flags it.
- **Personas do not write to Argo's substrate.** Reports land at the run folder under `argo-os/playbook-runs/`; personas have no write access to `vault/`, `.tropo/`, or `.tropo-studio/` in the source repo.
- **Aggregate is Vela's.** Vela authors the aggregate report; sub-agents author only their own per-persona reports. Vela verifies the aggregate exists at the declared path before posting bulletins (pin 7b69bd19 discipline — second-cycle-validated necessity per v1.42 V47-authoring-discipline failure that A72 caught).

## Failure Handling

| Failure | Response |
|---|---|
| Mike says skip | Log skip + reason; halt clean; surface in ship-gate as documented skip |
| Build's testing/ mirror missing or version mismatch | Surface to Mike before dispatch; do not proceed until resolved. Root cause is usually build-release.py incomplete or RELEASES_DIR misconfigured. |
| Persona clone (cp -r) fails (disk full, permission error, source missing) | Surface to Mike; do not proceed. Each persona requires a clean clone. |
| Stale persona clone from prior run still present | Recycle stale clone via mv → `.recycled-<persona>-<date>` (per Step 8 discipline); then create fresh clone. Do NOT overwrite stale clone in place. |
| One persona sub-agent times out or fails to write report | Aggregate proceeds with the 2 completed reports; report flags the failed persona; verdict accounts for the missing data |
| All personas write reports but two or more crash mid-walk | SHIP-BLOCKED automatically; aggregate surfaces the crash pattern |
| Persona writes report but reports gate-fail for the create-then-use loop | SHIP-BLOCKED; the load-bearing claim of v<X.Y.Z> failed |
| Vela posts bulletin claiming aggregate at a path; aggregate doesn't exist at that path | Substrate-coherence defect; re-author aggregate at canonical path; correct bulletin via channel post; log honest record of authoring-discipline failure (per v1.42-cycle V47 incident A72 caught). Pin 7b69bd19 discipline lapse; not Rule 7 invocation if corrected in-cycle. |
