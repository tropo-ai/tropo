# Fleet Ops â€” Scheduled Agent Dispatch

*Lightweight playbook. Read the registry, check last run dates, dispatch any due agents as sub-agents, scan for urgent findings. Run at boot and on demand.*

---

## Intent

The ops fleet keeps the vault healthy â€” channels clean, freshness monitored, integrity checked, repairs applied. This playbook is the scheduler. It replaces the Cowork cron mechanism with a vault-native, agent-driven dispatch model that works regardless of whether Mike's laptop is running a scheduled process.

---

### Step 2 â€” Determine what's due

For each enabled agent, check whether it should run today:

**Daily agents:** Check the most recent activation record in `agents/sa/sa.<name>/activation-log/`. If the most recent record was created today (filename date or `*Spawned by: ... | Date: YYYY-MM-DD*` header matches today), the agent already ran â€” **skip**. If no record exists today, the agent is **due**.

**Monday-only agents (`sa.vault-integrity-auditor`, `sa.governance-validator`):** First check if today is Monday. If not Monday, skip. If Monday, apply the same activation-log check as daily agents.

**`sa.daily-vault-health` shortcut:** Check if `shared/orientation/daily-health-report.md` has `report_date` matching today. If yes, skip.

**Result:** A list of agents to dispatch and a list to skip.

---

### Step 3 â€” Dispatch due agents (in order)

Execution order matters. `repair-agent` reads the other agents' reports and must run after them.

**Group A â€” dispatch in parallel (or sequence):**
1. `daily-vault-health`
2. `vault-janitor`
3. `freshness-monitor`
4. `channel-health-monitor`
5. `vault-integrity-auditor` (if Monday and due)
6. `governance-validator` (if Monday and due)

**Group B â€” dispatch after Group A completes:**
7. `repair-agent`

For each due agent in the group, spawn as an SA using the activation-log pattern:

1. Determine the next record number for `agents/sa/sa.<name>/activation-log/`
2. Create the record file: `agents/sa/sa.<name>/activation-log/NNN-fleet-ops-record.md`
3. Write the record header: `# sa.<name> â€” Activation Record NNN` / `*Spawned by: fleet-ops | Date: YYYY-MM-DD*`
4. Spawn the agent with its SA activation file as the prompt, passing the record path
5. No termination response needed â€” all ops fleet SAs are `archetype: one-shot` and self-terminate after writing `[DONE]`

Results land in the activation-log record. Any urgent findings are emitted as `tropo.broadcast.crew` events (`category: ops`, or `severity: flash` for urgent) by the SA itself (`channels/ops.md` + `channels/alerts.md` retired per Rule 13).

If an agent is skipped (already ran today), note it. Do not dispatch.

If no agents are due (all ran today), skip Step 3 entirely and note "Fleet current â€” no dispatch needed."

---

### Step 3B â€” Dispatch stalled playbook runs (the autonomic layer)

After the ops fleet dispatch, check for playbook runs that need resumption. This is the autonomous nervous system layer â€” paused runs restart themselves because the governance says so.

**Read:** `.tropo-studio/registries/playbook-runs.jsonl`

For each entry where `status: "paused"`:

1. Read the corresponding run folder's `run.state.json`
2. Check `pause_since` against the current time and the playbook's milestone timeout
3. If the pause timeout has been exceeded AND the external trigger has been received (human responded, system callback recorded in run.jsonl) â†’ **dispatch the playbook executor:**
   ```
   Spawn: agents/operations/playbook-executor/activate.md
   With: "Run UID: <run_uid> | Vault root: <path>"
   ```
4. If the pause timeout has been exceeded but NO trigger received â†’ emit a `tropo.broadcast.crew` event (`category: ops`) with a WARNING (`channels/ops.md` retired per Rule 13):
   ```
   [fleet-ops] STALLED RUN | Run {run_uid} | Paused since {pause_since} | Waiting: {pause_reason} | Milestone timeout exceeded â€” human attention needed
   ```
5. If within timeout window: skip silently

**If no stalled runs:** skip this step.

---

### Step 3C â€” Dispatch scheduled playbooks (v2.1 addition)

After sa.* dispatch (Step 3) and stalled playbook-run check (Step 3B), scan for **scheduled playbooks** that are due.

**Scan scope:** `playbooks/*.playbook.md` files (vault-level playbooks; not `.tropo/playbooks/` kernel).

For each playbook, read frontmatter and check:

1. `state: active` â€” playbook is currently activated (skip if `state: archived` or absent)
2. `cadence:` declared â€” string value (`monthly` = 30 days; `weekly` = 7 days; `daily` = 1 day; OR explicit `cadence_interval_days: <N>` integer)
3. **Compute last_run** â€” find most recent folder matching `playbook-runs/<playbook-uid>-*` (or `playbook-runs/<playbook-name>-*`); folder mtime = last_run. If no folder exists, treat as "never run" â†’ DUE.
4. **Compute due-ness** â€” if `(today - last_run) >= cadence_interval_days`, playbook is **DUE**.

**For each DUE playbook, emit a `tropo.broadcast.crew` event (`category: ops`) with the bulletin (`channels/ops.md` retired per Rule 13):**

```
[fleet-ops] PLAYBOOK-DUE | <playbook-title> (<uid>) | last_run: <date or "never"> | cadence: <cadence> | days_overdue: <N> | Vela run this session.
```

**Do NOT auto-execute the playbook itself.** Fleet-ops surfaces the due signal; Vela picks it up during boot/session and executes per the playbook's own protocol. This preserves the "Vela owns Build / executive owns execution" lane discipline.

**Convention for the playbook-run folder:** when Vela executes the playbook, she creates `playbook-runs/<playbook-name>-<date>/run.jsonl` (mirrors agent-activation run pattern). The new folder's mtime then becomes the next "last_run" anchor.

**Currently-scheduled playbooks** (as of v2.1 ship 2026-05-05):

| Playbook | UID | Cadence | First scheduled run |
|---|---|---|---|
| `vault-pruning.playbook` | 2e0e56a1 | monthly (30 days) | 2026-06-04 (30 days from 2026-05-05 inaugural execution) |

Future playbooks adopting the convention: amend their frontmatter to include `state: active` + `cadence: <interval>`. No registry registration needed â€” fleet-ops scans `playbooks/` directly. (This differs from sa.* which require explicit registry registration; the simpler scan-by-frontmatter pattern earns its place here because playbooks are fewer and self-contained.)

---

### Step 4 â€” Scan for urgent findings

After all dispatches complete, query the event log for `severity: flash` crew events (`query-events --type tropo.broadcast.crew`; `channels/alerts.md` retired per Rule 13). Look for any new events emitted during this run (timestamp = today's date).

- **No new alerts:** Note "No new alerts â€” fleet ran clean."
- **New FLASH or CRITICAL alerts:** Surface them immediately to the executive. These require attention before the startup signal or before returning to session work.

---

### Step 5 â€” Emit dispatch summary

Emit a `tropo.broadcast.crew` event (`category: ops`) with a one-line summary (`channels/ops.md` retired per Rule 13):

```
[fleet-ops] DISPATCH | [DATE] | Ran: <agent list>. Skipped (current): <agent list>. Alerts: <N new>. Reports at agents/operations/<name>/reports/[DATE].md.
```

If nothing was dispatched: `[fleet-ops] STATUS | [DATE] | Fleet current â€” all agents ran today. No dispatch needed.`

---

## Triggering

**At boot:** Run during Phase 3 (Operational Grounding) of the boot playbook, after channel scan, before startup signal. This ensures any missed runs are caught before the executive engages the human.

**On demand:** Any executive can dispatch this playbook directly as a sub-agent. Use when vault health status is uncertain mid-session or after a gap in operations.

**Periodic (session):** During sessions longer than ~4 hours, re-run at natural pause points. No automated intra-session trigger is implemented in v1.0 â€” this is a manual discipline until a session-timer mechanism is established.

## Failure Handling

| Failure | Response |
|---------|----------|
| Registry file not found | **Stop.** Emit a `tropo.broadcast.crew` event with `severity: flash` â€” fleet-ops could not read registry (`channels/alerts.md` retired per Rule 13). Also emit a `tropo.broadcast.crew` event (`category: ops`) with file path and error. Do not fall back to the embedded convenience table â€” it is not the source of truth. |
| Registry unparseable (malformed frontmatter or table) | **Stop.** Emit a `tropo.broadcast.crew` event with `severity: flash` â€” fleet-ops could not parse registry (`channels/alerts.md` retired per Rule 13). Include the parse error. Also emit a `tropo.broadcast.crew` event (`category: ops`). |
| Agent activate.md not found | Skip that agent. Emit a `tropo.broadcast.crew` event (`category: ops`) naming the agent and the missing path. Continue with remaining agents. |
| Sub-agent dispatch fails | Note failure in dispatch summary. Emit a `tropo.broadcast.crew` event (`category: ops`) with the agent name and failure reason. Continue to next agent. |
| Report file write fails (inside dispatched sub-agent) | Sub-agent handles â€” not fleet-ops's concern. |
| All agents already ran | "Fleet current" â€” no dispatch, no error. Emit a single `tropo.broadcast.crew` event (`category: ops`) for observability. Normal at mid-session re-run. |
| Playbook itself errors during dispatch loop (unhandled exception) | **Stop.** Emit a `tropo.broadcast.crew` event with `severity: flash` â€” fleet-ops aborted (`channels/alerts.md` retired per Rule 13). Include stage reached (which agent was being processed). Also emit a `tropo.broadcast.crew` event (`category: ops`). Parent executive must continue boot â€” fleet-ops failure MUST NOT block activation. |

**Visibility rule:** every fleet-ops run â€” successful or not â€” emits at least one `tropo.broadcast.crew` event (`category: ops`) to the event log (`channels/ops.md` retired per Rule 13). Silent success is not acceptable. An executive who boots and sees no fleet-ops event in the event log should treat that as a signal the dispatch never ran.

## Updating the Fleet

When adding a new scheduled agent:
1. Create `agents/sa/sa.<name>/` with SA activation file and `activation-log/` folder
2. Add a row to `agents/sa/README.md` catalog
3. Add a row to `agents/operations/scheduled-agent-registry.md`
4. Update the convenience reference table in Step 1 of this playbook
5. Ensure the SA is `archetype: one-shot` and writes `[DONE]` when complete
