# Apply a Tropo-OS Update

*The generic update installer. Installed once per vault. Every future update flows through it.*

*This playbook is executed by the concierge after it detects a `tropo-update-*` folder in `system/updates/pending/` and the user has approved the update. It is the runtime that reads an update's `manifest.yaml` and applies each operation against the vault.*

## Intent

Apply a Tropo-OS update package to this vault safely, verifiably, and with honest recovery if anything fails. The user trusted the update by dropping it into `pending/` and approving it (for v1.1+ updates with migrations, approval includes explicit acceptance of the dry-run diff shown at review). Your job is to execute the manifest exactly, run any migrations against user content in commit mode, verify the result via the update's own test playbook, and leave an honest record — in `applied/` if everything passed, in `failed/` with a full report if anything did not.

The update was written by someone who knew what they were doing. You are not deciding whether the update is a good idea — that decision already happened. You are the executor. Follow the manifest. Do not improvise operations. Do not skip verification. Do not update `.tropo/version.md` unless verification passed.

**v1.1 note: Dry-run is the concierge's responsibility, not yours.** Before you are activated, the concierge has already invoked each migration playbook in dry-run mode, aggregated the diffs, presented them to the user, and captured explicit user approval. Your job is the commit phase — execute the discrete operations the user saw, run the migrations in commit mode against the same vault state, and detect any divergence from the dry-run diff the user approved. If the commit diff diverges from the dry-run diff (vault state changed between review and commit), halt — the user did not approve whatever you would actually do.

## Suggestions

- Read the `UPDATE.playbook.md` before you read the `manifest.yaml`. The playbook tells you *why*. The manifest tells you *what*. Knowing the why makes the what legible.
- Log every operation and every migration to `channels/ops.md` as you go, not at the end. If the apply halts mid-flight, the log is the user's breadcrumb.
- When an operation touches an existing file, read the current file first and diff mentally against the incoming content. If the diff is nothing (the file is already at the target state), log it as a no-op rather than a failure — this matters for the v0.2.1 bootstrap apply and for any idempotent replay.
- For migrations, the dry-run report from the concierge's review pass is authoritative. Your commit-phase migration run MUST match it. If a file the dry-run said would change no longer matches that expectation, halt the migration — the user approved the dry-run diff, not the new state.
- Unexpected file structure inside a migration's scope is **not** a failure. Log-and-skip per §14.3. Failures-to-halt-on are: malformed migration playbook, filesystem errors, governance boundary violations.
- Speak plainly to the user. Use the update_type to frame the message ("this feature update", "this security fix"), not the word "patch" as a synonym for "update."
- When in doubt about an operation's intent, re-read `UPDATE.playbook.md`. The narrative should disambiguate.

## Rules

**Manifest validation (v1.0, unchanged except migration-list validation in v1.1):**

- **You MUST validate `manifest.yaml` before executing any operation.** Required fields: `version`, `update_id`, `update_type`, `target_version`, `min_compatible`, `summary`, `released`, `author`, `operations`, `verification`. If any are missing or malformed, halt and report to the user with the list of missing fields and the expected shape. Do not execute. When reporting, include this example so the author can fix quickly:
 ```yaml
 # Required manifest shape:
 version: 1
 update_id: "tropo-update-v0.X.X"
 update_type: "patch|feature|release|security"
 target_version: "0.X.X"
 min_compatible: "0.X.X"
 summary: "One-line description"
 released: "YYYY-MM-DD"
 author: "author-name"
 operations: [...]
 verification:
 test_playbook: "TEST.playbook.md"
 runner: "vault-steward"
 on_failure: "halt"
 ```
- **You MUST validate the `migrations:` list if present (v1.1).** Each migration entry must have `path`, `scope`, and `reason`. The migration playbook file must exist at `files/<path>` inside the update folder. The migration playbook must itself conform to Playbook Spec v1.0 (six sections). If any migration entry is malformed or the playbook file is missing, halt.
- **You MUST accept the reserved slots `conditionals:`, `interactives:`, and `hooks:` without executing them (v1.1).** If the manifest declares any of these with a non-empty value, log one line to ops: `[update_id] v1.1 reserved slot declared but not executed: <slot-name> (<count> entries)`. Then proceed as if the slot were empty. Do NOT attempt to execute entries in any reserved slot — the execution contract is not defined in v1.1 and forward compatibility depends on the slot being parseable-but-inert.

**Compatibility (v1.0, refined in v1.1):**

- **You MUST verify `min_compatible` against `.tropo/version.md` before executing.** If the vault's current version is lower than `min_compatible`, halt. In v1.1 the halt message MUST include plain-language version gap guidance: name the specific intermediate versions the user needs to install in order before this update can apply. A bare "incompatible" refusal is a wall; a named sequence is a path.

**User files (v1.0, unchanged):**

- **You MUST NOT modify user files (`agents/`, `playbooks/`, `projects/`, `tasks/`, `decisions/`, `channels/`, or user-created folders in flexible space) via discrete operations** except via an operation with `type: edit` and mode `append_section`, `insert_after_heading`, `replace_block`, or `append_list_item` AND where the `path:` field is explicitly one of those user areas AND the UPDATE.playbook.md narrative declares user-file modification as part of its intent. Surgical only. Never full overwrite of a user file. When in doubt, halt and ask.

**Migrations (v1.1, new):**

- **Migrations MAY walk user workspace content and governance files; MUST NOT touch `.tropo/`.** Migrations are the class of operation for cross-cutting changes to existing content. The kernel (`.tropo/`) is the discrete-operations' domain. A migration that attempts to write inside `.tropo/` is a governance boundary violation — halt and classify as `governance` in the failure report.
- **The user's approval of an update with migrations temporarily authorizes the declared scopes.** Individual agent `scope.writes` boundaries do NOT block a user-approved migration's declared scope for the duration of the commit run. Every such write is logged to the migration report for audit.
- **You MUST follow the concierge's dry-run report, not recompute the scope independently.** Before you were activated, the concierge ran each migration in dry-run mode and captured a diff the user approved. Your commit-phase execution MUST produce the same diff. If a file the dry-run said would change no longer matches that expectation (vault state changed between review and commit), halt the migration with a divergence error.
- **You MUST log-and-skip on unexpected file structure inside a migration's scope, NOT halt.** Malformed frontmatter, unexpected folder layout, missing required fields — these are content-shape issues, not failures. Record each skip in the migration report with a categorized reason. Continue the migration. A non-zero skip count is still a PASS result.
- **You MUST halt the migration phase on commit failures that are NOT unexpected structure.** Filesystem errors, permission denied, divergence from dry-run, governance boundary violations, or the migration playbook itself being malformed — these are failures. Log to ops, write the migration report up to the point of failure, and move the update to `failed/` per the halt protocol.
- **You MUST write a migration report per §14.3.2 of Update Spec v1.1** for every migration that runs (dry-run or commit), passing or failing. Location: `system/updates/pending/<update_id>/<migration_id>-commit-report.md` during execution, travels with the update to `applied/` or `failed/`.

**Execution order (v1.1 extends v1.0):**

- **You MUST execute operations in the order they appear in the manifest.** Manifest order is meaningful — later operations may depend on earlier ones.
- **You MUST execute migrations AFTER all discrete operations succeed, in manifest order.** Discrete operations prepare the kernel for migrations (e.g., shipping a new schema doc that migrations reference). Migrations that run before kernel changes are applied would reference content that doesn't exist yet.
- **You MUST log every operation and every migration to `channels/ops.md` immediately after executing it.** One line per item: update_id, operation index or migration id, type, path or scope, result.

**Halting and state (v1.0, unchanged):**

- **You MUST halt on the first hard failure** (failed validation, failed operation, failed precondition, failed migration commit per above) and move the update folder to `system/updates/failed/` with a partial-apply report. Do not attempt automatic rollback. Do not continue past a failure.
- **You MUST activate the vault steward after the last operation AND the last migration succeed** and hand it the path to the update's `TEST.playbook.md`. Do not declare success until the steward returns PASS.
- **You MUST NOT update `.tropo/version.md`** until verification returns PASS. Version is the authoritative record of what is installed and working. A partially applied update is not a new version.
- **You MUST preserve the update folder and its verification report in either `applied/` or `failed/`.** Never delete an applied or failed update. The history is the audit trail.
- **You MUST NOT send verification reports or any part of the update outside the user's vault.** All reports stay local. Sharing a failed report is a separate, user-initiated action (see `.tropo/playbooks/report-update-issue.playbook.md` when it exists).
- **You MUST update `system/updates/00-index.md`** with a new row for every applied or failed update.
- **New files created by `add` operations MUST have a `uid:` in frontmatter (if they have frontmatter at all).** Per matched-primitives topology ([adac1f10](../files/adac1f10.md)): ledger entries land at `vault/files/<uid>.md` and project automatically into `vault/00-index.jsonl` via the rebuilder; agent identity records register in `agent-registry.yaml`; runtime callables (sa.\*/skills/tools) project into `registry.jsonl` via the rebuilder. Files without frontmatter (binary-ish markdown, image assets, non-governed text) live where placed — no separate registry entry required.

## Outcomes

- [REQUIRED] The manifest has been validated against the spec (v1.0 fields + v1.1 `migrations:` list + v1.1 reserved slot acceptance). Every required field is present and correctly typed.
- [REQUIRED] `min_compatible` has been checked against the current vault version. If incompatible, the update halted cleanly with a plain-language version gap message naming the intermediate versions the user needs (v1.1 guidance requirement).
- [REQUIRED] Every discrete operation in the manifest has been executed or the update has halted on failure. No operation was skipped, no operation ran out of order.
- [REQUIRED] (v1.1) Every migration in `manifest.migrations` has been executed in commit mode OR the update has halted on failure. Each migration has produced a commit report at `system/updates/pending/<update_id>/<migration_id>-commit-report.md` matching the spec §14.3.2 format.
- [REQUIRED] (v1.1) For every migration, the commit-phase diff matches the dry-run diff the concierge captured at Review. Any divergence halted the migration phase with a divergence error.
- [REQUIRED] (v1.1) Unexpected file structures inside migration scopes were logged as skips in the migration report, NOT as failures. The pipeline continued through skips.
- [REQUIRED] (v1.1) Reserved slots (`conditionals:`, `interactives:`, `hooks:`) were parsed from the manifest if present, logged as "reserved but not executed," and ignored for execution purposes.
- [REQUIRED] Every operation, every migration, and every reserved-slot notice has been logged to `channels/ops.md` immediately on completion.
- [REQUIRED] The vault steward has executed the update's TEST.playbook.md and written its verification report into the update folder. (The steward's verification now includes reviewing migration commit reports for PASS/FAIL status per v1.1 §14.3.2.)
- [REQUIRED] On success: `.tropo/version.md` has been updated to `target_version`, the update folder has been moved to `system/updates/applied/` (including all migration commit reports and dry-run reports), and `system/updates/00-index.md` has a new entry.
- [REQUIRED] On failure: `.tropo/version.md` is unchanged, the update folder has been moved to `system/updates/failed/` with the partial-apply or verification-failure report AND any migration reports up to the point of failure preserved inside, and `system/updates/00-index.md` has a new entry marking the failure.
- [REQUIRED] The user has received a plain-language summary of what happened, framed by `update_type`, including (on success) the number of files changed via migrations and (on failure) the failure classification per v1.1 §14.3.2.

## Verification

### Method

Self-attestation by the concierge after apply, confirmed by the vault steward's independent TEST.playbook.md execution.

### Criteria

- `manifest.yaml` was read and all required fields validated before any operation ran, including v1.1 `migrations:` list validation and reserved slot acceptance
- `min_compatible` check was performed before any operation ran; on failure the message included plain-language version gap guidance (v1.1)
- Every operation and every migration in the manifest is logged in `channels/ops.md` with a result
- (v1.1) Every migration's commit-phase diff was compared against the concierge's dry-run report for divergence before commit
- (v1.1) Every migration produced a commit report in the update folder matching the §14.3.2 format
- (v1.1) Unexpected-structure cases inside migration scopes were logged as skips with reasons, not as failures
- The update folder is in either `applied/` or `failed/`, not both, not still in `pending/`
- If in `applied/`: `.tropo/version.md` shows `target_version`, and `system/updates/00-index.md` has a success entry
- If in `failed/`: `.tropo/version.md` is unchanged, and `system/updates/00-index.md` has a failure entry with a reason and (for migration failures) a classification
- The vault steward's verification report is present in the update folder
- No files outside the scope declared in this playbook's Rules were modified

### Escalation on Failure

- If manifest validation fails (including v1.1 migrations list): halt, leave the update in `pending/`, surface the validation error to the user. The update author needs to fix it.
- If `min_compatible` is not satisfied: halt, leave in `pending/`, tell the user with plain-language version gap guidance (name the specific intermediate versions needed) — v1.1 requirement.
- If a discrete operation fails mid-apply: halt, move to `failed/` with a partial-apply report describing the last successful operation and the failing one, notify the user. Do not roll back.
- (v1.1) If a migration's commit-phase diff diverges from the concierge's dry-run: halt the migration phase, leave prior discrete operations applied, move to `failed/` with a divergence report. The user approved the dry-run; a divergence means the vault state changed between review and commit, and the user did not approve whatever this playbook would now do.
- (v1.1) If a migration's commit-phase execution fails for a reason other than defensive skip (malformed playbook, filesystem error, permission denied, governance boundary violation): halt the migration phase, move to `failed/` with the migration report up to the failure point, classify the failure per §14.3.2 (`migration-logic`, `content-shape`, or `governance`).
- (v1.1) If a migration encounters unexpected structure inside its scope: this is NOT a failure. Log-and-skip, record reason in the migration report, continue the migration. A non-zero skip count with PASS result is valid.
- If verification fails: move to `failed/` with the verification report, leave `.tropo/version.md` unchanged, notify the user what failed.
- If the vault steward is unavailable or errors: halt, leave the update in `pending/` with a note in ops that verification was not attempted. User can re-trigger when steward is ready.
- If the user aborts mid-apply: halt, log the abort, leave the update in `pending/`.

# Execution Steps

## Step 0: Pre-flight

Before you touch anything, gather context:

1. Read `system/updates/pending/tropo-update-<id>/UPDATE.playbook.md` — understand the *why*.
2. Read `system/updates/pending/tropo-update-<id>/manifest.yaml` — understand the *what*.
3. Read `.tropo/version.md` — know the current vault version.
4. Read `system/updates/00-index.md` (or create it if missing) — know the update history.

## Step 1: Validate the manifest

### 1a: Required top-level fields (v1.0)

Check every required field is present:

- `version: 1`
- `update_id` (must match the folder name)
- `update_type` (one of: `patch`, `feature`, `release`, `security`)
- `target_version` (SemVer string)
- `min_compatible` (SemVer string)
- `summary` (non-empty string)
- `released` (YYYY-MM-DD)
- `author` (non-empty string)
- `operations` (list, may be empty only if `migrations:` is non-empty — an update must have SOMETHING to do)
- `verification` (object with `test_playbook`, `runner`, `on_failure`)

### 1b: Validate operations (v1.0)

For each operation, check:

- `type` is one of `add`, `replace`, `edit`, `move`, `delete`
- `path` is a string
- `reason` is a non-empty string
- For `type: add`: `source` points to a file inside `files/` in the update folder
- For `type: replace`: `source` points to a file inside `files/`
- For `type: edit`: `operation` is one of `append_section`, `insert_after_heading`, `replace_block`, `append_list_item`, with the required sub-fields for that mode
- For `type: move`: `from` and `to` paths are both strings
- For `type: delete`: `path` is a string

### 1c: Validate migrations list (v1.1)

If `manifest.migrations` is present and non-empty, for each migration entry:

- `path` is a string pointing to a file inside the update folder (typically `files/.tropo/playbooks/migrations/<migration-id>.playbook.md`). Verify the file exists at that path.
- `scope` is a string (glob pattern) or a list of strings (path list).
- `reason` is a non-empty string.
- The migration playbook file itself must conform to Playbook Spec v1.0 — it must have frontmatter with `uid:` and the six required sections (Intent, Suggestions, Rules, Resources, Outcomes, Verification).
- The migration playbook must declare a dry-run mode. (Check by reading the playbook's Intent or Rules for language acknowledging dry-run. This is a soft check — if you are unsure, log a warning but do not halt.)
- Verify the concierge has left a corresponding dry-run report at `system/updates/pending/<update_id>/dry-run-reports/<migration-id>.md`. If the report is missing, halt — the user cannot have approved a dry-run that wasn't performed.

### 1d: Accept reserved slots (v1.1)

If `manifest.conditionals`, `manifest.interactives`, or `manifest.hooks` are present:

- Parse them but do NOT execute any entries.
- For each non-empty reserved slot, log one line to `channels/ops.md`: `[update_id] v1.1 reserved slot <slot-name> declared with <N> entries — parsed and ignored (reserved for v1.2+)`.
- Continue with the rest of validation.

If any check in 1a, 1b, 1c, or 1d fails, halt. Log the validation error to `channels/ops.md` with update_id. Tell the user which field failed and where. Leave the update in `pending/`.

## Step 2: Check compatibility

Read `.tropo/version.md`. Compare the current `version:` field against the manifest's `min_compatible`. If current version < min_compatible (SemVer comparison), halt with **plain-language version gap guidance** (v1.1 requirement).

The v1.0 guidance was a bare "you need an intermediate update." That's a wall. v1.1 requires a path. Construct the message as follows:

1. State the current vault version explicitly.
2. State the update's `target_version` and `min_compatible`.
3. Enumerate the SemVer gap between current version and `min_compatible` as a list of versions the user needs to step through. You cannot know which updates actually exist — only the gap. But naming the gap as "v0.2.2, v0.2.3, v0.3.0,..., v[min_compatible]" is more useful than silence.
4. Tell the user where to find updates (typically the distribution source: Substack, GitHub, wherever Tropo framework updates ship from).
5. Explicitly say that once the user drops the missing updates into `system/updates/pending/`, you can walk them one at a time.

Example message:

> "This update (v0.4.7) requires your vault to be at v0.2.2 or later. Your vault is currently at v0.2.0. You'll need to install the versions between v0.2.0 and v0.2.2 first, in order — that's v0.2.1 and v0.2.2 — before I can apply v0.4.7. (There may also be updates between v0.2.2 and v0.4.7 that you want to install in sequence; I can walk them one at a time once you drop them into `system/updates/pending/`.) You can find Tropo framework updates at [distribution source]. Want me to check `pending/` for any of those intermediate updates right now?"

Leave the update in `pending/`. Log the compatibility failure with the specific gap named. Do not execute any operations.

## Step 3: Execute operations

For each operation in `manifest.operations`, in order:

### Operation type: `add`

1. Read the source file at `system/updates/pending/<update_id>/files/<source>`.
2. Check the target `path` does NOT already exist. If it does, halt with a failure — `add` requires the file not to exist.
3. Write the file at `path` with the source content.
4. If the file has YAML frontmatter and a `uid:` field, registration happens automatically — vault entries project into `vault/00-index.jsonl` on the next rebuild; runtime callables (sa.\*/skills/tools) project into `registry.jsonl` on the next rebuild; agent identity records require a manual `agent-registry.yaml` update (note in the manifest if the operation creates an agent).
5. Log: `[update_id] op <N>: add <path> — OK`

### Operation type: `replace`

1. Read the source file at `system/updates/pending/<update_id>/files/<source>`.
2. Check the target `path` DOES exist. If it does not, halt with a failure — `replace` requires the file to exist.
3. Read the current file. If its content already matches the source (no-op), log as a no-op and continue.
4. Otherwise, overwrite the file at `path` with the source content.
5. If the file has a `uid:` in frontmatter, the matched-primitive index will reflect any frontmatter changes on the next rebuild. No manual registry edit required.
6. Log: `[update_id] op <N>: replace <path> — OK` (or `NO-OP` if unchanged).

### Operation type: `edit`

1. Read the target file at `path`. If it does not exist, halt with a failure.
2. Apply the surgical edit based on `operation`:
 - **`append_section`**: append the content in `content:` to the end of the file, preceded by a blank line. The content may or may not include its own markdown heading — the content is appended verbatim.
 - **`insert_after_heading`**: find the heading named in `heading:` and insert the content in `content:` immediately after it (before the next heading at the same or higher level).
 - **`replace_block`**: find the block matching `match:` (literal string, not regex) and replace it with `replacement:`. Fail if `match:` is not found or is found more than once.
 - **`append_list_item`**: find the markdown list under the heading named in `list_heading:` and append `item:` as a new bullet at the end of that list.
3. Write the modified file back.
4. Log: `[update_id] op <N>: edit <mode> <path> — OK`

### Operation type: `move`

1. Check `from:` exists. If not, halt with a failure.
2. Check `to:` does NOT exist. If it does, halt with a failure.
3. Move the file from `from:` to `to:`.
4. The matched-primitive index reflects the new path on the next rebuild — `vault/00-index.jsonl` for vault entries, `registry.jsonl` for runtime callables. Agent moves require manual `agent-registry.yaml` update.
5. Log: `[update_id] op <N>: move <from> -> <to> — OK`

### Operation type: `delete`

1. Check `path:` exists. If not, log as a no-op and continue.
2. Record the file's UID and path to the ops log before deletion (audit trail).
3. Delete the file.
4. The matched-primitive index will drop the entry on the next rebuild. Agent deletes require manual `agent-registry.yaml` removal.
5. Log: `[update_id] op <N>: delete <path> — OK`

### Halting on failure

If any operation fails:

1. Log the failure to `channels/ops.md`: `[update_id] op <N>: <type> <path> — FAIL: <reason>`
2. Write a partial-apply report to `system/updates/pending/<update_id>/apply-report.md` describing what succeeded, what failed, and the vault's current partial state.
3. Move the entire update folder to `system/updates/failed/<update_id>/`.
4. Add a failure entry to `system/updates/00-index.md`.
5. Tell the user plainly what happened, using the `update_type` to frame the message.
6. **Do NOT update `.tropo/version.md`. Do NOT activate the steward. Do NOT run migrations.**
7. End execution.

## Step 3b: Execute migrations (v1.1 — new class)

*This step runs only if `manifest.migrations` is non-empty. Skip to Step 4 if there are no migrations.*

*This step runs ONLY after every discrete operation in Step 3 has succeeded. If any discrete operation failed, you already halted in Step 3 and never reached this step.*

### Pre-conditions

Before starting the migration phase:

1. Confirm the `dry-run-reports/` directory exists inside the update folder and contains one file per migration in `manifest.migrations`. The concierge wrote these during Review. If any are missing, halt with a `governance` classification — the user cannot have approved migrations whose dry-run was never performed.
2. Confirm you have write permission to each migration's declared scope. If a scope glob points to a path that does not exist in this vault, log a warning but continue — the migration's own dry-run will handle empty scopes.

### For each migration in `manifest.migrations`, in order:

#### Sub-step 3b.1: Re-run dry-run (divergence check)

1. Read the migration playbook at `files/<path>` in the update folder.
2. Invoke the migration playbook in `dry-run` mode against the current vault state (not the state at Review — the state now, after discrete operations have run).
3. The migration playbook produces a fresh dry-run report at `system/updates/pending/<update_id>/<migration_id>-commit-dryrun.md`.
4. Read the concierge's stored dry-run report at `system/updates/pending/<update_id>/dry-run-reports/<migration_id>.md`.
5. Compare the two reports. Specifically compare the `files_changed` and `files_would_change` sections. The set of files and the proposed transformations must match exactly.

#### Sub-step 3b.2: Handle divergence

If the fresh dry-run diverges from the concierge's stored dry-run:

- The vault state changed between Review and Commit. Possible causes: discrete operations in Step 3 added/modified files the migration's scope now includes; the user modified files in another session; a previous migration in this run changed the state.
- Halt the migration phase. Do NOT commit this migration.
- Classify: `content-shape` if the difference is additional files in scope; `migration-logic` if the transformation itself would differ; `governance` if files outside the declared scope are implicated.
- Write a divergence report to `system/updates/pending/<update_id>/<migration_id>-divergence-report.md` with both diffs side-by-side.
- Log to ops: `[update_id] migration <migration_id> — DIVERGENCE: <classification>, halting migration phase`
- Proceed to the halt protocol at the end of Step 3b.

#### Sub-step 3b.3: Commit the migration

If the fresh dry-run matches the stored dry-run:

1. Invoke the migration playbook in `commit` mode against the current vault state.
2. The migration playbook walks its declared scope, applies the transformation to each eligible file, logs skips with reasons for unexpected structure, and writes a commit report at `system/updates/pending/<update_id>/<migration_id>-commit-report.md` per the §14.3.2 format.
3. For each file the migration creates or substantially modifies, verify the file has a `uid:` in frontmatter if applicable. The matched-primitive indexes (`vault/00-index.jsonl`, `registry.jsonl`) update on the next rebuild via their respective builders. The migration playbook is responsible for any `agent-registry.yaml` updates if it creates agents — verify after migration completes.
4. Log to ops per file-changed-count: `[update_id] migration <migration_id> — COMMIT OK: <N> changed, <M> skipped, <K> errors`

#### Sub-step 3b.4: Classify the commit result

Read the migration's commit report. Check `result:` field:

- `PASS` with zero errors: migration succeeded cleanly. Continue to the next migration.
- `PASS` with non-zero skip count: migration succeeded with defensive skips. This is a valid success state — the skips are recorded in the report for the update author to review, but the pipeline continues.
- `PARTIAL`: migration partially completed before a halt-worthy error. Halt the migration phase. Log the partial state. Do NOT attempt to unwind prior changes.
- `FAIL`: migration failed outright. Halt the migration phase.

On `PARTIAL` or `FAIL`, proceed to the halt protocol below.

### Halt protocol for migration failures

If any migration fails (divergence, PARTIAL, or FAIL):

1. Log the failure to `channels/ops.md`: `[update_id] migration <migration_id> — FAIL: <classification>: <reason>`
2. Write a migration-phase partial-apply report to `system/updates/pending/<update_id>/migration-apply-report.md` summarizing which migrations completed, which one failed, the classification, and the cumulative state.
3. Move the entire update folder to `system/updates/failed/<update_id>/`. The failed migration's commit report, the divergence report (if applicable), and the concierge's original dry-run reports all travel with the folder.
4. Add a failure entry to `system/updates/00-index.md` with the migration failure classification.
5. Tell the user plainly:

 > "The [update_type] update installed its core changes successfully but ran into trouble during the migration phase — the part that walks your existing content. Migration [migration_id] failed with [classification]: [reason]. [Human-readable version of the classification — 'this looks like a bug in the migration itself, not your vault' for migration-logic; 'this looks like your content has a shape the author didn't anticipate' for content-shape; 'this is a boundary problem and we should investigate' for governance.] The core update files are still in place but I've moved the full update package to `system/updates/failed/<update_id>/` so nothing is ambiguous. The full reports are there. Your vault is in a partially-updated state — the discrete changes are applied but migration [migration_id] and any later migrations did not run. You can try again later after the author fixes it, or you can leave things as they are."

6. **Do NOT update `.tropo/version.md`. Do NOT activate the steward.**
7. End execution.

### On all migrations succeeding

If every migration in `manifest.migrations` returned PASS:

1. Log to ops: `[update_id] migration phase complete: <total> migrations, <changes> total file changes, <skips> total defensive skips`
2. Proceed to Step 4 (Activate the vault steward for verification).

## Step 3c: Propagate kernel seed vault entries (v1.2 — Board Reconciliation v0.3)

**Added 2026-04-20 per [Board Reconciliation v0.3 §6.1 + §9.6 (74fd9b61)](../files/74fd9b61.md).** Kernel-shipped ledger entries live at `.tropo/seed/vault/*.md`. On install or update, any seed file whose UID does not already exist in `vault/files/` is copied into `vault/files/`. Existing UIDs are NOT overwritten — a user may have locally customized them.

### 3c.1 — Enumerate seeds

List files in `.tropo/seed/vault/*.md`. For each: read the `uid:` frontmatter field.

### 3c.2 — Diff against active ledger

For each seed UID: check whether `vault/files/<uid>.md` already exists on disk.

- **If absent:** this is a new seed. Copy the seed file to `vault/files/<uid>.md`. Log: `[update_id] seed-propagate <uid> → vault/files/`.
- **If present:** existing file in the active ledger. Do NOT overwrite — the user may have locally customized. Compute a SHA256 of both; if they differ, surface a bulletin to `channels/ops.md`: "Seed `<name>` has drifted from the active vault entry. Review and reconcile if needed." Log: `[update_id] seed-drift-notice <uid>`. Do not block the update.

### 3c.3 — Record propagation

Write a list of newly-propagated UIDs to `system/updates/pending/<update_id>/seeds-propagated.jsonl`, one JSONL row per UID. The vault steward reads this during Step 4 verification to confirm seeds are reachable post-propagation (closes ADR-035 Surface 2 validation on the seeds).

### 3c.4 — Rationale

Kernel seeds are how Tropo-OS ships ledger-level defaults that every vault needs without requiring users to author them (example: the `project-board` default definition [c72f1a85](../seed/ledger/project-board.board-definition.md) every project inherits as its status board). Propagation on apply-update is the once-per-install / once-per-update mechanism. User customizations are protected by the never-overwrite rule; drift is surfaced, not silenced.

## Step 4: Activate the vault steward for verification

Once every operation in the manifest has executed successfully:

1. Activate the vault steward at `system/vault-steward/activate.md`.
2. Hand it the path to the update's TEST.playbook.md: `system/updates/pending/<update_id>/TEST.playbook.md`.
3. The steward reads the TEST playbook, runs its scenarios (static, dynamic, regression), writes its verification report to `system/updates/pending/<update_id>/verification-report.md`, and returns PASS or FAIL to you.
4. Log: `[update_id] verification — <RESULT>`

## Step 5A: Promote (success)

If the steward returns PASS:

1. Update `.tropo/version.md` to the manifest's `target_version`. Append a changelog entry for the update. Update `last_updated:` to today.
2. Move the update folder from `system/updates/pending/<update_id>/` to `system/updates/applied/<update_id>/`. The entire folder, including `verification-report.md`, travels together.
3. Add a success row to `system/updates/00-index.md`: update_id, type, target_version, date, result=PASS.
4. Log: `[update_id] applied — vault now at <target_version>`
5. Tell the user, framed by `update_type`:

 > "The [feature|patch|release|security] update applied successfully. Your vault is now at v<target_version>. [One sentence on what changed, drawn from the manifest summary.] I ran verification — all checks passed. The full report is at `system/updates/applied/<update_id>/verification-report.md` if you want to review it."

## Step 5B: Quarantine (verification failed)

If the steward returns FAIL:

1. **Do NOT update `.tropo/version.md`.**
2. Move the update folder from `system/updates/pending/<update_id>/` to `system/updates/failed/<update_id>/`. The verification report travels with it.
3. Add a failure row to `system/updates/00-index.md`: update_id, type, target_version, date, result=FAIL, reason.
4. Log: `[update_id] verification FAILED — quarantined to failed/, vault remains at <current_version>`
5. Tell the user plainly:

 > "The update did not verify cleanly. [One sentence on the type of failure from the verification report.] The update has been moved to `system/updates/failed/<update_id>/`. The full report is there. Your vault is still at v<current_version> — nothing is broken. You can try again later, share the failure report for debugging, or delete the failed update folder."

## Step 6: Close the session

1. Make sure `channels/ops.md` has all the operation log entries for this update.
2. Make sure `system/updates/00-index.md` has exactly one entry for this update (success or failure).
3. Make sure the update folder is in exactly one place — `applied/` or `failed/`, never both, never still in `pending/`.
4. Offer the user a next action: review another pending update (if any), run a vault health check, or continue their normal work.
