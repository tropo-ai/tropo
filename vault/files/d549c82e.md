## §1 — Why this cycle exists

Three things compose the case for v1.20.0.

**First — Lock A wants to be whole.** v1.17.0 declared the destination. v1.18.0 proved the pattern on kernel substrate. v1.19.0 advanced the convergence on safe substrate (KB articles + Studio-root playbooks + 10 capsules). What's left is the load-bearing substrate — the files that boot the agents themselves. Until v1.20.0 lands, "everything converges to `vault/files/<uid>.md`" has an asterisk on it: *"except agents/* and except AGENTS.md and except the boot chain at fixed paths."* That asterisk is the wedge that splits the structural invariant; closing it is the cycle's job.

**Second — agent portability needs the structural prerequisite.** Mike-A57 surfaced the question at walk: "Can we extract Agent X from a Studio?" Today, an agent is a hand-crafted bundle — charter at one path, soul at another, status card here, generation log there, briefing-package folder, reflections folder, transfers folder, memory in `.tropo-capsule/`, plus cross-references in channels, plus identity in a registry. Hand-extraction requires knowing the layout by heart. After v1.20.0, "export Agent X" is a vault query (`member_of:` includes the agent's UID) plus a 1-hop traversal. The same `extraction_scope:` machinery that ships the Tropo-OS zip applies directly. Without v1.20.0, when v1.22.0 (Vault Import + Export) comes to ship the federation surface, the engineering team would first have to do v1.20.0's structural work under pressure of shipping the feature instead of under careful walk discipline. v1.20.0 isn't a detour from agent portability — it's the structural prerequisite that makes it cheap.

**Third — v1.20.0 frees v1.21.0+ to focus.** v1.21.0 (sa.* + verification substrate hardening) wants to ship against a uniformly-typed substrate. v1.22.0 (Vault Import + Export) wants the same. v1.23.0 (Memory System) inherits cleaner. If we delay v1.20.0, every subsequent cycle pays the tax of "we need to teach this script / this validator / this migration tool about the special agent folder structure" — and the tax compounds. v1.20.0 is the cycle that makes every cycle after it composable against one substrate pattern.

The combined effect at ship: the convergence is structurally complete; agent portability is unlocked at the substrate level (Stream E proves it end-to-end); v1.21.0+ inherits a uniformly-typed surface; Studios that download the zip see one substrate pattern, not three different "where do files live" rules.

## §3 — The six substrate moves, concretely

### Stream A — Governance-contract typed primitive

**Source state.** Today, folder governance is per-folder `AGENTS.md` files. They use YAML frontmatter for structured metadata and a markdown body for human-readable rules. Schema is implicit — different folders' AGENTS.md files have drifted in shape. No validator enforces consistency. No state machine. The pattern works in practice but is structurally invisible to the typed-primitive substrate.

**The new primitive.** Authored at `.tropo/capsules/governance-contract.capsule.md` (kernel substrate, kernel exception class per Lock B). Schema includes:

- **Required frontmatter:** `governed_path:` (the folder this contract governs), `write_scope:` (who can write — agent UIDs or roles), `belongs:` (what artifact types/patterns belong in this folder), `protocols:` (naming + lifecycle + approval-flow rules), `member_of:` (subsystem hub UID)
- **Optional frontmatter:** `exception_class:` (declares folder as legacy-AGENTS.md exception if applicable), `validator_extension:` (link to validator check for this folder's invariants), `governed_by_release:` (which release locked this contract — tracks contract history)
- **State machine:** `draft → active → archived`. `active → archived` is supersession-only (one active contract per `governed_path:`).
- **Validation rules:** 8-12 rules covering shape, scope, invariants
- **Body shape:** 4 required sections — §1 Intent + §2 Who Writes + §3 What Belongs + §4 Protocols (per AGENTS.md convention but typed)
- **Composes-with:** AGENTS.md retirement path (Stream D); subsystem-hub.capsule (parent governance)

**Instance authoring:** Stream A authors the typed primitive only. Stream D authors the per-folder governance-contract instances at `vault/files/<uid>.md` for each folder being migrated from AGENTS.md.

**Validator extension at WARN:** `check_governance_contract_*` in `tropo-validate.py` — confirms every governance-contract instance resolves to a real folder + has non-empty required fields + matches schema. WARN-severity through v1.20.0+early-v1.20.x grace window per established grace-period pattern; ERROR at v1.21.0+.

### Stream E — Agent-extraction proof of concept (Mike-A57 authorized dogfood)

**Intent:** prove "export Agent X" is a vault query, not a hand-crafted bundle. Living validation that v1.20.0's structural pattern composes the way the theory says.

**Concrete mechanic:** author `.tropo-studio/scripts/v1.20.0-extract-agent-2026-05-XX.py` that, given an agent name (`argus`), produces:

1. Resolve `argus`'s UID set: identity files (charter / soul / status / gen-log / boot extension) at `vault/files/<uid>.md` per Stream B's migration.
2. 1-hop traversal: every vault entry whose `member_of:` includes any of Argus's UIDs (reflections / transfers / briefing-package entries IF they live in vault; cross-cutting referenced specs). Filter by `extraction_scope: ship` to match the existing extraction machinery.
3. 2-hop traversal (optional): every vault entry that *references* Argus by UID (channels with Argus's UID in active threads; ops bulletins; release entries crediting Argus).
4. Package the result into an extraction folder (`export/argus/<timestamp>/`) matching the structure `extraction_scope: ship` produces for the OS-zip.

**Dogfood pass:** run the extraction script against Argus once Stream B completes. Verify the resulting export folder contains a complete, self-describing agent bundle (identity + canonical substrate + cross-references resolved).
