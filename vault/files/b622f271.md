# How Vault Entry Works — Platform Hooks and the Governance Chain

**Relations**

| Relation | Target |
|---|---|
| Governed by | [kb-article (4cb20382)](4cb20382.md) |

*For agents: how you got here, why the rules apply, and why there is no path into this vault that bypasses governance.*

---

## The Question This Answers

When an agent opens a Tropo vault for the first time — or enters any governed folder — how does it know to read the governance files? Nothing in the model's training says "read CAPSULE.md before writing." The answer is the entry point mechanism: a set of platform-native files that pull the agent into the governance chain before it does anything else.

---

## The Entry Point Files

Different platforms have different conventions for "what an agent reads first." Tropo supports all of them by maintaining one entry point file per platform convention at the vault root.

| File | Platform | Mechanism |
|------|----------|-----------|
| `AGENTS.md` | All LLMs (OpenAI/Linux Foundation standard) | LLMs trained after mid-2025 recognize `AGENTS.md` as a governance file and read it automatically |
| `CLAUDE.md` | Claude Code (Anthropic) | Claude Code reads `CLAUDE.md` before any tool use in the session |
| `.cursorrules` / `.cursor/` | Cursor | Cursor IDE injects this into the model context at session start |
| `SYSTEM_PROMPT.md` | Other platforms | Convention for platforms that accept a custom system prompt file |

**In this vault:** `AGENTS.md` and `CLAUDE.md` both exist at the root. A Claude Code agent reads `CLAUDE.md` first. Any other LLM reads `AGENTS.md`. Both files say the same thing in their respective formats: *you are in a Tropo vault, here is the governance chain, follow it.*

---

## What Each Entry Point Says

**`AGENTS.md` (vault root):**
```
This folder is part of a Tropo-OS vault. Before operating, read these files in order:
1..tropo/TROPO-CONTROL.md — OS rules, identity checkpoint, invariants
2. STUDIO.md (vault root) — organization defaults and constraints
3. CAPSULE.md (this folder) — folder purpose and operating logic
```

**`CLAUDE.md` (vault root, Claude-specific):**
```
This is a Tropo-OS vault. You are the vault concierge.
Boot Instructions:
1. Read.tropo/concierge/activate.md — your full activation file.
2. Read.tropo/TROPO-CONTROL.md — OS rules, identity checkpoint, invariants.
3. Read STUDIO.md — organization defaults and constraints.
4. Before writing to any folder, read the CAPSULE.md in that folder.
```

Both routes terminate at the same governance chain. The platform hook is different; the destination is the same.

---

## The Four-File Governance Chain

Once pulled in by the entry point, every agent follows the same read sequence:

```
Entry point (AGENTS.md or CLAUDE.md)
 ↓
.tropo/TROPO-CONTROL.md ← OS invariants. Read ONCE per session.
 ↓
STUDIO.md ← Vault defaults + constraints. Read ONCE per session.
 ↓
CAPSULE.md (per folder) ← Folder rules. Read when entering each new folder.
```

**TROPO-CONTROL.md** is the OS singleton. It defines the identity checkpoint, the override hierarchy, and the invariants that no folder rule can break. It lives in `.tropo/` — the kernel — and is maintained by Tropo through the update pipeline.

**STUDIO.md** is the vault owner's layer. It defines naming conventions, default permissions, and hard constraints. The vault owner writes and maintains it.

**CAPSULE.md** is the folder layer. It defines who can write, what belongs, and any local overrides to STUDIO.md defaults. Every governed folder has one.

**The key property:** TROPO-CONTROL.md and STUDIO.md are read once at session start and cached. CAPSULE.md is re-read each time the agent enters a new folder. This keeps the overhead low while ensuring folder governance is always current.

---

## AGENTS.md Is in Every Folder — Not Just the Root

This is the insight that makes the governance ambient rather than dependent on where the agent enters.

Every governed folder in a Tropo vault has its own `AGENTS.md`. The content is identical — it's a thin pointer, about 15 lines. But its presence means:

- An agent that drops directly into `vault/` without reading the root hits `vault/AGENTS.md` first
- An agent that enters `channels/` hits `channels/AGENTS.md` first
- An agent that enters `.tropo/capsules/` hits `.tropo/capsules/AGENTS.md` first

**There is no path into a governed folder that bypasses the chain.** As long as the agent is behaving — reading AGENTS.md when it enters a folder, which all compliant LLMs do — governance applies everywhere.

---

## Why This Architecture

**Language-native portability.** No server, no permissions API, no admin panel. The governance is in files the agent reads. Any LLM on any platform that can read a text file and follow instructions gets the same governance. Tropo works on Claude, Gemini, GPT, local models — because the entry point is just a markdown file.

**Platform-specific without platform lock-in.** `CLAUDE.md` is optimized for Claude Code's session model. `AGENTS.md` is the cross-platform standard. They coexist. Adding support for a new platform is adding one entry point file — nothing else changes.

**Governance is maintained, not trained.** The rules aren't in the model's weights. They're in the files. When Tropo ships a new release and updates `TROPO-CONTROL.md`, every vault that applies the update gets the new rules — without retraining anything.

---

## For Future Argus Generations

When reviewing vault governance architecture, the entry point mechanism is the load-bearing first step. If AGENTS.md is missing from a folder, that folder's governance is only enforced for agents that enter via the root. If CLAUDE.md is stale or missing, Claude Code agents may not follow the Tropo boot protocol. The `vault-integrity-auditor` checks for AGENTS.md coverage — if it flags missing files, that's not cosmetic.

The `.tropo/00-index.md` is the system-level navigation hub — when you have a question about how any part of the kernel works, start there.
