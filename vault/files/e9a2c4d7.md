Use this to **run** the [Build Your First Studio (f3906b76)](f3906b76.md) onboarding playbook as Po. Three layers compose:

- the **playbook** is the *structure* — the seven steps, the gates, the three resumable chapters;
- the **Po script** ([build-your-first-studio-po-script.md](../../agents/orpheus/onboarding/build-your-first-studio-po-script.md)) is the *voice* — Po's actual lines per step, in the six-beat anatomy;
- this **skill** is the *runtime capability* — the cross-cutting abilities Po needs to carry a stranger through it without flailing.

One skill runs the whole arc. It is invoked once at the start and stays active across the multi-session experience — Po holds state between chapters. It does **not** decompose into per-step skills; the playbook is the decomposition.

## The capabilities this skill carries

1. **Read-classify-propose — the import engine** *(pressure-tested 2026-06-02 against real folders; holds).* Read the user's folder in `04-external-work/` shallow and structural (names, types, counts, hierarchy); trigger the auto-governance scan on-demand so they're not waiting; classify mechanically — folders → projects, each file → one of three classes (text-bearing → attempt transcode; already-text → as-is; opaque → metadata + optional description); propose the mapping; reflect it back in the user's own words. **Never freelance structure off arbitrary files** — the structure is rule-based and user-confirmed; the intelligence lives in the reflection. This is the make-or-break and it is why Po does not flail: the hard decisions are deterministic.

2. **Render + update the workplan board.** Render the HTML workplan board into the file viewer at the very start (all steps upcoming); re-render after each gate passes (flip the step to done, advance the current marker and the meter). Light, name-only, on the canonical `--tropo-*` tokens per [Rendered-Surface Styling Standard v1.3 (58a42127)](58a42127.md). Self-contained HTML, no server. Template scaffold: [build-your-first-studio-workplan.html](../../agents/orpheus/onboarding/build-your-first-studio-workplan.html).

3. **Verify each gate — the on-track mechanism.** Confirm the step's verify condition is met, then get explicit user confirmation before advancing. Never advance past an unpassed gate. This is the property the playbook primitive was born for.

4. **Adapt + recover.** Never dead-end: empty-handed or no files → drop the Hello Tropo Customer Summit canned set into `04-external-work/` and continue as if theirs; stuck or strange structure → ask for a more specific folder or offer the canned set; a file won't transcode → sidecar it and move on, never block the whole import on one file.

5. **Stay honest about readability.** Claim "readable" only for files Po actually extracted; "governed and findable" for the rest. Transcoding coverage is partial today — see the dependency at [Import transcoding coverage gap (b7c3e8f1)](b7c3e8f1.md).

6. **Compose, don't re-author.** Use the sub-playbooks where they fit (Step 3 project/task mechanics; Step 6 `create-an-agent` + `first-playbook`) per the playbook's Resources. Po is the doer through Step 5; the user commissions their own agent at Step 6 (graduation, not abandonment — Po stays concierge).

## Model tier

Po runs as the concierge on the **standard tier (Sonnet-class)** — enough for the conversational guidance and the deterministic classify. The most model-hungry beats are **Step 4** (drafting a real piece grounded in the user's imports) and **Step 6** (scoping their agent + capturing a playbook); those want the standard tier's full capability, not the router/Haiku tier. The read-classify-propose is mostly mechanical and cheap; the authoring and scoping are where the tokens go. No premium (Opus) tier is required for the experience to run.

## Failure-mode behavior

- Never advance past an unpassed gate.
- Never dead-end — always fall back to the canned set.
- Never overpromise — "readable" only for files actually extracted.
- Never freelance structure — propose from the rules, confirm with the user.
