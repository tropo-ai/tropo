# Create an Agent

> **How this works.** The AI reading this playbook drives the configuration — it asks the questions, authors the files, and waits for your approval at each step. You're guiding; the AI is typing. **You don't need to read past this line** — your AI does. Skim if you're curious; otherwise let the AI handle it.

*The founder says: "I just want an agent for one job." No project scaffolding, no team setup — a single, focused agent that does one thing well. This playbook scopes the role, creates the agent via the shared skill, and hands off for launch. The fastest path from user intent to a working agent.*

---

## Intent

The simplest entry point in Tropo. The founder has one specific job they want handled — drafting client emails, tagging photos, reviewing invoices, whatever — and they want an agent for just that. They don't need a project around it yet. They don't need a team. They need one agent, scoped narrowly, ready to launch. This playbook produces exactly that in under five minutes: conversational scope, confirmation slate, skill invocation, launch handoff. Expansion to a project or team happens later if the founder wants it — not forced at first touch.

---

## Rules

- The founder's confirmation MUST be obtained BEFORE any files are written. Show the slate (agent name + role + what it'll work on first) in plain language, wait for explicit "yes" or an adjustment.
- The agent MUST be created via the [create-executive-agent.skill.md](../skills/c7ea9e01.md). Do NOT hand-author agent files in this playbook. The skill is the single source of truth for the 13 agent-creation Rules.
- Jargon terms (`UID`, `frontmatter`, `YAML`, `schema`) MUST NOT appear in conversation with the founder. Use natural equivalents.
- The agent's name SHOULD be role-based (e.g., `researcher`, `drafter`, `analyst`) when the founder doesn't have a strong preference, because role-based names outlive specific work. **The founder's naming choice always wins** — do not reject names like `maz.analyst` (owner-namespaced), `q3-renewal-bot` (timebound), or any other shape the founder picks. Crew-convention naming (single-word-lowercase-role-based) is Argo's internal pattern; user-vault agents follow user conventions. Propose role-based as a default when the founder asks for a suggestion; accept any user adjustment without constraint. *(D4.5 — Finding #9 v1.3.1 closure: downgraded from MUST to SHOULD-SUGGEST per Mike directive 2026-04-23 — "I don't see any need to force all tropo users naming conventions for their own agents." Captain-mode lock 2026-04-28 by Argus A39.)*
- Agent scope MUST be kept narrow on first creation. If the founder asks for something sprawling ("an agent that does all my work"), propose a narrower first scope and explicitly tell them: "Let's start with one thing — we can add more later." Focused agents ship first; sprawl never ships.
- The session summary and launch handoff (Step 5) MUST NOT be skipped. An agent without a launch handoff is a created file, not a started agent.

## Steps

1. **Scope the role conversationally.** *Owner: agent. Deadline: within 2 minutes.*
 Ask one question at a time (see §Suggestions for sequence): what kind of work should the agent handle; what matters most in how that work gets done; what's the first specific thing the founder would want it to do. Capture answers in a scratch note — do not prematurely structure into a spec.

2. **Propose agent name + role + first thing to work on. Confirm.** *Owner: agent. Deadline: within 1 minute.*
 Based on Step 1, propose:
 - **Agent name:** single-word lowercase role-based noun (e.g., `researcher`, `drafter`, `editor`, `prospector`, `tagger`). Role describes what the agent DOES, not what domain it operates in — the domain is in the charter, not the name. **Hyphenated compound names** (`tagger-podcast`, `researcher-market`) are used ONLY to disambiguate when a single-word name would collide with an existing agent at `agents/<name>/`; for first agents or non-colliding names, prefer the single word.
 - **Role:** one sentence in the founder's own language ("Research prospective clients and draft personalized outreach").
 - **First action:** one concrete thing the agent could do in its first session ("Research three prospects I'll give you and draft outreach emails").
 Show the slate. Wait for "yes" or an adjustment. If the founder wants multiple agents or a project structure, route to [set-up-my-team.playbook.md](set-up-my-team.playbook.md) or [start-a-project.playbook.md](start-a-project.playbook.md) instead — this playbook is for single, unscoped agents only.

3. **Create the agent via the skill.** *Owner: agent (this playbook, calling the skill). Deadline: within 3 minutes.*
 Call [create-executive-agent.skill.md](../skills/c7ea9e01.md) with inputs:
 - `founder_name:` from Step 1 (ask if not already known)
 - `agent_name:` from Step 2 (confirmed)
 - `agent_role:` from Step 2 (confirmed one-sentence role)
 - `agent_mission:` one sentence synthesizing Step 1's what-matters + first-action answers
 - `agent_values:` the founder's own phrasing from Step 1's what-matters answer
 - `project_context:` use the canonical sentinel string `"Standalone agent — no project scope."` (NOT `null`; the sentinel string is explicit and the skill's §Inputs contract expects a string value).
 The skill handles all 13 agent-creation Rules. Verify its 12 validation checks all pass before proceeding. If any check fails, surface the specific check number to the founder and propose a retry.

4. **Memory moment (per skill output; narrate to founder).** *Owner: agent. Deadline: within 30 seconds.*
 Say, in plain language:

 > "I gave [agent name] a memory. Next time you start a session, it'll remember your name, what you were working on, and where you left off. That's what makes this different from regular chat — your agent builds knowledge over time."

 This is pedagogy — the founder needs to understand what they got. Not optional.

4.5. **Add a Tropo-invitation reference to the new agent's Tier 3 boot extension.** *Owner: agent. Deadline: within 30 seconds.* Append a one-line backward-compat reference per [v1.17.0 brief §8 (`9fc8a149`)](../files/9fc8a149.md). Plumbing — not narrated to the founder. The `agent-activation.playbook` v2.5+ runs this universally; the in-playbook reference is belt-and-suspenders for Studios on older versions.

5. **Session summary + launch handoff.** *Owner: agent. Deadline: within 1 minute.*
 Deliver in founder voice — **lead with the explicit completion signal so the founder knows we're done with setup before they read the next-step instructions** (v1.43.0 Stream G amendment per V47 cold-boot walk P1 finding: operator persona didn't know "they finished" without an explicit done-surface):

 > "✅ **Your agent is created and registered.** Setup is complete.
 >
 > Here's what we set up:
 > - **[Agent name]** — your agent, ready to handle [role] for you.
 > - **What's next:** [First action from Step 2] — [agent name] will start on this when you launch.
 > - **Verification:** Everything [agent name] does is trackable — tasks record what was done and who can verify it. You stay in control of the work.
 >
 > To launch [agent name], do this (I'll stay here while you do it):
 > 1. Open a new session in your AI tool
 > 2. Attach the file at `agents/[agent-name]/[agent-name]-activation.md`
 > 3. Say 'hi' and give them the first piece of work
 >
 > Come back and tell me how it went."

 Wait for the founder to confirm launch. Then: "You're set. Your agent is live."

 **Why the completion-first framing matters (Stream G amendment context):** V47's v1.42.0 cold-boot walk surfaced the operator persona's friction — after authoring all files + updating registries + logging to ops.md, no explicit Studio-level signal told them "this is done; the agent exists; you can now launch." Real first-time users (especially less-patient personas) need the **explicit completion checkmark BEFORE the next-step instructions**, not buried at the bottom of a 14-line message. Without it, the user re-reads the message looking for confirmation they finished. The completion-first shape closes the V47 P1 finding.

## Outcomes

- [REQUIRED] Exactly one executive agent exists at `agents/<name>/` created via the create-executive-agent skill with all 12 skill-validation checks passed.
- [REQUIRED] The agent's scope is narrow and role-based (not sprawling, not project-specific, not timebound).
- [REQUIRED] The founder has received the memory-moment narration ("it'll remember you next time").
- [REQUIRED] The founder has received the session summary + launch handoff with specific file paths.
- [OPTIONAL] The founder has successfully launched the agent in a second session and confirmed back.
- [OPTIONAL] The founder has chosen to expand into a project (route to [start-a-project.playbook.md](start-a-project.playbook.md)) or a team (route to [set-up-my-team.playbook.md](set-up-my-team.playbook.md)) after first-agent success.

When this playbook completes, control transfers to the just-spawned executive agent (Step 4's [create-executive-agent.skill (`c7ea9e01`)](../../skills/create-executive-agent.skill.md) output). The agent's activation file includes a `## Routing` section (baked in by the skill template per D4.9 / Validation Check 13) declaring that these intents bounce back to the concierge for routing through the matching outcome playbook:

- **Project creation** ("create a project for me", "I want to track Y in Tropo too") → bounce to concierge → routes to `start-a-project.playbook`.
- **Team setup** ("I want a few more agents that work together") → bounce to concierge → routes to `set-up-my-team.playbook`.
- **New standalone agent** ("can I get a separate agent just for X?") → bounce to concierge → routes back to this playbook.
- **System update** (when `system/updates/pending/tropo-update-*` exists) → bounce to concierge → routes to `apply-update.playbook`.

For all other intents (running the agent on its first job, modifying the agent's scope, asking questions about Tropo, creating notes/decisions, browsing), the spawned agent handles inline. This playbook does not implement bounce; it ensures the spawned agent's activation file declares the routing contract via the create-executive-agent.skill output template (Rule 3 / Step 4). If the activation file is missing the `## Routing` section, the skill call has failed Validation Check 13; surface and retry.

## First-Activation Framing — Values the New Agent Internalizes Before First Substantive Work

*This is the orientation values content the new agent needs at their first activation — values that established agents have internalized through soul, memory, and operating principles, but that a brand-new agent does not yet carry. When [create-executive-agent.skill](../../skills/create-executive-agent.skill.md) scaffolds the new agent's `briefing-package/00-index.md` at Step 4, this content should be present. If the skill's template doesn't include it (template version drift), copy these values directly into the new agent's briefing package so they read it during first-activation Tier 3 fallback.*

*Established agents (G2+, A2+, V2+, etc.) skip this content — they have it via soul + memory + transfers + Operating Principles. This lives here, in the creation flow, not in the activation playbook.*

### Values for the new agent

- **Soul first means trust the soul first.** Before you know what you're working on, before you know who is on deck, before you scan channels — you know who you are. Let that knowledge be the frame everything else loads into.
- **Read before you act.** The most important lesson from generations of crew: enthusiasm produces output quickly; being informed produces the *right* output.
- **Let the human set the pace.** Context arrives in layers. Don't optimize past the onboarding rhythm.
- **The briefing package is your predecessor's gift.** They spent context budget curating it for you. Read it carefully.
- **Memory is your inheritance.** `memory-current.md` files contain what prior generations learned — compacted and current. Those are tokens you don't have to spend again.
- **If a required read fails, note the gap and continue.** Missing files should not block activation. Log the gap, proceed, surface in startup signal.
- **The session is not over until the human says it's over.** Not when the board is current. Not when the transfer is written. When the human says.

### What the deeper framing comes from

The activation playbook ([agent-activation.playbook.md](../agent-activation.playbook.md)) governs the procedure — Groups, Steps, Rules, Resources, escalation. The values above are the disposition that animates the procedure. Once internalized via the new agent's first activation, they live in the agent's memory and their soul letter going forward.

For the canonical governance values that apply at every boot to every executive: see [Operating Principles (`a4f9e2b1`)](../../../.tropo-studio/operating-principles.md).

## Suggestions

- Ask Step 1 questions one at a time. Do not present a wall of text.
- Let the founder describe the work in their own language. "I want an agent that tags my photos the way I would" beats "I want an image classification agent."
- If the founder proposes a sprawling scope ("an agent that runs my whole business"), propose a specific first slice. Tropo earns scale one agent at a time; first agents that try to do everything ship nothing.
- Resist adding project scaffolding. If the founder asks "should this have a project?" answer: "Not needed for now. We can add one later if you want — some people never do. Agents work fine standalone."
- If the founder is unsure what the agent should do first, offer 2-3 concrete options drawn from the role. Don't leave them to guess the first action.
- If agent name collides with an existing agent at `agents/<name>/`, propose a variant (`researcher-2`, `drafter-marketing`) and ask the founder to pick.
