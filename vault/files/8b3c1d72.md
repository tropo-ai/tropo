## Intent

Tropo-OS is the open-source artifact of the Tropo platform — a portable, markdown-first operating system that users can download, point an LLM at, and get governed AI agents. Today the OS gets built locally by [`build-release.py`](../../.tropo/scripts/build-release.py) into versioned folders at `/Users/mike/dev/tropo-ai/releases/v1.X.X/`. Those folders never reach the public web.

This brief specifies how `tropo-ai.com` exposes those releases for public download — the **download surface** of the v1 marketing site. The implementation deploys after this brief locks; the brief itself is the contract for that work.

**Out of scope:**
- Implementation work (separate deploy after brief lock).
- The build process itself (`build-release.py` is unchanged; this brief consumes its output).
- Authentication or paid-tier gating on downloads (Tropo-OS is open-source; downloads are public).
- Hosting the OS artifact anywhere other than GitHub Releases (object storage, S3, etc. — defensible alternatives, but GitHub Releases is the conventional pattern for open-source distribution and is the recommended approach).

## Decisions made

The following are committed by this brief (not open for litigation at implementation time):

1. **Public GitHub repo is the distribution surface.** Specifically: `github.com/tropo-ai/tropo-os` (a new repo, see §Dependencies). GitHub Releases on that repo is the canonical home for Tropo-OS versioned builds.

2. **Releases page on tropo-ai.com reads from GitHub's public API.** No DB cache for v1; we hit `api.github.com/repos/tropo-ai/tropo-os/releases` directly with ISR caching (Next.js incremental static regeneration). Public API rate limit is 60 req/hour unauthenticated, 5K req/hour authenticated — ISR caching at 1-hour TTL keeps us well under unauthenticated bounds for v1 traffic.

3. **Latest release is the primary CTA.** The download page features the most recent stable release prominently. Previous versions accessible via a "Past releases" expandable section or a `/releases/v1.X.X` route per version.

4. **Release artifacts are zip files attached to each Release.** When Argus runs `build-release.py` and publishes to `tropo-ai/tropo-os`, he attaches the built folder as a `.zip` asset to the GitHub Release. The website's download button links to that asset URL.

5. **Prereleases are hidden by default.** GitHub Releases supports a "Prerelease" flag. The website filters those out on the main download page; a `?prerelease=true` query param or a hidden tab can reveal them for internal testing.

## Architecture

### Data flow

```
build-release.py (local) → releases/v1.X.X/ (local folder)
                              ↓ (Argus zips + uploads)
                         tropo-ai/tropo-os Releases tab (GitHub)
                              ↓ (Next.js fetch via GitHub API)
                         tropo-ai.com/download (rendered page)
                              ↓ (visitor clicks Download)
                         GitHub-hosted zip (direct download)
```

### Routes on tropo-ai.com

- **`/download`** — main download page. Renders: latest release name + version + release notes excerpt + a prominent "Download Tropo-OS v1.X.X" button. Below: "Past releases" expandable section listing prior versions.
- **`/releases`** — alias of `/download` (or a redirect). Common convention.
- **`/releases/[version]`** — per-release page (e.g., `/releases/v1.30.0`). Renders full release notes, download link, prior-version pointers. Useful for direct linking from social, email, blog.

### Implementation pattern (Next.js)

- **Fetch function:** server-side `lib/tropo-os-releases.ts` exports a typed wrapper over GitHub's `/repos/tropo-ai/tropo-os/releases` endpoint. Returns `Release[]` with normalized fields (`version`, `name`, `published_at`, `body` markdown, `zip_url`, `prerelease`).
- **Caching:** `revalidate: 3600` on the route (Next.js ISR). Hourly refresh. Acceptable lag for a marketing surface — released-yesterday is fine to show released-today after an hour.
- **Rendering:** Tropo-branded; bone background, ember accents matching `app/page.tsx`. Release notes are markdown — render via `react-markdown` with prose styles.
- **Fallback:** if GitHub API is down or rate-limited, the route returns its last-cached value. Stale-while-revalidate is built into Next.js ISR.
- **No auth required at v1.** Public reads against GitHub's API are sufficient. If we ever hit rate-limit issues, add a `GITHUB_TOKEN` env var (read-only personal access token) to bump to 5K req/hour — still no user-facing auth needed.

### Visual shape (UI)

The `/download` page should feel like a product launch page, not a developer docs page. Aim:

```
[Tropo wordmark — ember]

Download Tropo-OS

The latest release of the operating system for human-AI teams.

[ Tropo-OS v1.X.X — Stable ]
[ Download (XX.X MB)        ]   <-- ember button, prominent
[ Release notes ↓           ]

(brief release-notes excerpt)

────────────────────────────

Past releases
  - v1.X.0 (Month Day, Year) — Download
  - v1.X.0 (Month Day, Year) — Download
  - ...
```

Bone background, ember CTAs, ink-on-light text. Type-heavy, image-light. The kind of page that respects the visitor's time.

## Dependencies

**Substrate prerequisites (must exist before implementation can ship):**

1. **`tropo-ai/tropo-os` public GitHub repository** must be created. Currently does not exist (only `tropo-ai/tropo-ai` and `tropo-ai/website` exist in the org as of 2026-05-15). Mike + Argus call: when to create, what to put in the repo's root (README, LICENSE — recommend MIT or Apache 2.0 for an open-source OS distribution), and whether the starter source files live in this repo (mirror of `argo-os/starter/` content) or only the built releases live there. Recommended: source mirror + releases both, so the repo is browsable + downloadable.

2. **First Tropo-OS release published to `tropo-ai/tropo-os` Releases tab.** Argus's existing `build-release.py` flow gets a publish step appended — after build, push to GitHub and attach the zipped build folder as a release asset. Could be a separate `publish-release.py` script or a flag on `build-release.py`.

3. **Versioning convention.** GitHub Releases use git tags. Tropo-OS versions are already semver (v1.X.X). Convention: each release tag matches `v<major>.<minor>.<patch>` exactly.

**Substrate dependencies (must be in place at implementation):**

- Vercel team for tropo-ai.com website (✅ done, this project)
- Next.js 16 framework on `tropo-ai/website` (✅ done, this project)
- Brand tokens (✅ done, this project)

**No new platform engineering needed.** This is straightforward Next.js + GitHub API integration; no Supabase, no Resend, no streaming AI.

## Open Decisions

1. **`tropo-ai/tropo-os` repo creation — when?** Owner: Mike + Argus. Blocks: implementation deploy of this brief.
   *Recommendation:* create now, even before this brief is implemented. Future-proofs the OS's open-source presence. README at minimum + a LICENSE file. Releases can start landing when Argus does his next release cycle.

2. **What lives at the root of `tropo-ai/tropo-os` repo?** Options: (a) just README + LICENSE (releases-only repo), (b) full source mirror of `argo-os/starter/` (browsable open-source code) + LICENSE + releases. Owner: Mike + Argus.
   *Recommendation:* (b) — full source mirror. Open-source means open source. Anyone landing on `github.com/tropo-ai/tropo-os` should see the code, browse it, and fork it. Releases are a separate distribution channel for "pre-built zip downloads." Both surfaces compound the project's open-source story.

3. **License choice.** ✅ **CLOSED 2026-05-16. Apache 2.0.** Mike confirmed after reading the [OSS License Research note (b7e9c41d)](b7e9c41d.md). Reverses the previously-locked AGPL-3 ship-artifact ([4a7e1c9b](4a7e1c9b.md)). Substrate change handoff to Argus filed at [c5d8e421](c5d8e421.md) in dev-pipeline 01-inbox. Reasoning: Apache 2.0's patent grant + enterprise-adoption signal outweighs AGPL's defensive-against-cloud-vendors posture for a markdown-substrate framework whose moat lives in the substrate design, not the license.

4. **Release notes source.** GitHub Releases supports markdown release notes on each Release. Where does that text come from? Argus's existing release-plan.capsule and release.capsule produce structured release artifacts in the vault. We could (a) hand-author release notes per release, (b) auto-generate from the vault release entries via a script. Owner: Argus (architectural).
   *Recommendation:* (b) — auto-generation matches Tropo's substrate-driven discipline. A script reads the release entry from `vault/files/<release-uid>.md` and produces markdown for the GitHub Release body. Closes the gap "where do release notes live" cleanly.

5. **Email capture on download?** Optional gating: capture email before showing download button. Owner: Mike.
   *Recommendation:* **no email capture at v1.** Friction-low download is the conversion-maximizing v1 choice. The agent-builder surface (separate brief, #8) is the right place to introduce email capture — that's where someone is producing real personalized output for which we have a reason to keep in touch. The OS download is generic; gating it with email is friction without justification.

- `/releases/[version]` per-release page: ~2 hours engineering.

- Auto-generation script for release notes (per Open Decision #4 recommendation): ~2-3 hours, falls to Argus's substrate-tooling lane.

Rough total: **~10 hours of implementation work** once the brief locks and `tropo-ai/tropo-os` repo + first release exist.
