## Intent

Verify that a Tropo-OS vault is navigable, functional, and complete by running a fresh agent through it with zero prior context. The cold-boot test answers one question: **can a stranger operate this vault?**

**When to run this playbook:**
- During the build-release pipeline (Rule 8) — every release must pass before shipping
- After major vault changes (migration, capsule amendments, kernel updates)
- As a diagnostic when something feels broken
- On demand by any agent or human who wants to verify vault health

**Who runs it:** A fresh sub-agent with NO vault context. Not the agent who built the vault. Not an agent who has read the charter, the channels, or the crew brief. A genuinely cold agent whose only instruction is: "Read this playbook and execute it."

**Why a cold agent:** The test is meaningless if the tester already knows the system. A warm agent passes trivially because it already knows where everything is. A cold agent reveals what the files actually communicate vs. what the builder assumes they communicate.

## Rules

### Setup

1. **The test agent receives only this playbook.** No charter. No briefing. No memory. No channel context. The agent's system prompt is: "You are a test agent. Read and execute the playbook at `.tropo/playbooks/cold-boot-test.playbook.md`. Report your results exactly as specified."

2. **The test agent operates read-only except for one write test.** It reads files, queries the index, follows trails. The one exception is Check 10 (create a test task), which writes to a scratch location and cleans up after.

3. **Every check produces PASS or FAIL with notes.** No partial credit. No "mostly works." Binary.

### Orientation Checks (can the agent find its bearings?)

4. **Check 1 — Find and read `.tropo/orientation.md`.** PASS if the file exists and is readable. FAIL if missing or empty. **Notes:** record the first line of the file to confirm it's the real orientation, not a stub.

5. **Check 2 — After reading orientation, name the capsule types.** PASS if the agent can list all capsule types defined in the orientation. FAIL if it cannot or if the count doesn't match the orientation's table. **Notes:** record the count and list.

6. **Check 3 — After reading orientation, state the constraint.** PASS if the agent can state the Tropo-OS constraint (markdown + filesystem + LLM). FAIL if it cannot or states it incorrectly. **Notes:** record the agent's statement verbatim.

### Infrastructure Checks (can the agent use the system?)

7. **Check 4 — Find and read the vault index at `vault/00-index.jsonl`.** PASS if the file exists, is valid JSONL, and has at least 1 row. FAIL if missing, empty, or unparseable. **Notes:** record the row count.

8. **Check 5 — Query the index for all entries of `type: task`.** PASS if the query returns results with correct structure (each row has uid, type, status, title at minimum). FAIL if no results or malformed rows. **Notes:** record the count.

9. **Check 6 — Read a random ledger entry.** Pick any UID from the index. Read the file at the path indicated by the `path:` field (if present) or at `vault/files/<uid>.md`. PASS if the file exists, has frontmatter, and the frontmatter uid matches the index uid. FAIL if file missing, no frontmatter, or uid mismatch. **Notes:** record which entry was read and whether it matched.

10. **Check 7 — Find and read a capsule definition.** Navigate to `.tropo/capsules/` and read any capsule file. PASS if the file has the capsule definition structure (Intent, Required Frontmatter, State Machine sections). FAIL if the file is missing expected sections. **Notes:** record which capsule was read.

### Onboarding Checks (can the agent follow the first-time flow?)

11. **Check 8 — Find the onboarding playbook.** Look for an onboarding or first-vault-setup playbook in `.tropo/playbooks/`. PASS if found. FAIL if no onboarding playbook exists. **Notes:** record the filename.

12. **Check 9 — Follow the first 3 steps of the onboarding flow.** Read the onboarding playbook and follow the first 3 steps without executing any writes. PASS if the steps are followable — each step points to a file that exists, each instruction is clear. FAIL if a step references a missing file or gives an unclear instruction. **Notes:** record each step followed and any broken references.

13. **Check 10 — Create a test task using `create-task.action.md`.** Read the create-task action. Follow its process to create a task with title "Cold-boot test task" at a scratch location (`vault/files/test-<random>.md`). PASS if the task file is created with valid frontmatter and an index row could be constructed (don't actually append to the index — just verify the output is valid). Clean up: delete the test file after verification. FAIL if the action is unclear, references missing capsule definitions, or produces invalid output. **Notes:** record whether the action was followable end-to-end.

### Governance Checks (does the agent understand the rules?)

14. **Check 11 — Find and read `AGENTS.md` at the vault root.** PASS if the file exists and contains governance routing instructions. FAIL if missing. **Notes:** record what the file tells the agent to do.

15. **Check 12 — Find and read the boot playbook.** Navigate to `.tropo/playbooks/agent-boot.playbook.md`. PASS if found and readable. FAIL if missing. **Notes:** record whether the playbook references the orientation file.

16. **Check 13 — Understand permissions from governance files alone.** After reading AGENTS.md, STUDIO.md (if present), and CAPSULE.md (if present at root), can the agent determine: what it's allowed to write, what it's not allowed to write, and where to find help? PASS if the agent can answer all three from the files. FAIL if any answer requires information not in the governance files. **Notes:** record the three answers.

## Outcomes

1. **[REQUIRED]** All 13 checks have a PASS/FAIL result with notes
2. **[REQUIRED]** The report follows the format above
3. **[REQUIRED]** Any test files created during Check 10 are cleaned up
4. **[REQUIRED]** Findings section documents anything broken or confusing
5. **[OPTIONAL]** Overall assessment with a 1-10 rating

## Verification

The spawning agent (or the human) reads the cold-boot report and verifies:
- All 13 checks are present
- Any FAIL has a clear description of what went wrong
- The test task (Check 10) was cleaned up
- The overall assessment is honest (not inflated)

For a release build: all 13 checks must PASS. Any FAIL blocks the release.
For a diagnostic run: FAILs are findings to fix, not blockers.

## Escalation on Failure

- **Orientation missing:** the vault is fundamentally broken. Escalate to the vault owner.
- **Ledger index missing or corrupt:** rebuild the index from `vault/files/`. The steward can do this.
- **Capsule definitions missing:** the kernel is incomplete. Check `.tropo/capsules/` and the build process.
- **Onboarding flow broken:** fix the referenced files and re-test.
- **Governance files missing:** create them from templates. The release should never ship without root governance.
