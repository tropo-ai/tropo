## Purpose

The implementation step of build stage. Talos (or whichever engineering agent is running the pipeline) writes the code for the files or features in the confirmed scope. Edits land at the `tropo-ai/tropo-ai` repo root in `app/`, `lib/`, `components/`, `tests/`, `public/`, `scripts/`, and project-config files as needed. The step closes when the changes compile in dev mode and render correctly at localhost:3000 for an eyeball check.

This is the only step in build that produces source code changes. verify-build (step-2) verifies what this step produced; it doesn't author new code.

## Structure

Leaf step. No children.

## Behavior

No confirmation gate. Executor enters the step; agent does the work; step closes when local dev-mode rendering is correct.

**Step work:**

1. Read the activation-root project's `changes_in_scope` to confirm what's being built. Read `expected_visual_diff:` — drives the verification approach (zero-diff baseline check for refactor-class, content-check for feature-class).
2. Open or create the files needed. Stay within `changes_in_scope`. Touching files outside scope is grounds for a pause-and-confirm with Mike (see Scope-expansion discipline below).
3. Run `npm run dev` (if not already running). Open localhost:3000 at the affected routes. For zero-diff refactor changes: compare against the pre-change baseline (eyeball or screenshot diff). For feature changes: confirm the new behavior renders correctly.
4. Author any per-deploy decisions, alias mappings, or dependency notes as documents under the activation-root project (e.g., "Aliased --accent → --tropo-accent — exact mapping at <link>").
5. Write `step_completed` event to `run.jsonl`. Stage advances to verify-build.

**On in-flight scope expansion:** if the implementation surfaces a need for scope beyond what was confirmed at step-0, pause and ask Mike before expanding. Do not silently grow the scope.

## Inputs

- Activation-root project (from step-0): `changes_in_scope`, `theme`, `expected_visual_diff`, deploy number.
- Current state of the repo at the deploying branch (typically `main`).

## Outputs

- Source code changes within `changes_in_scope`.
- Optional per-deploy documents under the activation-root project (decisions, alias mappings, dependency notes).
- `run.jsonl` events: `step_started`, `step_completed`.
