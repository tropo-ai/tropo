# How Pipelines Work

**Relations**

| Relation | Target |
|---|---|
| Governed by | [kb-article (4cb20382)](4cb20382.md) |

*A pipeline declares the shape of repeating work. A pipeline-run executes one instance of it. The pattern survives across sessions; the run records what happened.*

---

## The one-line answer

A pipeline is a Tropo primitive — a declarative workflow template. You author it once; you run it many times. Each run is its own audit trail.

If you've worked with Airflow (DAG / DAG-Run), or BPMN (Process / Instance), or GitHub Actions (workflow YAML / workflow run), you already know the shape. Pipelines in Tropo are the same idea, written in markdown, governed by the Vault, and executable by any AI agent that can read files.

---

## The mental model — template and instance

Two concepts. Keep them apart.

| Concept | Capsule | What it is |
|---|---|---|
| **Pipeline** | [`pipeline.capsule`](../capsules/pipeline.capsule.md) | The template. A declaration of how the work flows: stages, steps, and the transitions between them. Versioned. Re-usable. |
| **Pipeline-run** | [`pipeline-run.capsule`](../capsules/pipeline-run.capsule.md) | A single execution. Pins the pipeline version at start. Has its own folder, event log, and resumption anchor. Ephemeral. |

The split matters. Edit the pipeline → new runs use the new version, in-flight runs continue on the version they pinned. Delete a run → the pipeline itself is unchanged. Audit a past release → read its run folder; the events tell the truth.

**Analogy:** the pipeline is the recipe; the pipeline-run is the meal you cooked on Tuesday. The recipe sits in the cookbook. The meal happened once, and you remember exactly what you did because you took notes.

---

## Why pipelines (and why not just playbooks?)

Tropo already has playbooks — governed step-by-step procedures any agent can execute. Why a separate pipeline primitive?

Three differences:

1. **Pipelines persist state across sessions.** A playbook is a procedure you run linearly. A pipeline-run carries its own folder with `definition.md` + `context.md` + `thread.md` + `run.jsonl` + `run.state.json`. Pause it, walk away, come back next week — the next agent reads the run folder and continues exactly where the prior agent stopped.
2. **Pipelines bundle members.** A pipeline-run carries `members:` — the work-items or projects this execution is operating on. The run is a coordination container; processor-agents filter the members they own and execute their slice. A playbook doesn't have this concept.
3. **Pipelines have audit-grade event logs.** Every transition (`step_started`, `stage_entered`, `pause_started`, `step_completed`) writes a structured event to `run.jsonl`. The log is append-only. Months later, you can replay exactly what happened — when, by whom, with what.

If your work is **a procedure run once** → use a playbook. If your work is **a workflow with stateful execution that may pause for humans, span multiple sessions, and produce an audit trail** → use a pipeline.

## The shape — pipelines compose recursively

A pipeline is a tree of WorkflowNodes. Same capsule, fractal composition.

```
Pipeline (root WorkflowNode)
├── Stage (composite WorkflowNode with children)
│   ├── Step (leaf WorkflowNode, no children)
│   └── Step
└── Stage
    ├── Step
    └── Step
```

- **Pipeline** — the root. Has `domain:` (e.g. `tropo-work`, `release`, `marketing-campaign`).
- **Stage** — a phase of the work. Has children.
- **Step** — atomic unit. No children. This is where work actually happens.

Each box is its own vault entry (`type: pipeline, subtype: workflow-node`) referenced by UID. Composition is via UID-references, never inline objects. Soft-cap on nesting depth: 3 levels (BPMN's complexity-at-scale lesson).

Flow is expressed via `next_steps:` — successors in execution order. Linear by default (length-1 array). Branching is schema-supported, behavior shipping incrementally per [pipeline.capsule §Known Enforcement Gaps](../capsules/pipeline.capsule.md).

## How execution actually happens

A pipeline-run is a coordination + audit container. It does NOT execute by itself. Execution happens via **processor-agents** operating within the run.

The flow at runtime:

1. **Invoker authors a pipeline-run.** An entity (human or AI) creates the run folder, sets `pipeline:` to the template's UID, pins `pipeline_version:`, declares `members:` (the work-items or projects in scope), and seeds `run.jsonl` with a `run_created` event.
2. **Processor-agent picks up the current step.** Any agent that reads the run's `definition.md` sees the current stage + step. The agent filters `members:` by `requested_of: <own entity-uid>` — the slice it owns.
3. **Agent executes the step on its slice.** The work itself happens via the member's own request-lifecycle (e.g. for tasks, the state machine in [task.capsule](../capsules/task.capsule.md)). The run observes; the agent mutates.
4. **Agent appends a `step_completed` event.** The run's `current_step:` advances per the pipeline's `next_steps:` schema.
5. **Repeat or complete.** The next step's processor (same agent or different) picks up the new `current_step:`. Continues until the terminal step → `status: complete`.

**Confirmation gates.** A leaf step may declare `requires_confirmation: true`. When the executor enters that step, the run pauses (`status: active → paused`), logs a `pause_started` event with `data.reason: "human-confirmation-required"`, and waits. An authorized confirmer (owner / principal / authorized entity) appends a `pause_resumed` event with `data.confirmation_granted_by:`, the run resumes, and the step's work begins. This is how pipelines hold for human approval mid-workflow.

## The first pipeline in Tropo — `dev-pipeline`

Tropo's own development workflow is itself a pipeline. The scaffold ships at [`agents/dev-pipeline/`](../../agents/dev-pipeline/) with three stages and seven steps:

| Stage | Steps |
|---|---|
| **Specify** | 0 accept-work-item → 1 generate-release-plan → 2 spawn-arch-specs |
| **Build** | 3 author-arch-spec-artifacts |
| **Deploy** | 4 generate-release-notes → 5 produce-release-folder → 6 external-test |

Step 1 (`generate-release-plan`) carries `requires_confirmation: true` — the pipeline-run pauses on entry until the principal confirms the proposed release number. That's the v2.1 confirmation-gate pattern in production.

The first dev-pipeline activation ([`7c0402b8`](../../agents/dev-pipeline/activations/7c0402b8/run.jsonl)) ran the v1.5 cycle of Tropo-OS itself end-to-end on 2026-05-04. The run.jsonl from that activation is a worked example — read it to see how a real pipeline-run unfolds across stages.

## When to author your own pipeline

Author a pipeline when:

- **The same workflow happens repeatedly** — releases, marketing campaigns, audit cycles, onboarding sequences. Each instance gets its own run; the template is shared.
- **The workflow needs to pause for humans** — confirmation gates, review checkpoints, sign-offs. `requires_confirmation: true` is the primitive for this.
- **You want an audit trail** — every step, every transition, every actor recorded permanently. The `run.jsonl` is your evidence.
- **Multiple agents coordinate over time** — one agent handles Stage 1, a different agent handles Stage 2, a human signs at the gate. The run folder holds the shared state.

Do NOT author a pipeline for:

- **A one-shot procedure.** Use a [playbook](how-playbooks-work.md) — simpler, no run-folder overhead.
- **A static checklist.** Use a [collection](how-the-tropo-vault-works.md) of tasks.
- **A document workflow with two stages.** Probably overkill — start with a project + tasks, graduate to a pipeline if the pattern repeats.

The rule of thumb: if you wouldn't author a recipe for it (because you'll only cook it once), don't author a pipeline.
