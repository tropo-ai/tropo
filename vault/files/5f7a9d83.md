*Every governed file in Tropo is a typed file. The type is defined by a *capsule*. The capsule says what fields are required, how status changes over time, what validation runs, and what governance applies. This article explains the primitive — what it is, why it exists, how to read one, how to author one.*

---

## The two file shapes: capsule definition + capsule instance

Every type in Tropo has two file shapes:

- **Capsule definition** — at `.tropo/capsules/<name>.capsule.md`. Declares the contract for a type. Schema, state machine, validation rules, governance.
- **Capsule instance** — at `vault/files/<uid>.md`. A specific file of that type. Obeys the definition's contract.

Concrete example. The `task` type has:

- A definition file at `.tropo/capsules/task.capsule.md`
- Instances at `vault/files/a4f8c291.md`, `vault/files/b7e3f582.md`, etc. — each one a specific task

Read the capsule definition once to understand the type. Read individual instances to read the actual work.

---

## What a capsule definition declares

Every capsule definition follows a **five-section pedagogy** (the pattern that capsules like `task.capsule.md` and `playbook.capsule.md` model):

1. **Intent** — what the type is, what it's for, what failure mode it prevents.
2. **Schema** — what frontmatter fields are required, what types they hold (string, list, UID reference, enum, etc.).
3. **State Machine** — the lifecycle for instances of this type. Each capsule defines its own state names + transitions; there's no universal lifecycle.
4. **Validation Rules** — what checks run against instances. Numbered Rule format ("Every instance MUST have a `title:` field").
5. **Composes-With** — how this type composes with other types in the graph (which fields reference which other types; which capsules this one extends or is extended by).

The definition is the *contract*. Every instance of the type obeys it.

The root capsule is `core.capsule.md` — every other capsule extends from it. The shape is `core` at the floor + specialized capsules (`task`, `project`, `decision`, `pipeline`, etc.) that extend it with type-specific fields and rules.

---

## How an instance obeys the contract

An instance is just a markdown file with YAML frontmatter following the definition's schema. Skeleton for a task instance:

```yaml
---
uid: a4f8c291
type: task
title: "Author the engineer FAQ"
status: closed
resolution: done
state: archived
owner: argus
created: 2026-05-18
created_by: argus-a73
governed_by: <task-capsule-uid>
member_of:
  - <parent-project-uid>
---

# Author the engineer FAQ

[body content describing the task]
```

The frontmatter fields are what the agent reads + writes programmatically. The body is plain prose — for humans, and for agents to reference as they work.

Note that `status` and `state` are different fields with different jobs: `status` carries the task's lifecycle position (task uses `new → accepted → active → closed`), while `state` is an orthogonal lifecycle-visibility flag (`active` / `archived`) shared across all capsules.

## Why typed files (instead of free-form prose)

Three concrete benefits:

**Validation by construction.** Because every governed file has a declared type, a script can walk every file and check the schema. Run `python3 .tropo/scripts/tropo-validate.py` to see this in action — it surfaces broken instances by rule number.

**Composition by reference.** Typed files reference each other by UID via fields like `member_of:`, `aligned_with:`, `governed_by:`. The result is a graph that any tool can walk. The Tropo Platform walks it. Validators walk it. Agents walk it. Everyone gets the same answer from the same graph.

**State machines that compose.** Each capsule has a lifecycle. A project starts `state:active`; its tasks start `state:active`; when all tasks reach `done`, the project can move to `done`. The state machines compose so high-level state reflects the underlying work.

Without types: every file is freeform prose. Every agent has to read the body of every file to figure out what it is. The graph collapses; the validation surface disappears; the audit trail loses signal.

## How to read a capsule definition cold

Open `.tropo/capsules/<name>.capsule.md` in your AI tool. Skim in this order:

1. **§Identity** — what does this type DO?
2. **§Schema** — what fields are required?
3. **§State machine** — how does this type's lifecycle work?
4. **§Validation rules** — what rules apply?
5. **§Governance** — who can author this; what requires approval?

You don't need to read every line; the section structure means you can scan for what matters.

---

## How to find capsule definitions

- **`ls .tropo/capsules/`** — the directory listing IS the catalog. Roughly 60 capsules ship with the OS, covering core primitives (project, task, decision, agent, capsule) and extension types (release, design-brief, pipeline, kb-article, etc.).
- **`grep "^type: capsule-definition" vault/00-index.jsonl`** — vault-resident definitions.

For one-shot lookup: ask your AI *"what capsule governs `<file>`?"* — the AI reads the file's `governed_by:` field and resolves the linked definition.

---

## How to author a new capsule

If your team needs a new type that doesn't exist:

1. Read `core.capsule.md` — the root type every capsule extends. Then read a similar existing capsule (e.g., `task.capsule.md` for work-item-shaped types) as a pattern exemplar.
2. Author your definition at `.tropo/capsules/<name>.capsule.md` following the five-section pedagogy (Intent / Schema / State Machine / Validation Rules / Composes-With).
3. Add `extends: core` in your definition's frontmatter to declare inheritance.
4. Run `rebuild-vault.py` to register the new type.
5. Validate: `tropo-validate.py` will check instances of your new type once they exist.

Most teams don't need new capsules — the shipping set covers common primitives. New definitions are reserved for genuinely novel types in your team's workflow.

---

## How capsules relate to other Tropo primitives

- **Projects** are instances of type `project`.
- **Tasks** are instances of type `task`. Tasks live `member_of:` a project.
- **Decisions** are instances of type `decision`.
- **Agent charters** are instances of type `charter`.
- **Releases** are instances of type `release`.
- **Subsystem hubs** are instances of type `subsystem-hub`.
- **The capsule library itself** is governed by `core.capsule.md` (the root capsule from which all type definitions inherit).

Everything governed is typed. The graph composes through typed references. The validator enforces by walking.

---

## When this article doesn't answer your question

The source-of-truth is the capsule definition itself. Open `.tropo/capsules/<name>.capsule.md` for any type you care about — the definition IS the contract.

For the root type every capsule extends: `.tropo/capsules/core.capsule.md`.

For browsing the type catalog: `ls .tropo/capsules/` or filter `vault/00-index.jsonl` by `type:capsule-definition`.
