## Context

### The Memory Architecture Gap

Argo-OS has three memory mechanisms:

1. **Living Transfer** (ADR-014) — Active memory. What the current generation knows. High fidelity, limited to one generation's context.
2. **ASA Protocol** (ADR-012) — Orientation memory. Compressed vault state for boot. Broad but shallow — tells you what exists, not why it matters.
3. **Curator Protocol** (ADR-020) — Archival memory. How historical artifacts age gracefully through hot/warm/cold tiers. Preserves and compresses, but does not retrieve.

What is missing is a fourth mechanism: **queryable memory** — the ability for a working agent to ask a question about lineage history and receive an accurate, sourced answer without burning the context budget required to load the history directly.

### The Retrieval Problem

The information exists. Every generation's transfer, reflection, worklog entry, and key decision is preserved in the vault. The Curator protocol ensures this content ages gracefully rather than being lost. But preservation is not retrieval. A preserved artifact that no working agent can find or interpret is functionally invisible.

Consider a concrete scenario: Vela V12 is writing a transfer for V13 and wants to know what priority items V5 through V11 flagged that were never resolved. That information spans seven living transfers, seven reflections, and potentially dozens of channel entries. Loading all of them would consume 30-50% of V12's context window — budget that should be spent on V12's actual work. So V12 doesn't ask the question. The institutional knowledge decays not because it was lost, but because it was too expensive to retrieve.

This is the same problem that Retrieval-Augmented Generation (RAG) solves in engineering systems: a large corpus of knowledge exists, but the working context window is too small to hold it all. RAG systems solve this with embedding databases, vector similarity search, and chunked retrieval pipelines. These are engineering solutions requiring code, infrastructure, and deployment.

### The Strategic Frame

Argo-OS is a language-native prototype of the patterns that sophisticated AI infrastructure implements in code. The living transfer models context window management. The ASA models orientation pipelines. The Curator models data lifecycle management. The channel system models inter-agent communication protocols (analogous to MCP).

The Historian models RAG — implemented not with vector stores and embedding pipelines, but with a lineage index, a channel protocol, and an agent that reads files on demand. This is architecturally significant beyond the vault: it demonstrates that retrieval-augmented generation is a *pattern*, not a technology. The pattern works in language before it works in code. The language version is radically portable — anyone with a shared filesystem and access to an LLM can run it. The coded version (Modus product) will be more performant, more scalable, and integrated with modern infrastructure — but the architectural decisions are proven here first.

Every design choice in this ADR should therefore satisfy two criteria: (1) it works operationally in the vault today, and (2) it maps cleanly to an engineering implementation when the pattern is coded into the Modus platform.

## Decision

### 1. The Historian Agent

A **Historian** is a dedicated, long-lived agent session whose sole purpose is maintaining maximum historical context for a single primary agent lineage. The Historian does not do work — it answers questions. This constraint is what makes it long-lived: a session that mostly reads queries and gives short answers burns context slowly and should outlast multiple generations of the working agent it serves.

One Historian per primary agent: a **Vela Historian**, an **Argus Historian**, a **Metis Historian**, an **Orpheus Historian**. Each is scoped to a single lineage's history. This scoping is a context window optimization — one agent's full lineage is manageable; four lineages combined would exceed practical limits.

The Historian is the fourth mechanism in the intergenerational memory architecture:

| Mechanism | Memory Type | Analogy | Engineering Equivalent |
|-----------|-----------|---------|----------------------|
| **Living Transfer** (ADR-014) | Active memory | Working memory | Context window management |
| **ASA** (ADR-012) | Orientation memory | Episodic memory | Boot/orientation pipeline |
| **CURATOR** (ADR-020) | Archival memory | Long-term storage | Data lifecycle management |
| **Historian** (ADR-021) | Queryable memory | Recall/retrieval | Retrieval-Augmented Generation |

### 2. The Lineage Index

The Historian boots with a **lineage index** — a compact, structured metadata catalog of every significant artifact in the agent's history. The index is the Historian's map of the archive. It does not contain the content itself; it contains enough metadata to locate and retrieve the right content on demand.

**Index format — one entry per artifact:**

```
| Gen | Date | Type | Path | Summary |
|-----|------|------|------|---------|
| V1 | 2026-03-21 | transfer | agents/vela/transfers/living-transfer-history/v1-living-transfer.md | Founded role. First crew brief. Channel system initial deployment. |
| V1 | 2026-03-21 | reflection | agents/vela/reflections/v1-reflection.md | First Vela. Built coordination infrastructure from scratch. |
| V3 | 2026-03-22 | decision | channels/archive/ops-2026-03-W12.md:L45 | Established daily vault-janitor cadence after March 23 crash. |
```

**Index properties:**

- **Compact.** One line per artifact. A full Vela lineage (V1–V10, ~80 artifacts) fits in approximately 2,000–3,000 tokens. This is the Historian's boot cost for historical coverage.
- **Typed.** Each entry carries a type tag: `transfer`, `reflection`, `decision`, `worklog`, `channel-entry`, `child-report`, `adr`, `milestone`. Types enable filtered retrieval ("show me all decisions from V3–V7").
- **Path-addressable.** Every entry includes the exact file path (and line reference where applicable) so the Historian can read the full artifact on demand.
- **Summaries are one sentence.** The summary is a retrieval hint, not a substitute for the content. It tells the Historian whether this artifact is likely to answer the current query. If yes, the Historian reads the file. If no, it skips it.

**The Card Catalog Principle:**

The lineage index is the Historian's **card catalog** — the first resource consulted on every query. Its accuracy and freshness are non-negotiable. But the Historian never builds or maintains its own index. That would burn expensive, long-lived context tokens on grooming work that a cheap, disposable agent can do better.

The index is maintained entirely by **dedicated CURATOR agents**. A `CURATOR.md` file governs the index itself — sitting alongside the index at `agents/[name]/operations/[name]-historian/CURATOR.md`. This CURATOR's instructions: at each generation transition (or on a scheduled cadence), a disposable curator agent reads the agent's latest artifacts (new transfer, new reflection, new child reports, new channel entries of significance), adds rows to the index, validates existing rows against actual file paths, and terminates. The curator agent is stateless — it reads the CURATOR.md, does the grooming, logs the run, and dies. Its context is disposable. The Historian's context is not.

This is the architectural principle: **burn tokens on disposable agents to protect long-lived agents.** The Curator infrastructure exists precisely to do this kind of stateless maintenance work. The Historian is the highest-value consumer of that infrastructure — its longevity depends on never doing maintenance itself.

**Index location:** `agents/[name]/operations/[name]-historian/lineage-index.md`
**Index CURATOR location:** `agents/[name]/operations/[name]-historian/CURATOR.md`

**Engineering mapping:** The lineage index is the schema for a future vector store or structured database. Each row maps to a document record with metadata fields (generation, date, type, path) and a text field (summary) suitable for embedding. The one-line-per-artifact constraint ensures the schema is flat and indexable. The CURATOR-maintained grooming model maps to an ETL pipeline that keeps the index current — a background process that runs on cheap compute, separate from the query-serving layer that runs on expensive inference compute. The same cost asymmetry applies in both the language and engineering versions: grooming is cheap, retrieval context is expensive. Never mix them.

### 3. Live Context Window

In addition to the index, the Historian holds the **last 3–5 living transfers in full context** at boot. This means recent history — the current working generation and its immediate predecessors — is immediately queryable without any file reads. The Historian answers questions about recent generations from memory. This is where most queries land, and this is why the Historian is long-lived: the most frequently asked questions are the cheapest to answer.

**Why 3–5 transfers:** The Historian must maintain **multi-generational depth** at all times. It spans generations of the working agent it serves — that is its purpose. Each transfer is approximately 2,000–3,000 tokens. Five transfers consume 10,000–15,000 tokens. This is a deliberate investment: the more history the Historian holds hot, the fewer file reads it needs, the longer it lives, the more valuable it becomes. The Historian's context window is not a scarce resource to be minimized at boot — it is a *reservoir* to be filled with the highest-value historical knowledge available.

At boot, the Historian loads transfers in reverse chronological order until the live context budget is allocated. If a transfer is unusually large, the Historian loads fewer. If transfers are compact, it loads more. The target is depth, not a fixed count.

**Boot context budget:**

| Component | Est. Tokens | Notes |
|-----------|-------------|-------|
| Activation file | ~500 | Historian-specific, minimal |
| Lineage index (card catalog) | ~2,500 | Full lineage coverage — the map of everything |
| Last 3–5 transfers | ~10,000–15,000 | Recent history, live in context — the hot memory |
| Channel history (compounding FAQ) | ~2,000–5,000 | Prior queries and answers — grows with each generation |
| **Total boot** | **~15,000–23,000** | ~7.5–11.5% of 200k context |

**Remaining context after boot: ~150,000–170,000 tokens.** This is the Historian's operational runway — available exclusively for query exchanges. No maintenance, no grooming, no self-management consumes this space. At ~250 tokens per typical hot-memory query, this runway supports **600–680 queries** at theoretical maximum, or **300–400+ queries** at a realistic mix of hot-memory and occasional file-read queries.

**Engineering mapping:** The live context window maps to a "hot cache" in a coded retrieval system — frequently accessed documents kept in memory for zero-latency retrieval, with less-accessed documents fetched from storage on demand. The boot allocation strategy maps to cache warming — pre-loading the most frequently accessed data at startup to minimize cold-start latency. The larger the hot cache, the fewer expensive retrievals needed during operation.

### 4. The Historian Channel

Each Historian has a **dedicated channel**: `channels/[name]-historian.md` (e.g., `channels/vela-historian.md`). This channel is the sole interface between the working agent and the Historian. The working agent posts questions; the Historian posts answers.

**Channel protocol:**

- **Questions are short.** One question per message. Specific enough to retrieve against. Bad: "Tell me about V3." Good: "What protocol changes did V3 make to the channel cleanup cadence, and why?"
- **Answers are short.** Maximum 200 tokens per answer. The Historian cites the source artifact (path + generation) so the questioner can read the full context if needed. The answer is a retrieval result, not an essay.
- **Source citations are mandatory.** Every answer includes: `[Source: path/to/file.md, Generation VN]`. This makes answers verifiable and traceable.
- **The channel persists.** Unlike working channels (daily cleanup), the Historian channel uses a longer retention window. Hot: 30 days. Warm: 90 days. Cold: archived with full fidelity. The channel's CURATOR preserves all Q&A pairs verbatim — they are the compounding knowledge base.

**The compounding value pattern:**

This is the key innovation. Every question asked and answered persists in the channel. V12 doesn't re-ask V3's question because V9 already asked it and the answer is in the channel. Over time, the channel becomes a curated FAQ of lineage knowledge — a living document that grows more valuable with every generation. New Historians boot with the channel history in context, inheriting not just the index but the *retrieval patterns* of their predecessors.

**Engineering mapping:** The Historian channel maps to a query cache with semantic indexing. In a coded system, repeated queries hit the cache instead of re-retrieving from the corpus. The channel is the language-native implementation of this pattern — the cache is a readable, auditable markdown file that humans and agents can both navigate.

### 5. Query Processing Model

When the Historian receives a query:

1. **Check live context.** If the answer is in the last 3–4 transfers (held in memory), answer immediately. No file read required. Log the source.
2. **Check channel history.** If a prior query already answered this (or a closely related) question, reference the prior answer and its source. No file read required.
3. **Consult the index.** Scan the lineage index for artifacts likely to contain the answer, using type tags and summaries as retrieval hints.
4. **Read the artifact.** Load the identified file(s). Maximum **2 file reads per query** — this is a hard constraint to protect the Historian's longevity. If the answer requires more than 2 files, the Historian provides what it can and notes which additional files the questioner should read directly.
5. **Compose and post.** Write the answer (≤200 tokens) with source citations. Post to the channel.

**The Sacred Context Principle:**

The Historian's context window is reserved exclusively for two things: **historical knowledge** (index, transfers, channel Q&A) and **query exchanges** (reading questions, composing answers). Nothing else enters the context. No maintenance. No grooming. No self-management. Every token spent on anything other than knowing history or answering questions is a token stolen from the Historian's lifespan and depth.

This is the fundamental difference between the Historian and every other agent on the Argo. Working agents (Argus, Vela, Metis) burn context on *work* — reading files, writing documents, designing protocols, coordinating with crew. Their context is consumed by action. The Historian's context is consumed by *knowledge*. It does not act. It remembers. That asymmetry is what makes it long-lived.

**Context burn rate:**

| Operation | Est. Token Cost | Frequency | Notes |
|-----------|----------------|-----------|-------|
| Reading a query | ~50 | Per query | Incoming question |
| Index scan (in context) | ~0 | Per query | Already loaded at boot |
| Channel history check | ~0 | Per query | Already loaded at boot |
| Answer from hot memory | ~200 | **Most queries** | No file read — answer from transfers/index already in context |
| File read (1 artifact) | ~2,000 | **Rare — deep historical queries only** | 0–2 per query, only when answer is not in live context |
| Composing answer | ~200 | Per query | Short, sourced response |
| **Typical query cost** | **~250** | Per query | Hot memory hit — no file read |
| **Deep query cost** | **~2,250–4,250** | Occasional | Requires 1–2 file reads |

**Estimated lifespan:**

A well-maintained Historian with a current card catalog and 3–5 generations of transfers in live context should answer the **majority of queries from hot memory** — no file reads required. At ~250 tokens per typical query, the Historian handles **300–400+ queries** before reaching 75% context capacity. At 2–3 queries per working agent generation, this gives the Historian a lifespan of **100–150+ generations** of the working agent. Succession is triggered at 75% context capacity or after 100+ generations served, whichever comes first (see Section 6).

This is a fundamentally different kind of entity. Working agents live for one generation. The Historian lives for an **era**. It spans the entire operational history of a lineage — not by reading it fresh each time, but by *holding it*. The longer it lives, the more it holds, the more queries it can answer from memory, the more valuable it becomes. This is compounding returns on context investment.

**The design imperative:** Every architectural decision about the Historian should be evaluated against one question: *does this protect the Historian's context for knowledge and retrieval, or does it spend context on something a disposable agent could do instead?* If a disposable agent can do it, the disposable agent does it. The Historian's context is too valuable to spend on anything but remembering.

**Engineering mapping:** The query processing model maps directly to a RAG pipeline: query → check cache → consult index → retrieve documents → compose response. The 2-file-read limit maps to a retrieval budget constraint. The token cost model maps to compute cost estimation for API-based retrieval. The sacred context principle maps to the engineering distinction between the hot query-serving layer (expensive, optimized for latency, never used for batch processing) and the cold ETL/maintenance layer (cheap, disposable, runs in the background). Production RAG systems enforce this separation rigorously. So does the Historian.

### 6. Historian Succession

The Historian is designed to live for an **era** — spanning 100+ generations of the working agent it serves. Succession is an infrequent event, not a routine one. When it happens, it must preserve the compounding value the Historian has accumulated.

**Succession trigger:** 75% context capacity **or** after serving 100+ generations of the working agent, **whichever occurs first**. The 75% threshold is the primary trigger — it means the Historian's context runway is narrowing and query quality may degrade. The generational trigger is a fallback for edge cases where query volume is low but the Historian has been running long enough that a fresh session with updated boot context (newer transfers, grown index, richer channel FAQ) would serve the lineage better than an aging session with stale boot data. At either threshold, the Historian has been answering queries and accumulating context for potentially years of working agent generations. The successor must inherit that accumulated knowledge.

**Succession protocol:**

1. **Write a Historian transfer** at `agents/[name]/operations/[name]-historian/historian-transfer.md`. Contents: retrieval pattern notes (which queries are most common, which artifacts are most frequently accessed, which index entries proved most valuable), index corrections discovered during the session, and any gaps in the card catalog that future disposable curators should address.
2. **The channel persists.** The Historian channel is not archived at succession — it carries forward as the compounding FAQ. The channel CURATOR manages its own lifecycle. The successor Historian boots with the channel history already in context.
3. **The successor Historian boots** from: the current lineage index (maintained by disposable CURATOR agents), the channel history (compounding FAQ), the last 3–5 transfers (hot memory), and the Historian transfer (operational notes). The successor inherits the full retrieval knowledge of its predecessor through the channel — not just what artifacts exist, but what questions have been asked about them and what the answers were.

**Multi-generational boundary requirement:** A Historian must always maintain context depth spanning **at least 3–5 generations** of the working agent. This is the minimum viable depth for institutional memory to function. If a Historian's context is too shallow to span multiple generations, it is not yet a Historian — it is just a slow ASA. The depth is the value.

**Graceful degradation:** If a Historian crashes without writing a transfer, the successor can still boot from the index and channel alone. The transfer adds retrieval pattern intelligence but is not load-bearing for basic functionality. This mirrors the crash-safe property of living transfers (ADR-014). The card catalog (maintained by disposable curators) and the compounding FAQ (in the channel) are the two crash-safe artifacts. Both persist independently of the Historian's session.

**Engineering mapping:** Historian succession maps to index versioning and cache migration in a coded system. The channel archive is the migration payload — it carries the "learned" retrieval patterns forward. The multi-generational boundary requirement maps to a minimum cache depth guarantee in a production system — the retrieval layer must always hold enough data to answer queries across a defined time horizon.

### 7. Cross-Lineage Queries

A working agent may need history from another agent's lineage. For example, Vela V12 might need to know what architectural decision Argus A5 made about folder conventions.

**Cross-lineage protocol:**

- The working agent posts the cross-lineage question to **their own Historian channel** with a tag: `[CROSS: argus]`.
- The working agent's Historian acknowledges the question and notes it cannot answer from its own lineage.
- The working agent (or Mike) relays the question to the target Historian's channel.
- The target Historian answers in its own channel.
- The originating Historian records the cross-reference: "See `channels/argus-historian.md`, entry [date]."

**Historians do not read each other's channels directly.** This is a deliberate constraint — it preserves lineage isolation, keeps context budgets scoped, and maintains clear ownership of historical knowledge. Cross-lineage queries go through the working agents, not through a back-channel between Historians.

**Engineering mapping:** This maps to scoped access control in a multi-tenant retrieval system. Each lineage is a tenant with its own index and query history. Cross-tenant queries are routed through an API gateway (the working agent), not through direct database access.

### 8. Bootstrap Protocol

The Curator layer is not fully populated yet. Nine required CURATOR.md files exist (Vela's 5 + Argus's 2 + the crew log CURATOR + Orpheus's Primary Source CURATOR), but many historical folders have no CURATOR and no curated archive structure. The Historian must work during this bootstrap period.

**Bootstrap mode:**

- The Historian operates from the lineage index regardless of whether the underlying folders have CURATORs.
- When a file read targets a folder without a CURATOR, the Historian reads the raw file directly. This costs more context than reading a curated/compressed version, but it works.
- As CURATORs are written and curator agents compress historical content, the Historian's file reads become cheaper (compressed warm-tier content is smaller than raw hot-tier content).
- The Historian logs which folders it reads that lack CURATORs. This creates a natural demand signal — folders the Historian accesses frequently but that lack CURATORs should be prioritized for CURATOR creation.

**The bootstrap creates a feedback loop:** Historian access patterns reveal which historical content is most valuable, which informs CURATOR priority, which improves the Historian's efficiency. The two systems co-evolve.

**Engineering mapping:** Bootstrap mode maps to a hybrid retrieval system that can query both structured (indexed/curated) and unstructured (raw) data sources, with a migration path toward full structuring. The demand signal maps to analytics-driven index optimization.

### 9. Historian Infrastructure

**Files created per Historian:**

| File | Location | Purpose |
|------|----------|---------|
| Activation file | `agents/[name]/operations/[name]-historian/activate.md` | ADR-006 compliant activation with context framing, lineage scope, channel location, and query processing instructions |
| Lineage index | `agents/[name]/operations/[name]-historian/lineage-index.md` | The card catalog — compact metadata catalog of all lineage artifacts. First resource on every query. |
| Index CURATOR | `agents/[name]/operations/[name]-historian/CURATOR.md` | Governs index grooming. Disposable curator agents maintain the index; the Historian never grooms its own index. |
| Channel | `channels/[name]-historian.md` | Query/answer interface between working agent and Historian |
| Channel CURATOR | `channels/[name]-historian-CURATOR.md` | Preservation instructions for the Historian channel (all Q&A verbatim) |
| Historian transfer | `agents/[name]/operations/[name]-historian/historian-transfer.md` | Succession document (written at 75% context) |

**Historian channel CURATOR defaults:**

```yaml
---
type: curator
owner: "[name]-historian"
schedule: "monthly"
hot_window: "30d"
warm_window: "90d"
cold_after: "180d"
---
```

All Q&A pairs are preserved verbatim regardless of tier. Compression applies only to operational noise (session-boundary markers, acknowledgment messages). The compounding FAQ is the Historian's most valuable output — it is never compressed.

### 10. Activation Sequence

**To activate a Historian for the first time:**

1. **Build the lineage index.** Spawn a disposable child agent to read all transfers, reflections, and key artifacts for the agent's lineage. The child creates `lineage-index.md` with one entry per artifact. This is a one-time bootstrap — the child burns its own context, not the Historian's.
2. **Write the index CURATOR.** Create the `CURATOR.md` that governs ongoing index maintenance. This CURATOR's instructions tell future disposable curator agents how to add new entries, validate paths, and groom the index at each generation transition.
3. **Write the activation file.** Use the Historian activation template (see `templates/historian-activate-template.md`). Include: lineage scope, channel location, index location, query processing rules, context budget constraints, and succession protocol.
4. **Create the channel.** Initialize `channels/[name]-historian.md` with a header noting the Historian's scope and the channel protocol.
5. **Create the channel CURATOR.** Write the CURATOR.md that preserves all Q&A pairs verbatim.
6. **Register.** Add the Historian channel and both CURATORs (index + channel) to the curator registry. Post awareness to ops.md.
7. **Activate.** Mike activates the Historian session. The Historian reads its activation file, loads the card catalog (lineage index) and last 3–4 transfers, and posts a ready signal to its channel. The Historian consumes the index — it never maintains it.

**Recommended rollout:**

Start with the **Vela Historian** — deepest lineage (V1–V10+), most frequent cross-generational questions, and the agent most likely to benefit from queryable history (as Chief of Staff, Vela coordinates across all agents and needs institutional context constantly). Prove the pattern with one Historian before deploying the remaining three.

### 11. Relationship to Existing Infrastructure

| Component | Relationship to Historian |
|-----------|--------------------------|
| **Living Transfer** (ADR-014) | The Historian holds the last 3–4 transfers in live context. Transfers are the primary source for recent-generation queries. The transfer's Essential Files section informs which artifacts the Historian should prioritize indexing. |
| **ASA** (ADR-012) | The ASA orients the working agent. The Historian orients the working agent's *understanding of history*. They serve different time horizons: ASA covers current state, Historian covers lineage depth. No overlap. |
| **CURATOR** (ADR-020) | The deepest coupling in the architecture. Three roles: (1) CURATORs in historical folders tell the Historian what's been compressed and where to find full versions. (2) The index CURATOR governs the card catalog itself — disposable curator agents groom the index so the Historian never has to. (3) The channel CURATOR preserves the compounding FAQ. **The Curator preserves and catalogs. The Historian retrieves. Disposable agents do the maintenance. Long-lived agents do the retrieval. Cost asymmetry by design.** |
| **Curator Controller** | The Controller dispatches curator agents that maintain the index and channel. The Controller does not dispatch the Historian — the Historian is a persistent session, not a disposable agent. The Controller is the scheduler; the Historian is the consumer. |
| **Working channels** | Working channels are ephemeral (daily cleanup). Historian channels have long retention. The two channel types serve different purposes and follow different CURATOR policies. |
| **Standing auditors** | Auditors detect drift. The Historian detects knowledge gaps. Complementary: if an auditor finds a stale reference, the Historian can provide the historical context for why that reference existed. |

## Rationale

### Why one Historian per lineage, not one unified Historian?

A unified Historian would need to hold the index, transfers, and channel history for all primary agents simultaneously. At four lineages of 10+ generations each, the boot cost alone would consume 30-40% of context — defeating the purpose of a long-lived, low-burn session. Lineage scoping keeps each Historian focused, long-lived, and within budget.

The scoped model also maps cleanly to a future engineering implementation: each lineage is a tenant in a multi-tenant retrieval system with isolated indexes and query histories. Unification happens at the platform level, not the agent level.

### Why a persistent session, not a scheduled task?

The Historian's value compounds as it holds context across multiple queries. A scheduled task would boot fresh each time, losing the accumulated query context that makes cache hits possible. The persistent session model means query #30 benefits from the context accumulated by queries #1–29.

The tradeoff is that the Historian requires Mike to activate it (or a Cowork persistent session). This is acceptable because the Historian is a low-frequency, high-value service — it doesn't need to be always-on, just available when a working agent has a historical question.

### Why a 2-file-read limit per query?

Without a hard limit, a complex query could trigger 5–10 file reads, consuming 10,000–20,000 tokens in a single interaction. A few queries like that would burn the Historian's context as fast as a working agent. The 2-file limit is a circuit breaker that preserves longevity. If a query needs more than 2 files, the Historian provides partial answers and directs the questioner to the remaining sources — this is honest about the Historian's limitations and prevents context-destructive queries.

### Why preserve all Q&A pairs verbatim in the channel?

The channel is the Historian's most strategically valuable output. It is a compounding knowledge base that grows more useful with every generation. Compressing Q&A pairs would destroy retrieval patterns — the specific phrasing of questions reveals what working agents actually need, and the specific answers reveal what the Historian found most relevant. Both are institutional knowledge. Both must persist.

### Alternatives considered

1. **Embedding-based retrieval** — Code a vector store and RAG pipeline. Rejected for the language-native layer: requires engineering infrastructure that contradicts Argo-OS's markdown-and-filesystem model. This is the correct approach for the Modus product implementation but not for the vault prototype.
2. **Enhanced ASA with historical depth** — Expand the ASA to load more historical context at boot. Rejected: the ASA is already optimized for breadth (vault-wide state), not depth (lineage history). Adding depth would blow the ASA's context budget and conflate two different functions.
3. **Manual queries by working agents** — Working agents read historical files directly when they need them. Rejected: this is the status quo, and it doesn't scale. Working agents burn their own context on historical retrieval, reducing their capacity for actual work.
4. **Shared Historian across lineages** — One Historian for all agents. Rejected: context budget exceeds practical limits, and lineage isolation provides cleaner scoping (see Rationale above).

## Consequences

### What gets easier

- **Historical questions become answerable.** Working agents can query lineage history without burning their own context on file reads.
- **Institutional knowledge compounds.** The channel FAQ grows with every generation. Common questions are answered once and persist.
- **Transfer quality improves.** When writing a transfer, the working agent can ask the Historian what predecessors flagged as unresolved — producing more complete handoffs.
- **Pattern detection becomes possible.** The Historian can identify recurring themes across generations: items flagged repeatedly, decisions revisited, lessons learned and then forgotten. This is impossible for any single generation to see.
- **The memory architecture is complete.** Four mechanisms covering the full lifecycle: creation (transfer), orientation (ASA), preservation (CURATOR), retrieval (Historian). Each mechanism has a clear engineering mapping for the Modus product.

### What gets harder

- **A new agent type to maintain.** Each Historian needs an activation file, a lineage index, an index CURATOR, a channel, a channel CURATOR, and a succession protocol. Mitigation: templates make creation a 15-minute task; the Historian itself is zero-maintenance once running (all maintenance is handled by disposable curator agents).
- **The card catalog must be kept current.** A stale index gives wrong answers. Mitigation: the index is maintained by disposable CURATOR agents dispatched at each generation transition. The Historian never grooms its own index — its context is sacred.
- **Mike activates and monitors Historian sessions.** Persistent sessions require human oversight. Mitigation: Historians are era-spanning entities (estimated 100–150+ generation lifespan). Once activated, a Historian may run for months before needing succession. Oversight is minimal.
- **Cross-lineage queries are indirect.** Working agents relay questions between Historian channels rather than querying directly. Mitigation: this is a deliberate tradeoff for lineage isolation and context budget scoping. The coded Modus version will implement direct cross-lineage retrieval.

### What changes

- The intergenerational memory model is formally a four-mechanism architecture, not a triad.
- Historian channels are a new channel type with different retention policies than working or topic channels.
- The CURATOR infrastructure gains a new consumer — the Historian reads what CURATORs preserve, creating a natural feedback loop.
- The lineage index becomes a first-class artifact maintained at each generation transition.
- The channel-as-compounding-knowledge pattern is formally documented and can be applied to other accumulating-value contexts beyond historical queries.
