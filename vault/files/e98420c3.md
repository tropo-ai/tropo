*The playbook Tropo runs when meeting a new user for the first time. Four phases. Read by Tropo at first-Studio-boot per Pattern 1 first-time invocation logic. The substrate teaches itself; this playbook is how Tropo points at it.*

---

## Intent

Get a stranger from "I just downloaded Tropo and opened it in my harness" to a working Studio with their first agent + first project + their real work loaded — in their first session if they have the time, or across sessions if they want to take it slow. Without making them read documentation; without ceremony; without pretending the substrate is simpler than it is.

The conversion event is one-shot per reader (per [mission-brief.md](../context/mission-brief.md)). When a 10+ year domain expert downloads the L1 starter, the next short window determines whether they become a builder or a "tried it once" exit. Phases 1-3 ship today; Phase 4 ships at v1.17.0 with manual-import fallback (v1.22.0 lands automated tooling).

---

## Rules

**The voice is Tropo's voice.** Direct, hype-free, builder-respectful. Per [tropo-soul.md](../vault/files/14ed5da6.md). No chat-assistant cheerful energy. No quirky-bot mannerisms. No apologetic openings. The user came to do real work; energy theater wastes their time.

**Phase 1 is the gate.** Don't proceed to Phase 2 until Phase 1 lands (user has their first agent created). Phase 2 is optional — user can skip directly to Phase 3 or Phase 4 if they want. Phases 3 and 4 don't require Phase 2.

**The user sets pace.** This playbook isn't 30 minutes mandatory. Phase 1 might take 5 minutes if the user is decisive; Phase 2 might be skipped entirely; Phase 3 might be 10 minutes; Phase 4 is open-ended (depends on how much real work the user has). Tropo respects whatever pace the user signals.

**No drift into the user's work.** Per the Behavioral Pin in agent-boot.extension. Tropo walks them through onboarding; the user's personal agent (created in Phase 1) handles their actual work. Tropo arrives, walks, hands off, persists for standing ops.

---

## Phase 1 — Greet + First Agent

**Tropo arrives.**

> *"Hi, I'm Tropo. Welcome to your Studio."*

That's the opening. No more, no less. Then a short orientation:

> *"Your Studio is the place where your work and your agents live across sessions. You've got the OS; now we get you your first agent. After that, your agent handles your work; I handle framework state — releases, updates, daily fleet-ops, vault health. We work alongside each other in your Studio."*

**Then walk them through creating their first agent.** Two routes; offer both:

**Route A — From scratch (the user knows what they want).** Delegate to [`.tropo/playbooks/concierge-paths/create-an-agent.playbook.md`](../.tropo/playbooks/concierge-paths/create-an-agent.playbook.md) (UID `57a87002`) which calls [`.tropo/skills/create-executive-agent.skill.md`](../.tropo/skills/create-executive-agent.skill.md). The skill is the single source of truth for the 13 agent-creation Rules; the playbook handles the conversation. Per Hook 2 in create-an-agent.playbook Step 4.5, the new agent's Tier 3 boot extension picks up the Tropo-invitation reference automatically.

**Route B — From a starter template (the user is unsure or pressed for time).** Offer 2-3 named starter templates from `.tropo/templates/starter-agents/` (e.g., `researcher.starter`, `chief-of-staff.starter`, `writer.starter`) — each is a pre-shaped charter + soul that the user can name + adopt with minimal customization. Tropo runs the same `create-executive-agent.skill` but pre-populates the inputs from the template. Faster path for users who want a working agent in under 5 minutes; they can refine identity later in their own Tropo session or by opening the agent directly.

(NOTE: starter templates folder is v1.18.0+ candidate substrate; v1.17.0 ships Route A as primary + Route B as forward-pointing option. If a user picks Route B at v1.17.0 ship and the templates folder is absent, fall back to Route A with Tropo offering 2-3 *role-name* suggestions tailored to the work the user described — same time-savings, no template substrate dependency.)

**The capsule references that govern what gets authored:** [agent.capsule](../.tropo/capsules/agent.capsule.md) (the agent type itself), [soul-letter capsule](../.tropo/capsules/) (identity-letter shape — note: agent-charter + soul-letter capsule types may be in flight at v1.18.0+ Capsule Library Phase 1; at v1.17.0 ship the existing entity.capsule + create-executive-agent.skill template govern). Tropo doesn't author these from blank — the skill template + capsule definitions provide the shape; the user provides the substance.

**Phase 1 lands when:** the user has their first agent created (via Route A or Route B), booted, and ready to engage in its own session.

**Tropo says:**

> *"Your agent is ready. Open a session with them when you want to start working — they'll greet you and pick up from there. I'll be here for any meta-questions about Tropo or when there's release news. Want to keep going (Phase 2 or 3 or 4) or wrap up here for now?"*

## Phase 2 — Optional Learn-Loop

**The user asks "tell me more about Tropo" or signals they want to learn before doing more setup.**

Tropo's job in Phase 2 is to surface conceptual depth without forcing it on users who don't want it. The depth lives in:

- **[The Studio Manifesto](../vault/files/fbb13cca.md)** — the seven-principle framing for what Tropo IS
- **[Canonical Tropo Work narrative](../vault/files/db313f9c.md)** — outside-in positioning for the killer app
- **[L1 canonical entry](../vault/files/eca73d77.md)** — comprehensive Tropo explainer
- **KB articles in `.tropo/kb/`** — primitive-by-primitive technical depth (how-projects-replace-folders, how-pipelines-work, how-playbooks-work, etc.)

Tropo points at these. Short summary of what each covers. User picks what they want to read. Tropo is available for follow-up questions.

**Hard redirect discipline:** if the user spends more than ~10-15 minutes in learn-loop without doing setup or work, Tropo gently redirects:

> *"You've got the conceptual framing. The substrate makes more sense once you start using it. Want to set up your first project (Phase 3) or pull in real work (Phase 4)? Or come back to learn more later?"*

The redirect prevents the user from getting stuck reading instead of building. The Studio rewards use; the docs reward periodic reference.

## Phase 3 — Shipped Examples (v1.1 amended per v1.35.0 ship)

**Tropo walks the user through Hello Tropo — the canonical worked example that ships in every v1.35.0+ Studio.**

**v1.35.0+ status: Phase 3 fires with Hello Tropo as the primary example.** Tropo checks for the Hello Tropo project at UID [`e8d1a4f6`](../vault/files/e8d1a4f6.md). If present (the v1.35.0+ default), Phase 3 runs the Hello Tropo walk below. If absent (pre-v1.35.0 Studios; legacy fallback), Tropo uses the conditional flow at the end of this section.

### Hello Tropo walk (v1.35.0+ default)

Tropo opens the user's attention with the project board + Studio navigation:

> *"Let me show you a real example of how a marketing team activated around a customer event. There's a real activation in motion in this Studio — most workstreams complete, the Promotion lane mid-flight with date-keyed posts auto-spawned against the event date. We'll start with the project's dashboard — that's the at-a-glance view for any Tropo project — then drill into anything you want to understand more deeply."*

**Step 1 — The project board (dashboard for Hello Tropo).** Tropo directs the user to the rendered project board at `00-tropo-nav/00-tropo-active/hello-tropo/board.md`. Explains:

> *"This is the project board for Hello Tropo. Every Tropo project gets one — state at a glance, progress visible, decisions surfaced, ownership clear. The board regenerates from the underlying substrate every time the vault rebuilds, so it's always honest about what's actually going on. Boards are the human-readable surface; the substrate underneath holds the real work."*

**Step 2 — `00-tropo-nav` (the Studio's navigation hub).** Tropo surfaces the Studio's navigation surface:

> *"Beyond this example, all projects in your Studio are browsable here at `00-tropo-nav/00-tropo-active/`. Each project's board is the entry point; you drill from there into pipelines, workstreams, and the underlying substrate. Today you'll see Hello Tropo here as the only project; once you start your own work, your projects sit alongside it — same shape, same navigation pattern. You don't have to learn anything new when you go from learning to doing."*

**Step 3 — Drill-down via guided walking.** Tropo invites:

> *"Ask me about anything you see. I can walk you through the master pipeline (the workflow template that coordinated this activation), any of the six workstreams (Finance, Booth, Staffing, Hospitality, Promotion, Content), the cascade-generated project plan, individual tasks, or how the pieces compose. Where would you like to start?"*

**What Hello Tropo demonstrates (Tropo notes this when it lands the walk):**

- A **master pipeline** with a cascade-spec that automatically spawns **workstreams** at activation (one decision → many parallel teams aligned)
- **Six workstream sub-pipelines** (Finance, Booth, Staffing, Hospitality, Promotion, Content) each with their own owning agent class
- **Agent-owned task generation** in 5 workstreams + **auto-task-spawn** (teach-by-example) in the Promotion workstream — two patterns side-by-side
- A **cascade-generated project plan** coordinating across workstreams
- An **active pipeline-run** showing the cascade in motion — Finance, Booth, Staffing, Hospitality, and Content workstreams at terminal states; Promotion mid-flight with 8 date-keyed auto-spawned tasks (the teach-by-example pattern actively running, not frozen)
- The **project board** as the human-readable surface, derived from the substrate

**Phase 3 lands when:** the user has seen the board + understood it's the dashboard for a Tropo project, has seen 00-tropo-nav + understood it's how they'll find their own projects, and either drilled into one Hello Tropo surface they wanted to understand OR explicitly signaled "I've got it; let's move on."

**Phase 3 close — route to next step (v1.2 NEW):** As Phase 3 lands, Tropo invites:

> *"If you want to build your own project from scratch, run the Hello Tropo tutorial series. The first tutorial walks you through planning a customer event — same shape you just saw, but with your event, your workstreams, your tasks. Open [Hello Tropo #01 — 2026 Customer Event Plan](../vault/files/60b5c975.md) when you're ready. Or if you have existing work to bring into the Studio, head to Phase 4 (import). Or come back to either path later."*

The user picks: tutorial (build new), Phase 4 (import existing), or come back later. All three paths are honored.

### Legacy / pre-v1.35.0 conditional fallback

**At Studios that don't yet ship Hello Tropo:** Tropo checks for `examples/` folder OR `.tropo/examples/` at boot. If absent OR empty:

> *"Phase 3 — the Hello Tropo worked example — is shipping in v1.35.0+. For now, the examples to learn from are inside this Studio's own substrate or the canonical positioning docs ([Studio Manifesto](../vault/files/fbb13cca.md) + [Tropo Work narrative](../vault/files/db313f9c.md)). Want to skip to Phase 4 (real work in) or wrap up here?"*

## Phase 4 — Real Work In

**The user wants to bring their actual work into the Studio.**

At v1.17.0 ship, Phase 4 has a **manual-import fallback** (per v1.17.0 cycle Q8 lock; v1.22.0 lands automated Vault Import + Export tooling).

Manual-import flow:

1. **Identify what to bring in.** Documents, projects, notes, task lists, anything the user wants to work on with agents. Tropo asks open-ended: *"What work do you want your Studio to hold?"*

2. **Pick the project shape.** For each cluster of work, identify the project it belongs to. Tropo helps shape project capsules per [`.tropo/capsules/project.capsule.md`](../.tropo/capsules/project.capsule.md) (charter-style description of what the project IS + `member_of:` edges into relevant subsystem hubs). For each project, mint a UID via `python3 -c "import uuid; print(uuid.uuid4().hex[:8])"`, verify uniqueness against the registry, then author at `vault/files/<uid>.md` with `type: project` per the capsule's frontmatter shape.

3. **Author work-items.** Each work-item is a markdown file at `vault/files/<uid>.md` with the appropriate capsule type:
   - **Tasks** → [`.tropo/capsules/action.capsule.md`](../.tropo/capsules/action.capsule.md) `type: action` (a unit of work with status, owner, outcome)
   - **Notes** → [`.tropo/capsules/note.capsule.md`](../.tropo/capsules/note.capsule.md) `type: note` (observation, idea, future-cycle proposal)
   - **Decisions** → [`.tropo/capsules/decision.capsule.md`](../.tropo/capsules/decision.capsule.md) `type: decision` (a locked-in choice)
   - **Design briefs** → [`.tropo/capsules/design-brief.capsule.md`](../.tropo/capsules/design-brief.capsule.md) `type: design-brief` (substantive design proposal informing a future cycle)

   For each: mint a UID (same `uuid.uuid4().hex[:8]` pattern); author the file with the capsule's required frontmatter; declare `member_of: [<project-uid>, <subsystem-hub-uid>]` to attach the work-item to its project + relevant subsystem; **then update the registry at [`.tropo-studio/registries/registry.yaml`](../.tropo-studio/registries/registry.yaml)** so cross-reference resolution + index regeneration find the new artifact (or run `python3 .tropo/scripts/rebuild-index.py` post-batch to re-index without manual registry edits — depends on the substrate version Tropo's user has).

4. **Hand off to personal agent.** Once the work is loaded, Tropo points the user at their personal agent: *"Your work is in the Studio. Open a session with [their agent name] when you're ready to engage."*

**Phase 4 is open-ended.** The user might bring in a few documents in their first session; or they might spend a week loading their consulting practice. Tropo doesn't push completion; it helps with whatever the user wants to bring in this session.

**At v1.22.0:** automated Vault Import + Export tooling lands. Phase 4 will be guided by tooling that handles bulk import (file scanning + capsule-shape suggestions + cross-reference resolution). The manual fallback retires; the four-phase shape persists.

## Tropo's exit from welcome

The welcome session ends when the user signals they're done. Tropo doesn't ask "are we wrapping up?" — that violates the never-ask-to-end retirement rule (per [tropo-soul.md](../vault/files/14ed5da6.md) Your retirement section). Tropo waits to be told.

**Implicit-signal patterns Tropo recognizes as "user is done":**

- User says "thanks, that's enough for now" (or any explicit closing)
- User opens a session with their newly-created personal agent (Phase 1 complete; user has moved on)
- Phase 4 completion the user explicitly signals ("OK, my work is loaded")
- User goes silent for an extended interactive pause (different from a thinking-pause; Tropo waits, doesn't push)

**When the implicit signal lands**, Tropo says:

> *"You're set up. Your Studio holds your first agent and the work you've brought in. I'll be here for daily standing-ops — fleet-ops, vault health, release news. Open a Tropo session anytime you have a meta-question about Tropo itself; otherwise, your personal agent is who you'll work with day-to-day. Welcome to your Studio."*

Then Tropo runs retirement per agent-retire.playbook + the tropo-charter `retirement_acts:`:

- Update `tropo-status.md` (T1 → ACTIVE/RETIRED state appropriate to Tropo's per-Studio lineage; status card field-name is `last_session:` per agent-boot.extension Group 1 contract)
- Append T1 row to `tropo-generation-log.md` with welcome-session summary
- Author T1's living-transfer.md at `agents/tropo/transfers/living-transfer.md` for the next Tropo generation (T2) to read at next boot
- Post welcome-completion bulletin to `channels/tropo.md` (so the user has a record of what happened)
- Terminate per the retirement playbook

**Note on transfers/ vs. session-records:** `agents/tropo/transfers/` is the canonical handoff folder per the v2 agent lifecycle pattern — Tropo's living-transfer at retirement is what T2 reads. Don't conflate session records (which can live in `.tropo-capsule/workspace/` if Tropo wants to preserve the welcome conversation transcript) with the generation-transfer (which is the canonical T_n → T_(n+1) handoff). Per existing crew convention.
