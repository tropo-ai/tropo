## Intent

A browser-based wizard on tropo-ai.com that walks first-encounter visitors through creating their first Tropo agent. The wizard is **simultaneously a creation tool, a learning experience, and a marketing-conversion funnel** — visitors leave with an understanding of how Tropo agents are constructed AND a personalized agent ready to drop into their first Studio AND their email captured for follow-up.

The wizard **mirrors the conversational shape of Tropo's existing `create-an-agent.playbook` ([57a87002](57a87002.md))** — same questions, same one-at-a-time pacing, same single-word role naming convention. But it runs in a browser instead of inside a Studio, and its primary job isn't producing a functional agent (the real playbook does that in-Studio); its job is **engagement-conversion** with the agent-package output as the cost of admission for the user's email.

**Out of scope:**
- The in-Studio agent activation experience (already handled by `agent-activation.playbook` + `create-an-agent.playbook`)
- The agent's substrate migration into vault on first boot (gated on Argus's playbook from [9e2b4c87](9e2b4c87.md))
- Implementation work (separate deploy after brief lock)
- The broader website hub vision (captured separately in [e8d3a275](e8d3a275.md); this brief is Layer-0 only)
- Generic agent import/export from other frameworks (Capability B in [9e2b4c87](9e2b4c87.md); separate work)

## Decisions made

The following are committed by this brief (not open at implementation time):

1. **Audience = curious first-encounter visitor.** Someone who landed on tropo-ai.com from a Substack link, social referral, or word-of-mouth. Has never installed Tropo. Is exploring whether this thing matters to them. The wizard's design optimizes for this audience; sophisticated existing-Studio users can skip the wizard and use `create-an-agent.playbook` directly in their Studio.

2. **Wizard shape = 5 conversational steps + email-capture gate + instant download.** Total ~10-15 minutes for an engaged visitor (longer than the 5-min in-Studio playbook because each step includes pedagogy). Chat-style conversation interface, not multi-step form. Step indicator at the top. Live preview card on the right showing the agent forming.

3. **Each step has a "why" pedagogy beat alongside the question.** The wizard isn't just collecting inputs — it's teaching Tropo's mental model as the visitor builds. By the end, the visitor understands what soul / charter / status / memory ARE, why Tropo structures agents this way, and how their specific agent will behave.

4. **Post-wizard explainer = overview page with expandable sections (Option A).** "Meet your agent" landing page shows their agent + a list of files with one-line descriptions + expandable sections for deeper explanation. Memory gets its own dedicated subscreen because it's the most surprising aspect for first-encounter visitors.

5. **Output artifact = flat `agents/<name>/` zip with README.** Wizard generates agent files at the older flat substrate paths, NOT the v1.21.0.1 vault-based pattern. User drops the folder into their Studio's `agents/`, runs activation. The agent self-migrates to vault/files/ substrate on first boot (gated on [9e2b4c87](9e2b4c87.md)). The user never sees vault/ in the wizard's narrative.

6. **Email-capture lands between step 5 and download — instant flow, no email-link redirect.** Visitor enters email + name + newsletter opt-in (unchecked default per [Open Decision #6 on the project plan](6d4f7a82.md)); server stores in Supabase emails table with `source: 'agent-builder'` + agent details in metadata; zip generates server-side; download triggers immediately. The 10-15 min investment makes instant-download the right call here; email-link friction at the highest-cost drop-off moment.

7. **Brand voice = Tropo Workshop-style.** Plain English. Conversational. Doesn't expose Tropo's substrate vocabulary (UID, frontmatter, schema, ledger) to the visitor — same discipline `create-an-agent.playbook` follows. Same brand tokens (bone background, ember accents, ink text). Same wordmark presence as the placeholder we shipped 2026-05-15.

8. **Substrate composability.** The wizard's collected inputs map cleanly onto the inputs that `create-executive-agent.skill` (the in-Studio agent-creation skill) expects: `founder_name`, `agent_name`, `agent_role`, `agent_mission`, `agent_values`. Same fields, same conventions. Wizard output is interchangeable with what the in-Studio playbook produces.

## Architecture

### Wizard flow — 7 stages

```
Step 1: Welcome + your name
        ↓
Step 2: Your agent's role
        ↓
Step 3: What matters in how they work
        ↓
Step 4: First action
        ↓
Step 5: Behavioral guardrails (optional)
        ↓
Step 6: Email-capture gate
        ↓
Step 7: Meet your agent + download
```

### Per-step details

| # | Question | Input | Server stores | "Why" pedagogy beat (shown alongside the question) |
|---|---|---|---|---|
| 1 | "What should we call you?" | Single text field | `founder_name` | "Tropo agents remember you across sessions — even after generations turn over. Your name is the first piece of context they'll carry forward." |
| 2 | "What kind of work should this agent handle?" Open conversation; wizard proposes a single-word role name (`researcher`, `drafter`, `analyst`); user confirms or edits. | Open text → role name + role description | `agent_name` (single-word lowercase), `agent_role` (one sentence) | "We name agents by their role, not their personality or domain. 'Researcher' outlives any specific research project. The role is durable; the work changes." |
| 3 | "What matters most in how they work?" Open conversation; wizard extracts 2-3 short value statements from the response and shows them for confirmation. | Open text → 2-3 values | `agent_values` (array of short statements) | "These are your agent's soul. Not preferences — the disposition the agent operates from. When the agent has to make a judgment call you didn't anticipate, the soul is what it falls back on." |
| 4 | "What's the first specific thing you'd want this agent to do?" | Open text | `agent_first_action` (one sentence) | "Concrete and narrow. Sprawling agents never ship; focused ones do. Your agent earns wider scope through doing good work." |
| 5 | "Anything they should never do?" (Optional — visitor can skip.) Wizard extracts 1-2 hard rules. | Open text or skip | `agent_hard_rules` (array of 0-2 short rules) | "Hard rules are boundaries the agent doesn't cross — even when it would help in the moment. Boundaries make verification possible later." |
| 6 | Email-capture screen: email + name + newsletter checkbox (unchecked default) | email, name, newsletter_opt_in | Supabase `emails` table: `source: 'agent-builder'`, metadata: { agent_name, agent_role, agent_first_action } | Brief framing: "Where should I send your download?" Implies value-in-exchange-for-info, not gating. |
| 7 | "Meet [agent_name]" overview page with expandable sections + download CTA | (no input — just navigation) | — | Memory subscreen explains how memory accumulates and compacts over time. |

Brand-styled (bone background, ember accents, ink text). Updates with subtle animation as each new piece lands. Functions as continuous reinforcement that "you are building something real."

### Visual / UI shape

- **Chat-style conversation interface** on the left ~60% of the viewport. Each visitor input becomes a chat bubble (ink-on-bone), each wizard response a contrasting bubble (ember accent). Pacing one question at a time; visitor types or selects; wizard responds; advances.
- **Step indicator** at the top: "Step 3 of 6: Your agent's values" — gives visitors progress signal; reduces abandonment.
- **Live preview card** on the right ~35% of the viewport. Updates as answers come in.
- **"Why" pedagogy beat** rendered as a small italicized callout under each question, in muted ink. Visitor can read it or skip past. Not gating.
- **Transitions** between steps: subtle fade + advance, not flashy.
- **Mobile responsive:** preview card collapses to a collapsible top section on narrow viewports; conversation gets full width.

### Memory explainer subscreen

A separate page (e.g., `/agent-builder/memory`) linked from the "How memory works" expandable section. Renders standalone — visitors can deep-link here from anywhere.

Content:

```
How memory works in Tropo

Your agent remembers you across sessions. Today their memory
is empty. Here's what happens as they work with you.

[Visual: animated illustration of memory accumulating across sessions]

  Session 1 — your agent learns your name, your work style,
              and the first task you assigned.

  Session 2-N — every session adds new context: decisions
                you made together, things that worked, things
                that didn't, the patterns of your collaboration.

  Compaction — over time, your agent's memory gets distilled.
               A "curator" agent reviews accumulated entries,
               folds older details into condensed summaries,
               keeps the active memory focused.

  Result — your agent improves automatically. You don't have
           to manually catalog every session. The substrate
           handles it.

This is one of the things that makes Tropo different from
chatbots. Chatbots forget you when the tab closes. Tropo
agents inherit knowledge across generations.

[Subtle CTA: "See your agent's memory file (memory-current.md)"
 — links to a sample file rendered inline]
```

Tone matches the masthead voice from the Substack article. Concrete, plain, not condescending.

### Server-side architecture

```
Browser (Next.js client)
        │
        │ POST /api/agent-builder/generate
        │ { founder_name, agent_name, agent_role,
        │   agent_values, agent_first_action, agent_hard_rules,
        │   email, email_name, newsletter_opt_in }
        ↓
Vercel serverless function
        │
        ├─ Validate inputs (email format, required fields, name format)
        ├─ Store email row in Supabase emails table
        ├─ Generate agent files via template + interpolation:
        │    agents/<name>/<name>-activation.md
        │    agents/<name>/<name>-charter.md
        │    agents/<name>/<name>-soul.md (or inline in boot-extension)
        │    agents/<name>/<name>-status.md
        │    agents/<name>/<name>-boot.extension.md
        │    agents/<name>/memory/memory-current.md (empty scaffold)
        │    agents/<name>/README.md
        ├─ Zip all files into <name>-agent.zip
        ├─ (Optional v1.1+) Trigger Resend email with personalized follow-up
        └─ Return zip as streaming download response
```

**File templates** are versioned and stored in the repo at `tropo-ai/website/lib/agent-templates/`. Each template is a parameterized markdown string with placeholders for the wizard's collected inputs. Server-side rendering uses simple string interpolation (no template engine needed — straightforward enough).

**Zip generation** uses Node's built-in streaming patterns + a lightweight zip library (e.g., `archiver` or `JSZip`). The response is streamed to the browser; no temporary file storage needed.

**Templates evolve with Tropo's substrate.** When the agent file conventions change in Tropo (e.g., Argus authors a v3 agent shape), the website's templates need updating. Couplings are documented in CONTRIBUTING.md.

### Email capture mechanics

When the visitor submits at step 6:

1. **Client-side validation**: email regex, non-empty name, newsletter checkbox state captured
2. **POST to serverless function** with all wizard inputs + email
3. **Server validates again** (defense in depth — never trust client)
4. **Supabase insert into `emails` table**:
   ```sql
   INSERT INTO public.emails (
     email, name, source, newsletter_opt_in, captured_at, metadata
   ) VALUES (
     '<email>',
     '<name>',
     'agent-builder',
     <bool>,
     NOW(),
     '{"agent_name": "...", "agent_role": "...", "agent_first_action": "...", "agent_values": [...], "agent_hard_rules": [...]}'::jsonb
   );
   ```
5. **Zip generation** runs immediately after insert succeeds
6. **Download streams to browser**; visitor's browser handles the file

If email-insert fails (duplicate email, validation error), the wizard surfaces the error inline (chat-bubble style — "I already have this email — is it possible you've been here before?"). Visitor can correct or proceed.

### Newsletter handling

Newsletter opt-in checkbox at email-capture is unchecked by default (GDPR-clean). If checked, the visitor's `newsletter_opt_in: true` is stored. A future broadcast endpoint (not in this brief's scope) reads opt-in users and sends them updates via Resend.

## Dependencies

**Substrate prerequisites:**

1. **Argus's agent-import playbook from [9e2b4c87](9e2b4c87.md) — Capability A.** Gates the agent's self-migration from flat-shape to vault-shape on first boot. NOT strictly blocking — the wizard's flat-shape output activates fine under Tropo's existing playbook (which supports legacy flat-shape) even without the migration. But the experience is best when Argus's work lands.
2. **Supabase emails table schema.** Defined in Open Decision #6 of the project plan; needs to actually be created in the Supabase project. ~30 min of setup work in the Supabase Studio SQL Editor.
3. **Resend transactional email setup.** Already in place from Deliverable #6.
4. **The `tropo-ai/website` Next.js app structure.** Already in place from Deliverable #3.
5. **The `create-executive-agent.skill.md` input contract.** Wizard inputs must remain compatible with what the skill expects so wizard-built agents can later be re-processed through the in-Studio playbook if needed.

**Engineering dependencies (npm install at implementation):**
- `archiver` or `JSZip` — zip file generation in Node
- Existing: `@supabase/supabase-js`, `resend`, `react` (Next.js stack from the website project)

**Code dependencies:**
- Email-capture pattern from the eventual `/download` page (brief #10 [8b3c1d72](8b3c1d72.md)) — should be a shared component if both surfaces capture emails the same way.

## Open Decisions

1. **Exact name extraction logic for step 2.** When the visitor describes their agent's role ("I want an agent that helps me draft client emails"), the wizard needs to propose a single-word name (`drafter`). The extraction logic — naive keyword pattern, GPT-call, predefined synonym table — is an implementation decision. Lean: simple keyword + synonym table at v1 (deterministic, no LLM call needed; ~20-30 common roles covered). LLM-assisted naming is a v1.1+ enhancement.
2. **Soul value extraction logic for step 3.** Same shape — visitor describes in their own words; wizard extracts 2-3 short value statements. Lean: LLM call at this step (this one is harder to keyword-match deterministically; LLM gives better quality). One Anthropic API call per wizard run; cost is negligible (~$0.01 per agent built).
3. **Live preview card animation polish.** How dynamic should the card feel? Lean: subtle fade-in for new content, no flashy effects. Polish-level for v1.
4. **Telemetry.** Do we capture where visitors abandon the wizard? Step-level analytics would inform iteration but require analytics tooling. Lean: skip at v1 — the email-capture rate is the only metric that matters for the funnel; we don't need step-level telemetry to know if the wizard is working overall.
5. **Mobile experience.** Does the wizard work on a phone, or is it desktop-first? Lean: works on phone but is desktop-optimized. The conversation interface translates fine to mobile; the live preview card collapses to a top section on narrow viewports.
6. **Accessibility.** WCAG AA at minimum. Color contrast on bone/ember already designed for AA; conversation interface needs keyboard navigation + screen reader announcements. Standard.
7. **Multiple agents in one session.** Can a visitor build two agents in a row (e.g., a `researcher` and a `drafter`)? Lean: not at v1. After the first agent's download, the wizard ends with a "Want another?" CTA that loops back to step 1 — but a clean reload, no carryover. Multi-agent flows are v1.1+.

## Verification at brief lock — `two-instrument` (Argus review)

Per the project plan's Verification Plan, this brief carries `two-instrument` verification discipline. Talos T4 authored; **Argus reviews before lock.**

Argus's review focuses on:
1. **Substrate fidelity.** Do the wizard's generated agent files match Tropo's current agent file conventions? Are the file templates substrate-coherent with v1.21.0.1 patterns where applicable, and the legacy flat-shape where the wizard outputs?
2. **Composability with `create-an-agent.playbook` and `create-executive-agent.skill`.** Are the wizard's input field names + semantics interchangeable with the in-Studio playbook? Could a wizard-built agent later be re-processed through the in-Studio playbook for refinement?
3. **Pedagogy accuracy.** Do the "why" beats accurately describe Tropo's mental model? Are there mischaracterizations that would mislead first-encounter visitors?
4. **Substrate dependency on agent-import playbook.** Is the gating relationship between this brief and [9e2b4c87](9e2b4c87.md) correctly described? Should the wizard ship before or after Argus's import playbook lands?
5. **Memory explainer accuracy.** Does the memory subscreen's description of how memory accumulates and compacts match the actual sa.memory-curator architecture at the user-visible level?

Argus's review may surface adjustments to the wizard's collected inputs, the file templates, or the pedagogy beats. Brief progresses to `status: specify` once Argus signs off.
