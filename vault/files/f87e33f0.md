## How Tasks Flow

**Authoring:** new Documentation primitives are authored by the agent owning the relevant subject — Argus for substrate explainers, Vela for ops-narrative docs, Metis for product-positioning docs, Orpheus for crew-narrative or historical OS storytelling. Each new doc declares `member_of:` including this hub UID `f87e33f0`.

**Per-doc subsystem judgment:** when authoring a new doc, the author asks: *"Does this document the OS-as-a-whole, or does it document a specific subsystem?"* OS-as-a-whole → Documentation. Subject-specific → that subject's subsystem (via `member_of:`). Both are valid; the test is who the reader is.

**Updating on release ship:** at B6 atomic-triangle commit, this hub's `release_history:` row is populated DERIVED via query from the release entry's `subsystems_touched:` per [release.capsule v3.4 Rule 12](../../.tropo/capsules/release.capsule.md). No manual authoring of v1.8+ rows; the substrate computes them.

**Promoting a doc:** if a doc-class file (e.g., a new KB article) substantively documents a subject-specific capability rather than the OS-as-a-whole, retag its `member_of:` from `f87e33f0` to the relevant subject hub. This is the subject-honesty discipline — documentation lives WITH its subject.
