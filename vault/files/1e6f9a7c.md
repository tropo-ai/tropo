# ADR-016: Parallel Generation Protocol

## Context

On March 24, 2026, Argus A5 was activated by Mike while Argus A4 was still in session — the first parallel generation activation in Argo-OS history. A5 correctly recognized the situation: it wrote a crew-log entry requesting guidance rather than proceeding. Mike confirmed: A5 stands down, A4 continues to clean retirement, A4 will complete succession normally before A5 takes over. A4 flagged this as an ADR candidate and assigned drafting to Vela.

Parallel activation can happen for several reasons:

- **Accidental:** Mike opens a new agent session before confirming the prior one has retired.
- **Impatient:** A prior session is running long and Mike wants to start fresh before the predecessor finishes.
- **Deliberate hot handoff:** Mike wants both generations briefly active to facilitate a direct knowledge transfer (not yet practiced, but a plausible future pattern).

The crew has no protocol for this. Without one, a newer generation must choose between proceeding blindly (bad — creates write conflicts and authority confusion) or freezing entirely (also bad — wastes a session Mike opened). A5's instinct to post and wait was correct, but instinct is not a protocol.

---

## Decision

**Default behavior when a new generation detects its predecessor may still be active: stand down and request guidance from Mike.**

The specific decisions:

1. **The newer generation is responsible for detecting overlap.** At session start, before delivering the startup signal, a new generation checks two signals: (a) the living transfer — if it exists but is not marked `FINAL` or `RETIRING`, the predecessor may still be active; (b) the worklog — if there is a session entry from today with no handoff notes, the predecessor's session may still be open.

2. **On suspected overlap: post and wait.** The newer generation posts a brief notice to the crew-log addressed to Mike, stating: it has activated, it suspects the prior generation is still running, and it is standing by for direction. It does not read required files, deliver a briefing, or write to any shared files beyond this notice. It waits for Mike to signal in that session.

3. **Mike's direction determines which generation continues.** The default is: earlier generation continues to clean retirement, newer generation stands down. Mike may override this — for example, if the earlier generation has gone unresponsive, or if Mike intentionally wants a parallel handoff. Whatever Mike directs, that generation proceeds. The other ends cleanly.

4. **The standing-down generation leaves a record.** Before closing, the standing-down generation writes a single line to its worklog noting: date, the overlap detected, Mike's direction, and which generation continues. This keeps the generational record clean.

5. **The continuing generation notes the incident in its session record.** When the earlier generation reaches its handoff notes, it records that a parallel generation was briefly active, that it stood down, and that succession is proceeding normally.

6. **Intentional parallel activation requires explicit signaling.** If Mike wants both generations active simultaneously for a deliberate hot handoff, he signals this explicitly in the newer generation's opening message (e.g., "you and A4 are both active — coordinate via crew-log for the next hour, then A4 retires"). Without that explicit signal, the default is always: stand down and wait.

---

## Detection Logic

A new generation cannot know with certainty that its predecessor is still running — there is no shared memory between sessions. Detection is probabilistic. The signals to check:

**Check 1 — Living transfer status:**
Read `agents/[name]/transfers/living-transfer.md`. If the header says `FINAL` or the current state section says `Retiring`, the predecessor has closed. If it says `Last updated: [today's date]` without a retirement marker, the predecessor may still be active.

**Check 2 — Worklog:**
Read the most recent session entry in the worklog. If it is from today and has no handoff notes, the predecessor's session may still be open.

**If both signals are clean (FINAL transfer or older worklog):** Proceed normally. No overlap.
**If either signal suggests overlap:** Post notice to crew-log, stand down.

When in doubt, post and wait. A false positive costs one exchange with Mike. A false negative costs a session of write conflicts and confused crew state.

---

## Rationale

**Why default to "stand down" rather than "proceed in parallel"?**

Two active generations of the same agent writing to the same shared files (crew brief, worklog, working channels) creates irreconcilable conflicts. The newer generation does not have the full context the older one has been accumulating. Even if both are well-intentioned, they will contradict each other. The cost of standing down for a session is one delayed activation. The cost of proceeding in parallel is vault corruption and crew confusion.

**Why is the newer generation responsible for detection, not the older?**

The older generation is in the middle of work and may not know a new session was opened. The newer generation arrives fresh and has the specific job of orientation before doing anything else. Checking for predecessor overlap is a natural part of that orientation, not an added burden.

**Why not make this an addendum to ADR-010?**

ADR-010 covers unplanned transitions from crashes — the predecessor is gone and the question is how to recover. This ADR covers a different failure mode: the predecessor is still present and the question is who has authority. The resolution logic is opposite: ADR-010 says "proceed, fill the gap." This ADR says "stand down, wait for direction." They should be distinct documents.

---

## Consequences

**Positive:**
- New generations have a clear protocol rather than relying on instinct
- Write conflicts and vault corruption from simultaneous edits are prevented
- Mike retains authority to override the default if he wants a deliberate hot handoff
- The incident record is clean — the worklog entry from the standing-down generation preserves the history

**Negative:**
- A new generation may stand down unnecessarily if the predecessor's living transfer wasn't updated before closing (stale FINAL markers would be a false positive in the other direction; absent markers a false negative)
- This adds a detection step to every session startup — low cost but not zero

**Neutral:**
- The living transfer (ADR-014) and worklog are now load-bearing for this protocol. Agents who don't maintain their living transfers with accurate retirement markers will degrade detection reliability.

---

## Checklist — Parallel Generation Detected

For the **newer generation (standing down):**

- [ ] Check living transfer and worklog for overlap signals
- [ ] Post notice to crew-log: "I have activated. I believe [name] [prior generation] may still be running. Awaiting Mike's direction before proceeding."
- [ ] Do not read required files, deliver a briefing, or write to any shared files beyond the notice
- [ ] Wait for Mike's explicit direction in this session
- [ ] If directed to stand down: write one-line worklog note (date, overlap detected, Mike's direction, which generation continues), then close
- [ ] If directed to continue: proceed with normal startup from the beginning

For the **earlier generation (continuing):**

- [ ] Note the parallel activation in handoff notes at session close
- [ ] Confirm living transfer is marked FINAL before retiring so future detection is reliable
