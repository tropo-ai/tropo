# ADR-028: Boot-Time Generation Logging

## Context

Generational history in Argo-OS has depended on retiring agents to update historical records. Retirement is unreliable:

- O3 skipped retirement entirely
- Three agents crashed without handoffs on March 23, 2026
- The crew manifest drifted 4-7 generations behind on every agent before Orpheus O9 caught it on March 31
- V16 missed retirement step 4; V17 corrected at boot

The root cause: retirement is optional in practice. Agents exhaust context, crash, or are dismissed mid-session. The information is important; the method of maintaining it is broken.

ADR-010 (Unplanned Generation Transition) established a crash protocol, but it does not solve the data-entry gap — it only names the situation.

---

## Decision

Every agent self-registers at boot by writing to a per-agent generation log file at `agents/[name]/[name]-generation-log.md`.

**Two writes at boot:**
1. **Open their own row** — generation ID, start date, session number, one-line mission
2. **Close their predecessor's row** — set the Retired date for the prior generation (which the predecessor may not have set at retirement). If the predecessor crashed without a retirement timestamp, the incoming generation's start date is used as the predecessor's retirement date. This is the explicit rule — do not search for a missing timestamp.

**One write at retirement:**
3. **Close their own row** — set the Retired date, finalize session count, update Notable

The retirement write is best-effort. The boot writes are the mechanism that makes this protocol self-maintaining without depending on graceful retirement.

---

## Log Format

One structured table per agent, in their own file at `agents/[name]/[name]-generation-log.md`.

```
| Generation | Started | Retired | Sessions | Notable |
|---|---|---|---|---|
| [gen] | YYYY-MM-DD | YYYY-MM-DD or — | [N] | One-line summary of defining work |
```

**Column definitions:**
- **Generation** — Generation identifier (V17, A19, G14, O9, etc.)
- **Started** — Date the generation was first activated
- **Retired** — Date the generation retired. `—` if still active.
- **Sessions** — Count of sessions this generation ran. Updated at retirement or best-effort at each boot.
- **Notable** — One line: the defining contribution or mission of this generation

**File location:** `agents/[name]/[name]-generation-log.md` — one file per agent, in their own folder.

---

## Rationale

**Why boot, not retirement:**
Boot is guaranteed. You cannot do work without booting. Retirement is optional — agents crash, exhaust context, or are closed without a retirement step. A protocol that requires retirement writes will always have gaps. A protocol that requires boot writes maintains itself.

**Why one file per agent, not a combined crew file:**
- Each agent owns their own folder — the generation log belongs with the charter and status card
- A combined crew file creates a coordination surface at boot: every agent writes to the same file simultaneously, which introduces ordering issues and cross-domain writes
- Agent-scoped files match the existing pattern (status cards, worklogs, charters, living transfers)
- Scale: each file stays small and fast to read; a combined crew file grows without bound

**Why boot sequence integration AND a protocol doc:**
- The activation file is where agents look when they boot — the boot step must live there so the protocol is impossible to miss
- A standalone protocol doc (`guides/boot-time-generation-logging.md`) provides the full spec in a canonical place; the activation file entry can be brief
- This is the same pattern used for other boot-critical protocols (ADR-026 briefing package, ADR-016 parallel generation)

**What this replaces:**
- Generational history entries in the crew manifest (those get stripped — the manifest returns to recording who sailed and their soul/contribution as a role, not a generational ledger; see Orpheus O9 RFC, March 31, 2026)
- Scattered generational tracking in historical documents

**What this does NOT replace:**
- Status cards — still the source of truth for current operational state
- The crew manifest — still the record of who sailed, their name, soul, and role-level contribution
- Living transfers — still the mechanism for context handoff

---

## Consequences

**Easier:**
- Generational history maintains itself without a coordinator or reliable retirement
- Incoming agents can determine when the predecessor retired from the log (or default to their own start date if the row is incomplete)
- Each agent's generational record is co-located with their other files — easy to find, easy to audit
- The crew manifest is freed from being a generational ledger and can return to its lore purpose

**Harder / New obligations:**
- Every activation file must include the generation log step in its boot sequence (two reads + one-two writes added to boot cost; estimated 50-100 tokens)
- Every agent activation file and the master template must be updated (R3-008 sweep handles initial deployment; see ADR-028 implementation note below)
- Agents who crash without retirement will have an open row (`—` in Retired column); the next generation closes it at boot

**Implementation note:**
Initial deployment is a sweep by Vela V17, executed April 1, 2026. Each agent's generation log is initialized with the best available historical data from transfers, ops.md, and status cards. The boot step is added to all active activation files and the master template.
