*A canonical explainer for the v1.8 Capability Subsystem Membership substrate — what the typed link between governed primitives and their owning subsystem hub means, how to author it, how releases consume it, and how the rollup query produces self-documenting `subsystems_touched` for free. Authored as v1.8 Stream A5 by Argus A46 (2026-05-05) consolidating the three substrate primitives v1.8 introduced (release-plan.capsule v1.2 + release.capsule v3.4 + subsystem-hub.capsule v1.4) plus the validator at v1.8 Stream A4. ## The TL;DR

v1.7 typed the subsystem taxonomy at 6 hubs anchored to [tropo-subsystems root (aae9a37b)](../../vault/files/aae9a37b.md). v1.8 types the capability taxonomy: every governed primitive declares its owning hub via `member_of:`. **v1.8 also promotes Documentation to a 7th first-class subsystem hub** ([tropo-documentation (f87e33f0)](../../vault/files/f87e33f0.md)) per Mike-A46 mid-cycle resolution of brief D4 (was PUNTED to v1.9; elevated to in-cycle per Mike's importance-calibration on 2026-05-05: *"documentation is a release deliverable, first-class subsystem"*). Releases declare `capabilities_touched: [UIDs]`. The substrate computes `subsystems_touched: [UIDs]` via 1-hop graph traversal at ship time. Hub `release_history:` rows + `subsystem-registry.jsonl` rows are derived from the same query result.

**The rollup is a query, not an authored summary.** Drift between capability membership and subsystem rollup becomes structurally impossible because the rollup IS the query result.

## The model

### What's a "capability"?

A **capability** is a governed primitive — a typed thing the OS knows how to use:

- A capsule definition (`.tropo/capsules/*.capsule.md`)
- An action (`.tropo/actions/*.action.md`)
- A skill (`.tropo/skills/*.skill.md`)
- A playbook (`.tropo/playbooks/*.playbook.md` or `playbooks/*.playbook.md`)
- A sa.* agent activation file (`agents/sa/sa.<name>/sa.<name>.md`)
- A KB article (typed `kb-article` at `vault/files/<uid>.md`; migrated from `.tropo/kb/` at v1.19.0)
- A library document (`library/*.md`)
- A definitions file (`.tropo/definitions/*.jsonl`)

Capabilities are the operational atoms of Tropo. A release SHIPS capabilities (new ones, amended ones, retired ones) — never raw "subsystems" in the abstract.

### The typed link

Every capability's frontmatter `member_of:` array MUST include exactly one of the 7 subsystem hub UIDs:

| Hub UID | Subsystem | Scope |
|---|---|---|
| [`8dd772a0`](../../vault/files/8dd772a0.md) | tropo-governance | Capsule definitions, validation infrastructure, ADRs, governance skills |
| [`dbc1cbbf`](../../vault/files/dbc1cbbf.md) | tropo-rendering | Build scripts, ledger rebuilders, project-tree renderers |
| [`2d083137`](../../vault/files/2d083137.md) | tropo-work | Work-management primitives (task/note/design-brief/project capsules; pipelines; tropo-work L0 root) |
| [`99ed55fd`](../../vault/files/99ed55fd.md) | tropo-agents | sa.* agents, agent.capsule, session-agent.capsule, executive activation files, agent skills |
| [`76bab75f`](../../vault/files/76bab75f.md) | tropo-playbooks | Playbook files, playbook.capsule, action.capsule, skill.capsule, playbook orchestration skills |
| [`1aba710c`](../../vault/files/1aba710c.md) | tropo-library | (NARROWED v1.8) Starter packs (`library/packs/*`), crew-narrative files (captains-log, crew-status-reports), library content-storage substrate |
| [`f87e33f0`](../../vault/files/f87e33f0.md) | **tropo-documentation** *(NEW v1.8 — 7th subsystem)* | KB articles (typed `kb-article` at `vault/files/`, navigable via this hub's member list; migrated from `.tropo/kb/` at v1.19.0), library/docs/ how-tos, the Tropo Handbook, the Studio Manifesto, cross-cutting OS-level explainers |

A capability MAY have additional `member_of:` entries (e.g., a sa.* agent that's `member_of: [99ed55fd, <project-context-uid>]`). The subsystem hub UID MUST be present; multiple subsystem hub UIDs are NOT permitted (a capability belongs to ONE subsystem).

**Per-doc judgment for documentation-class capabilities:** the question is *"Does this document the OS-as-a-whole, or does it document a specific subsystem?"* OS-as-a-whole → tropo-documentation. Subject-specific → that subject's subsystem (e.g., `how-pipelines-work` → tropo-work, `how-to-write-a-playbook` → tropo-playbooks). Both are valid; the test is who the reader is.

### Why `member_of:` not a new `subsystem:` field?

L1 thesis applied recursively. The project-graph primitive (per Mike's directive 2026-04-14: *"Projects are a graph; the tree is a rendering of relationships"*) extends naturally to capabilities. A capability is a graph-child of its subsystem hub. The 6 hubs are typed projects already (Stream A1 of v1.7 locked `subsystem-hub.capsule v1.3`); v1.8 just makes capability membership in those hubs typed. No new schema invented; existing primitive applied at the next-level-down granularity.

This honors Mike's "let's not overthink it" directive (2026-05-05) on the v1.8 brief: Decision 2 explicitly chose graph reuse over schema invention.

## How releases consume the typed link

### `release-plan.capsule v1.2` declares `capabilities_touched:`

When you author a release plan under v1.2 schema (declared via `capsule_version: "1.2"` opt-in), you populate:

```yaml
capabilities_touched:
  - "<capability-uid-1>"
  - "<capability-uid-2>"
  - "..."
```

Empty `[]` is legal at `stage: design`; non-empty REQUIRED at `stage: specify` onward (Check 19).

**When to populate:** capability-tagging-at-stream-start sub-pattern. As you open each stream of work, identify the governed primitives the stream will touch and append their UIDs. The field is queryable mid-cycle for stream-status views; populate it incrementally, not retrospectively at ship.

**Pre-v1.2 release-plans (v1.4.4 / v1.5 / v1.6 / v1.7) use the older informal `sub_systems:` strings field** — those plans are grandfathered through v1.2's `capsule_version` opt-in marker. v1.2+ release-plans use the typed UIDs.

### `release.capsule v3.4` mirrors + derives

At B6 atomic-triangle commit, the release entry mirrors the release plan's `capabilities_touched:` (Check 23) and computes the derived `subsystems_touched:`:

```
subsystems_touched = unique({
  hub_uid
  for cap_uid in capabilities_touched
  for hub_uid in graph.member_of[cap_uid]
  if hub_uid in {6 valid hub UIDs}
})
```

**The release author NEVER types `subsystems_touched:` directly.** Rule 12 (release.capsule v3.4) forbids it. The query produces it.

### `subsystem-hub.capsule v1.4` populates `release_history:` from the same query

For releases dated ≥ v1.8.0 ship, each touched-subsystem hub gets a `release_history:` row generated from the release entry's `subsystems_touched:`. Hub authors no longer write `release_history:` rows independently — the substrate computes them.

Each derived row carries `derived_from: capabilities_touched` to distinguish it from manually-authored v1.7-and-earlier rows (which carry `derived_from: manual_authoring` or omit the field — both treated as manual authoring).

### `subsystem-registry.jsonl` rows derived from the same query

The vault-root append-only event log gets one row per touched subsystem at B6, generated from the same `subsystems_touched:` derivation. Hub `release_history:` and registry rows are TWO FACES of the same query result.

**Drift collapse:** under v1.7 Rule 11, registry rows + hub release_history could disagree (manual authoring on both sides). Under v1.8 Rule 12, both are computed from the same source-of-truth (`capabilities_touched:` on release entry); divergence is structurally impossible.

## The validator

[`scripts/validate-capability-membership.py`](../scripts/validate-capability-membership.py) (v1.8 Stream A4) checks:

1. **C1 — every governed primitive's `member_of:` includes exactly one of the 6 subsystem hub UIDs.** WARNING-level at v1.8-v1.9; ERROR at v1.10. Documentation-class capabilities with placeholder `member_of: [1aba710c]` emit INFO not WARNING (v1.9 Documentation thesis decides their final home — see PUNT below).
2. **C2 — no capability has multiple subsystem hub UIDs in `member_of:`.** WARNING at v1.8-v1.9; ERROR at v1.10. (D2 of v1.8 brief: a capability belongs to ONE subsystem; cross-cutting expressed via `relationships:` not `member_of:`.)
3. **C19 — release-plan v1.2 instances have non-empty `capabilities_touched:` at `stage: specify` onward.** WARNING at v1.8-v1.9; ERROR at v1.10.
4. **C20 — every UID in `capabilities_touched:` resolves to a capability with valid hub `member_of:`.** Honor-system at v1.8-v1.9; mechanical at v1.10.
5. **C23 + C24 (release entries)** — `capabilities_touched:` mirrors release-plan exactly; `subsystems_touched:` equals derived value. Honor-system at v1.8-v1.9; mechanical at v1.10.

Run: `python3 .tropo/scripts/validate-capability-membership.py [--strict] [--json]`

`--strict` enables v1.10+ exit-code semantics (1 on any ERROR). Default is v1.8-v1.9 SOFT mode (always exit 0; report findings).

## The Documentation 7th-subsystem question (RESOLVED IN v1.8 — Mike-A46 elevation)

v1.8 brief D4 originally PUNTED the Documentation question to v1.9 or v1.10. **Mike-A46 elevated it mid-cycle to in-cycle resolution on 2026-05-05** ("I don't care about the locked 6 subsystems. it is more honest to my intent of having documentation become a first class subsystem"). Three candidate resolutions had been considered:

1. **Library covers it** (the placeholder approach — every documentation-class capability tagged `member_of: [1aba710c]`)
2. **Documentation becomes the 7th subsystem** (new hub authored; library docs + KB articles + handbook + manifesto retag to it; Library narrows to "content storage substrate") — **THIS WAS PICKED**
3. **Per-doc judgment** (some docs land in the subsystem they describe — `how-pipelines-work.md` → tropo-work — others stay at Documentation) — **THIS WAS ALSO PICKED (hybrid 2+3)**

**Resolution shipped in v1.8:** new [tropo-documentation hub (`f87e33f0`)](../../vault/files/f87e33f0.md) authored as the 7th subsystem (Option 2). Subject-specific docs re-homed to their describing subsystem via per-doc judgment (Option 3) — `how-pipelines-work` → tropo-work; `how-to-write-a-playbook` → tropo-playbooks; `how-the-registry-works` → tropo-governance; etc. Library hub narrowed to content-storage substrate + packs (`library/packs/*`) + crew-narratives (captains-log, crew-status-reports). Stream E1 audit note [`8945d2e2`](../../vault/files/8945d2e2.md) was the input data driving this resolution.

**Why elevated mid-cycle:** the v1.7 → v1.10 chain is fundamentally about making documentation structurally first-class. Per Mike's 2026-05-05 doctrine: *"documentation is a release deliverable, not an artifact."* Punting Documentation as a 7th subsystem would have left v1.8 shipping with placeholder pollution Library hub became 36.7% of all governed primitives) — substrate-incoherent. Resolving in-cycle honors patient honing + the chain's load-bearing thesis.

## How to author capability membership for a new primitive

When you create a NEW governed primitive (capsule, action, skill, playbook, sa.* agent, KB article):

1. **Pick the subsystem.** Use the table above — what's the PRIMARY function?
   - A new capsule that defines work-management semantics → tropo-work
   - A new sa.* agent → tropo-agents
   - A new playbook → tropo-playbooks
   - A new KB article → tropo-library (placeholder until Documentation question resolves)
   - Something genuinely cross-cutting → use PRIMARY function; cross-cutting via `relationships:` not `member_of:`
2. **Add to `member_of:` frontmatter.** Include the hub UID alongside any other parent UIDs (project containers, etc.).
3. **Run the validator.** `python3 .tropo/scripts/validate-capability-membership.py` — should report no WARNINGs against the new primitive.
4. **Tag in release-plan if shipping.** When the release plan is opening, append the new capability's UID to `capabilities_touched:`.

That's it. The substrate handles the rest at ship.

## How to backfill capability membership for an existing primitive

v1.8 Stream B does this mechanically for ~100 existing primitives via [`scripts/backfill-capability-subsystem.py`](../../.tropo-studio/scripts/v1.8-stream-b/backfill-capability-subsystem.py). The script:

1. Reads each governed primitive's frontmatter
2. Infers subsystem from folder location + capsule type + content heuristics (Pass 1 — high confidence)
3. Flags ambiguous cases for manual review (Pass 2)
4. Outputs a manifest of proposed changes for reviewer approval
5. Applies approved changes

If you're authoring a primitive after v1.8 ship, follow the "new primitive" steps above. If you discover a primitive that wasn't backfilled (perhaps newly imported from elsewhere), run the backfill script in single-file mode or add `member_of:` manually.

## Worked example — v1.8 ship's own dogfood

v1.8 ships its own substrate AND tags its own ship in `capabilities_touched:`. The release plan [1fe60f02](../../vault/files/1fe60f02.md) declares (at draft, populated incrementally as streams execute):

```yaml
capabilities_touched:
  - "a3f1e7b2"   # release-plan.capsule v1.2 (Stream A1)
  - "b19e8d43"   # release.capsule v3.4 (Stream A2)
  - "8a4e21c5"   # subsystem-hub.capsule v1.4 (Stream A3)
  - "<validator-uid>"   # scripts/validate-capability-membership.py (Stream A4)
  - "0c2b6aa4"   # this KB article (Stream A5)
  - "..."   # populated as Stream B + C + D + E + F execute
```

At B6 ship, the substrate computes:

```
subsystems_touched = unique({
  member_of[a3f1e7b2] ∩ {6 hubs},   # release-plan.capsule → tropo-work
  member_of[b19e8d43] ∩ {6 hubs},   # release.capsule → tropo-work
  member_of[8a4e21c5] ∩ {6 hubs},   # subsystem-hub.capsule → tropo-governance
  member_of[<validator>] ∩ {6 hubs},   # validate-capability-membership.py → tropo-rendering
  member_of[0c2b6aa4] ∩ {6 hubs},   # this KB article → tropo-library (placeholder)
  ...
})
= {2d083137, 8dd772a0, dbc1cbbf, 1aba710c, ...}   # tropo-work + tropo-governance + tropo-rendering + tropo-library + ...
```

Hub `release_history:` rows + registry rows for each touched hub generated from the same query result. v1.8 eats its own dog food on the very first ship under the new model. If F6 produces inconsistent or empty rows, the cycle hasn't shipped — this IS the substrate validation.

## Pitfalls

- **Adding multiple subsystem hub UIDs to `member_of:`** — Rule violation. A capability belongs to ONE subsystem. Cross-cutting via `relationships:` not `member_of:`.
- **Manually authoring `subsystems_touched:` on a v1.8+ release entry** — release.capsule v3.4 Rule 12 violation. The field is DERIVED via 1-hop graph traversal at B6. Author `capabilities_touched:` only.
- **Authoring a v1.8+ release entry whose `capabilities_touched:` differs from the linked release-plan's `capabilities_touched:`** — Check 23 mismatch. The release entry mirrors the release plan at B6.
- **Forgetting to tag a new capability when adding it to a release plan mid-cycle** — Check 19 misses if the field is empty at specify; downstream rollup misses the capability if tagged-too-late. Tag at stream open, not at ship.
- **Treating documentation-class placeholder as permanent** — `member_of: [1aba710c]` for KB articles is a PUNT, not a decision. v1.9 Documentation thesis resolves; expect retagging in v1.9 cycle if the resolution moves docs to a 7th subsystem.
