## Purpose

**Block 4 close-criterion empirical-validation discipline** (Block 4 = Hardening Through Test, per Captain's Read v2.0 [a5f4b26b](a5f4b26b.md); the cycle-phase block where Tropo-OS validates stranger-encounter readiness through three persona cold-boot-walks before each release ships). Runs the release-cold-boot-walk.playbook (6f3d2a18) against the .zip artifact produced by step-5 (8654900a) AFTER step-6 external-test confirms vela-test-plan smoke clean, BEFORE step-7 git-commit fires.

Three personas (engineer / operator / enterprise) cold-boot the release in isolated test-Studios; their walks produce per-persona scores (1-10) + binary-gate verdicts (9 gates). Aggregate verdict gates git-commit: only releases with sustained 8.0+ aggregate AND ≥8 per-persona AND all-binary-gates pass land in production.

The step formalizes what was memory-resident agent behavior since v1.41 (Vela authored the playbook at v1.41 + dogfooded across v1.41/v1.42); v1.47 makes the step substrate-resident so pipeline-runs structurally cannot skip the gate.

## Flow Rules

**Completes when:** the release-cold-boot-walk.playbook (6f3d2a18) has executed all three persona walks AND the playbook orchestrator has fired `pipeline-runtime.py verify-step c6b61fb9 --verification-data-stdin` with the structured JSON payload (see §Verification Data Contract below). The orchestrator — NOT the engine — evaluates the five threshold criteria:

- `aggregate_avg_score >= 8.0` (mean of three persona scores)
- `per_persona_engineer >= 8`
- `per_persona_operator >= 8`
- `per_persona_enterprise >= 8`
- `all_binary_gates 9/9 pass`

ALL five criteria must pass for the orchestrator's `verdict` field to be `pass`. A failing criterion produces `fail` verdict; the pipeline-run pauses (`trust_level: auto-with-verification` permits the engine to surface the verdict but halts auto-advance to step-7; Vela ratifies pass OR Mike intervenes on fail).

**Advances to:** git-commit (3e0bb81e) on pass natural verdict.

**verification_class: true with `--verification-data-stdin`** — walks produce the natural verdict via the playbook orchestrator's structured JSON. The engine does NOT evaluate exit_criteria via DSL for this step; instead it accepts the orchestrator's verdict + structured data as substrate-resident verification_receipt. See §Verification Data Contract.

## Verification Data Contract

The playbook orchestrator (6f3d2a18) fires verify-step with structured JSON on stdin per `pipeline-runtime.py:905-910` action_verify_step `verification_data_stdin` branch:

```bash
echo '<json-payload>' | python3 .tropo/scripts/pipeline-runtime.py \
  --activation-uid <activation-uid> --actor vela-v<N> \
  verify-step c6b61fb9 --verification-data-stdin
```

**JSON payload schema:**

```json
{
  "verdict": "pass" | "fail",
  "verifier_role_resolved": "vela",
  "aggregate_avg_score": 8.33,
  "per_persona": {
    "engineer": {
      "score": 8.5,
      "binary_gates": {"<gate-name>": "pass" | "fail" | "n/a", ...},
      "report_uid": "<persona-walk-report-uid>",
      "duration_minutes": 12
    },
    "operator": { /* same shape */ },
    "enterprise": { /* same shape */ }
  },
  "binary_gates_summary": "9/9 pass" | "<N>/9 pass",
  "rationale": "<one-line summary of verdict>",
  "rubric": {"aggregate_threshold": 8.0, "per_persona_threshold": 8, "binary_gates_required": 9}
}
```

**Unhappy-path handling** (per release-cold-boot-walk.playbook 6f3d2a18 §Failure Handling):

- **Persona walk crashes mid-walk** (sub-agent timeout, harness error): orchestrator marks `per_persona.<persona>.score: null` + `binary_gates: {}` + rationale `"<persona> walk failed: <reason>"`. Aggregate excludes failed persona from mean (denominator = 2 not 3); if remaining 2 personas pass thresholds, verdict can still be `pass` IF orchestrator's rubric permits — otherwise `fail`.
- **Malformed persona report JSON**: same as crash; orchestrator's responsibility to detect + emit honest verdict.
- **Binary gate result `n/a`** (gate not applicable to this persona-walk): counted as pass for that persona's `9/9` tally; aggregate-level `binary_gates_summary` reflects raw fail count across all personas.
- **Orchestrator failure** (orchestrator itself crashes before firing verify-step): pipeline-run pauses at step 7.5 with no verification_receipt event; ops.md BULLETIN named "cold-boot-walk orchestrator failed pre-verdict; manual triage required."

## Cold-Boot Walk-Through

A pipeline-run enters step 7.5 with:
- `.zip` artifact produced by step-5 (8654900a)
- vela-test-plan smoke verdict from step-6 (bc6b17ec) — must be pass (gate)

Vela (step_owner_role) executes the release-cold-boot-walk.playbook (6f3d2a18):
1. Copies build artifact → `/Users/mike/dev/release-testing/<persona>/` per Vela's release-testing copy discipline.
2. Dispatches 3 sub-agents (one per persona substrate file at `.tropo/personas/cold-boot/<persona>.md`) into the test-Studios.
3. Each sub-agent boots fresh against the unzipped release + executes the persona-specific walk (engineer first-build evaluation; operator daily-ops scenarios; enterprise governance scenarios).
4. Sub-agents return per-persona scores + binary-gate verdicts to the playbook orchestrator.
5. Orchestrator computes aggregate + writes verification_receipt to this pipeline-run event log with natural verdict (pass | fail) per the exit_criteria evaluation.

If pass: pipeline-run advances to step-7 (3e0bb81e git-commit). Vela ratifies the verdict via human-signoff if `trust_level` policy requires.

If fail: pipeline-run pauses at step 7.5; ops.md gets a BULLETIN naming the failing criteria; Mike + Vela decide remediation (fix-and-rebuild via re-entering build stage OR sign off and proceed with explicit principal authorization).
