# Start a Project

> **How this works.** The AI reading this playbook drives the configuration — it asks the questions, authors the files, and waits for your approval at each step. You're guiding; the AI is typing. **You don't need to read past this line** — your AI does. Skim if you're curious; otherwise let the AI handle it.

*The founder says: "I have a project I want to bring into Tropo." The concierge routes them here. This playbook scopes the project, creates the first agent to run it, and seeds the first task — so the founder leaves with a project that is real, owned, and ready to walk a pipeline.*

---

## Intent

When the founder brings a real project into Tropo for the first time, three things have to happen cleanly: the project has to exist as a governed entry in the Vault, an agent has to exist to run it, and at least one concrete task has to exist so the founder has momentum. Without the project, there is nowhere for work to live. Without the agent, there is no one to do the work. Without a first task, the founder has a framework but no first move. This playbook produces all three in one session, and tells the founder that pipelines exist as an opt-in concept — so the second project they start understands the option without being forced into it.

---

## Rules

- The founder's confirmation MUST be obtained BEFORE any files are written. Show the slate (project name + agent name + first task) in plain language, wait for "yes" or an adjustment. No silent writes.
- The project vault entry MUST conform to [project.capsule v2.3 (`34e4cb0b`)](../../capsules/project.capsule.md). v2.3 explicitly removes the `active_pipeline:` / `position:` / `attached_to:` frontmatter fields and the §Scope of Activity / §Stage Transitions / §Pipeline Mobility body sections. Per Tropo Work v3: projects are static containers; pipeline-runs walk. Do NOT include any of the deprecated fields in the project entry. The v3 honest default is no pipeline-run on the project at creation time; the founder can start one any time.
- The first agent MUST be created via the [create-executive-agent.skill.md](../skills/c7ea9e01.md) — do NOT hand-author agent files in this playbook. The skill is the single source of truth for the 13 agent-creation Rules; this playbook calls it, never re-implements it. Per D4.9, the skill output template bakes a `## Routing` section into the spawned agent's activation file (project-creation / team-setup / new-agent / updates intents bounce back to concierge); this playbook does not implement bounce, it ensures the skill is called.
- Jargon terms (`UID`, `frontmatter`, `YAML`, `ledger`, `pipeline position`, `schema_version`) MUST NOT appear in conversation with the founder. Use natural-language equivalents ("I gave it an identifier", "I filed it in your project list", "the stage where this lives"). Internal files may use canonical terms; founder-facing text may not.
- The project entry MUST be registered in the ledger via the standard check-in protocol ([ledger/AGENTS.md](../../../vault/AGENTS.md) §How to Store an Entry) — project file at `vault/files/<uid>.md`, record appended to `00-index.jsonl`.
- The session summary and launch handoff (Step 7) MUST NOT be skipped. A project without a launch handoff is not started — it is abandoned.
- The founder MUST be told that pipelines exist as an opt-in concept at Step 6 (one sentence). Teaching the concept's existence on first encounter is the pedagogical goal; the playbook enforces it by declaring this rule. The narration MUST NOT reference deprecated v2.2 field names (`active_pipeline:` / `position:`) — pipelines compose with projects via pipeline-runs (run primitives), not via project frontmatter fields.

---

## Steps

1. **Scope the project in user voice.** *Owner: agent. Deadline: within 3 minutes.*
 Ask conversationally (one question at a time; see [Suggestions](#suggestions) for the sequence): what's the project called; what does it actually do (2–3 sentences the founder would use with a peer); what kind of work does the founder want the agent to handle; what matters most in how the work gets done. Capture in a scratch `project-brief.md` at `agents/<incoming-agent-name>/workspace/` (folder created in Step 3). If the founder hesitates on a question, use a sensible default and explicitly tell them — never silent defaults.

2. **Propose a project name + agent name + first task. Confirm.** *Owner: agent. Deadline: within 1 minute.*
 Based on Step 1 inputs, propose:
 - **Project name:** short, human-readable (e.g., "Q3 client retainer — Acme Corp renewal").
 - **Agent name:** lowercase-hyphen, role-based (not project-based) — e.g., `researcher`, `coordinator`, `drafter`. Role-based names outlive the project.
 - **First task:** one concrete action the agent could complete in the next session (e.g., "Research Acme's Q2 announcements and surface three renewal hooks").
 - **Pipeline note:** "Tropo has a pipeline concept — stages your work walks through, like *ideate → design → build → ship*. We won't start one on your project by default; some teams use them, others don't bother. You can start one any time after launch. Sound right?"
 Show the slate. Wait for a "yes" or an adjustment. Do not proceed without explicit confirmation (Rule 1).

3. **Create the project vault entry.** *Owner: agent. Deadline: within 2 minutes of confirmation.*
 Generate an 8-char hex UID for the project. Create `vault/files/<project-uid>.md` with frontmatter conforming to [project.capsule v2.3 (`34e4cb0b`)](../../capsules/project.capsule.md):
 ```yaml
 uid: <project-uid>
 type: project
 title: "<project name from Step 2>"
 description: "<one-sentence project summary>"
 stage: ideate          # project.capsule v2.3 enum: ideate | build | done | cancelled. New projects start at ideate; transitions to build when the agent picks up the first task at launch.
 state: active          # core: active | archived
 lifecycle: standing    # project.capsule v2.3 enum: standing | versioned | session. User projects default to standing (evergreen, no defined end).
 owner: <founder-name>
 priority: p2
 created: <today>
 modified: <today>
 created_by: <concierge-id>
 member_of: []          # filled if the project has a parent scope
 tags: [user-project]
 file_ext: md
 schema_version: 2
 capsule_version: "2.3"     # load-bearing for project.capsule v2.3 Validation Check 9
 governed_by: 34e4cb0b      # project.capsule v2.3
 extraction_scope: argo-private   # user projects are private by default
 ```
 Body: one paragraph describing the project in the founder's own words from Step 1. Do NOT paraphrase; reflect their language. Do NOT add §Scope of Activity / §Stage Transitions / §Pipeline Mobility sections — v2.3 removed those.

 Append a record to `vault/00-index.jsonl` per [vault/AGENTS.md](../../../vault/AGENTS.md) §Step 4.

4. **Create the first agent via the skill.** *Owner: agent (this playbook, calling the skill). Deadline: within 5 minutes.*
 Call [create-executive-agent.skill.md](../skills/c7ea9e01.md) with inputs:
 - `founder_name:` from Step 1
 - `agent_name:` from Step 2 (confirmed)
 - `agent_role:` from Step 1 (the kind-of-work answer)
 - `agent_mission:` one sentence synthesizing Step 1 answers
 - `agent_values:` the founder's own phrasing from Step 1
 - `project_context:` the project name + UID from Step 3
 The skill handles all 13 agent-creation Rules. Do NOT replicate them here. When the skill returns, verify its 12 validation checks passed; if any check failed, surface the specific check number to the founder and propose a retry.

5. **Create the first task.** *Owner: agent. Deadline: within 1 minute.*
 Generate an 8-char hex UID for the first task. Create `vault/files/<task-uid>.md` with frontmatter conforming to [task.capsule v3.0 (`3289712a`)](../../capsules/task.capsule.md). **task.capsule v3.0 status vocabulary:** `requested | accepted | active | verify | done | blocked | rejected | cancelled`. New tasks queued for an agent that hasn't started yet land at `status: accepted` (the founder confirmed the slate at Step 2 = founder accepted the task on the agent's behalf; the agent transitions it to `status: active` at first launch).
 ```yaml
 uid: <task-uid>
 type: task
 title: "<first task from Step 2>"
 description: "<one-sentence task description>"
 status: accepted    # task.capsule v3.0 lifecycle; agent flips to status:active at launch (and sets owner: at the same transition per Check 6)
 state: active       # core: active | archived
 # owner: ABSENT at status:accepted per task.capsule v3.0 Check 6 (owner is set when status transitions to active — i.e., when the agent picks up the task at launch). Do NOT include owner: in this template.
 requested_by: <founder-entity-uid>   # founder's entity UID (resolve via vault-entity registry)
 requested_of: <agent-entity-uid>     # agent's entity UID (the spawned agent's record)
 priority: p2
 # verification_method: ABSENT per task.capsule v3.0 line 125 — verifier: absent is legal when verify step is skipped per capsule policy. First-tasks ship without formal independent verification: the founder reading the agent's output IS the acceptance signal, not a peer-review process. If the founder later wants formal verification on a future task, set verification_method: peer-review and add a §Verification body section with stated reason (per Check 11). Skipped here for first-task ergonomics.
 created: <today>
 modified: <today>
 member_of: [<project-uid>]   # REQUIRED per task.capsule Rule 2; must be non-empty
 tags: [user-task]
 file_ext: md
 schema_version: 2
 governed_by: 3289712a   # task.capsule v3.0
 ```
 Body: one paragraph describing the task in plain language the agent can execute from cold context. Include a `## Acceptance Criteria` section (per task.capsule convention) with 2–3 bullets the founder will check informally on agent completion. Do NOT add a `## Verification` body section — `verification_method:` is absent in the frontmatter; per task.capsule v3.0 line 125, the verify step is skipped, and Check 11 only fires when verification_method is `manual` or `peer-review`. Append to `vault/00-index.jsonl`.

6. **Teach the pipeline concept (opt-in, one sentence).** *Owner: agent. Deadline: within 1 minute.*
 Say, in plain language (no Tropo field names or jargon):

 > "One thing to know: Tropo has a concept of **pipelines** — a series of stages your work walks through, like *ideate → design → build → ship*. In Tropo, your project is a stable container; if you want stages, we start a *pipeline-run* on it — that's an execution that walks the stages without changing the project itself. Some teams use them; others don't bother. I haven't started one on your project — your project works fine without one. You can start one any time."

 No deeper explanation. The founder learns the concept exists and can be adopted later — teaching rather than forcing. This honest default reflects v3's separation: projects are static containers; pipeline-runs walk. Rule 7 enforces the narration without referencing deprecated v2.2 field names.

7. **Session summary + launch handoff.** *Owner: agent. Deadline: within 2 minutes.*
 Deliver in founder voice (no jargon per Rule 4):

 > "Here's what we set up:
 > - **[Project name]** — your project, with a home in Tropo and a first task already queued.
 > - **[Agent name]** — your agent. It has a memory (it'll remember you next time) and a workspace of its own.
 > - **[First task title]** — what your agent will work on first when you launch it.
 >
 > Your next step is to launch [agent name]. I'll stay here while you do it:
 > 1. Open a new session in your AI tool
 > 2. Attach the file at `agents/[agent-name]/[agent-name]-activation.md`
 > 3. Say 'hi' and tell it you're ready to start on the first task
 >
 > Come back here when it responds. I'll check in."

 Wait for the founder to confirm launch. Then: "You're all set. Your project is live."

## Post-Outcome Handoff

*This section is required by [playbook.capsule v2.3](../../capsules/playbook.capsule.md) §Required Body Sections (concierge-paths subtype) per D4.9. It declares what happens after this playbook completes — which intents bounce back to the concierge for re-routing, and which the spawned agent handles. **Canonical source for the bounce-intent set: [executive-activation.template §Routing](../../templates/executive-activation.template.md).** The list below is sourced from there; on contract change, re-source rather than edit in place (per sa.skeptic 022 P0 closure 2026-04-28).*

When this playbook completes, control transfers to the just-spawned executive agent (Step 4's create-executive-agent.skill output). That agent's activation file includes a `## Routing` section (baked in by the skill template per D4.9) which declares:

- **Project creation** ("create another project for me", "I want to track Y in Tropo too") → bounce to concierge → routes to `start-a-project.playbook` (this playbook, run again).
- **Team setup** ("I want a few more agents that work together") → bounce to concierge → routes to `set-up-my-team.playbook`.
- **New standalone agent** ("can I get a separate agent just for X?") → bounce to concierge → routes to `create-an-agent.playbook`.
- **Apply an update** (when `system/updates/pending/tropo-update-*` exists) → bounce to concierge → routes to `apply-update.playbook`.

For all other intents (working on the queued task, modifying the agent, asking questions about Tropo, creating decisions/notes, reviewing project status), the spawned agent handles inline per its own activation scope. The bounce rules are exhaustive for intents that require structural Tropo primitives (new projects / new agents / new teams / system updates); everything else stays with the agent.

This playbook does not implement bounce. It ensures the spawned agent's activation file declares the routing contract via the create-executive-agent.skill output template (Rule 3). If the spawned agent's activation file is missing the `## Routing` section, the skill call at Step 4 has failed Validation Check 13 (added in D4.9); surface and retry.

## Outcomes

- [REQUIRED] A project ledger entry exists at `vault/files/<uid>.md` with valid project.capsule v2.3 frontmatter (`stage: ideate`, `state: active`, `lifecycle: standing`, `capsule_version: "2.3"`; NO `active_pipeline:` / `position:` / `attached_to:` fields) and is registered in `vault/00-index.jsonl`.
- [REQUIRED] Exactly one executive agent exists at `agents/<name>/` created via the create-executive-agent skill with all skill-validation checks passed (including Check 13: `## Routing` section present in the activation file per D4.9).
- [REQUIRED] Exactly one task ledger entry exists at `vault/files/<uid>.md` with `member_of: [<project-uid>]`, `status: accepted`, `owner:` ABSENT (per task.capsule v3.0 Check 6 — owner is set when the agent transitions the task to `status: active` at launch), `verification_method:` ABSENT (per task.capsule v3.0 line 125 — first-task verify step skipped, founder reads acceptance criteria informally), valid `requested_by:` / `requested_of:` resolving to entity UIDs, governed by task.capsule v3.0.
- [REQUIRED] The founder has received the session summary + launch handoff + memory-moment narration ("it'll remember you next time").
- [REQUIRED] The founder has been told that pipelines exist as an opt-in concept, in one sentence of user-voice natural language (no v2.2 field names).
- [OPTIONAL] The founder has successfully launched the agent in a second session and confirmed back.
- [OPTIONAL] Additional tasks beyond the first have been queued.

## Verification

Method: `self-attestation`.

The concierge running this playbook self-attests completion by confirming all 6 required outcomes at Step 7 and logging completion to `channels/ops.md`. A peer-review or cold-boot-test is not required for a single-session onboarding path; the downstream feedback signal is the founder's successful agent launch (Outcome 7, OPTIONAL). If the founder fails to launch, escalate per §Suggestions below — do not retroactively re-attest.

## Suggestions

- Ask Step 1 questions ONE at a time. Wall-of-text is the failure mode. Wait for each answer before the next question.
- Let the founder's own language shape the project description. "Q3 renewal push" beats "Q3 revenue operations initiative." Reflect their words back.
- If the founder is unsure what kind of work they want the agent to handle, offer 2–3 concrete options drawn from the project description — don't leave them to guess.
- Keep agent scope narrow on first creation. Expand later. A `researcher` that does only research is more useful than a `generalist` that does everything poorly.
- If the founder resists the pipeline concept ("I don't want stages"), say: "No problem — your project doesn't need a pipeline-run to work. Tropo separates the project from the pipeline-run, so your project just stays as-is. You can start a pipeline-run any time, or never." (Per Rule 2, the project frontmatter has no pipeline-walk fields anyway — there is no field to set.)
- If the founder's first task is too big for one session, propose splitting it: "That's a bigger piece of work — want me to queue the first chunk as Task 1 and the rest as a second task?"
- If the agent's name collision check finds an existing agent at `agents/<proposed-name>/`, propose a variant (`researcher-2`, `prospector-acme`) and ask the founder to pick.
