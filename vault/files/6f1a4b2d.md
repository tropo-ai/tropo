# ADR-011: Agent Operations and Archive Directory Structure

## Context

Two structural gaps emerged from the first month of Argo-OS operation:

**The children/operations conflation.** Standing operational processes (vault auditors, channel monitors, vault-janitor) were stored under `agents/vela/children/` alongside one-time task children (V3.1, V3.2). This was architecturally misleading: task children are ephemeral context-offloading tools that exist for a single job. Operational processes are persistent system infrastructure that runs on recurring schedules. The same folder implied they were the same kind of thing. They are not.

**No per-agent archive.** Agents had no designated location for retiring completed workspace files, superseded drafts, or historical artifacts. Without an archive pattern, agents were forced to either leave stale material in their workspace (cluttering the active working space) or delete it (losing the record).

---

## Decision

**1. Create `agents/operations/`** — a crew-level directory for all recurring system processes that serve the Argo-OS as infrastructure. These are not owned by any one agent; they are owned by the operating system. Vela coordinates them but does not parent them.

**2. Create `agents/[name]/operations/`** for every agent — a per-agent directory for reusable activation templates and recurring task patterns the agent wants easy access to. Vela's `operations/` holds her operational toolkit (e.g., prepared activation files for common jobs). Argus's `operations/` might hold architectural tooling. Each agent manages their own.

**3. Create `agents/[name]/archive/`** for every agent — a per-agent directory for retiring completed work, superseded drafts, and historical materials that should be preserved but not surface in active working directories.

**Standard agent folder structure (v3.1+, updated):**
```
agents/[name]/
 [name]-activate.md
 [name]-worklog.md
 [name]-status.md
 workspace/ ← active working documents
 published/ ← completed, shared outputs
 operations/ ← reusable activation templates and recurring task patterns
 children/ ← one-time task children (V[N].1, V[N].2, etc.)
 archive/ ← retired work, superseded drafts, historical materials
 reflections/ ← generational retirement reflections
```

**Naming convention for operations/ activation files:** `[job-name].activation.md`

---

## Rationale

**Why separate operations/ from children/ at the crew level:** The five standing processes (vault-janitor, vault-integrity-auditor, freshness-monitor, governance-validator, channel-health-monitor) audit and maintain the entire Argo-OS. Locating them inside a single agent's folder implied ownership that didn't match reality. `agents/operations/` makes their system-level nature explicit.

**Why per-agent operations/ folders:** As agents mature, they will develop repeatable activation patterns — standard jobs they want to launch without redrafting. Giving each agent an `operations/` directory applies the same pattern consistently and supports the broader principle that agents manage what's in their own folders.

**Why archive/ at the agent level rather than a shared archive:** Agent work is personal. A draft Metis wrote for the playbook spec is Metis's material to retire when it's superseded, not a crew-level concern. Each agent archiving into their own folder keeps the ownership clear and prevents `archive/` from becoming a catch-all dump.

---

## Consequences

**What gets easier:**
- Operational processes have a clear home that reflects what they actually are
- Agents have a designated place to retire completed work without losing it
- The distinction between "this agent's task work" and "system infrastructure" is legible from the folder structure

**What gets harder:**
- Existing references to `agents/vela/children/[operational-process]` need updating (done as part of this migration)
- The onboarding guide and agent activation template need updating to include `operations/` and `archive/` in standard setup

**What changes:**
- Five operational processes migrated: `agents/vela/children/{vault-janitor, vault-integrity-auditor, freshness-monitor, governance-validator, channel-health-monitor}` → `agents/operations/{same}`
- Old locations soft-deleted to `recycle/operational-migration-2026-03-23/`
- `operations/` and `archive/` directories created for all eight agents (vela, argus, metis, talos, silas, jules, nestor, tiphys)
- Child registry `Standing Children` section updated to reference new `agents/operations/` paths
- Activation files for all five operational processes updated with corrected report paths
