## Purpose

Auto-fires Rule 12 hub release_history derivation at the close of build stage. For every subsystem hub touched by the cycle's release (derived from `capabilities_touched:` via 1-hop `member_of:` graph traversal), this step writes:

1. A new `release_history:` row on the hub frontmatter (with `derived_from: capabilities_touched`)
2. A `last_release_reflected:` bump on the hub frontmatter
3. A corresponding row in `subsystem-registry.jsonl`
4. The derived `subsystems_touched:` set on the release entry

**Frontmatter only.** Hub body sections (`## Change Log`, `## Current State`) are NOT written by this step — they are human documentation per v1.9.2 Q7 walk reframe (carry-forward note [`4b7a9d3f`](4b7a9d3f.md)). Body section authoring stays with the agent as a separate manual or sa.hub-groomer-handled concern.

Closes the manual hub-row substitute pattern that ran across v1.8 + v1.9.0 + v1.9.1 (each cycle hand-wrote an `append-v1-X-hub-rows.py` script because this executor didn't exist). Per build-the-executor-when-introducing-doctrine pin (a46-reflection 2026-05-05; A47 + A48 inheritance + A49 commitment): ship the rule + ship the executor in the same cycle.

Does NOT govern: hub body section updates, subsystem-hubs.collection.md updates, RELEASE-NOTES authoring, orientation §What's-new updates, or any human-canonical layer work. Those are manual per cycle Stream C5.

## Flow Rules

**Pre-conditions to fire:**
- Release-plan, release-entry, and build-entry are all at `status: locked`
- Release-plan's `capabilities_touched:` is populated (per release-plan.capsule v1.3 Check 19)
- Release-plan's `hub_summaries:` is populated for every derived subsystem hub (per release-plan.capsule v1.3 Check 21)

**Step logic:**
1. Read release-plan + release-entry + build-entry by UID
2. Derive `subsystems_touched:` via 1-hop `member_of:` graph traversal over `capabilities_touched:` (filtered to subsystem hub UIDs — children of `aae9a37b`)
3. Validate: every derived subsystem hub has a `hub_summaries[hub_uid]` entry; FAIL with clear message if missing
4. Allocate registry UIDs (8-hex; one per touched subsystem)
5. For each touched hub: prepend `release_history:` row (with `derived_from: capabilities_touched` + `summary:` from `hub_summaries[hub_uid]`); bump `last_release_reflected:` to release version
6. Append rows to `subsystem-registry.jsonl` (one per touched subsystem)
7. Write derived `subsystems_touched:` back to release entry
8. Emit machine-readable manifest of writes for run.jsonl audit log

**Post-conditions on success:**
- Each touched hub has new `release_history:` row at top of array
- Each touched hub's `last_release_reflected:` matches release version
- `subsystem-registry.jsonl` has new rows for all touched subsystems
- Release entry's `subsystems_touched:` matches derived set

**Failure modes:**
- Missing `hub_summaries:` entry for derived hub → FAIL with named missing hub_uid
- Hub UID resolves to non-hub project → FAIL with named UID
- Registry write conflict → FAIL with conflict details
- Vault index inconsistency → FAIL with index error

**On any FAIL:** pipeline halts; no atomic-triangle commit; no `.tropo/version.md` bump; no run.jsonl pipeline-complete milestone. Agent reviews + amends release-plan or fixes substrate + re-fires.

**ZERO-touch cycles:** if `capabilities_touched: []` derives empty `subsystems_touched: []`, the step no-ops cleanly (zero rows written; zero registry rows; zero subsystems_touched). Pipeline advances to deploy stage. ZERO-touch cycles are rare (e.g., pure rollback patches that touch no governed primitives) but the step must support them.

**Terminal within build stage.** On completion, build stage advances to deploy (3a7dbdda) via the build stage's `next_steps:`.
