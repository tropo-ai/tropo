# ADR-004: Agent Folder Consolidation

## Context

Argo-OS v2 grew organically. Agent activation files and worklogs were placed at the `agents/` root level (`agents/metis/metis-activate.md`, `agents/metis/metis-worklog.md`) while agent-specific subdirectories (`agents/metis/children/`) were created alongside them. This produced a hybrid layout: some agent files inside a named folder, some beside it.

At 8 agents with 12 root-level files plus 3 named directories, the `agents/` directory is cluttered but manageable. At 50 agents — the scale Argo-OS is designed for as a product — it becomes unmanageable. The v3 design brief introduces new per-agent artifacts (workspace, published, status card) that compound the problem.

The question: consolidate all agent files into one directory per agent, or continue the hybrid pattern?

---

## Decision

Consolidate. Every agent's files live inside `agents/[name]/`. The `agents/` root contains only shared resources (`README.md`, `agent-activate-template.md`, `child-agent-registry.md`) and agent directories.

Files keep their agent-name prefix even inside the named folder: `agents/metis/metis-activate.md`, not `agents/metis/activate.md`. The name is identity — it should be readable out of context, in a directory listing, a cross-reference, or when pulled into another session. Mike's decision.

---

## Rationale

**Why consolidate:** One directory per agent means one `ls` to see an agent's full footprint. It scales cleanly. It aligns with v3.1's workspace/published/status additions — those artifacts need to live inside the agent folder anyway. Leaving activation files and worklogs outside while putting everything else inside creates an arbitrary split.

**Why keep the name prefix:** The alternative — bare `activate.md` — is shorter but loses identity when the file is referenced outside its directory context. When Vela lists files across agents, or a cross-reference is read in the crew log, `metis-activate.md` identifies itself. `activate.md` does not.

**Alternative considered:** Leave v2 layout as-is, only add new artifacts inside agent folders. Rejected because the hybrid gets worse as more artifact types are added, and the cross-reference problem grows with each new agent.

---

## Consequences

**What gets better:** `agents/` root is clean — only shared files and named directories. Each agent's full footprint is one directory. New agent onboarding is simpler: create one folder, populate it.

**What gets harder:** Migration. 12 files move. Every cross-reference in the vault needs updating. Historical crew-log entries will reference old paths (they are not modified — a header note is added instead).

**What changes:**
- All activation file paths change from `agents/[name]-activate.md` to `agents/[name]/[name]-activate.md`
- All worklog paths change from `agents/[name]-worklog.md` to `agents/[name]/[name]-worklog.md`
- All guides, the OA, the crew brief, and agent activation files must be updated
- The onboarding guide must reflect the new structure
- A child agent (A1.1) executes the migration and cross-reference sweep
