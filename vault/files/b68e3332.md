## Purpose

The post-deploy verification step of app-pipeline. Verifies the deployed instance of the Tropo platform app renders the changes correctly. Two modes per the activation-root project's `live_url:` declaration:

- **Live mode** (when `live_url:` is set): waits for the host to report deploy complete, then curls + content-checks the live URL.
- **Local mode** (when `live_url:` is unset — current v1.0.0 default for the platform app, which has no public deploy target yet): verifies the production preview at localhost (deeper than verify-build's check — full route walk, key interactive surfaces, no console errors during click-through).

Either mode is a real gate. The naming is `smoke-test` (not `smoke-test-live`) to reflect the dual-mode design.

This is the terminal step of app-pipeline. On close, the pipeline-run completes and the activation-root project flips from `state: standing` to `state: done`.

## Structure

Leaf step. No children. Terminal (`next_steps: []`); pipeline-run closes when this step closes.

## Behavior

No confirmation gate. Executor verifies the deployed instance per the mode the activation-root project declares.

**Step work — Live mode (`live_url:` is set):**

1. **Wait for deploy completion.** Mechanism is host-dependent (see Known Enforcement Gaps). Options once host is wired:
   - Vercel: poll the Vercel API for the deploy status matching the pushed commit SHA.
   - Netlify / Cloudflare Pages: similar pattern, host API.
   - Fallback: fixed wait (e.g., 60-90s) before attempting the live curl.
2. **For each route in `changes_in_scope` (or representative routes for non-route scope):**
   - `curl -s -o /dev/null -w "%{http_code}\n" <live_url>/<path>` — confirm HTTP 200.
   - `curl -s <live_url>/<path> | grep -o "<expected-content-string>"` — confirm key content strings render.
3. **Report results to Mike.** Write a summary to `run.jsonl` and surface the live URLs in the agent's response for eyeball check.

**Step work — Local mode (`live_url:` is unset):**

1. **Ensure production preview is running.** If verify-build's `npm start` is still up, reuse it; otherwise start fresh: `npm run build` then `npm start`.
2. **For each route in `changes_in_scope` (or representative routes for non-route scope):**
   - `curl localhost:3000/<path>` — confirm HTTP 200 + key content strings.
3. **For `expected_visual_diff: zero` refactor runs:** open localhost:3000 in a browser, click through the affected routes, confirm zero perceptible visual diff against the pre-change baseline. (Manual at v1.0; visual-diff tooling is v1.1+.)
4. **For feature changes:** verify the new behavior end-to-end through the affected user flows.
5. **Console error check:** open the browser console at affected routes, confirm no new errors / warnings introduced by the change.
6. **Report results to Mike** for eyeball confirmation.

**Common terminal logic:**

- **On all-green:**
  - Write `step_completed` event to `run.jsonl`.
  - Write `run_complete` event to `run.jsonl`.
  - Flip the activation-root project's `state:` from `standing` to `done`.
  - Pipeline-run flips `status:` to `complete`.
- **On failure (HTTP non-200, content missing, visual diff for zero-diff refactor, console error introduced):**
  - Capture to `run.jsonl` (`step_failed` event with `data.reason`, `data.failed_routes`, `data.evidence`).
  - Pause the run.
  - Mike directs next action (rollback is separate work at v1.0.0 — see Known Enforcement Gaps).

## Inputs

- Activation-root project: `changes_in_scope`, `expected_visual_diff`, `live_url:` (mode selector).
- Commit SHA from step-3.
- Host status (live mode only).

## Outputs

- Confirmation that the deployed instance renders correctly per mode.
- `run.jsonl` events: `step_started`, `step_completed` or `step_failed`.
- Final `run_complete` event (pipeline-run terminator).
- Activation-root project state flip: `standing → done`.

## Known Enforcement Gaps

| Gap | What closes it | Target release | Owner |
|-----|----------------|----------------|-------|
| Platform app has no public live URL at v1.0.0 — Live mode unavailable until a deploy target is wired (Vercel etc.) | When the platform app gains a public deploy target, declare `live_url:` on activation-root projects; codify host polling pattern | v1.0.0 → v1.1 | talos |
| Visual / behavioral regression beyond HTTP 200 + content strings — relies on manual click-through at v1.0 | v1.1+ Playwright smoke tests against deployed instance | v1.1+ | talos |
| Rollback on failure — no rollback mechanism declared at v1.0.0; current behavior is "Mike directs rollback as separate work" | v1.1+ amendment: optional rollback step (revert commit + push) or `restart_strategy: rollback` semantics | v1.1+ | talos |
| CDN cache staleness (live mode only) — newly deployed pages may hit stale cached versions | Step body documents cache-bust pattern; v1.1+ could automate | v1.0.0 → v1.1+ | talos |
| Console error baselining — "no new errors / warnings" compares to subjective memory; no automated baseline | v1.1+ could capture pre-change console snapshot and diff | v1.0.0 → v1.1+ | talos |
