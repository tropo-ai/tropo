## Purpose

The push-approval gate of deploy stage. Authors a structured commit message, pauses for Mike's approval, then commits and pushes to the platform remote (`tropo-ai/tropo-ai`). The push triggers the host's auto-deploy if wired.

This step encodes Talos's hard rule structurally: **never push to remote without Mike's approval.** The `requires_confirmation: true` flag means the pipeline cannot accidentally push — the gate is in the substrate, not in agent memory.

## Structure

Leaf step. No children. `next_steps` points to smoke-test.

## Behavior

**`requires_confirmation: true`.** Executor enters the step, authors the commit message, then pauses BEFORE the git operations fire.

**Step work:**

1. **Author the commit message.** Read the activation-root project for deploy number, theme, `changes_in_scope`. Read the working-tree diff (`git diff --stat HEAD`) for files-changed. Compose a structured message:
   ```
   <theme> (app-deploy-N)

   Scope: <changes_in_scope summary>
   Files changed: <count> files, +<additions>/-<deletions>
   Expected visual diff: <zero | described | n/a>

   <optional: per-deploy notes from activation-root documents>
   ```
2. **Pause.** Write `pause_started` event to `run.jsonl` with:
   - `data.reason: "push-confirmation-required"`
   - `data.commit_message: <full message>`
   - `data.files_changed: [<list>]`
   - `data.target_branch: main`
   - `data.target_remote: tropo-ai/tropo-ai`
3. **Wait.** Mike reviews. He confirms with `pause_resumed` (granting push approval, optionally redirecting the commit message) or rejects with explicit `pause_resumed` carrying `data.confirmation_granted: false` — in which case the run aborts at this step and the agent fixes whatever Mike named.
4. **On approval:** executor runs `git add <files-in-scope>` (NOT `git add -A` — only the files listed in `data.files_changed`, to avoid sweeping in unrelated working-tree changes), then `git commit -m "<message>"`, then `git push origin <branch>`.
5. Write `step_completed` event to `run.jsonl`. Stage advances to smoke-test.

**Hooks and signing:** never skip pre-commit hooks (no `--no-verify`) or bypass signing unless Mike explicitly requests it. If a hook fails: capture to `run.jsonl`, pause, agent fixes the issue, re-runs.

## Inputs

- Activation-root project: deploy number, theme, `changes_in_scope`, `expected_visual_diff`.
- Working-tree diff at the deploying branch.
- Mike's approval (via `pause_resumed` event).

## Outputs

- Git commit with structured message + tagged files only.
- `git push` to the platform remote (which fires the host's auto-deploy if wired).
- `run.jsonl` events: `step_started`, `pause_started`, `pause_resumed`, `step_completed`.

## Known Enforcement Gaps

| Gap | What closes it | Target release | Owner |
|-----|----------------|----------------|-------|
| Commit message schema — currently a recommended format in the step body; not validated | v1.1+ could require a structured commit-message capsule or validator | v1.1+ | talos |
| Files-in-scope discipline — relies on `git add <specific-files>` rather than `git add -A`; agent could still mis-include unrelated files | v1.1+ could diff `changes_in_scope` against staged-files and flag mismatches | v1.1+ | talos |
| Pre-existing dirty working tree — if there are unrelated uncommitted changes when the run starts, scope discipline is harder; no enforcement | Step-0 could check working-tree cleanness at run-start; v1.1+ enhancement | v1.0.0 → v1.1+ | talos |
| Branch / push target — currently `main` by convention; no canonical declaration if the project moves to a different branching strategy | Step body or activation-root project could declare `target_branch:` at step-0 | v1.0.0 → v1.1+ | talos |
| Unpushed prior commits — if the local branch has commits ahead of remote unrelated to this deploy (e.g., T4 left `cbd89df` unpushed), they will travel with this push | Step-0 should detect ahead-of-remote state and surface; v1.0 relies on agent awareness | v1.0.0 → v1.1+ | talos |
