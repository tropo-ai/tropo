## 1. "What's the governance contract?"

Tropo's governance is explicit, file-based, and inspectable end-to-end.

The model is **four files in three tiers**:

- **OS-tier (two files)**: `AGENTS.md` (in every folder; cross-platform entry point) + `TROPO-CONTROL.md` (the kernel: universal invariants that apply in every Tropo Studio — UID requirement, governance file requirements, operations logging, registry tracking, OS read-only protection).
- **Studio-tier (one file)**: `STUDIO.md` at vault root + companion `.tropo-studio/operating-principles.md` — your organization's defaults and constraints: registration policy, write scopes, agent class rules, custom invariants.
- **Folder-tier (one file per folder)**: `CAPSULE.md` — folder-specific purpose, access control, and operating logic. Each governed folder declares who can write, what belongs, and what protocols apply.

The override contract is explicit: TROPO-CONTROL.md invariants override STUDIO.md; STUDIO.md overrides CAPSULE.md. Nothing slips in unannounced.

Every governed artifact also has a typed contract (a `capsule` — see the [How Capsules Work](5f7a9d83.md) KB). The capsule declares the type's schema, its state machine, validation rules, and what changes require explicit approval. Each capsule defines its own state machine; there's no universal lifecycle. Validation runs via `python3 .tropo/scripts/tropo-validate.py`.

If a rule needs to change, the change itself is a governed artifact (a decision, an ADR, an amendment to the capsule). Nothing slips in unannounced.

## 2. "How do I audit changes?"

Three audit surfaces, each independently verifiable:

- **Git history** — every Tropo Studio is a git repo. Standard `git log` / `git blame` / `git diff` works on every file. Authorship and timing are git-truth.
- **`channels/ops.md`** — append-only operations log. Every meaningful agent action gets a one-line entry: agent ID, action, file path, timestamp. Human-readable in any text editor.
- **Agent registry + activation entries** — `.tropo-studio/registries/agent-registry.yaml` lists every registered agent (canonical roster). Each agent session opens an activation entry at `vault/files/<uid>.md` with `type: activation` recording who activated, which agent, what generation, when — closing with the session outcome. The activation entries maintain an immutable lineage across generations, queryable via `vault/00-index.jsonl`. Playbook runs separately log to `.tropo-studio/registries/playbook-runs.jsonl`.

For incident review: filter `vault/00-index.jsonl` by date range, by agent, by file type, or by any frontmatter field. Combine with `git log` for change attribution.

For continuous audit: standard CI can run `tropo-validate.py` on every commit and surface drift before it lands.

## 3. "Can my team run this with our own data?"

Yes. Tropo is filesystem-local and runs entirely within whatever environment you put it in.

- **No external dependencies** — Tropo doesn't phone home, doesn't require an API key, doesn't ship telemetry. The LLM you use is the only external surface, and you choose which one.
- **Local-first by design** — Tropo vaults are folders on disk. Sync via git, sync via your enterprise file system, sync via whatever your team uses. No SaaS dependency.
- **Your data stays where you put it** — agent-authored files live at `vault/files/<uid>.md` on your filesystem. Whatever data-handling policies apply to your other filesystem content apply here.
- **Air-gapped operation** — if your LLM runs locally (Llama, your own fine-tuned model, etc.), Tropo runs fully offline.

For sensitive workloads: place the Tropo vault inside your existing data-classification boundary. The vault inherits whatever protections that boundary provides.

## 4. "What about compliance?"

Tropo's compliance posture is "the substrate is auditable; the runtime is your choice."

- **SOC 2 / ISO 27001** — Tropo doesn't directly implement these frameworks, but its file-based + append-only + git-versioned design composes cleanly with them. Your existing controls (access management, encryption-at-rest, change-management) apply to Tropo files the way they apply to any other markdown in your repos.
- **HIPAA / data-residency** — Tropo doesn't transmit your data anywhere. The LLM you run is the only data-egress point; choose one that meets your requirements (on-premise model, BAA-covered API, etc.).
- **Audit trails** — git history + `channels/ops.md` + activation registry give you a triple-recorded audit surface that's stronger than most SaaS tools provide.
- **Right-to-be-forgotten** — every governed file is markdown; deletion is `tropo-recycle.py <uid>` (soft-delete to a recycle folder) or `git rm` (hard-delete from history). Both are auditable.

What Tropo doesn't claim: certified compliance with specific frameworks. What it provides: a substrate your compliance team can inspect, attest to, and integrate into your existing posture.

## 5. "How does this work at multi-team scale?"

Tropo's multi-team posture lands fully at v2.0 (Federation Foundation). Today's single-Studio model works well for one team of any size; federation across multiple Studios ships at v2.0.

Within a single Studio:

- **Multiple agents coordinate via channels** — each agent has its own scope (charter), its own files, its own memory. Agents communicate via shared `channels/<agent-a>-<agent-b>.md` files. No agent-to-agent direct messaging; coordination is auditable.
- **Multiple humans share a Studio** — git-based collaboration. Standard PR/review workflow. Multiple humans can direct multiple agents in parallel; the vault tracks who did what.
- **Multiple work-streams compose via projects + pipelines** — each project owns its own work; pipelines define repeatable workflows; the dev-pipeline pattern (used by Tropo itself) shows how to compose them.

Across multiple Studios (post-v2.0):

- **Studio Registration** — each Tropo Studio registers in a federation directory.
- **Studio-Prefixed UIDs** — UIDs become globally unique across the federation namespace.
- **Studio Manifests** — each Studio publishes what it federates; other Studios can subscribe.

For enterprise adoption today: start with one Studio per team. v2.0 lets those Studios federate.

## When this FAQ doesn't answer your question

The architectural source-of-truth lives in the substrate itself. For governance specifics: read `.tropo/TROPO-CONTROL.md` + your Studio's `STUDIO.md`. For audit specifics: run `tropo-validate.py` and inspect `vault/00-index.jsonl`. For compliance integration: bring your compliance team into a working Tropo Studio and let them inspect the substrate directly — the design principle is "you can read every rule that applies to your data."
