## Context

Argo-OS has three classes of documents that require different governance models:

1. **Static documents** — written once, never edited. ADRs, reflections, captain's log entries. Permanent record.
2. **Governed Documents (ADR-023)** — foundational, slow-changing, multi-stakeholder. Operating Agreement, crew-manifest. Require deliberative governance with amendment queues.
3. **Living Documents** — maintained continuously, updated frequently, single owner. The crew brief, agent status cards, briefing-package indexes, intake-feeds, living transfers.

ADR-023 recognized the need for managed documents but defined only the deliberative, slow-changing case. The crew brief is the most-read document in the vault — every agent reads it at every boot — yet it fails ADR-023's criterion 3 ("slow-changing but drift-prone"). It changes daily. It cannot be a Governed Document.

Without a formal class for high-frequency, owner-maintained documents, these documents either get managed ad hoc (current state) or get forced into an inappropriate governance model. The result is either unpredictable maintenance or unnecessary bureaucracy.

This ADR establishes the **Living Document** pattern to fill that gap.

---

## What Qualifies as a Living Document

A document qualifies as a Living Document when it meets all of the following:

1. **Single owner** — one agent or crew member is responsible for its accuracy and has sole authority over its content. No amendment queue; the owner makes changes directly.
2. **Continuous update cadence** — it is expected to be updated on a regular schedule (daily, per-session, or on-demand), not once and left alone.
3. **Broadly read** — it is read by other agents or crew members as a source of current state, not as historical record.
4. **Non-foundational** — it does not define how the crew operates at a constitutional level. It conveys operational state. (If a document is foundational, it belongs in ADR-023's Governed Document class.)

**Current instances:** `00-crew-brief.md`, agent status cards (`agents/[name]/[name]-status.md`), briefing-package indexes (`agents/[name]/briefing-package/00-index.md`), briefing-package intake-feeds, living transfers (`agents/[name]/transfers/living-transfer.md`).

**Not Living Documents:** ADRs, reflections, captain's log, ship's journal (static). Operating Agreement, crew-manifest (Governed Documents). Working channels (ephemeral — governed by ADR-005 and ADR-013).

---

## Required Structure

Every Living Document must have:

### 1. YAML Front-Matter
Machine-readable header per ADR-015:

```yaml
---
owner: "[agent or role responsible]"
update_cadence: "daily | per-session | on-demand"
last_updated: "YYYY-MM-DD"
status: "current | stale | archived"
---
```

### 2. Designated Owner
Named in the front-matter. The owner is responsible for maintaining the document at the declared cadence. If the owner changes (e.g., due to generation transition), the new owner updates the front-matter at their first session.

### 3. CURATOR.md (for folder-based Living Documents)
When a Living Document lives in a dedicated folder (e.g., briefing-package), a `CURATOR.md` in that folder defines the maintenance protocol: what to include, how to update, when to archive, and how to pass the document between generations. See ADR-020 for the CURATOR protocol.

For single-file Living Documents (e.g., crew brief, status cards), the maintenance protocol is defined in the document's own header section or in the relevant activation file.

### 4. Archive Convention
When a Living Document is versioned at a lifecycle boundary (e.g., at agent retirement), the current version is copied to an `archive/` subfolder with a generation or date stamp. The live document is then rewritten for the new context. The archive is a historical record; the live document is always current.

---

## Maintenance Obligations

**Owners must:**
- Update the document at or above the declared cadence
- Update `last_updated` in the front-matter whenever the document changes
- Archive the document at lifecycle boundaries before rewriting
- Not let the document go stale without flagging it to Vela or Mike

**Readers must:**
- Check `last_updated` to assess freshness before relying on content
- Flag apparent staleness to the owner via their working channel (not directly editing the document)

**Vela maintains:**
- The staleness monitor (daily health report will flag Living Documents overdue per their cadence)
- The intake-feed write protocol for briefing-package intake-feeds (ADR-026 consequence)
- Bootstrap protocol for creating initial Living Documents for newly commissioned agents

---

## Relationship to ADR-023 (Governed Documents)

ADR-023 remains in effect and is **not superseded** for its defined scope. The two patterns are complementary, not competing:

| Criterion | Governed Document (ADR-023) | Living Document (ADR-025) |
|---|---|---|
| Change frequency | Slow-changing, drift-prone | Continuously updated |
| Governance | Deliberative — amendment queue, governance CURATOR | Direct — owner updates in-place |
| Stakeholders | Multi-stakeholder, contested | Single owner, broadly read |
| Examples | Operating Agreement, crew-manifest | Crew brief, status cards, briefing-package |

A document cannot be both. If a document is foundational and slow-changing, use ADR-023. If it is operational and continuously updated, use ADR-025.

**ADR-023 is amended to add this clarifying note:** Governed Documents must meet criterion 3 ("slow-changing"). Documents that update on a daily or per-session cadence are not eligible and should be classified as Living Documents under ADR-025.

---

## Consequences

**What becomes easier:**
- The crew brief has a formal governance model that matches how it actually works
- Staleness detection is standardized — `last_updated` front-matter makes it machine-readable
- Briefing-package indexes and intake-feeds have a formal class they belong to
- The distinction between Living Documents and Governed Documents is explicit, preventing future misclassification

**What requires work:**
- All existing Living Documents need YAML front-matter added (ADR-015 front-matter, extended with `owner` and `update_cadence`)
- CURATOR.md files for folder-based Living Documents need to declare which documents in their folder are Living Documents
- Vela: define intake-feed write protocol (who can post, format, clearing cadence) — open item from ADR-026
- Rollout order: crew brief first (blocking crew-brief Phase 2), then briefing-packages, then status cards

**Open items:**
- Crew brief Phase 2 migration: convert crew brief to Living Document standard (Vela leads, unblocked by this ADR once locked)
- Bootstrap protocol for new agent commissioning: Vela to define with Mike
- Living Document template: Argus to produce (or incorporate into briefing-package template per ADR-026)
