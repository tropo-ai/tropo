## What ships

`app/projects/[id]/page.tsx` shrinks from **992 → 456 lines** (54% reduction) by extracting chat orchestration into a project-aware hook + sharing the SSE parser across both hooks.

**New files:**
- `lib/sse-parser.ts` (32 lines) — extracted from useChatSession; shared by both useChatSession + useProjectChatSession.
- `lib/use-project-chat-session.ts` (648 lines) — project-scoped chat orchestration sibling to useChatSession. Differences:
  - API payloads include `projectId` (backend loads instructions/memory/files)
  - Power/plan flow triggered by powerMode only (no auto-mode escalation — projects don't escalate)
  - `onArtifactProduced` callback fires when an artifact is produced (page persists it to project)
  - Exposes setMessages + clearArtifact + displayArtifact for page-driven hydration (e.g., loading a historical chat or artifact)

**Modified:**
- `lib/use-chat-session.ts` — now imports parseSSELines from the shared parser (was inlined locally)
- `app/projects/[id]/page.tsx` — rewritten as a thin orchestrator (456 lines). Project state + project-specific callbacks stay inline; chat orchestration delegated to the hook. Setup-hint banner extracted as private sub-component for readability.

## Why two hooks instead of one parameterized hook?

The state shapes overlap heavily but the orchestration logic differs:
- Main `/` has auto-mode escalation; project page does not
- Main page handles `output_format === "chat"` (chatOnlyResponse); project page does not
- Project page persists artifacts to the project; main page does not
- Project page exposes hydration setters (setMessages, displayArtifact, clearArtifact) for chat-loading + history-loading flows; main page never needs these

Parameterizing useChatSession to handle both would create branchy options surface. Two siblings sharing the SSE parser is cleaner.
