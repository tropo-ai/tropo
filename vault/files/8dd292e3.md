## Intent

Ship the first tutorial in the Hello Tropo series — a concierge playbook that walks a fresh user through building their own activated project end-to-end. The user ends the tutorial with a real project in their Studio (their event, their workstreams, 2-3 first tasks) — not a copy of Hello Tropo, but their own.

The conversion lever loads here: downstream channel content (Substack / LinkedIn / Reddit / HN / tropo-ai.com) points AT this tutorial as the load-bearing artifact. v2.0.0's 1000+ public users land at tropo-ai.com, download the zip, hit Welcome Playbook, then run this tutorial. The tutorial converts download → first activation.

## Decisions Locked (Mike-walked Q1-Q9)

1. **Load-bearing artifact: in-Studio playbook.** Lives at `vault/files/<uid>.md` with `type: playbook`; ships with every Studio. Channel-specific content (Substack drafts, LinkedIn posts, etc.) is downstream and your ongoing creative work — not v1.36.0 scope. tropo-ai.com copy is a sibling concern, separate cycle.
2. **Audience: in-Studio new user post-Welcome-Playbook.** They've completed Welcome Phases 1-3 (greet + first agent + Hello Tropo walk). Want step-by-step "build my own first real thing" guidance. Voice: Tropo T1's curious-curator register — same person they've been talking to.
3. **Tutorial as separate playbook + Welcome amendment as route-in.** Tutorial complements Welcome by addressing the "build new from scratch" path. Welcome Phase 4 stays as-is for "import existing work." Tutorial invocable standalone for experienced users skipping Welcome.
4. **Concrete example end-to-end + substitution prompts at each step.** Concrete teaches; abstract gets skimmed. Walking a real example linearly with explicit substitution invitations — *"This is where I picked X for the customer event; what would you pick for YOUR project?"* — gives shape without locking. Medium depth: ~30 minutes reader effort first pass.
5. **Adapt-as-you-go.** User builds their own activated project in parallel with the example. Substitutes their event, their workstreams, their tasks at each prompt. Tutorial ends with the user having a real activated project — that's the conversion moment.
6. **Example: Hello Tropo's existing substrate (not a different example).** Composes with v1.35.0; user already absorbed the shape via Welcome Phase 3. Two examples doubles cognitive load and substrate cost without proportional teaching benefit. Sticking with Hello Tropo.
7. **6-phase structure with teach-along-the-way.** Phase 0 covers meta-concepts (tropo-work, typed work-items, capsules). Phases 1-5 each teach the primitive they exercise (project / pipeline+workstream / activation+cascade / project board+plan / task). Just-in-time learning beats front-loading.
8. **Example project name: "2026 Customer Event Plan."** Substrate gets renamed (Hello Tropo project + master pipeline + workstream pipelines + activation + project plan) from "Industry Conference Activation" → "2026 Customer Event Plan." Series name "Hello Tropo" preserved; first tutorial = **"Hello Tropo #01 — 2026 Customer Event Plan"** (zero-padded; future-proofs past #9). Folder slug stays `hello-tropo/` (slug = series name; project inside renames only).
9. **Welcome Playbook amendment route-in.** One-line addition at Phase 3 end or Phase 4 entry: *"If you want to build your own project from scratch, run the Hello Tropo tutorial series. The first tutorial walks you through planning a customer event — same shape you just saw, but with your event, your workstreams, your tasks."* Light touch; doesn't restructure Welcome.

**Plus binding tutorial-authoring discipline (Mike-A68 directive 2026-05-17):** user-facing playbooks describe WHAT happens, not HOW the kernel does it. No `.py` file names. No schema jargon (cascade_spec, owner_agent_class, member_of). No validator vocabulary. Tropo T1 says *"I'll create your project structure now,"* not *"I'll run pipeline-activate.py to fire the cascade."* This is memory-pin-worthy beyond this cycle — binding for ALL future user-facing playbook authoring.

## Streams

### Stream A — Tutorial playbook authoring (load-bearing)

Author `Hello Tropo #1 — 2026 Customer Event Plan` playbook at `vault/files/<new-uid>.md` with 6 phases per Q7 lock:

- **Phase 0 — Welcome + Definitions** (~150 lines). What tropo-work is + typed work-items + capsules + common attributes. Series framing. "Ready?" prompt. No substrate authored.
- **Phase 1 — Frame your project** (~80 lines). Teaches **project**. Asks for event name + date + goal. Captures + authors user's project entry.
- **Phase 2 — Design your shape** (~120 lines). Teaches **pipeline** + **workstream**. Walks workstream identification (Hello Tropo has 6; user's might have fewer). Captures + authors user's master pipeline.
- **Phase 3 — Tropo creates your project structure** (~80 lines). Teaches **activation** + **cascade** (conceptually; no script names). One step creates the structure. ~10-15 work-items appear.
- **Phase 4 — Walk your substrate** (~120 lines). Teaches **project board** + **project plan**. Reads + explains: dashboard, coordinating plan, workstream sub-projects. No new substrate.
- **Phase 5 — Start working** (~80 lines). Teaches **task** as a work-item shape. User picks a workstream + names 2-3 tasks they would do. Captures + authors tasks. Tutorial closes with what's next.

Voice: Tropo T1 curious-curator. Conversational. Pauses for user input at substitution prompts. References Hello Tropo's reference example at each phase as concrete anchor.
