## Context

This note surfaces a substrate-engineering work-stream that emerged during the tropo-ai.com Launch Foundations project's design walk on Deliverable #8 (Agent-Builder Design Spec) on 2026-05-16.

The conversation started narrow — the website's agent-builder wizard generates agent files; should they go to `agents/<name>/` (the older flat substrate pattern) or `vault/files/<uid>.md` (the v1.21.0.1 governed-substrate pattern Argus locked)?

Mike resolved the narrow question cleanly: **the wizard ships files at `agents/<name>/`, and the agent itself handles its own substrate migration into vault on first boot.** User stays at flat-shape forever; agent self-cares.

Then Mike extended the thinking: **if we're building a "thing that moves agent files into vault" playbook anyway, we should generalize it.** Make it smart enough that a user can bring their own agent from ANY source — LangChain, CrewAI, AutoGen, a hand-written markdown file, another Tropo Studio's export — and Tropo's playbook reads it intelligently and maps it into Tropo substrate.

That generalization is the real opportunity. The narrow wizard-migration case is a special instance of the broader capability.

---

## The two related capabilities

### Capability A — Agent first-boot migration (narrow)

**The case:** Wizard-generated agent files (or hand-authored starter files) live at the flat `agents/<name>/<name>-*.md` paths. On first activation, the agent itself runs a migration step: reads its own identity files, mints UIDs, copies content into `vault/files/<uid>.md`, updates its activation file's frontmatter with the new UIDs, rebuilds the vault index, continues activation.

**After first boot:**
- Identity files live at `vault/files/<uid>.md` (governed substrate)
- `agents/<name>/<name>-activation.md` still exists as the human-readable entry point, with frontmatter pointing at UIDs (matches the argo-os pattern Argus locked at v1.21.0.1)
- Indistinguishable from a "born-in-vault" agent like Talos/Argus/Vela/Metis/Orpheus

**This is the table-stakes capability** — the website agent-builder design (Deliverable #8 of the Launch Foundations project) depends on it. Brief lock for #8 is gated on this playbook landing.

### Capability B — Agent import/export (broad)

**The case:** A user wants to bring an agent from elsewhere into Tropo, or take a Tropo agent out for use somewhere else. The "elsewhere" could be:

- **LangChain agent definitions** (Python class structures with prompts, tools, memory configs)
- **CrewAI YAML agents** (declarative agent + crew configs)
- **AutoGen agent configs** (Python or JSON)
- **OpenAI Assistants API** (JSON blob with instructions, tools, file IDs)
- **Anthropic Claude project files** (system prompts + tool definitions)
- **Plain markdown notes** describing an agent in informal English
- **Tropo wizard output** (the flat `agents/<name>/` shape we just locked) — this is the simplest input format, identical to Capability A
- **Another Tropo Studio's export** — round-trip portability

**The playbook:**

1. **Detect input format.** File structure inspection, content patterns, frontmatter signals. If ambiguous, ask the user.
2. **Map source concepts to Tropo concepts.** Each input format has its own primitive vocabulary; Tropo has its own. Examples:
   - LangChain `system_prompt:` → Tropo soul (3 values extracted) + charter §Role
   - CrewAI `goal:` → Tropo charter §Mission
   - AutoGen `description:` → Tropo charter §Scope
   - LangChain `tools:` → Tropo boot-extension §Resources
   - OpenAI Assistants `instructions:` → Tropo soul + charter combined
3. **Surface gaps conversationally.** Tropo expects specific structures (3 soul values, status card, boot-extension Tier 3 behavioral rules). If the source doesn't have these, the playbook prompts the user to fill them in — using the same conversational pattern as `create-an-agent.playbook`.
4. **Author Tropo substrate.** Generate the canonical files: `agents/<name>/<name>-activation.md` (thin pointer) + `vault/files/<uid>.md` for each identity file type.
5. **Rebuild vault index.** Make the new files queryable.
6. **Hand off to activation.** Optional: trigger first activation. Or just hand off to the user with "Your agent is ready; run activation when ready."

**Export side:** Reverse direction. Read Tropo substrate, produce a file in the target format. Useful for: porting a Tropo-built agent to another framework, sharing an agent with someone who doesn't run Tropo, archiving an agent definition.

### The architectural insight

**Capability A is a special case of Capability B** where the input format is "Tropo's own flat `agents/<name>/` shape." The import playbook detects that format and runs a trivial mapping (no concept translation needed; just UID minting + relocation).

So Argus could author this as **one unified playbook** — `agent-import.playbook.md` — that handles all input formats, with Tropo's flat shape as one supported format among many. The first-boot migration case is the lightest path through that playbook.

Or **two playbooks** — `agent-first-boot-migration.playbook.md` (narrow, runs automatically at first activation) and `agent-import.playbook.md` (broader, user-invoked) — that share underlying machinery. Trade-off: cleaner conceptual separation vs more code duplication.

Argus's call which architecture is right.

## Why this matters strategically

### Lowers switching cost for new Tropo users

Someone who's already built agents in LangChain, CrewAI, AutoGen, OpenAI Assistants, etc. can migrate to Tropo without rebuilding from scratch. They run the import playbook, point it at their existing agent files, answer the gap-filling questions, get a fully-Tropo-shaped agent in their Studio. The friction of "rewrite all my agents from scratch to try Tropo" disappears.

This is meaningful because **every other agent framework currently makes you choose between their primitives or starting over.** Tropo offering a clean import path is genuine differentiation.

### Validates Tropo's substrate as a universal target

If LangChain, CrewAI, AutoGen, and OpenAI agents can all be mapped cleanly into Tropo substrate, that's a strong claim about Tropo's primitives — they're general enough to be the lingua franca of agent definitions. That validates the "Tropo is the operating system for AI agents" positioning at a substrate level, not just a marketing level.

### Establishes interoperability as a Tropo capability

Once import exists, export naturally follows. Once export exists, round-trip portability exists. Tropo agents become portable substrate — you can build them in Tropo, take them elsewhere, bring them back. That's the open-source ethos in action.

### Aligns with the website's hub vision

Per [website strategic expansion brainstorm (e8d3a275)](e8d3a275.md), the tropo-ai.com website is becoming a community + learning hub. A natural section of that hub: "Bring your existing agents to Tropo." Walks visitors through the import playbook, shows them the mapping their LangChain/CrewAI/AutoGen agent gets translated into, gives them a fully-Tropo agent as the output. Conversion funnel for existing-framework users.

## Use cases this enables

1. **Wizard agent first-boot** — narrow case, depends on Capability A.
2. **User has a hand-written agent in plain markdown** — drops it into `agents/<name>/`, runs import playbook, gets fully-Tropo agent.
3. **User migrates from LangChain** — points import at their LangChain agent file, answers gap-fill questions, gets Tropo agent.
4. **User shares an agent with a friend** — exports their agent to a portable format, friend imports it on their Tropo Studio.
5. **User wants their Tropo agent in AutoGen** — exports to AutoGen config format, runs it elsewhere.
6. **User has agents across multiple Studios and wants to consolidate** — exports from one Studio, imports into another, rebuilds.
7. **Recovery / forensics** — old agents from archived Studios can be imported into current substrate via the same playbook.

## Design considerations for Argus

### Input format detection

Heuristics on file structure + content patterns. Examples:

- **Tropo flat shape:** `agents/<name>/<name>-*.md` files exist with expected names + frontmatter
- **LangChain:** Python class definition with `langchain` imports
- **CrewAI:** YAML with `role:`, `goal:`, `backstory:` keys
- **AutoGen:** Python or JSON with `assistant_agent` / `user_proxy_agent` patterns
- **OpenAI Assistants:** JSON with `object: "assistant"` + `instructions` field
- **Plain markdown:** None of the above; informal description

When detection is ambiguous, ask the user: "I see [X, Y, Z] files; which best describes your existing agent's source?" Multiple-choice. Cheap conversational step.

### Mapping table (recommended substrate)

A canonical mapping document — `vault/files/<uid>.md` — that declares for each supported source format, how each source concept maps to Tropo concept. Maintained by Argus; extended as new formats are added. Used by the playbook at runtime. Acts as the documentation for "what does Tropo expect from a [LangChain/CrewAI/AutoGen] agent."

### Gap-fill conversation

Reuse the same conversational pattern as `create-an-agent.playbook`. If the source agent doesn't have a status card, the import playbook just asks the questions create-an-agent would ask to generate one, then writes the file. No new conversational engineering needed — compose with what exists.

### UID minting + collision avoidance

Each imported agent's identity files get fresh UIDs minted in the destination Studio. UID collision with existing vault entries is handled the same way the rebuild-vault.py validator handles it — surface and ask user to resolve. Should be rare in practice (8-hex UID space).

### The "save the original" question

When importing, should we preserve the source file at its original path, or replace it with Tropo substrate? Lean: preserve the source. Add a `imported_from:` field to the activation file's frontmatter pointing at the original. If the user wants to clean up the source after a successful import, they can — but the import playbook doesn't delete user-authored content unilaterally.

### Versioning

If the mapping table changes (e.g., LangChain v2 introduces new primitives), already-imported agents may not need re-mapping — but the import playbook for new imports should use the latest mapping. Standard versioning hygiene: capsule it.

### Export side

Export is the simpler half. Read Tropo substrate, output to target format. Each supported target format gets its own export sub-playbook (or one playbook with format-specific output sections). Should be a sibling to import.

### Composition with existing substrate

This playbook family composes with:
- [create-an-agent.playbook (57a87002)](57a87002.md) — gap-fill conversation reuses its patterns
- [create-executive-agent.skill](../../.tropo/skills/create-executive-agent.skill.md) — the validation rules apply to imported agents too
- [agent-activation.playbook (99341618)](99341618.md) — the first-boot migration case integrates with the standard activation flow
- [vault/AGENTS.md governance](../AGENTS.md) — imported agents must respect destination Studio's governance

## Naming proposals

Bikeshed-prone but worth naming intentionally:

- `agent-import.playbook.md` + `agent-export.playbook.md` — clearest, two playbooks, plain English
- `agent-onboarding.playbook.md` — single unified playbook; "onboarding" covers both first-boot and external-source cases
- `agent-migration.playbook.md` — emphasizes the substrate transformation aspect
- `agent-customs.playbook.md` — whimsical; immigration metaphor (Tropo as country, agents as immigrants)

Talos T4 lean: **`agent-import.playbook.md` + `agent-export.playbook.md` as separate playbooks**, with the first-boot migration case as an automatic invocation of `agent-import` triggered at activation. Two-playbook clarity beats one-playbook flexibility for substrate. The "wizard-output → Tropo substrate" path is just `agent-import` running silently with the Tropo-flat input format detected.

## Phasing recommendation

This work doesn't need to ship at once. Reasonable phasing:

**Phase 1 (gates the website agent-builder brief #8 — needed soon):**
- `agent-import.playbook.md` with one supported input format: Tropo's own flat `agents/<name>/<name>-*.md` shape.
- This unblocks Capability A (wizard agents migrate on first boot).
- Estimated effort: 4-6 hours (mostly playbook authoring + UID minting + index rebuild integration).

**Phase 2 (when website agent-import surface ships, probably Phase 2-3 of website hub work):**
- Extend `agent-import` with LangChain, CrewAI, AutoGen, OpenAI Assistants format support.
- One format at a time. Each adds a mapping section + format-detection branch.
- Authoring effort per format: 3-5 hours.

**Phase 3 (later, when export use cases surface):**
- `agent-export.playbook.md` mirroring import in reverse.

**Phase 4 (community surface from e8d3a275):**
- "Submit a format" community pattern — users contribute new format mappings via PR to the public Tropo-OS repo. Format library grows from community contributions.
