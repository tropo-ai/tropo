# tropo-work-pipeline — Pipeline v1.0

*The canonical pipeline Tropo-OS projects walk. Formalizes the existing `work-pipeline/` [020274e0] bucket structure as a declared pipeline artifact governed by [pipeline.capsule v1.0 (e4c8a6b2)](../../capsules/pipeline.capsule.md). Five stages × five positions each = 25 positions. Projects that declare `active_pipeline: 7c5a2b8f` walk this pipeline.*

---

## What This Pipeline Governs

**In scope:** the work-flow for Tropo-OS architectural and governance work — concepts, design briefs, arch-specs, builds, releases, and the projects that carry them through stages. This is the pipeline for Tropo-OS's own development.

**Not in scope:** workflows in other domains (marketing, finance, customer-facing product work). Those teams should author their own domain-specific pipelines — `pipeline.capsule v1.0` is the governance; this pipeline is one instance.

**The boundary test:** if the question is "how does a Tropo architectural or governance artifact move from idea to ship," that's this pipeline. If the question is "how does our marketing team move a campaign from concept to launch," that's a different (yet-to-be-authored) pipeline.

---

## Stages

Five ordered stages: `ideate → design → specify → build → deploy`. Each stage produces a typed artifact; projects walking the pipeline accumulate scope as they advance.

### Ideate

**Purpose:** capture raw ideas and shape them into triageable concepts.
**Artifact produced:** `concept` (per [concept.capsule v1.0 (c9e1a5b7)](../../capsules/concept.capsule.md))
**Positions + typical owners (role-class):**
- `ideate-inbox` — anyone (open intake)
- `ideate-accepted` — director (triage decision)
- `ideate-active` — anyone (capture + shape)
- `ideate-GATE` — director (advance/attach/close decision)
- `ideate-archived` — director (terminal state post-decision)
**Completion criteria (§GATE):** concept reaches `status: shaped` per concept.capsule; all 4 required body sections populated.

### Design

**Purpose:** form concepts into fully-shaped design briefs that frame architecture and name open questions.
**Artifact produced:** `design-brief` (per [design-brief.capsule v2.1 (de5181b0)](../../capsules/design-brief.capsule.md))
**Positions + typical owners:**
- `design-inbox` — director (accept concepts arriving from Ideate)
- `design-accepted` — architect
- `design-active` — architect (author the brief)
- `design-GATE` — director (advance/attach/close decision)
- `design-archived` — director
**Completion criteria (§GATE):** design-brief has all required sections, open questions identified, architecture framework proposed. Specify-ready.

### Specify

**Purpose:** formalize briefs into structurally-sound specifications with named contracts.
**Artifact produced:** `arch-spec` (per [arch-spec.capsule v1.0 (a7f2e9c4)](../../capsules/arch-spec.capsule.md))
**Positions + typical owners:**
- `specify-inbox` — architect (accept briefs arriving from Design)
- `specify-accepted` — architect
- `specify-active` — architect (author the spec)
- `specify-GATE` — architect (advance/attach/close decision)
- `specify-archived` — architect
**Completion criteria (§GATE):** spec reaches `status: locked` per arch-spec.capsule; required contracts declared; build-ready.

### Build

**Purpose:** form specs into fully-built, tested, documented packages.
**Artifact produced:** `build` (per [build.capsule v1.0 (b3d7e5a1)](../../capsules/build.capsule.md))
**Positions + typical owners:**
- `build-inbox` — architect (accept specs arriving from Specify)
- `build-accepted` — architect
- `build-active` — architect (produce the build)
- `build-GATE` — architect (advance/attach/close decision)
- `build-archived` — architect
**Completion criteria (§GATE):** build reaches `status: locked` per build.capsule; §Test Results populated; deploy-ready.

### Deploy

**Purpose:** ship builds as versioned releases. Deploy stage uniquely produces TWO paired artifacts: the `release-plan` that coordinated the ship closes, and the `release` that records what shipped is born.
**Artifact produced:** `release` (per [release.capsule v2.0 (b19e8d43)](../../capsules/release.capsule.md)) + paired `release-plan` closure per [release-plan.capsule v1.0 (a3f1e7b2)](../../capsules/release-plan.capsule.md).
**Positions + typical owners:**
- `deploy-inbox` — releaser (accept builds arriving from Build)
- `deploy-accepted` — releaser
- `deploy-active` — releaser (execute ship)
- `deploy-GATE` — releaser (advance to another pipeline, or close; rarely attach)
- `deploy-archived` — releaser
**Completion criteria (§GATE):** release entry filed (`status: done`); release-plan transitions to `stage: done` with `shipped_release:` pointing at the new release entry; build's `composes_into:` set atomically. This is the atomic-commit triangle per release.capsule Rule 3.

## Forward-Only Motion Rule

Projects walking `tropo-work-pipeline` advance their `position:` field monotonically forward in the `positions:` array order declared above. A project at `design-GATE` may advance to `design-archived`, `specify-inbox`, `specify-accepted`, etc. — any position at a later index in the array. Backward moves (e.g., `specify-active → design-active`) are prohibited.

**Exception: pipeline mobility.** A project may exit this pipeline entirely and enter another by flipping `active_pipeline:` to a different pipeline UID + setting `position:` to a valid position in the new pipeline. This is documented in [project.capsule v2.2 (34e4cb0b)](../../capsules/project.capsule.md) §Pipeline Mobility.

**Enforcement:** honor-system in v1.3; ADR-037 trigger-based enforcement in v1.4.

## Stage Transition Rules (advance / attach / close)

At every GATE position, the project owner declares one of three transitions per the typed pipeline brief Typed Pipeline Architecture brief §4:

### Advance

Project continues forward within this pipeline to the next stage's inbox.

- `position:` flips from `<stage>-GATE` to `<next-stage>-inbox` (or further, up to the owner)
- A new next-stage artifact is authored with `derived_from:` including the completed artifact
- The completed artifact's `composes_into:` is populated with the new artifact's UID
- Scope of activity widens to include the new stage

### Attach

Project's completed artifact composes into another project's in-progress artifact; this project archives.

- The target project's next-stage artifact's `derived_from:` gets the completed artifact's UID appended
- The completed artifact's `composes_into:` gets the target artifact's UID
- This project's `state:` flips to `archived`
- This project's `attached_to:` is set to the target project's UID
- This project's `position:` stays at the GATE (provenance)

### Close

Project's work stands as historical; no downstream composition.

- This project's `state:` flips to `archived`
- This project's `position:` stays at the GATE
- Completed artifact remains, attached to its stage-archive evergreen project as secondary parent:
 - ideate → [ideate-archive (a1c2e3d4)](../files/a1c2e3d4.md)
 - design → [design-archive (b2d3f4e5)](../files/b2d3f4e5.md)
 - specify → [specify-archive (c3e4a5f6)](../files/c3e4a5f6.md)
 - build → [build-archive (d4f5b6a7)](../files/d4f5b6a7.md)
 - deploy → [deploy-archive (e5a6c7b8)](../files/e5a6c7b8.md)

**Deploy stage specifically:** the Attach path is rare at deploy-GATE because the release is a terminal output — attaching to another release's in-progress artifact rarely makes sense. `gate_rules.deploy: [advance, close]` reflects this. Advance at deploy usually means "switch pipelines" (e.g., a shipped release becomes the foundation for a marketing-campaign-pipeline project).

## Cold-Boot Walk-Through

Hypothetical project walking the pipeline end-to-end, showing each transition.

**Project:** a hypothetical "Tropo Telemetry Starter" — someone wants to ship a reference implementation of Tropo-OS telemetry governance.

1. **Ideate.** Someone drops a concept at `ideate-inbox`. A director triages (`ideate-accepted`). The concept gets shaped at `ideate-active` until all 4 required sections have substance. At `ideate-GATE`, the director declares **advance** — the concept is promising and warrants a design brief.

2. **Design.** The concept's project advances to `design-inbox`. The architect accepts (`design-accepted`) and begins authoring the design brief (`design-active`). The brief references Typed Pipeline Architecture brief (typed pipeline), proposes a telemetry subsystem architecture, names 6 open questions. At `design-GATE`, the director declares **advance** — the brief is spec-ready.

3. **Specify.** The architect picks up the brief at `specify-inbox`, accepts, and authors the arch-spec (`specify-active`). The spec formalizes data model, query protocol, storage layer. Locks at `specify-GATE`. Architect declares **advance**.

4. **Build.** The architect produces the build (`build-inbox → accepted → active`): scaffolding, sample queries, verification harness, cold-boot-tested. At `build-GATE`, architect declares **advance**.

5. **Deploy.** The releaser picks up the build at `deploy-inbox`, files the release-plan, executes the ship (`deploy-active`), and files the release entry (`release.derived_from: [<build-uid>]`; `release-plan.shipped_release: <release-uid>`; `build.composes_into: <release-uid>`). At `deploy-GATE`, releaser declares **close** — the ship is terminal.

Each transition is a deliberate declaration; each composability edge is set atomically.
