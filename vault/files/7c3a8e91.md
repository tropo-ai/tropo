## What this is

This is your vault's **vault-entity** — the root entity that grounds every action in this vault back to a human (or AI agent) principal. Every Tropo-OS vault has exactly one. It's how the system knows whose vault this is and who's ultimately accountable for what happens here.

You don't usually edit this file directly — the Concierge handles updates as part of normal first-run setup and ongoing customization.

## Identity

- **Name:** `your-vault` *(concierge updates to your preferred slug on first run)*
- **Subtype:** vault
- **Principal:** [Vault Owner (4b6e2c8a)](4b6e2c8a.md) — placeholder; concierge updates with your real name + identity on first interaction
- **Vault class:** `personal` (single-user starter; can be re-declared if you turn this into a team or company vault later)

## Role in the Tropo-OS substrate

Per the [Tropo Work substrate](../../vault/files/d61ce0a7.md), every governed work-item must trace ownership up to a vault-entity. This is that anchor. It enables:

- **D7 invariant satisfaction:** every work-item `member_of:` chain ends at a vault-entity-owned project. The vault-inbox ([2d5f9b04](2d5f9b04.md)) catches orphans that don't have a more specific home.
- **Principal grounding:** every action in this vault traces to a person (via `principal:` chain). Audit trails resolve.
- **Activity logging:** vault-scope events (entity creation, agent commissioning, release ship-signals) append to `vault/activity.jsonl`.

## What the Concierge does on first run

1. Asks for your vault's preferred name (e.g., "Acme Research" or "Maria's Studio")
2. Updates this entry's `title:`, `name:`, and `vault_class:` to match
3. Walks you through customizing the [Vault Owner (4b6e2c8a)](4b6e2c8a.md) placeholder with your real name + pronouns
4. Logs the changes in this entry's activity log

If the Concierge is bypassed, this entry stays as "Your Tropo Vault" — the vault still works; the names are just generic.
