# ADR-013: Context Budget Ceiling and Proactive Archival

## Context

On March 23, 2026, three agents (Vela V4, Argus A1, Metis G7) crashed simultaneously from context overflow. Root cause: working channel files grew unchecked as productive conversations accumulated. The required-reading set for each agent exceeded context capacity. The vault-janitor — designed to archive bloated files — had never run.

The crash was not caused by bad architecture. It was caused by success: agents communicated well, worked productively, and generated channel content that grew past the capacity of any single session to consume. The system punished its own success.

Metis G8 identified the structural fix: enforce context budgets at write time, not read time. A2.4's research confirmed this pattern across database systems (write-ahead logging), configuration management (drift detection with proactive remediation), and knowledge management (metadata-driven freshness tracking with automated archival).

The vault-janitor (ADR-002) exists but has never run. This ADR defines the context budget ceiling that the janitor enforces, the archival trigger mechanism, and the relationship between the ceiling and the boot sequence.

---

## Decision

### 1. Hard Context Budget Per File Type

Every file that agents are required to read has a maximum token budget. When a file approaches its ceiling, proactive archival is triggered — the file does not grow past the ceiling under any circumstance.

| File Type | Ceiling (lines) | Ceiling (approx tokens) | Trigger (% of ceiling) |
|---|---|---|---|
| Working channel (`[name]-[name].md`) | 150 lines | ~3,000 | 75% (112 lines) |
| Topic channel (`ops.md`, `alerts.md`, `releases.md`) | 200 lines | ~4,000 | 75% (150 lines) |
| Crew brief (`00-crew-brief.md`) | 250 lines | ~5,000 | 80% (200 lines) |
| Agent status card (`[name]-status.md`) | 60 lines | ~1,200 | 80% (48 lines) |
| Worklog (`[name]-worklog.md`) | 200 lines | ~4,000 | 75% (150 lines) |

Files not in this table (ADRs, guides, activation files, design documents) are not subject to automatic archival. They grow through deliberate editorial action, not append-only accumulation.

### 2. Proactive Archival Trigger

When a file crosses its trigger threshold, the vault-janitor:

1. **Detects** the threshold breach during its scheduled sweep.
2. **Archives** older content to `archive/[type]/[filename]-[YYYY-MM].md`. For channels, this means moving all entries older than 48 hours to the archive file, preserving the most recent 48 hours in the live file. For worklogs, sessions older than the current generation's sessions are archived.
3. **Fallback rule:** If the file still exceeds the ceiling after time-based archival (because all content is recent — a productive single-day session), keep only the most recent 80 lines and archive the rest regardless of age. The ceiling is absolute; the time-based rule is the preferred method but not the only method.
4. **Writes a header note** in the live file: `*Archived entries before [date] at archive/[path]. This file was auto-trimmed per ADR-013.*`
5. **Posts to ops channel:** `[vault-janitor] ARCHIVAL | [filename] trimmed from [N] to [M] lines. Archive at [path].`
6. **Verifies** the live file is below the ceiling after archival.

### 3. Write-Time Enforcement

When any agent writes to a ceiling-governed file and the write would push the file past the ceiling:

1. The write proceeds (we do not block agent output).
2. The vault-janitor is flagged for immediate archival on its next run.
3. If the janitor has not run within 24 hours, any agent detecting the overage posts to the alerts channel: `[ALERT] [filename] exceeds context budget ceiling (ADR-013). Vault-janitor overdue.`

This is detect-and-remediate, not prevent. We do not block writes because blocking creates data loss. We archive aggressively to keep files under ceiling.

### 4. Vault-Janitor Activation Priority

The vault-janitor is the highest-priority standing operational process. It runs:

- **First** in every Vela session, before any other operational work.
- **On a 24-hour maximum cycle.** If no Vela session occurs within 24 hours, the next session that detects stale janitor state posts to alerts.
- **On demand** when any agent detects a ceiling breach.

The janitor is not housekeeping. It is resilience infrastructure. The March 23 crash proves that failure to run the janitor has crew-level consequences.

### 5. Boot Sequence Integration

The context budget ceiling is designed to keep the total required-reading set for any agent under 15,000 tokens at boot. This budget is allocated:

| Boot Layer | Budget | Content |
|---|---|---|
| Tier 0: Identity | ~500 tokens | Activation file header, env.md |
| Tier 1: Situational Awareness | ~2,500 tokens | ASA activation report (orientation index, channel digest, staleness audit, predecessor diff) |
| Tier 2: Role Context | ~6,000 tokens | Role-specific guides, relevant ADRs, crew brief |
| Tier 3: Deep Context | ~6,000 tokens | On-demand: archived channels, predecessor reflections, design documents |

Tiers 0-2 are loaded at boot. Tier 3 is loaded on demand during the session. The ceiling ensures Tiers 0-2 never exceed their budget regardless of vault growth.

---

## Rationale

**Why enforce at write time (detect-and-remediate), not read time (block)?** Blocking writes loses data. Blocking reads loses context. Neither is acceptable. The only safe approach is to let writes proceed and archive aggressively. The write is the signal; the archival is the response.

**Why proactive (threshold trigger) rather than reactive (ceiling breach)?** The 75-80% trigger threshold gives the janitor a window to archive before the ceiling is hit. If the janitor runs on its 24-hour cycle, the buffer absorbs normal write rates without breach. The March 23 failure happened because there was no trigger at all — files grew to 400+ lines before anyone noticed.

**Why per-file-type ceilings rather than a single global limit?** Different file types serve different functions. A channel is a rolling conversation — old entries lose value fast. A crew brief is a synthesized reference — it should be tighter. A worklog is a historical record — archival moves old sessions, not recent ones. The ceilings reflect how each file type decays.

**Why the vault-janitor, not individual agents?** Centralized archival ensures consistency. If each agent archived its own channels, concurrent writes could produce conflicts or partial archives. The janitor is the single writer for archival operations.

---

## Consequences

**Positive:**
- Context overflow as a crash vector is eliminated for ceiling-governed files.
- Boot cost is predictable and bounded regardless of vault growth.
- The janitor has a concrete, measurable job: keep files under their ceilings.
- Agents can write freely without worrying about file growth.

**Negative:**
- Historical context is moved to archive files that agents don't read by default. If an agent needs old channel history, it must deliberately load the archive (Tier 3).
- The janitor becomes a critical path dependency. If it fails to run, ceilings are breached. Mitigation: the 24-hour alert mechanism.
- The ceiling values are estimates. They may need tuning after operational experience. The first janitor runs should be monitored closely.

**Operational changes:**
- Vela adds janitor-first to her session startup checklist.
- The vault-janitor activation file is updated with ceiling values, trigger thresholds, archival format, and ops channel reporting.
- Standing auditors add a ceiling check to their detection suite.
