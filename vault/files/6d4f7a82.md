## Objectives

- **tropo-ai.com is live** on a production host with DNS wired from Cloudflare, SSL provisioned, a real page serving HTTP 200.
- **Tropo (the company) has clean founder-ops infrastructure** — Workspace email at the domain, GitHub org, Vercel team, Supabase project, Resend account, Cloudflare account — all owned by a Tropo identity (`mike@tropo-ai.com`), not Mike's personal accounts.
- **Web-pipeline ([7b2e94c1](7b2e94c1.md)) can fire its first end-to-end run** against the live host: commit → push → host auto-deploys → smoke-test-live confirms.
- **Layer-0 design surface is speced and locked**, so the next deploys after the placeholder are execution work, not design work. Agent-builder spec has architectural surface and is reviewed by Argus before lock.
- **Substrate choices made now don't have to be ripped out** when Layer 1 (accounts) and Layer 2-3 (hosted studios) arrive.

## Scope

**In-scope:**
- Host selection + Vercel team account
- Domain + DNS wiring (Cloudflare → Vercel)
- Google Workspace at tropo-ai.com (Mike's Tropo email + Google identity)
- GitHub org for Tropo (with `tropo-ai` repo migration)
- Supabase project (Postgres + Auth + Storage)
- Resend account (transactional email; domain-verified)
- Cloudflare account hygiene (Tropo-owned, not personal)
- First production deploy — placeholder page proving the pipeline end-to-end
- Agent-builder design spec (Layer 0 wizard shape; output package contents)
- Substack RSS pipeline design (how blog content reaches tropo-ai.com)
- GitHub Releases integration design (how the download surface reads releases)

**Out-of-scope (deferred to future projects):**
- Implementation of agent builder (a future deploy after the design brief locks)
- **Vault registration entirely** — design spec + implementation. Deferred per Mike's call on 2026-05-15: not a v1 launch surface. When the time comes, a future project commissions the architecture spec (and the UID-scheme architectural call gated on Argus).
- Implementation of blog routes + Substack ingestion (future deploy)
- Implementation of download UI (future deploy)
- Real account system + sign-in (Layer 1 — deferred)
- Hosted studios — read-only or read-write (Layer 2-3 — deferred)
- Collaboration, curated education surfaces, marketplace (Layer 4+ — deferred)

## Open Decisions

Items blocked on a human decision; close before `status: locked`. Each entry names: the decision, who decides, what it blocks, the recommendation.

**1. GitHub org name.** ✅ **CLOSED 2026-05-15.** `tropo-ai` org already exists in Mike's Gmail identity. Decision: keep the existing org; Deliverable #2 is re-framed from "create new org" to "transition org identity to `mike@tropo-ai.com` Workspace, then transfer the repo into it."

**2. Repo migration path.** ✅ **CLOSED 2026-05-15.** Transfer-in-place via GitHub Settings → Transfer ownership. Preserves history; GitHub auto-sets-up redirects from `mikemaziarz/tropo-ai` → `tropo-ai/tropo-ai` so external links keep working. Current state: repo lives at `github.com/mikemaziarz/tropo-ai`; target: `github.com/tropo-ai/tropo-ai`.

**3. Vercel plan tier.** ✅ **CLOSED 2026-05-15.** Hobby (free) for v1. Tropo is pre-revenue and pre-incorporation; the site is a marketing surface + free download, not a commercial operation. Vercel Hobby's "non-commercial" restriction targets active commercial activity (ecommerce, paywalls, revenue funnels), not entity status. **Upgrade triggers** — flip to Pro the moment any of these fires: (a) revenue starts (subscriptions, paid features, money flows), (b) team grows past 1 human + AI agents, (c) Vercel Hobby limits hit (100GB bandwidth or 100K function invocations / month), (d) any commercial feature ships (paid hosted studios in Layer 3+). Upgrade is one-click in Vercel dashboard; no migration cost.

**4. Supabase plan tier.** ✅ **CLOSED 2026-05-15.** Free for v1. Supabase Free has no commercial-use restriction; only usage-limited. **Upgrade triggers** — flip to Pro ($25/mo) when any of these fires: (a) auto-pause cold-start (~30s after 7 days zero DB activity) becomes a real UX issue, (b) DB approaches 500MB, (c) file storage approaches 1GB (relevant when hosted studios start in Layer 2-3), (d) MAU crosses 50K, (e) Point-in-Time Recovery becomes important for real user data. Upgrade is one-click in Supabase dashboard; no migration cost.

**5. Cloudflare account state.** ✅ **CLOSED 2026-05-15.** `tropo-ai.com` is currently on a Cloudflare account tied to Mike's personal Gmail. **Resolution:** once Workspace lands (Deliverable #1), migrate the domain from the personal-Gmail Cloudflare account to a new Cloudflare account at `mike@tropo-ai.com`. Cloudflare's "Move domain to a different account" feature handles this without DNS interruption — initiate transfer from source account, accept on destination; DNS records carry over. Net: Tropo Workspace identity controls the domain.

**6. Auth model details for Layer 0 email capture.** ✅ **CLOSED 2026-05-15.** Three sub-resolutions:
- **(a) Data model:** flat `emails` table in Supabase. Columns: `id`, `email`, `name` (nullable), `source` (enum: `agent-builder` / `newsletter`), `newsletter_opt_in` (boolean), `captured_at` (timestamp), `metadata` (JSONB for source-specific fields, e.g., agent name from the builder wizard). Simple and queryable; migrates cleanly to a normalized users+events model when Layer-1 accounts arrive.
- **(b) Newsletter opt-in default:** opt-in required (checkbox unchecked by default; user must affirmatively check). GDPR + CAN-SPAM clean.
- **(c) Email verification:** none at v1. Format validation + uniqueness check on submit; captured emails go directly into the table. Verification gets added at Layer 1 when real accounts arrive — that's the natural gating moment.

**7. Vault UID scheme for the public ecosystem.** ✅ **CLOSED 2026-05-15 (deferred).** Vault registration is out of v1 scope (see Out-of-scope section); this architectural call is deferred until a future project commissions the vault-registration architecture spec. When that project lands, the UID-scheme decision (keep 8-hex + central registry deduplication vs extend to 12-hex / UUID for vault-root identifiers) gets routed to Argus for review at that time.
