## Intent

The activation-root project for all setup work that must complete before [web-pipeline (7b2e94c1)](7b2e94c1.md) can fire its first real per-deploy run against the live tropo-ai.com host. Holds: account + infrastructure setup, founder-ops migration to a Tropo-owned identity, design briefs for Layer-0 surfaces (agent builder, vault registration, blog, downloads), and the first production placeholder deploy that proves the pipeline end-to-end.

Scoped by [tropo-ai.com Launch Foundations — Project Plan (6d4f7a82)](6d4f7a82.md).

**Predecessor:** none. First foundational project under web-pipeline.

**Successor pattern:** once this project closes (`stage: done`), web-pipeline begins firing per-deploy runs against the live site. Each subsequent deploy is a separate pipeline-run with its own activation-root project (e.g., `web-deploy-N`).

## Why this project exists

The Layer-0 product surfaces (marketing, blog, download, agent builder, vault registration) plus the Layer-1+ growth path (real accounts, hosted studios, collaboration) require substrate choices made up-front: a host that can stream AI responses and run real serverless functions; a database that scales from email-capture to multi-tenant; a Tropo-owned identity for every infrastructure provider (not Mike's personal accounts). This project executes those choices and produces the substrate that web-pipeline's first deploy lands on.

## Closure condition

Project transitions `stage: build → done` when all 11 deliverables in the [project plan (6d4f7a82)](6d4f7a82.md) have shipped per their acceptance criteria. §Lessons gets populated in the plan at that moment.

## Execution Notes

**Deliverable #11 (Cloudflare account migration) deferred — 2026-05-15.** During Phase 1 execution Mike discovered Cloudflare's UI no longer surfaces a simple "Move to Another Account" option for Free-tier registered domains. The plan committed to moving `tropo-ai.com` from Mike's personal-Gmail Cloudflare account to a new account at `mike@tropo-ai.com`. Three options were on the table: (a) add `mike@tropo-ai.com` as a Super Administrator Member on the personal account (pragmatic; doesn't actually migrate), (b) submit a Cloudflare Support ticket for internal account migration, (c) transfer registration to a different registrar entirely. Per Mike's call, Option A taken — `mike@tropo-ai.com` Member access gives full operational control without the support-ticket overhead. Full account separation deferred to a future project (triggered by incorporation, co-founder onboarding, or any concrete identity-isolation need).

This means Deliverable #11's locked acceptance criteria (per plan [6d4f7a82](6d4f7a82.md)) are not met by this project as authored. Resolution paths when the project moves toward `stage: done`: (i) ship #11 via Cloudflare Support ticket before closing, (ii) supersede the plan with looser criteria, (iii) close the project with the deviation noted explicitly in §Lessons of the plan. Decision deferred to that moment; tracking the open question here so it's not lost.

**Architectural note — separate website repo (decided 2026-05-15).** Plan committed to "Vercel team + repo connection" without specifying which repo. During Phase 2 execution we elected to author a **separate repository** for the website (`tropo-ai/website`) rather than deploying the existing platform repo (`tropo-ai/tropo-ai`). Rationale: the platform repo contains engineering substrate (vault KB resolver, Mo Author, internal KB articles, agent-routing logic) that doesn't belong on a public-facing surface; deploying it to `tropo-ai.com` would expose internal references. Separate repo keeps the website lean and the platform private. Platform repo stays put; will eventually deploy to `app.tropo-ai.com` or similar when there's a reason. Argus eyes welcome on this architectural pattern if it ever escalates.
