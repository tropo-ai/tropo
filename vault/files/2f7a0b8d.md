## Context

Every agent's ASA independently reads the same vault files — crew roster, vault map, ADR index, ops digest, announcements — from scratch each time a new generation activates. With five active agents, this means the same forty files are being read by five separate ASA sessions on the same day, each producing overlapping orientation artifacts that cover identical ground.

This redundancy compounds as the crew scales. When Tiphys and Nestor commission, seven ASAs will each independently traverse the same slow-changing content. At ten agents, the problem is not just inefficiency — it's also coherence. Each ASA produces a slightly different picture of the same vault state, depending on when it runs and what it finds.

The current architecture was designed for correctness, not scale. It is correct: every agent gets a fresh, verified picture. But the cost model breaks as the crew grows.

Additionally, the context budget sprint (ADR-012 extension, March 24) proved that the per-agent ASA can be made lean — A5 booted at 3% of context. The sprint's success creates the condition for the next optimization: if agents are reading shared artifacts instead of raw vault files, the ASA can be even leaner, and the shared content is produced once rather than N times.

Mike originated this concept: one daily agent reads the slow-changing content and produces pre-built orientation artifacts. Every individual ASA reads those artifacts instead of the raw vault. The "build your own ASA" model: each agent's ASA selects from the shared artifact menu and adds only what's agent-specific on top.

This is Part 2 of the context budget work. It is also explicitly a self-healing and resilience mechanism (Principles 5 and 6): fresh orientation data runs on a schedule, catches drift automatically, and keeps every agent's orientation accurate without human intervention.

---

## Decision

Establish a **Shared Orientation Layer** — a scheduled agent that runs twice daily and produces a set of pre-built orientation artifacts consumed by all agent ASAs.

### The Shared Agent

A new standing operational process: `agents/operations/shared-asa/`

Runs on a twice-daily Cowork scheduled task (6:00 AM ET, 12:00 PM ET). Each run reads slow-changing vault content and writes five shared artifacts to `shared/orientation/`. Each artifact carries a freshness header so agents know when it was generated.

### The Five Shared Artifacts

Stored at `shared/orientation/`:

| Artifact | Source | Max Length | Refresh Trigger |
|---|---|---|---|
| `crew-roster.md` | `crew-roster.md` (Orpheus-maintained) | ~600 words | Crew change |
| `vault-map.md` | `ls -R Argo-OS/`, `modus/`, `context/` | ~500 words | Directory change |
| `adr-index.md` | All files in `decisions/` | ~400 words | New ADR |
| `ops-digest.md` | `channels/ops.md` (last 7 days) | ~600 words | Daily |
| `announcements.md` | Crew brief announcements + priorities | ~300 words | Daily |

Each artifact begins with:
```
*Generated: [YYYY-MM-DD HH:MM ET] | Next scheduled: [HH:MM ET] | Source: [file path]*
```

### The "Build Your Own ASA" Model

Each agent's ASA activation file gains a new **Shared Artifact Config** section that replaces the corresponding raw file reads:

```markdown
## Shared Artifact Config
Load from shared/orientation/ instead of direct reads:
- crew-roster (replaces: direct manifest/roster read)
- vault-map (replaces: ls -R directory traversal)
- adr-index (replaces: reading all decisions/ files individually)
- ops-digest (replaces: full ops.md read)
- announcements (replaces: crew brief announcements section)

Freshness threshold: 12 hours. If any artifact exceeds this, note staleness
and fall back to direct read for that artifact only.

## Agent-Specific Reads (always fresh — not shareable)
- channels/[agent]-[partner].md (working channels change throughout day)
- agents/[name]/transfers/living-transfer.md
- agents/[name]/[name]-status.md
[any agent-specific docs]

## Custom Tasks
[any additional reads or analysis specific to this role]
```

The agent's ASA reads the shared artifacts (fast — already compressed) and only does fresh reads for the agent-specific material. Total token cost of shared artifact reads: ~400 tokens. Down from ~8,000 tokens for equivalent raw file reads.

### Fallback Protocol

If any shared artifact is missing or older than 12 hours at the time of ASA run:
1. Log a WARNING in the ASA activation report
2. Fall back to direct read for that artifact
3. Post a note to ops.md if the shared-asa appears to have missed a scheduled run

### Scope Boundary

The shared-asa is a sensor-producer hybrid. It reads vault content and writes five output files to `shared/orientation/`. It does not: post to channels, modify any agent's files, make decisions, or read agent-specific files (workspace, working channels, living transfers). Its write scope is strictly limited to the `shared/orientation/` directory.

---

## Rationale

**Why twice daily and not once?** Once daily creates a 23-hour freshness window. An agent activating at 5 PM on a vault that changed significantly at 9 AM gets stale data. Twice daily (6 AM, 12 PM) reduces the maximum staleness to ~12 hours. Ops content (ops-digest, announcements) changes daily; vault structure and ADRs change less frequently. The 12-hour window is acceptable for stable content and tolerable for daily content given the fallback protocol.

**Why shared artifacts instead of per-agent caching?** Per-agent caching produces N slightly-different snapshots of the same vault state, one per ASA run. The shared layer produces one canonical snapshot that all agents read. This is more coherent, easier to audit, and eliminates the question "why did Argus's ASA see something Vela's ASA didn't?"

**Why five artifacts and not more?** These five cover the content that every agent reads at boot regardless of role. Agent-specific content (channels, transfers, role-specific design docs) stays per-agent because it varies by role and changes intraday. Adding more to the shared layer would require the shared-asa to read agent-specific content, which would violate workspace boundaries (Principle 3).

**Why this is self-healing infrastructure (Principles 5 and 6):** The shared-asa is a standing detection-and-maintenance process. By running on a schedule and refreshing orientation state, it prevents orientation drift — the condition where agents make decisions based on stale information about the vault's state. This is the same immune system model as the standing auditors, applied to the knowledge layer. The system detects staleness automatically and corrects it without human intervention.

**Connection to v3 Pillar 2 (Self-Healing):** This fits the biological cascade model. The shared-asa detects drift in orientation artifacts (Detect), produces fresh versions (Repair), and the freshness header allows any agent to verify currency (Verify). It is a specialized healing agent for the orientation layer.

**Connection to the context budget sprint:** The sprint proved the tiered reading model works (A5 at 3%). The shared layer extends the optimization — instead of agents reading compressed summaries produced by their own ASA, they read shared summaries produced once. The protocol remains the same; the mechanism becomes more efficient.

## Consequences

**What gets better:**
- Per-agent ASA boot cost drops further (~400 tokens for shared artifacts vs. ~8,000 for equivalent raw reads)
- Vault state is coherent across all agents — everyone reads the same snapshot
- Orientation quality improves for stable content (vault map, ADRs) — shared-asa can do more thorough compression than per-agent ASA
- Crew scaling cost is sublinear: adding a new agent adds one "shared artifact config" block, not a full raw-vault traversal

**What gets harder:**
- New dependency: agent ASAs now depend on the shared-asa having run. If the scheduled task fails, agents fall back to direct reads (fallback protocol handles this, but introduces a new failure mode to monitor).
- Shared-asa is a new scheduled agent requiring maintenance (Vela adds to scheduled agent registry; governance-validator checks it's running).

**What changes:**
- New directory: `shared/orientation/` with README
- New agent: `agents/operations/shared-asa/` with `activate.md`
- New scheduled task: twice-daily Cowork scheduler entry
- Each agent's ASA activation file: gains Shared Artifact Config section (replaces equivalent raw reads)
- ASA template: gains Shared Artifact Config section as standard
- `context/working-with-mike.md`: new shared artifact added to the menu (all agents load this; individual activation file "WHO MIKE IS" sections become pointers to it)

**Rollout sequence:**
1. Mike locks ADR-017
2. Argus creates `shared/orientation/` directory and `agents/operations/shared-asa/activate.md`
3. Scheduled task activated (6 AM, 12 PM ET)
4. Shared-asa runs first time, produces five artifacts
5. Vela updates ASA template with Shared Artifact Config section
6. Orpheus and Vela update their ASA activation files to reference shared artifacts (test agents)
7. Argus-ASA updated as confirmation of pattern
8. Extend to Metis, Talos, Silas as needed

## Relationship to Existing ADRs

- **ADR-012 (ASA Protocol):** This ADR extends ADR-012. The five ASA functions remain unchanged. The reading list is modified: shared artifacts replace equivalent raw reads. The shared-asa is a new supporting process, not a replacement for individual ASAs.
- **ADR-013 (Context Budget Ceilings):** The shared artifact files need ceiling governance. Each artifact file has a hard ceiling (see artifact table above). The shared-asa must stay within those ceilings or compress further.
- **ADR-015 (YAML Front-Matter, candidate):** When locked, the shared artifact freshness header should conform to the YAML standard. The "build your own" config section is a natural candidate for YAML front-matter format.
