## What this subsystem holds

This subsystem is the home of everything that makes Tropo (the brand-as-agent) work in user Studios:

**The agent itself.** A permanent executive-class agent named "Tropo" — the brand walking through the door. Not a persona; not a chat-assistant character; the OS the user installed, arriving as an agent in their Studio. Per-Studio lineage (T1, T2, ... T_n); kernel-managed identity (charter, soul, activation, agent-boot.extension); user-owned lived state (status, generation log, transfers, memory).

**The first-user experience.** A four-phase welcome flow (Phase 1 greet + first agent → Phase 2 optional learn-loop → Phase 3 shipped examples → Phase 4 real work in) authored as a Studio-level playbook. Replaces the v1.4-era concierge path concepts (welcome.playbook v0.6 + tour-tropo + personal-chief-of-staff, all RECYCLED at v1.17.0 ship).

**The governance contract.** Kernel-managed-vs-user-owned split declared in `agents/tropo/CAPSULE.md`. Identity ships from HQ; relationships stay with the user. The update pipeline reads the contract and behaves accordingly. Brand evolves; user-Tropo continuity persists.

**The release-news substrate.** `briefing-package/00-index.md` + `briefing-package/current-release-notes.md` template + lifecycle pattern (`release-history/v<N>-notes.md` archive on supersession). Each Tropo release ships news content via this substrate; Tropo's release-liaison role delivers via consistent pattern.

**The harness-portable scheduling.** L1 portable schedule registry (`tropo-schedule.yaml`) + opt-in harness adapters (Claude Code shipped at v1.17.0; other harnesses get adapters as needed; non-adapter harnesses use universal L1 boot-time check fallback).

**The federation foundation.** Future-cycle architectural thesis (Studio Registration + Studio-Prefixed UIDs at [`7cac6473`](7cac6473.md)) for v2.x+ activation. Substrate for the marketplace + federation half of the strategic loop.

---

## Architectural property: HQ push, Studio pull

**At v1.17.0 ship, this subsystem is one-way by design.** HQ pushes (releases, news, kernel-managed file refreshes). Studios pull on next boot or via update-detection trigger. Nothing flows back to HQ. Not generation counts. Not memory contents. Not what users are working on. Not how often Tropo runs.

This is a positive design property, not a temporary gap. It composes with:

- **Brand story** — your Studio is yours; nothing leaves
- **Bounded-verification moat** — the human is the verifier; HQ has no way to "see into" a Studio to validate it
- **Stranger-encounter expectations** — the user downloaded a zip; the zip doesn't phone home

Eventual upward flow (marketplace publishing via explicit user action; federation sharing via explicit user action; opt-in crash reports) is always *user-initiated and explicit* — never telemetric, never automatic, never assumed. The federation foundation thesis at `7cac6473` is the substrate for those future moves; not v1.17.0 scope.

The v1.17.0 walk produced multi-cycle architectural locks that shape v1.18.0+ chain. This subsystem hub composes with all five (per [v1.17.0 brief §12 (`9fc8a149`)](9fc8a149.md)):

- **Lock B — Five exception classes.** `agents/tropo/` is the entry-point folder-shape exception. Thin-loader pattern preserves boot reliability; identity content migrates to vault/files/.

- **Lock D — AGENTS.md retires for converged classes.** This subsystem's `agents/tropo/` AGENTS.md (none today) would only exist if folder-shape persists; convergence Phase 2 retires the need.
