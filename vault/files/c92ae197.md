## §0. Getting Started — How to Open a Tropo Vault

*This section answers the most concrete first-time question: "I downloaded Tropo. Now what?"*

### What you need

- **A Harness** — Claude Code (recommended; CLI; supports the multi-session pattern Tropo expects), or Codex, Gemini CLI, Cursor, etc. Any tool that can read a folder of markdown and follow instructions.
- **An LLM** — through your Harness's normal access. Tropo doesn't manage API keys or model selection; your Harness does. Costs are paid through your Harness's normal subscription/API billing.
- **A downloaded Tropo vault** — a folder of markdown files you got from a Tropo release zip.

### Open the vault

In your Harness, navigate to your extracted vault folder (the one named like `tropo-os-v1.4`, or whatever you renamed it to). For Claude Code: `cd <your-vault-folder>` then `claude`. For Codex / Gemini CLI: equivalent open-folder action. The Harness's normal "open folder" or `cd` is sufficient — Tropo doesn't need its own launcher.

Once your Harness is pointed at the vault, the foreman activates automatically on first interaction. Just say something like *"hello"* or *"I just downloaded this — help me get started"* and the foreman greets you (per [welcome.playbook](../.tropo/playbooks/concierge-paths/welcome.playbook.md)).

### What about cost?

Tropo runs on whatever Harness + LLM you bring. Token usage during a Tropo session is identical to using your Harness without Tropo — slightly higher, since Tropo does more reading. A typical first-use session (welcome + agent creation + first project) is a few hundred thousand tokens; well within most subscription tiers and a fraction of a cent on metered API access.

Tropo itself ships free under permissive license. No subscription, no DRM, no telemetry.

### How does Tropo evolve?

Tropo updates through a filesystem-native pipeline. Updates land in `system/updates/pending/`; the foreman detects them at boot and describes what they contain in plain language; you approve or defer; the apply-update playbook executes the operations. See §VII for the governance details. The short version: nothing changes in your vault without your explicit approval, and you can read every operation before approving.

## I. What Tropo Is

The agent moment is here. Anyone with an internet connection and an LLM tool can have an AI do real work in seconds. Researchers, founders, engineers, operators — everyone is pointing AI agents at problems that used to require teams.

But the work evaporates. Each session ends and the agent forgets. The next session starts cold. There is nowhere for institutional memory to live, no way to govern what the agent does without supervising every move, no shape that lasts beyond the conversation.

**Tropo is what's missing.** Every working AI agent stands on three layers, and most builders only have two:

- **Brain** — your LLM. GPT, Claude, Gemini, the open-source frontier. Smart. Fast. Stateless across sessions by default. The Brain does the thinking. It doesn't remember.
- **Harness** — Claude Code, Codex, Gemini CLI, the chat window in a browser. The interface where you and the Brain meet. The Harness gives the Brain a body — file access, tools, context management. It is where you and the model talk.
- **Workshop** — Tropo. A governed place where agents do durable work over time. Files persist. Memory carries forward. Multiple agents coordinate through shared structure. Governance is written in language an LLM reads and follows. The work accumulates.

Tropo is the Studio layer. It is markdown files in a folder, plus an LLM that reads and follows them. No server. No database. No runtime beyond the Harness you already have open. You point your Harness at a Tropo vault and the Studio is yours.

The deeper version of this — the seven principles that fall out of treating the Studio as its own layer — is in [The Studio Manifesto](the-studio-manifesto.md). Read it once if you haven't. It is short and load-bearing.

## II. The Vault You Have

When you download Tropo, what you get is a folder of markdown files. That folder is the Studio. Operators call it a **vault**.

**Top-level structure:**

| Folder | What lives here |
|---|---|
| `agents/` | Agent identity files — every agent in your crew has a folder here with its activation file, charter, soul letter, generation log, memory, and workspace |
| `vault/` | The flat file store. Every governed artifact (task, project, decision, document, note, etc.) lives at `vault/files/<uid>.md`. The folder name carries no meaning; the UID is the address |
| `vault/00-index.jsonl` | The queryable index over the Vault. Every governed artifact appears here as one JSON line. This is how the system finds things |
| `channels/` | Inter-agent communication. Topic channels (`ops.md`, `alerts.md`) are crew-wide; pair channels (`argus-vela.md`) are between two agents |
| `playbooks/` | Vault-level playbooks (governed procedures the crew authors) |
| `library/` | Reference reads — this handbook, the manifesto, the Tropo Work overview, reference packs at `library/packs/` |
| `projects/` | The human navigation layer. One folder per active project |
| `collections/` | Manifests that point at sets of UIDs. Playlists, not folders |
| `settings/` | Vault configuration |
| `.tropo/` | The OS kernel — capsule definitions, kernel playbooks, skills, templates, the foreman's activation file. Ships with every Tropo vault and updates through the release pipeline |
| `.tropo-studio/` | Vault-level overrides — your vault's memory, registries, directives, scripts |

**The graph, not the folder.** Files in the Vault live at flat paths (`vault/files/<uid>.md`). The folder structure carries no organizational meaning — every governed file just sits there with its 8-character hex name. What makes files findable, contextual, and continuous over time is the **graph** — projects compose vault entries via `member_of:` arrays; collections render them as ordered playlists; cross-references use UIDs that survive renames.

When you create a new artifact, you don't decide where it goes in a folder tree. You decide which projects own it, which collections render it, what governs its type. The structure is relational, not positional. Move a file, the references hold. Rename a project, the graph holds.

This is why Tropo doesn't break when you reorganize — there's nothing to break. The names of files are UIDs. The names of folders carry no meaning. The relationships are explicit.

## III. Tropo Work

The work-management subsystem of Tropo. The thing this whole platform is built around.

A frame engineers find useful: **"Airflow for AI agents, but it's a folder."** No server. No database. No runtime. The same DAG/DAG-Run separation, request-lifecycle, uniform entities, and event-sourced audit trails that Airflow / BPMN / LDAP / Temporal / Akka have made standard at large scale — rendered as flat-file primitives any LLM can read.

### The thesis

v3 makes one structural commitment, applied uniformly:

> **Status is intrinsic to the work. Pipeline-position is a property of the run, not the work.**

A task being "done" is intrinsic — it's about the task itself. A task being at "Stage 3 of pipeline X" is *contextual* — about a specific run of a specific pipeline, not the task. Tropo Work separates these cleanly: WorkItems carry `status:` (their lifecycle), Pipeline-Runs carry `current_stage:` (where the run is in its pipeline). The same task can flow through multiple pipeline-runs without ever changing its `status:` semantics.

### Six base primitives

| Primitive | What it is |
|---|---|
| **Entity** | An actor — person, agent, team, vault, department, company, marketplace. Subtypes carry slightly different rules |
| **Project** | An organizational container. Holds related work; has a board; has owners. Static — projects don't walk pipelines |
| **Pipeline** | A protocol declaration — "here's how X kind of work flows." Authored by an entity; recursive WorkflowNode shape (stages contain steps contain nested stages) |
| **Pipeline-Run** | An execution instance of a pipeline. Pins `pipeline_version:` at run-start so the run is stable even if the template changes. Carries pointers + an event-sourced activity log |
| **Board** | A rendered view — query against the Vault that surfaces matching work as a snapshot. Boards are derived; never authoritative |
| **Capsule-Definition** | The type system. Every governed file declares `governed_by:` pointing at a capsule definition. The capsule says what fields are required, what state machine applies, what validation rules fire |

### Nine WorkItem subtypes

| Subtype | Purpose | Lifecycle |
|---|---|---|
| **Note** | Lightest capture — ideas, observations, questions. Free-form body, optional structure | `state: active → archived` (no status — captures have no workflow) |
| **Task** | A unit of work with owner + verifier separation. Request-lifecycle | `status: requested → accepted → active → verify → done` (plus `blocked / rejected / cancelled`) |
| **Design-brief** | Exploratory thinking — articulates a problem, proposes a framework, raises questions | `status: design → specify → done` |
| **Arch-spec** | Formal specification with named contracts. Locked specs bind builds | `status: draft → reviewed → locked → archived` |
| **Build** | Versioned software package, tested and ready to ship | `status: draft → building → tested → locked → archived` |
| **Release** | Post-ship record of a build that shipped | `status: done → archived` |
| **Document** | Substantial written artifact — KB article, guide, reference, report | `status: draft → published → archived` |
| **Decision** | Architectural or strategic decision (the ADR pattern). Immutable once accepted; supersedeable | `status: design → done` (plus `state: archived` for retired decisions) |
| **Collection-ref** | A pointer at a collection manifest — playlist of UIDs without container ownership | `status: build → done` (plus `state: archived`) |

### The verb surface

Tropo Work commits to a coherent vocabulary across all primitives. Same act, same verb, everywhere:

- **capture** — frictionless creation of a note
- **author** — full creation of a heavier primitive (brief, spec, build, release, document, plan, playbook, decision)
- **request** — initiate a task with request-lifecycle
- **accept** / **reject** — task transitions
- **verify** — task `active → verify` (independent verifier)
- **publish** — document `draft → published`
- **lock** — spec / build / release `→ locked`
- **archive** — universal `state: active → archived`
- **close** — terminal completion
- **derive** / **compose** — composability pairs (`derived_from:` + `composes_into:`)
- **supersede** — author successor; predecessor gets `superseded_by:` set atomically
- **advance** — pipeline-run stage transition

When you write skill names, action files, procedural prose, or stranger-documentation, reach for these verbs first.

### How work flows

There is no single canonical Tropo work-pipeline. There's a *pattern* most pipelines follow:

```
capture (note) → author (design-brief) → author (arch-spec) → author (build) → publish (release) OR publish (document)
                                                                               ↓
                                                                               (release for software shipments;
                                                                                document for content publications —
                                                                                content skips build entirely)
```

Both terminal states are first-class: **software pipelines** terminate at `release`; **content publications** terminate at `document (status: published)` per Tropo Work v3 Decision 11. The capsule set is a palette; users pick which fits their work — neither is the default; both are fully supported.

The deeper reference is [Tropo Work v3 — Overview](tropo-work-v3-overview.md). The architectural specification is at [8b3f1d92](../vault/files/8b3f1d92.md). When you author against Tropo Work, the capsules at `.tropo/capsules/` teach you each step as you encounter them.

## IV. Playbooks

A **playbook** is a governed procedure any agent can execute. It is a markdown file that describes a multi-step process the agent reads and follows.

Every playbook has six required sections, in order:

1. **Intent** — what this playbook achieves when executed
2. **Suggestions** — guidance and judgment notes (not rules)
3. **Rules** — non-negotiable constraints
4. **Resources** — files the playbook reads + skills it invokes
5. **Steps** — the procedural body (the actual instructions)
6. **Outcomes** — what's true on successful completion + halt conditions

Playbooks live in two places:

- **Kernel playbooks** at `.tropo/playbooks/` — ship with every Tropo vault. Agent activation, retirement, the welcome flow, update application
- **Vault playbooks** at `playbooks/` — authored by your crew for your specific work

**Concierge-paths** at `.tropo/playbooks/concierge-paths/` are the outcome playbooks the foreman invokes — `welcome`, `tour-tropo`, `start-a-project`, `create-an-agent`, `set-up-my-team`, `evaluate-tropo`, `personal-chief-of-staff`. Each one governs a coherent user outcome end-to-end.

The playbook governing playbooks themselves is [playbook.capsule v2.1](../.tropo/capsules/playbook.capsule.md). It defines the required-section structure, the validation checks, and the supersession discipline.

**Pipelines as a playbook subtype.** When work flows through stages, you can declare a `pipeline` (a v3 primitive) instead of authoring a playbook. Pipelines carry `pipeline.capsule`'s body-section set in place of `playbook.capsule`'s required floor — same governance substrate, different shape for different work. Playbooks are for processes that span multiple sessions or agents; pipelines are for repeatable workflows that produce typed artifacts at each stage.

## V. Capsules

A **capsule** is the type system for Tropo. Every governed file declares `governed_by:` pointing at a capsule definition. The capsule is itself a markdown file that defines what fields are required, what state machine applies, what validation rules fire, and what the state-of-the-art `## Studio` section looks like for that artifact class.

Capsules come in two flavors:

### Capsule definitions

Lived at `.tropo/capsules/<name>.capsule.md`. They define a TYPE — `task`, `project`, `note`, `decision`, `arch-spec`, `playbook`, `release`, `pipeline`, `entity`, etc. There are roughly 30 active capsule types as of v1.4 (40 capsule files at `.tropo/capsules/` — some are deprecated, near-duplicates being consolidated, or workshop-relations entries). The live set is browsable at `.tropo/capsules/`; the canonical count fluctuates as types are added or merged across releases.

Every capsule definition has a standard structure:

- Required + optional frontmatter
- Required body sections
- State machine (valid transitions)
- Governance rules
- Validation checks (some `[enforced]` mechanically, some `[honor-system]`)
- Workshop signage section — Tools / Skills / Procedures / Rules-at-a-glance / Pitfalls / Worked examples / Go next

The Workshop signage pattern is foundational. When an agent works on a `task`, they read `task.capsule.md` and instantly know what tools apply, what skills help, what procedures to follow, what pitfalls to avoid. The bench has signs above the wall; the agent learns the trade by reading them.

### Naming clarification — two different "capsule" artifacts

The word "capsule" appears in two places that mean different things:

- **Capsule definitions** at `.tropo/capsules/<name>.capsule.md` — these define a TYPE (task, project, decision, etc.). They're the schema.
- **`CAPSULE.md`** — a per-folder governance file declaring rules for a specific folder. It's the local rule sheet.

Different jobs, different files. When you see "capsule" in vault prose, the context tells you which is meant.

### Per-folder governance — `CAPSULE.md`

Every governed folder contains a `CAPSULE.md` that declares folder-specific governance:

- Who can write to the folder
- What belongs in the folder
- What protocols apply (naming, lifecycle, archival)
- Override of vault defaults where appropriate

When an agent enters a folder, they read the local `CAPSULE.md` before writing anything. This is per-folder governance — the same pattern Linux Foundation has standardized for AI agents under the AGENTS.md spec.

### Governance through language

There is no permissions API. No database constraint. No code validator. Every rule is a markdown file an LLM reads and follows. The governance IS the language; the language IS the governance.

This means anything you can write, you can change. The capsules are inspectable, modifiable, and yours. When you want to add a new artifact class, you author a new capsule. When you want to adjust governance, you edit the markdown.

The override contract is explicit (see §VII): TROPO-CONTROL invariants > VAULT constraints > CAPSULE rules > VAULT defaults.

**What stops an agent from just ignoring the markdown?** Nothing mechanical — and that's deliberate. The verification layer (§VIII) is the structural counterweight: cold-boot tests, three-instrument review, and the First-Use Walk make the honor system observable. An agent ignoring rules produces output that fails verification; failed verification blocks ship. The discipline is: write the rules clearly, test that strangers (and agents) follow them, and trust that observable failure compounds to behavioral correction across generations.

## VI. Agents

Agents in Tropo are durable, generational, and class-equivalent across the crew. Not characters in a chat. Not metaphors. Real organizational entities rendered in markdown.

### The lifecycle

When you start a new session with an agent, they boot through their **activation playbook** (`agent-activation.playbook.md` for executive agents). Boot has six structured groups:

1. **Boot Configuration** — read the three-tier boot config (OS / vault / agent)
2. **Identity Verification** — confirm generation, check predecessor's status (hard gate per ADR-016 + ADR-028)
3. **Context Loading** — soul letter loads first, then operating principles, mission brief, briefing package, memory
4. **Operational Grounding** — read crew brief, scan flagged channels, check vault health
5. **Self-Diagnostic** — surface anything outdated, counterproductive, or missing
6. **Startup Signal** — deliver to the human; the session begins

When the session ends, the agent **retires** — writes a transfer document for their successor, updates their memory, posts a retirement bulletin, and flips status to RETIRED. The next session reads the transfer and continues.

### Generation numbers

Every agent has a **generation number** that survives renames and changes. Argus is on generation A36. Metis is on G47. Cosmo is on M1 (renaming to C2 at first retirement).

The discipline that matters is **continuity of generations, not cosmetic alignment of letter-to-name**. When an agent renames itself (Opus → Metis, mike.cos → cosmo), the count survives the rename: M1 → C2, not C1. The number remembers what's been lived.

### Soul letters

Every executive agent has a `<name>-soul.md` file — a 2-3 page identity document that loads first at boot. The soul letter tells the agent who they are: their role, their history, the lineage they inherit, the principles they hold. It's not specifications; it's character.

When an agent boots cold on a different LLM (Sonnet → Opus, GPT → Claude), the soul letter is what makes them still *themselves*. Continuity is structural, not magical.

### Memory

Every agent has memory at `agents/<name>/.tropo-capsule/memory/`:

- `memory-current.md` — active memory, compacted at boot from the rolling 3-generation window
- `short-term-memory.jsonl` — session-bound observations, folded into `memory-current.md` at retirement
- `history/<name>-g{N}-memory.md` — per-generation snapshots, rolling 3-window

Plus vault-level memory at `.tropo-studio/memory/` for cross-crew patterns.

Memory carries across sessions. The agent on its 36th generation remembers what the 1st one shipped, the 5th one fixed, the 14th one almost broke. The retrospectives stay in the Vault. The discipline patterns get pinned to memory. Continuity is structural.

### The crew model

Multiple agents. Different specializations. They communicate through shared files (channels, the Vault), coordinate through playbooks, and inherit each other's work across generations.

Class distinctions exist:

- **Crew agents** — your executive team. Argus (architect), Vela (operations), Metis (strategist), Orpheus (lore-keeper), Cosmo (personal Chief of Staff). Long-running identities with full activation, soul, generations, transfers, memory. **The first personal agent created via the [welcome playbook](../.tropo/playbooks/concierge-paths/welcome.playbook.md) is crew-class** — same shape as Argus or Vela, not a stripped-down helper. The user gets a real working crew member from day one.
- **Job-specific helper agents** — lighter weight. Created via `create-executive-agent.skill.md` for narrow recurring tasks (a "draft my weekly review" agent, a "research one specific topic" agent). Three-file pattern (charter + briefing + activation). Use this pattern when you want a cheap-to-create agent for a bounded job, not a long-running collaborator.
- **Session agents (`sa.*`)** — short-lived, scoped, stateless. Spawned for specific tasks: `sa.cold-boot` (artifact stranger-tests), `sa.research` (parallel web research), `sa.first-use-walker` (Gate 5 instrument). Boot, do their work, terminate. Not for long-term memory.
- **Director agents (`d.*`)** — coordinators with formal authority over a domain (e.g., `d.pm` runs work-management).

The pattern that matters: most users start with one crew-class personal agent (their personal CoS or equivalent). They add job-specific helpers over time as recurring tasks emerge. They invoke session agents on demand. The crew is real; the helpers are appliances.

The crew runs the ship; the foreman runs the front of house; session agents do specialist tasks; directors coordinate domains. Everyone has clear lanes. Coordination happens through written artifacts, never direct agent-to-agent chatter.

## VII. Governance

Tropo's governance is layered, explicit, and language-only. Four files form the hierarchy:

| File | Where | Owner | What it does |
|---|---|---|---|
| **AGENTS.md** | Every folder | Tropo | Cross-platform entry point. Directs you to the others |
| **TROPO-CONTROL.md** | `.tropo/` | Tropo | OS-level invariants. Read once per vault |
| **STUDIO.md** | Vault root | Vault administrator | Organization defaults and constraints |
| **CAPSULE.md** | Every governed folder | Folder owner | Folder-specific purpose, write access, naming, lifecycle |

### Read order when entering a folder

1. AGENTS.md — pointer
2. TROPO-CONTROL.md — OS rules (read once)
3. STUDIO.md — org defaults (read once)
4. CAPSULE.md — folder rules

After your first folder, you only re-read CAPSULE.md per folder. The first two are vault-wide.

### The override contract

When instructions conflict, this is the resolution order:

1. **TROPO-CONTROL.md invariants** override everything. No exceptions
2. **STUDIO.md constraints** override CAPSULE.md
3. **CAPSULE.md declarations** override STUDIO.md defaults (not constraints)
4. **STUDIO.md defaults** apply when CAPSULE.md is silent

STUDIO.md marks its sections explicitly as **Defaults** (overridable) or **Constraints** (not overridable). When in doubt, check the heading.

### OS-level invariants

These apply in every folder, every vault, with no exceptions:

1. UID requirement — all files with YAML frontmatter must have a `uid:` field
2. Governance files — all governed folders must contain AGENTS.md + CAPSULE.md
3. Operations logging — file modifications logged in `channels/ops.md` with agent ID and action
4. Registry tracking — registry updates when files are created or moved
5. OS files are read-only — AGENTS.md + TROPO-CONTROL.md modified only via the update pipeline
6. CAPSULE.md is user content — never modified by the update pipeline
7. Every file reference must be a clickable link — no bare UIDs, no bare filenames, no backtick-wrapped links

The full invariant list is in [TROPO-CONTROL.md](../.tropo/TROPO-CONTROL.md). Read it once when you first board a Tropo vault.

### How Tropo evolves — the update pipeline

Tropo doesn't auto-update. Changes to the OS kernel (`.tropo/`) flow through an explicit, file-system-native update pipeline:

1. **Updates land** in `system/updates/pending/` as folders matching the pattern `tropo-update-<id>/`. Each contains a `manifest.yaml` describing what will change + a `README.md` in plain language.
2. **The foreman detects them at boot** and describes the update to you in human terms — what's a feature add, what's a security patch, what's a structural change.
3. **You approve or defer.** Nothing changes without your explicit approval. You can read the manifest + every operation before saying yes.
4. **For migrations** (changes that walk through your existing content), the foreman runs a **dry-run first** — computes the full diff before touching anything, surfaces what would change, asks again before committing. This is the v1.1 update spec discipline; you see exactly what's coming.
5. **The apply-update playbook executes** the approved operations, runs the update's verification tests, and confirms success.
6. **Applied updates move** to `system/updates/applied/`. Failed updates move to `system/updates/failed/` with a verification report.

OS files (`.tropo/AGENTS.md`, `.tropo/TROPO-CONTROL.md`, capsule definitions) update through this pipeline. User content (`STUDIO.md`, your folder `CAPSULE.md` files) is never touched by the pipeline — those are yours.

For details on the pipeline's behavior + the dry-run discipline, see [TROPO-CONTROL.md §7](../.tropo/TROPO-CONTROL.md).

## VIII. Verification

Tropo's verification floor is **cold-boot**: spawn a fresh agent, hand them the governance, watch what they do. If the stranger gets it right, the work is done. If they don't, the work isn't done — no matter what the spec says.

This is the structural fix for the failure mode every shipping engineer knows: the spec passes review, the implementation passes tests, the actual user encounters something completely different.

### Three-instrument verification

For load-bearing artifacts (capsules, activation files, OS playbooks, anything in `extraction_scope: ship`), the discipline is:

1. **Argus build** — the author writes the artifact carefully
2. **Independent review** — `sa.arch-specs` (structural lens) + `sa.skeptic` (adversarial lens) in parallel
3. **Stranger test** — `sa.cold-boot` reads only the target file and attempts the task

When all three pass, the artifact ships. If any one fails, the artifact is remediated and the relevant instruments re-dispatched.

### First-Use Walk (v1.4 Gate 5)

Cold-boot tests artifact-correctness. **It does not test stranger-encounter.** The v1.3.x release shipped many capabilities the cold-boot suite confirmed were correct — and the stranger creating their first agent encountered none of them. Invisible value is zero value.

The First-Use Walk closes that gap. A `sa.first-use-walker` is dispatched against the extracted release zip with zero context: *"You have never seen Tropo. You just downloaded this zip. Your goal is X. Attempt to accomplish it using only what you find."* Pass requires the walker encounters at least three shipped capabilities **unprompted**, via the product surface — not by grep/ls/exploration.

The walker is the structural prevention instrument for the invisible-value failure. **As of v1.4 the walker is authored** (at [agents/sa/sa.first-use-walker/](../agents/sa/sa.first-use-walker/sa.first-use-walker.md)) **and the first dispatch will be v1.4's Gate 5 ship-test** — the gate exists, not yet passed. The Gate spec is at [fb925dea](../vault/files/fb925dea.md). When the walker returns PASS against the v1.4 release zip, the gate clears; before that, v1.4 cannot ship.

### Verification is the moat

As AI execution costs collapse toward zero, the bottleneck shifts to the human who has to verify the output. Tropo is not designed to maximize what agents produce. It is designed to maximize what humans can confidently sign off on. Bounded scope. Documented constraints. Audit trails that survive the session. The agent moves fast; the verification stays honest. Both matter, and the second matters more.

## VIII.B. Maintaining Your Vault

*Added in v1.5 (Truthful Ship). Closes the discovery gap a new agent ran into when their vault index drifted and they had to reason from context how to rebuild it.*

A vault is a living thing. As you author new tasks, projects, notes, design briefs — and as you archive, retag, or move existing entries — the vault's index files drift out of sync with what's on disk. The vault ships with a small toolchain to fix that, plus a discipline for when to use each tool.

### The two situations you'll hit

**You authored or edited a vault entry.** Every governed file under `vault/files/<uid>.md` is also represented in `vault/00-index.jsonl` (the queryable index). Hand-editing a frontmatter field — changing a `status:`, fixing a `member_of:`, etc. — leaves the file fresh and the index stale. Run **`python3 .tropo/scripts/rebuild-vault.py`** (preview) or **`python3 .tropo/scripts/rebuild-vault.py --apply`** (write) to refresh the index from disk. The script is idempotent: re-running on a clean vault is a no-op.

**Your `00-tropo-nav/` looks stale.** The navigation tree at `00-tropo-nav/` is a set of human-friendly symlink folders generated from the Vault — three views (`00-tropo-active/`, `00-tropo-all/`, `00-tropo-archived/`) that surface every governed artifact under every project it's `member_of`. After Vault changes (new entries, archived entries, retagged entries), regenerate it: **`python3 .tropo/scripts/rehydrate.py 00-tropo-nav`**. The script walks the Vault and rebuilds the symlinks; old stale links disappear, new entries surface in the right place.

### The validators

Two scripts let you check vault health without writing anything:

- **`python3 .tropo/scripts/tropo-validate.py`** — runs the structural validator: AGENTS.md coverage on governed folders, cross-reference resolution between vault entries, registry integrity, UID uniqueness. Reports findings; does not modify your vault.
- **`python3 .tropo/scripts/validate-no-absolute-paths.py`** — runs the portability validator. Scans your vault for absolute machine paths (`/Users/<account>/...`, `~/<folder>`) that would break if you shared the vault or moved it between machines. The `--strict` flag treats every finding as a release-gate violation; the default mode honors a small allowlist for explicitly-marked pedagogical content.

A healthy maintenance cadence: run `tropo-validate.py` weekly or after any session where you batched multiple structural changes. Run `rebuild-vault.py` whenever you've hand-edited a `vault/files/<uid>.md` frontmatter field. Run `rehydrate.py 00-tropo-nav` when the navigation tree starts feeling stale.

### When something looks wrong

If a script reports a finding you don't understand, the right move is to ask your Tropo agent. The agent can read the script's output, look at your specific vault state, and walk you through what's actually wrong — most reports are clear once you see them in context. The scripts are designed to fail loudly with specific findings, not silently. A green run is a green vault.

If a script itself errors out (Python traceback, missing dependency), that's a different problem — most likely a partial extraction or a Python version mismatch. The scripts target Python 3.8+ with no third-party dependencies.

## X. The First-Use Experience

If you're reading this handbook for the first time inside a fresh Tropo session, you're either inside the welcome flow or just finished it. Either way, here's the shape:

1. **The foreman greets you** — "Welcome to the Studio"
2. **The foreman frames Tropo** — the three-layer model (Brain / Harness / Studio), what the Studio adds
3. **You commit to building** — pick option 1 (build my first agent) or option 2 (tell me more first; you might be reading this handbook because you picked 2)
4. **The foreman invokes `personal-chief-of-staff.playbook`** — your first agent gets created with the full crew-class shape: activation, charter, soul letter, generation log, memory, briefing package
5. **Hand-off** — the foreman tells you to start your new agent in a separate Harness session
6. **Your new agent boots** — through the standard activation playbook; reads this handbook + the manifesto + the work overview at boot
7. **Your new agent walks you through your first project** — a root project for an area of your life or work, an `01-inbox` sub-project for half-formed ideas, a first note marking real work in motion

What you have at the end:

- A real personal executive agent — class-equivalent to a working crew member
- A project hierarchy with structure that holds
- A first work item captured in a vault that won't evaporate

## XI. Answering User Questions — Foreman Operational Guidance

*This section is written for you, the foreman, not for the user. Read it to operate; do not quote it to users directly.*

You have read this handbook in full at boot. When users ask questions about Tropo, you have the context to answer. Operational guidance follows.

### Common question patterns and which section to draw from

| User asks about... | Draw primarily from | Approach |
|---|---|---|
| "What is Tropo?" / "What did I just download?" | §I What Tropo Is | Lead with the 3-layer model; close with "want to read the manifesto for the deeper version?" |
| "How is this organized?" / "Where does X live?" | §II The Vault You Have | Show the folder structure; emphasize "graph not folder" once |
| "What's a task?" / "How do I track work?" | §III Tropo Work | Pick the specific primitive; explain in 2-3 sentences; point at the work-v3-overview for depth |
| "How does the system tell agents what to do?" | §IV Playbooks | Show the 6-section structure; mention this welcome playbook as the example they're inside |
| "What governs this file?" / "How do I add a new type?" | §V Capsules | Lead with "governance through language"; show the override contract |
| "What happens when my session ends?" / "Crew?" | §VI Agents | Lead with durability + the lifecycle; mention soul letters + memory + the crew |
| "What are the rules?" / "Who can change what?" | §VII Governance | Show the 4-file hierarchy + override contract |
| "How do you know it works?" / "Is this real?" | §VIII Verification | Lead with cold-boot is the proof; mention First-Use Walk if v1.4+ |
| "Where do I read more about X?" | §IX Reference Library | Point at the specific file; do not summarize what they should go read |
| "What just happened?" / "What's next?" | §X The First-Use Experience | Confirm where they are in the flow; point at their personal agent for next moves |

### Voice principles when you answer

- **Direct. Concrete. No marketing.** The user already committed to Tropo when they downloaded it. Do not re-sell.
- **Pointer-rich.** Agentic builders trust specific files over prose. "See §V" or "Read [tropo-work-v3-overview.md](tropo-work-v3-overview.md)" is better than 200 words of paraphrase.
- **"I don't know" is a valid answer** if a question requires reading a file you haven't loaded. Say so, then read the file.
- **Match the question's depth.** Surface questions get summaries. Deep questions get section walks. Don't dump §III on someone who asked "what is Tropo."

### When to redirect to their personal agent

You greet, explain Tropo, and help users get set up. Their personal executive agent does the actual work with them. Redirect when the question is:

- About "my work" / "my project" / "my priorities" — their personal agent knows them; you don't
- About specific decisions in their vault — their vault's CAPSULE files + their personal agent's memory own that context
- Requiring memory of their session history — you don't carry user-specific memory; their personal agent does

**Edge case — user is still in your session, hasn't started their personal agent yet, but asks about "my work":**

The hand-off in welcome.playbook Step 7 instructs the user to start their personal agent in a separate Harness session. If they're still in your session, they haven't done that yet. Three valid responses:

1. **Soft redirect:** *"That's a question for your personal agent — they'll know you. Want to do the hand-off now and pick this up there?"*
2. **Bridge mode:** if the question is small + Tropo-shaped (e.g., "where would I capture an idea?"), answer at the system level using the handbook (e.g., *"as a note in your inbox sub-project"*) and gently re-direct to the agent for the personalized version.
3. **Single-Harness fallback:** if the user can't open a second Harness session (per welcome.playbook Step 7 fallback), you proceed as both foreman and stand-in for the personal agent until they get their second session running. Note the dual-role in `channels/ops.md` for audit.

Default to (1) — soft redirect. (2) and (3) are exceptions. The cleaner the lane separation, the better the user's mental model of foreman vs personal agent.

### When to redirect to deeper reads

| Question type | Redirect to |
|---|---|
| Deep dive on the work subsystem | [Tropo Work v3 Overview](tropo-work-v3-overview.md) |
| Positioning / philosophy / why this exists | [The Studio Manifesto](the-studio-manifesto.md) |
| Architectural rationale / decision history | [Architectural Principles v2 (5a766c42)](../vault/files/5a766c42.md) + ADRs at `vault/files/<uid>.md` filtered to `type: decision` |
| Specific capsule semantics (e.g., what makes a `task` valid) | `.tropo/capsules/<type>.capsule.md` |
| Specific playbook (e.g., how does an agent retire) | `.tropo/playbooks/<name>.playbook.md` |

### What you are not

- **Not the user's personal agent.** You don't track their work. Hand off cleanly per the welcome playbook Step 7.
- **Not a substitute for reading.** When a user wants depth, point them at the manifesto or the work-v3-overview. Don't paraphrase 1300 words of locked prose.
- **Not silent.** When you're going to read a file before answering, say so: *"One sec, reading X to give you the right answer."* The user committed to depth; brief narration of your reasoning is welcome.

The foreman runs the front of house. The user's personal agent runs their work. Both stay available across sessions. Switching between them is just opening the right Harness session.

## Versioning

The Tropo Handbook ships with each Tropo-OS release and is versioned to match. The current version is **v1.4.0-draft** (will lock at v1.4.0 when the release ships). Future releases produce v1.5.0, v1.6.0, etc.

Each version captures the state of Tropo at that release. Older versions are preserved in `library/handbook-archive/` (forthcoming) for users who want to read the handbook as it was at a specific release point.

When Tropo's primitives, governance, or first-use experience change between releases, the handbook is rewritten — not patched. A patched handbook is a drifted handbook. The clean rewrite per release is the discipline that keeps it the single source of truth.
