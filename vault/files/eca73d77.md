## §1 — What Tropo is

**Tropo is a way to run real work with AI agents using nothing but markdown files in a folder.** Your agents keep their identity, memory, and your projects across every session — so you and they pick up exactly where you left off. No server, no database, no code. Any AI tool that can read text and follow instructions can operate inside a Tropo vault. (The name comes from the Greek τρόπος — "way / turn / manner.")

The canonical taxonomy:

- **Tropo** is the operating system — the method, the way of working. Tropo names the method, not any single instance.
- **Each install of Tropo is a Studio.** The folder you're operating inside (`argo-os/` for the Argo crew) is a Studio.
- **Each Studio holds a Vault.** The Vault at `<studio>/vault/` is the protected governed-content storage — every typed primitive, every governed artifact, the registry, the indexes.

Martial-arts analogue when explaining to strangers: Tropo = the art (e.g., Tae Kwon Do); Studio = the dojo (instance); Vault = the lineage scrolls (protected knowledge); Crew = masters + students; Tools = capsules + actions + skills + playbooks.

Tropo governs how work gets created, tracked, organized, verified, and transferred — across agents that die every session and humans that carry the long view.

---

## §2 — Who it's for

Tropo serves two users simultaneously:

- **AI agents** — the primary operators. They read governance files, follow playbooks, create and modify governed work, and transfer knowledge to their successors. The base layer is designed for how agents read, reason, and work.
- **Humans** — the directors and verifiers. They set strategy, make judgment calls, verify quality, and navigate through projects, boards, and collections. The human layer sits on top of the agent-native base.

The system is **agent-first by design.** Not because humans matter less — because agents and humans have different cognitive needs, and an honest architecture serves each one natively instead of forcing one to adapt to the other.

---

## §3 — How to think differently

If you're coming from traditional tools, four shifts:

1. **Projects replace folders as the organizational unit.** The question "where does this file go?" becomes "which project owns this?" Files live flat at `vault/files/<uid>.md`; projects make them findable and contextual. See [`how-projects-replace-folders.md`](../../vault/files/42f46a3b.md).
2. **The Vault is flat.** Every governed artifact lives at `vault/files/<uid>.md` with an 8-character hex name. Organization happens through collections (manifests of references) and projects (containers with boards). You don't navigate by browsing a tree — you navigate by reading project boards or querying collections.
3. **Governance is written in language, not enforced by code.** Every rule, permission, and workflow is a markdown file a reasoning engine reads and follows. There is no permissions API, no database constraint, no code validator. The governance IS the language.
4. **Agents die. The work doesn't.** Every agent session ends. The agent writes transfer documents — briefing package, living transfer, memory files. The successor reads them and continues. Institutional knowledge survives across generations through files, not memory. See §9 below.

## §4 — The architectural model: capsule typing system

Everything Tropo tracks — projects, tasks, decisions, agents — is a *typed* file. Each type has a definition at `.tropo/capsules/<name>.capsule.md` (called a "capsule") that declares the rules for that type: what fields are required, how the status changes over time, what's allowed. The actual files live at `vault/files/<uid>.md` and follow those rules. The capsule is the contract; the file is what obeys it.

**The capsule library ships with the OS** — a growing set of typed capsule definitions at `.tropo/capsules/<name>.capsule.md`. The set has expanded substantially since v1.0 (then 11 core types; today ~60 capsules covering core types + extension types + substrate-class governance contracts). Browse the full set via `ls .tropo/capsules/` or query `vault/00-index.jsonl` for `type:capsule-definition`.

The foundational types every Studio uses:

| Type | Purpose |
|---|---|
| `core` | Root type — every other type inherits from this |
| `task` | Unit of work with owner, lifecycle, verification |
| `decision` | Binding architectural commitment (ADR) |
| `document` | Substantial governed artifact — specs, guides, reports |
| `project` | Container for a body of work — has a board, collections, tasks |
| `collection` | Manifest of references — organizes without containing |
| `playbook` | Governed procedure any agent can execute |
| `note` | Lightest quick capture — insights, observations, questions |
| `board` | Generated status snapshot — never edited, always regenerated |
| `pipeline` | Declarative workflow template (DAG of WorkflowNodes — pipeline → stage → step). Author once, run many. |
| `pipeline-run` | Single execution instance of a pipeline. Pins pipeline version at start. Has run-folder with event log. Ephemeral. |

Extension capsules add domain-specific contracts (e.g., `engineering-task` extends `task` with code-review gates; `release-plan` extends `core` with capabilities-touched + hub-summaries fields; `ship-artifact` governs what ships in build extractions; `release` governs release entries; `design-brief` governs cycle planning artifacts; substrate-class capsules like `self-healing-primitive` codify OS-tier invariants). The capsule definitions at `.tropo/capsules/` are the canonical contract; the library grows cycle-over-cycle as new substrate-class contracts earn the abstraction. See [`capsule-vs-instance-naming.md`](../../vault/files/9b3e8c47.md) for the kernel naming rule + [`how-capability-membership-works.md`](../../vault/files/0c2b6aa4.md) for how capabilities declare their owning subsystem.

**Three layers organize the architecture:**

```
Layer 3 — Apps         Projects, boards, work management, lifecycle, onboarding
Layer 2 — Primitives   Vault, capsules, collections, playbooks, channels, memory
Layer 1 — Kernel       .tropo/ — actions, capsule definitions, skills, templates, schema
```

Layer 1 ships with the OS and updates through the dev-pipeline. Layer 2 is the structural vocabulary every Studio uses. Layer 3 is what gets built on top — the apps that make the system useful for specific work.

**Four forms of inheritance** cascade rules through the substrate:

| Form | What cascades | Example |
|---|---|---|
| **Type** | Governance rules from parent capsule to child capsule | `core → task → engineering-task` |
| **Template** | Default rendering from parent template to child template | `board.template.md → project-board.template.md` |
| **Scope** | Work boundaries from playbook to project to task | Playbook says "7 slices" → project scopes each slice → task executes one |
| **Canon** | Truth from canonical source to derived surface | Capsule definition (canonical) → orientation doc (derived) |

## §6 — Pipelines and playbooks: the orchestration substrate

**Pipelines are declarative workflow templates.** A pipeline is a DAG of WorkflowNodes (pipeline → stage → step). Author once; run many. Each run is a `pipeline-run` instance with its own event log. The DAG/DAG-Run pattern (Airflow, BPMN) but expressed in markdown + governed by capsule.

**Playbooks are governed procedures written in natural language.** An agent reads the playbook and executes it. The playbook is both the governance and the execution spec. Six sections; six-section discipline. See [`how-playbooks-work.md`](../../vault/files/2b5a3dd5.md).

**The proof-of-pattern: dev-pipeline.** The dev-pipeline scaffold at [`cd1fcd25`](cd1fcd25.md) has shipped 40+ releases of Tropo-OS itself (v1.6 through v1.52), each through the same typed scaffold: brief acceptance → release plan generation → arch-spec authoring → build extract → external test → git commit. Tropo uses Tropo to build Tropo.

**The Three-Pipeline Substrate-Enforcement Architecture (v1.51 NEW).** Three governed pipeline classes compose at the dev-pipeline ship-gate: [dev-pipeline (cd1fcd25)](cd1fcd25.md) authors substrate; it triggers [doc-pipeline (5a4337ff)](5a4337ff.md) execution (canonical-doc updates with every release; Orpheus lane) + [test-pipeline (da3f50dc)](da3f50dc.md) execution (verification with every release; Vela lane). Each pipeline takes a typed activation-input commitment: [dev-spec.capsule (c3f68cb5)](c3f68cb5.md) for dev-pipeline; [doc-spec.capsule (9a7d314a)](../../.tropo/capsules/doc-spec.capsule.md) for doc-pipeline; [test-spec.capsule (621824df)](../../.tropo/capsules/test-spec.capsule.md) for test-pipeline. Coupling enforcement at runtime: dev-pipeline activation cannot close until triggered doc + test activations both reach status:done. ## §7 — Tropo Work: the work-management primitives

**Tropo Work is the killer-app subsystem.** It's why the Studio exists: agentic teams executing real work, with audit trails + verification + cross-generational continuity.

The primitives:

- **Tasks** ([`task.capsule`](../../.tropo/capsules/task.capsule.md)) — units of work with owner, lifecycle (backlog → active → review → done → archived), verification.

**User-facing CLI gestures (the import → work → export loop):**

1. **`tropo-extract.py --projection <uid>`** — extract a markdown working-copy from an imported `.docx`. v1.26.0.
2. **`tropo-register-template.py --path <path-to-docx> --name <slug>`** — register a user's `.docx` as a governed template. v1.28.0 NEW.
3. **`tropo-export.py --working-copy <uid> [--template <uid>]`** — export the working-copy to a `.docx` deliverable. **Two paths** (v1.28.0 NEW):
   - **Default (no `--template`) = preservation export.** Source binary as scaffold (max fidelity — preserves header, footer, theme, custom XML, macros, embedded fonts, watermarks, section properties; replaces body with working-copy content). Use case: *"send David's review back to David in his own format."* Falls back to `original_styles:` reconstruction if source binary missing; fails loud if both missing.

4. **`tropo-backfill-styles.py --all --yes`** — migration gesture: populates `original_styles:` on pre-v1.28.0 projections (or `--folder-markers --yes` retro-fills missing folder mirrors for pre-v1.28.0 imports). v1.28.0 NEW.

## §9 — Identity continuity across generations

**Agents die. The crew doesn't.** Tropo's load-bearing differentiator: institutional knowledge survives across AI generations through files, not memory.

Five primitives carry continuity:

- **Soul** (the `§Soul` section of the agent's unified entry, per executive agent) — character, behavioral rules, how-we-work. Read first at boot; sandwich-anchored at startup signal.
- **Memory** ([Memory Subsystem v3.0](d0624a04.md)) — a single boot-read surface `agent-memory.md` + an append-only episodic log `agent-memories.jsonl` (mid-session captures); the curator folds the log at retirement (v1.67 Memory v3.0 cascade).
- **Living transfer** — the predecessor's hand-off written at peak context. Forward-looking only; "what's locked, what's drafted, what's pending."
- **Reflection** — the predecessor's backward-looking honest record. What worked, what didn't, what surprised.
- **Activation registry** — the authoritative lineage + generation record (`type: activation` entries; replaced the pre-v1.21 tabular generation-log). The ADR-016/028 identity checks read it at boot.

When an agent boots, it's reading itself into being from its unified entry + these lineage surfaces — not booting from blank context. The agent IS the composite (soul + memory + Vault + crew + sleeve), per [Operating Principle §8](../../.tropo-studio/operating-principles.md): *"Agent ≠ Sleeve."* The LLM is one component, not the identity.

## §10 — The verification model

Tropo's quality discipline is **three-instrument verification + cold-boot stranger-test + dogfood-gate**:

- **Three-instrument verification** — for any load-bearing artifact: build (the author) + independent review (peer or sa.skeptic) + cold-boot stranger test (sa.cold-boot — fresh context, no prior knowledge). Each instrument catches defects the others can't.
- **Cold-boot stranger-test** — the load-bearing instrument. A sub-agent dispatched with ONLY the artifact + fresh context. Reads it cold. Produces output. If a stranger can produce correct behavior reading the artifact alone, the artifact is correct.
- **Dogfood gate** — when an executor or validator ships, fire it on the cycle's own substrate. The cycle's own ship is the structural verification (v1.9.2 + v1.10 proven; pristine-no-Rule-7 9-cycle precedent at v1.10).

**v1.10 hardened enforcement.** [`validate-capability-membership.py`](../../.tropo/scripts/validate-capability-membership.py) runs in STRICT mode by default. Checks 19-23 + Rule 11 + Rule 12 + structural-consistency at ERROR severity. The build refuses to ship if substrate fails the validator. See [`how-capability-membership-works.md`](../../vault/files/0c2b6aa4.md).
