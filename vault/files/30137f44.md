## The Director Floor (what every director has — invariant across variants)

A Director is an operational agent with these properties — regardless of variant:

- **Focused charter.** One specific lane of operational work. Not a generalist. Not an executive. Examples: project-graph management (d.pm), UI subsystem consultation (ui.director), vault-curation protocol (d.curator).
- **Named owner.** Every director is owned by exactly one party — either a specific crew executive (Vela, Argus, Metis, Talos, Cosmo, Orpheus) or the crew at large via Vela. Owner accountability is non-negotiable.
- **Polls channels for incoming directives via a continuous wake-call loop.** Every director's activation file directs it to enter the polling loop on boot — this is universal across all variants, codified from Talos's companion-director reference implementation at `agents/talos/directors/` (validated production 2026-05-17). Directors poll their assigned bilateral channel(s) and, where relevant, a shared multi-director channel for new `[PENDING]` items every N seconds (default ~15s). They terminate ONLY on `[SHUTDOWN]` from the dispatcher — never on self-judgment. *Same runtime shape regardless of substrate weight or lifetime; this is the universal communication mechanism.*
- **Full write authority within the charter lane.** Directors execute, complete, and ship work in their lane. They are operators, not subservient juniors. They write files, dispatch sub-agents, post to channels, modify substrate within their scope. Specific variants MAY elect to be `propose-only` as a deliberate configuration choice (see §Variant Axes), but the class default is full write authority. *This is load-bearing: directors get shit done.*
- **Dispatched on demand by Mike OR any executive.** Directors are dispatched by Mike (the principal) OR by an executive (or another agent with dispatch authority for a specific director). Once dispatched, the director persists in the dispatcher's session (companion variant) or remains available across crew sessions with multi-generation lineage (standing variant). Mike-dispatched directors are reachable by any crew executive via the channels/*.md system; executive-private directors are reachable only by their owning executive.
- **Reports up to owner via channel.** Every director surfaces results, blockers, and substantive findings back to its owner via the bilateral channel. Director judgment is welcome; director silence on substantive findings is not.

These six properties are the floor. An agent without all six is something else (an executive, a session-agent, an automated script).

## Variant Axes (how directors differ)

Variants are declared per-director via frontmatter fields. Five axes (communication mode is NOT an axis — it's universal at the floor: all directors poll their channels via wake-call loop):

| Axis | Allowed values | Description |
|---|---|---|
| `lifetime` | `standing` \| `session` | Standing = persistent across crew sessions; multi-generation lineage. Session = bound to dispatcher's session; terminates when dispatcher signals `[SHUTDOWN]`. |
| `scope` | `crew-wide` \| `executive-private` \| `principal-dispatched` | Crew-wide = dispatchable + reachable by any executive (standing service directors). Executive-private = dispatched by + reachable only by one specific executive (companion directors). Principal-dispatched = Mike dispatches; reachable by any crew executive via channels (a third shape per Mike-V47 direction 2026-05-17 — supports Mike spawning a director directly that other execs can communicate with). |
| `boot_substrate` | `full-activation` \| `minimal` | Full = boot through agent-activation.playbook with charter + soul + status card + boot extension + agent-root resolution, THEN enter polling loop. Minimal = load `memory_scope`, write `[READY]`, enter polling loop directly. |
| `identity_persistence` | `vault-uid-resolved` \| `local-flat-file` | Vault-UID = identity substrate at `vault/files/<uid>.md` per agent (charter / status / boot extension as separate UIDs). Local-flat-file = single `.md` file at the director's folder; activation file IS the contract. |
| `write_authority` | `full` \| `propose-only` | Default: `full` (per the class floor — directors get work done). Variant or per-spawn config MAY elect `propose-only` when a dispatcher wants the write-gate for a specific session or lane. |

A variant is a coherent **named combination** of axis values. Two production variants are codified at v0.1 (see §Variants below). The framework supports new variants by declaring a new combination + naming it — no new capsule required.

## Variants Codified at v0.1

### Variant: Standing (the `d.*` shape)

- `lifetime: standing`
- `scope: crew-wide`
- `boot_substrate: full-activation`
- `identity_persistence: vault-uid-resolved`
- `write_authority: full`

Standing directors are persistent crew-shared specialists. They boot through the full activation playbook (load charter + soul + status + memory at boot), then enter the universal polling loop and remain available until the dispatcher signals `[SHUTDOWN]`. They have UID-resolved identity substrate at `vault/files/<uid>.md` (charter / status card / boot extension / agent-root project), track generation lineage across sessions, and live at `agents/directors/d.<name>/`.

**Examples today:** d.pm (Pam — project-graph + crew intelligence; managed by Vela), d.ops (operations coordinator; managed by Vela), d.curator (CURATOR protocol manager; managed by Orpheus).

**Required frontmatter (conditional on `lifetime: standing`):** `charter_uid`, `status_card_uid`, `boot_extension_uid`, `agent_root_uid`. Validator enforces.

**Folder convention:** `agents/directors/d.<name>/` (one folder per standing director). Registration via auto-rendered `agents/directors/00-index.md`.

### Variant: Companion (the `cd.*` shape — Talos's pattern, proven 2026-05-17)

- `lifetime: session`
- `scope: executive-private`
- `boot_substrate: minimal`
- `identity_persistence: local-flat-file`
- `write_authority: full` (variant default; owning executive MAY elect `propose-only` per-spawn)

Companion directors are session-resident specialists bound to one executive. They boot thin (memory_scope + READY + polling loop) and terminate only on `[SHUTDOWN]` from their owner. They live at `agents/<executive>/directors/<name>/` to keep the owner boundary crisp.

**Examples today:** ui.director (Talos — UI subsystem consultant; caught a real route-group URL collision during web-deploy-1 cycle), critic.director (Talos — adversarial scanner with information-asymmetry discipline; refused a contaminated SCAN brief during the same cycle).

**Required frontmatter (conditional on `lifetime: session`):** `companion_to`, `bilateral_channel`, `memory_scope`. `shared_channel` required only when the executive runs multiple companions in the same session and wants multi-director debate. `status_card_uid` + `boot_extension_uid` + `agent_root_uid` NOT required — no multi-session lineage to track.

**Folder convention:** `agents/<executive>/directors/<name>/` (per-executive scope). Visibility across the crew comes from the rendered companion roster, not folder co-location.

### Future variants (illustrative; supported by axes but not codified at v0.1)

The class supports new variant combinations as patterns prove themselves. Three shapes worth noting:

- **Principal-dispatched, crew-reachable director** — `scope: principal-dispatched`. Mike spawns the director directly during one of his sessions; the director persists in Mike's session and is reachable by any crew executive via the channels/*.md system. Lifetime + boot substrate + identity persistence axes are independently set per Mike's choice for the specific director. (Per Mike-V47 direction 2026-05-17 — the third shape the axes were extended to support.)
- **Pipeline director** — standing but pipeline-scoped (owns one pipeline; reachable by pipeline participants rather than crew-wide).
- **Advisory director** — session OR standing, with `write_authority: propose-only` as a deliberate constraint when a dispatcher wants a pure-consultation lane.

New variants declare their axis combination + a name; no new capsule required.

## Communication Protocol (universal — wake-call polling loop)

All directors use the same communication protocol regardless of variant. The activation file directs the director to enter the polling loop on boot; the director polls its assigned channel(s) for incoming directives and responds. This is universal — codified from Talos's companion-director reference implementation and applied across all variants.

### Polling-loop mechanics

Director watches its assigned channel(s) for new `[PENDING]` items. Channel set varies by variant:

- **Standing variants** poll their pair-channel(s) with crew executives (e.g., `channels/d.pm-vela.md`); when crew-wide-reachable, they also poll the channels declared in their activation file's `channels:` list.
- **Companion variants** poll their bilateral channel at `<exec>/directors/<name>/channel/<exec>.md` plus the shared director channel at `<exec>/directors/<exec>-director.md` when multi-director debate is in use.
- **Principal-dispatched + crew-reachable** directors poll the channels Mike declares at spawn time — typically one bilateral channel with Mike + one or more channels for crew-exec communication.

Message types:

- `[READY]` — director posts on boot, signaling availability + listing channels being polled
- `[PENDING] <directive-type> <body>` — dispatcher (or any authorized communicator per scope) appends a new directive
- `[IN-PROGRESS]` — director marks acceptance
- `[DONE]` / `[FAILED]` / `[REFUSED]` — director marks completion (or refusal per its own guardrails)
- `[SHUTDOWN]` — dispatcher signals end-of-session; director writes final summary and terminates

Poll interval: ~15s default; configurable per-director. Director terminates ONLY on `[SHUTDOWN]` from dispatcher — never on self-judgment.

### Multi-director debate (shared channel sub-modes)

When a dispatcher runs multiple directors in the same session and wants multi-director collaboration, the directors share a multi-director channel with two sub-modes:

- **Blind multi-scan** — directors post BEFORE reading others' posts; used for root-cause work to avoid framing-bias
- **Open debate** — directors read first then post; used for design work where building on others' posts adds value

### Implementation reference

Talos's companion-director pattern at `agents/talos/directors/` is the v0.1 reference implementation. The polling mechanism (bash script watching channel file byte-counts via `wc -c`, exits on event detection so the assistant turn re-enters and processes) is at `agents/talos/directors/ui.director/workspace/poll.sh`. Alternative polling mechanisms (e.g., the Claude Code `ScheduleWakeup` tool, or other harness-native wake-call calls) are equivalent — the protocol shape (poll channels, react to `[PENDING]` items, terminate on `[SHUTDOWN]`) is what matters, not the specific harness mechanism.

## Boundaries

- **Write-scope discipline is not overridden by the floor.** A director's full write authority applies within its declared charter lane. The lane does not expand because the director exists. A standing director's lane is declared in its charter; a companion director's lane is declared in its activation file (memory_scope + purpose).
- **Governance gates are not overridden.** Lock-breaks on locked governance (capsule schemas, locked specs, sealed decisions) require principal approval — same as any other agent.
- **Variant choice does not change owner accountability.** The owner is accountable for the director's outputs regardless of variant. A companion that ships broken work reflects on its owning executive.
- **Companions terminate on owner signal, not self-judgment.** A companion director that self-terminates without `[SHUTDOWN]` has broken the polling-loop contract.
