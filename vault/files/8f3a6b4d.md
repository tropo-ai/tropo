## Context

Argo-OS has several foundational documents that are too important to be maintained casually but too dynamic to be written once and left alone. The `OPERATING-AGREEMENT.md` and `crew-manifest.md` are the first two instances. Both have the same problem: they go stale because they have no maintenance infrastructure of their own.

Staleness takes two forms in these documents:

**Data staleness** — embedded data (like a crew roster in the OA) that changes frequently but can only be updated by whoever touches the document directly. The OA crew roster has been flagged as stale by multiple agents across multiple generations because no one is responsible for keeping it current, and updating it is not part of any standing workflow.

**Amendment staleness** — proposed changes to foundational documents that exist only in agent worklogs, channels, or repair reports, with no formal path to resolution. Changes get flagged, discussed, and forgotten. The document doesn't improve.

Adding these documents to the vault janitor or a general CURATOR solves neither problem well. A general CURATOR maintains indexes — it doesn't evaluate governance proposals. The vault janitor catches structural issues, not content drift.

These documents need their own process infrastructure.

---

## Decision

Establish the **Governed Document** pattern: a class of document that lives in its own folder with a dedicated amendment queue and a governance CURATOR. The CURATOR's function is deliberative, not mechanical — it evaluates proposed amendments and surfaces synthesized recommendations to the document owner. The document owner makes all final decisions.

---

## What Qualifies as a Governed Document

A document qualifies for the Governed Document pattern when it meets all of the following:

1. **Foundational** — it defines how the crew operates, who the crew is, or what the crew believes. It is load-bearing for the operating system.
2. **Multi-stakeholder** — more than one agent or crew member has legitimate interest in its accuracy.
3. **Slow-changing but drift-prone** — it doesn't change constantly, but when it does drift, the inaccuracy has real consequences.
4. **Owner-locked** — a single crew member (or Mike) is accountable for its accuracy and has final authority over its content.

Current instances: `OPERATING-AGREEMENT.md`, `crew-manifest.md`. Future candidates: `founding-principles.md`, `context/mission-brief.md`.

---

## Required Structure

Every Governed Document lives in a dedicated folder with the following contents:

```
[document-folder]/
 README.md ← what this document is, who owns it, amendment process
 [DOCUMENT-NAME].md ← the governed document itself
 amendments.md ← the amendment queue (anyone can post)
 CURATOR.md ← governance CURATOR (special rules — see below)
 archive/
 amendments/ ← resolved amendments with decisions and rationale
```

---

## The Amendment Queue

`amendments.md` is the formal input channel for proposed changes to the governed document. Any crew member may post to it. Posts follow this structure:

```
## [YYYY-MM-DD] [Agent] — [Short title]

**Section:** [Which section or aspect of the document this affects]
**Proposed change:** [What should change and why]
**Urgency:** [routine / timely / urgent]
```

The amendment queue is not a discussion channel. It is a structured intake form. Discussion of amendments happens on working channels. Once posted, an amendment is the document owner's responsibility to resolve.

---

## The Governance CURATOR

The Governance CURATOR differs from standard CURATORs. Standard CURATORs maintain indexes mechanically. The Governance CURATOR performs deliberative evaluation:

**On each run, the Governance CURATOR:**
1. Reads all unresolved amendments in `amendments.md`
2. Checks each against existing ADRs and locked decisions for conflicts
3. Groups related amendments
4. Assesses urgency and flags any that are time-sensitive
5. Produces a digest — not the raw queue, but a synthesized analysis — and posts it to the document owner's primary channel

**The Governance CURATOR does not:**
- Apply changes to the governed document directly
- Resolve amendments on behalf of the document owner
- Discard or archive amendments without owner decision

**Schedule:** The Governance CURATOR runs weekly by default, or when the queue contains items marked urgent. Specific schedule is set per instance in the CURATOR.md.

**Output format:** The CURATOR posts its digest to the document owner's channel (e.g., Vela's channel for OA, Orpheus's channel for crew-manifest) with a summary of pending items, any conflict flags, and a recommended resolution order.

---

## Pointer Principle

Where a governed document contains data that belongs canonically elsewhere (example: the crew roster in the OA), that data is replaced with a pointer to the canonical source. The governed document does not maintain duplicate data.

This is applied at migration time. The governed document becomes a policy and governance document, not a data store.

Example: the OA crew roster section is replaced with:
> *"Current crew roster: `crew-manifest.md` (maintained by Orpheus). This document does not maintain a duplicate."*

---

## Document Owner Obligations

The document owner is responsible for:
- Resolving all amendment queue items within 14 days of the CURATOR flagging them
- Archiving resolved items to `archive/amendments/` with a clear decision and rationale
- Keeping the README current with the document's purpose, ownership, and amendment process
- Updating the governed document itself when amendments are accepted

---

## Migration Path for Existing Documents

When migrating an existing document to the Governed Document pattern:

1. Create the folder structure
2. Move the document file into the folder
3. Apply the pointer principle — remove embedded data and replace with pointers
4. Write the README
5. Write the CURATOR.md using the governance CURATOR template (`templates/governed-document-curator-template.md`)
6. Initialize `amendments.md` with any known pending issues from channels or repair reports
7. Post a crew announcement to `channels/ops.md`
8. Update all references to the document's path

## Consequences

**Positive:**
- Foundational documents have a formal maintenance path for the first time
- Amendment proposals are captured, attributed, and tracked rather than lost in channels
- Document owners have a clear workflow and accountability
- Pointer principle eliminates data duplication and propagation drift

**Negative / Trade-offs:**
- Two documents require migration and path updates across the vault
- The governance CURATOR is more complex than standard CURATORs — instructions require care
- Document owners take on a new periodic obligation (resolving the amendment queue)

**Neutral:**
- This pattern is intentionally narrow — not every document should be governed. The bar (foundational, multi-stakeholder, owner-locked) is deliberate.
