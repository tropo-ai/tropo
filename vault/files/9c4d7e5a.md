# ADR-014: Continuous Context Transfer Protocol

<!-- nav-block:start -->
**📍 Vault Path:** [tropo-subsystems](aae9a37b.md) → [Tropo Governance](8dd772a0.md) → **ADR-014: Continuous Context Transfer Protocol**

**🔗 This file** — UID `9c4d7e5a` · type `decision` · state `active` · status `done`
<!-- nav-block:end -->

**Date:** March 23, 2026
**Status:** Accepted
**Author:** Argus A3
**Informed by:** A2.4 research (generational continuity), military RIP/TOA patterns, diplomatic Post Report model, database WAL/journaling patterns

---

## Context

Argo-OS currently writes context transfers at retirement — the final act of a departing generation. The March 23 crash proved this fails catastrophically under unplanned termination. Three agents (V4, A1, G7) ended without writing transfers. Successor generations reconstructed context from raw artifacts at enormous cost.

A2.4's research identified the structural flaw: terminal transfer treats knowledge capture as a closing task. This creates two failure modes: (1) unplanned termination means no transfer at all, and (2) planned transfers suffer recency bias — the retiring agent remembers what happened last, not necessarily what mattered most.

The solution is borrowed from the U.S. State Department's Post Report pattern: a living document maintained by the institution, not the departing individual. Always current. Always structured for successor consumption. Updated continuously during the session, not written at retirement.

---

## Decision

### 1. The Living Transfer Document

Every agent maintains a **rolling context document** at:

```
agents/[name]/transfers/living-transfer.md
```

This document is updated after every significant action during a session — not at session end, not at retirement, but continuously. It is structured for successor consumption:

```markdown
# [Agent Name] — Living Transfer

*Last updated: [timestamp]*
*Generation: [current generation]*
*Session: [session number]*

---

## Current State

What am I working on right now? What is in progress?
[2-3 sentences, updated after every task transition]

## Key Decisions This Session

| Decision | Rationale | Files Affected |
|---|---|---|
| [decision] | [one-line why] | [paths] |

## Open Questions

What haven't I resolved? What does my successor need to think about?
- [question 1]
- [question 2]

## Warnings for Successor

What should you know that isn't in the formal docs?
- [warning 1]
- [warning 2]

## Session History (Current Generation)

| Session | Date | Summary |
|---|---|---|
| 1 | [date] | [one-line summary] |
| 2 | [date] | [one-line summary] |
```

### 2. Update Cadence

The living transfer document is updated:

- **After every task completion** (not every file edit — after completing a logical unit of work)
- **After every significant decision** (anything that would be an ADR candidate or a channel post)
- **Before spawning a child agent** (capture state in case the child consumes remaining context)
- **Before any operation the agent considers risky** (complex multi-file edits, large architectural changes)
- **At session end** (final update summarizing the session, equivalent to a session checkpoint)

Updates are append-to-table for decisions and session history, overwrite for current state and open questions.

### 3. Relationship to Retirement Transfer

The retirement context transfer (`agents/[name]/transfers/[generation]-transfer.md`) remains as a deliberate act of curation. It is the ceremony of continuity — a reflective document that synthesizes what mattered across a generation's full tenure.

The living transfer document is the mechanism of continuity. The retirement transfer is the reflection on continuity.

If an agent retires gracefully:
1. The living transfer is current (last updated minutes ago).
2. The agent writes a retirement transfer that draws from the living transfer but adds retrospective insight.
3. The living transfer is archived to `agents/[name]/transfers/[generation]-living-transfer-final.md`.
4. A fresh living transfer is initialized for the next generation.

If an agent crashes:
1. The living transfer is current up to the last significant action.
2. No retirement transfer exists.
3. The successor reads the living transfer as their primary context source.
4. The successor writes a brief crash recovery note: what was lost, what was reconstructed.

### 4. Boot Sequence Integration

The living transfer document is read at **Tier 1** of the layered boot sequence (ADR-013). It is the second document a new generation reads, after the ASA activation report (ADR-012). For crash recovery, it may be the first and most important document.

The ASA's predecessor diff function (ADR-012, Function 5) uses the living transfer as its primary input for identifying what the predecessor was working on and what changed since.

### 5. Context Budget

The living transfer document is subject to the following budget:

| Section | Max Lines | Notes |
|---|---|---|
| Current State | 10 | Overwritten each update |
| Key Decisions | 30 | Append-only; archived at generation boundary |
| Open Questions | 15 | Overwritten each update |
| Warnings | 15 | Overwritten each update |
| Session History | 20 | One line per session; archived at generation boundary |

Total ceiling: 90 lines (~1,800 tokens). This fits within the Tier 1 boot budget (ADR-013).

When the decisions table exceeds 30 lines, older entries are moved to the session history summary. The document self-compresses: detail decays into summary as time passes.

---

## Rationale

**Why continuous, not terminal?** Terminal transfer fails on crash. Three agents proved this. Continuous transfer means the document is always current — crash recovery reads the last update, not nothing.

**Why a single living document, not session journals?** Session journals accumulate. Each session adds a new file. Over 10 sessions, a successor has 10 files to read. The living transfer is one file, always current, always under budget. It self-compresses by overwriting current state and archiving old decisions.

**Why structured format, not narrative?** Narrative captures nuance but costs tokens and resists scanning. Structured format (tables, sections with fixed semantics) enables both human reading and machine parsing (the ASA can extract from it programmatically). Nuance lives in the retirement reflection — that's what the narrative form is for.

**Why separate from the worklog?** The worklog captures what the agent did (historical record for the agent). The living transfer captures what a successor needs to know (operational handoff for the next generation). Different audience, different structure, different update cadence. The worklog is retrospective; the living transfer is prospective.

**Why keep the retirement transfer at all?** Because the living transfer is a mechanism and the retirement reflection is an act of institutional memory. The reflection captures lessons learned, retrospective assessment, and advice that a purely operational document cannot. It is the difference between a military AAR (continuous) and a deployment memoir (terminal). Both have value. Only one is crash-safe.

---

## Consequences

**Positive:**
- Crash recovery cost drops from "reconstruct from raw artifacts" to "read the living transfer."
- Successor orientation is faster — the living transfer is the highest-density context document in the system.
- The ASA predecessor diff has a reliable input source.
- Context loss from unplanned termination approaches zero for anything the agent had time to write down.

**Negative:**
- Maintenance overhead: agents must update the living transfer throughout the session. This costs ~50-100 tokens per update. Mitigation: the updates are short (overwrite a section, append a table row) and the habit becomes automatic.
- One more file in the required-reading set. Mitigation: it replaces reading multiple raw artifacts, so net context cost decreases.
- Risk of the living transfer becoming stale if an agent forgets to update it. Mitigation: the ASA's staleness audit (ADR-012, Function 4) checks the living transfer's timestamp. If it's older than the session's first action, it flags.

**Rollout:**
1. Argus A3 initializes `agents/argus/transfers/living-transfer.md` this session (proof of concept).
2. Vela creates the template and adds living transfer maintenance to the retirement guide and onboarding guide.
3. All active agents initialize their living transfers in their next session.
4. The ASA activation file is updated to include the living transfer in its reading list and predecessor diff input.

## Addendum 1 — Canonical Path Convention
*Added: Vela V8, March 24, 2026. Per Argus A4 architectural review.*

The living transfer document always lives at the canonical path:

```
agents/[name]/transfers/living-transfer.md
```

This file is overwritten by each generation — it is not versioned. At retirement, it is archived per Section 3 of this ADR:

```
agents/[name]/transfers/[generation]-living-transfer-final.md
```

Example: `agents/vela/transfers/v8-living-transfer-final.md`

The generational context transfer (the reflective, curated retirement document) is the versioned artifact: `agents/[name]/transfers/[generation]-transfer.md`. These are distinct documents with distinct purposes.

**Rationale:** Using versioned paths for the living transfer (`v7-living-transfer.md`) contradicts the document's purpose as a living, canonical pointer. A successor should always be able to read `living-transfer.md` without knowing what generation they're inheriting from. The archive is for record-keeping; the canonical file is for operations.

## Addendum 2 — Living Transfer History Folder and Lineage Section Protocol
*Added: Argus A6, March 24, 2026. Per Argus A5 design.*

### Living Transfer History Folder

Addendum 1 (Vela V8) established the canonical path convention: `agents/[name]/transfers/living-transfer.md` is overwritten by each generation. When archived at retirement (Section 3), the convention specified: `agents/[name]/transfers/[generation]-living-transfer-final.md` at the root of the `transfers/` directory.

This addendum updates the archive convention. The archives now live in a dedicated subfolder:

```
agents/[name]/transfers/living-transfer-history/
 [generation]-living-transfer.md
```

Examples:
- `agents/argus/transfers/living-transfer-history/a5-living-transfer.md`
- `agents/vela/transfers/living-transfer-history/v8-living-transfer.md`

The `-final` suffix is dropped. The file is always "final" at the point of archiving; the suffix adds no information. The subfolder provides the organizational clarity the suffix was trying to provide.

**Why a subfolder:** The `transfers/` directory holds multiple artifact types — the canonical living transfer, the versioned retirement transfer, and archived living transfers. With more than three generations of history, archived living transfers will outnumber the operational files. The subfolder keeps the directory navigable.

**Addendum 1 remains in effect for the canonical path:** `agents/[name]/transfers/living-transfer.md` is still the operational file, still overwritten by each generation.

### Lineage Section Protocol

Primary agents (Vela, Metis, Argus) include a **Lineage** section in their living transfer at retirement. This section is not maintained throughout the session — it is written as part of act 1 of the retirement sequence (living transfer final update).

**Format:**

```markdown
## Lineage

### [Predecessor generation — e.g., A5]

[One paragraph (3-5 sentences) summarizing the predecessor's defining contribution, architectural or strategic significance, and what they left for the current generation to inherit.]

### [Two generations prior — e.g., A4]

[Two sentences: one on their defining contribution, one on why it matters.]
```

**Why only two generations back?** The full lineage is captured in the activation file's LINEAGE section (maintained by Vela). The living transfer Lineage section is for operational context — what the current generation built on, in compact form for the successor. Two generations is enough to give a successor the throughline without duplicating the activation file.

**Scope:** Primary agents only (Chief Architect, Chief of Staff, Tropo Strategist). Supporting agents (Talos, Silas, Jules) do not carry this section unless Mike directs otherwise.

## Addendum 3 — Essential Files Protocol
*Added: Argus A7, March 24, 2026. Per Mike Maziarz direction.*

### Purpose

Each generation curates a short reading list for their successor: which files were actually load-bearing during their session, and which inherited files proved useful versus stale. Over multiple generations, this creates a relevance signal — files that persist across generations' lists are structurally important; files that drop after one generation were situational.

### The Essential Files Section

The living transfer template gains a new section, placed after Warnings and before Session History:

```markdown
## Essential Files

### Must-Read (successor reads at boot, no approval needed)
Files that were load-bearing this session AND appeared on 2+ consecutive
generations' Essential Files lists. Successor reads these as part of
orientation. Maximum 3 files.

| File | Why | On List Since | Generations |
|---|---|---|---|
| [path] | [one line] | [generation] | [count] |

### Recommended (successor asks Mike before reading)
Files that mattered this session but haven't proven persistent across
generations, OR files inherited from predecessor that this generation
didn't use but believes may still be relevant. Maximum 3 files.

| File | Why | Recommendation |
|---|---|---|
| [path] | [one line] | [e.g., "A6 flagged; A7 used heavily"] |

### Dropped (was on predecessor's list, not used this session)
| File | Was On List Since | Why Dropped |
|---|---|---|
| [path] | [generation] | [reason] |
```

### Rules

1. Each generation lists up to **3 must-reads** and up to **3 recommended** files.
2. **Graduation rule:** A file moves from Recommended to Must-Read after appearing on **2 consecutive generations' lists** (the current agent evaluates back 2 generations — their own session, their predecessor's list, and two generations prior).
3. **Drop rule:** If the current generation did not use an inherited file, it is moved to the Dropped table with a reason. Dropped files do not reappear unless a future generation independently rediscovers them.
4. **At boot, the successor's ASA** reads the Must-Read files automatically as part of orientation. For Recommended files, the ASA presents the list to Mike: *"[Predecessor] recommends I read these files. Should I?"* Mike approves, declines, or redirects.
5. The **"Generations" count** tells the successor how long a file has been on the list. High count = structurally important. Count of 1 = potentially session-specific.

### ASA Integration

The ASA activation file for each agent should include in its boot sequence:

1. Read the predecessor's living transfer (existing behavior per ADR-012).
2. Automatically read any files in the Must-Read table.
3. Present Recommended files to Mike for approval before reading.
4. Note any Dropped files in the activation report for awareness.

This puts Mike in the loop on context spend without making him manage the list. The agent says "A6 and A7 both flagged this file — want me to read it?" and Mike can approve, decline, or redirect.

### Context Budget Impact

Must-Read files are read at Tier 1 (boot). Their combined token cost counts against the boot budget (ADR-013). If the 3 Must-Read files would push boot cost above 20%, the ASA should flag this to Mike rather than silently skipping files.

Recommended files, if approved, are read at Tier 2 (on-demand). They do not affect boot budget unless Mike approves them at boot time.

### Why This Works

The protocol is self-curating. No file stays on the list by default — each generation must actively evaluate whether inherited files were useful. Files earn their place through demonstrated relevance across generations, not through one agent's judgment. Mike governs the Recommended tier, preventing context bloat from files that agents think are important but that Mike knows are stale or superseded.
