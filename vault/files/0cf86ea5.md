## Purpose

After dev-pipeline build stage closes (step 4 update-subsystem-canonical-docs done; substrate authoring complete), the dev-cycle's owner authors a `doc-spec.md` describing what documentation must update across the three tiers (summary / subsystem / spec) to reflect the cycle's substrate changes. The doc-spec is the activation-input for a corresponding doc-pipeline activation; this step fires that activation.

By terminal close of the dev-pipeline activation, the triggered doc-pipeline activation MUST also be at `status:done` (engine-level enforcement per [51d171f3 §Three-Pipeline Coupling State Machine](51d171f3.md)). This step ensures the trigger fires + the substrate UID is recorded honestly.

Does NOT govern: the doc-pipeline activation's own work (that's the doc-pipeline definition's job; Orpheus O11 lane primary at Phase C). Does NOT cover: legacy cycle dev-pipeline activations pre-v1.51 — those grandfathered per dev-spec.capsule Rule 7.

## Flow Rules

**Activates when:** dev-cycle's pipeline-run reaches step 4 close (depends_on_steps: [9d4f7e21] update-subsystem-canonical-docs done). Engine surfaces this step as eligible-next in parallel with step 4.6 (trigger-test-pipeline-activation) — both have same dependency; engine fires both concurrently.

**Completes when:** all three exit_criteria pass:

- `triggered_doc_spec.uid exists` — a doc-spec.md entry exists at vault/files/<uid>.md
- `triggered_doc_spec.status == active` — doc-pipeline activation against the doc-spec entered status:active
- `dev_spec.triggered_doc_spec_uids contains <triggered-doc-spec-uid>` — the dev-cycle's dev-spec frontmatter has been atomically updated to record the triggered activation UID

**Advances to:** step 6 produce-release-folder (8654900a) — multi-input convergence with step 4.6 (4f64ec3c). Step 6's `depends_on_steps:` array is amended at v1.51 to require BOTH 4.5 + 4.6 complete (was `[9d4f7e21]` per v1.49.0 chain; v1.51 amendment makes the chain `3 → 5 → 5.5 → 4 → [4.5, 4.6] → 6`). Note: positioning per v1.49.0 reordered chain (Argus A78 substrate-coherence fix per 74f84c63); step 4 fires AFTER step 5.5, so the triggers sit between step 4 (last build/canonical-docs step) and step 6 (first ship-gesture step).

**Failure modes:** if doc-pipeline activation fails to enter status:active (e.g., doc-spec.md validation fails; engine refuses activation; race lock collision on triggered-spec UID), step 4.5 stays incomplete; dev-pipeline activation flips to `status: blocked_on_trigger_failure` per engine extension §Failure mode handling.

## Engine Contract — exit criteria DSL

Three exit_criteria evaluated mechanically by pipeline-runtime DSL evaluator (per v1.51 DSL extension at engine spec [51d171f3 §Exit-criteria DSL extension](51d171f3.md)):

- `triggered_doc_spec.uid exists` — DSL resolves `triggered_doc_spec` context handle to `dev_spec.frontmatter.triggered_doc_spec_uids[-1]` (most recent trigger); checks existence at vault/files/
- `triggered_doc_spec.status == active` — DSL resolves activation entry by that UID; checks frontmatter status field
- `dev_spec.triggered_doc_spec_uids contains <triggered-doc-spec-uid>` — DSL list-membership check; ensures the atomic write to dev-spec frontmatter succeeded

If all three pass: `verify-step 0cf86ea5` returns `pass`; step marks complete.

If any fail: step stays in `eligible` state; executor (Argus or doc-pipeline-primary) addresses + retries the trigger fire.

## Cold-Boot Walk-Through

A dev-pipeline-run enters step 4.5 with:

- Step 4 (9d4f7e21 update-subsystem-canonical-docs) closed; subsystem canonical docs updated for the cycle's substrate
- dev-spec entry exists at vault/files/<dev-spec-uid>.md with `triggered_doc_spec_uids: []` (empty at activation; populated at step 4.5 fire)

Argus (step_owner_role) executes:

1. **Author doc-spec.md** at vault/files/<new-uid>.md per doc-spec.capsule v1.0 schema. Required fields: `target_subsystem:` (UIDs of subsystems this cycle's substrate documents); `target_tier:` (summary / subsystem / spec / multi); `triggered_by_dev_cycle: <current-dev-spec-uid>`; `doc_changes_required:` (list of doc files needing update at each tier); `acceptance_criteria:`.
2. **Rebuild vault index** to register the entry (`npm run vault:rebuild`).
3. **Fire the trigger:** `pipeline-runtime.py --activation-uid <dev-activation-uid> --actor argus-a<N> trigger-step 0cf86ea5 --triggered-spec-uid <new-doc-spec-uid>`. Engine spawns doc-pipeline activation against the doc-spec + atomically writes `<new-doc-spec-uid>` into dev-spec's `triggered_doc_spec_uids:` array.
4. **Verify trigger fire:** doc-pipeline activation now exists at status:active; `dev_spec.frontmatter.triggered_doc_spec_uids` includes the new UID.
5. **Fire step-complete:** `pipeline-runtime.py step-complete 0cf86ea5 --artifact-links <new-doc-spec-uid>`.
6. **Fire verify-step:** `pipeline-runtime.py verify-step 0cf86ea5` — engine evaluates the three exit_criteria; returns pass.

Step 4.5 marks complete. Engine surfaces step 6 (8654900a produce-release-folder) as eligible once step 4.6 also completes (multi-input convergence on the downstream gate).
