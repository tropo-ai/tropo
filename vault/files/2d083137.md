## What This Subsystem Covers

Tropo Work is the killer-app subsystem of Tropo-OS — **Airflow for AI agents, but it's a folder.** It governs how work gets requested, executed, verified, and shipped across human-AI crews. Currently at **subsystem version 3.0.0** (locked 2026-04-24).

**Primitive stack (v3):** Entity / Project / Pipeline / Pipeline-Run / Board / Capsule-Definition + WorkItem subtypes (Note, Task, Design-brief, Arch-spec, Build, Release, Document, Decision, Collection-ref). Concept deprecated per [v3 Decision 1 (8b3f1d92)](8b3f1d92.md) — captures collapse into note.

**Core thesis:** *status* is intrinsic to the work; *pipeline-position* is a property of the run, not the work. WorkItems carry no `stage:` field. Pipelines are entity-authored protocol declarations. Pipeline-runs are ephemeral execution instances with pinned pipeline_version. Users author their own work-pipelines (Decision 10 — there is no canonical Tropo work-pipeline). Content-production pipelines terminate in `document (status: published)`, not `release` (Decision 11). Coherent verb surface across all primitives (Decisions 9 + 13).

### In scope

- **WorkItem capsule definitions** — note, task, design-brief, arch-spec, build, release, document, decision, collection-ref. The typed primitives that define what governed work IS.
- **Project + Board + Pipeline + Pipeline-Run capsule definitions** — the containers, surfaces, templates, and execution instances. Project capsule [(34e4cb0b)](34e4cb0b.md) anchors hierarchy; board-definition [(b0d1e4f2)](b0d1e4f2.md) renders surfaces; pipeline.capsule v3.0 + pipeline-run.capsule v2.0 are the workflow shape.
- **Activation-input *-spec capsules (v1.51 trio)** — dev-spec.capsule (engineering activation-input), doc-spec.capsule (documentation activation-input), test-spec.capsule (testing activation-input). The Three-Pipeline Substrate-Enforcement architecture per [c3dc9f00](c3dc9f00.md).
- **The four production pipelines** as activation trees + their per-cycle activation-root-projects + member work-items: dev-pipeline [(cd1fcd25)](cd1fcd25.md), doc-pipeline [(5a4337ff)](5a4337ff.md), test-pipeline [(da3f50dc)](da3f50dc.md), publish-pipeline / publish.pipeline class [(e2f4a8c1)](e2f4a8c1.md) + [(143c74d5)](143c74d5.md).
- **L0 root primitive** [`tropo-work` (b8e5f3a2)](b8e5f3a2.md) — the substrate root every governed work-item lives under (or under a sibling like `01-vault-inbox`).
- **Activation-root-project pattern** — invariant since v1.6 per Decision 3: "Everything is a child of the project. Step 0."

### Edge cases

- **Activation-root-projects** are typed `project` (Tropo Work primitive) but their `member_of:` chain often reaches subsystem hubs (Governance, Documentation, etc.) when the cycle's subsystem-scope is named. The activation-root is canonically Tropo Work; the cycle's substrate touches are subsystem-scoped via membership edges.
- **Capsule-definition entries** (e.g., task.capsule, dev-spec.capsule) are typed `capsule-definition` which is itself a Tropo Work primitive, but they're load-bearing as governance contracts. Cross-membership: Tropo Work owns the type; Tropo Governance often anchors the specific entries when content is OS-rules-class.

## How Work Flows (v3)

Work flows through user-authored work-pipelines. The default Tropo crew runs the [tropo-work-pipeline (7c5a2b8f)](7c5a2b8f.md) instance — a 5-stage flow (idea → design → spec → build → ship/publish) — but it's **one example**, not the canonical pattern. Strangers author their own pipelines; the platform ships the primitives.

Tasks use the **request-lifecycle**: `requested → accepted → active → verify → done` (plus `blocked`, `rejected`, `cancelled`). Inter-entity work is a request between entities; intra-entity work is self-assigned scaffolding via the `author` verb.

Pipeline-runs auto-scaffold stage-stub projects + working folders + default boards per the pipeline template's declarations (Decision 8). "Activate the pipeline" is user-friendly verb for "create a pipeline-run" — the primitive is pipeline-run.
