## Context

Argo-OS v2 has no self-monitoring. Document staleness, broken cross-references, schema drift, and inconsistent state across files are detected only when a human or agent happens to notice them during real work. At a crew of five, this is survivable — problems are caught quickly because everyone reads the same small set of files. At fifty, it's untenable — silent degradation compounds until someone discovers the vault is inconsistent, and the repair cost is proportional to how long the problem went undetected.

The v3 design brief (Pillar 2) proposes a biological cascade model: Detect → Diagnose → Repair → Verify. G7.2's research confirmed that no existing product provides self-monitoring for a document-based coordination system. This is Argo-OS's most commercially differentiating feature.

---

## Decision

Implement the Detect phase of the biological cascade model as four standing auditor agents. Defer Diagnose, Repair, and Verify phases.

### Four standing auditors

All are standing children in Vela's namespace, designed by Argus, activated by Mike on a recurring schedule:

1. **Vault Integrity Auditor** (weekly) — Cross-references, activation file structure, folder structure, README coverage, naming conventions. The Conformity Monkey.
2. **Freshness Monitor** (daily) — Status card staleness (48h threshold), crew brief consistency, document review cadence, topic channel archival compliance.
3. **Governance Validator** (weekly) — ADR consistency, OA vs. guides vs. activation files, role definitions, channel assignments.
4. **Channel Health Monitor** (daily) — Working channel cleanup compliance, topic channel archival, unresolved FLASH alerts, message format compliance.

### Reporting pattern

All auditors write to a standard report format (date, status CLEAN/FINDINGS, numbered findings with severity INFO/WARNING/CRITICAL and suggested actions). All post one-line summaries to `ops.md` (routine) or `alerts.md` (critical).

### What's deferred

- Automated repair agents — the crew needs detection data first to know what's worth auto-fixing
- Chaos testing — Release 3.4 scope
- Diagnostic children — spawn on demand, not standing infrastructure
- Verify phase — next scheduled audit run serves as verification
- Auditor-to-auditor cascade triggers — manual for now

---

## Rationale

**Why detect-only:** Detection is the highest-value, lowest-risk work. It surfaces problems without changing anything. Repair requires judgment about what to fix and how — that judgment should be human-supervised until the crew has enough data to define safe auto-fix boundaries. Progressive autonomy applies.

**Why four auditors instead of one:** Separation of concerns. Each auditor has a clear domain, a defined check set, and an appropriate cadence. A single monolithic auditor would burn excessive context on every run and make it hard to tune cadence (structural checks need weekly; freshness needs daily).

**Why Sonnet for all four:** Detection is pattern-matching and structural validation, not interpretation. Sonnet handles this reliably. The governance validator is the edge case (semantic consistency across documents), but its job is flagging potential issues, not resolving them. If Sonnet's first run misses semantic inconsistencies, upgrade to Opus for that auditor only.

**Why Vela's namespace:** Auditors are operational infrastructure. Vela owns operational infrastructure. Argus designs the specifications; Vela owns the running system. This matches the architect/operator split established in the OA.

---

## Consequences

**What gets better:** The vault monitors itself. Staleness, broken references, inconsistent state, and protocol drift are caught automatically. Problems are surfaced through channels before they compound. The crew brief becomes more reliable because the freshness monitor validates it. External adopters get a self-monitoring vault out of the box.

**What gets harder:** Four more standing children to manage. Mike needs to activate them on schedule (or they become standing scheduled tasks). Reports need review — detection without review is just noise. The crew brief needs to surface auditor findings.

**What changes:**
- Four new standing child directories under `agents/vela/children/`
- Auditor reports written to `agents/vela/children/[auditor]/reports/`
- Findings posted to `ops.md` and `alerts.md`
- Self-healing protocol guide written (`guides/how-self-healing-works.md`)
- Crew brief updated to include standing auditor status
- Child agent registry updated with all four auditors
