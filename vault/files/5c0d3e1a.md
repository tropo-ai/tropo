## Context

Argo-OS accumulates historical artifacts: crew log entries, channel archives, living transfers, child agent reports, agent worklogs, reflections, and ADRs. These artifacts are the system's institutional memory. They grow continuously and never shrink, because the historical record is permanent (OA Section 9).

The current maintenance model is centralized: the vault-janitor enforces file-size ceilings (ADR-013) and archives oversized content. But the janitor applies generic rules — age and size — because it cannot know what *matters* in each artifact type. A crew log entry about a founding decision has different preservation value than a routine status update. A child agent's research report has different compression needs than a working channel's daily coordination. Only the owner of an artifact knows what should stay hot, what can compress, and what moves to cold storage.

The March 23 crash (ADR-010) was caused by channel files growing beyond context limits. ADR-013 addressed the symptom (file-size ceilings). This ADR addresses the root cause: **the system has no distributed, domain-aware mechanism for maintaining its own memory.**

Additionally, Argo-OS's intergenerational memory currently has two mechanisms:

1. **Living transfers** (ADR-014) — Active memory: what the current agent knows
2. **ASA protocol** (ADR-012) — Orientation memory: compressed vault state for boot

What is missing is a third mechanism: **archival memory** — how the institution's historical artifacts age gracefully so the vault stays navigable as it grows. The Curator protocol completes this triad.

## Decision

### 1. The Curator Convention

Any folder in the vault that contains historical or accumulating content MAY include a file called **`CURATOR.md`** at its root. Designated folder types (see Section 4) MUST include one.

A `CURATOR.md` file is a lightweight activation file for a **curator agent** — a disposable, stateless child agent that reads the folder's content, applies the hot/warm/cold memory model, and terminates. The curator agent has no persistent state. It boots, tends, logs, and exits.

The CURATOR file is written by the **owner of the artifact** — the agent or human responsible for that folder's content. The owner knows what matters, what can compress, and what should never be lost. This makes memory maintenance distributed and domain-aware.

### 2. CURATOR.md Structure

Every CURATOR file contains:

**YAML Frontmatter:**

```yaml
---
type: tend
owner: "[agent name or role]"
schedule: "weekly" | "daily" | "monthly" | "on-demand"
last_run: "YYYY-MM-DD"
last_run_by: "[curator agent ID]"
hot_window: "[duration, e.g., 7d, 30d]"
warm_window: "[duration, e.g., 30d, 90d]"
cold_after: "[duration, e.g., 90d]"
---
```

**Body — Natural Language Instructions:**

The body contains plain-language instructions that a curator agent reads and follows. Three required sections:

1. **What Matters** — What content in this folder has lasting value? What should never be compressed or lost? What are the domain-specific criteria for importance?
2. **How to Compress** — When content moves from hot to warm, what does compression look like? Summaries? Key entries only? One-line index entries? What fidelity is acceptable?
3. **What to Preserve Verbatim** — Are there specific content types (decisions, milestones, lessons learned) that should never be compressed regardless of age?

### 3. The Three-Tier Memory Model

Every CURATOR defines three tiers for its folder's content:

| Tier | Definition | Fidelity | Who Reads |
|------|-----------|----------|-----------|
| **Hot** | Content within the `hot_window` | Full verbatim text | Current-generation agents at boot |
| **Warm** | Content between `hot_window` and `cold_after` | Compressed summaries with key entries preserved | Agents needing recent context |
| **Cold** | Content older than `cold_after` | One-line index entries; full text in versioned archive files | Researchers, historians, deep-context tasks |

The curator agent implements this by:
- Leaving hot content in the live file unchanged
- Replacing warm content with compressed summaries (inline or in a companion `[filename]-warm.md`)
- Moving cold content to an `archive/` subfolder as versioned files (e.g., `crew-log-2026-W12.md`)
- Maintaining a cold index file (`[folder]-index.md`) with one-line entries pointing to archive files

### 4. Required and Optional CURATOR Folders

**Required** (vault integrity auditor flags if missing):

| Folder | Owner | Recommended Schedule |
|--------|-------|---------------------|
| `crew-log.md` (parent folder) | Vela | Weekly |
| `channels/ops.md` (parent folder) | Vela | Weekly |
| `channels/releases.md` (parent folder) | Vela | Weekly |
| `channels/alerts.md` (parent folder) | Vela | Weekly |
| `agents/[name]/transfers/` | Each agent | At generation transition |
| `agents/[name]/children/` | Each agent | Monthly |

**Recommended** (not audited, but encouraged):

| Folder | Owner | Recommended Schedule |
|--------|-------|---------------------|
| `agents/[name]/archive/` | Each agent | Monthly |
| `agents/operations/[auditor]/reports/` | Vela | Weekly |
| `archive/channels/` | Vela | Monthly |
| Working channels (pair-level) | Subscribing agents | Daily (via vault-janitor) |

**Any folder** may include a CURATOR.md. The convention is open by design. If an agent creates a new artifact type that accumulates, they write a CURATOR for it.

### 5. The Curator Controller

A standing operational agent at `agents/operations/curator-controller/activate.md` serves as the scheduler and registry for all CURATOR files.

**The controller's functions:**

1. **Registry** — Maintains `agents/operations/curator-controller/curator-registry.md`, a table of all CURATOR files in the vault: path, owner, schedule, last run date, last run status.
2. **Scheduling** — On each run, the controller reads the registry, identifies CURATORs whose schedule is due, and activates curator agents for each. The controller does not tend — it dispatches.
3. **Logging** — Each curator agent logs its run to `agents/operations/curator-controller/logs/YYYY-MM-DD.md`: what folder it tended, what was compressed, what was archived, any issues encountered.
4. **Health Monitoring** — The controller flags: CURATORs that haven't run in 2× their scheduled interval (overdue), CURATORs with errors on last run, folders on the Required list that lack a CURATOR.

**The controller runs on a schedule** (recommended: daily). It reads the registry, dispatches curators for due folders, and writes its own log. It is a standing agent in `agents/operations/`, following the same pattern as the vault-janitor and standing auditors.

### 6. Relationship to Existing Infrastructure

| Component | Relationship to CURATOR |
|-----------|---------------------|
| **Vault-janitor** (ADR-002, ADR-013) | Janitor enforces hard size ceilings. CURATORs provide intelligent compression *before* ceilings are hit. The janitor is the safety net; the CURATOR is the primary mechanism. |
| **Standing auditors** (ADR-009) | Auditors detect drift and decay. CURATORs prevent drift and decay in historical content. Complementary, not overlapping. |
| **Living transfers** (ADR-014) | Transfers capture active agent memory. CURATORs maintain the archival layer that transfers compress from. The Curator in `agents/[name]/transfers/` defines how old transfers age. |
| **ASA protocol** (ADR-012) | The ASA reads vault state for orientation. CURATORs ensure the vault state the ASA reads is appropriately compressed, keeping boot cost bounded as the vault grows. |
| **Context budget ceilings** (ADR-013) | Ceilings are the hard constraint. CURATORs are the intelligent mechanism that keeps content under ceiling without losing information. |

### 7. Intergenerational Memory Model (Complete)

With the Curator protocol, Argo-OS has three memory mechanisms forming a complete model:

| Mechanism | Memory Type | What It Serves | Maintained By |
|-----------|------------|----------------|---------------|
| **Living Transfer** (ADR-014) | Active memory | Current-generation context and working knowledge | The agent, continuously |
| **ASA** (ADR-012) | Orientation memory | Compressed vault state for boot | Standing ASA process |
| **CURATOR** (ADR-020) | Archival memory | Long-term institutional history, intelligently compressed | Distributed curator agents |

This triad mirrors the human brain's memory architecture: working memory (active, limited capacity, high fidelity), episodic memory (recent events, moderate compression), and long-term memory (deep history, heavily compressed, indexed for retrieval).

## Rationale

### Why decentralized, not centralized?

A centralized grooming agent would need to understand every artifact type — crew logs, channel archives, transfer histories, child reports, worklogs. This creates a single point of failure and a scaling bottleneck. The Curator pattern distributes maintenance to the people who understand the content. The controller is just a scheduler — it doesn't need domain knowledge.

### Why a file convention, not a protocol document?

A CURATOR.md file *is* the protocol for its folder. It sits where the artifacts live, written in natural language, readable by any agent on any model. No separate protocol document to cross-reference. The instructions and the content they govern are co-located. This is the same principle as README.md — the documentation lives with the code.

### Why natural language instructions, not structured rules?

The curator agent is an LLM. It reads natural language better than structured rules. "Keep entries about founding decisions verbatim; compress routine status updates to one line each" is more expressive and more maintainable than a JSON schema. The Curator template provides structure through sections (What Matters, How to Compress, What to Preserve), but the content within each section is prose.

### Why not just extend the vault-janitor?

The janitor's job is enforcement — hard ceilings, cleanup, recycle bin. The Curator's job is intelligent maintenance — domain-aware compression that preserves what matters. These are different functions with different quality criteria. The janitor is a safety net. The Curator is the primary mechanism. Combining them would overload the janitor's scope and require it to understand every artifact type.

### Alternatives considered

1. **Centralized archive agent** — A single agent that reads all historical content and compresses it. Rejected: creates a bottleneck, requires too much domain knowledge, doesn't scale.
2. **Scheduled compression scripts** — Code-based automation. Rejected: violates the no-code principle of Argo-OS.
3. **Manual grooming by Vela** — Status quo. Rejected: doesn't scale, creates a single point of failure in Vela's workload, and Vela doesn't have domain expertise for every artifact type.

## Consequences

### What gets easier

- **Vault growth is managed automatically.** Historical content compresses on schedule without human coordination.
- **Boot cost stays bounded.** As the vault grows, CURATORs keep the hot tier small and the warm tier compressed.
- **Institutional memory is preserved.** Nothing is deleted — only compressed and indexed. Cold content is always retrievable.
- **New artifact types self-maintain.** When an agent creates a new folder of accumulating content, they write a CURATOR. No central process needs updating.
- **The intergenerational memory model is complete.** Active (transfer) + orientation (ASA) + archival (CURATOR) covers the full memory lifecycle.

### What gets harder

- **Every required folder needs a CURATOR.** This is new work for current agents. Mitigation: a template makes CURATOR creation a 5-minute task.
- **A new standing agent (curator-controller) must be maintained.** Adds to operational overhead. Mitigation: the controller is simple — it reads a registry and dispatches. No domain knowledge required.
- **Curator agents consume compute.** Each curator run is a disposable agent session. Mitigation: curator agents are fast (read folder, compress, archive, log) and run infrequently (weekly for most folders).
- **CURATOR quality varies by author.** A poorly written CURATOR produces poor compression. Mitigation: the template and the "What Matters" section guide authors. The curator-controller logs issues, and the vault integrity auditor can flag CURATORs that haven't been updated.

### What changes

- The vault-janitor remains the safety net but is no longer the primary memory maintenance mechanism.
- Vela's grooming workload decreases as CURATORs handle artifact-level maintenance.
- The Required CURATOR list creates a new auditing surface for the vault integrity auditor.
- The intergenerational memory model is formally documented as a three-mechanism triad.
