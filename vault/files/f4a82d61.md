## Context

Domain `tropo-ai.com` is registered via Cloudflare. No hosting attached yet. This is the first item that needs to land before web-pipeline can fire its first pipeline-run — the pipeline assumes a host with `git push triggers auto-deploy` mechanics, and several pipeline steps (verify-build, smoke-test-live, deploy-completion detection) are partially host-dependent.

This note is the placeholder for the research conversation Mike + Talos will have. It graduates to a setup project (`type: project`, child of [web-pipeline (7b2e94c1)](7b2e94c1.md)) once decisions are made — the setup project becomes the activation-root for the foundational substrate work that wires the host.

## Decisions to make

1. **Host selection.** Candidates to compare:
   - **Vercel** — Next.js-native, zero-config, push-triggers-deploy via GitHub integration. Free tier exists; paid tier scales with bandwidth. Made by the Next.js team.
   - **Cloudflare Pages** — Domain already on Cloudflare, so DNS is one less hop. Push-triggers-deploy via GitHub integration. Free tier generous. Next.js support via `@cloudflare/next-on-pages` adapter (some edge-case caveats around runtime APIs).
   - **Netlify** — Mature platform, push-triggers-deploy, decent Next.js support. No reason to prefer over Vercel or Cloudflare Pages unless there's a feature pull.
   - **Self-hosted (VPS + Node + reverse proxy)** — Maximum control, ~$5-20/mo, but you own ops (patching, certs, monitoring). Almost certainly overkill for a marketing site.

2. **DNS wiring.** Cloudflare-registered → host. The path depends on host:
   - Cloudflare Pages: trivial (same provider).
   - Vercel / Netlify: CNAME or A-record from Cloudflare to host's edge endpoint.

3. **CI/CD mechanism.** All modern hosts default to `git push to main triggers deploy`. Confirm that's the pattern; web-pipeline assumes it.

4. **Env vars.** Tropo-ai.com may need at least `ANTHROPIC_API_KEY` (if any AI features ship to the marketing site), plus host-specific build vars. Where they live (Vercel dashboard, Cloudflare secrets, .env via host UI).

5. **Branch strategy.** Production deploys from `main`? Preview deploys from PR branches? web-pipeline step-3 (commit-and-push) needs a target_branch declared.

6. **Deploy URL pattern.** `https://tropo-ai.com` as the production URL. Preview deploys at `<branch>-tropo-ai.<host>.app` or similar — useful for review without affecting prod.

## Best practices to cover

- Cache headers / CDN behavior (especially relevant for web-pipeline step-4 smoke-test-live cache-bust concerns).
- Build performance (Next.js 16 + Turbopack defaults; whether to use ISR/SSR/SSG).
- Image optimization (host-provided vs `next/image` defaults).
- Analytics + observability (Plausible / Vercel Analytics / Cloudflare Analytics).
- Robots.txt + sitemap.xml (host config or app/ route).

## Recommendation seed (Talos's lean — Mike redirects as needed)

**Lean: Vercel.** Next.js-native, zero ops, push-triggers-deploy out of the box, preview deploys per PR built-in. Cost is minimal for a marketing site (free tier likely sufficient until real traffic). The "same-provider-as-Cloudflare-Pages" argument loses to "same-vendor-as-Next.js-team." DNS wiring from Cloudflare → Vercel is a 5-minute CNAME edit.

**Second-lean: Cloudflare Pages.** Tempting because the domain is already on Cloudflare. Real risk: `@cloudflare/next-on-pages` adapter has known limitations for Next.js features that use Node runtime APIs. If we stay on the marketing-site happy path (static + light SSG), it's fine. If we ever surface platform features (live AI, streaming), Vercel removes that risk entirely.

Mike to decide; this is a conversation, not a commitment.
