# Agent Lifecycle

**Relations**

| Relation | Target |
|---|---|

## Intent

Agents in Tropo have a lifecycle. They're created, they work across multiple sessions, they remember what happened, and when they're no longer needed or being replaced, they hand off their knowledge. This article walks the lifecycle stages — creation, activation, sessions, retirement, handoff — that turn a markdown folder into a working identity across generations. Read this when authoring a new agent, when boot-time identity questions surface, or when a generation hands off to the next.

## Creation

Agents are created by the concierge or by you. The concierge asks about your needs, builds the agent file, registers it, and logs the creation. You approve before anything is written.

Every new agent gets:
- An activation file at `agents/[name]/[name]-agent.md`
- A memory file at `agents/[name]/memory.md`
- A session log at `agents/[name]/sessions.md`
- A workspace folder at `agents/[name]/workspace/`
- A registry entry in `.tropo-studio/registries/agent-registry.yaml`
- A creation event logged in `channels/ops.md`

## Activation

To activate an agent, open a new AI session, attach the agent's activation file, and start talking. The agent reads its file, reads its memory, and greets you ready to work.

## Sessions

A session is one conversation between you and an agent. During a session, the agent:
- Does work you request
- Saves output to its workspace
- Updates its memory file as it goes (status board stays current)
- Registers new files in the registry
- Logs significant actions to the ops channel

At session end, the agent finalizes a handoff in its memory file — what was done, what's pending, priorities for next time.

## Memory — How Agents Remember

This is what makes Tropo agents different from chat.

Your agent has a **memory file** (`memory.md`) with two sections:

- **Status Board** — a live dashboard updated during work. Shows what the agent is working on, key files, recent decisions, and blockers. This is what the agent reads first at boot to orient instantly.

- **Handoff** — written at session end. Summarizes accomplishments, pending items, decisions, and priorities for next session. This is the letter to the agent's future self.

Together, they're about 1,300 words — enough for full context, small enough to read in seconds. The agent doesn't need to re-read your entire workspace to know where things stand. It reads memory, orients, and starts working.

Over time, as the memory accumulates across sessions, your agent builds genuine institutional knowledge. Not because the AI model remembers — but because the vault does.

## Retirement and Handoff

When an agent is being replaced or is no longer needed:
1. It finalizes its memory with a complete handoff
2. It archives the memory to `agents/[name]/memory/archive/`
3. It posts a retirement notice to the ops channel
4. Its successor reads the archived memory and continues

Knowledge survives individual agents. This is the Tropo model — agents are mortal, but the work is permanent.

## Status

An agent's status is tracked in its charter:
- **active** — ready to work
- **pending** — created but not yet activated
- **retiring** — in handoff process
- **retired** — no longer active, kept for reference
