# Grooming Playbook — Backlog Review and Cycle-Commitment Protocol

*The protocol for moving work-items between phases of the Tropo Work pipeline. Run by Vela (or designated PM agent) with Mike — typically at cycle kickoff. Per [The Patient Honing Doctrine](../../.tropo-studio/memory/patient-honing-doctrine.md): one improvement at a time. Grooming surfaces candidates individually; doesn't bulk-commit.*

---

## When to run this playbook

- **At cycle kickoff** — deciding what to commit from backlog into the new cycle's release plan
- **When backlog grows beyond comfortable browse-size** — keep the standing project surveyable
- **On Mike's direction** — ad-hoc grooming sessions
- **After a finding-rich event** — cold-boot returns, walker returns, post-ship testing surfaces items; trigger a small grooming pass to capture them

## Required Reads

Before running:

1. [v1.4.2+ Backlog Project (c3e9f12a)](../files/c3e9f12a.md) — current backlog state
2. [The Patient Honing Doctrine](../../.tropo-studio/memory/patient-honing-doctrine.md) — grooming discipline
3. [Vault-inbox project (2d5f9b04)](../files/2d5f9b04.md) — upstream catch surface (any items here that should promote to backlog?)
4. The current cycle's release plan (if one is open) — what's the active scope to commit into?

## Steps

### 1. Survey the backlog

Query: all `member_of: [<backlog-project-uid>]` AND `status: requested` tasks.

Mechanically:
```
grep '"member_of":\[[^]]*"<backlog-uid>"' vault/00-index.jsonl | grep '"status":"requested"'
```

Or read the cascade index: `cat vault/00-cascade-<backlog-uid>.jsonl | grep requested`.

### 2. Survey the vault-inbox

Query: all items in vault-inbox (2d5f9b04 in fresh installs; 8a4c9e15 in argo-os) that haven't been triaged into either backlog or active cycle.

These are upstream items — first triage decides whether they're real backlog candidates or just orphan capture.

### 3. For each item — present + decide

Walk each item with Mike (or sponsoring agent for Mike-not-available cases). For each:

- Read the item's intent + estimated effort
- Surface its source (cold-boot N findings, walker P2, mike organic, architectural debt, etc.)
- Surface dependencies (does it depend on other items? on ADR signs? on cycle resolution?)

**Decision tree:**

| Decision | What to do |
|---|---|
| **Promote to current cycle** | Update `status: requested → accepted`. Add the cycle's release-plan UID to `member_of:`. The item is now committed for execution. |
| **Keep in backlog for next cycle** | Tag with target version (e.g., `target_cycle: v1.4.3`). Status stays `requested`. Item is tracked, not committed. |
| **Reject** | Update `status: rejected`. Add rejection rationale to the body's Activity log. Item is closed. |
| **Decompose** | Author smaller child task entries; update parent task `status: superseded` with `composes_into:` pointing at children. Children get the appropriate disposition (promote / defer / etc.). |

### 4. Log the grooming session

After all items reviewed, author a `note` vault entry:

```yaml
type: note
title: "Grooming session 2026-MM-DD"
member_of: [<backlog-project-uid>]
tags: [grooming, audit-trail]
```

Body: timestamped list of decisions made. For each item: UID + before-status + after-status + rationale. This is the audit trail; no decision unwitnessed.

### 5. Update related artifacts

- If items were promoted to the current cycle: update the release plan's stream(s) to reference the new committed members
- If a cycle is being scoped fresh: author the release plan + populate it with promoted items
- If items were rejected: ensure their rejection rationale is captured before closing

### 6. Rebuild ledger

Run `python3 .tropo/scripts/rebuild-vault.py --apply` so cascade indexes reflect the new memberships.

## Gates

- **Gate 1:** every backlog item reviewed (no skipped — patient honing means no item left unconsidered for too long)
- **Gate 2:** every decision logged in the grooming-session note (audit trail complete)
- **Gate 3:** rebuild-vault runs clean post-session (no broken references introduced)
- **Gate 4:** Mike-sign on the grooming session if cycle commitments were made (cycle-affecting decisions need principal sign)

## Verdict format

After grooming:
- `<N> items promoted to <cycle>`
- `<N> items deferred to <future-cycle>`
- `<N> items rejected`
- `<N> items decomposed`
- Grooming session note UID logged

## Anti-patterns to avoid

- **Bulk-promoting items** — violates one-improvement-at-a-time. Promote only what current cycle can absorb without bundling.
- **Skipping the audit trail** — every decision needs a record, or grooming becomes ad-hoc institutional drift.
- **Letting backlog grow without grooming passes** — the substrate's value is "we can find what's known." If backlog isn't groomed, items get rediscovered cold instead of remembered.
- **Grooming without Mike** for cycle-affecting decisions — strategic decisions are Mike's lane per the Cycle Kickoff Brief discipline.

## Patient honing note

Grooming is the substrate's act of patient honing at the work-pipeline level. Per the doctrine: *the workshop IS the system; the system compensates for forgetting.* Without grooming, items sit in backlog forgotten. With grooming, the substrate keeps them visible until they're either committed or closed. **Grooming is how the workshop remembers what to remember.**
