## Intent

Name the layers of the Tropo system so every agent, user, and prospect reads the same picture. Until today, "brain," "harness," "librarian," and "vault" were used loosely — overlapping, substituting, drifting. That ambiguity is now closed.

This spec also locks **Mike's L1-counterpart rule** (2026-04-19): every L2 Tropo service must have an L1 equivalent. L1 is the portability floor.

---

## The Stack — Four Layers

Read bottom-up. Each layer is a distinct concern.

```
┌─────────────────────────────────────────────────┐
│ Tropo-OS │
│ (the operating environment the agent enters) │
│ ┌──────────────────────────────────────────┐ │
│ │ L3 Tropo — external interop │ │
│ │ MCP, marketplace, third-party APIs │ │
│ ├──────────────────────────────────────────┤ │
│ │ L2 Tropo — active services │ │
│ │ The Librarian (first service) │ │
│ ├──────────────────────────────────────────┤ │
│ │ L1 Tropo — the governed substrate │ │
│ │ Library + Graph + Memory + Registry + │ │
│ │ Session Agents + Governance (markdown) │ │
│ └──────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
 ↑ agent enters Tropo-OS
┌─────────────────────────────────────────────────┐
│ Agent = Brain + Harness │
│ ┌──────────────────────────────────────────┐ │
│ │ Harness (Exoskeleton) │ │
│ │ Claude Code, Cursor, Cowork, Gemini │ │
│ │ CLI, Codex — gives the brain tools, │ │
│ │ sub-agents, web search, skills, python │ │
│ ├──────────────────────────────────────────┤ │
│ │ Brain │ │
│ │ The LLM — Claude, GPT, Gemini │ │
│ └──────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

### Layer 1 — Brain

The LLM. The reasoning engine. Claude, GPT-5, Gemini, etc. **"Brain" means LLM — nothing else gets to wear the word.**

### Layer 2 — Harness (Exoskeleton)

The agent's body and equipment. Provides the brain with tools, sub-agents, web search, python scripts, file access, a library of skills, shortcuts, and a working environment.

Examples: Claude Code, Cursor, Cowork, Gemini CLI, Codex, Claude Desktop, a fresh Claude.ai session.

A brain in a harness is powerful but generic — a minted MBA student with no company and no funding. It can do enormous amounts of work, but it has no durable role, no accumulated history, no team.

### Layer 3 — Agent

Brain + Harness. A reasoner with capabilities.

When people say "AI agent" generically (Anthropic, OpenAI, industry usage), they mean this: an LLM equipped to act via a harness. Powerful but rootless.

### Layer 4 — Tropo-OS

**The operating environment the agent enters and works within.** Not a layer on top of the harness — a workspace the agent walks into.

Analogy: the brain is a person; the harness is their body and equipment; Tropo-OS is the office building they walk into to work. The office building doesn't "sit on" the person. The person enters the office, is given a role, uses the office's rules and tools, and becomes someone specific there.

Tropo-OS provides: domain, rules, inline instructions, graph, memory, durable identity, crew, accumulated history. When the agent retires, the Tropo-OS persists — the next generation inherits it.

### A Tropo agent

**A Tropo agent = agent + Tropo-OS identity.** Not just a reasoner with capabilities, but someone with a role, a history, a crew, and durable continuity.

This is the term we use externally. "AI agent" is generic; "Tropo agent" is specific. The difference is the identity and the environment.

## Inside Tropo-OS — Three Levels

Tropo-OS has its own internal architecture. Three levels.

### L1 Tropo — The Governed Substrate

**Portable. Universal. Required.**

Everything at L1 is markdown + convention, readable by any LLM in any harness. An agent with L1 alone has a fully functional Tropo-OS workspace.

Five components:

1. **The Library** (currently `vault/` on disk; the concept is "The Library") — flat markdown, UID-addressed, schemaless document store per [ADR eda09d06](eda09d06.md). Every governed artifact — projects, tasks, decisions, design-specs, documents, releases, release-plans — lives here. Primary storage + retrieval.
2. **The Graph** — relational substrate. `member_of:` edges for containment, `refs:` edges for dependencies, cascade indexes for projection, [`00-project-tree.jsonl`](../00-project-tree.jsonl) for hierarchy. Per [graph-is-the-os](../../.tropo-studio/memory/graph-is-the-os.md).
3. **The Memory System** — agent-level + vault-level + short-term JSONL staging + rolling compaction per [spec d0624a04](d0624a04.md). Standing knowledge across generations.
4. **The Registry Topology** — matched primitives per domain ([adac1f10](adac1f10.md) — earn-the-abstraction). No single universal UID registry. Discovery surfaces: [`vault/00-index.jsonl`](../00-index.jsonl) for work artifacts, [`agent-registry.yaml`](../../.tropo-studio/registries/agent-registry.yaml) for agent class + identity, `registry.jsonl` for runtime callables (sa.\*/skills/tools), folder listings for kernel content (`.tropo/capsules/`, `.tropo/playbooks/`, `.tropo/skills/` — filenames are the addresses).
5. **The Session Agent Query Layer (sa.\*)** — specialists that load context once and serve queries freely. In-session acceleration without a dedicated L2 service. Per [vault/files/e863a1e0.md](../../vault/files/e863a1e0.md).

**Governance at L1** runs through [AGENTS.md](../../AGENTS.md) chain + [capsule definitions](../../.tropo/capsules/) + check-in protocol per [vault/AGENTS.md](../AGENTS.md). The LLM reads these and enforces governance through discipline.

### L2 Tropo — Active Services

**Optional. Accelerators. Not required for basic operation.**

The Librarian is the inaugural L2 service. It provides:

- Structured check-in/check-out (the Library's governed interface)
- Continuous compliance validation (rejects non-compliant writes automatically)
- Hybrid search (vector + keyword) over the Library
- Typed skill/tool/sa.\* lookup via the registry
- Filing and grooming assistance
- Security / write-scope enforcement

**The Librarian is not "an L2 brain."** It's a governance interface — a service that enforces what L1 declares. The brain (LLM) is still the reasoner; the Librarian structures and serves.

Other future L2 services follow the same pattern: markdown declares intent at L1, service enforces/accelerates at L2.

### L3 Tropo — External Interop

**Optional. Bridges.**

MCP bridges, marketplace publishing, third-party APIs (Linear, GitHub, Slack), ecosystem integrations. L3 is how a Tropo-OS workspace interoperates with the world outside it.

## Mike's L1-Counterpart Rule (locked 2026-04-19)

> **Every L2 Tropo service must have an L1 equivalent — a version that works by the agent reading markdown and enforcing governance through discipline. L1 is the portability floor. A capability that only works with L2 active is a portability violation.**

**Enforcement.** Before any L2 service locks, its L1 counterpart is named and working. The [Capabilities Matrix](e8b4d157.md) is the surface where this rule is checked — every row must have the L1 column populated (non-blank).

**Why.** Tropo-OS ships into contexts we don't control. Starter vault, Claude.ai cold-paste, marketplace distribution, harness-neutral deployments. If L2 is required for anything to work, the system isn't portable and the thesis fails.

## The Library (renamed from "ledger")

**Vocabulary lock (v1.2):** what we previously called "the Vault" is now called **"The Library."** The L2 service that governs it is **"The Librarian."**

**Filesystem path stays `vault/` for now.** Renaming the path is a vault-wide sweep (hundreds of references); it's scoped as a separate task (v1.3 or when convenient). In all new prose, spec text, matrix entries, and user-facing documentation, use "Library." When a path is cited literally, the path stays `vault/` until the sweep lands.

**Why the rename.** "Ledger" is accounting vocabulary — passive, line-item, financial. What the folder actually holds is organized knowledge for lookup, reuse, governance, and accumulation. "Library" is the word that already describes this class of system, and it pairs naturally with "Librarian" as the active service.

**What lives in The Library:**

| Entry type | Purpose |
|------------|---------|
| `project` | Graph node; container for work |
| `task` | Unit of work |
| `decision` (ADR) | Architectural or strategic commitment |
| `design-spec` | Locked architectural specification |
| `design-brief` | Proposed design, pre-lock |
| `document` | Reference material |
| `collection-ref` | Membership manifest |
| `board-def` / `board-snapshot` | Generated view + frozen render (per [74fd9b61](74fd9b61.md)) |
| `team-def` | Team identity |
| `playbook-run` | Per-execution state |
| `agent-configurator` | Activation artifact |
| `release` / `release-plan` | Ship coordination + record |

Every entry is UID-addressed. Any LLM can read the Library by path (`vault/files/<uid>.md`) or by index (`00-index.jsonl`).

For GTM, user onboarding, documentation landing pages, prospect conversations. One paragraph.

> *"A Tropo agent is an LLM (the brain) running inside a harness (Claude Code, Cursor, etc. — the agent's body and equipment) that has entered a Tropo-OS workspace. Inside Tropo-OS, the agent has a role, a history, a crew, and durable memory. Tropo-OS is built in three layers: L1 is markdown + governance any LLM can execute; L2 is active services that accelerate L1 (starting with the Librarian, which governs the Library); L3 is external interop (MCP, marketplace, APIs). ## Vocabulary Reference

Authoritative glossary. Use these terms in all new writing.

| Term | Definition |
|------|------------|
| **Brain** | The LLM. Reasons. Nothing else gets this word. |
| **Harness / Exoskeleton** | Claude Code, Cursor, Cowork, Gemini CLI, Codex, etc. Gives the brain tools, sub-agents, file access, skills, python, web search. |
| **Agent** | Brain + Harness. Generic "AI agent" in industry usage. |
| **Tropo-OS** | The operating environment the agent enters. Not a layer on top of the harness; a workspace. |
| **Tropo agent** | Agent + Tropo-OS identity. A reasoner with capabilities, a role, a history, a crew, durable memory. |
| **L1 Tropo** | Markdown + governance + graph + memory + registry + sa.\*. Portable. Required. |
| **L2 Tropo** | Active services. Librarian first. Optional accelerators. |
| **L3 Tropo** | External interop. MCP, marketplace, APIs. Optional. |
| **The Library** | L1 storage + retrieval. Renamed from "ledger." Filesystem stays `vault/` until a later sweep. |
| **The Librarian** | L2 governance interface over the Library. Check-in/check-out, compliance, search, filing, grooming, typed access. |
| **Capsule** | A governance schema. Declares how an entry of a given type must be structured. Lives at `.tropo/capsules/<name>.capsule.md`. |
| **AGENTS.md** | Per-folder governance. Declares who writes what, under what rules, in that folder. |
| **Vault** | The filesystem root of a Tropo-OS deployment (e.g., `argo-os/`). Contains the Library + governance + agents + channels. |
