# Create a Task

*Atomic action: create a single task entry in the Vault.*
*Reads the `task` capsule definition. Produces a valid task file and an index record.*

---

## 1. Intent

This action creates one new task entry in the vault's ledger. A task is a discrete unit of work with a clear owner, at least one parent project, a pipeline stage, and a verifier (for done tasks — owner ≠ verifier).

**When to invoke this action:**

The user says something like:
- "Create a task for me to do X"
- "Add a task to the Vault"
- "Log a task for Vela to review the registry"
- "I want to track X as a task"

**Do NOT invoke this action for:**
- Large bodies of work that span multiple sessions — use `create-project.action.md` instead.
- Notes, ideas, or references — use `create-note.action.md` instead.
- Architectural decisions — use `create-decision.action.md` instead.

**Reference:** The task capsule definition at [`.tropo/capsules/task.capsule.md`](../capsules/task.capsule.md) defines what a valid task IS. This action defines how to CREATE one.

---

## 2. Prerequisites

Before running this action, verify:

1. **The Vault exists.** `vault/` must be present at the vault root.
2. **The task capsule definition exists** at `.tropo/capsules/task.capsule.md`. If missing, the OS is incomplete — escalate to the vault principal.
3. **The vault index is writable.** `vault/00-index.jsonl` must exist and be appendable.
4. **The `vault/files/` directory is writable.**
5. **At least one project exists.** Tasks require `member_of:` with at least one valid project UID (capsule Rule 2 — no orphan tasks). If no projects exist, run `create-project.action.md` first.

If any prerequisite fails, stop and report the failure. Do NOT attempt to create the task with partial prerequisites.

---

## 3. Inputs

Gather these from the user. If a required input is missing, ASK before proceeding — do not guess.

### Required inputs

| Input | Type | Notes |
|-------|------|-------|
| `title` | string, ≤100 chars | Human-readable task name. Use imperative phrasing ("Do X", "Review Y"). |
| `description` | string, ≤120 chars | One-line summary. What is the agent being asked to do, in a single sentence? |
| `owner` | string | Agent or human identifier of who owns the task. Defaults to the invoking agent if not specified. |
| `member_of` | array of UIDs | At least one project UID. Each must exist as `type: project`. No orphans (capsule Rule 2). |

### Optional inputs

| Input | Type | Notes |
|-------|------|-------|
| `priority` | enum | One of `p0`, `p1`, `p2`, `p3`. If present, added to tags. |
| `tags` | array of strings | Up to 10 tags, each ≤30 chars, lowercase hyphen-separated. Merged with priority if present. |
| `due` | ISO 8601 date | Optional deadline. Not enforced — informational. |
| `stage` | enum | Defaults to `ideate`. May be set to `build` if the owner intends to start immediately. Must be one of: `ideate`, `build`, `verify`, `done`, `cancelled`. If set to `done`, `verifier:` is required and must differ from `owner:` (capsule Rule 1). |
| `relationships` | array | Typed references to other UIDs. Form: `[{rel: "references", to: "<uid>"},...]`. Use `depends-on` or `blocked-by` for blockers (capsule §Blocking). |

### Rule of thumb for gathering inputs

If the user says "create a task to review the registry, owned by Vela, in project `de5f0df9`," you have title, owner, description material, and member_of. You may need to ask for priority. Err on the side of one clarifying question rather than two, and err on the side of accepting defaults (priority = none, owner = requesting user, stage = ideate, state = active, tags = empty) rather than asking for every optional field.

**If the user does not name a project:** ASK which project this task belongs to. Do not default to a project — the user's intent matters. A task without `member_of:` is non-compliant and will fail validation.

---

## 4. Process

Execute these steps in order. Do NOT skip steps. Do NOT reorder steps.

### Step 1 — Generate UID

Produce an 8-character lowercase hex UID. Use `openssl rand -hex 4` or the equivalent cryptographic random function. Verify the UID does not already exist in `vault/00-index.jsonl` (grep for it). If collision detected, regenerate.

### Step 2 — Validate inputs

Check every input against the task capsule definition's validation rules:
- Title length ≤ 100 chars, no newlines, no tabs
- Description length ≤ 120 chars, no newlines, no tabs
- Owner is a known identifier (agent name or human name)
- `member_of:` present, non-empty, and every UID resolves to a `type: project` entry in the vault index
- If `priority` is present, verify it is one of `p0`, `p1`, `p2`, `p3`
- `stage:` is one of: `ideate`, `build`, `verify`, `done`, `cancelled` (defaults to `ideate`)
- `state:` is `active` (archived is a later lifecycle move, not a creation value)
- If `stage: done`, `verifier:` is present and differs from `owner:` (capsule Rule 1 — no self-verification)
- Tags: count ≤ 10, each ≤ 30 chars

If any validation fails, STOP. Report the failure to the user. Do not attempt to "fix" invalid inputs silently.

### Step 3 — Apply the template

Use the template in §5 below. Fill placeholders from the validated inputs. Generate `created` and `modified` as today's date in ISO 8601 format (`YYYY-MM-DD`). Set `file_ext: md` and `schema_version: 2`.

### Step 4 — Write the task file

Write the filled-in template to `vault/files/<uid>.md`. The filename is the UID. Nothing else.

### Step 5 — Append the index record

Add a JSONL record to `vault/00-index.jsonl`. The record must include all required schema fields:

```json
{"uid":"<uid>","type":"task","stage":"<stage>","state":"active","title":"<title>","description":"<description>","owner":"<owner>","member_of":[<project-uids>],"created":"<date>","modified":"<date>","tags":[<tags>],"file_ext":"md","schema_version":2}
```

The record must fit on a single line (no embedded newlines).

### Step 6 — Confirm to the user

Report back:
- The UID of the new task
- The location (`vault/files/<uid>.md`)
- The stage (typically `ideate`) and state (`active`)
- The project(s) it belongs to
- Any optional fields that were set

Example: *"Created task `a7e3d1f9` — 'Review registry for drift' (stage: ideate, owner: vela, priority: p1, member_of: [de5f0df9]). File at `vault/files/a7e3d1f9.md`, index record appended."*

---

## 5. Template

```markdown
---
uid: <generated-8-hex>
type: task
stage: <stage, defaults to ideate>
state: active
title: "<title>"
description: "<description>"
owner: <owner>
member_of: [<project-uid-1>, <project-uid-2>]
created: <today>
modified: <today>
tags: [<tags with priority if present>]
file_ext: md
schema_version: 2
priority: <if present>
due: <if present>
relationships:
 - rel: references
 to: <uid>
---

# <title>

## Intent

<One paragraph: why this task exists, what it will accomplish, and how success is measured.>

## What to Do

<Bullet list of concrete steps, or a numbered procedure the task owner will follow.>

## Verification

<How success will be verified. Who verifies it? By what criteria? If self-verification is acceptable (cold-boot-style only), state why.>

---

*Task created by <agent> | <date>*
```

If the user provided content for any of the body sections (Intent, What to Do, Verification), fill them in. If not, leave placeholders for the task owner to fill in later — do not fabricate content.

**Body sections** must match the capsule's `Standard Task Body` (Intent / What to Do / Verification). Do not substitute alternate headings — the capsule specifies these three.

---

## 6. Verification

Before declaring the action complete, the agent confirms:

1. **File exists.** `vault/files/<uid>.md` is present and readable.
2. **Index record exists.** The new JSONL line is in `vault/00-index.jsonl`.
3. **Field consistency.** Frontmatter in the file matches the index record field-by-field (uid, type, stage, state, title, description, owner, member_of, created, modified, tags, file_ext, schema_version).
4. **Schema compliance.** All fields conform to `schema_version: 2` per `.tropo/capsules/task.capsule.md`.
5. **Capsule compliance.** Validation checks pass:
 - `member_of:` present, array, ≥1 entry (capsule Rule 2)
 - `stage:` in enum, `state: active`
 - If `stage: done`, `verifier:` present and ≠ `owner:` (capsule Rule 1)
6. **UID uniqueness.** The new UID does not appear elsewhere in the index.

If any verification fails, report the failure to the user and recommend the appropriate fix (re-validate, re-write, or escalate to the vault principal).

---

## Failure Modes

| Failure | Action |
|---------|--------|
| UID collision | Regenerate UID and retry |
| Invalid input (length, enum, etc.) | Stop and ask user for correction |
| `member_of:` empty or references non-project UID | Stop and ask for a valid project UID. Do not proceed — orphan tasks are non-compliant. |
| `stage: done` with no verifier or verifier = owner | Stop and ask user for a different verifier. Self-verification is never valid (capsule Rule 1). |
| Ledger directory missing | Stop and escalate — the vault is not Phase 1 ledger compliant |
| Index file locked or unwritable | Stop and escalate — report to ops.md |
| Partial write (file written but index update failed) | Remove the orphaned file and report the failure; never leave the Vault half-committed |
