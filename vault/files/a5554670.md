## Purpose

Groom each subsystem hub touched by the release in flight, producing recommended drafts via [`sa.hub-groomer`](../../agents/sa/sa.hub-groomer/sa.hub-groomer.md) (worker + judge dual-mode swarm). The invoking pipeline-run executor reviews recommendations and writes the final groomed hub bodies + corresponding [`subsystem-registry.jsonl`](../../subsystem-registry.jsonl) rows per the Ship Gate ordering rule in [subsystem-hub.capsule v1.3 §Hub Update Discipline](../../.tropo/capsules/subsystem-hub.capsule.md).

**Position in dev-pipeline:** between step-4 (generate-release-notes; UID `804e339e`) and step-5 (produce-release-folder; UID `8654900a`). After release deltas are crystallized in release notes; before the vault is packaged into the ship folder. This ordering ensures registry rows + groomed hubs are PART OF the shipped release.

**First instance of the pipeline-step-with-sa.\* verifier pattern** locked in [v1.7 brief 6b5f7886 Decision 3 expansion](../../vault/files/6b5f7886.md). The generalized capsule (`pipeline-step-verifier.capsule` or similar) is formalized at v1.10 Enforcement; this step accumulates instance experience for that extraction.

## Inputs

- The release entry being authored (UID via run context).
- The release plan (UID via run context) — for `sub_systems:` array.
- The build entry (UID via run context).
- The release notes (output of step-4).
- All 6 active subsystem hubs (loaded from [`collections/subsystem-hubs.collection.md`](../../collections/subsystem-hubs.collection.md)).

## Procedure

For each subsystem hub listed in the release plan's `sub_systems:` array (or — under Correction 1 retrospective-not-predictive — for each hub the executor has empirically determined was touched by this release):

### Per-hub iteration

1. **Determine targets.** Map `sub_systems:` strings to hub UIDs via the live roster ([`collections/subsystem-hubs.collection.md`](../../collections/subsystem-hubs.collection.md)). Equivalence rule: `sub_systems:` strings match `subsystem_name:` exactly (canonical lowercase-hyphen form: `tropo-governance` / `tropo-rendering` / `tropo-work` / `tropo-agents` / `tropo-playbooks` / `tropo-library`).

2. **Author channel file** for this hub at `agents/dev-pipeline/activations/<run-uid>/groom-subsystems/<subsystem-name>-channel.md`. Initial content: header naming dispatch + verification context + section headers `## Worker 1 Draft` / `## Worker 2 Draft` / `## Judge Recommendation`.

3. **Dispatch via [`dispatch-sa-step.skill`](../../.tropo/skills/dispatch-sa-step.skill.md)** with parameters:
   - `sa_target: sa.hub-groomer`
   - `instance_count: 3` (2 workers + 1 judge)
   - `modes: [worker, worker, judge]`
   - `dispatch_params:` per-hub (target_hub_uid + release_uid + release_plan_uid + channel_file)
   - `verification_check`: judge wrote `## Judge Recommendation` + convergence verdict ∈ {HEALTHY, SUSPICIOUS} + unified draft covers 4 grooming surfaces (release_history row + Change Log entry + Current State + optional sections)

   The skill orchestrates: workers spawn in parallel → both `[DONE]` → judge spawns → judge `[DONE]`. Per hub: ~3 sa.* instances + ~1-3 minute wall clock per worker; judge is similar.

4. **Read judge recommendation** from channel file. Review for substantive issues. Convergence verdict guides action:
   - **HEALTHY:** proceed to write final groomed hub body + registry row. Most common path.
   - **SUSPICIOUS:** route both perspectives to reviewer-of-record (typically the dev-pipeline executor with optional Mike escalation). Step does NOT advance until human resolves.
   - **TRIVIAL:** re-dispatch workers with different framing prompts. Original channel file kept as historical; new channel file appended (or new sub-section).

5. **Write the final groomed hub** (executor performs, NOT sa.hub-groomer):
   - Update hub frontmatter: `last_release_reflected: <new-semver>`; append to `release_history:` array with `registry_uid: <pending>` placeholder.
   - Update hub body: append `## Change Log` entry; rewrite `## Current State`; update optional sections if present.

6. **Write the corresponding `subsystem-registry.jsonl` row** (executor performs):
   ```jsonl
   {"uid":"<NEW-8-hex>","subsystem_uid":"<hub-uid>","subsystem_name":"<canonical>","release_uid":"<release-entry-uid>","release_version":"<semver>","row_type":"per-release","summary":"<from judge recommendation>","streams_touched":[<from judge>],"created":"<ISO>","created_by":"<executor-id>"}
   ```

7. **Backfill `registry_uid:`** on the hub-side `release_history:` row from `<pending>` to the new registry row's UID. This closes the bidirectional pair per [v1.3 Ship Gate ordering rule](../../.tropo/capsules/subsystem-hub.capsule.md).

### After all hubs groomed

8. **Verify Ship Gate.** Per [release.capsule v3.3 Rule 11](../../.tropo/capsules/release.capsule.md): every subsystem in release plan's `sub_systems:` has a registry row with `release_uid:` matching this release. Soft-gate (v1.7-v1.9): WARNING if any hub missing; reviewer-of-record approves. Hard-gate (v1.10+): ERROR via validator.

9. **Log step completion** in run.jsonl + advance to step-5 (produce-release-folder; the registry rows + groomed hubs are now part of what produce-release-folder packages into the ship folder).

## Output

- N groomed hub bodies (one per touched subsystem; `<= 6` typical for substantial cycles).
- N rows appended to `subsystem-registry.jsonl`.
- N channel files in `agents/dev-pipeline/activations/<run-uid>/groom-subsystems/` (permanent history).
- 3N activation records in `agents/sa/sa.hub-groomer/activation-log/` (permanent history).
- 1 step_completed event in run.jsonl with verification verdict aggregate.

## Verification

Per Rule 11 (release.capsule v3.3) + Check 22:
- Each touched subsystem has a registry row with matching `release_uid:`.
- Each touched hub's `release_history:` references the registry row's UID.
- `last_release_reflected:` bumped on each touched hub.
- `## Current State` rewrite reflects post-ship reality (judgment call; reviewer-of-record reviews).

Soft-gated v1.7-v1.9; hard-gated v1.10+.

## Special cases

- **Empty `sub_systems:` (no subsystems touched):** step skips cleanly with `step_skipped` event noting "no subsystems touched per release plan." Most rare for substantive cycles but possible for pure-bugfix releases.
- **Hub creation/archive during release:** if a release creates a NEW hub (per [§Adding a New Subsystem](../../.tropo/capsules/subsystem-hub.capsule.md)) or archives an existing one, the step authors first-time `release_history:` rows or migration-window archived rows respectively. Both are within step scope.
- **TRIVIAL verdict on re-run:** if re-dispatched workers also produce TRIVIAL, escalate to human-review path. Don't re-run indefinitely.
