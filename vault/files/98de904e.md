*Terminal step. Writes back to vault-ops-spec, posts observability bulletin to ops.md, retires pipeline-run activation.*

**What it does:**
1. Composes DISPATCH bulletin (for scheduled run_type) or COMPLETE bulletin (for on-demand run_type) summarizing:
   - Run type + spec UID + activation UID
   - For scheduled: agents dispatched, agents skipped, alerts count, headline findings
   - For on-demand: substrate touches applied, evidence captured, headline findings
   - Bottom-appends to [channels/ops.md](../../channels/ops.md) per CAPSULE v1.21.0.1 write-order
2. Writes back to vault-ops-spec entry:
   - `closed_at`: ISO 8601 timestamp
   - `acceptance_evidence`: list of UIDs (sa.* activation-log records + run.jsonl event UIDs + post-condition substrate state)
   - Stage: `active` -> `done`
3. Retires pipeline-run activation entry:
   - `status:retired`
   - `closure_reason: cycle-complete` (or `closure_reason: cycle-partial-with-gaps` if step 4 forced skips with documented rationale)
   - `retired_at`, `retired_by: vela-v<N>`
4. If this activation was triggered by upstream (future composition - e.g., dev-pipeline triggers vault-ops-pipeline at some point): signals back upstream per coupling contract

**Observability is mandatory:** No silent close. Per the V48 dormancy lesson + Mike-V54 owner-doesn't-get-to-hide directive: every vault-ops-pipeline activation leaves at least one bulletin in ops.md. Silent success is failure of the doctrine.

**Failure modes:**
- ops.md write fails (filesystem error) - step retries; on second failure posts FLASH to alerts.md + manual surface to executive
- vault-ops-spec write-back fails - retry; on failure logs to run.jsonl with explicit close_failed event for manual cleanup
- pipeline-run activation entry write-back fails - same shape

**Pipeline-run terminal state.** When this step exits clean, the pipeline-run activation is retired; the vault-ops-spec is closed; the bulletin is on ops.md. Next activation starts fresh with a new vault-ops-spec.
