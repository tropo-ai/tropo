# ADR-002: File Safety Protocol

## Context

V3.1's activation required a permanent delete permission prompt from Cowork. This surfaced a gap: as the crew scales and child agents begin making vault changes routinely, permanent destruction of files without a recovery window is a real risk. Agents will make mistakes — wrong file, bad edit, mistaken delete. The vault needs a safety layer that mirrors how operating systems handle this: a recycle bin that holds deleted files temporarily before final destruction.

A second gap was also identified: child agents currently execute without a research phase. V3.1 dove straight into changes. A better pattern is for the child to map all affected files first, write a plan, then execute — giving Vela or Mike a review window before anything is modified.

---

## Decision

### 1. Recycle bin

All file deletions by any crew member or child agent are routed through `recycle/` before permanent destruction. No file is permanently deleted in a single step.

`recycle/` acts as its own root. A file's original path is preserved beneath it:

```
Original: Argo-OS/agents/vela-boot.md
In recycle: Argo-OS/recycle/Argo-OS/agents/vela-boot.md
```

This makes recovery unambiguous — you always know where a file came from.

A standing child agent — `vault-janitor` — runs weekly and permanently deletes everything in `recycle/`. Between janitor runs, all deleted files are recoverable. This window is the crew's rollback mechanism for mistakes.

### 2. V2 child agent protocol: research phase

Child agents executing file operations should follow a two-phase process:

**Phase 1 — Research.** The child reads all files it will touch. It writes a complete plan to its report file: what it found, what it will change, and any judgment calls. It then pauses — either waiting for explicit go-ahead, or proceeding automatically if the activation document specifies auto-proceed.

**Phase 2 — Execution.** The child makes all changes. Deletes route through recycle. On completion, the child updates its report with what was done, what was skipped, and any issues.

This two-phase approach gives Vela and Mike a review window on significant operations before files are modified.

---

## Recycle protocol

**Who moves files to recycle:** Any agent performing a delete. The agent is responsible for recreating the original folder path under `recycle/` before moving the file.

**Who empties recycle:** `vault-janitor` only. No agent should permanently delete files from recycle directly.

**Emergency recovery:** If a file is needed before janitor runs, Mike or Vela retrieves it from `recycle/` and restores it to its original path.

**What does NOT go in recycle:** Files being renamed (copy → delete old) may go through recycle or not, at the agent's discretion — the copy already exists. The recycle bin is primarily for files being removed without a replacement.

---

## The vault-janitor

A standing child of Vela V3. Activated once, runs weekly.

- **Location:** `agents/vela/children/vault-janitor/`
- **Task:** Permanently delete all contents of `recycle/`, preserving the folder structure itself
- **Reports:** `vault-janitor/reports/YYYY-MM-DD.md` — one report per run listing what was deleted
- **Schedule:** Weekly (Sundays, or first session of the week)

---

## Rationale

The recycle bin is the right pattern over per-file verify copies because:
- It's a single, well-understood location rather than scattered `.verify` files throughout the vault
- It only activates on deletes, not on every edit — lower overhead
- The folder structure preservation makes recovery unambiguous
- Weekly automated cleanup keeps it from accumulating indefinitely

The research phase is the right pattern for child agents because:
- It separates understanding from action — a better cognitive model for any complex task
- It produces an auditable record of what the agent intended before it acted
- It creates a natural review checkpoint without requiring a separate session

---

## Consequences

**What gets easier:**
- Recovering from accidental deletes
- Auditing what child agents changed and why
- Trusting agents with destructive operations — the safety net exists

**What gets harder:**
- Agents performing deletes must know the recycle protocol and implement it correctly
- Vault size grows slightly between janitor runs
- The vault-janitor must be running regularly or recycle accumulates

**What changes:**
- All activation documents for child agents performing deletes must include recycle instructions
- V3.1 is grandfathered — it ran before this protocol existed and had a manual backup as safety net
- Future child agents performing destructive operations follow this protocol by default
