## Intent

The tropo-ai.com website will host a blog as part of its Layer-0 product surfaces (Mike's framing during the project's authoring conversation: "host our blog, which will mirror what we publish on substack"). Substack is the authoring + distribution surface — that's where Mike publishes, where subscribers receive posts, where the comments + community live. Tropo-ai.com is the **owned mirror** — same content, our URLs, our branding, our SEO authority.

This brief specifies how Substack content reaches tropo-ai.com — the data flow, caching strategy, route shape, fallback behavior — so a subsequent deploy can implement the blog routes against a clean contract.

**Out of scope:**
- Implementation work (separate deploy after brief lock).
- Substack publication setup (Mike already owns or will own a Substack — outside this brief's surface).
- Comments / community features on tropo-ai.com (Substack stays the comment surface; the website mirror is read-only).
- Email subscription on tropo-ai.com (Substack handles its own subscriber list; we link to the Substack subscribe page rather than re-implementing).

## Decisions made

The following are committed by this brief (not open at implementation time):

1. **Source of truth = Substack's RSS feed.** Substack provides every publication a public RSS feed at `https://<publication>.substack.com/feed`. The feed includes title, publication date, author, full post content as HTML, and excerpt. This is the input contract.

2. **Render = native Next.js routes on tropo-ai.com.** No iframe of Substack. We pull RSS, parse into typed objects, render with our own components + brand. URLs are `tropo-ai.com/blog/<slug>` and `tropo-ai.com/blog` for the index.

3. **Caching = Next.js ISR with hourly revalidation.** Static at build, regenerated on demand at most once per hour. Acceptable lag — Substack publication frequency is well below hourly; visitors don't notice.

4. **Slug source = Substack's slug.** RSS doesn't include a clean slug field, but the post URL (e.g., `https://<pub>.substack.com/p/<slug>`) gives us one. We parse the slug from the URL and use it as our route's slug. Match Substack's URLs so backlinks from Substack to tropo-ai.com/blog work naturally — and so the SEO doesn't fragment.

5. **Subscribe link = link to Substack.** A "Subscribe" CTA on `/blog` and per-post pages routes the visitor to Substack's subscribe page. We don't try to capture email on tropo-ai.com for newsletter signups at v1 — Substack already does that better.

6. **No Substack-API-specific dependencies.** Substack doesn't have a stable public API; RSS is the only reliable contract. Implementation uses standard RSS parsing (`xml2js`, `fast-xml-parser`, or `rss-parser` npm package) — no Substack-vendor lock-in beyond the feed URL.

## Architecture

### Data flow

```
Mike publishes post on Substack
        ↓
Substack publishes RSS entry to <publication>.substack.com/feed
        ↓
Next.js ISR fetch (server-side, hourly cache TTL)
        ↓
Parsed into Post[] type, sorted by date
        ↓
Rendered at tropo-ai.com/blog (index) and /blog/[slug] (per-post)
```

### Routes on tropo-ai.com

- **`/blog`** — index. Renders: list of all posts, newest first. Each entry shows title, publication date, excerpt, link to per-post page. Pagination if/when post count exceeds ~20 (low priority for v1; pagination is a future enhancement).
- **`/blog/[slug]`** — per-post page. Renders: post title, publication date, author, full post HTML, "Subscribe to the Tropo Substack" CTA at the bottom linking to the Substack page.
- **`/feed.xml`** (optional) — re-export the Substack feed under our own domain. Useful for users who want to subscribe via RSS without going through Substack. Implementation: trivial proxy of the upstream feed. Low priority for v1.

### Implementation pattern (Next.js)

- **Fetch + parse:** `lib/substack-feed.ts` exports `getPosts()` and `getPostBySlug(slug)`. Both server-side. Fetches from `process.env.SUBSTACK_FEED_URL`. Parses XML via `rss-parser`. Returns typed `Post[]` (title, slug, publishedAt, excerpt, html, author).
- **Caching:** `revalidate: 3600` on `/blog` and `/blog/[slug]` routes. ISR handles refresh.
- **Rendering:** Substack RSS includes full post HTML. We render it directly in our own layout, with `prose` styling (Tailwind Typography plugin) to match Tropo brand. Inline images come through as `<img>` tags with Substack-hosted CDN URLs — those load fine over HTTPS.
- **Fallback:** if Substack RSS fetch fails, the route returns its last-cached value (Next.js ISR's stale-while-revalidate). If the cache is cold and the fetch fails (initial build moment), the route returns a graceful empty state ("Posts coming soon" or similar). Implementation should not throw on RSS failure; the rest of the site must keep serving.
- **Slug extraction:** RSS post URL is parsed at fetch time; we extract the last URL segment as the slug. Stored on the typed `Post` object.

### Environment variables

- `SUBSTACK_FEED_URL` — full URL of the Substack RSS feed. Server-side only (no `NEXT_PUBLIC_` prefix). **Value: `https://theagenticbuilders.substack.com/feed`** (Open Decision #1 closed 2026-05-15).

### Visual shape (UI)

The `/blog` index should feel like a publication, not a developer changelog. Aim:

```
[Tropo wordmark — ember]
[ Nav: Home · Blog · Download ]

The Tropo Blog

Notes on the operating system for human-AI teams.

──────────────────────────────────────────────

[Post Title — large, serif or sans]
Publication date · X min read
Excerpt of the post in muted ink...
Read more →

──────────────────────────────────────────────

[Next post...]

[Subscribe on Substack →]   <-- ember CTA at bottom of index
```

Bone background, ember accents, prose-styled post content. Type-driven. Matches the placeholder's brand voice and extends it for editorial content.

## Dependencies

**Substrate prerequisites:**

1. **Substack publication URL must be known.** See Open Decision #1.
2. **Substack publication has at least one published post** for routes to have content. If empty at implementation, the routes ship with the graceful empty state and post-arrival is automatic via ISR.

**Code dependencies (npm install at implementation):**
- `rss-parser` — battle-tested RSS feed parser.
- `@tailwindcss/typography` — prose-styled rendering of post HTML. Probably also useful for the Layer-0 download page (#10 brief).

**Engineering dependencies from this project:**
- `tropo-ai/website` Next.js app (✅ done)
- Brand tokens (✅ done)
- Vercel deployment + Cloudflare DNS (✅ done)

**No backend dependencies.** This is pure server-side fetch + render. No Supabase, no Resend, no auth.

## Open Decisions

1. **Substack publication URL.** ✅ **CLOSED 2026-05-15.** Publication: **The Agentic Builders** — `https://theagenticbuilders.substack.com/` — already established by Mike, with at least one published post ("Your Agentic Ambitions," 2026-04-28) that's Tropo-aligned content. RSS feed verified reachable + parseable. Content voice is Mike-as-agentic-builder writing about the space (of which Tropo is the central project), not a Tropo-product changelog — fine for v1 (Mike *is* Tropo). If a separate Tropo-product-blog publication ever becomes warranted as the company grows, the multi-source case is trivial: clone the design with a second `SUBSTACK_FEED_URL` and merge. Out of scope for v1.

2. **First post timing.** Owner: Mike.
   *Question:* will the blog have content at launch, or ship with an "empty state" page? Either is fine; just affects how the launch reads.
   *Recommendation:* publish one or two anchor posts on Substack before the blog routes go live (a "Welcome to Tropo" + maybe a short "What is Tropo" piece). Launching `/blog` with zero posts is a missed moment.

3. **Comments on tropo-ai.com/blog/[slug] pages?** Owner: Mike.
   *Question:* should per-post pages show comments inline (e.g., embedded Disqus, or fetched from Substack)? Or just a "Discuss on Substack →" link?
   *Recommendation:* "Discuss on Substack →" link. v1 keeps comment-engine implementation out of scope; Substack is already the community surface; visitors who want to engage go there.

4. **Cross-linking strategy.** Owner: Mike.
   *Question:* when Mike publishes a post on Substack, does he link from the Substack post to the tropo-ai.com mirror? Or vice versa? Or both?
   *Recommendation:* Substack is for distribution + email + comments; tropo-ai.com is for SEO + product context. Mike's call which way primary links flow. Lean: posts on Substack include a "Read on tropo-ai.com" link near the top — gives readers who prefer browsing-without-subscribing a path; concentrates SEO authority on the owned domain.

5. **SEO canonical URL.** Owner: Mike.
   *Question:* when both Substack and tropo-ai.com host the same post, which is the "canonical" for search engines? Substack will set its own canonical to the Substack URL by default. Should tropo-ai.com claim canonical = itself, or honor Substack's?
   *Recommendation:* **tropo-ai.com claims canonical = itself.** We want SEO authority concentrated on the owned domain, not the rented one. This means search engines may dedupe in favor of one or the other based on signals (links, freshness) — usually fine. Setting canonical = self is the standard owned-mirror pattern.

- Tailwind Typography integration + Tropo prose styles — ~1 hour

Rough total: **~6-8 hours of implementation work** once the brief locks and the Substack URL is known.
