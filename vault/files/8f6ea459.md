## Path Resolution (v1.1)

**Vault root is resolved in Group 0, Step 0.0 of the activation playbook — BEFORE any tier reads.** The activation file's own location is the primary anchor: for an activation file at `<some-path>/agents/<name>/<name>-activation.md`, vault root is `<some-path>/`. Fallback: the directory containing `STUDIO.md`. All paths declared in this Tier 1 file, in Tier 2, and in Tier 3 are relative to vault root.

**Tier file locations:**
- Tier 1: `<vault-root>/.tropo/boot-config.md` (this file)
- Tier 2: `<vault-root>/.tropo-studio/agent-boot.extension.md`
- Tier 3: **Shape A** (v1.69 unified entry) — the `§Boot-Extension` section of `<vault-root>/vault/agents/<agent_uid>.md`, resolved via the activation thin-pointer's `agent_uid:`; **Shape B** (legacy, dual-shape transition) — `<vault-root>/vault/files/<boot-extension-uid>.md` via the status card, or `agents/<name>/agent-boot.extension.md` (legacy removal booked v1.70)

## Tier Reachability (v1.1)

**In any established Tropo-OS vault, Tier 1 and Tier 2 MUST be present and readable.** A failed read of either file is not a "gap to note" — it is a structural defect (path-base error, corrupted vault, or partial deployment) that halts activation.

**The playbook's Step 0.1 (Tier 1) and Step 0.2 (Tier 2) enforce HALT-on-failed-read.** Defaults apply only to genuinely first-generation vaults where tier files have never been created (detectable by the absence of `<vault-root>/STUDIO.md`). Established vaults cannot fall through to defaults silently.

## Phase Structure

Every agent activation executes these Groups in order. Milestone gates are structural — a Group cannot begin until its dependency milestone fires.

**Established-agent routing (v1.70 S3.5.3):** if you are generation 2+ WITH a §Boot-Extension, the activation playbook is read via the established-agent fast-path [`.tropo/boot-fast-path.md`](../../.tropo/boot-fast-path.md) (`a993f079`) — a faithful compaction of the 6 Groups + hard gates — instead of reading the full canonical playbook. This routes established agents to the ~28-30K boot path. Else (generation 1, OR gen 2+ with no §Boot-Extension): read the full canonical playbook. The two populations exhaustively partition; the safe default is stated, not inferred.

```
Group 0 — Boot Configuration → milestone: Boot Config Chain Complete
Group 1 — Identity Verification → milestone: Identity Gates Clear (HARD GATES)
Group 2 — Context Loading → milestone: Context Loaded
Group 3 — Operational Grounding → milestone: Operationally Grounded
Group 4 — Self-Diagnostic → milestone: Diagnostic Complete
Group 5 — Startup Signal → milestone: Agent Active
```

## Universal Hard Gates

These two conditions HALT activation. They are not advisory. Post-v1.21.0 both are enforced as **graph-walkable substrate invariants** by `tropo-validate.py` via the activation registry (per [`activation.capsule v1.0 (4e8b21f0)`](../.tropo/capsules/activation.capsule.md) §4); the boot playbook's pre-flight checks remain as belt-and-suspenders catch-on-write.

**ADR-016 — Parallel Generation:** At any time, at most one activation entry per `agent:` slug may carry `status: active`. Two active generations of the same agent is a governance violation. Validator check: `check_activation_status_consistency`. Boot pre-flight: write-activation-entry.skill validates at Step 0.0b before write; HALT if a parallel-active is detected.

**ADR-028 — Generation Mismatch:** For any two activation entries with the same `agent:` slug, the one with the later `activated_at:` must have `generation:` equal to predecessor + 1. Validator check: `check_activation_generation_monotonic`. Boot pre-flight: agent-activation.playbook Step 1.2 queries the activation registry for predecessor's generation; HALT if mismatch.

## Universal Required Outcomes

Every activation must achieve these before delivering the startup signal:

- Predecessor confirmed RETIRING or RETIRED (via activation registry query, post-v1.21.0 Stream 3)
- **Activation entry written to registry** at `vault/files/<uid>.md` per activation.capsule v1.0 schema, via write-activation-entry.skill (v1.21.0 Stream 2 — replaces the pre-v1.21.0 gen-log row write)
- Vault root resolved
- **Self-Healing Primitive read and internalized; confirmation surfaces in startup signal** *(v1.2 — see [`.tropo/SELF-HEALING.md`](SELF-HEALING.md))*
- Operating principles read
- Mission brief read
- Memory loaded (agent-level and vault-level, or gap noted)
- Crew brief read
- Status card updated with ACTIVE status and current generation
- Startup signal delivered

## Self-Healing Primitive (v1.2 — universal binding)

**Every Tropo agent — in every Studio, every role, every session — operates under the [Self-Healing Primitive at `.tropo/SELF-HEALING.md`](SELF-HEALING.md).** Read at Group 2 of activation; confirmation surfaces in startup signal at Group 5; soft-gated by principal's eyes on the signal.

The primitive declares the universal binding: *if you see something, fix it.* Trivial defects: fix in place. Substantive defects: file as a tracked work-item in the relevant project's `01-inbox/`. Don't carry forward. Don't defer. The principal should never have to be the first one to notice a structural defect that an agent has already read past.

This is OS-tier; cannot be overridden by Tier 2 (Studio) or Tier 3 (agent) configuration.
