## Scope of this folder

This folder (`agents/directors/`) holds **STANDING-variant** director agents only — the persistent crew-shared specialists with multi-generation lineage. As of 2026-05-17 the standing roster is `d.pm` (Pam), `d.ops`, `d.curator`.

**Companion-variant directors live elsewhere** — at `agents/<executive>/directors/<name>/`, owned by their dispatching executive. Example: Talos's `ui.director` + `critic.director` at `agents/talos/directors/`. This split keeps owner accountability crisp at the folder level; companion roster discovery happens via per-executive `directors/README.md` files and the rendered crew brief, not via this folder.

The type contract for all director variants (standing + companion + future shapes) is at [`director.capsule v1.0`](30137f44.md).

---

## What belongs in `agents/directors/d.<name>/`

Per the v1.21.0+ thin-pointer pattern, each standing director's folder contains:

| File / Path | Purpose | Required |
|---|---|---|
| `d.<name>.md` | Thin-pointer activation file — declares identity UIDs (charter_uid, status_card_uid, boot_extension_uid, agent_root_uid) + `director_variant: standing` per director.capsule | YES |
| `briefing-package/` or `transfers/` | Generation handoff substrate (briefing-package legacy; living-transfer current per v1.24.0 playbook) | YES if generation 2+ |
| `directives/` | Per-director operational directives (standing convention; e.g., `d.ops/directives/director-health-standards.md`) | Optional; per-director judgment |
| `session-logs/` | Running record of activation sessions | Optional; per-director judgment |
| `.tropo-capsule/memory/` | Memory substrate per the v3 memory architecture | YES if memory active |

The director's **canonical identity** (charter / soul / status / boot extension) lives at `vault/files/<uid>.md` per the UIDs declared in the thin pointer — NOT in this folder. This folder holds the activation entry point + operational substrate.

---

## Activation pattern

All director-class agents (standing AND companion) follow the universal polling-loop runtime per director.capsule v1.0 §Floor: every director's activation file directs the director to enter a continuous wake-call polling loop on boot, watch its assigned channels for incoming `[PENDING]` directives, respond, return to polling, and terminate only on `[SHUTDOWN]` from the dispatcher.

Standing-variant directors specifically:
- Boot via the full `agent-activation.playbook` (Tier 1/2/3 + soul + charter + status + memory)
- Then enter the polling loop (universal floor)
- Persist across crew sessions (multi-generation lineage)
- Are dispatchable by any executive (`scope: crew-wide`) — typically via Vela who owns the directors layer; some directors managed by other crew members (e.g., `d.curator` managed by Orpheus)
