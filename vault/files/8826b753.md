## What ships

`app/page.tsx` shrinks from **799 → 97 lines** (88% reduction) by extracting the chat session logic into a custom hook.

**New file:**
- `lib/use-chat-session.ts` (802 lines) — owns the full chat orchestration:
  - Message/input state, artifact state, file attachments, loading + research state
  - Tropo Power plan state (currentPlan, routerContext, pendingQuery, pendingAttachments)
  - The full SSE pipeline (parseSSELines helper inlined)
  - `onSend` — main query orchestrator with research/plan branching
  - `executePlan` — multi-step Tropo Power execution
  - `onPlanApprove`, `onRetrySteps` — plan callbacks
  - `onEscalationApprove`, `onEscalationDecline` — auto-escalation flow
  - `onFileSelect`, `onFileRemove` — attachment handlers
  - `onCopyArtifact`, `onDownloadDocx`, `onDownloadMarkdown`, `onDownloadPdf` — artifact callbacks
  - `onNewChat` — reset (extended to clear plan state too — D8 hygiene improvement)
  - Unmount-time abort cleanup via useEffect

**Modified:**
- `app/page.tsx` — now a thin orchestrator (97 lines): consumes `useAppShell()` + `useChatSession()`, holds the config localStorage state, registers the New Chat slot, renders the 3-pane grid. No useState/useCallback/useMemo declarations beyond `config`.

## Acceptance

- `app/page.tsx` ≤ 400 lines: **97 ✓** (target was 400; achieved 24%)
- Each extracted abstraction has a clear prop type: `useChatSession({ config, mode, powerMode, selectedKbPath, debugOn })` → typed return shape (chat state + callbacks)
- `npx tsc --noEmit` clean

## Notes

The hook is large (802 lines) by intent — it absorbs ALL the chat-related state because the state is densely interconnected (e.g., `onSend` writes to messages/artifact/sources/plan; splitting them would create prop-drilling between hooks). One coherent hook is cleaner than three artificial splits.

This same hook can be reused by `app/projects/[id]/page.tsx` at D10 — that page has a near-duplicate of this logic. Out of scope for D8; D10 absorbs.

Also: `onNewChat` now also clears Tropo Power plan state (currentPlan, routerContext, pendingQuery, pendingAttachments) — the pre-D8 code left those set, which could cause stale plan UI on the next conversation. Hygiene improvement caught during extraction.
