*The doing step. For scheduled fleet runs: calls fleet-ops.playbook (Phase 1 wrap). For on-demand: applies substrate changes per plan.*

**What it does (scheduled run_type) - Phase 1 wrap of fleet-ops.playbook:**
1. Invokes [fleet-ops.playbook v2.2 (768c610e)](../../playbooks/fleet-ops.playbook.md) Step 3 (dispatch Group A in parallel) - daily-vault-health + vault-janitor + freshness-monitor + channel-health-monitor + vault-index-monitor + reconciler; Monday-only weeklies if applicable
2. Awaits Group A completion
3. Invokes Step 3B (stalled playbook runs) + Step 3C (scheduled playbooks due check)
4. Invokes Step 3 Group B (sa.repair-agent) - reads Group A reports + applies repairs
5. For each dispatch: writes `dispatch_completed` event to run.jsonl with sa.* activation-log path + summary
6. Aggregates: writes `fleet_dispatch_aggregate` event at step end

**What it does (on-demand run_type):**
1. For each substrate_touch in plan:
   - Applies the change (file edit / state flip / substrate authoring per touch shape)
   - Captures evidence per declared verification_method (machine_executable_script output / deterministic_assertion result / structural_check validator output / agentic_review sa.* verdict / manual_walk note)
   - Writes `touch_executed` event to run.jsonl with target_substrate_refs + verification_evidence + result
2. Aggregates: writes `substrate_touches_aggregate` event at step end

**Phase 2 future (later cycle):** fold fleet-ops.playbook dispatch logic directly into engine; retire playbook. Pipeline-runtime can do what playbook does today.

**Failure modes:**
- sa.* dispatch fails - logged with reason; next dispatch continues (per fleet-ops.playbook Failure Handling); aggregate captures partial-run state
- FLASH posted to alerts.md mid-execution - step pauses; executive triages before proceeding
- substrate touch FAIL with rollback trigger declared in plan - rollback applied; touch logged as FAILED-ROLLED-BACK
- substrate touch FAIL without rollback - touch logged as FAILED; aggregate captures; step exits with partial-state for step 4 validation

**Hands off to:** [step 4 validate-evidence (8cc1b5ea)](8cc1b5ea.md). Step 4 is the anti-box-checking gate; partial-state from step 3 is OK as input but won't pass step 4 unless evidence is honest.
