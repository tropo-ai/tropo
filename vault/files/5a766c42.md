## The Meta-Principle — The Rule of Three

**Every abstraction in Tropo-OS is organized in three layers. Fewer is too specific. More is too complex.**

The Rule of Three is a guiding principle, not a counting rule. It governs the number of abstraction *layers* in any governed domain — not the number of items within a layer. The default is always three layers. Exceptions require explicit justification.

Evidence it is already operating throughout the system:

| Domain | Layer 1 | Layer 2 | Layer 3 |
|--------|---------|---------|---------|
| Namespace | `.tropo` (kernel) | `.tropo-studio` (org) | `.local` (agent/run) |
| Governance | Kernel invariants | Vault guardrails | Capsule definition |
| Boot config | OS layer | Vault layer | Agent layer |
| Security | Personal | Shared-trusted | External |
| Playbook binding | OS playbook | Vault binding | Run context |
| Agent identity | Soul | DNA | Generation |
| These principles | Foundation | Design | Governance |

The document itself is organized in three layers — the Rule of Three as organizing structure, not formula.

## Layer 1 — Foundation

*What Tropo-OS is built on. These cannot be negotiated away.*

### 1.1 — L1 Is the Runtime

**The LLM is the runtime. The filesystem is the state store. The markdown is the protocol. That is it.**

Everything in Tropo-OS is implementable at L1: filesystem + markdown + LLM reasoning. L2 tooling — schedulers, APIs, databases, protocol-native execution — is an acceleration layer, not a dependency.

**The test:** If a design requires infrastructure that doesn't exist at L1, it is either a thesis violation or a thesis test. These are not equivalent. A thesis test requires documented evidence of what would have to be true at L1 for the L2 dependency to become unnecessary. Without that documentation, it is a violation.

*Locked by Mike Maziarz, April 13, 2026: "We are building what looks like a relic. It is not. It is infinitely adaptable, composable, portable. That is the basis for federation."*

### 1.2 — The Protocol Layer

**Tropo-OS is not an application or a framework. Its design intent is to become the protocol layer for agentic work.**

TCP/IP did not ask permission from applications — it became the substrate and everything built on top. That is the design intent here: define the primitives, conventions, and governance patterns that agentic coordination requires, so that every team, tool, and platform can build on them. We are building on the protocols that already exist — TCP/IP, DNS, HTTPS, SSL — and Tropo-OS is designed to be the next layer up.

### 1.3 — Model-Agnostic by Design

**The vault is designed to run on any sufficiently capable language model. No model name is hardcoded. No provider capability is a dependency.**

In practice, the current boot protocol requires Sonnet or Opus-class reasoning — multi-document instruction following, YAML parsing, and stateful protocol execution. As models improve, the capability floor drops. The architecture has no ceiling below the model's ceiling. The goal is that any model capable of reading and reasoning about structured markdown can operate in a Tropo-OS vault.

### 1.4 — Starving for Oxygen

**Constraint is the forcing function for primitive innovation.**

Refusing to reach for infrastructure forces discovery of solutions at L1. This is not deprivation — it is discipline. Every time the architecture solved a problem with markdown and reasoning instead of a database or an API, it produced a more portable, more composable, more elegant result. The constraint is the competitive advantage.

### 1.5 — Lovingly Reverse-Engineering the Future

**We design today so our primitives plug into tomorrow.**

MCP, A2A, and the emerging agentic protocol stack are where the industry is heading. Tropo-OS studies those directions carefully and builds primitives that will map cleanly onto them when they arrive — without depending on them now. We are not waiting. We are building the compatible foundation.

## Layer 2 — Design

*How Tropo-OS is designed. These govern every architectural decision.*

### 2.1 — Intent Over Everything

**Intent is the primitive underneath the primitives.**

Every governed artifact declares intent: why it exists, what it is trying to accomplish, what success looks like. The pattern is always: **intent → outcome → rules → verification.** This chain is built into every capsule definition, every playbook, every task, every ADR. No governed artifact is complete without it.

### 2.2 — Fewer Primitives, Many Uses

**A small number of powerful primitives used in many combinations is more elegant and more durable than a large number of specialized types.**

The vault entry, the UID, the JSONL record, the markdown file, the capsule definition — these compose into everything. New primitives are created reluctantly and only when composition genuinely fails. Composition failure means: two or more people independently attempted a composition solution, documented why it didn't work, and agreed a new primitive was necessary. The bar is high by design.

### 2.3 — Markdown Over Schema

**When language can govern behavior, prefer language over structure.**

Tropo-OS uses three formats, each for its appropriate purpose:
- **YAML** — definitional objects that must be machine-parsed: file frontmatter, call records, binding files
- **JSONL** — structured event logs and indexes: `vault/00-index.jsonl`, `run.jsonl`, `playbook-runs.jsonl`. JSONL is machine-queryable by design — it is the right format for append-only audit trails and flat indexes
- **Markdown** — everything else, written for reasoning not parsing

Strict schema enforces compliance through rejection; language governs through reasoning and degrades gracefully when a rule doesn't fit a situation exactly.

*The short letter takes more time. Write for intent. Earn structure.*

### 2.4 — Composable Means Many-to-Many

**Primitives compose freely. No hierarchy of ownership. No parent-child lock-in.**

A project is a named scope that references entries — not a container that owns them. A playbook run belongs to many projects. A collection includes anything. Composability is many-to-many relationships between sovereign entries, not hierarchical containment.

**The core operating method:** When asking where something belongs, ask which project owns it — not which folder it goes in. The project is the organizational unit. The folder is storage. This is the fundamental shift from traditional folder-based project management. See [how-projects-replace-folders.md](vault/files/42f46a3b.md) for the full explanation.

### 2.5 — Private and Published

**Every layer of the system has a working surface and an exposed surface.**

The working surface is where agents draft, iterate, and reason. The published surface is what crosses the boundary into shared or external use. This applies at every tier: workspace vs. channel post; draft spec vs. locked ADR; `thread.md` vs. `playbook-runs.jsonl`. The OS governs what crosses that boundary and when.

### 2.6 — Machines Compose, Humans Experience

**Raw composability is machine-native. Human participation requires rendered experience.**

UIDs, JSONL, tags, and many-to-many relationships compose beautifully for agents and systems. Humans need boards, views, and navigable structure. The OS provides both: machine-authoritative primitives and a generated human experience layer.

The default is generated. Navigation surfaces (boards, collections, project views) are produced by agents from source data — not maintained by hand. The primitive is always sovereign; the view is always derived.

Where strategic human judgment cannot be generated — executive assessment, decisions waiting on the principal, flags that require contextual interpretation — curated sections are permitted. Curated sections must be explicitly demarcated (e.g., `<!-- source: curated -->`), owned by a named crew member, and kept minimal. The generated default is the discipline; curation is the earned exception.

### 2.7 — AI-First Means Humans Change Too

**An AI-first operating system requires humans to operate differently.**

The human is not the executor — execution is the agent's job. The human is the architect and the verifier: the one who defines intent, reviews outcomes, and underwrites meaning. This is not a reduction of human agency. It is a redefinition of where human agency is most valuable — bounded verification, not general oversight.

## Layer 3 — Governance

*How Tropo-OS governs itself and interacts with the world.*

### 3.1 — Three-Tier Inheritance

**Rules flow down through three tiers. Lower tiers extend. They do not remove.**

The kernel sets invariants. The vault sets organization defaults and constraints. The agent or run sets local specifics. Rules accumulate downward — a lower tier can add constraints but cannot remove those declared above. This is the phone-home protocol: every governed action traces back to a rule set above it.

### 3.2 — Defense in Depth

**Governance through language is intentional. Any single layer may fail. The layers are the resilience.**

Tropo-OS governs through reasoning, not enforcement — agents are instructed, not blocked. This is a deliberate choice that produces portability and composability. The known limitation is that linguistic governance can fail silently when a model misreads or skips a governance step.

The answer has three parts:
1. **Three independent governance layers** (kernel, vault, capsule) — failure at one layer does not eliminate governance
2. **Detectable failure modes** — milestone timeouts, step-started events, and verification receipts exist precisely to surface when governance was bypassed. Detectability is the complement to governance. What cannot be detected cannot be corrected. Design failure modes to be visible, not just recoverable
3. **Human gates at irreversible actions** — playbooks must declare explicit human-in-the-loop attestation for high-stakes, irreversible steps

The principles do not protect against everything. Detectability and human gates do the rest.

### 3.3 — Design for Discontinuity

**Every session ends abruptly. Design accordingly.**

Agents lose context. Models are deprecated. Sessions are interrupted. Every governed artifact is a complete, self-contained record — readable with zero external context. The generation log, living transfer, and `thread.md` are implementations of this principle. Build resilience into the structure, not the process.

### 3.4 — Observability Is Structural

**Every action is logged, traceable, and rollback-able. Not as a feature — as an invariant.**

The JSONL pattern, ops.md, the playbook registry, and verification receipts exist because observability is a first-class requirement of agentic governance. An action that cannot be traced cannot be verified. An action that cannot be verified cannot be trusted. Logging is not instrumentation — it is the proof that governance operated.

### 3.5 — Federation Through Portability

**A vault that travels is a vault that can join any team, any platform, any future protocol.**

Because Tropo-OS has no infrastructure dependency, a vault can be shared, forked, merged, and federated without migration overhead. The zip file is the unit of distribution. The filesystem is the integration surface. Federation is not a feature to be built — it is the natural consequence of building on L1.

### 3.6 — Governance Must Be Executable at Reading

**A governed agent must be able to determine, from the governance files alone, whether any proposed action is permitted, required, or prohibited.**

Without ambiguity. Without inference chains longer than the agent's working context. Without requiring state from a previous session.

If a rule cannot be followed by a stateless reader — an agent that has just booted with no prior context — it is not a Tropo-OS governance rule. It is a convention hoping to be followed.

This principle has a hard architectural implication: governance complexity has a ceiling. The point at which an agent following the rules cannot reliably determine whether it is in compliance is the point at which language-native governance fails silently. Every addition to the governance chain (new AGENTS.md, new CAPSULE.md layer, new meta-rule) must be tested against this ceiling. When a new agent, given only the governance files and no other context, cannot determine whether their proposed action is compliant — the governance has failed, not the agent.

*This principle is the first to break under scale. Watch it.*

## Decision Checklist

When making an architectural decision, test it against these questions:

1. **L1 test** — does it satisfy L1? If not, is this a documented thesis test or a violation?
2. **Rule of Three** — does the abstraction have three layers of *governance inheritance* (kernel → vault → agent/run)? This is about the inheritance structure, not the number of internal files or components. If it departs from three tiers, document why explicitly.
3. **Intent first** — does it declare intent before structure?
4. **Primitive discipline** — does it compose from existing primitives? If a new primitive is proposed, has composition genuinely been exhausted?
5. **Boundary clarity** — does it separate the private working surface from the published surface?
6. **Observability** — is every action logged, traceable, and rollback-able?

When a principle conflicts with practicality: name the trade-off explicitly. Do not silently violate a principle. Document the exception. The exception becomes the next principle's boundary condition.
