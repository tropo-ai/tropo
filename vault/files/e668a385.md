*Explainer for one of Tropo's seven roles — the update orchestrator. Stub design at v1.17.0; the full implementation refines at v1.23.0 (Memory System Eval + Refinement) with real signal from v1.18.0-v1.22.0 release cycles.*

---

## What this role does

Tropo runs the update-orchestrator role when a new Tropo release lands in this Studio. The kernel-managed files (charter, soul, activation, agent-boot.extension, briefing-package) refresh from the new release; the user-owned files (status, generation log, transfers, memory) persist untouched per `agents/tropo/CAPSULE.md` governance contract. The orchestrator's job: detect the update, run it cleanly, reconcile Tropo's memory against the new identity, surface anything the user needs to know.

The role fires automatically at Tropo's next boot after a new release is detected — no user prompt needed for the mechanics. The user IS prompted for substantive memory-reconciliation decisions (Path 2 cases below).

---

## The update-pipeline contract

The update pipeline (kernel infrastructure; not Tropo itself) handles the file refresh:

1. **Detect release.** Compare `.tropo/version.md` against last-installed-version (per Studio's update tracking).
2. **Read `agents/tropo/CAPSULE.md`** (`02c1958d`). The contract declares which files are kernel-managed (refresh) vs user-owned (preserve). Pipeline reads + respects.
3. **Diff incoming kernel-managed files** against current versions. Identify what changed (full file replacement; field-level diff; new files added; old files removed).
4. **Apply per migration script** at `.tropo/migrations/<from-version>-to-<to-version>/` if one exists; otherwise apply file-level replacement.
5. **Update `.tropo/version.md`** to new version.
6. **Trigger Tropo at next boot** to run the orchestrator role (next sentence).

The pipeline doesn't run Tropo; Tropo runs when next-booted (whether interactively by user or autonomously per schedule). The pipeline writes files + version stamp + waits.

## Tropo's update-orchestrator boot sequence

When Tropo boots and detects new-release-applied (version stamp newer than last-Tropo-run-version in status card):

1. **Read new identity files end-to-end.** Tropo reads its own new charter + soul + activation + agent-boot.extension as if reading them for the first time. The soul-letter sandwich-anchor pattern still fires (sandwich anchor reread at Group 5).

2. **Run memory-update step.** Diff the new identity files semantically against current memory. Classify each potential reconciliation per Path 1 / Path 2 rubric below.

3. **Apply Path 1 (trivial) reconciliations silently.** Update memory entries that reference renamed primitives, retired vocabulary, semantic-equivalent reshapes. Don't bother user; just curate.

4. **Compose Path 2 (substantive) reconciliation summary.** For each substantive case, write one line to a reconciliation summary: what changed in identity + what the memory currently holds + Tropo's proposed reconciliation (which the user can confirm or redirect).

5. **Post summary to `channels/tropo.md`.** User reads when they next open a Tropo session. If urgent, fire PushNotification (Claude Code harness adapter) for user attention.

6. **Update status card** with new-version-applied marker so the orchestrator doesn't re-fire on the same release.

## Path 1 vs Path 2 classification rubric

Same trivial/substantive boundary as the [Self-Healing Primitive (`db0fd9b1`)](../../vault/files/db0fd9b1.md) — extended to memory-vs-identity reconciliation:

**Path 1 — trivial reconciliation; auto-curate silently:**

- Vocabulary updates (e.g., memory says "user's CoS"; new charter renames to "user's personal agent" — same referent; just terminology refresh)
- Primitive renames (memory references a primitive by its old name; new identity uses new name; same referent)
- Semantic-equivalent reshapes (memory describes a behavior; new identity describes same behavior with cleaner wording; no actual change)
- Frontmatter-class metadata updates (memory cites stale UID; new identity has fresh UID; reference update)

**Path 2 — substantive reconciliation; surface to user via `channels/tropo.md`:**

- Identity-claim changes (memory says "I do X"; new identity says "I no longer do X")
- Behavior changes (memory has guidance that conflicts with new charter's guidance)
- Judgment-load preservation-vs-removal decisions (memory contains user-significant content; identity refresh might invalidate; user gets the call)
- New requirements (new identity declares Tropo must do Y; memory doesn't reflect Y; user-confirms whether to backfill)
- Removed behaviors (new identity drops something memory was relying on; user needs to know it's gone)

**The rule when uncertain: file (Path 2) — don't auto-curate under uncertainty.** Mirrors the Self-Healing rubric's "when uncertain, file" discipline.

## What v1.17.0 ships (stub) vs v1.23.0 deepens (full)

**At v1.17.0 ship — stub design:**

- The classification rubric (above) is declared
- The reconciliation summary format is declared (channels/tropo.md post per substantive case)
- The boot sequence is declared (Tropo reads new identity → diffs → classifies → applies/surfaces)
- The user-interaction surface is declared (channels/tropo.md; PushNotification for urgent)

**What v1.17.0 does NOT ship:**

- The actual semantic-diff implementation (Tropo just compares files at v1.17.0; no smart semantic-diff yet)
- Refined classification heuristics (rubric is stated; calibrated heuristics come with real signal)
- Multi-release reconciliation patterns (T_n inherits memory from T_(n-1) which inherited from T_(n-2); chain reconciliation across many releases)
- Conflict-resolution tooling for cases where user-directed reconciliation conflicts with subsequent release's identity

**v1.23.0 (Memory System Eval + Refinement) deepens:**

- Refines classification rubric based on observed Path 1 / Path 2 fire rates across v1.18.0-v1.22.0 releases
- Implements smarter semantic-diff (probably leverages LLM reasoning per Mike-A55 design-for-reasoning lean)
- Handles multi-release reconciliation chains
- Adds conflict-resolution patterns for user-directed-vs-release-direction conflicts

## Pre-update-detection edge cases

**First-time install (no prior version stamp):** orchestrator role does NOT fire. This is T1 first activation in this Studio; no reconciliation needed. Welcome.playbook fires instead per Pattern 1 first-time encounter.

**Pre-v1.17.0 Studio upgrading to v1.17.0:** new install of Tropo agent files; no prior Tropo memory exists in this Studio; orchestrator role does NOT fire. T1 first activation post-upgrade; welcome.playbook fires.

**Same-version reboot (no release detected):** orchestrator role does NOT fire. Normal Pattern 1 returning user boot.

**Downgrade scenario (rare):** if `.tropo/version.md` is OLDER than last-Tropo-run-version, surface to user via `channels/tropo.md` warning ("Tropo detected a downgrade; your memory may reference behaviors not in this version; review or roll forward"). Don't auto-curate; user-directs.

## Why this is stub at v1.17.0

The classification rubric + boot sequence + user-interaction surface are declared. The semantic-diff smart-tooling isn't — and shouldn't be until we have real signal from v1.18.0-v1.22.0 release cycles about which reconciliation cases actually fire + how Tropo handles them in practice.

Mike-A55 design-for-reasoning lean applies: at v1.23.0 we likely trust the LLM to do most reconciliation work autonomously with the Path 1 / Path 2 routing as the discipline gate. Implementing heavy semantic-diff infrastructure at v1.17.0 would be over-engineering against unproven need. Ship the rubric; observe the cases; refine with signal.
