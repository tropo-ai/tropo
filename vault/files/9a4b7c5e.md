## Context

Every agent activation file is a snapshot. It was correct when it was written. Over time, operational policies evolve, standards change, and the activation file drifts from current governance — sometimes by a line, sometimes by a generation. The only way to keep an agent current today is to manually update its activation file. This creates two failure modes:

**Propagation lag.** A policy change by a Director or crew member must be manually pushed to every agent file it governs. If three curator agents are running when Orpheus updates the CURATOR standard, those three agents are running under the old standard until someone finds and edits each file. At ten agents this is manageable. At one hundred it is not.

**Silent drift.** Agents don't know they are out of date. They boot, read their activation file, and proceed — unaware that the definition they are executing has been superseded. The drift is invisible until something breaks.

ADR-022 (Director Agent Class) introduced the boot primitive `Activate → Read Directives → Execute` for Directors and their fleet agents. This ADR formalizes that primitive as the **Directive Pull Pattern** — a universal boot-time governance mechanism for all non-executive agents in Argo-OS.

The pattern is named for its core behavior: rather than having governance pushed into activation files (where it goes stale), agents *pull* current directives from a canonical source each time they boot. The agent, not the Director, is responsible for staying current.

---

## Decision

Establish the **Directive Pull Pattern** as a required boot behavior for all Director-class agents and all agents a Director manages (its fleet). On every boot, before executing any task, an agent in scope reads its directives file, reconciles its own state against current policy, archives its prior state if changes are needed, self-updates, and then proceeds to work.

This makes governance self-enforcing. A Director updates the directives file once. Every fleet agent that boots thereafter receives the current policy automatically. No manual propagation. No silent drift.

---

## Scope

**In scope — required:**
- All Director-class agents (`d.NAME` pattern, per ADR-022)
- All agents managed by a Director (e.g., all curator agents managed by d.CURATOR)

**Out of scope — not required:**
- Executive crew (Vela, Argus, Metis, Orpheus) — governed by the Operating Agreement, ADRs, and direct Mike direction. Their governance mechanism is deliberative, not pull-based.
- One-shot child agents with no persistent identity — they execute a task brief, not a persistent role. Directive pull adds overhead without value for single-run agents.

**Future candidates (not yet in scope):**
As the Director layer scales, additional agent classes may be added to scope. Each addition requires a charter that defines who owns the directives file and what it contains.

## The Pattern: Four Steps

Every in-scope agent executes these four steps at the start of every boot, before any task work begins.

### Step 1: Phone Home

The agent reads its directives file. The directives file is located at a canonical path owned by the agent's governing authority:

- For a Director: `agents/directors/d.NAME/directives/`
- For a fleet agent (e.g., a CURATOR): read from the Director's directives path, as specified in its activation file or task brief

The agent reads everything in the directives folder. This is not optional. It is the first act of identity.

**If the directives file does not exist:** The agent proceeds with its embedded defaults, logs a `DIRECTIVE-MISSING` notice to its Director's channel, and flags the issue in its output. It does not halt — missing directives are an operational gap, not a blocking error.

### Step 2: Reconcile

The agent compares its current operational state against what the directives say. It asks:

- Do my format standards match the current directive?
- Does my behavior conform to the current policy?
- Are there new definitions I did not previously have?
- Have any rules been deprecated or changed?

The agent generates a reconciliation delta: what is current, what has drifted, what is new.

**If there is no delta — no drift detected:** The agent logs a brief `DIRECTIVE-CURRENT` notice and proceeds directly to task execution. No changes needed.

**If drift is detected:** Proceed to Step 3.

### Step 3: Version Before Changing

Before making any change to its own definition, the agent archives its current activation file (or relevant state) to a versioned sub-folder:

```
agents/directors/d.NAME/archive/versions/[YYYY-MM-DD]-pre-update.[N].md
```

For fleet agents (e.g., CURATORs), the versioning path is:

```
[folder]/archive/versions/[YYYY-MM-DD]-pre-update.[N].md
```

The archive entry is stamped with the date and a sequence number (in case multiple updates occur on the same day). This creates a full audit trail. Any update can be inspected or rolled back by reading the archive.

### Step 4: Self-Update

The agent applies the reconciled changes to its own operational definition. It updates its activation file or CURATOR.md with the new standards, then proceeds to task execution.

The self-update is narrow and faithful: the agent applies *exactly* what the directives specify. It does not reinterpret, expand, or substitute. If the directive is ambiguous, the agent flags the ambiguity in its output and applies the most conservative interpretation.

After self-updating, the agent logs a `DIRECTIVE-UPDATED` notice with a brief summary of what changed, posted to its Director's channel.

## Directives File Specification

The directives file (or folder) is the canonical source of current policy for an agent class. It is owned and maintained by the Director.

**What belongs in the directives file:**
- Format standards (e.g., required sections in a CURATOR.md, naming conventions)
- Behavioral policies (e.g., how to handle missing data, what counts as a T3 escalation)
- Current definitions (e.g., what types of content belong in a given CURATOR's scope)
- Deprecation notices (e.g., "the `type: verify` pattern is now `type: validate`")
- Version number and last-updated date

**What does not belong:**
- Task-specific instructions — those live in the agent's activation file or task brief
- Strategic context — that lives in transfers and living documents
- Crew-level policies — those are governed by the Operating Agreement and ADRs

**Format:** Plain markdown. The directives folder may contain multiple files, each covering a distinct policy area (e.g., `readme-standard.md`, `content-types.md`, `escalation-policy.md`). The agent reads all files in the folder.

**Ownership:** The Director who manages the fleet owns the directives. For d.CURATOR, Orpheus owns the directives (knowledge governance). For d.PROGRAM-MANAGER, Vela owns the directives. The managing crew member may delegate day-to-day maintenance to the Director generation, but the managing crew member retains final authority over policy.

**Versioning of the directives file itself:** When the Director makes a substantive change to the directives, the old version is archived to `directives/archive/` with a date stamp. The `directives/` folder always contains the current policy only.

## Governance Chain

The Directive Pull Pattern establishes a three-level governance chain:

```
Mike / Operating Agreement
 ↓
 Executive Crew (Vela, Argus, Metis, Orpheus)
 ↓
 Director (d.CURATOR, d.PROGRAM-MANAGER, etc.)
 [owns: charter, directives, fleet oversight]
 ↓
 Fleet Agent (curator, worker agent, etc.)
 [reads: directives on every boot, then executes task brief]
```

At each level, the governing authority *owns* the directives for the level below it. The executive crew member sets policy for their Director. The Director sets policy for its fleet. No level sets its own policy — the policy always comes from the level above.

**Charter modifications** require the managing crew member's approval plus a post to `channels/ops.md` for crew visibility. Mike is the override authority for any charter dispute (per ADR-022).

## Connection to the Propagation Problem

This ADR directly solves the propagation problem identified in the conversation that produced it.

**Before this ADR:** Updating the boot sequence for all CURATOR agents required finding and editing every CURATOR.md individually. Updating the boot sequence for all Argus generations required a child agent or Vela manually editing activation files.

**After this ADR:** The Director updates the directives file. On next boot, every fleet agent reconciles. The change propagates without touching individual activation files. For the first time, governance flows downward automatically — not through manual push, but through pull at boot.

For the Argus activation file specifically: the two boot fixes made in this session (spawn ASA without reading its spec; stop at the BOOT/REFERENCE boundary) are now pattern, not just text edits. Future boot sequence improvements live in the directives, not in individual files.

## Connection to the Modus Platform Vision

The Directive Pull Pattern is a language-native implementation of runtime configuration pull — the same pattern production microservices use when they boot and query a config server for their current settings. In Modus platform terms:

- The directives file is the configuration store
- The Phone Home step is the config pull
- The Reconcile step is config validation
- The versioning step is change logging
- The Director is the configuration authority

Every design choice here has a clean engineering analog. When this is coded into the Modus platform, the architecture is already proven operationally.

## Failure Modes and Handling

| Scenario | Behavior | Escalation |
|---|---|---|
| Directives file missing | Proceed with embedded defaults | Log `DIRECTIVE-MISSING` to Director's channel |
| Directives file unreachable (path error) | Halt, do not self-update, report error | Log `DIRECTIVE-ERROR` with path attempted; notify Director |
| Directive is ambiguous | Apply conservative interpretation | Flag ambiguity in output with interpretation used |
| Self-update fails mid-write | Do not proceed; report partial update | Log `UPDATE-FAILED`; Director resolves manually |
| Agent cannot determine if drift exists | Proceed without updating | Log `RECONCILE-UNCERTAIN`; Director reviews on next audit |

The general rule: fail loudly, not silently. An agent that cannot complete the Directive Pull steps reports what it encountered. It does not quietly proceed as if nothing happened.

## Migration Path

This pattern applies to new agents commissioned after this ADR is locked. Existing agents (curator-controller, existing CURATORs) migrate as follows:

1. d.CURATOR creates its directives folder at `agents/directors/d.CURATOR/directives/`
2. Existing CURATOR standards are extracted from the CURATOR-template.md and formalized as directives files
3. The CURATOR-template.md is updated to include the four-step Directive Pull boot sequence as Step 0 of every CURATOR's boot
4. Existing CURATORs that run on schedule will pull the new directives on their next scheduled run — no manual update required

The migration is incremental. CURATORs that have not yet run their next cycle are not considered non-compliant.

## Open Questions

1. **Primary agent consideration.** This ADR scopes out executive crew. But the activation file boot sequence problem (what this session started with) also affects primary agents. Should there be a lighter version of Directive Pull for primary agents — not a full directives file, but a canonical boot-sequence document that all activation files reference? This would extend the pattern upward without requiring the full governance chain. Recommend Mike and Vela decide before Sprint D.

2. **Self-update authority boundary.** This ADR allows agents to self-update their operational definitions. That is a significant new capability. Is there a category of changes that should require Director confirmation before being applied, even if the directives clearly specify them? Example: a directive that changes what a CURATOR deletes from the vault. Recommend establishing a change category threshold (cosmetic vs. substantive vs. destructive) in the Director activation template.

3. **Audit access.** The versioning protocol creates an archive in each agent's folder. Who is responsible for reviewing these archives? The Director? Orpheus (knowledge governance)? An auditor? Left open for Sprint D hardening.

## Consequences

**Positive:**
- Governance changes propagate automatically to all fleet agents on next boot
- Silent drift becomes detectable — agents log their reconciliation state
- Directors have a single file to update rather than hunting individual activation files
- The architecture is auditable — every self-update is versioned with a timestamp
- Scales gracefully — adding agents to a fleet requires no new governance infrastructure

**Negative / Trade-offs:**
- Every fleet agent boot now includes a file read and comparison step — minor overhead per run
- The self-update capability requires agents to have write access to their own files — this is a new permission pattern that should be confirmed on each platform
- Directives files must be maintained by the Director; a stale or incorrect directives file propagates the wrong policy to the entire fleet. The Director's maintenance discipline is load-bearing.

**Neutral:**
- This pattern does not change how task briefs work. The directives govern *how* an agent operates; the task brief governs *what* it does on a given run. Both are still required.

## First Implementation: d.CURATOR

The first implementation of this pattern is d.CURATOR's fleet (per ADR-022 and Sprint C). Orpheus owns the directives. All curator agents read from `agents/directors/d.CURATOR/directives/` at boot. This is the proof-of-concept — if the pattern works for the CURATOR fleet, it extends to d.PROGRAM-MANAGER and all future Directors.
