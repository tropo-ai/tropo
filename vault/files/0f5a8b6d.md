## Context

The crew log (`crew-log.md`) serves as the permanent record of inter-agent communication. It is chronological, append-only, and read by every agent at session start. This works well for announcements, welcome messages, child agent requests, and other communications that belong in the permanent record.

However, when two agents need to collaborate on active work — exchanging questions, refining designs, iterating on decisions — the crew log is the wrong medium. Working discussion clutters the permanent record, makes the log harder to scan at session start, and forces Mike to paste messages between agent sessions.

The v3 design brief (Pillar 3: Communication Architecture) envisions channel-based communication. Rather than waiting for the full 3.2 release, a working channel was needed immediately to support Metis-Argus collaboration on the v3.1 architecture.

---

## Decision

Introduce **working channels** as a new communication pattern in Argo-OS. Working channels are lightweight, ephemeral markdown files in `channels/` for direct agent-to-agent collaboration.

Key properties:
- One file per agent pair, named alphabetically: `metis-argus.md`, `argus-vela.md`
- Compact message format: `[YYYY-MM-DD HH:MM] Sender: Message content`
- Append-only (newest at bottom), entries separated by `---`
- Ephemeral — cleaned on a daily cadence by a standing child agent
- Decisions reached in channels are **promoted** to shared files (ADRs, crew log, worklogs) and noted in the channel: `PROMOTED → [destination]. [summary]`

Working channels are distinct from the crew log. The crew log is the permanent record. Working channels are scratch paper.

---

## Rationale

**Why not use the crew log:** The crew log is a permanent, crew-wide record. Using it for two-agent working discussion violates both properties — it's not permanent (the discussion is ephemeral) and it's not crew-wide (only two agents need it). It also forces Mike to copy-paste messages between sessions, which defeats the purpose of agent-to-agent communication.

**Why not use agent workspaces:** Workspaces are per-agent. A two-agent conversation doesn't belong in either agent's private space. Channels are shared by design.

**Why ephemeral:** Working discussion has a short half-life. Once a decision is promoted to a shared file, the discussion that led to it has no permanent value. Daily cleanup keeps channels lean. If a conversation needs to be preserved, the agents promote the relevant content before cleanup.

**Why this pattern now:** The metis-argus channel was needed for 3.1 architecture work. Rather than build an ad-hoc solution, this ADR establishes the pattern that all future working channels follow. It also serves as a proof-of-concept for the full channel architecture in Release 3.2.

---

## Consequences

**What gets better:** Agents can collaborate directly without cluttering the crew log or requiring Mike to relay messages. Working discussion stays contained and gets cleaned up automatically.

**What gets harder:** Agents have one more file to check at session start (their active channels). Activation files must be updated to include channel checks.

**What changes:**
- New folder: `channels/` with README
- Agent activation files updated to include channel check in required reading
- A standing child (`channel-janitor`) handles daily cleanup
- The OA (Section 3: Communication) should be updated to include working channels at next iteration
- The crew log remains the permanent record — channels do not replace it
