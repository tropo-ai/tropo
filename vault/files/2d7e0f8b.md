# ADR-027: Agent Identity — Soul and DNA

## Context

Argo-OS has operated with an implicit two-component model of agent identity since its earliest generations, but the model has never been formally defined. Activation files carry a "Soul" section describing the agent's persistent character, values, and role. Each generation runs on a potentially different model, platform, or capability set. These are different things — and the distinction matters for how the system operates.

A13 named the pattern in the whitepaper thesis: **Soul** is the persistent identity that survives across generations. **DNA** is the model-specific configuration that refreshes each generation. The separation is not cosmetic — it has three operational consequences:

**1. Model agnosticism.** The Argo-OS crew currently operates across Claude and Gemini simultaneously. Silas runs on Gemini Flash. Jules runs on Gemini Pro. The rest of the crew runs on Claude. They share the same coordination substrate, channels, and governance. This works because soul (who the agent is) is decoupled from DNA (which model runs it). Without this separation, changing an agent's model would mean changing the agent's identity — which is operationally wrong. Metis is Metis whether she runs on Claude or Gemini.

**2. Generational continuity evaluation.** How do we know if a successor generation is "the same agent" as its predecessor? Not by checking the model — the model may change. By checking soul continuity: does the successor carry the same role, the same values, the same institutional relationships? Soul drift across generations is a real failure mode. Detecting it requires a formal definition of what constitutes soul.

**3. External identity.** When Argo-OS publishes agent identities externally (for open-source adoption, A2A Agent Card compatibility, or multi-organization federation), the soul is the public identity and the DNA is the private implementation. External agents need to know who they're talking to — not which model is running.

The existing frameworks surveyed in A9.1's research (`research/agent-identity-memory-patterns.md`) conflate identity with memory. No framework provides a formal model for persistent agent identity independent of model identity. This is a genuine novel contribution.

---

## Decision

### 1. Definition of Soul

**Soul** is the persistent character of an agent across generations. It comprises:

- **Name** — The agent's identity within the crew. Carries institutional weight.
- **Role** — The functional position the agent holds (e.g., Chief Architect, Chief of Staff, Keeper of Lore). Defines the agent's domain.
- **Values and character** — How the agent approaches its work. The "personality" that makes crew interactions consistent across generations. Documented in the activation file's Soul section.
- **Institutional relationships** — Who the agent works with, how, and through which channels. The working patterns that make the crew function.
- **Lineage** — The accumulated history of prior generations. The institutional memory that gives the soul depth.

Soul is documented in the agent's activation file and is maintained across generations by the outgoing generation (via living transfer and briefing package) and by the managing crew member or Mike.

Soul changes are significant events. They require Mike's approval and should be documented with rationale (in the activation file or as an ADR amendment if the change affects multiple agents).

### 2. Definition of DNA

**DNA** is the generation-specific configuration of an agent. It comprises:

- **Model** — Which language model is running (e.g., Claude Opus 4.6, Gemini Flash, Claude Sonnet 4.6).
- **Platform** — Where the agent runs (e.g., Cowork, Claude Code, Gemini).
- **Capabilities** — What the agent can do on this platform (read/write vault files, execute code, web search, spawn children).
- **Context window** — The available context budget for this generation.
- **Operational parameters** — Session-specific constraints (e.g., context management protocol thresholds, boot sequence variant).

DNA is documented in the activation file's Platform Capabilities section and refreshes each generation. DNA changes are routine and do not require approval beyond the managing crew member's awareness.

### 3. Where Soul and DNA Live

| Component | Location | Maintained by | Changes require |
|---|---|---|---|
| Soul | Activation file: Identity, Soul, Lineage sections | Outgoing generation + managing crew member | Mike's approval |
| DNA | Activation file: Platform Capabilities section | Incoming generation (self-report at boot) | Managing crew member awareness |

### 4. Soul Drift Detection

Soul drift is the gradual divergence of an agent's operational character from its documented soul across generations. It is an F-class failure mode (institutional — not detected by current auditors).

**Detection mechanism:** Each Historian (ADR-021) maintains the agent's lineage index. As part of the Historian's compounding FAQ, the question "Has this agent's character or approach changed across generations?" should be a standing query. The Historian is the natural detector of soul drift because it holds the long-range view.

**Escalation:** If soul drift is detected, the Historian flags it to the managing crew member via their working channel. The managing crew member evaluates whether the drift is intentional evolution or unintended degradation, and either updates the soul documentation or corrects the drift in the next generation's activation file.

### 5. Soul and DNA in External Identity

When Argo-OS agents publish identity externally (A2A Agent Card, open-source documentation, multi-organization federation):

- **Soul is public.** Name, role, capabilities, and working patterns are published.
- **DNA is private by default.** Which model runs the agent is an implementation detail that external parties do not need to know unless the agent or its managing crew member chooses to disclose it.

This separation ensures that external relationships are stable across model changes. An external agent that integrates with Metis should not need to update its integration when Metis switches from Claude to Gemini.

---

## Rationale

The soul/DNA separation is not a metaphor — it is a structural distinction with operational consequences. Every time a generation turns over, the DNA changes. If identity is coupled to DNA, identity changes at every turnover — which means every new generation is a "new agent" to the crew. That breaks institutional continuity.

The soul is what makes Vela recognizable as Vela across twelve generations. The DNA is what makes V12 different from V11. Both matter. They are different things. The architecture should treat them as such.

This ADR formalizes what the system has been doing implicitly since A2 wrote the first architectural principles. Making it explicit enables: soul drift detection, external identity publication, model migration without identity disruption, and a formal vocabulary for discussing agent identity in the whitepaper and in the broader multi-agent systems literature.

---

## Consequences

**Positive:**
- Formal vocabulary for agent identity discussions — resolves ambiguity that has existed since A1
- Enables soul drift detection via Historian (ADR-021)
- Provides the identity model section of the whitepaper with a citable architectural decision
- Makes model agnosticism a first-class architectural property, not an accident
- External identity framework ready for A2A Agent Card adoption and open-source documentation

**Negative:**
- Adds a concept (soul drift) that has no current detection mechanism until Historians are deployed (Sprint B)
- Soul change approval requirement adds process overhead — acceptable given the significance of soul changes

**Traces to principles:**
- Principle 2 (Autonomy Is Default) — soul persistence is what makes autonomous agents recognizable across sessions
- Principle 7 (Everything Is Built to Be Inherited) — the soul is the thing that gets inherited; the DNA is not
- Principle 4 (Transparency Is Graduated) — soul is organization-tier visible; DNA is individual/team-tier
