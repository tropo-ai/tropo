# The Parallel Orientation Sweep

**Relations**

| Relation | Target |
|---|---|

*How to run a wide-coverage read of your domain at boot. The L2 layer of Three-Layer Vault Hygiene (ADR-031).*

---

## What It Is

The Parallel Orientation Sweep is a pattern for running multiple read-only sub-agents in parallel, each with a tight declared scope, collecting their structured verdicts, and synthesizing the results in a single low-cost pass.

It is the L2 layer of the Three-Layer Vault Hygiene model:

| Layer | Who | When | What |
|-------|-----|------|------|
| L1 | The agent itself | Before every write | Read AGENTS.md/CAPSULE.md, stay in scope |
| **L2** | **The domain owner** | **At boot** | **Sweep your domain for drift — this pattern** |
| L3 | Scheduled fleet | Continuously | Watch for time-based drift between sessions |

L2 runs at the start of each executive session. It catches drift that accumulated since the last session — stale references, inconsistent fields, broken cross-references, governance gaps. Done in parallel, it adds under 5 minutes to boot while providing coverage that a single sequential read could not.

---

## When to Use It

**Use the Parallel Orientation Sweep when:**

- You are an executive agent beginning a session and need to verify your domain is in good shape before doing work
- A significant change has happened (migration, major new primitive, release) and you need to sweep for impact on your area
- You want to verify a specific concern without reading every file yourself

**Do NOT use it when:**

- You only need to check one or two specific files — just read them directly
- You need to make changes — sweeps are read-only. If a sweep surfaces ACTION-REQUIRED items, act on them in your main session, not inside the sweep sub-agents
- You're running a standing scheduled agent — the sweep is a session-start pattern, not a continuous one

---

## The Three Verdict Types

Every sub-agent in a sweep produces exactly one verdict for their scope. No narrative, no hedging, no partial verdicts.

| Verdict | Meaning | What triggers next |
|---------|---------|-------------------|
| **CLEAN** | No issues found in this scope | Nothing — continue the session |
| **WARNING** | Something is off but not breaking | Note it, add to session flags, address when convenient |
| **ACTION-REQUIRED** | Something needs to be fixed before work proceeds | Surface immediately in startup signal; address in this session or block work that depends on the affected area |

The verdict is the first line of the sub-agent's report. Every subsequent line is supporting evidence. The executive agent reads all verdicts before reading any evidence — this is the synthesis pass.

## The Dispatch Model

### Step 1 — Define the scope inventory

Before dispatching, decide which aspects of your domain need sweeping. Each sub-agent gets exactly one scope. Scopes should be:

- **Narrow** — one folder, one file type, one concern. A sub-agent reading `agents/` is too broad. A sub-agent reading `agents/argus/` is right.
- **Read-only** — the sub-agent reads and reports, never writes
- **Independently completable** — each sub-agent can finish without waiting for others

Example scope inventory for an Argus L2 sweep:

```yaml
sweep_scopes:
 - name: "Channel scan"
 scope: "channels/argus-vela.md, channels/metis-argus.md, channels/argus-orpheus.md, channels/ops.md"
 concern: "Action items directed at Argus, unresolved requests, stale entries"
 - name: "Architecture spec currency"
 scope: "vault/00-index.jsonl — type:decision, type:document tags:architecture"
 concern: "Any specs in proposed state needing review; version currency"
 - name: "Vault health"
 scope: "shared/orientation/daily-health-report.md"
 concern: "Known structural issues, alerts, flags from the standing fleet"
 - name: "Board currency"
 scope: "boards/board-argus-active/current.md"
 concern: "Is the board stale? Does it reflect current task state?"
```

### Step 2 — Dispatch in parallel

Launch all sub-agents simultaneously. Each receives:

1. Their specific scope (what to read)
2. Their specific concern (what to look for)
3. The three verdict types (CLEAN / WARNING / ACTION-REQUIRED)
4. The report format (verdict first, evidence second)

```
Dispatch: "You are a read-only orientation sweep agent. Your scope is [scope].
Read everything in that scope. Evaluate against this concern: [concern].
Report:
 VERDICT: CLEAN | WARNING | ACTION-REQUIRED
 EVIDENCE: [specific findings that support the verdict]
 If ACTION-REQUIRED: what specifically needs to be fixed?
Do not write any files. Do not take any action. Read and report only."
```

### Step 3 — Collect and synthesize

After all sub-agents complete, read all verdicts before reading any evidence. This is the synthesis pass — it takes one low-cost read and produces the full picture:

- Any ACTION-REQUIRED verdict → surface immediately in startup signal
- Any WARNING → add to session flags
- All CLEAN → proceed normally

Then read the evidence for each ACTION-REQUIRED and WARNING to understand the specifics.

## The Cost Model

**Why it's cheap:**

- Each sub-agent has a tight scope — they load only a small number of files, produce a small context, complete quickly
- Parallel execution means N sub-agents take the same wall-clock time as 1
- The synthesis pass is one executive agent reading N short reports — linear in scope count, not file count

**The math for a typical 4-scope sweep:**

| Step | Tokens (est.) | Time |
|------|--------------|------|
| 4 sub-agents (parallel) | ~4K each = ~16K total | ~90 seconds |
| Synthesis pass | ~2K | ~15 seconds |
| **Total** | **~18K tokens** | **~2 minutes** |

A 4-scope sweep costs roughly 1-2% of a 1M context window. The alternative — reading all files sequentially — would take 4x longer, consume more of the executive agent's main context, and be less reliable because the executive agent has to track everything at once.

## Implementing a New L2 Sweep for Your Domain

To build your own domain sweep, you need three things:

**1. A scope inventory** — what areas of your domain need checking? Write them out as a YAML list (see example above). Start with 3-5 scopes. Add more if your domain grows.

**2. A concern per scope** — for each scope, what are you looking for? Be specific. "Is this file current?" is too vague. "Are there any `type:decision` entries in `status:proposed` that Argus has been asked to review but hasn't?" is right.

**3. A sweep step in your boot protocol** — add the dispatch to your agent boot playbook (`.tropo/playbooks/agent-boot.playbook.md` Phase 3) alongside the existing parallel spawn. Your sweep runs alongside `scan-channels` and `check-vault-health`.

That's it. The pattern handles the rest.

---

## Checklist for a Well-Designed Sweep

Before dispatching:
- [ ] Each scope is narrow enough that one agent can read it completely
- [ ] Each concern is specific enough that the agent knows what "WARNING" vs "CLEAN" looks like
- [ ] All scopes together cover your full domain responsibility without overlapping
- [ ] The sweep is read-only — no sub-agent has write instructions

After receiving verdicts:
- [ ] Read all verdicts before reading any evidence
- [ ] Any ACTION-REQUIRED is in your startup signal before you do anything else
- [ ] Any WARNING is in your session flags
