*For software engineers evaluating Tropo. Five honest questions, technical answers.*

---

## 1. "How is this different from a config file?"

A config file is static, declarative, and scoped to one tool. Tropo is:

- **Dynamic** — agents read, amend, and author it during a session
- **Multi-tool** — any LLM (Claude, GPT, Gemini, local model) can operate inside the same Tropo vault
- **Composable** — typed primitives (capsules) compose into projects, projects into subsystems, subsystems into the Studio
- **Versionable** — every file is markdown; git is your version control

The closest analogy: Tropo is to AI agents what `package.json` is to a Node project — except the package IS the project, the dependencies are the agents, and the install script is the agent activation playbook.

Concretely: a config file tells one tool how to behave. A Tropo vault tells a team of AI agents how to work together, with each other, across sessions, with a permanent audit trail.

---

## 2. "Where's the API?"

Markdown is the API. Tropo's runtime is the LLM you're already running — there's no separate Tropo daemon, no SDK, no client library to install.

For programmatic interaction:

- **Read** — parse markdown frontmatter + the indexes (`vault/00-index.jsonl` for the artifact index; `vault/00-graph-index.json` for relationships). Stable YAML schemas defined by capsule definitions at `.tropo/capsules/<name>.capsule.md`.
- **Write** — author markdown files following the relevant capsule's schema; assign an 8-character hex UID; place at `vault/files/<uid>.md`; run `rebuild-vault.py` to re-index.
- **Validate** — `python3 .tropo/scripts/tropo-validate.py` runs ~30 checks against substrate integrity. Exit code 0 = clean.

You can build tools on top of Tropo (the [Tropo Platform](https://github.com/tropo-ai/tropo-ai) is a Next.js app that reads/writes vault files via filesystem access). Those tools are clients of the markdown substrate — they don't replace it.

---

## 3. "How do I script against this?"

Several scripting surfaces ship with every Studio:

- **`.tropo/scripts/`** — Python scripts that ship with the kernel (validators, rebuilders, extractors, ship-artifact builder). Pattern: stdlib-only + standalone CLI + exit codes. Read `.tropo/orientation.md` for the catalog.
- **`.tropo-studio/scripts/`** — Studio-specific scripts you add for your team's automation. Same pattern.
- **`agents/sa/`** — session agents (sa.*). Short-lived task-specific agents your executive agents commission for parallel work (e.g., sa.skeptic reviews a draft; sa.cold-boot tests a new playbook).

For one-off automation: write a Python script that walks `vault/files/` or queries `vault/00-index.jsonl`. The vault is just files — standard Unix tools work (`grep`, `find`, `jq`, `yq`).

For recurring automation: define a `pipeline` capsule instance + author a playbook the AI runs to execute it. The pipeline becomes typed substrate other agents can query.

---

## 4. "What's the build process?"

Three build artifacts:

1. **Indexes** — `python3 .tropo/scripts/rebuild-vault.py` regenerates `vault/00-index.jsonl` + the graph index from frontmatter. Run after any substantive substrate edit. Idempotent; safe to run anytime. (Companion scripts: `rebuild-index.py` for the artifact index alone; `rebuild-project-index.py` for the project tree.)
2. **Runtime catalog** — `python3 .tropo/scripts/register-kernel.py` updates `.tropo-studio/registries/registry.jsonl` from sa.* / skill / tool source files. The catalog is what agents query at boot to learn what's available.
3. **Release** — `python3 .tropo/scripts/build-release.py` produces a downloadable `.zip` of the Studio's ship-artifact entries (used to publish Tropo-OS updates).

Single-gesture health check: `npm test` (wraps the validator + reports green/yellow/red). If you don't have Node, `python3 .tropo/scripts/tropo-test.py` works standalone.

For CI: any runner that can install Python 3.10+ and run scripts can validate a Tropo vault. Standard `pytest`-style failure mode.

---

## 5. "How does this integrate with my existing pipeline?"

Tropo composes with your existing stack as a substrate layer — it doesn't replace your CI, your data layer, or your API gateway.

Common patterns:

- **Git source-of-truth** — Tropo vaults are markdown files in a git repo. Standard PR/review/merge workflow. Vault rebuilds happen post-merge via CI hook.
- **CI validation** — run `npm test` in CI to catch substrate drift before merge (broken UID references, schema violations, etc.). Fast (~10s on a typical Studio).
- **API extraction** — write a script in `.tropo/scripts/` that walks the vault and emits JSON for downstream consumers (your dashboard, your data warehouse, your billing system). The vault is the source-of-truth; everything else is a projection.
- **Event hooks** — when a Tropo file changes (e.g., a task moves to `state:done`), a git hook or CI step can fire any downstream API. The state transition is the trigger.
- **Multi-repo** — Tropo vaults can be subdirectories of larger repos. Path resolution is filesystem-relative; no global state.

What Tropo does provide that your existing pipeline doesn't: a structured-markdown substrate that AI agents read, reason against, and write into — with version history, typed schemas, and a permanent audit trail. The substrate is the contract between you, your team, and the AI agents working alongside both.

---

## When this FAQ doesn't answer your question

The source-of-truth is the substrate itself: read `.tropo/capsules/<name>.capsule.md` for type definitions, `.tropo/scripts/` for the runtime tools, and `.tropo/orientation.md` for the navigation map. Tropo's design principle is "the substrate explains itself" — if a question isn't answered here, the answer lives in the file the question is about.
