## What ships

Hotfix on top of app-deploy-19 ([f1b8c7d3](f1b8c7d3.md)) — closes an infinite render loop in `useHeaderSlots` that surfaced on `/projects/[id]` after deploy-19 + the SSR hydration hotfix (commit `e8e5fe2`) landed. Designed by [sa.ui record 004](../../agents/sa/sa.ui/activation-log/004-talos-t6-record.md) (verdict `pass`).

### Root cause

Three layered defects compounded into the loop:

1. **`useChatSession` + `useProjectChatSession`** return fresh `{...}` literals on every render (no `useMemo` on the return). Any caller that captures the WHOLE returned object in a `useCallback` dep array gets a new callback identity every render.

2. **`app/projects/[id]/page.tsx`** does exactly that — `onNewChat` is `useCallback(..., [projectId, currentChatId, chat])`. So `onNewChat` invalidates every render, defeating the page-level `useMemo` patch that wraps `primaryActionSlot`.

3. **`AppShellProvider`'s `value` memo** had `breadcrumb` / `primaryAction` / `settingsMenuExtras` in its dep array. So any slot setState (which `useHeaderSlots` does on every effect fire) rebuilt the broad context value, re-rendering every `useAppShell()` consumer — including the page that just fired the effect. Loop accelerator.

`app/page.tsx` only survived by accident: it dot-accessed `chat.onNewChat` directly (a `useCallback(..., [])` value inside the hook with empty deps — fully stable identity). It never captured the whole `chat` object, so the loop didn't propagate.

### Source-level fixes (sa.ui's recommendation)

**`lib/use-chat-session.ts`** — wrapped the 31-property return in `useMemo` keyed on its actual contents. `useState` setter identities (`setInput`) omitted from deps per the React contract.

**`lib/use-project-chat-session.ts`** — same treatment on the 28-property return.

**`lib/app-shell-context.tsx`** — split the single `AppShellContext` into two contexts:
- `AppShellContext` — broad app state (theme / mode / power / KB / debug). Stable across header-slot churn.
- `HeaderSlotContext` — per-page chrome slots (breadcrumb / primaryAction / settingsMenuExtras + setters). Volatile by design.

`useAppShell()` preserved as an aggregate hook — reads from both contexts and returns the merged view for backward compat. Existing call sites (Header, page files) need no changes. `useHeaderSlots` now subscribes only to `HeaderSlotContext` — slot churn no longer invalidates broad app-state subscribers.

Removed the `// eslint-disable-next-line react-hooks/exhaustive-deps` comment on `useHeaderSlots`'s effect deps — with the upstream stability fix, the deps array is now naturally complete (added `setBreadcrumb` / `setPrimaryAction` / `setSettingsMenuExtras`).

### Caller-side memos retained for now

`app/page.tsx` (`useMemo` for `primaryAction`) + `app/projects/[id]/page.tsx` (`useMemo` for `breadcrumb` / `primaryAction` / `settingsMenuExtras`) — applied earlier in this debugging session before sa.ui's audit. Sa.ui's recommendation is to leave them in place initially (defensive, bisect-friendly) and remove in a follow-up cleanup pass after observing dev-mode stability.

## Cadence note

This hotfix surfaced the importance of using the sa.\* fleet for systemic analysis. The original deploy-19 went out clean by smoke-test standards but exposed brittle hook contracts that linear caller-side patching couldn't fix. The first three iterations (hydration fix, caller-side memos, deeper caller-side memos) each closed one symptom level without modeling the full chain.

Sa.ui's dispatch 004 was the first analysis that read the full call chain end-to-end and identified the source-level defects. Going forward:
- **Systemic surface bugs** (hook contracts, cross-file refactors, architectural change) → sa.\* dispatch first; Talos verifies the audit, writes the fix, Mike tests, push.
- **Single-file mechanical bugs** → Talos directly.
- **Trigger:** if patching the same symptom twice, that's the signal — stop, dispatch.
