## Context

The v3 design brief defined four pillars and four releases. Releases 3.1 (agent autonomy), 3.2 (communication), and 3.3 (self-healing) are complete. Release 3.4 was originally scoped as "Error taxonomy + recovery protocols + chaos testing."

During R3.3 implementation, A3 pulled the repair agent, verification loop, and circuit breaker forward from the 3.4 scope into the 3.3 completion spec. These are now operational. The remaining 3.4 scope needed re-evaluation to avoid building what already exists.

A7 re-scoped 3.4 to cover the structural and defensive capabilities that remain undelivered: failure taxonomy, compensating transactions, input validation, and chaos testing.

## Decision

Release 3.4 delivers four capabilities:

1. **Failure taxonomy** — 14-category classification of document-based failure modes, each mapped to detection method, repair tier, and escalation path. Published as a standalone reference document. Referenced by the repair agent for classification.

2. **Compensating transactions** — Pre-change snapshot protocol for coordinated multi-file changes (3+ files as a unit). Manifest + backup files in agent workspace. Manual rollback by protocol. 72-hour retention, then archive.

3. **Input validation** — Five-tier trust model for data entering the vault: parent agent (high trust, no validation), child agent (medium, structural validation), standing process (medium, format validation), external import (low, full validation), unknown source (quarantine). Enforced by convention, audited by governance validator.

4. **Chaos testing** — 10-category adversarial test suite run against a vault snapshot. Verifies detection coverage of the self-healing pipeline. Run once to validate, then quarterly.

## Rationale

The failure taxonomy formalizes what the auditors and repair agent already detect informally. Making it explicit enables consistent classification, trend analysis across audit cycles, and clear escalation paths. Without it, repair tier assignment is ad hoc.

Compensating transactions address the rollback gap that has existed since R3.1's 40-file migration. The protocol is deliberately manual — automated rollback in a document-based system is fragile and error-prone. The value is in the snapshot discipline, not in automation.

Input validation establishes a perimeter. The vault has no defense against bad data from child agents or external sources. The quarantine protocol prevents unattributed content from entering the active vault silently.

Chaos testing proves the self-healing pipeline works. Organic findings validate that the auditors detect real drift. Chaos testing validates that they detect failure categories they haven't encountered organically.

## Consequences

**Positive:**
- Every failure mode in the system has a name, a detection method, and an owner
- Multi-file changes have a rollback path for the first time
- External data cannot enter the vault without attribution and review
- The self-healing pipeline has measurable, repeatable validation
- Argo-OS v3 is complete — all four design brief pillars delivered

**Negative:**
- Compensating transactions add overhead to multi-file changes (snapshot creation time)
- Input validation adds friction to external imports (receiving agent must review)
- Chaos testing requires a child agent and vault snapshot — context cost per run
- BOUNDARY-VIOLATION detection (taxonomy row) requires a future auditor enhancement not built in 3.4

**Trade-offs accepted:**
- Manual rollback over automated rollback (simplicity over coverage)
- Chaos testing proves detection only, not repair (repair testing is future scope)
- Input validation is convention-enforced, not structurally enforced (consistent with Argo-OS's design philosophy — convention over code)
