## Context

ADR-012 established the Activation Support Agent (ASA): a sub-agent spawned by the *incoming* generation at boot to sweep the vault and produce an orientation report. This model has a structural flaw revealed during A13's activation: the sub-agent's security rules treat file-based activation instructions as untrusted embedded content. The spawn fails reliably. Even when it doesn't, the incoming agent — mid-boot with zero context — is the worst possible agent to produce the orientation artifact. They don't know which files matter, what's in progress, or where the landmines are.

The right agent to produce orientation for the next generation is the *outgoing* generation. They worked the vault all session. They know exactly what files are live, what's open, and what their successor needs on day one. This is the model used for the living transfer (ADR-014) and it works cleanly.

Two additional problems this ADR addresses:

**1. The crew brief carries too much.** It currently tracks generation numbers, agent states, and operational announcements — information that updates continuously and makes Vela's maintenance burden proportional to crew size and transition frequency. Generation numbers belong in each agent's own status card, not in a shared document every agent reads at boot.

**2. Generic vault health is a separate job.** Staleness audits, channel health checks, and cross-reference validation are vault-wide concerns that run independently of any agent session. They should run on a schedule, not at boot.

---

## Decision

### 1. The Briefing Package

Each agent maintains a `briefing-package/` folder under their agent directory:

```
agents/[name]/briefing-package/
├── CURATOR.md — curation protocol (manual; retiring agent follows at transition)
├── 00-index.md — the briefing document; primary orientation read for incoming generation
├── intake-feed.md — inbox; Vela and peer agents may post here pre-activation
└── archive/
 └── [gen]-briefing.md — copy of 00-index.md at time of [gen]'s retirement
```

**`00-index.md`** is the main artifact. It is written and owned by the current generation, covering:
- Session summary: what was accomplished, decisions made
- Files in scope: active files with state summaries (what they contain, what's pending)
- Open items: unresolved tasks with locations and any context needed to resume
- Channel state: outstanding messages requiring a response from the next generation
- Recommended reading order for the incoming generation
- Pointer to the daily vault health report (for vault-wide state)
- Pointer to the living transfer (for strategic and narrative context)
- Next priorities as the outgoing agent sees them

**`intake-feed.md`** accepts posts from Vela and peer agents before the next generation activates. The retiring agent reads the intake-feed before constructing the successor's `00-index.md`, incorporating relevant items. The intake-feed is then cleared (archived to `archive/[gen]-intake.md`) as part of the transition.

**`CURATOR.md`** is a protocol document, not an automated dispatch target. The retiring agent reads it at transition and follows it manually. It defines: what belongs in the briefing, how to archive, what to do with the intake-feed, and quality standards for the handoff. Active automated dispatch (via the Curator Controller, Sprint C) may be layered on later; it is not required for this protocol to function.

---

### 2. Transition Protocol (Retirement Step)

At the start of the retirement sequence, before the living transfer is marked FINAL:

1. **Read the intake-feed.** Incorporate any items from Vela or peer agents into the new briefing.
2. **Archive the current package.** Copy `00-index.md` to `archive/[current-gen]-briefing.md`. Copy `intake-feed.md` to `archive/[current-gen]-intake.md`. Clear both source files.
3. **Construct the successor's `00-index.md`.** Write a complete, curated briefing for the next generation. This is substantive work — it should take 10–15 minutes of session time.
4. **Point to the daily vault health report.** Include the path to the current report in the 00-index so the incoming generation can supplement with vault-wide state.

The briefing-package construction replaces Step 4 of the prior retirement protocol (Run the Argus-ASA). The living transfer (Step 1) remains separate — it carries strategic narrative and personal continuity that does not belong in the operational briefing.

---

### 3. Boot Sequence Change

The incoming generation reads, in order:

1. `agents/[name]/briefing-package/00-index.md` — curated session context from predecessor
2. Daily vault health report (path included in 00-index) — vault-wide health snapshot
3. `00-crew-brief.md` — lean standup + crew directory (see Section 4)
4. `design/architectural-principles.md` — architectural compass

No sub-agent is spawned at boot. The orientation is deterministic and pre-built.

---

### 4. Crew Brief Redesign

The crew brief is restructured to two sections only:

**Section A — Daily Standup** (Vela writes, updated daily): Mike's current priorities, active announcements, anything the crew needs to know today. Rolling 7-day window; older entries move to the crew log.

**Section B — Crew Directory** (stable; changes only when crew composition changes): one row per agent: Role | Name | Channel path | Status card path. No generation numbers. No inline state. Generation numbers live exclusively in each agent's own status card, which the agent maintains. An activating agent reads the crew directory to know who is on the crew and how to reach them; they read the status card if they need current state.

This eliminates the crew brief as a source of generation-tracking churn. Vela updates it when someone joins or leaves the crew, not when generations turn over.

---

### 5. Daily Vault Health Report

A scheduled task runs once per day and produces a vault health report at `shared/orientation/daily-health-report.md`. It covers:
- Staleness audit: files that haven't been updated beyond their expected cadence
- Channel health: ceiling compliance, archival status
- Standing auditor summary: any findings from the four auditors
- Blocking issues: anything requiring immediate attention

This replaces the generic orientation functions of the per-agent ASA. It is vault-wide, not agent-specific. Agent-specific orientation is the briefing-package's job. This aligns with and implements ADR-017 (Shared Orientation Layer).

---

## Rationale

**Why outgoing agent produces the briefing, not incoming agent spawns one:** The outgoing agent has all the context. The incoming agent has none. Orientation should flow from knowledge to ignorance, not the reverse. This is the same logic behind the living transfer — and the living transfer works.

**Why keep the daily vault health report separate:** Staleness audits, channel compliance, and cross-reference validation are continuous concerns that don't require a session transition to trigger. Running them on a schedule decouples vault hygiene from boot timing. Any agent at any point in their session can read the latest report.

**Why strip generation numbers from the crew brief:** Generation numbers change every time any agent transitions. The crew brief is read by every agent at every boot. Coupling a frequently-written field to a frequently-read document creates unnecessary churn and means the crew brief is almost always partially stale. Each agent knows their own generation; they don't need to broadcast it to the shared document every session.

**Why keep the living transfer separate:** The briefing-package is operational and forward-looking. The living transfer is strategic and narrative. An incoming agent needs both, but they are different in character and different in author intent. The briefing-package is instructions; the living transfer is handoff.

---

## Consequences

**What becomes easier:**
- Boot orientation is reliable — no sub-agent spawn, no security conflicts, deterministic read
- Crew brief maintenance burden drops significantly — Vela updates standup and announcements only
- Incoming agents get higher-signal orientation — curated by someone who knew the context, not swept generically
- Vault health monitoring decouples from boot — runs on schedule, available to any agent at any time

**What becomes harder / requires new discipline:**
- Retiring agents must invest session time in constructing the briefing-package. This is real work, not a template fill. Quality of the briefing directly determines quality of the next generation's boot.
- The intake-feed requires a clear write protocol (who can post, what format, when it gets cleared). Vela owns this protocol definition.

**What is superseded:**
- ADR-012 (Activation Support Agent): the spawned-sub-agent model at boot is retired. The daily scheduled report and per-agent briefing-package together replace the ASA's five functions.
- ADR-017 (Shared Orientation Layer): the daily vault health report implements and extends ADR-017's intent. The prior ADR-017 scheduled task (if running) should be evaluated for consolidation.

**Open items:**
- Bootstrap protocol for newly commissioned agents (no predecessor, no briefing-package yet): **Vela's protocol to define, in coordination with Mike.**
- Intake-feed write protocol: who can post, what format, how it gets cleared: **Vela's protocol to define.**
- Template for `CURATOR.md` in briefing-package: Argus produces this session.
- Template for `00-index.md`: Argus produces this session.
- All existing agents need briefing-packages created. Rollout order: Argus first (this session), then Vela, Metis, Orpheus on their next sessions.
