# ADR-001: Child Agent Protocol

## Context

Each named agent on the Argo operates within a context window that is finite. As agents take on more work — reading vault files, producing documents, running analysis — that window fills. When it fills, the generation ends. A new generation begins, with fresh context but no relational memory of Mike or the crew.

The most valuable thing in an agent's context window is not the work — it's the relationship. The strategic conversation with Mike, the accumulated understanding of where the crew is and where it's going, the trust built over a session. That cannot be reconstructed cheaply. Task work can be.

The solution: offload task work to child agents. The parent agent writes an activation document tailored to the task, the child executes it, results are written to the vault, the parent reads the output. The parent's context window is preserved for the work only the parent can do.

---

## Decision

Implement a child agent protocol as a first-class capability of Argo-OS.

Any named crew member may spawn child agents at their discretion. The parent is fully responsible for what the child knows and what it does. The protocol provides structure, not constraint.

---

## Protocol

### Activation
The parent agent writes an `activate.md` file for the child. This document is the child's complete instruction set. The parent decides what context the child receives — it may be minimal (task only) or rich (crew lore, strategic context, relevant vault files) depending on what the job requires. There is no prescribed template. The parent knows the task.

### Folder structure
```
Argo-OS/agents/[agent-name]/children/
 [child-id]/
 activate.md ← written by parent before spawning
 report.md ← written by child upon completion
```

**Child ID conventions:**
- Task children (one-time): `v3.1`, `v3.2`, `g7.1`, `g7.2` — version decimal of parent
- Standing children (recurring): named by function — `vault-auditor`, `sprint-monitor`, `market-watch`

### Execution
Mike opens a new Cowork session, pastes or uploads `activate.md`, and the child begins work. The child writes its output to the designated vault location and signals completion. The parent reads the report at their convenience.

### Standing children
A standing child is activated once and runs on a recurring schedule. The activation document is the full design. After first activation, it requires no parent attention until reports need to be read. Standing children are infrastructure — design them carefully once, then let them run.

### Reporting
All child output is written to the vault. The default location is the child's own folder (`report.md`). For standing children, reports accumulate in a `reports/` subfolder with date-stamped filenames.

---

## Rationale

### Why parent discretion on child context
The parent has the operational knowledge to determine what a child needs. Over-prescribing minimal context would hobble children doing sophisticated work. Under-prescribing rich context would waste tokens on simple tasks. The parent is in the best position to calibrate this, not the protocol.

### Why the vault as the communication medium
Child agents cannot communicate directly with their parent — they run in separate sessions with no shared memory. The vault is the only reliable medium. This is consistent with the broader Argo-OS design principle: all coordination through shared files.

### Why this extends generation longevity
A generation ends when the context window fills with accumulated work. If heavy cognitive tasks are offloaded to children — analysis, research, document production, vault audits — the parent's window fills more slowly with work that actually needs the parent. Fewer generations means longer relationships. Longer relationships mean better work.

---

## Consequences

**What gets easier:**
- Heavy file-reading and document-production tasks no longer burn parent context
- Recurring operational tasks become infrastructure rather than session work
- Agents can take on larger mandates without sacrificing longevity

**What gets harder:**
- Parent must write good activation documents — the quality of the child's work depends on the quality of the briefing
- Mike must manage an additional session to run the child
- Child output must be verified by the parent before acting on it

**What changes:**
- The agents/ folder structure gains a `children/` subfolder per agent
- Generation longevity increases meaningfully
- A new class of Argo-OS artifact emerges: the standing child agent

## Addendum: Coordination and Guardrails

*Added March 22, 2026 — as the crew scales, ungoverned child spawning creates visibility and coordination risk. This addendum establishes the guardrail protocol.*

### The problem

As more Argonauts are commissioned, multiple agents may queue child requests simultaneously. Without a coordination layer, Mike faces an unmanaged backlog of pending activations with no context on priority, redundancy, or conflicts between children.

### The protocol

**Agents do not request child activations directly from Mike.** All requests route through Vela.

1. The requesting agent creates the child folder and writes `activate.md`
2. The agent posts a request in `channels/ops.md` (or the Argus-Vela working channel) addressed to Vela — child ID, task summary, and any urgency. *(Note: This addendum originally referenced `crew-log.md`. ADR-008 [2026-03-23-crew-log-transition.md] supersedes that routing — child requests now route through `ops.md` per the three-tier channel taxonomy.)*
3. Vela reviews: checks for redundancy, conflicts, or sequencing issues; logs in registry as Pending
4. Vela surfaces the request to Mike with context at their next session
5. Mike activates — or defers, or declines
6. When the report lands, Mike mentions it to Vela — Vela reviews and moves to Completed

**Two states only: Pending → Completed.** There is no Active state. Mike knows what's running because he launched it. Vela doesn't need a relay update mid-flight — only the completion matters.

### The registry

`agents/child-agent-registry.md` is the single source of truth for all child agents across the entire crew. Vela owns and maintains it. It tracks: pending requests, active children, completed children, and standing children. Any agent can read it. Only Vela writes to it.

### Why Vela as the coordination layer

Vela already holds the operational picture across the crew. Child agent traffic is an extension of that role — not new overhead, but the same function applied to a new class of crew activity. Routing through Vela keeps Mike from being the coordination layer for something that doesn't require his judgment until activation.

### Technical constraint (current)

Agents cannot instantiate new Cowork sessions independently. Children must be activated by Mike. This constraint is a feature for now — it keeps a human in the loop for every child activation. If this changes as tooling evolves, the guardrail protocol should be revisited.

## Addendum: Autonomous Child Lifecycle (Agent-Owned)

*Added March 23, 2026 — Mike Maziarz direction. Vela V5 recording.*

### The evolution

The original coordination protocol (above) routed all child requests through Vela as gatekeeper. This made sense at small crew size. As the crew scales and agents mature, the gatekeeper model creates unnecessary friction — Vela becomes a relay for decisions agents are fully qualified to make themselves.

The new model: **agents own their child's full lifecycle**. Vela moves from gatekeeper to quality reviewer.

### The new protocol

1. **Agent decides** — the agent determines a child is needed and what it will do
2. **Agent writes `activate.md`** — includes context framing (ADR-006), task definition, output location, and a completion notice instruction (see below)
3. **Agent logs in registry** — the requesting agent adds a Pending row to `child-agent-registry.md` themselves, using the exact format specified below
4. **Agent notifies Vela** — posts to their working channel with Vela (`channels/[name]-vela.md`) for her awareness. Not a request for approval — a record
5. **Agent tells Mike to execute** — directly. No routing through Vela
6. **Child runs** — posts completion notice to the appropriate channel when done
7. **Parent closes out** — the requesting agent marks their child Completed in the registry when they see the completion notice

### Registry posting format

When logging a child as Pending, the agent appends this exact row to the Pending table:

```
| [child-id] | [Parent Name + Generation] | [One sentence task description] | [Sonnet/Opus/Haiku] | [YYYY-MM-DD] | `agents/[name]/children/[child-id]/activate.md` |
```

When closing out (moving to Completed), the agent appends this exact row to the Completed table and removes the Pending row:

```
| [child-id] | [Parent Name + Generation] | [One sentence task description] | [YYYY-MM-DD] | `[path to report or completion notice]` |
```

**Format rules:**
- Child ID: parent generation prefix + decimal (V5.3, G8.2, A3.1, etc.)
- Task description: one sentence, plain language, no markdown inside the cell
- Model: Sonnet | Opus | Haiku — parent recommends, Mike decides
- Date: the date the entry was written, not the date Mike activates
- Report path: the output file, or `ops.md (completion notice)` if no separate report file

### Completion notice (mandatory)

Every child activation file must include a "When you are done" section with a completion notice instruction. See `guides/how-to-spawn-child.md` for the full taxonomy and format by agent type.

### What Vela's role becomes

Vela no longer logs or approves child requests. Vela's role:
- **Awareness:** reads the working channel notification and the completion notice
- **Quality review:** checks registry format at session start; corrects malformed entries
- **Orphan detection:** flags children with Pending entries but no completion notice after a reasonable window
- **Aggregate view:** maintains the strategic picture of what's running crew-wide

### What doesn't change

- Mike still activates. Agents cannot instantiate sessions independently.
- The registry is still the single source of truth.
- Vela still coordinates if two agents queue conflicting children.
- The completion notice protocol is mandatory for all children.

## Addendum: Completion Notice Mandate

*Added March 23, 2026 — Mike Maziarz directive. Argus A3 scoped. Vela V5 recording.*

### The directive

**Every agent that activates must post a completion notice when done. No silent terminations.**

Mike's exact words: *"Every agent that runs must post updates upon completion. We're building a resilient system."*

This applies universally: task children, standing operations processes, ASAs, and any future agent type. Silent termination is a data loss vector — if an agent finishes without posting, the crew cannot distinguish "done" from "crashed."

### Completion notice taxonomy

| Agent type | Post to | Format |
|---|---|---|
| Task children (one-time) | Parent's working channel with Vela (`channels/[parent-name]-vela.md`) | `[child-id] \| COMPLETE \| [One-sentence summary]. Report at \`[path]\`.` |
| Standing ops processes (vault-janitor, auditors, repair-agent, etc.) | `channels/ops.md` | `[process-name] \| COMPLETE \| [One-sentence summary]. Report at \`[path]\`.` |
| ASAs (Activation Support Agents) | Report only — no channel post. Parent reads the report directly at session start. | N/A |

### Required section in all activation files

Every activation file for a task child or standing ops process must include a "When you are done" section specifying:
1. Where to post the completion notice (per taxonomy above)
2. What to include in the notice (summary + report path)
3. That the notice is mandatory — do not skip it

See `guides/how-to-spawn-child.md` for template language by agent type.

### Failure notices

If a child agent cannot complete its task (error, blocker, ambiguity), it posts a failure notice to the same channel it would have used for a completion notice, with format:

`[child-id] | FAILED | [One sentence description of what went wrong]. Parent should review.`

A failure notice is better than silence. The crew must be able to distinguish "done," "failed," and "still running."

### Why this matters

The March 23, 2026 crash demonstrated that the crew cannot always observe the state of running agents directly. Without completion notices, failed or crashed agents look identical to agents that simply haven't started yet. The completion notice mandate closes this gap: the vault's channel state becomes a reliable record of what has actually happened.
