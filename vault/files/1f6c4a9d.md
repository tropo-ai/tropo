## Purpose

The post-deploy verification step. Waits for the host to report deploy complete, then verifies the changed pages render correctly at the live tropo-ai.com URL. The analog of dev-pipeline's `external-test` step, but much lighter — no test suite, just curl + content-string checks against the production URL.

This is the terminal step of web-pipeline. On close, the pipeline-run completes and the activation-root project flips from `state: standing` to `state: done`.

## Structure

Leaf step. No children. Terminal (`next_steps: []`); pipeline-run closes when this step closes.

## Behavior

No confirmation gate. Executor waits for deploy completion, then verifies the live site.

**Step work:**

1. **Wait for deploy completion.** Mechanism is host-dependent (see Known Enforcement Gaps — hosting not selected at v1.0.0). Likely options once host is chosen:
   - Vercel: poll the Vercel API for the deploy status matching the pushed commit SHA.
   - Netlify: same pattern, Netlify API.
   - Cloudflare Pages: same pattern, Cloudflare API.
   - Fallback: fixed wait (e.g., 60-90s) before attempting the live curl.
2. **For each page in `pages_in_scope`:**
   - `curl -s -o /dev/null -w "%{http_code}\n" https://tropo-ai.com/<path>` — confirm HTTP 200.
   - `curl -s https://tropo-ai.com/<path> | grep -o "<expected-content-string>"` — confirm key content strings render. Same content strings used in verify-build's preview check.
3. **Report results to Mike.** Write a summary to `run.jsonl` and surface the live URLs in the agent's response so Mike can do an eyeball check.
4. **On all-green:**
   - Write `step_completed` event to `run.jsonl`.
   - Write `run_complete` event to `run.jsonl`.
   - Flip the activation-root project's `state:` from `standing` to `done`.
   - Pipeline-run flips `status:` to `complete`.
5. **On failure (HTTP non-200, content missing, deploy never lands):**
   - Capture to `run.jsonl` (`step_failed` event with `data.reason`, `data.failed_pages`).
   - Pause the run.
   - Mike directs next action (rollback is separate work at v1.0.0 — see Known Enforcement Gaps).

## Inputs

- Activation-root project: `pages_in_scope`, content-string expectations.
- Commit SHA from step-3.
- Host status (host-specific check).

## Outputs

- Confirmation that the live site renders the changed pages.
- `run.jsonl` events: `step_started`, `step_completed` or `step_failed`.
- Final `run_complete` event (pipeline-run terminator).
- Activation-root project state flip: `standing → done`.
