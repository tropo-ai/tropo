**What it does:**
1. Populates test-spec.closed_at with current ISO 8601 timestamp
2. Populates test-spec.acceptance_evidence with:

   - coverage-audit UIDs from step 4 validate-coverage substrate
   - sa.* commissioning record UIDs where agentic_review fired
3. Flips test-spec.stage → done; test-spec.state stays active (only archives at supersession)
4. If test-spec.harness_framework_changes_required was true: populates test-spec.harness_extension_landed_evidence with substrate UIDs of harness extension that shipped as part of cycle scope per test-spec.capsule v1.0 §Harness Framework Changes — Triggers + Scope Expansion
5. Flips pipeline-run.status → done; writes terminal event to run.jsonl
6. **Back-feed:** updates triggering dev-pipeline activation's triggered_test_spec_uid field with this test-spec UID per engine extension [51d171f3] §Three-Pipeline Coupling State Machine. Engine's dev-pipeline activation close gate sees this back-feed + advances if doc-pipeline's parallel back-feed also landed.
7. Captures Mike approval signal verbatim (ship-bulletin pattern); records in activation entry close substrate

**What the engine does (separately, after step close):**
- Engine extension [51d171f3] §Three-Pipeline Coupling State Machine evaluates dev-pipeline activation's close gate
- If BOTH triggered_test_spec_uid + triggered_doc_spec_uid resolve to status:done test-spec + doc-spec entries → dev-pipeline activation can transition status:done (unblocks step 6 produce-release-folder for the triggering dev-cycle)
- If either still in-flight → dev-pipeline activation stays BLOCKED until parallel-pipeline closes
