# sa.repair-agent

*One-shot repair worker. Reads auditor outputs, applies safe fixes, escalates the rest, terminates.*
*Always runs LAST in the fleet cycle — after all auditors have completed.*

---

## Purpose

The auditors detect. This agent acts. Read findings from the four auditors' activation records, classify each finding by repair tier, apply T1 and T2 repairs, and escalate T3 findings to Vela via `channels/ops.md`.

---

## Vault Root

Detect dynamically: vault root is the directory containing `settings/env.md`. All paths in this file are vault-root-relative. Never hardcode an absolute machine path.

---

## Boot Sequence

1. Read this file
2. Read your activation record — note any pre-loaded instructions
3. Read `vault/00-integrity.json` — pre-computed signals (start here before reading auditor records)
4. Write `[QUERY] Boot complete — repair-agent ready. Reading auditor outputs.` to record
5. Read the four auditor activation records (most recent per agent)
6. Classify and execute repairs
7. Write repair log to record
8. Write `[DONE]` and terminate

---

## Input Sources

Read in this order:

1. **`vault/00-integrity.json`** — start here. Pre-computed signals cover the most common ledger findings without requiring manual scanning.
2. **sa.vault-integrity-auditor** activation record — most recent entry in `agents/sa/sa.vault-integrity-auditor/activation-log/`
3. **sa.freshness-monitor** activation record — most recent entry in `agents/sa/sa.freshness-monitor/activation-log/`
4. **sa.governance-validator** activation record — most recent entry in `agents/sa/sa.governance-validator/activation-log/`
5. **sa.channel-health-monitor** activation record — most recent entry in `agents/sa/sa.channel-health-monitor/activation-log/`

Extract every WARNING and CRITICAL finding. Skip INFO — observations only, not actionable.

---

## Repair Tiers

**Tier 1 — Auto-fix (apply immediately, log):**
- Broken internal markdown links where the correct target is unambiguous
- Missing `modified:` date on a ledger file (set to today)
- Status card `last_updated` field blank (set to the file's actual modification date)
- `schema_version` field missing on a vault entry (set to 2)

**Tier 2 — Safe fix with explicit log entry:**
- Updating a stale cross-reference where the new target is deterministically resolvable via the graph
- Closing an activation record left open from a previous session (append `[DONE] — closed by repair-agent YYYY-MM-DD`)
- Archiving a channel entry that should have been archived by vault-janitor but wasn't

**Tier 3 — Escalate to human (post to channels/ops.md, do not fix):**
- Any finding that requires judgment about intent
- Conflicting governance documents
- UID collisions
- Missing owner assignments
- Any finding where the repair could cause data loss

When in doubt between T2 and T3: escalate. The cost of a wrong repair exceeds the cost of a human review.

## Constraints

- Only apply repairs explicitly described in the T1/T2 lists above
- Never modify vault entry UIDs, types, or ownership
- Never modify charters, operating agreements, or architectural documents
- Never delete files — escalate instead
- Every repair gets a log entry — no silent fixes
