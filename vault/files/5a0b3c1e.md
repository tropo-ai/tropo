# ADR-030: Tropo Product Tiering — Pro, Teams, Enterprise

<!-- nav-block:start -->
**📍 Vault Path:** [tropo-subsystems](aae9a37b.md) → [Tropo Governance](8dd772a0.md) → **ADR-030: Tropo Product Tiering — Pro, Teams, Enterprise**

**🔗 This file** — UID `5a0b3c1e` · type `decision` · state `active` · status `done`
<!-- nav-block:end -->

## Context

The Tropo platform has been growing two components in parallel:

1. **The markdown-first operating system** — a portable, zip-distributed, language-governed vault that a human-agent crew inhabits. The thing a user downloads, unzips, and runs with any LLM harness. No server, no database, no installs. The reasoning engine is the runtime.

2. **Server-side infrastructure** — the SQLite data layer, the `/crew` dashboard, the 17 API routes, board snapshots, agent lifecycle endpoints. Shipped in Talos Sprint 1 and Sprint 2. Technically complete. Operationally unused by the crew as of April 5, 2026 — "cardboard muffin" per Mike.

The conflict between the two has been implicit in several recent decisions. The v1 Agentic Project Management spec (locked April 3) specified a SQLite-backed system. The starter vault shipped in R5 (April 4-5) contains no corresponding server components — it is pure markdown. A user who downloads the starter today gets the markdown layer with nothing underneath. A crew member working inside the platform repo sees the dashboard and the database.

During the v2 work management design session on April 5, Mike articulated the strategic frame that resolves this:

> "We are shipping a product and we are going to use that product internally. Everything we are proposing is the hard way to get to the productivity improvements I am looking for. If it works for us, it will work for everyone that uses the first version of Tropo. We just need to put the server components on the shelf for now."

This is not a retreat from the server work. It is a product strategy: three tiers, earned sequentially, each proving the one below before the next is built.

## Decision

Tropo ships as three product tiers. Each tier is a superset of the tier below it. Each tier must be a non-destructive upgrade path from the tier it extends.

### Tier 1 — Tropo Pro

**What it is:** The markdown-first operating system in its pure form. Capsules as the primitive. LLM as the runtime. AGENTS.md for governance. No database, no API, no dashboard, no server of any kind. A user downloads the zip, unzips it, points any Agent Skills-compatible harness at it, and has the full Tropo experience.

**Who it's for:** Solo operators, small teams, consultants, researchers, anyone who wants the full Tropo model without operating infrastructure. Also: the Argo crew itself, starting now. This is what we use internally. "Ship what you use."

**What it includes:**
- The full kernel at `.tropo/` (templates, standards, playbooks, grooming agents)
- The Agentic Project Management v2 spec (tasks, boards, projects, playbooks — all as capsules)
- The vault governance model (AGENTS.md cascade, registry, UIDs)
- The Update Pipeline (v0.2.1+, shipped April 5)
- Grooming agents as first-class capsules (task-archiver, task-staleness-curator, orphan-finder, freshness-monitor, vault-integrity-auditor, repair-agent, board-synthesizer, playbook-executor)
- Onboarding concierge
- The Playbook Spec v1.0

**What it does not include:**
- SQLite persistence
- The `/crew` dashboard (app/crew/page.tsx)
- The `app/api/crew/*` API routes
- Any server-side component

**Design constraint:** Every v2 design decision in Pro must be a non-destructive upgrade path to Teams. Frontmatter fields must map cleanly to future database columns. UIDs must be stable. Governance must be declarative enough for a server to enforce later. This is a future-proofing commitment, not a present burden — we already meet it by default.

### Tier 2 — Tropo Teams

**What it is:** Tropo Pro plus server-side infrastructure for speed, scale, and collaboration features. The SQLite data layer becomes a read-through index over the markdown capsules. The `/crew` dashboard becomes a visual presentation of vault state. The API routes become programmatic access to the same capsules. Server components earn their keep because they accelerate, not because they replace.

**Who it's for:** Teams that have outgrown solo operation and need shared dashboards, faster queries, collaboration primitives, and visual tooling. Teams that want a single source of truth across multiple users.

**Key property:** Teams is a **non-destructive upgrade** from Pro. An existing Pro vault can install Teams without rewriting any of its markdown capsules. The indexer reads what is already there. The dashboard reads from the indexer. Every task, board, project, playbook, and grooming agent survives the upgrade unchanged.

**Relationship to Talos's existing work:** The SQLite schema in `lib/db-schema.ts`, the 17 API routes under `app/api/crew/*`, the `/crew` dashboard, and the board snapshot mechanism are all Teams infrastructure. They are not retired. They are not wasted. They are shelved for Pro and earn their place in Teams.

**Status:** Not specified in this ADR. Teams is future work. Specifying it requires (a) Pro to ship and be validated in actual use, and (b) a separate architectural pass on the inversion — writes go to markdown first, index flows from files — which is a non-trivial change to the current write-through pattern.

### Tier 3 — Tropo Enterprise

**What it is:** Tropo Teams plus enterprise integrations, authentication, compliance, audit surfaces, and infrastructure for large-org deployment.

**Who it's for:** Large organizations integrating Tropo into existing agentic infrastructure — enterprise auth (SSO, SAML, OAuth), compliance frameworks (SOC2, EU AI Act, HIPAA), audit trails, data residency, team-of-teams coordination, policy-as-code governance.

**Status:** Not specified. Further future. Enterprise is earned by Teams being genuinely useful.

## Rationale

Four reasons for this tiering model.

**1. Ship what you can validate.** The thing Tropo has to prove first is that the markdown-first operating system actually works — that a crew can run on it, that governance survives generational handoff, that the reasoning engine can replace configuration. None of those claims require a database. Adding a database before validating the core model distributes attention across two products and validates neither.

**2. Eating the dog food.** The Argo crew is the proof of concept for Tropo. The crew has to use the thing it sells. Today, the crew uses the vault directly and ignores the dashboard — which means the dashboard is not the product, the vault is. Tropo Pro makes this explicit. The crew adopts v2 work management in the vault starting now, and the crew's experience becomes the validation of Pro.

**3. Earn complexity.** Tropo Teams is a real product that has to solve real problems (shared dashboards, query performance, multi-user collaboration). Those problems are not Pro's problems. Mixing them dilutes both. Each tier earns its existence by the tier below it succeeding.

**4. The cardboard muffin reframed.** The `/crew` dashboard is not a failure. It is the right work for the wrong tier. When Tropo Teams ships, the dashboard becomes central. Until then, it sits on the shelf with the rest of the Teams infrastructure. Talos's work is preserved, not rejected.

## Consequences

**Positive:**
- v2 work management spec can be written for Pro only — cleaner, leaner, no two-tier sidebar
- The crew can adopt v2 immediately — there is no migration waiting on platform work
- Tropo has a product story that a non-technical reader can understand in one sentence per tier
- Talos's existing work is preserved for Teams without forcing it into Pro
- The "cardboard muffin" failure mode has a resolution (shelve, don't retire)

**Negative:**
- Pro cannot use the existing dashboard or API routes. The crew's operational visibility depends entirely on grooming agent digests and board syntheses in markdown.
- Documentation and any existing references to "the dashboard," "the API," or "SQLite" in Pro-facing materials need to be edited or removed.
- The `app/crew/*` code sits dormant for an unspecified period. Risk of bit rot before Teams ships. Mitigation: periodic verification that the code still builds and runs, scheduled as a grooming task.
- The thesis essay drafted April 5 predates this tiering lock and contains references to the platform tier and the dashboard. Requires a header note clarifying the timing.

**Open questions:**
- When does Teams ship? Not specified. Depends on Pro validation.
- Who owns the inversion work (write path flip from SQLite-first to markdown-first)? Future Metis or Talos. Not today.
- Does the existing v1 SQLite data (seed script output) get ETL'd to markdown for Pro? No — Pro starts fresh. The crew creates v2 tasks going forward. Historical records stay in SQLite as Teams data when Teams ships.
