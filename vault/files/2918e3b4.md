## Purpose

app-pipeline governs the build-and-ship lifecycle for the **Tropo platform application** — the Next.js app at the `tropo-ai/tropo-ai` repo root that serves Tropo's user-facing platform surfaces (currently `/`, `/author`, `/projects`, `/projects/[id]`, `/crew`; future: agent-builder wizard, downloads, more). One pipeline-run = one shippable change to the platform.

A pipeline-run can carry a single component refactor, a new feature, a foundation step, a bug fix, or any set of changes that ship together as a single coherent platform update. The run packages the scope, builds it, verifies the production artifact, pushes to the platform's remote (`tropo-ai/tropo-ai`), and confirms the deployed instance renders correctly.

**Sister pipelines (the three-pipeline routing):**
- [`web-pipeline (7b2e94c1)`](7b2e94c1.md) — governs deploys to **tropo-ai.com** (the public website served from `tropo-ai/website`).
- **app-pipeline (this)** — governs deploys to the **Tropo platform app** (this repo, `tropo-ai/tropo-ai`).
- [`dev-pipeline (cd1fcd25)`](cd1fcd25.md) — governs **Studio governance work** in `argo-os/` (will rename to `studio-pipeline` in a future cycle).

Each pipeline is anchored to a product surface, not a repo. The three surfaces are: the website, the platform app, the Studio substrate. Multiple repos can feed one pipeline if they ship the same surface.

**Does NOT govern:** marketing copy or content decisions (those are MOS / website concerns); Studio governance (dev-pipeline); the website (web-pipeline); product strategy authoring (pre-pipeline work).

```
└── deploy (4b71b571)
    ├── step-3: commit-and-push (6a3bda96) — requires_confirmation: true
    └── step-4: smoke-test (b68e3332)
```

Three stages, five leaf steps. Linear, forward-only. Two human-confirmation gates: scope acceptance at step-0 (before any build work fires) and push approval at step-3 (before the deploy hits remote).

**Shape mirrors web-pipeline.** The pipeline shape (3 stages × 5 steps × 2 confirmation gates) is identical to web-pipeline by design — same discipline, two different product surfaces. The differences are scope (platform code paths vs website routes), push target (`tropo-ai/tropo-ai` vs `tropo-ai/website`), and the terminal step's verification approach (smoke-test against deployed instance, which at v1.0.0 may be local-only until the app gains a public live URL).

## Flow Rules

Linear, forward-only. specify → build → deploy.

**Stage transitions.** Each stage's `next_steps:` points to the next stage. The pipeline-run executor completes a stage (all its steps at `done`) before starting the next.

**Step transitions within a stage.** Each step's `next_steps:` points to the next step in its parent stage. The executor advances when the prior step reaches `done`.

**Terminal nodes.** deploy stage (`next_steps: []`) and smoke-test step (`next_steps: []`) are terminal. Pipeline-run closes when smoke-test completes.

**Branching.** None at v1.0.0. Fully linear.

**Human pause points.**
- **Step-0 (accept-deploy-request)** declares `requires_confirmation: true`. Pipeline-run pauses on entry. Mike confirms the deploy number + theme + files/features-in-scope before any build work begins.
- **Step-3 (commit-and-push)** declares `requires_confirmation: true`. Pipeline-run pauses on entry. Mike approves the commit message and authorizes the push to remote. Once confirmed, the deploy mechanism fires (auto-deploy if wired; manual otherwise).

The push-approval gate encodes Talos's hard rule: never push to remote without Mike's approval. The pipeline makes that gate structural, not memory-dependent.

## Differences from web-pipeline (parameterizations)

The shape is shared; the parameterizations differ:

| Dimension | web-pipeline | app-pipeline |
|---|---|---|
| Product surface | tropo-ai.com (public website) | Tropo platform app (Next.js app) |
| Source repo | `tropo-ai/website` | `tropo-ai/tropo-ai` (this repo) |
| Deploy name | `web-deploy-N` | `app-deploy-N` |
| Scope grain | pages on the website | files / features in the app (often code paths like `app/styles/tropo-tokens.css` or features like `agent-builder-wizard`) |
| Terminal step | smoke-test-live (against tropo-ai.com) | smoke-test (against deployed instance — public or local) |
| Domain tag | `tropo-website` | `tropo-platform` |
| Push target | `tropo-ai/website` remote | `tropo-ai/tropo-ai` remote |

## Cold-Boot Walk-Through

A pipeline-run has been started against app-pipeline v1.0.0. Mike has named the deploy: "UI Foundation Step 1a + 1b — token consolidation" (deploy number `app-deploy-1`, theme `ui-foundation-step-1a-1b`).

**Stage 1 — specify, step-0 (accept-deploy-request):** Executor enters step-0. `requires_confirmation: true` triggers; run pauses (`pause_started` event in run.jsonl, `data.reason: human-confirmation-required`, `data.proposed_deploy: app-deploy-1`, `data.proposed_theme: ui-foundation-step-1a-1b`, `data.changes_in_scope: ["app/styles/tropo-tokens.css", "app/globals.css"]`, `data.expected_visual_diff: zero`). Mike confirms or redirects via a `pause_resumed` event. Run resumes; step-0 work executes: activation-root project `app-deploy-1` authored at `vault/files/<deploy-project-uid>.md` per pipeline.capsule Rule 10. The project becomes the graph parent for all downstream deploy artifacts (per-deploy notes, commit message, smoke-test results). Step closes; stage advances to build.

**Stage 2 — build, step-1 (implement):** Talos opens the files in scope (`app/styles/tropo-tokens.css`, `app/globals.css`), implements the token extension + alias mapping, runs `npm run dev`, eyeball-confirms the dev preview renders identically to the pre-change baseline. Authors any per-deploy notes (decisions taken, alias mappings chosen) as documents under the activation-root project. Step closes when the local dev check passes; flow advances to step-2.

**Stage 2 — build, step-2 (verify-build):** Executor runs `npm run build` from the repo root. Full production build with typecheck and route generation. On build failure: step pauses with the error captured in `run.jsonl`; Talos fixes and re-runs. On green build: executor starts the production preview (`npm start`), curls each route in scope (or the full set for visual-diff verification), confirms HTTP 200 + key content strings. Step closes; build stage's `next_steps` fires; flow advances to deploy.

**Stage 3 — deploy, step-3 (commit-and-push):** Executor authors a structured commit message: deploy number, theme, files-touched summary, files-changed list. `requires_confirmation: true` triggers; run pauses (`pause_started` event, `data.reason: push-confirmation-required`, `data.commit_message: <message>`, `data.files_changed: [...]`, `data.target_branch: main`). Mike approves or redirects the commit message. Run resumes; executor runs `git add <specific-files>`, `git commit`, and `git push origin main`. If the platform app is wired to auto-deploy (Vercel etc.), the host's deploy fires from the push. Step closes; flow advances to step-4.

**Stage 3 — deploy, step-4 (smoke-test):** Executor verifies the deployed instance. If the activation-root project declares a `live_url:`, executor waits for host deploy-complete and curls the live URL with HTTP-200 + content-string checks. Otherwise (local-only at v1.0.0), executor verifies the production preview (`npm start` at localhost:3000) with deeper checks than verify-build did: every changed route, key content strings, no console errors during a click-through. Executor reports results to Mike for eyeball check. On failure: step pauses; rollback is a separate concern documented in Known Enforcement Gaps. On success: step closes, deploy stage closes, `step_completed` and `run_complete` events written, activation-root project `app-deploy-1` flips `state: standing → state: done`, pipeline-run flips to `status: complete`.

## Known Enforcement Gaps

| Gap | What closes it | Target release | Owner |
|-----|----------------|----------------|-------|
| Platform app has no public live URL at v1.0.0 — smoke-test runs against local production-preview only until a deploy target is wired (Vercel etc.) | When the platform app gains a public deploy target, smoke-test step body documents the polling/wait pattern per host; until then, local-preview verification is the gate | v1.0.0 → v1.1+ | talos |
| Smoke-test deploy-completion detection — same as web-pipeline, host-dependent | When a deploy target is wired, codify per host | v1.0.0 → v1.1 | talos |
| Rollback on smoke-test failure — no rollback mechanism declared at v1.0.0; current behavior is "Mike directs rollback as separate work" | v1.1+ amendment: optional rollback step or `restart_strategy: rollback` semantics | v1.1+ | talos |
