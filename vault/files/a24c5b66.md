# How to Maintain Your Vault

**Relations**

| Relation | Target |
|---|---|
| Governed by | [kb-article (4cb20382)](4cb20382.md) |

*Authored in v1.5 (Truthful Ship). Closes the discovery gap a new agent ran into when their vault index drifted and they had to reason from context how to rebuild it.*

A vault is a living thing. As you author new tasks, projects, notes, design briefs — and as you archive, retag, or move existing entries — the vault's index files drift out of sync with what's on disk. Your vault ships with a small Python toolchain to keep things in sync, plus a discipline for when to use each tool.

**Everything here runs as plain Python. No npm, no Node.js, no third-party dependencies.** The scripts target Python 3.8+ which is bundled with macOS and most Linux distributions.

---

## The toolchain

Four scripts live at `.tropo/scripts/`. All are user-runnable; all are idempotent (re-running on a clean vault is a no-op).

| Script | Purpose | When to run |
|---|---|---|
| [`rebuild-vault.py`](../scripts/rebuild-vault.py) | Regenerates `vault/00-index.jsonl` from the frontmatter of every `vault/files/<uid>.md`. Catches stale-index-vs-disk drift. | Whenever you've hand-edited a ledger entry's frontmatter |
| [`rehydrate.py`](../scripts/rehydrate.py) | Regenerates the `00-tropo-nav/` symlink tree (active/all/archived views) from the current ledger. | When the navigation tree feels stale or after batch changes |
| [`tropo-validate.py`](../scripts/tropo-validate.py) | Structural validator: AGENTS.md coverage, cross-reference resolution, registry integrity, UID uniqueness. Reports findings; doesn't write. | Weekly, or after any session that batched multiple structural changes |
| [`validate-no-absolute-paths.py`](../scripts/validate-no-absolute-paths.py) | Portability validator: scans for absolute machine paths (`/Users/<account>/...`, `~/<folder>`) that would break sharing or relocating. | Before sharing a vault, or as a release-gate (use `--strict`) |

---

## When to run each one

### Right after you hand-edit a vault entry

You opened `vault/files/<some-uid>.md`, changed a `status:` or fixed a `member_of:` link. The file is now fresh; the index is stale. Run:

```
python3 .tropo/scripts/rebuild-vault.py            # preview what would change
python3 .tropo/scripts/rebuild-vault.py --apply    # actually rebuild the index
```

The script walks all `vault/files/<uid>.md` files, parses their frontmatter, and rewrites `vault/00-index.jsonl` to match. Any stale entries (file gone, frontmatter changed) get cleaned up. If you've moved entries out of `extraction_scope: ship`, the rebuild surfaces that too.

### When `00-tropo-nav/` looks stale

The navigation tree at vault root is a set of human-friendly symlink folders generated from the Vault. Three views: `00-tropo-active/` (state: active only), `00-tropo-all/` (everything), `00-tropo-archived/` (archived only). Every governed artifact appears under every project it's `member_of`.

If a project shows up that you archived, or a new entry doesn't appear yet, regenerate:

```
python3 .tropo/scripts/rehydrate.py 00-tropo-nav
```

The script removes the old tree and rebuilds from the current ledger state. Stale links disappear; new entries surface in the right places.

### To check vault health (no writes)

Two read-only validators tell you whether your vault is in a good state without modifying anything:

```
python3 .tropo/scripts/tropo-validate.py
```

Reports: AGENTS.md coverage on governed folders, cross-reference resolution, UID consistency, registry integrity. A green run means the structural invariants hold. Findings are specific (file path + line number when relevant), so you can navigate to the issue and fix it directly.

```
python3 .tropo/scripts/validate-no-absolute-paths.py            # vault mode (with allowlist)
python3 .tropo/scripts/validate-no-absolute-paths.py --strict   # release-gate mode
```

Vault mode honors a small allowlist for explicitly-marked pedagogical content (lines tagged `<!-- portability:exempt -->`). Strict mode treats every absolute machine path as a violation — this is the discipline a release zip should pass before being shared.

## A healthy cadence

For most users:

- After a session where you authored or edited vault entries by hand: **rebuild-vault.py**
- After a session where you archived or retagged a batch: **rebuild-vault.py** then **rehydrate.py 00-tropo-nav**
- Weekly (or before any external sharing): **tropo-validate.py**
- Before sharing a vault zip publicly: **validate-no-absolute-paths.py --strict**

For agents working with you: the scripts produce structured output, so an agent can run any of them, read the result, and tell you in plain language what (if anything) needs attention.

---

## When something looks wrong

If a script reports a finding you don't understand, ask your Tropo agent. The agent can read the script's output, look at your specific vault state, and walk you through what's actually wrong — most reports are clear once you see them in context.

If a script itself errors out (Python traceback, missing module): the most common cause is a partial extraction (some scripts couldn't read each other's expected paths). Re-extract the vault zip to a clean folder and try again.

If a script reports many findings after a long stretch of work without maintenance: that's normal. Run `--apply` to fix what's fixable, surface the rest to your agent, and you'll find the toolchain has caught drift you didn't have to track yourself.

---

## What this isn't

These scripts maintain the **structural invariants** of the vault — they don't author content for you, don't merge conflicts, don't make judgment calls about what should ship vs. stay private. Those decisions are yours, and the agent's. The scripts just ensure that when you make a decision, the index reflects the decision and the navigation tree reflects the index.

The vault is durable because the indexes match the disk. These scripts are how you keep them matched.
