*Planning step. Vela authors plan substrate naming each behavior's test substrate path + verification_method dispatch shape before any test substrate is authored or run.*

**Why this step exists:** the plan is the bridge between test-spec declaration (what behaviors will be covered) and execute stage (authoring + running). Without an explicit plan, step 2 + step 3 happen without coherent per-behavior orchestration. The plan substrate becomes the working document the execute stage consumes.

**What it does:**
1. Loads test-spec entry (already validated at step 0)
2. For each behaviors_covered entry, authors a plan section:
   - behavior_description (mirrors test-spec)
   - test_substrate_path resolution (file exists at path → confirm; OR new-file → flag for step 2 authoring)
   - verification_method dispatch shape:
     - `machine_executable_script` → command line to invoke at step 3; expected exit code semantics
     - `deterministic_assertion` → DSL expression evaluation grammar per pipeline.capsule v3.0 §6
     - `structural_check` → validator extension name; tropo-validate.py invocation
     - `agentic_review` → sa.* class to dispatch (dispatch_target field) + commissioning record path + [QUERY]/[RESPONSE] protocol
     - `manual_walk` → Mike-readable walk prompt + acceptance criterion
   - target_substrate_refs (mirrors test-spec; cross-validation reference)
3. Flags harness_framework_changes_required if any verification_method requires runner/library/aggregator extension per test-spec.capsule §Harness Framework Changes — Triggers + Scope Expansion
4. Writes plan substrate to workspace path; pipeline-run references the path

**Hands off to:** step 2 author-test-substrate.
