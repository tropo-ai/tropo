## Project Description

**Lantern** is the recurring workstream the operator uses to produce a monthly newsletter for their audience (any audience — the starter deliberately does not specify; the operator substitutes their own). One newsletter per month. Each month's newsletter is a first-class unit of governed work that walks through a stable 6-task sequence under the declared pipeline.

**Why "Lantern":** a lantern is what you light for the people finding their way in the dark — the audience the newsletter serves. A recurring illumination.

### The 6-Task Sequence (Per Newsletter)

Each newsletter cycles these 6 tasks. At the starter's snapshot date (2026-04-22), the current newsletter is at Task 2.

1. **Ideate topics** — choose the 3–5 themes for this month's newsletter. Status: **DONE** for current month.
2. **Draft** — write the first draft of the newsletter. Status: **ACTIVE** — in progress.
3. **Review** — pass draft through editorial review (internal + Canon for brand voice compliance). Status: **BACKLOG** — waiting on Task 2.
4. **Edit** — apply review findings. Status: **BACKLOG**.
5. **Schedule** — set publication date + channel placement. Status: **BACKLOG**.
6. **Ship** — publish the newsletter + log delivery. Status: **BACKLOG**.

See [`library/packs/governed-work/lantern/tasks/`](../../library/packs/governed-work/lantern/tasks/) for the 6 task files. See [`lantern/board/project-board.md`](../../library/packs/governed-work/lantern/board/project-board.md) for the board regenerated from task state.

## Why This Project Exists

This project exists for one reason: **to demonstrate how governed work looks when it's in-flight, not finished.** Most real work is in-flight. The Verification Starter showed a completed end-to-end walk (Helm, archived at deploy-GATE). This project shows the honest state of a project mid-stream — 1 task done, 1 active, 4 pending, walker-walked through ~60% of the pipeline and legitimately paused at specify-GATE because the build hasn't happened yet.

A stranger reading the Governed Work Starter sees a project that looks like their own projects — not every-task-complete, not every-stage-walked, but with governance applied consistently across what IS present.

**The Lantern substance is incidental to the pack's value.** The value is: how do tasks compose under a project? How does the project board regenerate from task state? How does the typed pipeline play with task lifecycle? How does partial-walk legitimacy work? A stranger auditing the starter can answer all four from the filesystem alone.
