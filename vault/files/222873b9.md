## §0. The Core Claim

The Tropo ledger is the central nervous system. Schema v2 makes it structurally correct at the foundation level: pipeline stage and lifecycle state are separated, membership replaces the too-narrow `projects:` field, relationship vocabulary is unified, and the node index is clean metadata — no relationship data, no three-way divergence.

These changes apply to every capsule type uniformly. They must be right before v1.1 ships publicly on GitHub.

---

## §0.1 — What Changed (v1 → v2)

| v1 | v2 | Why |
|----|-----|-----|
| `status: active` | `stage: build` + `state: active` | Stage and lifecycle are orthogonal. A task at `stage: verify` can be `state: archived`. Conflating them made filtering require application logic. |
| `projects: [uid]` or `project: uid` | `member_of: [uid]` | Any capsule can belong to any container — not just projects. The name reflects set membership, not strict hierarchy. |
| `refs:`, `blocked_by:`, `implements:`, etc. | `relationships: [{rel:..., to:...}]` | Unified typed relationship vocabulary. The graph edge log derives from one canonical source. |
| Relationship fields in `00-index.jsonl` | Metadata only in `00-index.jsonl` | Three-way divergence (frontmatter, index, graph) produced 112 measured inconsistencies. Index is now metadata-only; graph derives from frontmatter. |
| No formal stage vocabulary | Stages registry at `.tropo/definitions/stages.definitions.jsonl` | Controlled vocabulary with canonical terms and aliases. Agents write canonical terms; natural language input normalizes automatically. |
| No unified relationship vocabulary | Edge types at `.tropo/definitions/edge-types.definitions.jsonl` | Same alias-resolution pattern. Agents always write canonical `rel:` values. |
| No tag vocabulary | Tags at `.tropo/definitions/tags.definitions.jsonl` | `bug`, `feature`, `spike` etc. with aliases. Type-as-task, label-as-tag — no `bug` type needed. |
| `schema_version: 1` | `schema_version: 2` | Version trigger for tooling. Index records carry it so readers don't need to open files. |

---

## §0.2 — Why This Enables

**Composable board queries.** `stage: specify AND state: active AND member_of contains node in subtree(X)` is a clean composite index lookup. In v1, this required interpreting `status:` values in application code and joining against the project tree.

**Graph integrity.** The three-way divergence between frontmatter, index, and graph produced silent data corruption at scale (112 inconsistencies measured at 815 entries). v2 has one source of truth — frontmatter — with the index and graph derived by the Python rebuilder.

**d.pm and sa.* efficiency.** The PM Director (d.pm) queries the index for stage/state filters and the graph for membership traversal. With clean metadata separation and a dual-adjacency graph index, every query is O(1) lookup rather than linear scan.

**Controlled vocabulary.** Any agent on any platform writes canonical terms because the definitions files tell them what to write. Natural language from Mike normalizes to canonical terms automatically. The vocabulary extends without code changes — append a JSONL row.

**Cold-boot testability.** Every v2 capsule definition now governs files whose frontmatter a cold agent can interpret unambiguously. The schema is the governance.

## §1. Frontmatter v2 — Complete Field Specification

Every governed ledger file must have this frontmatter. `schema_version: 2` is the version trigger.

```yaml
---
uid: <8-char lowercase hex — generate new, never reuse>
type: <registered capsule type — see §7>
title: <string, ≤100 chars>
description: <string, ≤120 chars, one-line summary>
owner: <agent-id or human identifier — see §1.6>
stage: <registered stage value — see §2>
state: <active|archived>
created: YYYY-MM-DD
modified: YYYY-MM-DD
tags: [tag, tag,...]
schema_version: 2
member_of: [uid, uid,...]
relationships:
 - rel: <vocabulary type — see §3>
 to: <uid>
created_by: <agent-generation-id>
---
```

### §1.1 — `uid`

8 lowercase hexadecimal characters. Generated at file creation. **Never reused, never changed.** The uid is the file's permanent identity — it survives rename, move, and content change.

Generation: `openssl rand -hex 4` or `python3 -c "import secrets; print(secrets.token_hex(4))"`.

Validation regex: `^[0-9a-f]{8}$`

**Exception:** Capsule definition files may use mnemonic UIDs (e.g., `c0re0001`) where the `name:` field carries the human identifier. This exception is restricted to `type: capsule-definition` files only. All other file types must use valid hex UIDs.

### §1.2 — `type`

Registered capsule type. See §7 for the type registry and required capsule definitions. Unknown types produce a governance-validator warning.

### §1.3 — `stage`

Pipeline position. A core OS primitive — required on every capsule. See §2 for the full stage specification, vocabulary, and registry.

### §1.4 — `state`

Lifecycle visibility. Exactly two values:

- `active` — alive, relevant, navigable in active views
- `archived` — historical record, removed from active navigation

`state: archived` is orthogonal to `stage`. A capsule at `stage: done, state: active` is completed work still in use. A capsule at `stage: done, state: archived` is completed historical work. A capsule at `stage: specify, state: archived` is work abandoned mid-pipeline.

### §1.5 — `member_of`

Array of UIDs. Declares which containers this capsule belongs to. Any capsule can declare membership in any other capsule.

Replaces all three of: `projects:` (array), `project:` (singular), `project_uid:`. The Python rebuilder maps all three v1 forms to `member_of:` during migration.

Declared by the child — the parent never maintains a members list. Membership is derived by graph traversal.

### §1.6 — `owner`

String identifier. The current responsible party. Must match a known identifier in the agent registry or be a human name. ≤30 chars.

Distinct from `created_by:`. Ownership may transfer; `created_by` is immutable.

### §1.7 — `relationships`

Array of typed directed edges. See §3 for the full vocabulary and semantics.

V1 scattered fields (`refs:`, `blocked_by:`, `blocks:`, `implements:`, `supersedes:`, `depends_on:`, `informs:`) remain valid in frontmatter. The Python rebuilder normalizes all of them into the `relationships:` vocabulary when building `00-graph.jsonl`. New v2 files use the `relationships:` structure.

### §1.8 — `created_by`

Agent generation identifier at the time of creation (e.g., `argus-a24`, `vela-v28`, `metis-g42`). Immutable once set. Optional but strongly recommended — approximately 62% of v1 files carry it.

### §1.9 — `resolution`

Optional. Captures why a terminal stage was reached. Free-form string. Examples: `shipped`, `wont-do`, `duplicate`, `failed-verification`, `superseded-by-v2`. Only meaningful when `stage: done` or `stage: cancelled`.

### §1.10 — Optional Type-Specific Fields

Capsule definitions for each type may declare additional optional fields (e.g., `priority:`, `verifier:`, `locked_by:`, `version:`, `slug:`). These are permitted in v2 frontmatter. They do not appear in the node index unless explicitly declared by the capsule definition as index-visible fields.

## §2. Stage Vocabulary and Registry

### §2.1 — Stage as OS Primitive

`stage:` is a required field on every capsule. It is an OS-level primitive — all capsule types carry it. The VALUES are configurable per vault via the stages registry.

### §2.2 — OS Default Vocabulary

```
ideate — exploring possibilities, not yet committed to design
design — in active design (Metis domain)
specify — specification being written (Argus domain)
build — implementation in progress (Talos domain)
test — under testing and verification
deploy — being deployed or released (Vela domain)
verify — post-deploy governance check (cold-boot tests, sign-off)
done — completed the pipeline successfully [terminal]
cancelled — deliberately abandoned at any stage [terminal, parallel]
```

### §2.3 — Terminal Stage Semantics

`done` and `cancelled` are the only terminal stages in the OS defaults. Terminal means: the work has reached a final state. Neither prevents future transitions — terminal stages are terminal by convention, not by architectural block.

- `done → stage:X` (reopen): allowed. Requires `resolution_note:` field on the reopened capsule + a different verifier on second closure. Represents legitimate regression or scope change.
- `cancelled → stage:X` (uncancel): allowed. Rare. Requires explicit human decision.

### §2.4 — Stages Registry

Location: `.tropo/definitions/stages.definitions.jsonl`

One JSON object per line. Each entry defines a canonical stage term, its meaning, and the natural language aliases that map to it.

```jsonl
{"term":"ideate","definition":"Exploring possibilities, not yet committed to design","aliases":["explore","brainstorm","discovery","pre-design"],"terminal":false,"type":"stage"}
{"term":"design","definition":"In active design — defining the problem and approach","aliases":["designing","in-design"],"terminal":false,"type":"stage"}
{"term":"specify","definition":"Specification being written — translating design into a locked technical contract","aliases":["speccing","writing-spec","in-spec"],"terminal":false,"type":"stage"}
{"term":"build","definition":"Implementation in progress","aliases":["building","implementing","developing","coding","in-progress"],"terminal":false,"type":"stage"}
{"term":"test","definition":"Under testing and verification","aliases":["testing","qa","in-test","under-review"],"terminal":false,"type":"stage"}
{"term":"deploy","definition":"Being deployed or released","aliases":["deploying","releasing","shipping"],"terminal":false,"type":"stage"}
{"term":"verify","definition":"Post-deploy governance check — cold-boot tests, sign-off, validation","aliases":["verifying","validating","checking","sign-off"],"terminal":false,"type":"stage"}
{"term":"done","definition":"Completed the pipeline successfully","aliases":["complete","finished","shipped","closed","resolved"],"terminal":true,"type":"stage"}
{"term":"cancelled","definition":"Deliberately abandoned — work was stopped before completion","aliases":["abandoned","wont-do","wont-fix","dropped","out-of-scope"],"terminal":true,"type":"stage"}
```

**Alias resolution:** When an agent encounters a natural language term (from human input or channel communication) that matches any `aliases` array, it writes the canonical `term` in frontmatter. Natural language in — canonical term out. Always.

Users add new stage values by appending a new JSONL row. No code changes required. Capsule definitions validate `stage:` against the registry; unknown values produce a governance-validator warning.

### §2.5 — Stage Transitions

No transition is architecturally blocked. Stages should progress forward through the pipeline. Backward transitions (reopen, regression) are allowed with an explicit reason. The governance-validator flags backward transitions for human review, but does not prevent them.

## §2.5 — Definitions Architecture

All controlled vocabularies in Tropo-OS use the same JSONL definitions pattern. Three files live at `.tropo/definitions/`:

```
.tropo/definitions/
 stages.definitions.jsonl — pipeline stage vocabulary
 edge-types.definitions.jsonl — relationship type vocabulary
 tags.definitions.jsonl — tag vocabulary
```

**Every definitions file shares the same structure:**

```jsonl
{"term":"<canonical>","definition":"<prose meaning>","aliases":["<alt>","<alt>"],"type":"<stage|edge|tag>","<type-specific-fields>":...}
```

| Field | Required | Meaning |
|-------|----------|---------|
| `term` | Yes | The canonical term — always what agents write |
| `definition` | Yes | Human-readable prose definition |
| `aliases` | Yes | Natural language synonyms that map to this term |
| `type` | Yes | Which vocabulary class this belongs to |

**Alias resolution — the key behavior:**

When an agent receives natural language input that matches any `aliases` value, it writes the canonical `term` in frontmatter output. Natural language in, canonical term out. Always.

Examples:
- Human says "this is blocked by task X" → agent writes `rel: depends-on`
- Human says "this is a fix" → agent writes `tags: [bug]`
- Human says "we're shipping this" → agent writes `stage: deploy`

This creates natural vocabulary normalization without enforcement. Agents want to use canonical terms because they read the definitions first.

**Extending the vocabulary:** append a new JSONL row to the appropriate definitions file. No code changes. No schema changes. The new term is immediately available to any agent that reads the file.

## §3. Relationship Vocabulary

**Definitions file:** `.tropo/definitions/edge-types.definitions.jsonl`

Full content:

```jsonl
{"term":"depends-on","definition":"A cannot proceed until B is complete","aliases":["blocked-by","requires","needs","waiting-on","depends on","is blocked by"],"directed":true,"type":"edge"}
{"term":"implements","definition":"This artifact is a concrete realization of the target spec","aliases":["built-from","realizes","fulfills","is an implementation of","built against"],"directed":true,"type":"edge"}
{"term":"supersedes","definition":"This artifact replaces the target — target is now historical","aliases":["replaces","replaces-with","newer-version-of","is the new version of"],"directed":true,"type":"edge"}
{"term":"informs","definition":"This earlier-stage artifact is a design input to the target","aliases":["feeds-into","input-to","basis-for","was used to design","informed by"],"directed":true,"type":"edge"}
{"term":"references","definition":"Lateral dependency — these artifacts must be read together","aliases":["refs","see-also","related-reading","see","read with"],"directed":false,"type":"edge"}
{"term":"related-to","definition":"Loose association, no directional dependency","aliases":["related","associated-with","connected-to","linked-to","see also"],"directed":false,"type":"edge"}
{"term":"bound-to","definition":"Runtime binding — playbook instance to vault context","aliases":["binds-to","runtime-binding","configured-for","bound"],"directed":true,"type":"edge"}
{"term":"derived-from","definition":"This artifact was generated from or based on the target source","aliases":["generated-from","based-on","produced-from","output-of","came from","written from"],"directed":true,"type":"edge"}
{"term":"member-of","definition":"This capsule is an element of the target container","aliases":["belongs-to","in","part-of","contained-by","lives in","under"],"directed":true,"type":"edge","note":"derived from member_of: frontmatter — not declared in relationships: array"}
```

**Supersedes:** existing `edge-types.md` (uid `ae4c277c`) at `.tropo/vocabulary/`. Migrate during v2 build.

### §3.1 — Canonical Types

All relationships are typed directed edges: `{"from":"uid","rel":"type","to":"uid"}`.

| `rel` value | Direction | Semantics |
|-------------|-----------|-----------|
| `depends-on` | A → B | A cannot proceed until B is complete |
| `blocks` | A → B | B depends on A (operational alias — A blocks B) |
| `blocked-by` | A → B | A is blocked by B (operational alias for `depends-on`) |
| `implements` | A → B | A is a concrete realization of abstract spec B |
| `supersedes` | A → B | A replaces B; B is now historical |
| `informs` | A → B | A is design input to B (earlier-stage artifact → later-stage) |
| `references` | A → B | A has a lateral dependency on B; they must be read together |
| `related-to` | A → B | Loose association; no directional dependency |
| `bound-to` | A → B | A is runtime-bound to B (playbook instance → vault context) |
| `derived-from` | A → B | A was generated from or based on B (AI provenance) |
| `member-of` | derived | Derived from `member_of:` frontmatter field; not declared in `relationships:` |

**Operational aliases:** `blocks` and `blocked-by` are preserved for readability. They are normalized to directed `depends-on` edges in the graph: `blocks` → `{"from":"blocker","rel":"depends-on","to":"blocked"}`.

### §3.2 — `supersedes` Provenance

When a capsule supersedes another, the edge record should carry a reason:
```jsonl
{"from":"uid","rel":"supersedes","to":"prior-uid","reason":"v2 schema replaces v1","created":"2026-04-16"}
```

The `reason:` field is optional but strongly recommended. Preserves succession chain provenance for audit purposes.

### §3.3 — Edge Removal

Full rebuild from frontmatter is the primary mechanism for edge removal. When a relationship is removed from frontmatter and the graph is rebuilt, the edge disappears.

For `supersedes` specifically: the edge record with `reason:` is the audit trail. Do not remove supersedes edges from `00-graph.jsonl`; they are permanent provenance records.

### §3.4 — Tag Vocabulary

Tags are informal labels on any capsule. `type: task` with `tags: [bug]` is the correct model — bugs and issues are tasks, not separate types. The tag vocabulary is the canonical reference for what each tag means and what natural language maps to it.

**Definitions file:** `.tropo/definitions/tags.definitions.jsonl`

Full content:

```jsonl
{"term":"bug","definition":"A defect in existing functionality — something that worked and now doesn't","aliases":["defect","error","fix","regression","broken","issue","problem"],"type":"tag"}
{"term":"feature","definition":"New capability being added to the system","aliases":["enhancement","improvement","new-capability","addition","new feature","request"],"type":"tag"}
{"term":"chore","definition":"Maintenance, cleanup, or housekeeping work — no new capability added","aliases":["maintenance","cleanup","housekeeping","tech-debt","refactor","tidy"],"type":"tag"}
{"term":"spike","definition":"A time-boxed investigation to reduce uncertainty before committing to a design","aliases":["poc","research-task","investigation","proof-of-concept","discovery","explore","experiment"],"type":"tag"}
{"term":"issue","definition":"A user-reported problem or concern broader than a specific defect","aliases":["complaint","feedback","report","concern","user-issue"],"type":"tag"}
```

Tags are additive — a task can carry multiple tags: `tags: [bug, p0]`. Extend the tag vocabulary by appending new rows to the definitions file.

### §3.6 — `derived-from` for AI Provenance

When an agent generates an artifact based on another (a spec generated from a design brief, a task generated by a playbook run, a document generated from research), declare:
```yaml
relationships:
 - rel: derived-from
 to: <source-uid>
```

This is the governance record of AI-generated lineage. It enables impact analysis: "what was generated from this research report?" and "what is this spec based on?"

## §4. JSONL Index Architecture

Three files. Each has a distinct role. Never edit the graph files directly — they are always rebuilt from frontmatter.

### §4.1 — Node Registry: `vault/00-index.jsonl`

One JSON object per line. Contains node metadata only — no relationship fields.

**Required fields in every index record:**
```json
{
 "uid": "abc12345",
 "type": "task",
 "title": "Write PM Director spec",
 "description": "Architecture spec for the vault-wide PM Director agent",
 "stage": "done",
 "state": "active",
 "owner": "argus",
 "member_of": ["826ee57b"],
 "created": "2026-04-15",
 "modified": "2026-04-15",
 "tags": ["pm-director", "p0"],
 "file_ext": "md",
 "schema_version": 2
}
```

**What is NOT in the index:** `refs`, `blocked_by`, `implements`, `blocks`, `depends_on`, `relationships`, `board`, `primary_collection`, `tasks_collection`, `scope_ref`, `supersedes`, `informs`, `verifier`, `locked_by`, or any other relationship field.

`member_of` is the one exception: it is included because it is the primary board query field. Filtering by project membership must work without a graph lookup.

`schema_version:` and `file_ext:` are included in every index record — `schema_version` so tooling can detect v1 vs. v2 without opening files; `file_ext` for type-safe file operations.

**Canonical index record (all 13 required fields):**
```json
{
 "uid": "abc12345",
 "type": "task",
 "title": "Example task",
 "description": "One-line summary ≤120 chars",
 "stage": "specify",
 "state": "active",
 "owner": "argus",
 "member_of": ["826ee57b"],
 "created": "2026-04-16",
 "modified": "2026-04-16",
 "tags": ["example"],
 "file_ext": "md",
 "schema_version": 2
}
```

**Maintenance:** Updated by all agents at check-in (new file → append a row; field change → update the row in place). Written atomically: write to `00-index.jsonl.tmp` then `os.replace()`.

### §4.2 — Edge Log: `vault/00-graph.jsonl`

One JSON object per typed relationship. Rebuilt from frontmatter by the Python rebuilder. Append-only between rebuilds.

```jsonl
{"from":"abc12345","rel":"implements","to":"f4188576","created":"2026-04-15"}
{"from":"abc12345","rel":"member-of","to":"826ee57b","created":"2026-04-15"}
{"from":"abc12345","rel":"supersedes","to":"old-uid","reason":"v2 supersedes v1","created":"2026-04-16"}
```

**Source of truth:** The edge log is rebuilt from frontmatter. It is never hand-edited. If the edge log diverges from frontmatter, frontmatter wins — rebuild.

**`member_of:` generates edges.** Every UID in a file's `member_of:` array produces a `member-of` edge in the graph: `{"from":"file-uid","rel":"member-of","to":"container-uid","created":"date"}`. This edge is derived from the `member_of:` field — it is NOT declared in the `relationships:` array.

### §4.3 — Traversal Index: `vault/00-graph-index.json`

Pre-computed dual adjacency. Rebuilt from the edge log by the Python rebuilder. Never edited directly.

```json
{
 "abc12345": {
 "member-of": ["826ee57b"],
 "children": [],
 "implements": ["f4188576"],
 "depends-on": [],
 "blocks": [],
 "blocked-by": [],
 "supersedes": [],
 "informs": [],
 "references": [],
 "related-to": [],
 "derived-from": [],
 "bound-to": []
 }
}
```

Both forward and inverse directions are pre-computed. "What does A block?" = forward lookup. "What is blocked by A?" = inverse lookup. O(1) for both.

**Rebuild:** Always write to `vault/00-graph-index.json.tmp` first, then `os.replace()`. Crash-safe — the old index remains intact if rebuild fails.

## §6. Python Rebuilder Specification

### §6.1 — Command

`python scripts/rebuild-vault.py` or `npm run ledger:rebuild`

### §6.2 — What It Does

1. Scans all `vault/files/*.md`
2. Parses frontmatter from each file using `python-frontmatter` (PyPI)
3. Validates `uid` in filename matches `uid` in frontmatter — hard error on mismatch
4. Detects duplicate UIDs — hard error, never silently overwrite
5. Rebuilds `00-index.jsonl` from all valid files
6. Rebuilds `00-graph.jsonl` from all relationship fields (both `relationships:` array and v1 scattered fields)
7. Rebuilds `00-graph-index.json` from the edge log
8. Rebuilds all specialized indexes
9. Rebuilds `00-integrity.json`
10. Writes a rebuild manifest to `00-graph-registry.jsonl` (timestamp, counts, duration, trigger)
11. Prints summary: N files scanned, N nodes written, N edges written, N warnings, N errors

### §6.3 — V1/V2 Compatibility

The rebuilder handles both schema versions:
- Files with `schema_version: 2` — read `stage:` + `state:` + `member_of:` + `relationships:`
- Files with `schema_version: 1` (or missing) — read `status:` and apply the stage/state lookup table; read `projects:`/`project:` as `member_of:`; read scattered relationship fields

This shim gives the app a consistent v2 view before migration is complete.

### §6.4 — V1 → V2 Field Mappings

**`status:` → `stage:` + `state:`:**
See §8.3 for the complete lookup table. Any value not in the table is written to `00-rebuild-warnings.jsonl` and the file is indexed with `stage: ideate, state: active` as a safe default.

**`projects:` / `project:` / `project_uid:` → `member_of:`:**
All three map to `member_of:` as an array. Deduplicated.

**V1 relationship fields → `relationships:` edges:**

| V1 field | `rel` value in edge log |
|----------|------------------------|
| `refs: [uid]` | `references` |
| `blocked_by: uid` | `depends-on` (also written as `blocked-by`) |
| `blocks: uid` | `blocks` |
| `implements: uid` | `implements` |
| `supersedes: uid` | `supersedes` |
| `informs: uid` | `informs` |
| `depends_on: uid` | `depends-on` |

### §6.5 — Failure Modes and Handling

| Failure | Response |
|---------|----------|
| Malformed YAML frontmatter | Log to `00-rebuild-errors.jsonl`, skip file, continue |
| UID in filename ≠ UID in frontmatter | Hard error — rebuild fails, file identified |
| Duplicate UID | Hard error — rebuild fails, both files identified |
| Unknown `rel` value in relationships | Warning logged, edge written with `rel: unknown` tag |
| File not UTF-8 | Log error, skip file |
| Unknown `stage` value | Warning logged, value preserved in index |

### §6.6 — Libraries

- `python-frontmatter` — YAML frontmatter parsing; handles all edge cases
- `ruamel.yaml` — when field order and YAML comment preservation is required
- `pykwalify` — post-write schema validation (optional; run as a separate validation pass)

### §7.2 — The Power of the Capsule Library

Every governed type has a capsule definition — a file that any cold-boot agent reads before doing work of that type. The agent needs no prior context, no SDK, no training. The capsule tells it: what fields are required, what states are valid, who can make each transition, and what governance rules apply.

The richer the capsule library, the more capable any agent is on first encounter. `release.capsule.md` means any capable agent can create a governed release correctly. `research.capsule.md` means any agent can produce a governed research report. The governance chain is readable at every level of the pipeline.

Capsules compose. A `release` capsule references `playbook-run` instances governed by the `playbook-run` capsule, whose execution produces `task` completions governed by the `task` capsule. The entire Innovation Pipeline is navigable through capsule definitions alone.

### §7.3 — `research` Type (Governed, not informal)

Research reports are real pipeline artifacts — they inform design briefs, are cited in specs, and are produced by the research panel pattern. Keeping them informal loses the governance chain between research findings and design decisions.

**Capsule definition:** simple lifecycle. `draft → published`. No verifier required. Required fields: uid, type, title, description, owner, stage, state, created, modified, tags, member_of, schema_version. Optional: `refs:` (what this research informed), `created_by:`.

### §7.4 — `release` Type (New, Governed)

The Deploy stage produces a Release. No `release` type currently exists — releases are tracked as documents or in the projects/ folder. A formal `release` type closes the pipeline cleanly.

**What a `release` represents:** a versioned, announced capability increment. It bundles one or more Tropo Drops, has a version number, and carries the verification status of all included capabilities.

**Capsule definition:** `draft → published → archived`. Required fields: uid, type, title, description, owner, stage, state, created, modified, tags, member_of, schema_version, version (semver string). Relationships: `member-of` (the Deploy project), references to the Tropo Drops it bundles.

### §7.5 — `board-def` Deprecation

The vault uses `board` (70 files) but the proposed v2 list had `board-def` (2 files). Standardize on `board`. The 2 `board-def` files are migrated to `board` type as part of the v2 migration fleet pass.

### §7.6 — Pre-v1.1 Capsule Requirement

Every GOVERNED type must have a capsule definition before v1.1 ships. Capsule definitions live at `.tropo/capsules/<type>.capsule.md`. The cold-boot test for each capsule: dispatch sa.cold-boot with only the capsule file and ask it to create a valid instance of that type. If it succeeds without additional reads, the capsule passes.

### §7.3 — Capsule Definition Required Fields for v2

Every v2 capsule definition must declare:
1. **Required frontmatter fields** — including `stage:` valid values for this type
2. **State machine** — valid stage transitions, who can make each transition
3. **Governance rules** — behavioral constraints as testable invariants
4. **Validation checks** — concrete checks at write time
5. **Archival trigger** — when and by whom a record may be archived

### §8.4 — Ambiguous Case Protocol

For `type: task` files where `status: active` could map to multiple stages (backlog vs. build vs. verify), the agent uses these tie-breakers in order:
1. If `tags:` contains a stage name → use that stage
2. If `verifier:` field exists → stage is `verify`
3. If `projects:` is empty → stage is `ideate`
4. Default → `build`

## §9. Validation Rules

A valid v2 ledger file must:

1. Have `schema_version: 2` in frontmatter
2. Have `uid:` matching `^[0-9a-f]{8}$` (except `type: capsule-definition`)
3. Have `uid:` matching the filename stem (`vault/files/abc12345.md` → `uid: abc12345`)
4. Have `type:` in the registered type list
5. Have `stage:` in the stages registry
6. Have `state:` equal to `active` or `archived`
7. Have `title:` ≤100 chars
8. Have `description:` ≤120 chars
9. Have `owner:` ≤30 chars
10. Have `created:` and `modified:` in YYYY-MM-DD format
11. Have `member_of:` as an array (empty array `[]` is valid for top-level items)
12. Have `relationships:` as an array if present; each entry has `rel:` in the vocabulary and `to:` as a valid UID format
