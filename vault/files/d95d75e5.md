## §4 — Capsule body refactor pattern specification

The load-bearing architectural deliverable of v1.18.0. Two paired patterns: the active-capsule pattern + the history-companion pattern.

### §4.1 — Active capsule body — 5-section canonical structure

Capsules are AGENT-READ substrate (Mike-A55 directive 2026-05-11: *"These capsules need to be clear, concise, and only need to be focused on agent use, not human."*). Active body is current-state-only, tight, performative. ALL historical content extracts to the `.history.md` companion (per §4.2). The history pointer lives in frontmatter (`history_file: <UID>`), not in the body — agents resolving history queries find it via the index.

Every refactored capsule body follows this 5-section order:

```markdown
# <type-name> — Capsule Definition v<N>

## 1. Intent
What this primitive IS for. How to reason about it. What it governs.
What failure modes it prevents. 100-200 words; agent-focused; no
narrative essay; fast orientation. The agent reading needs to know
what this type does before it can use the type correctly.

## 2. Schema
Frontmatter required + recommended fields. Field types + canonical
values. Examples of well-formed frontmatter. The structural shape.

## 3. State Machine (if applicable)
Lifecycle states + transitions. State diagram (text or table).
Which states are terminal; which require approval; which are reversible.
Skip section entirely if the type has no state machine.

## 4. Validation Rules
Structural invariants. What makes a valid instance. What the validator
enforces. Severity per rule (ERROR / WARN / INFO).

## 5. Composes-With
Other capsule types this composes with. Dependencies (what this type
references). Subtypes (what types extend this). Composition examples.

---

*<capsule-name> | UID <uid> | v<N> | history at <history-file-UID>*
```

**Why this order:** Intent first because that's what the agent reading needs FIRST to use the type correctly. Schema second because once you know what the type is FOR, you can absorb the shape. State Machine third because it's only relevant if the type has one. Validation Rules fourth because they enforce what the schema declares. Composes-With fifth because it's relational context. History pointer is frontmatter-class, not body-class — agents that need history resolve it via index lookup, not body parse.

**Word budget per active capsule body:** ~300-600 words for typical capsule; ~150-300 for simple types. Down from current 290-636 line average. The reduction is the changelog + amendment-block + deprecated-content + supersession-lineage extraction to history.

**Frontmatter additions to support the pattern:**

```yaml
history_file: <history-file-UID>   # NEW — declared on refactored capsules; agents resolve via index
last_body_refactor: 2026-05-11      # NEW — marker for v1.18.0 cycle refactor; helps validator detect non-refactored capsules
```

**What goes IN the active body:** current-state schema + current state machine + current validation rules + current composition rules + current intent.

**What goes OUT to history (per §4.2):** changelog tables (all of them), amendment blocks (all of them), deprecated-content sections (all of them), supersession-lineage footers (all of them), prior-author lineage (the active footer keeps just current author + current date + current version).

### §4.2 — History companion file pattern

For each refactored capsule, a sibling file:

**File name:** `<capsule-name>.history.md` (e.g., `task.history.md`, `playbook.history.md`)
**Location:** Same folder as the active capsule (`.tropo/capsules/`)
**UID:** New UID minted per file; declared in frontmatter
**Type:** `type: capsule-history` (new typed primitive — but kept lightweight; doesn't need its own capsule definition for v1.18.0; canonical kb article documents the pattern)
**Frontmatter:**
```yaml
---
uid: <new-uid>
type: capsule-history
governs: <parent-capsule-uid>   # the capsule this history serves
governs_path: ".tropo/capsules/<capsule-name>.capsule.md"
extracted_at: 2026-05-11
extracted_by: argus-a55
extracted_in_release: dbb0bace   # or the v1.18.0 release UID once authored
schema_version: 2
governed_by: <kb-article.capsule UID>   # treated as a kb-article-class artifact
member_of:
  - "8dd772a0"   # tropo-governance (governance-substrate class)
  - "<parent-capsule-subsystem>"
extraction_scope: argo-private   # historical record; not user-shipping content
tags: [capsule-history, extracted-from-<capsule-name>, v1.18.0-extraction]
---
```

**Body sections:**
```markdown
# <capsule-name> — Capsule History

*Historical record extracted from the active capsule definition at v1.18.0
to keep the active body pedagogy-first. The active capsule lives at
.tropo/capsules/<capsule-name>.capsule.md (UID <parent-uid>). This file
preserves the changelog + amendment-block + deprecated-content sections
that previously lived in the active body.*

## Changelog
[Full extracted changelog table from prior active body]

## Amendment Blocks (if applicable)
[Full extracted amendment-block content]

## Deprecated Content (if applicable)
[Full extracted deprecated-content sections — what was tried; what was retired; why]

## Provenance
- Extracted: 2026-05-11
- Extracted by: argus-a55
- Extracted in cycle: v1.18.0 (Capsule Library Phase 1)
- Active capsule version at extraction: v<N>
- Active capsule version after extraction: v<N+1> (non-breaking; body-only)

---

*<capsule-name> capsule history | UID <uid> | extracted from <parent-capsule-uid>*
```

**Cross-reference contract:**
- Active capsule frontmatter declares `history_file: <history-file-UID>` so agents resolving history queries find the companion via index
- History file frontmatter declares `governs: <parent-capsule-uid>` so it's discoverable in reverse
- Both files appear in the index; both have UIDs; both compose with their subsystem hubs
- No body-class reference to history in the active capsule — frontmatter-class only (agent-read performant; humans who want lineage find the history file via the index just like agents do)

## §5 — kb-article.capsule schema design

The new typed primitive that closes [32bd09cf](32bd09cf.md). Schema specification:

### §5.1 — Frontmatter required fields

```yaml
---
uid: <8-hex-uid>
type: kb-article
title: "<concise title; ≤80 chars>"
description: "<one-line description; ≤120 chars>"
status: <draft | published | archived>
state: <active | inactive>
audience: "<comma-separated audience tags: agentic-builders, agents, strangers, etc.>"
category: <how-to | reference | concept | glossary | decision-support>
author: "<agent UID or human name>"
created: <YYYY-MM-DD>
created_by: <agent-UID>
modified: <YYYY-MM-DD>
modified_by: <agent-UID>
schema_version: 2
governed_by: <kb-article.capsule UID>
extraction_scope: <ship | argo-reference | argo-private>
member_of:
  - "<subsystem-hub-UID>"   # at least one; the subsystem(s) the article documents
tags: [<list>]
---
```

### §5.2 — Body required sections

Following Lock C pedagogy-first principle (mirrors §4.1 capsule body pattern at the article scale):

```markdown
# <Article Title>

*<Optional one-line subtitle/tagline>*

## Intent
What does this article teach? Who's it for? What context is the reader
in when they reach for this article? What will they know after reading?

## <Body content>
The actual KB content. Structure varies by category:
- how-to: numbered steps + examples + common pitfalls
- reference: tables + field definitions + cross-references
- concept: explanation + analogues + composes-with
- glossary: term + definition + cross-references
- decision-support: question framing + options + trade-offs

## Composes-With
Related articles + capsule definitions + concepts. UID-linked.

## Provenance
- Authored: <date> by <author>
- Last revised: <date> by <agent>
- Supersedes (if applicable): <prior article UID>
- Validated by (if applicable): <verification dispatch UIDs>
```

### §5.3 — State machine

```
draft → published → archived
       ↓
       (revision in place; published stays published; modified date updates)
```

Simple lifecycle. Most articles ship at `status: published, state: active`.

### §5.4 — Validation rules

- **ERROR** — required frontmatter fields present (uid, type, title, description, status, state, created, created_by, modified, modified_by, schema_version, governed_by, member_of)
- **ERROR** — `governed_by` resolves to the kb-article.capsule UID
- **WARN** — body has at least an "Intent" section (h2 or h1 named "Intent")
- **WARN** — at least one entry in `member_of:` list
- **INFO** — `audience` and `category` declared

### §5.5 — Composes-with

- **Subsystem hubs** — every kb-article declares `member_of:` pointing at the subsystem(s) it documents
- **Capsule definitions** — when an article documents a typed primitive, link to that capsule's UID
- **Other KB articles** — cross-references via UID
- **The .history.md companion files (per §4.2)** — `type: capsule-history` is treated as a kb-article-class artifact for governance purposes; same governed_by

### §5.6 — Migration sweep mechanics

For each existing untyped KB article (~12 files identified in §1):
1. Read frontmatter; identify what's there + what's missing
2. Add `type: kb-article` + minimum required fields per §5.1
3. Inspect body; add "Intent" section if absent (synthesize from existing intro paragraph)
4. Validate via tropo-validate.py extension
5. Index regeneration picks up new typing

The sweep is mechanical for most fields (uid + type + schema_version + governed_by + member_of can be derived from filename + folder context); body Intent section may need light authoring per article.

## §6 — Risk analysis + safety mechanisms

Refactoring 10-13 kernel capsule bodies has real risk. Each risk + safety mechanism:

### Risk 1 — Body refactor accidentally drops governance content

**Scenario:** A capsule's active body may contain inline governance rules (validation rules; structural invariants) that were authored as part of an amendment block. The refactor extraction may misidentify them as historical changelog when they're load-bearing.

**Safety mechanisms:**
- **Pre-refactor inventory per capsule:** before extraction, scan the capsule body for load-bearing content (validation rules, structural invariants, schema declarations); document what should stay in active body vs. what's extractable.
- **Per-capsule extraction-fidelity diff:** post-refactor, mechanical diff comparing pre-extraction body content vs. post-extraction body content + history content. Confirm: every line of pre-content appears in either active body OR history file (no silent drops).
- **Skeptic verification on representative capsules:** sa.skeptic Round 1 on 2-3 refactored capsules; specifically asks "did anything load-bearing get extracted to history?"

### Risk 2 — Backwards compatibility break

**Scenario:** Existing artifacts reference capsules via `governed_by: <UID>`. The UID stays stable. But if any existing artifact relied on body content (e.g., a script parsed the capsule body for field definitions), refactor could break it.

**Safety mechanisms:**
- **Validator pass before + after:** tropo-validate.py runs full pass pre-refactor + post-refactor; cross-reference resolution should remain at 1501 PASS.
- **Capsule version bumps marked non-breaking:** v4.0.1 → v4.1.0 (minor bump = non-breaking body-only change per existing convention).
- **No script today parses capsule bodies for field definitions** (validated by audit; rebuild-vault.py + tropo-validate.py read frontmatter, not body); if any future script needs body-parsed fields, the schema section is named consistently.

### Risk 3 — History-file proliferation

**Scenario:** Adding 10-13 .history.md files = 10-13 new substrate files. Adds substrate weight even as it removes from active capsules.

**Acceptable cost:** net substrate weight reduces (active capsules trimmed by ~50-60% on average; ~3000-5000 lines of active substrate moved to history). Discoverability via convention: `.history.md` naming + `governs:` frontmatter field. History files have UIDs and member_of edges; they're indexable + queryable.

### Risk 4 — kb-article capsule + sweep introduces typing where none existed

**Scenario:** Existing KB article cross-references may not have type-aware behavior; some references may break or change.

**Safety mechanisms:**
- **Validator extension at WARN severity:** untyped KB articles surface as WARN (not ERROR) for grace period; existing cross-references continue to resolve.
- **Frontmatter additions are additive:** no field renames; no field removals; pure addition of `type: kb-article` + supporting fields. Existing readers ignore unknown fields.

### Risk 5 — Selection criteria for Stream B may be wrong

**Scenario:** "Top 10-15 by line count" is a proxy; some smaller capsules may be more load-bearing.
