## What ships

Foundation deploy for the multi-KB architecture locked with Mike-T6 2026-05-17. Vault becomes the registry (KBs + release history), bundles become the runtime content store. The platform reads bundles directly via `lib/kb-bundle.ts`; no more 2-hop vault traversal at query time.

### Engineering changes

**`lib/kb-bundle.ts` (new, ~430 lines)** — bundle reader + discovery:
- `discoverKbReleases(platformRoot, vaultRoot)` — discovers every shipped KB release. Tries vault first (`type: release` entries with `bundle_path`, `release_version`, `is_latest`, member_of a `subtype: knowledgebase` project), falls back to an on-disk scan of `tropo-ai/kb/` + `releases/kb/` when no release entries exist yet.
- `loadBundleIndex(bundlePath)` — reads `manifest.json` from a bundle, returns `KbIndex`-shaped metadata (same contract as the prior `buildVaultProjectIndex` so Haiku consumes it unchanged).
- `loadBundleContent(bundlePath, maxChars)` — loads bundle content files, returns `LoadMosCoreResult`-shaped output (drop-in for `loadVaultProjectKb`).
- Inlined minimal YAML frontmatter parser (no new deps; the bundle reader doesn't depend on `gray-matter` server-side).

**`app/api/kb/list/route.ts`** — switched from `discoverKBs` (legacy `.kb-index.json` scan of `kb/` directory) to `discoverKbReleases`. Returns the same `KBEntry[]` shape so the AppShell context + Header selector pick it up without UI changes. Entries now look like `{id: "kb-marketing-v14.0", name: "kb-marketing v14.0 (latest)", path: "<abs path to bundle>"}`.

**`app/api/query/route.ts`** — `vaultProjectUid` + `buildVaultProjectIndex` + `loadVaultProjectKb` retired from the runtime path. New flow:
1. Resolve a bundle path: take the request's `kb_path` if it points at a valid bundle (contains `manifest.json`); else fall back to the latest release of `DEFAULT_KB_PREFIX` discovered via `discoverKbReleases`.
2. Build the Haiku index via `loadBundleIndex(bundlePath)`.
3. Load content via `loadBundleContent(bundlePath, KB_CHAR_BUDGET)`.
4. Selective-mode (when Haiku picks specific files) still uses `loadSelectedKb(intent.relevant_kb_files, ...)` — no change; the file paths Haiku sees are now bundle-resident absolute paths.

**`lib/app-config.ts`** — added `DEFAULT_KB_PREFIX = "kb-marketing"`. Kept `DEFAULT_VAULT_PROJECT_UID = "93f7ff23"` as a LEGACY export for the V1 vault-resolver smoke test only; no runtime callers remain.

### Test coverage

**V1 (existing, offline tier)** — vault-resolver test still passes; vault traversal is still valid code, just not the runtime KB path anymore.

**T1 / T2 / T3 / T4** — all green post-wiring. T1 in particular validates intent extraction against the new bundle-loaded KB index (Haiku selected 2 KB files; previously selected 1 — within expected non-determinism).

## Defensive design — works today AND after the pipeline-side extension lands

`discoverKbReleases` tries vault release entries first; falls back to scanning `releases/kb/` (current pipeline output) and `tropo-ai/kb/` (anticipated future location). Today, only the on-disk fallback fires — Mike + Cosmo haven't shipped the vault-release-entry extension to the kb-pipeline yet. Once that lands (writing `type: release, member_of: <kb-uid>, bundle_path, release_version, is_latest` per build), the code automatically prefers the vault entries — no redeploy.

This means: **app-deploy-19 ships AHEAD of the pipeline-side change without coordination friction.**

## Visual-diff expectations

- **Header KB selector** — entry labels change from generic name (was empty or single legacy entry) to version-stamped `kb-marketing v14.0 (latest)`. Visible UI change.
- **Query results** — content semantically identical (same MOS files, same content), but loading path differs internally. Users should not see content drift.
- **`/api/kb/list` payload** — `id` field changes from directory name to bundle release name (e.g. `kb-marketing-v14.0`). Any external client cataloging KB ids will see new ids.
