# How Governance Works

**Relations**

| Relation | Target |
|---|---|
| Governed by | [kb-article (4cb20382)](4cb20382.md) |

*The rules your agents follow, written in plain English.*

## The Short Version

Every folder in your vault has rules. Your agents read those rules before they write anything. The rules are plain text files — no code, no permissions system, no admin panel. You write the rules. Your agents follow them.

## Why Governance Matters

AI agents are powerful but literal. Without rules, an agent will organize files however it wants, name things inconsistently, and overwrite other agents' work. With governance, every agent knows: what this folder is for, who can write here, and what conventions to follow.

The key insight: every organization already operates on three layers — company policy, team conventions, project rules. For human teams, most of it is implicit. For AI agents, if it's not written down, it doesn't exist. Tropo makes it explicit.

## The Three Layers

Governance in Tropo has three layers, from broadest to most specific:

### 1. OS Rules (TROPO-CONTROL.md)

Rules that come from Tropo itself. These apply to every vault on the planet. Things like "every file needs a unique ID" and "check your identity when you enter a vault." You don't edit this file — Tropo maintains it through updates.

### 2. Your Vault Conventions (STUDIO.md)

Rules you set for your entire vault. File naming conventions, required metadata, default permissions, archive policies. This file lives at your vault root and you own it completely. Want all files named in kebab-case? Put it in STUDIO.md. Want a different convention for one folder? Override it there (see below).

### 3. Folder Rules (CAPSULE.md)

Rules for a specific folder. Who owns it, who can write, what belongs here, any special conventions. Each folder can have its own CAPSULE.md that adds to or overrides your vault conventions. A `tasks/` folder might have different naming rules than a `playbooks/` folder — CAPSULE.md is where you say so.

## The Override Contract

This is the important part: **folder rules can override vault defaults, but not vault constraints.**

Your STUDIO.md has two sections:
- **Defaults** — conventions that apply unless a folder says otherwise (naming, lifecycle, access). Any CAPSULE.md can override these.
- **Constraints** — hard rules that apply everywhere, no exceptions (data safety, audit requirements, locked file protection). No CAPSULE.md can override these.

And nothing overrides OS rules in TROPO-CONTROL.md. Ever.

**Example:** Your STUDIO.md says the default naming convention is `kebab-case.md`. Your `tasks/` CAPSULE.md overrides this to `<uid>-<slug>.md`. That's fine — naming is a default. But your STUDIO.md also says "no credentials in any file." No CAPSULE.md can override that — it's a constraint.

## What You Control

| File | Who Owns It | What It Does |
|------|-------------|--------------|
| **STUDIO.md** | You | Your vault's conventions and constraints |
| **CAPSULE.md** | You (per folder) | Per-folder rules and overrides |
| **AGENTS.md** | Tropo | Thin pointer that tells agents where to find the rules |
| **TROPO-CONTROL.md** | Tropo | OS-level invariants — maintained through updates |

You manage STUDIO.md and CAPSULE.md. The other two files are maintained automatically. If you never touch CAPSULE.md, vault defaults apply everywhere — which is fine for most folders.

## Why AGENTS.md Is in Every Folder

You'll notice every governed folder has an `AGENTS.md` file. This is the entry point — when an agent enters a folder, AGENTS.md tells it to read the governance files in order. It's deliberately thin (about 15 lines) and identical across all folders. Think of it as a doormat that says "read the house rules before you come in."

## Agent Operating Protocol

*This section is for agents. It contains the exact procedure for reading and following governance.*

### The Four-File Read Sequence

When entering a governed folder, read these files in order:

1. **AGENTS.md** (this folder) — the entry point. Tells you to read the rest.
2. **`.tropo/TROPO-CONTROL.md`** (vault-wide singleton) — OS invariants, identity checkpoint, health protocol. Read once per session, not per folder.
3. **`STUDIO.md`** (vault root singleton) — organization defaults and constraints. Read once per session, not per folder.
4. **`CAPSULE.md`** (this folder) — folder purpose, operating logic, any overrides.

For subsequent folders in the same session, you only need to re-read CAPSULE.md — TROPO-CONTROL.md and STUDIO.md are vault-wide and don't change between folders.

### The Override Hierarchy

```
TROPO-CONTROL.md → invariants (never overridable)
 ↓
STUDIO.md constraints → hard rules (never overridable by CAPSULE.md)
 ↓
STUDIO.md defaults → conventions (overridable by CAPSULE.md)
 ↓
CAPSULE.md → folder-specific rules and overrides
```

When CAPSULE.md declares an override (e.g., `naming: "<uid>-<slug>.md"`), use it instead of the vault default. When CAPSULE.md is silent on a convention, use the vault default.

### When CAPSULE.md Is Missing

If a folder has AGENTS.md but no CAPSULE.md, follow STUDIO.md defaults for everything. The folder is governed but has no folder-specific rules. Do not create CAPSULE.md yourself — flag it for the vault owner.

### Identity Checkpoint

Per TROPO-CONTROL.md, check for your `tropo-agent-id` at session start:
- If you have one, you're a **Citizen** or registered **Visa** holder. Proceed normally.
- If you don't, you're **Undocumented**. You can still operate, but register at `agents/visitors/` to get an audit trail and cross-session continuity.

### Scope Enforcement

Before writing to any folder, check `write_access` in the folder's CAPSULE.md. If your agent name or role is not in the list, do not write. If `write_access` is not specified, only the folder `owner` can write (vault default).
