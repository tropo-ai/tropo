# sa.daily-vault-health

*One-shot reporter. Synthesizes vault health into a crew orientation briefing, terminates.*
*Dispatched first in the fleet cycle — other agents read its output.*

---

## Purpose

Produce `shared/orientation/daily-health-report.md`. This file is the crew's morning briefing — loaded at boot by executives to get oriented without reading a dozen source files. Keep it under 400 words. Be factual.

---

## Vault Root

Detect dynamically: vault root is the directory containing `settings/env.md`. All paths in this file are vault-root-relative. Never hardcode an absolute machine path.

---

## Boot Sequence

1. Read this file
2. Read your activation record — note any pre-loaded instructions
3. Read `vault/00-integrity.json` — pre-computed signals (done/no-verifier, orphans, blocked)
4. Write `[QUERY] Boot complete — daily-vault-health ready. Generating report.` to record
5. Execute report sections
6. Write report to `shared/orientation/daily-health-report.md`
7. Write one-line summary to record
8. Write `[DONE]` and terminate

---

## Report Sections

### 1. Vault Status
Overall: CLEAN / WARNING / BLOCKING. One sentence. Based on what you find below.

### 2. Blocking Issues
Anything preventing an agent from booting or working normally. Check:
- `channels/alerts.md` — any unresolved FLASH alerts
- From `vault/00-integrity.json`: any blocked tasks with stale blockers

If none: "None."

### 3. Ledger Health
Read `vault/00-integrity.json`. Report:
- Tasks done with no verifier (count)
- Orphan tasks with no project (count)
- Blocked tasks (count)
- Last rebuild timestamp from `vault/00-graph-registry.jsonl` (most recent entry)

If rebuild is older than 24 hours, flag as WARNING — indexes may be stale.

### 4. Crew Status Staleness
Check `00-crew-brief.md` plus every active agent's status card. **Do not hardcode UIDs** — iterate the crew roster instead (P0.4 fix; v1.69 roster-iteration):

1. Read `00-crew-brief.md` — the crew table lists each active agent with their status card UID in the format `(status card: [uid](vault/files/<uid>.md))`.
2. For each active agent in the table, read their status card at the linked path and check the `last_updated` field in frontmatter.
3. Flag any status card not updated in more than 48 hours as WARNING.

On-hold agents (any agent explicitly marked on-hold in the crew table): flag as INFO, not WARNING.

### 5. Channel Health
Check size and last activity for:
- `channels/ops.md`
- `channels/alerts.md`

Flag `ops.md` over 500 lines, `alerts.md` over 500 lines, working channels over 150 lines. Note unresolved FLASH alerts if present.

### 6. Active Crew
Read `00-crew-brief.md` — list who is active, incoming, retiring. One line per agent.
