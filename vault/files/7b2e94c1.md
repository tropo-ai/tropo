## Purpose

web-pipeline governs the build-and-ship lifecycle for **tropo-ai.com**, the public-facing Tropo website. One pipeline-run = one deploy. A deploy can carry a single page edit, a batch of new pages, or any set of changes that ship together to production as a single visible update to the live site. The pipeline-run packages the scope, builds it, verifies the production artifact, pushes to the host, and confirms the live site renders correctly.

**Does NOT govern:** authoring of marketing strategy or content decisions (those happen as pre-pipeline work — research, drafting, design); hosting selection and DNS/CI-CD setup (one-time substrate work that lives outside the per-deploy pipeline); content-only edits that don't require a code push (if the site grows a CMS surface, those changes follow that surface's flow, not this pipeline).

## Flow Rules

Linear, forward-only. specify → build → deploy.

**Stage transitions.** Each stage's `next_steps:` points to the next stage. The pipeline-run executor completes a stage (all its steps at `done`) before starting the next.

**Step transitions within a stage.** Each step's `next_steps:` points to the next step in its parent stage. The executor advances when the prior step reaches `done`.

**Terminal nodes.** deploy stage (`next_steps: []`) and smoke-test-live step (`next_steps: []`) are terminal. Pipeline-run closes when smoke-test-live completes.

**Branching.** None at v1.0.0. Fully linear.

**Human pause points.**
- **Step-0 (accept-deploy-request)** declares `requires_confirmation: true`. Pipeline-run pauses on entry. Mike confirms the deploy number + theme + page/feature scope before any build work begins.
- **Step-3 (commit-and-push)** declares `requires_confirmation: true`. Pipeline-run pauses on entry. Mike approves the commit message and authorizes the push to remote. Once confirmed, the host's auto-deploy fires.

The push-approval gate encodes Talos's hard rule: never push to remote without Mike's approval. The pipeline makes that gate structural, not memory-dependent.

## Cold-Boot Walk-Through

A pipeline-run has been started against web-pipeline v1.0.0. Mike has named the deploy: "ship the pricing page" (deploy number `web-deploy-3`, theme `pricing-page-launch`).

**Stage 1 — specify, step-0 (accept-deploy-request):** Executor enters step-0. `requires_confirmation: true` triggers; run pauses (`pause_started` event in run.jsonl, `data.reason: human-confirmation-required`, `data.proposed_deploy: web-deploy-3`, `data.proposed_theme: pricing-page-launch`, `data.pages_in_scope: ["app/pricing/page.tsx"]`). Mike confirms or redirects via a `pause_resumed` event. Run resumes; step-0 work executes: activation-root project `web-deploy-3` authored at `vault/files/<deploy-project-uid>.md` per pipeline.capsule Rule 10. The project becomes the graph parent for all downstream deploy artifacts. Step closes; stage advances to build.

**Stage 2 — build, step-1 (implement):** Talos opens the files in scope, implements the pricing page (new route, components, content, styles), runs `npm run dev`, eyeball-confirms the page renders correctly at localhost:3000/pricing. Authors any per-deploy notes (decisions taken, dependencies added) as documents under the activation-root project. Step closes when the local dev check passes; flow advances to step-2.

**Stage 2 — build, step-2 (verify-build):** Executor runs `npm run build` from the repo root. Full production build with typecheck and route generation. On build failure: step pauses with the error captured in `run.jsonl`; Talos fixes and re-runs. On green build: executor starts the production preview (`npm start`), curls the pricing page, confirms HTTP 200 + key content strings. Step closes; build stage's `next_steps` fires; flow advances to deploy.

**Stage 3 — deploy, step-3 (commit-and-push):** Executor authors a structured commit message: deploy number, theme, pages-touched summary, files-changed list. `requires_confirmation: true` triggers; run pauses (`pause_started` event, `data.reason: push-confirmation-required`, `data.commit_message: <message>`, `data.files_changed: [...]`). Mike approves or redirects the commit message. Run resumes; executor runs `git commit` and `git push` to remote. Host's auto-deploy fires from the push. Step closes; flow advances to step-4.

**Stage 3 — deploy, step-4 (smoke-test-live):** Executor waits for the host to report deploy complete (polls deploy status; mechanism depends on host — Vercel/Netlify/Cloudflare Pages API, or fixed wait). Once live, executor curls each page in scope at the production URL (`https://tropo-ai.com/pricing`), confirms HTTP 200 and key content strings, and reports the live URLs to Mike for eyeball check. On failure: step pauses; rollback is a separate concern documented in Known Enforcement Gaps. On success: step closes, deploy stage closes, `step_completed` and `run_complete` events written, activation-root project `web-deploy-3` flips `state: standing → state: done` (deploy is closed; project is historical record), pipeline-run flips to `status: complete`.
