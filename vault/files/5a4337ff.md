## Intent

A doc-pipeline activation publishes the documentation impact of a triggering dev-cycle's NEW substrate across three tiers (summary / subsystem / spec) per Captain's Briefing v3.0 §Structural-Enforcement Requirement 1. The pipeline accepts a doc-spec entry (the activation-input contract per doc-spec.capsule v1.0), authors documentation where needed, runs voice review for tiers requiring it, validates cross-references against the rigor split per tier, and closes back into the triggering dev-pipeline's activation gate.

The substantive call this pipeline embodies: **documentation pristineness is real, not aspirational.** Every closed doc-pipeline activation has machine-checkable evidence that the dev-cycle's NEW substrate is documented at the tiers that matter, with voice consistency on reader-facing tiers and cross-reference integrity on substrate-citing tiers. Not "docs were updated" but "doc_changes_required at tier:summary updated the L1 entry to mention the new substrate; tier:subsystem updated the hub member list; tier:spec body reads present-tense canon against the new state; voice review passed for summary + subsystem; cross-reference check passed for spec including nav-block render verification."

## Composition with doc-spec.capsule v1.0

The doc-spec capsule (Phase B work LOCKED at [9a7d314a]) defines what valid activation-input looks like. The doc-pipeline (this entry) defines what the pipeline does WITH that input. Key composition points:

- **Step 0 validates the input** against all 12 doc-spec capsule validation checks (Checks 1-13 minus Check 11 close-invariant which fires at Step 5)
- **Step 1 plan-doc-substrate** consumes `doc_changes_required` from the doc-spec; one plan-entry per change-required entry
- **Step 2 author-doc-substrate** writes the changes per plan; preserves the doc-spec's `target_tier` (multi or per-tier shape) per Rule 6
- **Step 3 voice-review-substrate** dispatches voice-review per `voice_review_required` field default + spec-tier procedural note
- **Step 4 validate-cross-references** runs Check 9 EXTENDED with the per-tier rigor split from §Pristine Three Tiers
- **Step 5 close-activation** populates `acceptance_evidence` with voice-review UIDs + cross-reference audit UID per doc-spec Rule 3 + Check 11

## What this pipeline does NOT do

- Author the doc-spec itself (that's the upstream dev-cycle's job; doc-spec arrives as activation-input)
- Author voice-review skill definition (that's a separate Phase C deliverable; this pipeline references it)
- Run dev-pipeline or test-pipeline activations (those are independent pipelines; engine handles cross-pipeline coupling)
- Engine integration coupling (Argus's lane per c3dc9f00 §3 + engine extension 51d171f3)
