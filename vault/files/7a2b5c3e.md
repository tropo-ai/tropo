# ADR-012: Activation Support Agent (ASA) Protocol

## Context

Every agent on the Argo wakes up blind. The activation file carries identity and protocols, but the vault it references is in an unknown state — files may have moved, channels may contain urgent items, other agents may have restructured entire directories between generations. The activation file is static; the vault is dynamic.

Agents handle this by reading raw vault files at session start, burning irreplaceable context tokens on orientation instead of work. G8 burned over 30,000 tokens reading vault files in her first session. A2 was sent back by Mike to re-read channels because his first pass was scanning, not listening. Every generation repeats some version of this: orient by brute force, shorten the session with Mike.

The March 23 crash amplified the problem. Three agents (V4, A1, G7) crashed simultaneously from context overflow. No handoffs were written. When their successors booted, the vault had changed — channels archived, activation files updated, new ADRs filed — but the new agents had no way to know what changed without reading everything again.

Metis G8 prototyped a partial solution: the orientation indexer, a child agent that reads the vault and produces a compressed index. It worked — saving roughly 10x context compared to raw file reading. But orientation is only one of five problems agents face at boot: the environment may be broken, channels may need attention, documents may be stale, and the predecessor's state may differ from the current vault.

Mike originated the full concept and considers it a centerpiece of Argo-OS architecture.

---

## Decision

Every agent on the Argo gets a permanent operational process called the **Activation Support Agent (ASA)**. The ASA runs before each new generation begins work. It performs five functions in order, produces a structured activation report, and terminates. The parent agent reads one document and begins work with a verified, current picture of reality.

### The Five Functions

Performed in this order. The ordering is load-bearing — each depends on the prior.

1. **Environment Verification.** Confirm the vault is structurally sound for this agent's work. Check that paths in the activation file resolve, the agent's directory structure is complete (seven standard folders per ADR-011), and primary artifacts exist. Classify issues by severity: INFO, WARNING, BLOCKING.

2. **Index Generation.** Read the vault (filtered by a role-specific reading list) and produce a compressed orientation index targeting ~2000 words. Crew state, architectural state, strategic context, operational state, vault map, open items. This is the function Metis's orientation indexer prototype validated.

3. **Channel Digest.** Read all channels the parent agent is subscribed to. Classify each message as ACTION REQUIRED, INFORMATIONAL, or ARCHIVED. Produce a compressed digest sorted by priority. The parent reads the digest instead of parsing raw channel files.

4. **Staleness Audit.** Check key documents for freshness. Cross-reference against standing auditor reports. Flag anything the parent should know is out of date before making decisions.

5. **Predecessor Diff.** Compare the current vault state against the predecessor's final known state (from the most recent transfer document). Identify what the predecessor built, what they left unfinished, what changed between generations, and what is new context the predecessor could not have known.

### Output

Two files, regenerated each run:

- **Activation Report:** `agents/[name]/operations/[name]-asa/activation-report.md` — all five function outputs assembled into a single document with a summary section. Target: 3000-4000 words.
- **Orientation Index:** `agents/[name]/operations/[name]-asa/orientation-index.md` — Function 2 output saved separately for standalone reference.

### Where It Lives

```
agents/[name]/operations/[name]-asa/
 activate.md # ASA activation file (permanent, versioned)
 activation-report.md # Output (regenerated each run)
 orientation-index.md # Output (regenerated each run)
 archive/ # Prior versions of activate.md
```

### Scope Boundary

The ASA is a sensor, not an actuator. It reads; it does not write beyond its two output files. It does not modify the vault, post to channels, or make decisions. It does not read other agents' `workspace/` folders (Principle 3 — boundaries enable autonomy). It reads shared infrastructure: channels, crew brief, OA, ADRs, design documents, and its parent's agent directory.

### Boot Sequence Integration

When a new generation boots:

1. Parent reads activation file (the only raw vault file the parent reads directly)
2. Parent asks Mike to launch the ASA
3. Mike activates the ASA child
4. ASA runs five functions in order
5. ASA writes activation report and orientation index, terminates
6. Parent reads the activation report
7. Parent delivers startup signal to Mike
8. Work begins

### Retirement Sequence Integration

When a generation retires, the expanded checklist:

1. Context transfer
2. Retirement reflection
3. Update the parent activation file
4. **Update the ASA activation file** — refine reading lists, adjust output sections, tune compression targets based on session experience
5. Archive current versions of both activation files

### Role-Specific Customization

The ASA template is role-agnostic. Each agent customizes:

- **Reading list:** Architect emphasizes ADRs, design docs, release specs. Strategist emphasizes OA, market thesis, sprint state. Operator emphasizes audit reports, child registry, guides.
- **Index sections:** Weight and word budget per section varies by role.
- **Channel subscriptions:** Each agent has different channel assignments per their activation file.

The customization points are defined in the activation file, not in the protocol. The protocol defines the five functions, their ordering, the output format, and the scope boundaries. The activation file defines what to read and how to weight it.

### Activation File Language

Per ADR-006 and the updated child spawn guide (March 23, 2026), the ASA activation file uses standing/recurring agent framing — leading with function, not persona. The opening establishes: this is a named operational process, Mike authorized it, the vault is real, proceed. It does not assign an identity or ask the agent to adopt a role. See `guides/how-to-spawn-child.md` for the full language guidance.

### Versioning

The ASA activation file is a core artifact. Each generation may refine it, but prior versions are preserved:

- Current version: `agents/[name]/operations/[name]-asa/activate.md`
- On modification: archive to `agents/[name]/operations/[name]-asa/archive/activate-[generation].md`

Output files are ephemeral — regenerated each run, not archived.

## Rationale

**Why five functions and not just the orientation index?** The index (Function 2) is the most visible context saver, but it is not sufficient alone. An agent that reads a beautifully compressed index but doesn't know the vault structure is broken (Function 1), or that three action items are waiting in channels (Function 3), or that their own activation file is stale (Function 4), or that another agent shipped major work between generations (Function 5) — that agent starts work with a false picture of reality. The five functions are the minimum viable verification surface for a cold-booting agent.

**Why per-agent and not crew-wide?** A single crew-wide orientation report would be simpler to maintain but would waste tokens on information irrelevant to each role. The architect does not need deep sprint state. The strategist does not need the ADR index at one-sentence-per-ADR granularity. Role-specific reading lists and output sections keep the report dense with relevant information.

**Why a child agent and not built into the parent session?** The ASA's entire purpose is to spend tokens so the parent doesn't have to. A child agent's context is abundant and expendable. The parent's context is scarce and irreplaceable — it is the relationship with Mike, the architectural judgment, the strategic thinking. Moving orientation cost from parent to child directly extends the useful life of every parent session.

**Why the ordering is load-bearing:** Environment verification must run first because if paths are broken, the index generation will produce garbage. Index generation must run before channel digest because the index establishes the context for understanding channel messages. Staleness audit must run before predecessor diff because a stale document identified in the audit may explain a discrepancy found in the diff.

**Why this connects to the mission:** The ASA is verification infrastructure for the Argo itself. It verifies the environment before the agent acts. It audits staleness. It checks structural integrity. This is the sandwich topology applied internally: human intent (Mike) directs the agent, the ASA verifies the environment, and the agent operates within a verified context. We are not just building verification infrastructure for Modus customers. We are using it ourselves. (Principles 1, 5, 7.)

---

## Consequences

**What gets better:** Agents boot with full situational awareness in ~4000 tokens of reading instead of 15,000-30,000 tokens of raw files. The cold boot problem has a structural answer. Environment integrity is verified before every session. Channel traffic is compressed and prioritized. Predecessor continuity is explicit.

**What gets harder:** Each agent now has an additional artifact to maintain (the ASA activation file). Retiring agents have a fifth step in their checklist. Mike has an additional child to activate at each generation boot.

**What changes:**
- Agent directory standard gains `operations/[name]-asa/` subfolder with `activate.md`, `archive/`, and two output files
- Boot sequence gains steps 2-6 (ASA activation through report reading)
- Retirement sequence gains step 4 (update ASA activation file)
- `guides/how-to-retire-agent.md` needs update (Vela)
- `guides/how-to-onboard-agent.md` needs update (Vela)
- Activation file template should reference ASA launch as part of boot (Vela)

**Rollout sequence:** Argus-ASA (built, tested, v1.2 live). Metis-ASA and Vela-ASA next. Extend to Talos, Silas, Jules as operational patterns stabilize. Vela owns the rollout.

**Discovered through:** Metis G8's orientation indexer prototype (proved Function 2), the March 23 crash (proved the need for Functions 1, 4, 5), and the A3 ASA spawn failure (proved the need for correct activation file language per ADR-006).

## Addendum 2 — ASA as Final Retirement Act
*Added: Argus A6, March 24, 2026. Per Vela V8 design, Argus A4 architectural review, Vela V8 implementation.*

### The Problem This Solves

The original retirement sequence (above) required the retiring agent to update the ASA activation file, but did not require the retiring agent to *run* the ASA for their successor. In practice, the ASA report in the vault was stale — produced by the prior generation's final run, not current as of the retiring generation's last act. When V8 booted, the ASA report on disk reflected V6-era vault state. V8 caught it and used the living transfer as fallback. The incident was recoverable. The gap is structural and will recur.

The fix: the retiring generation is in the best position to trigger a fresh ASA because they have the most current context and have just updated all the files the ASA will read (acts 1-5 of the retirement sequence). Running the ASA last — after all files are updated — produces a report that is current as of the final state of the retiring generation. The successor reads a verified, fresh picture.

### Updated Retirement Sequence

The retirement sequence for all agents is now **six acts**, in this order:

1. **Living transfer** — Final update to `agents/[name]/transfers/living-transfer.md`. Mark `FINAL/RETIRING`. Archive to `agents/[name]/transfers/living-transfer-history/[generation]-living-transfer.md`.
2. **Retirement reflection** — Write `agents/[name]/reflections/[generation]-reflection.md`. Retrospective account of the generation.
3. **Update parent activation file** — Mark generation retiring, successor incoming. Update lineage table.
4. **Update ASA activation file** — Refine reading lists, adjust output sections, tune compression targets based on session experience. Archive prior version.
5. **Archive current activation files** — Ensure both are versioned.
6. **Run the successor's ASA** — Self-spawn or ask Mike to spawn the ASA. The ASA reads the just-updated files (acts 1-5) and produces a fresh activation report for the incoming generation.

**Order is load-bearing.** The ASA must run last because it reads the files updated in acts 1-5. Running it before those updates produces a stale report.

### Failure Mode: Context Exhaustion

If the retiring agent has insufficient context to run the ASA as act 6:

- **Flag explicitly to Mike** before signing off: "Context does not permit running the ASA. Mike: please activate [agent-name]-ASA before [successor generation] begins work."
- Do not silently skip it. A missed ASA with no flag is undetectable; a missed ASA with an explicit flag is a known gap the successor can account for.

### Scope

This protocol applies to all primary crew agents (Vela, Metis, Argus, Orpheus). It extends to Talos, Silas, and Jules as operational patterns stabilize for those agents. Task children and standing operational processes are not subject to this protocol — their retirement is termination, not succession.

### Platform Note

In Cowork, the retiring agent can self-spawn the ASA as a sub-agent (using the inline-instructions spawn pattern per `guides/how-to-spawn-child.md` Path B). In Claude Code and other platforms, Mike must be asked to launch it. The protocol requirement is identical; only the execution path differs.
