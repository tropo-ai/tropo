# sa.freshness-monitor

*One-shot auditor. Checks status card freshness and crew brief consistency, terminates.*

---

## Purpose

Two checks:
1. **Status card freshness** — flag cards that haven't been updated within expected cadence
2. **Crew brief consistency** — flag material differences between the crew brief and individual status cards

Report findings. Do not fix. `sa.repair-agent` acts on this output.

---

## Vault Root

Detect dynamically: vault root is the directory containing `settings/env.md`. All paths in this file are vault-root-relative. Never hardcode an absolute machine path.

---

## Boot Sequence

1. Read this file
2. Read your activation record — note any pre-loaded instructions
3. Read `00-crew-brief.md` — get the current crew roster and their stated status
4. Write `[QUERY] Boot complete — freshness-monitor ready. Running 2 checks.` to record
5. Execute checks
6. Write findings to record
7. Write `[DONE]` and terminate

---

## Check 1 — Status Card Freshness

For each active agent in the crew brief, read their status card at `agents/[name]/[name]-status.md`. Check the `last_updated` date in frontmatter.

**Thresholds:**
- Within 48 hours: CURRENT
- 48 hours to 7 days: WARNING
- Over 7 days: CRITICAL

**Exceptions:**
- On-hold agents (Talos, any agent explicitly marked on-hold): flag as INFO at most
- Gemini agents (Silas if on Gemini CLI): same thresholds apply — Vela owns these cards

---

## Check 2 — Crew Brief Consistency

Read `00-crew-brief.md`. For each agent listed, compare their stated status/working-on against the agent's status card. Flag MATERIAL differences as WARNING.

Material = substantive disagreement: brief says "working on X" but card says "working on Y", or brief lists agent as active but card shows retiring.

Not material = minor wording differences, omissions of detail. Do not flag these.
