## Status

**LOCKED.** Mike Maziarz + Argus A22 proposed April 13, 2026. Locked by Mike with Metis G41, April 13, 2026.

### Locked Decisions

1. **[default.collection.md](../../projects/) — all types, always.** Member projects render as nested sections mirroring a folder hierarchy, not a flat list.
2. **[all-projects.board.md](../../projects/) — top-level projects only.** Member projects listed under each parent row. A project that belongs to multiple parents appears under each — intentional, not a bug.
3. **Completed projects archive to [projects/archive/slug/](../../projects/archive/).** Mirrors the Vault archive pattern.
4. **Every project gets its own [projects/slug/](../../projects/) folder, always.** Every project is top-level. Membership in another project's collection is a relationship, not a demotion. The term "sub-project" is retired — use "member project."

---

## Context

The ledger is agent-first by design. Governed artifacts live in `vault/files/<uid>.md` — a flat structure that is queryable and schema-validated but not human-browsable. A human who wants to understand "what is the architecture-specifications project working on right now?" must either know the project's UID, query the index, or ask an agent.

The DASHBOARD.md addresses this at the executive summary level — it shows all active projects with health indicators. But it is a curated/generated hybrid owned by Metis, not a navigation portal. Clicking a project row opens the project's ledger file, which is a governed record, not a human-oriented workspace.

The result: there is no place in the vault where a human can simply browse to understand the shape of the work. The file tree — the primary human navigation surface in VS Code — does not reflect the work structure.

Boards address this at the individual project level, but they live in `vault/files/` (project boards) or `boards/<name>/` (owner boards) — neither is organized by project name in a way that makes the tree navigable.

---

## Decision

**Introduce `/projects/` as a dedicated human navigation layer for all work management.** This layer sits alongside the Vault — it does not replace it. The Vault remains the governed source of truth. `/projects/` is the navigation surface.

### The Structure

```
/projects/
 DASHBOARD.md ← executive dashboard (moved from vault root)
 all-projects.board.md ← master navigation board (generated)
 <project-slug>/
 project.board.md ← this project's generated board
 default.collection.md ← expanded view of everything in the project
 <project-slug>/
 project.board.md
 default.collection.md
...
```

### The Two Root Files

**`all-projects.board.md`** — the master navigation board. Generated by the board synthesizer. Shows every active project with: name (as link to its `project.board.md`), owner, health indicator, task count, and one-line status. This is the primary entry point for any human who wants to see "what is everyone working on?" It replaces the DASHBOARD's project table as the navigation hub — DASHBOARD becomes the executive strategic summary, not the navigation entry point.

**`DASHBOARD.md`** — moved from vault root into `/projects/`. It is a work management artifact. The vault root stays lean: governance files only (`AGENTS.md`, `STUDIO.md`, `CLAUDE.md`, `00-crew-brief.md`). A human who opens the vault and wants to understand governance reads the root. A human who wants to understand the work goes to `/projects/`.

### The Per-Project Folder

Each active project gets a named folder: `projects/<slug>/`. The slug matches the project's `slug:` field in the vault entry.

Two files per folder:

**`project.board.md`** — the project's generated board. Equivalent to the project board vault entries we've been creating, but living at a human-navigable path. Shows: project status, task tables by priority, decisions and specs referenced, flags. Generated by the board synthesizer. Uses wikilinks (`[[uid]]`) per STUDIO.md cross-reference convention for board files.

**`default.collection.md`** — the expanded collection. Not a flat UID list — a structured, generated view of everything that belongs to the project, grouped by type:

```markdown
## Decisions
- [ADR-032](../../vault/files/e6c3f410.md) — Three-Layer Boot Config — proposed

## Specs
- [2d016ecf](../../vault/files/2d016ecf.md) — Tropo Work Architecture v0.3 — locked

## Tasks
- [[5f1b9525]] Design cold-boot test protocol — P1, backlog
- [[ec566124]] Tighten check 7 grep — P2, backlog

## Notes
- (empty)
```

Generated by the board synthesizer (or a new director). Regenerated when the project's membership changes. Uses markdown links for cross-type references, wikilinks for task UIDs per board file convention.

### The Dual-System Pattern

The ledger entry (`vault/files/<uid>.md`) remains the governed record — state machine, validation, UID stability across renames. The `projects/<slug>/` folder is the navigation surface — what humans browse.

This is the same dual-system already proven by boards: `boards/board-vela-active/current.md` is the navigation surface; the Vault has the registration (`path: boards/board-vela-active/current.md`). ADR-033 extends this proven pattern to projects.

The vault entry for each project gets a `path: projects/<slug>/` field registered, same as kernel files and boards.

### Governance

Each `projects/<slug>/` folder gets `AGENTS.md` (standard Tropo governance pointer) and `CAPSULE.md` (owner: project owner, write_access: project owner + board synthesizer). The board synthesizer is the only agent authorized to write `project.board.md` and `default.collection.md` — they are generated, never hand-edited.

The root `/projects/` folder gets `AGENTS.md` and `CAPSULE.md` declaring: all agents may read, board synthesizer writes board files, vault admin writes DASHBOARD.md and all-projects.board.md (or delegates to board synthesizer).

## Rationale

**Why `/projects/` not `/tropo-work/` or `/views/`?** The folder contains project folders. The name is immediately obvious to any human or agent who opens the vault. `/tropo-work/` describes a system; `/projects/` describes the contents.

**Why move DASHBOARD.md?** The vault root is governance. DASHBOARD.md is work. Mixing them makes the vault entry point ambiguous. A lean root — governance files only — is clearer for new agents and new humans.

**Why `all-projects.board.md` at the root of `/projects/`?** Without it, a human must know which project folder to open. The master board is what makes the whole layer navigable from the top — open `/projects/`, see every active project, click in. Two levels: master board → project board → tasks.

**Why `default.collection.md` alongside the board?** The board shows status. The collection shows membership. They answer different questions: "how is this project going?" (board) vs. "what does this project contain?" (collection). Both are needed for full project understanding without opening the Vault.

**Why generated, not hand-maintained?** Hand-maintained files drift. Generated files stay current. The board synthesizer already exists. The same Pattern that regenerates Vela's active-work board can regenerate `project.board.md` and `default.collection.md`. Agents trigger regeneration on project state changes.

---

## Consequences

**What becomes easier:**
- Human navigation in VS Code — browse the file tree to find a project, open its board, understand its work in seconds
- Onboarding — new humans open `/projects/`, see `all-projects.board.md`, navigate the whole vault's work without any instruction
- The file tree IS the UI — no external tool required
- DASHBOARD becomes a true executive summary, not a navigation crutch

**What becomes more complex:**
- `create-project.action.md` must create the `projects/<slug>/` folder alongside the vault entry — compound action gains more steps
- The board synthesizer must handle a new target type: `project-folder:<slug>` — regenerates both `project.board.md` and `default.collection.md`
- Migration required: all existing projects need `projects/<slug>/` folders created

**What changes for agents:**
- Boot playbook Phase 3: agents may use `projects/<slug>/project.board.md` as an orientation surface for projects they own, instead of (or alongside) querying the Vault
- The `regenerate-board` skill needs a new mode for project folders

**Migration path:**
- Phase 1: Create `/projects/` folder structure for all active projects (Vela executes, board synthesizer generates content)
- Phase 2: Update `create-project.action.md` to v3.0 (creates folder + vault entry)
- Phase 3: Move DASHBOARD.md from vault root to `/projects/`
- Phase 4: Update `STUDIO.md` system map, boot playbook references, and agent charters

## Open Questions (for Mike to lock)

1. **Does `default.collection.md` always show all entry types, or does the project owner configure which types to show?** Current assumption: all types (decisions, specs, tasks, notes, documents) grouped by type.
2. **Does `all-projects.board.md` show ALL active projects or only top-level projects (excluding sub-projects)?** Recommendation: top-level only, with sub-projects shown under their parent row.
3. **When a project is completed and archived, does its `projects/<slug>/` folder move to `projects/archive/<slug>/`?** Recommendation: yes — mirrors the Vault's archive pattern.
4. **Does every project get a `projects/<slug>/` folder, including sub-projects?** Recommendation: yes — sub-projects are still navigable on their own.
