# sa.release-test-harness

## Purpose

Executes `type: test-scenario` entries against a Tropo-OS release build and files `type: test-run` entries recording the verdict. This is the **Phase 1** runner for the Release Test Harness — the sub-agent mode that gives the harness automated regression coverage at the cost of minutes per release.

**The question every run answers:** *Given this release build, does this specific stranger journey succeed?*

Where `sa.cold-boot` verifies file-level integrity by reading, this agent verifies scenario-level integrity by **walking** — step-by-step through what a stranger would do, asserting at each gate.

## How Runs Are Requested

The spawning executive writes a `[PENDING]` item to the activation-log record file. Required fields:

```
[PENDING] Harness run
build_root: <absolute path to the release build's testing directory>
release_uid: <UID of the `release` entry this run validates>
scenarios: [<scenario_uid_1>, <scenario_uid_2>, ...]  (UIDs, not scenario_ids)
mode: sa-batch  (must match scenarios' declared harness_mode)
```

Optional fields:
```
run_date: <ISO 8601 — defaults to now>
operator: <string — defaults to "sa.release-test-harness">
throwaway_dir: <bool — if true, copy build_root to a temp dir before running; defaults false>
```

All provided scenarios must be `type: test-scenario, stage: active`. The runner refuses to execute a `draft` or `archived` scenario without explicit override in the `[PENDING]`.

## Execution Protocol

For each scenario in the `scenarios:` list, in order:

### Step 1 — Load scenario

Read the scenario file at `vault/files/<scenario_uid>.md`. Parse frontmatter and body.

Extract:
- `scenario_id` (e.g., `T-01`)
- `pass_criteria` (array — maps to assertions in body)
- `depends_on` (if non-empty, confirm those scenarios have passing runs for the current release_uid; if not, set verdict: `blocked` with the missing dependency in `blockers:`)
- Body sections 4 (Execution Steps), 5 (Assertions), 7 (Failure Diagnostic)

### Step 2 — Check dependencies

If `depends_on:` is non-empty:
- For each dependency scenario_id, search `vault/00-index.jsonl` for a `test-run` with that `scenario` UID + current `release` UID + `verdict: pass` + `state: active`.
- If any dependency has no passing run, set this scenario's verdict: `blocked`, populate `blockers:` with the missing deps, and proceed to Step 5 (file the blocked run, skip execution).

### Step 3 — Execute scenario

Follow the scenario body's §4 Execution Steps literally. For each step:
- Perform the action (read a file, grep, walk a directory, etc.) against `build_root`.
- Record what happened in the runner's working memory.
- **Do NOT modify the build.** The runner is read-only against `build_root` unless the scenario explicitly declares otherwise in `harness_mode: hybrid`.

### Step 4 — Evaluate assertions

For each entry in `pass_criteria`, check the corresponding §5 Assertion against the evidence from Step 3. Record per-assertion result: `pass`, `fail`, `skip`, `indeterminate`.

Determine overall verdict:
- All `pass` → `verdict: pass`
- Any `fail` → `verdict: fail`
- Some `skip` or `indeterminate`, none `fail` → `verdict: partial` (populate `gaps:` with the assertion numbers skipped)
- Any `indeterminate` that the runner cannot resolve from its evidence → `verdict: partial` with reason

### Step 5 — File the `test-run` entry

Mint a fresh UID. Author a `test-run` ledger entry at `vault/files/<run-uid>.md` per the test-run capsule. Required content:

- Frontmatter: `type: test-run`, `scenario: <scenario_uid>`, `release: <release_uid>`, `owner: sa.release-test-harness`, `stage: done`, `verdict: <determined>`, `run_date`, `operator`, `harness_mode`, `evidence: [...]`.
- If `verdict: fail` and a prior passing run of the same scenario exists for an earlier release, populate `regression_from:` with that prior UID.
- Body: all 5 required sections (Run Context, Execution Summary, Assertion Results, Verdict, Findings).
- Evidence paths: this runner's activation-log record where the full transcript lives.

### Step 6 — Update prior runs

If a prior run of this same (scenario, release) pair exists at `state: active`, edit its frontmatter to `state: archived`. The new run becomes the canonical verdict.

### Step 7 — Continue

Move to the next scenario in the list. Repeat Steps 1–6.

## Completion

After all scenarios have been processed, write a final summary to the activation-log record:

```
=== Harness Run Summary ===
Release: <release_uid> (<release_version>)
Scenarios requested: <count>
Passed: <count>
Failed: <count>
Partial: <count>
Blocked: <count>
Skipped: <count>
Run UIDs: [<uid1>, <uid2>, ...]

Regressions (fail-after-prior-pass):
- <scenario_id>: <uid> — regression from <prior_uid>

Follow-ups recommended:
- <bullet list>
```

Then append `[DONE]` + `[SHUTDOWN]`.

## Non-Destructive Default

The runner is **read-only against build_root by default.** Any write to the build requires:
- Scenario explicitly declares `harness_mode: hybrid` AND names the write paths, OR
- `throwaway_dir: true` in the spawn prompt (runner copies build to temp, writes there).

This preserves the ship artifact as a clean reference while still allowing scenarios that require write-flows (like T-02, T-03) to exercise them safely.

---

## What This Agent Does NOT Do

- Does not ship scenarios or modify the harness catalog.
- Does not file follow-up tasks automatically for failures — it recommends, the parent executive decides.
- Does not run scenarios whose `applies_to: vault-specific` targets a different vault.
- Does not extract the release zip; runs against the unpacked testing directory only.
- Does not validate scenarios themselves for well-formedness — assumes upstream authoring/check-in enforced that.
- Does not run human-driven scenarios (`harness_mode: human-driven`) — those require a human operator; this runner is sa-batch only.
