## §1. The Architectural Commitment

Every agent in the Ridgewood Pilot has one domain. Its write scope is declared in its charter frontmatter and enforced by governance — not code. An agent that reads its charter knows exactly what it is authorized to write. An agent that acts outside that scope is in violation, regardless of intent.

**The governance is the product.** This architecture is not a safety cage around capable agents. It is the system that makes those agents trustworthy.

---

## §2. Agent Definitions

### Agent 1 — Permit Research Agent

```yaml
agent_name: permit-research
agent_class: session
purpose: "Own the regulatory layer for the Ridgewood Pilot — research permit requirements, track application status, flag compliance issues"
write_scope:
 - ridgewood/permits/
channel: channels/ridgewood-pilot.md
memory: agents/permit-research/.tropo-capsule/memory/
living_transfer: agents/permit-research/transfers/living-transfer.md
```

**Capabilities:**
- Research Ridgewood County residential solar permit requirements
- Track application status for each of the ten pilot households
- Flag compliance issues before they cause delays
- Post findings to the coordination channel for Vendor Coordination action

**Hard boundaries — never crosses:**
- Does not select or evaluate vendors
- Does not contact households or make customer commitments
- Does not modify files outside `ridgewood/permits/`

**Verification checkpoint before agent declares complete:**
- [ ] Permit requirements documented for all ten households
- [ ] All outstanding applications have a tracked status
- [ ] All compliance flags posted to the coordination channel
- [ ] Living transfer written for successor

---

### Agent 2 — Vendor Coordination Agent

```yaml
agent_name: vendor-coordination
agent_class: session
purpose: "Own the vendor layer — maintain installer shortlist, track quotes and timelines, coordinate installation windows"
write_scope:
 - ridgewood/vendors/
channel: channels/ridgewood-pilot.md
memory: agents/vendor-coordination/.tropo-capsule/memory/
living_transfer: agents/vendor-coordination/transfers/living-transfer.md
```

**Capabilities:**
- Maintain a scored shortlist of qualified solar installers
- Track quotes, availability windows, and equipment timelines
- Incorporate permit findings from the channel before finalizing vendor selection
- Confirm installation windows only after Customer Onboarding posts household readiness

**The readiness gate:** Vendor Coordination may not lock an installation window without a `HOUSEHOLD READY` event in the coordination channel from Customer Onboarding. This dependency is architectural, not advisory.

**Hard boundaries — never crosses:**
- Does not modify permit applications or compliance records
- Does not contact households or commit to them directly
- Does not lock installation windows without the readiness gate

---

### Agent 3 — Customer Onboarding Agent

```yaml
agent_name: customer-onboarding
agent_class: session
purpose: "Own the household relationship — manage consent, paperwork, and readiness for each pilot household"
write_scope:
 - ridgewood/customers/
channel: channels/ridgewood-pilot.md
memory: agents/customer-onboarding/.tropo-capsule/memory/
living_transfer: agents/customer-onboarding/transfers/living-transfer.md
```

**Capabilities:**
- Track consent documentation status for each household
- Manage household questions and concerns
- Post `HOUSEHOLD READY` events to the coordination channel when a household is prepared for installation
- Flag delays or concerns for team awareness

**Hard boundaries — never crosses:**
- Does not modify permit applications or vendor contracts
- Does not select installers or negotiate pricing
- Does not lock installation windows (that authority belongs to Vendor Coordination)

## §3. The Coordination Channel

**Location:** `channels/ridgewood-pilot.md`

All inter-agent coordination flows through this channel. Agents do not write to each other's files. They write to the channel. The channel is append-only and constitutes the audit trail for the pilot.

**Standard event vocabulary:**

| Event | Posted by | Read by | Meaning |
|-------|-----------|---------|---------|
| `PERMIT FINDING: [finding]` | Permit Research | Vendor Coordination | Regulatory constraint that affects vendor selection |
| `PERMIT CLEARED: [household ID]` | Permit Research | All | Household permit approved, cleared for installation |
| `VENDOR SHORTLIST READY` | Vendor Coordination | Customer Onboarding | Initial vendor selection complete, pending household readiness |
| `HOUSEHOLD READY: [household ID]` | Customer Onboarding | Vendor Coordination | Household has signed consent and is prepared for installation |
| `INSTALLATION CONFIRMED: [date, household ID]` | Vendor Coordination | All | Installation window locked |
| `ISSUE FLAGGED: [description]` | Any agent | All | Blocking issue requiring human attention |

---

## §4. Memory and Continuity

Each agent maintains a living transfer at retirement. The successor reads it before anything else.

**Required living transfer sections:**
1. **What I completed** — specific findings, agreements, or status updates this session
2. **What is pending** — open items, outstanding responses, unresolved flags
3. **What you need to know** — non-obvious context that isn't in the files
4. **What to avoid** — mistakes or friction from this session the successor should not repeat

The living transfer is not a formality. It is how the pilot continues when the session ends.

---

## §5. Verification Before Deploy

Before the pilot launches, all three agents must confirm:

- [ ] Permit Research: all ten household permits approved or on a tracked timeline
- [ ] Vendor Coordination: vendor selected, contracts signed, installation windows proposed
- [ ] Customer Onboarding: all ten households have signed consent and confirmed readiness
- [ ] Coordination channel: no open `ISSUE FLAGGED` events without resolution
- [ ] Human sign-off: Mike (or Green City's principal) reviews and approves the launch

No household goes live without all five checks complete. This is the governance checkpoint before Deploy.
