# How Projects Replace Folders

**Relations**

| Relation | Target |
|---|---|
| Governed by | [kb-article (4cb20382)](4cb20382.md) |

*The most important shift in how Tropo-OS organizes work.*

---

## The old question and the new question

In traditional tools — Notion, Confluence, Google Drive, SharePoint — the organizing question is:

> **"Where does this go?"**

You create a folder structure. Marketing docs go in Marketing/. Engineering specs go in Engineering/Specs/. New hire onboarding goes in HR/Onboarding/. Every file must be placed in exactly one location. The folder IS the organization.

In Tropo-OS, the question is different:

> **"Which project owns this?"**

There are no folders for work. There is the ledger (flat storage, every file at `vault/files/<uid>.md`) and there are projects (organizational context). A file's location in the ledger tells you nothing about what it is or who cares about it. The project tells you everything.

---

## What this means in practice

**Creating a new document:**

Old way: "I need to put this architecture spec somewhere. Does it go in design/? specs/? docs/? What's the right folder?"

Tropo way: "This is an architectural decision. It belongs to the Architecture Specifications project." The file goes in the Vault. The project's collection references it. Done.

**Finding a document:**

Old way: Navigate the folder tree. Remember whether it was in specs/ or design/specs/ or the old specs/ folder that got renamed.

Tropo way: Open the project. The project's board shows what's active. The project's default collection shows everything that belongs to it — decisions, specs, tasks, notes — grouped by type.

**A document that belongs to multiple projects:**

Old way: Copy the file. Now there are two copies that diverge. Or use a shortcut/alias. Now there's a maintenance burden.

Tropo way: The document has one UID, one file in the Vault. Both project collections reference that UID. One file, two organizational homes. No copies. No drift.

---

## The three-part structure every project gets

When a project is created, it automatically gets:

1. **A board** — the current state: what's active, what's blocked, what's done. The agent reads this at boot for orientation. The human opens this first.

2. **A default collection** — everything that belongs to the project, grouped by type: decisions, specifications, tasks, notes. The full membership roster.

3. **A tasks collection** — the work queue. Just tasks.

These three artifacts are the project's navigable surface. They live alongside the project in [/projects/<slug>/](../../..) — a folder in the file tree, named by the project's slug, so humans can browse without knowing UIDs.

---

## Why this is better than folders

**Folders impose a hierarchy. Projects impose ownership.**

A folder says: "This file is located here." A project says: "This file is owned here, by this team, for this purpose, with these other related files."

Folders break when:
- A document is relevant to two teams (where do you put it?)
- The organization restructures (everything needs to move)
- A new person joins (they don't know the folder conventions)

Projects work because:
- A document can belong to multiple projects by reference — one file, multiple contexts
- Projects don't move when the org changes — the project itself is the stable home
- A new person reads the project board and immediately understands the work, without needing to know the folder structure

**The flat ledger makes this possible.** Because all files live at `vault/files/<uid>.md` — a location that carries no organizational meaning — the only organization that matters is project membership. This is by design: the ledger is the stable, UID-based storage layer, and projects are the human-meaningful organizational layer on top.

---

## The new habit

When you're about to create something — a document, a task, a decision, a spec — ask:

1. **Which project does this belong to?** If no project exists for this work, create one first.
2. **Does the project already exist?** Check [/projects/](../../..) or query `type:project status:active` in the Vault.
3. **If a new project is needed**, create it with `create-project.action.md`. It automatically creates the board, the default collection, and the tasks collection.

The project is the answer to "where does this go." Always.

---

## For agents: what this means at boot

When you boot into a vault, you do not navigate a folder tree. You read the project boards. Each project's board tells you what's in flight, what's blocked, and what comes next. The [all-projects.board.md](../../..) in the `/projects/` folder shows every active project at a glance.

If you're given a task, find its project. Read the project board. That is your orientation. Not the folder structure — the project.
