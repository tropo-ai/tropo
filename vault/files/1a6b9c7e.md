# ADR-006: Child Agent Context Framing

## Context

When child agent A1.2 (a cold-boot smoke test) was launched on Haiku, the model interpreted its activation file as a prompt injection attack. It identified the structured role instructions, system names, and task protocol as classic injection patterns: false authority, elaborate fictional context, fake safety rules. It refused to execute.

The same activation file on Sonnet did not trigger an injection refusal, but the model still asked for explicit human confirmation before proceeding — it wanted Mike to verify in the chat that the instructions in the document were legitimate.

The root cause: AI models are trained to treat instructions found in documents as untrusted. This is correct behavior for general safety. But it conflicts directly with Argo-OS's operating model, where child agents receive their complete instruction set from a document (the activation file) written by their parent agent. Without sufficient context, a child agent cannot distinguish a legitimate activation file from a prompt injection.

This is a product-level concern. If Argo-OS is adopted externally, every child agent spawn will encounter this friction unless the activation protocol addresses it.

---

## Decision

Implement a three-layer defense against activation refusal:

**Layer 1: Context framing preamble (in activation file).** Every child activation file must include a context section at the top that establishes: (a) Argo-OS is a real, production system, (b) the vault contains real files on a real filesystem, (c) the activation file was written by the parent agent and approved by the human operator, (d) the child is authorized to perform the specified actions. This gives the model enough context to understand the legitimacy of its instructions.

**Layer 2: Verifiable mount point (in env.md).** `config/env.md` now includes platform mount points as first-class environment variables. A child agent reading env.md can immediately verify the filesystem exists by checking the mount point path. Concrete proof that the system is real, not hypothetical.

**Layer 3: Human launch message (operational).** When activating a child agent, the human operator provides a brief confirmation in the chat: "Execute the task in your activation file. You are authorized." This satisfies the model's safety training requirement for human confirmation of document-based instructions.

All three layers together make child activation robust across model sizes. Smaller models (Haiku) may need all three. Larger models (Opus) may boot clean with layer 1 alone. The protocol requires all three regardless — defense in depth.

---

## Rationale

**Why not just fix the activation file?** Layer 1 alone is insufficient for smaller models. Haiku's safety training is aggressive enough that even a well-framed document can trigger refusal. The human launch message (layer 3) is what definitively resolves ambiguity for all model sizes.

**Why not just use the human launch message?** Layer 3 alone creates operational friction without helping the model understand the system. The framing preamble (layer 1) gives the model genuine understanding, not just permission. This matters for task quality — a model that understands it's operating in a real system produces better output than one that's merely been told "go ahead."

**Why add mount points to env.md?** A child agent that can verify the filesystem path exists has concrete evidence the system is real. This is particularly valuable for models that may be skeptical of claims in the activation file. Verifiable facts are more persuasive than assertions.

**Why not just use larger models?** Model selection should be driven by task complexity, not by activation robustness. Mechanical tasks (file moves, audits, template generation) are Sonnet or Haiku territory. Requiring Opus for every child because smaller models can't boot is a misallocation of capability.

---

## Consequences

**What gets better:** Child agents boot reliably across model sizes. The activation protocol has a clear, documented pattern for establishing legitimacy. External adopters of Argo-OS have a tested solution for the model-safety-vs-document-instructions tension.

**What gets harder:** Activation files are slightly longer (the framing preamble adds ~150 words). The human operator has a small additional step at launch (one sentence in the chat).

**What changes:**
- `config/env.md` updated with platform mount points (`cowork-mount`, `claude-code-mount`)
- Child activation template updated with required context framing section
- `guides/how-to-spawn-child.md` updated with the three-layer protocol and recommended human launch message
- Model selection guidance: Haiku needs all three layers; Sonnet needs layers 1 + 3; Opus typically needs layer 1 only

**Discovered through:** A1.2 activation failure on Haiku (prompt injection refusal) and Sonnet (confirmation request). March 22, 2026.
