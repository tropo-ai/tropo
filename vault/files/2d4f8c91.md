*Tropo Work is the headline application of Tropo — the work-management surface you and your agents use day-to-day. Tasks, projects, decisions, and pipelines, all tracked together in plain markdown. This article explains how the surface composes and how to operate inside it.*

---

## The four primitives

- **Project** — a container with a name, a purpose, and a board. Owns work. Lives at `vault/files/<uid>.md` with `type: project`. Stage moves `ideate → build → done` (with `cancelled` as the early-out terminal).
- **Task** — a unit of work that lives `member_of:` a project. Status moves `new → accepted → active → closed`, with a `resolution:` field at `closed` (one of `done` / `rejected` / `cancelled`). Lives at `vault/files/<uid>.md` with `type: task`.
- **Decision** — a recorded choice (what was decided, why, what alternatives were considered). Status moves `design → done`; superseded decisions stay `done` but flip `state: archived`. Lives at `vault/files/<uid>.md` with `type: decision`.
- **Pipeline** — a repeatable workflow. Defines what stages a project moves through (e.g., specify → build → ship). Status moves `draft → active → locked → archived`. Lives at `vault/files/<uid>.md` with `type: pipeline`.

Each primitive is governed by its capsule (`.tropo/capsules/project.capsule.md`, `task.capsule.md`, etc.). Note that every capsule defines its own state machine; there's no universal lifecycle. For the underlying typing system, see [How Capsules Work](5f7a9d83.md).

---

## How they compose

Tasks belong to projects (`task.member_of: [project-uid]`). Decisions can belong to projects or stand alone. Pipelines define structures projects can follow.

State machines compose (each primitive carries its own lifecycle field — `stage:` for projects, `status:` for tasks/decisions/pipelines):

- A project sits at `stage: build` while tasks remain open
- When all tasks reach `status: closed` with `resolution: done`, the project can move to `stage: done`
- Pipelines surface what stage each project is in (specify / build / ship / etc.)
- The `state:` field is orthogonal — every primitive has `state: active / archived` as a lifecycle-visibility flag, separate from its primary status

The composition produces a graph: project → tasks; project → decisions; project → pipeline. Agents walk this graph to understand work-context.

## How you actually use it day-to-day

The shape of a working day:

1. **Start a project.** Use the [start-a-project playbook](../../.tropo/playbooks/concierge-paths/start-a-project.playbook.md) or just ask your agent: *"I want a project for X."* The agent creates a project entry, assigns an agent, and seeds the first task.
2. **Add tasks as work surfaces.** Tell your agent *"add a task to clean up the Q3 report"* — they author it at `vault/files/<uid>.md` with `member_of: [project-uid]`. No special syntax; just say what you want.
3. **Move tasks through the board.** As work happens, your agent updates `status:` on tasks (`new → accepted → active → closed`). When a task closes, the agent sets `resolution:` (`done` / `rejected` / `cancelled`). The project board reflects current state.
4. **Record decisions when judgment lands.** When a strategic call gets made (architecture choice, pricing call, lane decision), say *"let's record this as a decision"* — your agent authors a decision entry future agents can read for context.
5. **Define a pipeline when a workflow repeats.** When you find yourself walking the same sequence across projects (specify → build → ship; intake → analyze → recommend), author it as a pipeline so future projects plug into it.

## How agents pick it up across sessions

Each agent's status card lists active work. When an agent boots, they read:

- Their **status card** — what they were last working on
- The **project boards** for projects they own
- **`vault/00-index.jsonl`** to filter active tasks by owner
- Recent entries in **`channels/ops.md`** to see what landed since their last session

Continuity works because the work persists in files, not in the agent's memory. New session; same context.

---

## How to surface what's happening

For a high-level view:

- **The project board** at `vault/files/<project-uid>.md` shows tasks, their status, and recent activity.
- **The vault index** at `vault/00-index.jsonl` is queryable JSONL — every governed file with its type, status, state, and owner.
- **`channels/ops.md`** is the rolling activity log — append-only, human-readable.
- **Ask your AI** *"what's open right now?"* — agents read the index and summarize.

For a specific project: open its UID file. Frontmatter shows owner, status, and members; body shows the board.

---

## When typed work outperforms freeform notes

You'll feel the difference at three moments:

- **Across sessions** — your agent picks up exactly where you left off because work-state lives in files, not in the agent's memory.
- **Across agents** — multiple agents (analyst + writer + reviewer) work on the same project; each one reads the same project board; coordination is auditable.
- **Across time** — six months later, when you ask *"why did we decide X?"*, the decision file is right there. No archeology.

That's what "headline application" means. Tropo Work is the workhorse; the rest of Tropo (memory, governance, capsules) exists to make it work cleanly.
