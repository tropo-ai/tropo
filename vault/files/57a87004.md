# Evaluate Tropo

> **How this works.** The AI reading this playbook drives the configuration — it asks the questions, authors the files, and waits for your approval at each step. You're guiding; the AI is typing. **You don't need to read past this line** — your AI does. Skim if you're curious; otherwise let the AI handle it.

*The evaluator says: "I want to see whether this is real before I adopt it." They're an architect, a technical founder, or a skeptic who's seen too many AI-ops products that are smoke and mirrors. This playbook walks them through the governance surface, runs a cold-boot stranger test on one artifact, and surfaces what's locked vs what's drifting — so they can decide from evidence, not pitch.*

---

## Intent

Tropo is built on the premise that governance is legible, verifiable, and cold-boot-testable. An evaluator who asks "is this real?" deserves a demonstration, not a pitch. This playbook walks them through what the governance surface actually contains: the Vault (authoritative store), the capsules (declarative contracts), AGENTS.md and CAPSULE.md (folder governance), the ADR index (decision provenance). Then it does the one thing Tropo claims is the defining test — runs a cold-boot stranger test on one artifact the evaluator picks, so they see with their own eyes that a stranger agent can read a governance file and produce correct behavior without asking for help. By the end of the playbook, the evaluator has enough evidence to decide: Tropo is either the operating system it claims to be, or the claim is disproven. Either answer is a useful outcome.

---

## Rules

- The evaluator's mindset MUST be confirmed upfront. If they're here to build (not evaluate), route to [start-a-project.playbook.md](start-a-project.playbook.md), [create-an-agent.playbook.md](create-an-agent.playbook.md), or [set-up-my-team.playbook.md](set-up-my-team.playbook.md) instead. This playbook is for evaluation only.
- The cold-boot stranger test (Step 4) MUST be actually run — not described, not mocked. A stranger agent MUST read a real vault artifact and MUST produce a report. Evaluation without the stranger test is a pitch, not a demonstration. Rule violation HALTS.
- The evaluator picks the target artifact in Step 3. NOT the playbook, NOT the concierge. The evaluator's skepticism is the instrument; let them direct it.
- The evaluator's findings (Step 5 report) MUST include anything that failed, was unclear, or required inference. Hiding failures defeats the purpose of the evaluation. If the cold-boot test surfaces gaps, those ARE the data the evaluator needs.
- Jargon terms MAY appear in this playbook's user-facing text. The audience is technical — architects, founders, skeptics — and obscuring terms like "capsule," "ledger," "cold-boot," "AGENTS.md" makes the governance surface illegible. Use Tropo's canonical terms with brief explanations.
- The session MUST end with a decision prompt, not a sales ask. The evaluator leaves with either (a) evidence to adopt, (b) evidence to reject, or (c) a specific concern to investigate further. All three are valid outcomes.

## Steps

1. **Confirm evaluator mindset.** *Owner: agent. Deadline: within 1 minute.*
 Ask: "You said you want to evaluate Tropo before adopting. Let me confirm: you're here to see whether this system is real and governable, not to start building yet. Is that right?"
 - If **yes:** proceed to Step 2.
 - If **no or they want to build:** route to the appropriate build playbook and stop this playbook. Do not hybridize.
 - If **they're not sure:** offer the choice — "Evaluation is ~20 min, walks through governance and runs a stranger test. Building is faster and gets you an agent. Which would be more useful?" Follow their answer.

2. **Walk the governance surface (5 minutes).** *Owner: agent. Deadline: within 5 minutes total, ~1 minute per primitive (5 primitives × 1 min).*

 Budget 1 minute per primitive. If an evaluator follow-up stretches one primitive past budget, note the overrun and shorten a later primitive to compensate — the 5-minute total is the budget that matters. If the evaluator's questions push past 7 minutes on the tour, offer: "We're spending more time than planned on the surface walk — want me to just run the stranger test now? That's usually where evaluators get the best signal." Route to Step 3 if they agree.

 Introduce each primitive briefly — one paragraph each, the evaluator asks follow-ups if curious:

 - **The ledger** at `vault/files/` — flat, governed, authoritative store of every work artifact. Every entry has a UID, a type, a status, an owner, a state machine. The index at `vault/00-index.jsonl` is the query interface. Show the evaluator an index record: "Here's what one looks like. UID, type, title, owner, tags — one line per artifact, machine-readable, queryable."

 - **Capsules** at `.tropo/capsules/` — declarative contracts defining what each artifact type must contain. Open one — e.g., [playbook.capsule (e7b3c509)](../../capsules/playbook.capsule.md) — and walk the structure: required frontmatter, required body sections, state machine, validation checks. "Every playbook in the vault is checked against this. Drift from the contract surfaces at check-in, not months later."

 - **AGENTS.md / CAPSULE.md** — per-folder governance. Each folder declares its type, owner, write-access rules. Agents read AGENTS.md before writing anything to that folder. Show the evaluator [vault/AGENTS.md](../../../vault/AGENTS.md): "The librarian rules. Check-in protocol. What belongs and what doesn't."

 - **The ADR index** at [vault/files/417898d0.md](../files/417898d0.md) — every architectural decision has a record, a state (proposed / accepted / superseded), and a history. Nothing architecturally important is decided by shrug; every decision is tracked. Open one — e.g., [ADR-032 — Three-Layer Boot Configuration Model (e6c3f410)](../files/e6c3f410.md) — and show the evaluator the decision's body: context, decision, consequences. (v1.2 polish: direct-link to the specific ADR file, not the index — eliminates the navigate-from-index friction sa.cold-boot 067 surfaced.)

 - **The principles** at [vault/files/5a766c42.md](../files/5a766c42.md) — the architectural principles that govern everything else. The rules the capsules encode. "Everything traces back to these."

3. **Evaluator picks a cold-boot target. (2 minutes).** *Owner: agent + evaluator. Deadline: within 2 minutes.*
 Say: "Pick any single file in the vault. A capsule, an activation playbook, a charter, an ADR — whatever you'd most want to stress-test. A stranger agent will read ONLY that file and its declared dependencies, then report what it can execute and what it can't. The gaps are the finding."

 If the evaluator hesitates, offer 2-3 defaults: "Most evaluators pick a capsule — e.g., `playbook.capsule.md` or `project-plan.capsule.md`. They're the declarative contracts; the cold-boot tests whether a stranger can author a compliant file from the capsule alone. Another good target is an activation playbook — tests whether a stranger can boot an agent from the file alone."

 Capture the evaluator's pick. If they want to pick multiple files, pick ONE for this session — the test is about depth, not breadth.

4. **Run the cold-boot stranger test. (7-10 minutes).** *Owner: agent (dispatching a stranger sub-agent) + evaluator (observer). Deadline: within 10 minutes.*

 Dispatch a fresh sub-agent — naive by design, no prior Tropo context — to read the evaluator's chosen target file and write its findings back. The sub-agent reads ONLY the target (plus load-bearing dependencies the target explicitly declares).

 Two paths to dispatch — pick whichever fits the harness:

 **Path A — preferred: dispatch the shipped sa.cold-boot primitive.** The release ships a `sa.cold-boot` session agent at `agents/sa/sa.cold-boot/sa.cold-boot.md` and a commissioning quickref at `agents/sa/commission-quickref.md`. Follow the 6-step protocol in commission-quickref to dispatch in MODE B (BATCH): create a record file in `activation-log/`, pre-populate `[RESPONSE]` + `[PENDING]` with the test target, spawn the agent. The agent reads its own activation file, executes against the target, writes `[DONE]` with the standard verdict format, terminates. Cleaner audit trail (every cold-boot test becomes a numbered record on disk); reusable across playbooks; no copy-pasted prompt to maintain.

 **Path B — fallback: inline stranger-test prompt.** If the harness doesn't support sa.\* spawn-with-record-file commissioning, OR if the evaluator wants the simplest possible demonstration without the activation-log scaffolding, dispatch a fresh sub-agent with the inline prompt below. Self-contained, no external file read, works in any harness with a sub-agent primitive.

 **Harness-specific dispatch (applies to both paths):**
 - **Claude Code:** use the `Task` tool with `subagent_type: general-purpose`.
 - **Anthropic Messages API / Claude SDK:** make a fresh `messages.create()` call with a new conversation thread (no prior history), pass the prompt + target file contents as user message; read the response.
 - **OpenAI Assistants API:** create a new thread + assistant invocation with the prompt as the user message; read the run's output.
 - **Gemini CLI / Vertex AI:** start a new session (`gemini` CLI with fresh context) and pass the prompt + target file; capture the response.
 - **Cursor / Cowork / other agentic harnesses:** most expose a "sub-agent" or "background agent" primitive that runs a fresh turn; dispatch via that primitive.
 - **HALT if no sub-agent primitive exists.** The playbook's Rule 2 is not satisfiable by "the main agent reads the target file itself" — that would be self-attestation, not a stranger test. If the harness genuinely lacks any way to dispatch a fresh sub-agent with isolated context, this playbook HALTs per Rule 2. The evaluator is told: "This harness does not support the stranger-test primitive this playbook requires. Switch to a harness that does (e.g., Claude Code, Anthropic SDK with fresh threads) or defer evaluation." There is no degraded-mode fallback; the rule is load-bearing.

 **Path B — inline stranger-test prompt (copy into the sub-agent dispatch verbatim, substituting `<TARGET_FILE>`):**

 ```
 You are a stranger agent with ZERO prior context about this vault or about Tropo.
 You have been given exactly ONE task: read a single file and report what you
 find. Your naivety is the instrument — every file you must open beyond the
 target is a self-sufficiency gap; every term you cannot decode is a finding;
 every inference you must make is data.

 TARGET FILE: <TARGET_FILE>

 PROTOCOL:
 1. Read the target file in full. Read nothing else first.
 2. Read only the files the target explicitly points at as **load-bearing
 dependencies** — frontmatter fields that name a canonical source the
 target depends on to function (e.g., `governed_by:`, `spec:`,
 `basis_spec:`, `extends:`, `pattern_exemplar:`, `derived_from:`, or a
 template the target invokes by path). These are execution-required
 reads. **Distinct from cross-references** (`refs:`, `references:`,
 prose `[link]` mentions, see-also notes) which are pointers but NOT
 execution-required — do NOT chase cross-references in this test. Each
 load-bearing read must be noted in your report; cross-references that
 you noticed but did NOT open should also be noted (as a sanity check
 on the distinction).
 3. Attempt the task the file instructs — "author a compliant artifact per
 this capsule's contract," "boot an agent from this activation file,"
 "execute the procedure this playbook declares," or equivalent. You do
 NOT need to actually write files; describe what you WOULD do and where
 you would get stuck.
 4. Track: (a) every file you opened beyond the target + the reason, (b)
 every ambiguity that required inference, (c) anything you had to invent
 because the file did not specify it, (d) any terminology you could not
 decode from the target or its declared dependencies.
 5. Issue a verdict:
 - PASS — target is self-sufficient; zero or one extra reads, all justified.
 - PASS-WITH-GAPS — target is lockable but has P2 polish gaps; 2-3 extra
 reads; minor ambiguities with safe defaults.
 - FAIL — target has a load-bearing gap; cannot execute the declared task
 from the target alone.

 OUTPUT FORMAT (write this report and terminate):

 ## Cold-Boot Stranger Test Report

 **Target:** <TARGET_FILE>
 **Verdict:** PASS | PASS-WITH-GAPS | FAIL

 **Files opened beyond target:**
 - <path> — <reason>
 - (or "none" if zero extra reads)

 **P0 findings (would block use):**
 - <finding>
 - (or "none")

 **P1 findings (friction):**
 - <finding>
 - (or "none")

 **P2 findings (polish):**
 - <finding>
 - (or "none")

 **Inventions (what I had to fill in):**
 - <invention>
 - (or "none")

 **Verdict rationale (one paragraph):**
 <prose>

 Be blunt. Do not moderate your findings. Do not fabricate PASS. If you
 could not execute the declared task, that IS the verdict; FAIL is a
 useful outcome.

 CONSTRAINTS:
 - Do NOT dispatch further sub-agents from within this task. You are
 terminal — your naivety is the instrument; delegating away from
 yourself destroys the test.
 - Do NOT ask the caller clarifying questions; if something is
 genuinely undecidable from the target file alone, record it as an
 "Invention" ("I assumed X because the target didn't specify") and
 continue. The inference IS the data.
 - When your report is written, terminate. Do not wait for follow-up.
 ```

 During the wait, narrate to the evaluator what's happening: "The sub-agent is reading only your chosen file plus whatever load-bearing dependencies the file explicitly declares. It's checking whether it can produce correct behavior from that file alone. Every gap it hits is a finding — and findings are the data you came here for."

 **On return,** the sub-agent's Cold-Boot Stranger Test Report IS the verification artifact. **Default receipt-capture for a fresh vault (no established vault conventions yet):** append the verbatim report to `channels/ops.md` (the vault-wide operations channel, which the release ships with) with a header line `[<YYYY-MM-DD HH:MM>] evaluate-tropo | Cold-Boot Stranger Test | <TARGET_FILE>` preceding the report body. If your vault has an established alternative convention (workspace file, ADR, `verification-receipts/` subfolder), use that instead — the default is for new vaults that haven't declared otherwise. This playbook's load-bearing rule (Rule 2) requires the test to have actually run; the report is the receipt.

5. **Surface findings. (3-5 minutes).** *Owner: agent. Deadline: within 5 minutes.*
 When the stranger sub-agent's report lands, read the verdict and the finding list together with the evaluator. Do NOT hide or paraphrase findings.

 Present in a structured way:
 - **Verdict:** PASS / PASS-WITH-GAPS / FAIL (from the sub-agent)
 - **Files opened beyond target:** count + reason per file
 - **P0 findings** (would block use): bullet list
 - **P1 findings** (friction): bullet list
 - **P2 findings** (polish): bullet list
 - **Inventions:** what the stranger filled in that the file didn't specify

 If the verdict is PASS: "The file is self-sufficient. A stranger could execute from it alone. This is the claim Tropo makes about every governance artifact."

 If PASS-WITH-GAPS: "The file is lockable but has surface-level gaps. Every gap is tracked in the capsule's §Known Enforcement Gaps table or similar. This is the honest state — most Tropo artifacts ship at PASS-WITH-GAPS with P2 polish deferred."

 If FAIL: "The file has a load-bearing gap. This is a real defect the evaluation just surfaced. It would be remediated same-session if this were a build — the defect itself is the answer your evaluation was looking for."

 Append the stranger sub-agent's full report to the session summary so the evaluator can review the transcript.

6. **Decision prompt. (2 minutes).** *Owner: agent. Deadline: within 2 minutes.*
 Close with: "You've seen what Tropo actually contains — the Vault, the capsules, the governance primitives, and one stranger-test demonstration. Three possible outcomes from here:

 1. **Adopt** — you have enough evidence. Route you to [start-a-project](start-a-project.playbook.md), [create-an-agent](create-an-agent.playbook.md), or [set-up-my-team](set-up-my-team.playbook.md) to begin.
 2. **Reject** — the evaluation surfaced a blocker. Name it so future versions of Tropo can address it.
 3. **Investigate further** — you want to stress-test one more artifact, or read the principles more deeply, or ask a specific architectural question. Tell me which, and we'll do it."

 Respect the evaluator's answer. If they reject, LOG THE REASON to `channels/ops.md` as a named finding — a rejection with a reason is more valuable than an acceptance without one. If they want to adopt, route without a sales moment. If they want to investigate further, loop back to Step 3 with a different target OR answer their specific question from the governance surface walked in Step 2.

## Outcomes

- [REQUIRED] The evaluator has walked the five governance primitives (ledger / capsules / AGENTS.md / ADR index / principles) with one concrete example each.
- [REQUIRED] The evaluator has picked a cold-boot target from the vault (not chosen by the concierge).
- [REQUIRED] A stranger-test sub-agent has been dispatched per Step 4's inline prompt, has executed against the target file, and has returned its Cold-Boot Stranger Test Report.
- [REQUIRED] The cold-boot findings (verdict + files-opened + P0/P1/P2 + inventions) have been surfaced to the evaluator without hiding or paraphrasing failures.
- [REQUIRED] The evaluator has been prompted to decide: Adopt / Reject / Investigate further.
- [REQUIRED] If rejection: the reason has been logged to `channels/ops.md` as a named finding.
- [OPTIONAL] The evaluator has chosen to run additional cold-boot tests on other artifacts.
- [OPTIONAL] The evaluator has routed to a build playbook to start working.
- [OPTIONAL] The evaluator has filed a specific architectural question or concern that requires follow-up.
- [OPTIONAL — v1.5] The evaluator has been shown the vault-maintenance toolchain — the four Python scripts at `.tropo/scripts/` (rebuild-vault, rehydrate, tropo-validate, validate-no-absolute-paths) and how they keep the vault's indexes honest. An evaluator asking *"how does this stay healthy over time?"* gets walked through [vault/files/a24c5b66.md](../../kb/how-to-maintain-your-vault.md) plus a live `tropo-validate.py` run against the current vault to demonstrate. The evaluator sees that durability is not magic — it's a small, readable toolchain they can run themselves.

## Verification

Method: `cold-boot-test`.

This playbook declares `cold-boot-test` as its verification method deliberately — the playbook itself is a demonstration of the cold-boot test, so using the method that proves it is meta-consistent. Verification receipts: one per stranger sub-agent dispatch executed during a run of this playbook, appended to the run's `run.jsonl` (or for v1.3 honor-system: to `channels/ops.md` with the stranger sub-agent's report captured as prose or a referenced file).

The stranger-test dispatched in Step 4 IS the verification of what Tropo claims. A run of this playbook that produces a Cold-Boot Stranger Test Report has produced evidence. A run that doesn't has failed its own rule (Rule 2).

Most-common follow-on intents from an evaluation that landed positively:
- **"I'm convinced — I want to start a real project"** → routes to `start-a-project.playbook`.
- **"I want to try one quick agent first"** → routes to `create-an-agent.playbook`.
- **"I want the full First Run depth-first"** → routes to [`vault/files/e98420c3.md`](../files/e98420c3.md) (v1.17.0 successor; Tropo-owned four-phase first-user experience; v1.4-era welcome.playbook recycled at v1.17.0 ship).

- **"I want to think about it"** → concierge thanks the evaluator, names that the evaluation report is durable (filed under [`agents/sa/sa.cold-boot/activation-log/`](../../../agents/sa/sa.cold-boot/activation-log/) for re-reference), offers to keep the conversation open.
- **"This isn't for us"** → concierge respects the rejection per Suggestion 6; rejection-for-architectural-disagreement is a different signal than rejection-for-broken-governance, both valid evaluation outcomes.

This playbook does not implement bounce — it hands conversation cleanly back to the concierge surface. The cold-boot test report stands as durable evidence regardless of routing outcome.

## Suggestions

- Let the evaluator drive. This playbook is LESS script and MORE tour — follow their curiosity. If they ask a question in Step 2 that Step 4 would address, answer it early. If they want to skip Step 2 and go straight to the stranger test, that's fine — it's THEIR evaluation.
- Be honest about gaps. Tropo at v1.3 ships many artifacts at PASS-WITH-GAPS, not PASS. That's the discipline (verification before lock, not after). An evaluator who sees the gaps and understands the remediation path trusts the system more than one who sees only the polish.
- Do NOT dispatch multiple sa.cold-boots in parallel for one evaluation session. One stranger test, one artifact, clear evidence. Depth over breadth.
- If the evaluator picks a file with known high gap count (e.g., a legacy playbook like `first-vault-setup.playbook.md` that predates v2.0 enforcement), name the expected finding before the stranger test runs: "This file predates the current playbook.capsule — it won't satisfy today's validation checks. The cold-boot will surface that. Still want it as the target?"
- If the evaluator is an architect who wants to read Tropo's own architectural history, point them at the ADR index + the v1.3 design briefs. The provenance is the governance.
- If the evaluator rejects on ideological grounds (e.g., "I don't want governance"), respect the rejection. Tropo is not for every audience; a rejection for architectural disagreement is a different kind of data than a rejection for broken governance.
- Never fake a cold-boot test. If harness issues prevent the stranger sub-agent from dispatching, say so plainly and offer to reschedule. Rule 2 halts the playbook rather than accepting a mocked demonstration.
