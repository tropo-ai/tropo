## Purpose

Frames the release. Produces a committed `release-plan.capsule` instance naming the release number, intent, completion outcome / definition-of-done, and sub-systems to be touched. The user confirms the release number before the plan is committed.

The release-plan is the contract for this dev cycle. Step-2 reads it to know how many arch-specs to spawn; deploy stage reads it to produce release notes. The release-plan must be committed before this step is done.

## Structure

Leaf step. No children.

## Nodes

No child nodes. This is a leaf WorkflowNode.

## Flow Rules

Completes when: a release-plan.capsule instance is committed to the Vault, linked from this pipeline-run, with (a) release number confirmed by user, (b) intent declared, (c) completion outcome / definition-of-done declared, (d) sub-systems list declared.

Human confirmation required: the executor pauses for user confirmation of the release number before committing. *(Pause mechanism is a Known Enforcement Gap at v2.0.)*

Advances to: spawn-arch-specs (e1b819c4).

## Cold-Boot Walk-Through

The executor checks the registry for the next incremental release number. It drafts a release-plan instance with that number plus the intent and sub-systems derived from the accepted work-item. The draft is surfaced to the user for confirmation of the release number. User confirms (or adjusts). The release-plan is committed. Step done.
