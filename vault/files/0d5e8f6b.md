## Context

Argo-OS operational files contain structured data encoded as prose — status fields, blockers, priority flags, timestamps. Humans read this well. Machines cannot query it without parsing full markdown files. The repair agent can catch broken paths but cannot answer "which agents have open blockers?" without reading every status card in full. This is a gap that grows more expensive as the vault scales.

YAML front-matter is the standard mechanism for adding machine-readable headers to markdown files. Obsidian supports it natively. The agent profile spec (A3) already uses it. Extending the pattern to operational files closes the query gap without changing any prose.

---

## Decision

**Add YAML front-matter to three file types in V1: status cards, crew brief, and living transfers.**

Full schema definitions, field specifications, and conventions are in the design spec at `design/yaml-frontmatter-standard.md`.

Key design choices:
- **Additive, not substitutive.** YAML does not replace prose. Strip it and the file remains fully usable.
- **Obsidian-native.** Schema is immediately useful with existing vault tooling; no new infrastructure required for V1.
- **Schema-light.** Only fields that enable a concrete operational query. No speculative fields.
- **ADRs, activation files, and channels excluded from V1.** ADRs are stable and low-churn. Activation files already have Section 0 context framing (YAML would conflict). Channel files are ephemeral.

---

## Rationale

The ASA (ADR-012) currently reads full prose files to detect stale status cards, open blockers, and transfer warnings. YAML front-matter lets it read structured headers instead. This reduces ASA token consumption and increases detection accuracy. The repair agent gains a new class of T1–T3 checks: schema compliance, freshness, cross-file consistency. The vector query layer (P5) needs structured data to be useful — this is the prerequisite.

---

## Consequences

**Positive:**
- Status cards, crew brief, and living transfers become machine-queryable
- ASA staleness audit reads YAML headers instead of full prose
- Repair agent can validate front-matter as a formal T1 check
- Path opened for Obsidian dataview queries and eventual P5 vector indexing

**Negative:**
- Agents must maintain front-matter in sync with prose on every update
- Front-matter drift (YAML stale, prose current) is a new class of error to catch
- V1 rollout requires a touch-pass across all active status cards and living transfers

**Neutral:**
- Status card front-matter partially overlaps with agent profile YAML. When profiles roll out, dynamic fields migrate to profiles and status cards shrink. This is the intended path.
