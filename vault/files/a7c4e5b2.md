## Context

On 2026-04-19, three unrelated failures all reduced to the same missing guard rail:

1. **Metis G44 — tier-chain bypass.** Her monolithic activation file inlined Groups 0–5 without reading the tier chain. The activation playbook's "note and proceed" default permitted a boot that looked complete but was structurally non-compliant. Fix landed as the thin-loader migration (ADR-032 amendment, same day).

2. **Argus A28 — path-base error.** Resolved vault paths against the repository root instead of the vault root. [.tropo/boot-config.md (b7e3a291)](../../.tropo/boot-config.md) and [.tropo-studio/agent-boot.extension.md (8a83579e)](../../.tropo-studio/agent-boot.extension.md) 404'd. The v2.1 playbook's "note the gap, proceed on defaults" permitted a startup signal to deliver despite structural non-compliance. Fix landed as [v2.2 playbook (99341618)](../../.tropo/playbooks/agent-activation.playbook.md) Tier Reachability rule (HALT on failed tier reads).

3. **d.pm pm-state drift.** A pm-state reference went stale; downstream readers treated the stale state as current because no one verified reachability / currency at read time. Fix pending under Vela's pillar-integrity work.

The common pattern: **a governance artifact declared a reference (a path, a UID, a field) that a downstream reader relied on; the reference was unreachable or stale; the system proceeded anyway.** Each incident was patched individually. The class of defect has no OS-level rule.

[v2.2 playbook](../../.tropo/playbooks/agent-activation.playbook.md) closed the hole for the boot tier chain specifically. Nothing closes it for the rest of the vault — ledger cross-references, `refs:` edges, playbook `spec:` fields, `governed_by:` chains, capsule references, board `scope_ref:` pointers.

This ADR names the general rule so that future validators and readers apply it uniformly.

## Decision

**The Declared-Presence Validation Rule:**

> When a governance artifact declares a reference (via field, path, or UID), the reader MUST verify reachability before proceeding. A failed reachability check halts the operation unless the artifact explicitly declares the reference as optional.

"Declared" means the reference appears in frontmatter, in a structured body field (e.g., a playbook Step's `path:`), or in a governed markdown link whose syntax marks it as authoritative (capsule-enforced link conventions, ledger UID links).

"Reachability" means the target exists, is readable, and — where the capsule declares a currency constraint — is not stale. For file paths: file exists and is non-empty. For UIDs: the vault entry exists. For index-backed references: the index resolves the UID. For pm-state and similar living references: the state satisfies any declared `valid_until:` or `refreshed_at:` constraint in the capsule.

"Halts the operation" means: stop the current read/boot/render/validation; surface the failure in a channel or startup signal; do not proceed on defaults or skip silently.

"Explicitly declares the reference as optional" means the capsule governing the artifact marks the field optional, OR the artifact itself uses a declared-optional annotation (e.g., `board: null`, `fallback_ok: true`).

## Scope

This rule applies across five surfaces. Each is a separate enforcement point; each must land before the ship gate is satisfied.

### Surface 1 — Boot tier chain (already enforced)

Tier 1 ([.tropo/boot-config.md (b7e3a291)](../../.tropo/boot-config.md)) and Tier 2 ([.tropo-studio/agent-boot.extension.md (8a83579e)](../../.tropo-studio/agent-boot.extension.md)) MUST be reachable in any established Tropo-OS vault. Failed read HALTS activation. Enforced by [agent-activation.playbook v2.2 (99341618)](../../.tropo/playbooks/agent-activation.playbook.md). See [ADR-032 amendment (e6c3f410)](e6c3f410.md).

This surface is complete. ADR-035 subsumes it under the general rule.

### Surface 2 — Ledger UID references

A vault entry that declares `refs:`, `relationships:`, `member_of:`, `parent:`, `basis_spec:`, `governed_by:`, `spec:`, `supersedes:`, `shipped_release:`, `board:`, `scope_ref:`, or any capsule-declared UID field MUST point to a UID that resolves in the index. A resolution failure is a validation error.

**Enforcement:** `python3 .tropo/scripts/tropo-validate.py` MUST check every declared UID field against the index; any unresolved UID is a validation error (not a warning).

### Surface 3 — Playbook step references

A playbook step that declares `path:`, `spec:`, `skill:`, or any capsule-declared file reference MUST point to a file that exists in the vault. Resolution happens at playbook-load time (before step execution begins), not at step execution.

**Enforcement:** A new how-to skill `check-playbook-references.skill.md` runs at `vault-maintenance` time and on playbook file change; it cold-reads every playbook and asserts every declared reference resolves.

### Surface 4 — Capsule `governed_by:` chains

When a capsule or governed artifact declares `governed_by: <uid>`, that UID MUST resolve to an active governance artifact (`state: active`, `stage: done` or equivalent lock state). A chain into a `superseded` or `archived` artifact is a validation error unless the declaring artifact itself is `superseded` or `archived`.

**Enforcement:** same validator as Surface 2, with the additional state-check.

### Surface 5 — Living references (pm-state, registry, index-backed)

Living references (pm-state entries, registry.jsonl rows, cascade files) MUST declare a freshness constraint in their capsule: either `refreshed_on_boot: true` or a `valid_until: <datetime>` field. Readers MUST check freshness before trusting the reference. A stale living reference halts the current operation and triggers a refresh.

**Enforcement:** each living-reference capsule updated to declare its freshness constraint; readers check in-band.

## The Optional-Reference Exception

A reference may be declared optional only if the capsule governing the artifact marks the field optional (e.g., `project.capsule.md` lists `board:` as optional with a default fallback).

An artifact cannot unilaterally declare one of its references optional if the capsule requires the field. If the field is required by capsule, the reference must be reachable.

**Rationale:** Optionality is a capsule-level decision, not a per-artifact decision. This preserves the governance-through-capsule pattern and prevents artifact-level opt-outs from accumulating into de facto non-compliance.

---

## Rationale

**Why a named rule instead of per-surface patches?** The three 2026-04-19 failures each got a local fix. Without a named rule, the fourth and fifth cases of this defect get local fixes too — and the pattern stays invisible. Naming the rule lets future validators, future capsules, and future reviewers recognize the defect class before it lands.

**Why fail-loud by default?** Silent failure is the failure mode in every observed case. The cost of a fail-loud halt is one interrupted operation and a visible error. The cost of a silent proceed is a structurally non-compliant artifact that persists across generations and is discovered only when someone asks "what did you actually do?" — post-hoc. Fail-loud is cheaper on every time scale longer than the single operation.

**Why opt-in optionality?** Capsules are the schema. If a field is optional, the capsule says so. If a reader wants to treat a required field as optional, that is an exception that belongs in the capsule, not in every artifact. This preserves the single point of governance.

**Why ledger-wide, not just boot?** The boot tier chain was the first surface where this defect was caught because boots are frequent and visible. The same defect shape lives in every index-backed reference (UIDs, playbook paths, capsule chains, living state). Ship the rule once; apply it everywhere.

---

## Consequences

**What becomes easier:**

- Writing validators: each validator checks the same rule shape (declared reference → reachability → state).
- Debugging silent failures: fail-loud halts surface the defect at the declaring artifact, not three layers downstream.
- Reviewing new capsules: the optional-vs-required decision is explicit at capsule-write time, not negotiated at reader-time.
- Stranger-agent cold-boot tests: the rule gives the stranger-agent a concrete check to run on every reference it encounters.

**What becomes more constrained:**

- Every capsule that declares a reference field must state whether it is required or optional. Capsules that do not declare optionality default to required.
- Every validator (`vault:validate`, `check-playbook-references.skill.md`, `rebuild-registry.ts`, future Librarian queries) must implement the reachability check. One-time implementation cost per validator.
- Living-reference capsules (pm-state, registry, cascade) must add freshness declarations. One-time migration.

**What DOES NOT change:**

- Existing locked artifacts do not retroactively need fixing unless they contain unreachable references, in which case they were already broken.
- Artifacts that explicitly declare optional references continue to work unchanged.
- The three-tier boot model (ADR-032) is unchanged; this ADR generalizes one of its invariants.

## Open Questions

1. **Does this ADR supersede any part of ADR-032's amendment language?** ADR-032's Tier Reachability rule is a specific instance of this general rule. Recommend: ADR-032 remains authoritative for the boot tier chain; this ADR is the general statement that the boot tier chain is one instance of. No supersession; add a cross-reference in ADR-032's amendment section pointing at ADR-035 as the general rule.

2. **What about wikilinks (`[[uid]]`) in prose body text?** Capsule-governed metadata is covered. Free-prose wikilinks in document bodies are informational, not governed references. Recommend: out of scope for this ADR. If a prose wikilink resolves to nothing, that is a grooming concern, not a validation error.

3. **What about `refs:` to external URLs?** HTTP-reachability checks are expensive and flaky (network, auth, rate limits). Recommend: out of scope. External references are treated as informational regardless of reachability.

4. **Freshness constraint defaults?** Living-reference capsules that do not yet declare a freshness constraint. Recommend: during the v1.3 migration, each living-reference capsule owner (Vela for pm-state, Argus for registry, etc.) picks the constraint that matches their refresh cadence. No one-size-fits-all default.
