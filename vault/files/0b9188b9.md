## What is a Tropo subsystem?

A **subsystem** is a coherent slice of the Tropo platform — a domain that has its own primitives, its own evolution, and its own readers. **Seven active subsystems** exist as of v1.8 (6 anchored at v1.7 + Documentation promoted v1.8 mid-cycle per Mike-A46 elevation of v1.8 brief D4):

| Subsystem | Hub UID | What it covers |
|---|---|---|
| **Tropo Governance** | [`8dd772a0`](../../vault/files/8dd772a0.md) | Verification + ADRs + capsule discipline + ship gates |
| **Tropo Rendering** | [`dbc1cbbf`](../../vault/files/dbc1cbbf.md) | Python rebuilder + cascade indexes + project-tree + graph index |
| **Tropo Work** | [`2d083137`](../../vault/files/2d083137.md) | Tasks + projects + pipelines + the L0 work substrate |
| **Tropo Agents** | [`99ed55fd`](../../vault/files/99ed55fd.md) | Executive agents + session-agents (sa.\*) + directors (d.\*) |
| **Tropo Playbooks** | [`76bab75f`](../../vault/files/76bab75f.md) | Playbooks + skills + concierge paths |
| **Tropo Library** | [`1aba710c`](../../vault/files/1aba710c.md) | (NARROWED v1.8) Content-storage substrate — packs, crew-narratives, library content storage |
| **Tropo Documentation** *(NEW v1.8 — 7th subsystem)* | [`f87e33f0`](../../vault/files/f87e33f0.md) | Canonical OS documentation: KB articles, library/docs/, the Tropo Handbook, the Studio Manifesto, cross-cutting OS-level explainers |

The active roster lives at [`collections/subsystem-hubs.collection.md`](../../collections/subsystem-hubs.collection.md). Subsystems are **typed projects** (`type: project` with the `subsystem-hub` tag) rooted at [`tropo-subsystems` (aae9a37b)](../../vault/files/aae9a37b.md).

## How a subsystem is documented

Each subsystem has one **subsystem hub** — a markdown file that answers one reader question: *"what is the truth-today state of this subsystem?"* Hubs are governed by [`subsystem-hub.capsule v1.3`](../capsules/subsystem-hub.capsule.md). The contract is intentionally minimal:

**Required body sections (4):**
1. `## What This Subsystem Covers` — scope + boundary tests against adjacent subsystems
2. `## How Tasks Flow` — operational practice; the `projects:` array pattern
3. `## Current State` — 3-6 sentences answering "where is this subsystem TODAY"
4. `## Change Log` — append-only release impact log

**Optional body sections:**
- `## Open Questions` — present-truth open questions (not historical)
- `## Architectural Constraints` — present-truth invariants (not historical)

**Required frontmatter:**
- `lifecycle: standing`, `tags: [subsystem-hub]`, `subsystem_name`, `subsystem_scope`
- `member_of: [aae9a37b, ...]`
- `last_release_reflected:` (semver or null)
- `release_history:` (append-only event log; bidirectional with the registry)
- `aligned_with: 8a4e21c5` (this capsule's UID)

**Crucially, the contract is concentrator-not-archive.** Historical questions route to:
- `key_decisions` → ADR set (`vault/00-index.jsonl` query: `type: decision`)
- `health_read` → `shared/orientation/daily-health-report.md`
- `related_work` → `vault/00-graph-index.json`

These three sections were dropped from the contract in v1.3 per [Decision 4 of v1.7 brief](../../vault/files/6b5f7886.md). Recapitulating their content in hubs created drift surface; pointing at the canonical primitives eliminates it.

## The documentation-as-release-deliverable doctrine

Mike pinned this doctrine on 2026-05-05: ***"Documentation is a release deliverable, not an artifact."*** Before v1.7, hub updates were honor-system. Result: 7 cycles of unreflected releases (every cycle from v1.4.0 forward) per A44's [swarm audit (2026-05-05, 145 documents across 6 corpora)](../../agents/argus/workspace/v1.7-audit/04-corpus-audit-results.md) — Cluster 3 drift.

v1.7 closes the loop with **three substrate primitives**:

### 1. `release_history:` on every hub

Every subsystem hub carries an append-only `release_history:` array in frontmatter. Each entry: `{release_uid, release_version, summary, registry_uid, streams_touched?}`. Bidirectional pair with the registry.

### 2. `subsystem-registry.jsonl` at vault root

A vault-wide append-only event log at [`subsystem-registry.jsonl`](../../subsystem-registry.jsonl). Every release writes one row per touched subsystem. The registry consolidates the cross-cut over per-hub `release_history:` arrays — a single greppable file answering "what subsystems did each release touch + with what doc updates?"

### 3. `release.capsule v3.3` Rule 11 + Check 22

[`release.capsule v3.3`](../capsules/release.capsule.md) introduces:
- **Rule 11** — Subsystem documentation as release deliverable. Bidirectional pair with `release_history:` field on hubs.
- **Validation Check 22** — for releases dated `>= v1.7.0` ship: every subsystem in the linked release plan's `sub_systems:` array MUST have a registry row with matching `release_uid:`.
- **Soft-gated v1.7-v1.9** (honor-system; reviewer-of-record verifies).
- **Hard-gated v1.10+** via `scripts/validate-subsystem-hubs.ts` (deferred per [v1.3 §Pending Sub-Requirements](../capsules/subsystem-hub.capsule.md)).
- **Grandfathering:** releases dated `< v1.7.0` are EXEMPT.

## How hub grooming actually happens at ship gate

The dev-pipeline ship gate runs a NEW step `groom-subsystems` ([`a5554670`](../../vault/files/a5554670.md)) between step-4 (release-notes) and step-5 (produce-release-folder). The step exercises the **pipeline-step-with-sa.\* verifier pattern** — the first instance of which lives in v1.7; the generalized capsule is formalized at v1.10 Enforcement.

### Per touched hub

The step's executor uses [`dispatch-sa-step.skill`](../skills/dispatch-sa-step.skill.md) to orchestrate three sa.* instances:
- **2 workers** (sa.hub-groomer in `mode: worker`) — each independently produces a recommended grooming draft via [`groom-subsystem-hub.skill`](../skills/groom-subsystem-hub.skill.md)
- **1 judge** (sa.hub-groomer in `mode: judge`) — reads both worker drafts, identifies divergences, produces a unified recommendation with **convergence-by-disagreement verdict** (HEALTHY / SUSPICIOUS / TRIVIAL)

[`sa.hub-groomer`](../../agents/sa/sa.hub-groomer/sa.hub-groomer.md) does NOT write to the hub directly. The executor reviews the judge's recommendation, makes the call, and writes:
1. The hub-side `release_history:` row first (with `registry_uid: <pending>` placeholder)
2. The corresponding row in `subsystem-registry.jsonl`
3. Backfills `registry_uid:` on the hub-side row to the registry row's UID

This atomicity-by-ordering pattern mirrors merger-window discipline.

### Convergence-by-disagreement as quality signal

The pattern A44's swarm proved on the v1.7 audit (5 agents, IPC via channel file, cross-corpus reasoning) is the same primitive sa.hub-groomer uses at single-hub scale:
- **HEALTHY** verdict — workers diverged on details (wording / framing / minor inclusions) but converged on substance. Most common; expected pattern.
- **SUSPICIOUS** verdict — workers diverged on substance (different facts about what shipped). Flag for human review.
- **TRIVIAL** verdict — workers near-identical. Honest signal that one worker may have copied OR both missed the same context. Re-run.

The 2-worker + 1-judge swarm is **executor-orchestrated**, not sa.* sub-spawning (sa.* terminal-one-level constraint). The executor (a primary agent like Argus or Vela) dispatches all three instances directly via channel-file IPC.

## How a stranger uses this

If you're a stranger reading this article fresh:

**To find the active subsystems:** read [`collections/subsystem-hubs.collection.md`](../../collections/subsystem-hubs.collection.md).

**To find what a subsystem covers + its current state:** open the hub instance (UIDs in the table above). Read `## What This Subsystem Covers` + `## Current State`.

**To find what's been shipped affecting a subsystem:** read the hub's `## Change Log` + `release_history:` frontmatter.

**To find what changes any specific release made across all subsystems:** grep [`subsystem-registry.jsonl`](../../subsystem-registry.jsonl) for the release UID.

**To author a release that touches subsystems:** declare them in the release plan's `sub_systems:` array (or — under Correction 1 retrospective-not-predictive — populate at ship). The `groom-subsystems` step at ship gate auto-generates groomed drafts; you review + commit.

**To create a NEW subsystem:** follow [§Adding a New Subsystem in v1.3 capsule](../capsules/subsystem-hub.capsule.md). Crew decision required first.

**To merge or archive a subsystem:** follow [§Migration Window in v1.3 capsule](../capsules/subsystem-hub.capsule.md). Atomic supersession.
