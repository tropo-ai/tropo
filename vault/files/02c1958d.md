# agents/tropo/ — Governance Contract

*The permanent Tropo agent's filesystem split. Kernel-managed identity refreshes on Tropo release. User-owned lived experience persists.*

---

## Why this contract exists

Tropo is a permanent agent that ships with every Tropo release. The release pushes new identity (charter, soul, activation procedure, briefing) into every Studio that has Tropo installed. But Tropo also accumulates real relationships with each user — memory of what the user worked on, generation log of every Tropo session in this Studio, transfers between Tropo generations.

If the update pipeline doesn't know which files to refresh and which to leave alone, two failure modes follow. **Refresh too much** → user loses Tropo's accumulated memory of them; the Tropo they've known for six months is suddenly a stranger after every release. **Refresh too little** → identity files stale; Tropo speaks for an old version of the brand; bug fixes never reach users; new features Tropo should know about don't land.

This contract solves it. Two named classes of files; explicit update-policy for each; the update pipeline reads this contract and behaves accordingly. The user keeps their Tropo; Tropo gets every release update.

The hotel-concierge analogue: standard training (manuals, brand voice, chain procedures) ships from chain HQ — refreshes when HQ updates training. Accumulated relationships with the guests at this specific hotel are the concierge's and the hotel's — stay through every training update. Same architectural shape; different substrate.

---

## The contract

```yaml
kernel_managed_files:
  - tropo-charter.md
  - tropo-soul.md
  - tropo-activation.md
  - agent-boot.extension.md
  - briefing-package/        # 00-index.md + current-release-notes.md + supporting briefing artifacts
  - CAPSULE.md               # this file is itself kernel-managed (the contract refreshes with the agent)

user_owned_files:
  - tropo-status.md          # this Studio's current Tropo state
  - tropo-generation-log.md  # this Studio's Tropo lineage (T1, T2, ... per-Studio)
  - transfers/               # generation-to-generation handoffs in this Studio
  - .tropo-capsule/memory/   # accumulated memory of this user + this Studio
  - .tropo-capsule/workspace/   # scratch space

write_access:
  tropo:
    - all user_owned_files (sole writer)
    - none of kernel_managed_files (Tropo cannot rewrite its own identity files; the brand evolves through release process, not through self-modification)
  update-pipeline:
    - all kernel_managed_files (sole writer; refreshes per release)
    - none of user_owned_files (never touches; ironclad)

update_policy:
  - On Tropo release detected (.tropo/version.md updated):
    - Diff incoming kernel_managed_files against current versions
    - Apply replacement or migration per .tropo/migrations/<from-version>-to-<to-version>/
    - Leave user_owned_files untouched (zero exceptions)
    - Trigger Tropo's update-orchestrator role at next boot — Tropo reads new identity, runs Path 1/Path 2 memory-reconciliation, posts summary
  - On user request to reset Tropo identity (uninstall + reinstall scenarios):
    - kernel_managed_files refresh from current release
    - user_owned_files: user-explicit choice (keep, archive, or wipe — never automatic)
```

---

## Important — `update-pipeline` is a v1.18.0+ infrastructure component

**At v1.17.0 ship, the `update-pipeline` named in `write_access:` above does NOT exist as a runnable system.** This contract declares the *expected behavior* of an update pipeline; v1.17.0 ships the contract substrate; the pipeline itself is future infrastructure work (likely v1.18.0 or v1.19.0 cycle scope).

**Until the pipeline lands:** kernel-managed file refresh on Tropo release happens manually — when Mike (or whoever directs the next Tropo release in the Argo dev Studio) updates kernel-managed files, the refresh is a deliberate edit pass, not an automated sync. User Studios receiving Tropo releases get the refresh through whatever release-distribution mechanism Tropo eventually ships (also v1.18.0+ infrastructure).

**Why declare the contract now if the enforcer doesn't exist:** the contract IS the substrate the future pipeline will read. v1.17.0 ships the agreement (this CAPSULE.md). v1.18.0+ ships the enforcer. Until then, the contract is also a *behavioral commitment* — anyone manually editing files in `agents/tropo/` honors the kernel-managed-vs-user-owned split as if the pipeline were enforcing it. Discipline by construction; mechanical enforcement when the substrate catches up.

**The contract's protection of user-owned files is structural even without the enforcer:** Tropo (the agent) has read-only access to its own kernel-managed identity files per write_access table; only `update-pipeline` is named as kernel-managed-file writer. Until the pipeline exists, NO automated process writes to kernel-managed files. Kernel-managed file changes are deliberate human or Argus-during-cycle work; not pipeline-driven.

---

## What each class guarantees

**Kernel-managed files** carry **brand consistency across the install base**. A user in a Tropo Studio in Boston and a user in a Tropo Studio in Buenos Aires get the same Tropo soul, the same charter, the same activation procedure, the same briefing — because those files ship from one source and refresh together. When v1.18.0 lands a refined character pin, every Tropo in every Studio gets it on next update. When a behavioral rule changes, every Tropo updates together. Brand evolves coherently, not Studio-by-Studio.

**User-owned files** carry **the relationship Tropo has with this specific user in this specific Studio**. The memory of what work this user does, what decisions they've made, what release news they've already seen, who their personal agent is, what their projects are. None of it ships from HQ. None of it leaves this Studio. None of it gets overwritten by an update. The user's Tropo stays the user's Tropo through every release.

The split is not policy — it's structural. The update pipeline can't write to user_owned_files because the contract declares it can't. The user can't accidentally lose their Tropo's memory by accepting an update because the contract protects it. Discipline by construction; not enforcement by exception.
