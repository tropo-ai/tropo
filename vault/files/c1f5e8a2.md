## Context

Tropo is being built to free users from harness-provider lock-in. Portability is its strategic counter-positioning, not an incidental property.

On 2026-04-24 Mike switched MacBooks (account `mikemaziarz` → `mike`). Moving the vault — which should have been a `cp -R` and a fresh activation — turned into a half-day of friction. The cause: 134 vault files contained user-account-anchored absolute paths (`/Users/mikemaziarz/...`) accumulated across 30+ Argus generations and many crew sessions. Vault portability had been treated as an aspiration rather than an enforced architectural property. <!-- portability:exempt -->

Three structural failures enabled this:

1. **The rules-bearing doc contradicted itself.** `settings/env.md` declared *"No folder prefix is ever hardcoded in path references"* (A17 rule, March 27, 2026) and then in the same file used `/Users/mikemaziarz/dev/tropo-ai/...` as a "current deployment" example. <!-- portability:exempt -->
2. **No structural enforcement.** No validator checked for absolute paths; nothing in CI failed when one was committed; no release-build gate caught absolute paths in shipped artifacts. The rule lived in prose and decayed.
3. **No principle stated.** No ADR named portability as a sacred architectural property. Each generation inherited the unenforced rule and the contaminating examples; drift compounded.

ADR-032 (Three-Layer Boot Configuration Model) addressed the path-base resolution mechanism at boot time, but did not declare portability as a system-wide property of vault content.

---

## Decision

**Vault portability is a sacrosanct architectural property of Tropo.** No vault content — ever — references an absolute machine path. The rule applies uniformly across all Tropo vaults, all releases, all generations. Enforcement is structural, not aspirational.

### 1. The Two-Surface Architecture

Tropo recognizes exactly two surfaces, with hard separation:

| Surface | What it is | Path discipline |
|---|---|---|
| **Crew Work Surface** | The monorepo (`tropo-ai/`) where the Tropo crew develops the OS, dogfoods it via the `argo-os/` vault, and builds shippable artifacts. Source of truth. | Within vault content (`tropo-ai/argo-os/...`): paths are vault-root-relative (`agents/...`, `channels/...`, `vault/...`). The `argo-os/` and `tropo-ai/` prefixes are implicit. |
| **Shippable Product Surface** | The versioned product zip (`tropo-os-vN.n.n.zip`) that users download. Built from `tropo-ai/releases/vN.n.n/dist/`. | Inside the zip: paths are root-relative to whatever the user's extracted root is (commonly `tropo-os/` or whatever they name the folder). NEVER absolute. |

The crew's filesystem is the crew's concern. The user's filesystem is the user's concern. These surfaces never reference each other's absolute paths.

### 2. Forbidden Pattern (universally banned)

In any vault content (active or shippable), the following are forbidden:

- macOS user-account paths: `/Users/<account>/...`
- Linux home-account paths: `/home/<account>/...`
- Windows profile paths: `C:\Users\<account>\...`
- Shell-anchored paths: `~/<...>` in vault prose
- Any other absolute reference to a specific machine's filesystem layout

These appear in vault content only as `<account>` symbolic placeholders (in pedagogical content explaining what NOT to write).

### 3. Allowed Patterns

When a vault doc must reference a path:

| Need | Use | Example |
|---|---|---|
| Reference content within the vault | Vault-root-relative path | `vault/files/c407f4c0.md`, `channels/ops.md` |
| Reference content within the monorepo (engineering, scripts, releases) | Repo-root-relative path | `tropo-ai/`.tropo/scripts/rebuild-vault.py`, `tropo-ai/argo-os/...` |
| Symbolic / pedagogical reference | Angle-bracket placeholder | `<vault-root>/agents/...`, `<user-home>/.claude/...` |
| Runtime-resolved path (forensic logs only) | Whatever the runtime returns | logged in `playbook-runs/.../run.jsonl`; never referenced from elsewhere |

### 4. Enforcement (structural, not aspirational)

Three mechanisms enforce this rule:

**4a. Vault-side validator** (`scripts/validate-no-absolute-paths.ts`, wired into `python3 .tropo/scripts/tropo-validate.py`). Recursively scans vault content for forbidden patterns. Fails the build if any active file violates. Allowlist exempts historical / runtime-forensic / ephemeral / dated-execution-script content; per-line `<!-- portability:exempt -->` marker exempts pedagogical content that legitimately shows a forbidden form as an example.

**4b. Ship-time gate** (`npm run release:validate`, runs the validator with `--strict` against `releases/vN.n.n/dist/` contents). NO allowlist on the ship side. Any absolute path anywhere in a shippable artifact halts the release. Wires into the existing harness regime as part of Gate 5.

**4c. Move-the-vault portability test** (`python3 .tropo/scripts/validate-no-absolute-paths.py --strict`). Exercises the portability claim by copying the vault to an alternate path, re-resolving from `settings/env.md`, and validating that all governance reads succeed and all cross-references resolve. Becomes a release-verification step. Required to PASS before any release ships.

### 5. Bulk Amendment of Pre-Discipline Locked Entries

Approximately eleven previously-locked vault entries contain user-account-anchored absolute paths from earlier (pre-discipline) authoring. Examples: release-build records (5a0e0c62, 7027ccf6, b7f77186, b8478a09, dc27f748, 174a832a, 1f5e740d, 16fc1e73), Claude Code data audit records (3bce45ef, aa72620e, 5b166097), test-folder reference (d674de1c).

**These entries are bulk-amended with UID preservation under this ADR's authority.** The amendment replaces user-account-anchored paths with their portable equivalents (`tropo-ai/argo-os/...` for repo-relative, `<user-home>/.claude/...` for symbolic). No content semantics change; only the path prefix is corrected. UIDs preserve; supersession chains preserve; ledger queryability preserves.

**Justification for overriding capsule immutability for this class of fix:** the prior lock contract was made under earlier (pre-portability-discipline) governance. This ADR establishes the new discipline as a sacrosanct property. The amended literal paths are factually wrong NOW — the machines they reference no longer exist in this environment. Preserving the wrong literal paths under "immutability" would prefer formal correctness over factual correctness, in violation of the core operating value (NO CARDBOARD MUFFINS — verify substance, not surface). The amendment is a correction, not a rewrite.

**Future amendments of this class** require Mike's direct authorization via this ADR's pattern: an ADR establishing a new sacrosanct property + naming the bulk-amendment scope explicitly + Mike's principal acceptance.

### 6. Governance Composition

This ADR composes with — does not supersede — earlier ADRs:

- **ADR-032 (Three-Layer Boot Configuration Model):** governs path-base resolution AT BOOT. ADR-041 governs path-discipline IN VAULT CONTENT. Composes cleanly; the boot resolution mechanism produces the runtime path, the content rule keeps the runtime path out of content.
- **decision.capsule v3.0 immutability rule:** preserved by default. ADR-041 names a specific class of bulk amendment as the carve-out (Section 5). Future class-amendments require an analogous ADR.

## Alternatives Considered

**Alt 1 — Per-line escape only, no bulk amendment.** Mark every `/Users/mikemaziarz/...` reference in locked entries with `<!-- portability:exempt -->`, leaving the literal paths in place. **Rejected** because it preserves factually-wrong content (the machine doesn't exist anymore) for the sake of formal immutability — exactly the cardboard-muffin antipattern.

**Alt 2 — No bulk amendment; let entries decay naturally.** Locked entries stay as-is; only new content follows the rule. **Rejected** because future readers (including stranger users in the v3 cold-boot test) will trip on the stale paths and lose confidence in the system; portability becomes "true going forward" rather than "true."

**Alt 3 — Ship a migration script that runs at vault clone-time.** A script that detects user-account contamination and rewrites paths after clone. **Rejected** because (a) it papers over the rule rather than enforcing it; (b) it doesn't fix the source-of-truth in `tropo-ai/argo-os/`; (c) "your vault works because we run a fix-it script after you download it" is exactly the lock-in pattern Tropo opposes. The ship surface must be portable by construction, not by post-hoc repair.

**Alt 4 — Rely on env.md's existing rule + an updated style guide.** Strengthen the prose, no validator. **Rejected** because that's what's been done for 10 months and it produced 134 contaminated files.

**Alt 5 — Validator with no per-line exempt mechanism.** Reject all matches without exception. **Rejected** because pedagogical content (e.g., a doc about *why* absolute paths break portability) legitimately needs to *show* a forbidden pattern as an example. The `<!-- portability:exempt -->` marker is the standard escape pattern (cf. `eslint-disable`, `# noqa`). Used sparingly, with comments explaining why.

---

## Consequences

**What becomes easier:**

- A user who downloads `tropo-os-vN.n.n.zip`, extracts it anywhere, and points an LLM at it gets a working vault on first try. No symlink hacks, no path-base errors, no half-day of cleanup.
- A crew agent who switches machines (worktrees, alternate clones, fresh checkouts) gets the same.
- New generations inherit a structurally-portable vault — drift cannot accumulate silently because the validator fails the build.
- Stranger cold-boot tests against shipped releases test what the user actually receives, not what was true on the build machine.

**What becomes more constrained:**

- Every commit that touches vault content runs the path validator. Forbidden patterns block merge.
- Every release-build run runs the strict validator on the dist artifact. Forbidden patterns block ship.
- Every release-verification run runs the move-the-vault portability test. Failure blocks ship.
- Pedagogical content that legitimately needs to show forbidden patterns must explicitly opt-in via `<!-- portability:exempt -->` markers.
- Authors of locked-class artifacts (release entries, ADRs, capsules) write portable paths from the start; the rule does not bend for "it's a record of a one-time event."

**What this signals strategically:**

Tropo's portability is not a marketing claim. It is a property the vault has, enforced by infrastructure the vault carries. Counter-positioning to harness-provider lock-in becomes architecturally legible. *"Spend whatever time we need to make Tropo more portable for others"* (Mike, 2026-04-24) is the founder directive that this ADR encodes; the validator + ship gate + portability test are the mechanism.
