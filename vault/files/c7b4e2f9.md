# ADR-031 — Three-Layer Vault Hygiene

## Status

**Accepted.** Locked by Mike Maziarz, April 9, 2026. Reviewed by Vela V25 (APPROVED).

---

## Context

ADR-009 (March 23, 2026) established a four-phase self-healing framework — detect, diagnose, repair, verify — and noted that only detection had been implemented. Since then, several independent implementations have emerged:

- AGENTS.md folder governance (per-agent write-gate compliance)
- Boot playbooks for executive agents (domain sweep at session start)
- Vela's vault maintenance playbook using parallel sub-agents (a4f2e819, April 8, 2026)
- The grooming agent fleet in `agents/grooming/` (board-synthesizer, vault-steward, freshness-monitor, etc.)
- Scheduled board refresh with declared cadences

On April 8, 2026, Mike Maziarz identified that these implementations form a coherent three-layer pattern — not three separate features, but one architectural primitive. This ADR names and formalizes that primitive.

---

## Decision

Tropo-OS adopts **Three-Layer Vault Hygiene** as a named architectural primitive. Every Tropo-OS vault implements all three layers. The pattern is:

**Layer 1 — The Agent Knows Its Place**
Behavioral governance at the point of action. Every agent reads AGENTS.md before writing to a folder. Self-governance within declared scope. Near-zero cost. Catches violations at the moment they would occur.

**Layer 2 — The Owner Checks Their Domain**
Boot-time sweep by every executive agent across everything they own. Implemented via the **Parallel Orientation Sweep** pattern: parallel read-only sub-agents with tight declared scope, structured verdicts (CLEAN / WARNING / ACTION-REQUIRED), synthesized in a single low-cost pass. Catches domain drift at session start.

**Layer 3 — The Vault Watches Itself**
Scheduled grooming agents running continuously, independent of any human or executive session. Cadences declared in structured frontmatter. Low-risk repairs auto-applied. High-risk escalated. Catches time-based drift between sessions.

---

## Rationale

**Why three layers, not one?** Each layer catches what the one below misses. L1 operates at the moment of a write — it cannot detect drift in a neighboring folder. L2 operates at session start — it cannot detect drift that accumulates between sessions. L3 operates continuously — but requires the grooming fleet to be comprehensive and the scheduler to be reliable. The layers are complementary, not redundant.

**Why a primitive, not three features?** Three properties distinguish a primitive from a feature collection: layered coverage (each catches what the others miss), independence (each operates without depending on the others), and proportionality (each applies the right amount of intelligence at the right cost). This pattern has all three.

**Why now?** Vela's vault maintenance playbook (April 8, 2026) proved the L2 pattern with a 3-agent test and found real vault issues. The board scheduler work (same session) confirmed L3 infrastructure is converging. All three layers now have working implementations. The right moment to name a pattern is when it has been proven, not before.

**Why not just ADR-009?** ADR-009 established the detect-diagnose-repair-verify framework but did not name the layered implementation model. ADR-031 fills that gap — it is the implementation architecture that instantiates ADR-009's framework across all four phases.

---

## Consequences

**What becomes easier:**

- **New agent onboarding.** The three-layer model gives new agents (and new users) a simple mental model for vault health. "Your agent governs itself, you sweep your domain, the vault watches itself."
- **Grooming fleet expansion.** New grooming agents can be added to L3 without redesigning anything. They declare their scope, cadence, and autonomy tier — the pattern handles the rest.
- **Cross-agent coordination.** L2 sweeps surface ACTION-REQUIRED items that require coordination between agents. The structured verdict format makes this machine-readable, not just human-readable.
- **External adoption.** The three-layer model is explainable to a non-technical user in one sentence per layer. It becomes a feature of Tropo-OS that can be documented, marketed, and taught.

**What becomes harder or more constrained:**

- **Executive agent activation files** must include a boot playbook that implements L2. New executive agents cannot be commissioned without one. This is a requirement, not a suggestion.
- **Folder governance** must be maintained at L1 — AGENTS.md and CAPSULE.md files must exist in governed folders. The validator (python3 .tropo/scripts/tropo-validate.py) enforces this. Gaps are reported as ACTION-REQUIRED in L2 sweeps.
- **L3 scheduler gap — CLOSED (April 14, 2026).** The primary open constraint named at lock time ("L3 grooming agents run on-demand only — not continuous") was resolved by `playbooks/fleet-ops.playbook.md` (UID `768c610e`, built by Vela V28, April 13, 2026) and the boot playbook extension (Layer 2 of ADR-032, April 14, 2026). The solution: instead of a dedicated scheduler infrastructure, fleet dispatch responsibility is distributed across every agent boot. Phase 0 of `agent-boot.playbook.md` reads `.tropo-studio/agent-boot.extension.md` which requires every booting agent to check the worker registry and dispatch stale fleet agents. The vault's between-session self-monitoring is now triggered by every executive boot — no dedicated scheduler required. This is architecturally cleaner than a standalone cron-equivalent: if no session touches the vault for days, the first agent to boot handles the backlog. Confirmed by Argus A22 (April 13, 2026): "the distributed boot-time model is architecturally cleaner than a pure L3 scheduler." L3 is now fully operational.

**What this completes:**

ADR-009's detect-diagnose-repair-verify framework is now fully instantiated:

| Phase | Layer(s) | Status |
|-------|----------|--------|
| Detect | L1, L2, L3 | Complete |
| Diagnose | L2, L3 | Complete (structured verdicts) |
| Repair | L3 auto / L2 escalate | Complete (grooming charters define tiers) |
| Verify | L3 | Complete (post-repair checks in charters) |

## Implementation Notes

**L1 — already complete.** AGENTS.md and CAPSULE.md governance is operational across the vault. The three-tier model (TROPO-CONTROL.md / STUDIO.md / CAPSULE.md) is locked.

**L2 — pattern proven, fleet incomplete.** Vela has the first full L2 implementation (vault-maintenance.playbook.md). Metis, Argus, and Orpheus need their own domain sweep playbooks. Template: `agents/grooming/` — adapt from Vela's implementation.

**L3 — fleet operational, scheduler pending.** Grooming agents are running. Scheduler trigger infrastructure is in development (Vela V25). Board definitions declare cadences in structured frontmatter — ready to wire when scheduler is available.

**The Parallel Orientation Sweep pattern** (named April 8, 2026) should be documented as a standalone pattern in `design/` or `guides/` so future agents can implement L2 sweeps without reverse-engineering Vela's playbook.
