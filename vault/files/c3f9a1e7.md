# How Naming Conventions Work

**Relations**

| Relation | Target |
|---|---|
| Governed by | [kb-article (4cb20382)](4cb20382.md) |

*A short explainer on Tropo's numerical-prefix navigation convention. The convention is declared in [STUDIO.md](../../STUDIO.md) under §Vault Defaults; this article is the "why and how" that backs it.*

---

## The convention in 30 seconds

Files and folders whose role is structural — they organize what's around them, not what they themselves contain — get a numerical prefix:

| Prefix | Role |
|---|---|
| `00-*` | Index / navigation |
| `01-*` | Inbox / orphan-catcher |
| `02-*` through `98-*` | Author's discretion (sub-categories) |
| `99-*` | Archive |

The convention works at any depth — vault-top, project-internal, even nested.

---

## Why this exists

Tropo vaults are read primarily through file viewers (VS Code, Finder, Obsidian). Those viewers default to lexicographic sort. Without prefixes, structural folders interleave alphabetically with content folders — your inbox lands wherever its name falls in the alphabet, your archive too. Reading a folder becomes a scan rather than a glance.

With prefixes, structure becomes glanceable:

```
my-project/
├── 00-index.md          ← always first; tells me what's here
├── 01-inbox/            ← always second; my orphan-catcher
├── 02-reading/          ← my own organizing
├── 03-decisions/
├── 04-network/
└── 99-archive/          ← always last; out of sight
```

The prefixes telegraph intent. A new agent or human reading the folder for the first time knows what's index, what's inbox, and what's archive without inspecting any file.

---

## Why double-digit (`01-` / `99-`) and not single-digit (`1-` / `9-`)

Lexicographic sort is character-by-character ASCII order. It is **not numeric** order. The order matters because mixing single and double digits produces surprises:

```
1-foo       ← first char `1`, then `-` (0x2D)
10-foo      ← first char `1`, then `0` (0x30); `-` < `0`, so `1-foo` first
11-foo
2-foo       ← first char `2` (0x32) > `1` (0x31); ALL `1*` items first
9-foo
```

Sort order: `1-foo`, `10-foo`, `11-foo`, `2-foo`, `9-foo`. Numerically wrong.

Double-digit fixes this:

```
01-foo
02-foo
09-foo
10-foo
11-foo
99-foo
```

Sort matches numeric up to 99. The cost is one extra character per prefix. The benefit is 99 sort-correct slots versus 9.

**Single-digit is fine for bounded sets.** If you know a folder will never have more than 9 children — like a fixed pipeline (`1-ideate`, `2-design`, `3-specify`, `4-build`, `5-deploy`) — single-digit is a cheaper notation that still sorts correctly *within the bounded set*. Just don't extend.

---

## Where the convention applies

**Vault-top level**

```
argo-os/
├── 00-tropo-nav/        ← index/navigation
├── 01-vault-inbox/      ← orphan-catcher (D7 enforcement)
├── ... (other top-level projects, alphabetical)
└── 99-* (no vault-archive yet, but the slot exists)
```

**Project-internal**

```
agents/mike.cos/
├── 00-* (none yet)
├── briefing-package/    ← (named, not prefixed — first-class artifact)
├── transfers/           ← (named, not prefixed — first-class artifact)
├── .tropo-capsule/      ← (private; outside the convention)
└── 99-* (no archive yet, but the slot exists)
```

**Nested project-internal**

```
my-personal-project/
├── 00-index.md
├── 01-inbox/
│   └── 01-this-week/    ← inbox-of-inbox; convention nests
├── 02-active/
├── 99-archive/
└── ...
```

---

## When NOT to prefix

The convention is for **structural** files and folders. Don't prefix:

- Content files (the actual notes, drafts, decisions, briefs themselves)
- First-class artifacts that have their own identity (`briefing-package/`, `transfers/`, `<name>-charter.md`, `<name>-soul.md`)
- Files governed by a specific schema (frontmatter UIDs, vault entries)
- Anywhere the prefix would obscure rather than clarify

If a folder has only 2–3 children and they're stable, prefixes add noise. If a folder organizes ongoing work and structural roles repeat across projects, prefixes earn their cost.

---

## What's already in the wild (and stays as-is)

Some pre-existing single-digit prefixed projects exist in this vault:

- `work-pipeline/` children: `1-ideate`, `2-design`, `3-specify`, `4-build`, `5-deploy`
- Per-stage children: `1-inbox`, `2-accepted`, `3-active`, `9-archive` (Specify / Build)

These are **bounded-set pipeline scaffolding**. They don't extend; the single-digit notation is intentional and **does not migrate** to double-digit. The convention specifically allows single-digit for bounded sets.

What's new (post-convention-lock 2026-04-25):

- `01-vault-inbox/` (renamed from `Argo Vault — Inbox`) — the founder's first exercise of the convention at vault-top level.

---

## Override

Folder owners may override the convention via their CAPSULE.md per the standard [override contract](../../.tropo/TROPO-CONTROL.md). The convention is a [STUDIO.md](../../STUDIO.md) Default, not a Constraint — folder-local needs win.

---

## Future considerations

- **Tooling support.** A future kernel script could validate convention adherence (e.g., flag a folder with 5+ children of which only some use prefixes — that's the inconsistency that breaks sort).
- **Concierge guidance.** When the concierge helps a user create a project with internal structure, it can suggest the convention as default scaffolding.
- **Renaming pressure.** If a vault grows enough that existing alphabetically-sorted projects feel "in the way," users may opt into a one-time rename pass. The convention is permissive about that — rename when it helps; don't force.
