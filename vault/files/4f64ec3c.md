## Purpose

After dev-pipeline build stage closes (step 4 update-subsystem-canonical-docs done; substrate authoring complete), the dev-cycle's owner authors a `test-spec.md` describing what test coverage must extend to cover the new behaviors the cycle's substrate introduces. The test-spec is the activation-input for a corresponding test-pipeline activation; this step fires that activation.

By terminal close of the dev-pipeline activation, the triggered test-pipeline activation MUST also be at `status:done` (engine-level enforcement per [51d171f3 §Three-Pipeline Coupling State Machine](51d171f3.md)). The anti-box-checking gate lives on the test-spec.capsule (Vela walks Phase B semantics): `behaviors_covered:` must list at least one behavior per new feature substrate the dev-cycle introduces; validator enforces. This step fires the trigger; the capsule's validator + Vela's gate ensure the test-spec describes real coverage, not box-checking.

Does NOT govern: the test-pipeline activation's own work (that's the test-pipeline definition's job; Vela V50 lane primary at Phase C). Does NOT cover: legacy cycle dev-pipeline activations pre-v1.51 — those grandfathered per dev-spec.capsule Rule 7.

## Flow Rules

**Activates when:** dev-cycle's pipeline-run reaches step 4 close (depends_on_steps: [9d4f7e21] update-subsystem-canonical-docs done). Engine surfaces this step as eligible-next in parallel with step 4.5 (trigger-doc-pipeline-activation) — both have same dependency; engine fires both concurrently.

**Completes when:** all three exit_criteria pass:

- `triggered_test_spec.uid exists` — a test-spec.md entry exists at vault/files/<uid>.md
- `triggered_test_spec.status == active` — test-pipeline activation against the test-spec entered status:active
- `dev_spec.triggered_test_spec_uids contains <triggered-test-spec-uid>` — the dev-cycle's dev-spec frontmatter has been atomically updated to record the triggered activation UID

**Advances to:** step 6 produce-release-folder (8654900a) — multi-input convergence with step 4.5 (0cf86ea5). Step 6's `depends_on_steps:` array is amended at v1.51 to require BOTH 4.5 + 4.6 complete (was `[9d4f7e21]` per v1.49.0 chain). Positioning per v1.49.0 reordered chain (Argus A78 substrate-coherence fix per 74f84c63); step 4 fires AFTER step 5.5, so the triggers sit between step 4 (last build/canonical-docs step) and step 6 (first ship-gesture step).

**Failure modes:** if test-pipeline activation fails to enter status:active (e.g., test-spec.md validation fails on anti-box-checking gate; engine refuses activation; race lock collision on triggered-spec UID), step 4.6 stays incomplete; dev-pipeline activation flips to `status: blocked_on_trigger_failure` per engine extension §Failure mode handling.

## Engine Contract — exit criteria DSL

Three exit_criteria evaluated mechanically by pipeline-runtime DSL evaluator (per v1.51 DSL extension at engine spec [51d171f3 §Exit-criteria DSL extension](51d171f3.md)):

- `triggered_test_spec.uid exists` — DSL resolves `triggered_test_spec` context handle to `dev_spec.frontmatter.triggered_test_spec_uids[-1]` (most recent trigger); checks existence at vault/files/
- `triggered_test_spec.status == active` — DSL resolves activation entry by that UID; checks frontmatter status field
- `dev_spec.triggered_test_spec_uids contains <triggered-test-spec-uid>` — DSL list-membership check; ensures the atomic write to dev-spec frontmatter succeeded

If all three pass: `verify-step 4f64ec3c` returns `pass`; step marks complete.

If any fail: step stays in `eligible` state; executor (Argus or test-pipeline-primary) addresses + retries the trigger fire.
