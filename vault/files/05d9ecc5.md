*Execution step. Talos T9 engineering implementation does the heavy lifting (subprocess orchestration + DSL evaluation + validator dispatch + sa.* commissioning + result aggregation). Vela orchestrates manual_walk + agentic_review dispatch shapes that need executive-class context. Step is engine-orchestrated; deterministic pass/fail capture per behavior.*

**Why Talos is step_owner_role:** the execution layer is engineering. Subprocess orchestration, DSL grammar evaluation, validator extension dispatch, sa.* commissioning wiring, result aggregation into run.jsonl event types — all engineering implementation work. Vela owns the overall pipeline + the dispatch shape (which verification_methods need what), Talos owns the actual runtime.

**What it does (per verification_method):**

| verification_method | Runtime action | Result capture |
|---|---|---|
| `machine_executable_script` | subprocess.run(command_line); capture exit_code, stdout, stderr | test_executed event: pass = exit 0, fail = exit non-zero; stdout/stderr tail (last 50 lines) for diagnostic |
| `deterministic_assertion` | Engine DSL evaluator parses expression per pipeline.capsule v3.0 §6; evaluates against current substrate state | test_executed event: pass = expression true, fail = expression false; expression-tree-result for diagnostic |
| `structural_check` | Engine invokes named validator extension function in tropo-validate.py | test_executed event: pass = check returns clean, fail = check returns violations; violations list for diagnostic |
| `agentic_review` | Engine dispatches sa.* class per dispatch_target via standard commissioning protocol [e863a1e0]; awaits [RESPONSE] block in activation-log record | test_executed event: pass/fail per sa.* verdict in [RESPONSE]; sa.* findings list for diagnostic |
| `manual_walk` | Engine surfaces Mike walk prompt; Vela orchestrates Mike's walk session; captures Mike's pass/fail verdict + rationale | test_executed event: pass/fail per Mike verdict; rationale ≥50 chars |

**Aggregation:** after all behaviors executed, engine writes test_aggregate event to run.jsonl with summary (pass_count, fail_count, per-behavior verdicts, total runtime). Aggregate becomes step 4 validate-coverage's input.

**Failure semantics:** individual behavior failures don't halt step execution — all behaviors execute; aggregate captures the full pass/fail picture. Step 4 validate-coverage decides whether aggregate satisfies coverage requirements.

**Hands off to:** step 4 validate-coverage.
