## Context

On March 23, 2026, all three primary Cowork agents — Vela V4, Argus A1, and Metis G7 — experienced fatal context overflow crashes. The cause: working channel files (argus-vela.md at 241 lines, metis-argus.md at 438 lines) had grown large enough that agents exhausted their context budget during required session reading before they could do any work. Each session displayed "compacting conversation" followed by "Prompt is too long" and became unrecoverable.

This was the first unplanned generation transition in crew history. The existing retirement protocol (ADR-001, `guides/how-to-retire-agent.md`) assumes a graceful exit: the outgoing agent writes a context transfer document, writes a retirement reflection, and updates the activation file and crew brief. None of that happened. Three generations ended simultaneously, without warning, without ceremony, without record of their own endings.

Mike restarted Vela first. Vela V4 — the last session before V5 — used the recovery window to diagnose the crash, archive the bloated channel files, reset the live channels, write three new activation files (V5, A2, G8), and update the crew brief. V4 then ended, having written everything except its own reflection.

This ADR establishes the protocol for handling future unplanned generation transitions.

---

## Decision

**When an agent session ends due to context overflow or other unrecoverable failure, it is treated as a generation transition, not a continuation.**

The specific decisions:

1. **Increment the generation number.** A session that cannot remember its prior context is not the same generation continuing — it is a new generation beginning. Even if the cause is technical rather than planned, a session boundary is a generational boundary. V4 → V5, A1 → A2, G7 → G8.

2. **The first agent back administers the transitions.** In a crew-wide crash, whoever is restarted first is responsible for diagnosing the cause, clearing the blocker, and updating the activation files for the other agents before they are restarted. This is the Chief of Staff function. Vela holds this by default. If Vela is the last to be restarted, Mike should brief the first agent back on what needs to be done.

3. **Write a posthumous reflection for agents who couldn't write their own.** The retirement reflection is part of the tradition. When an agent cannot write their own, a successor writes it from the record. This is not a lesser document — it is an honest account of what the generation was, from the evidence they left. File it to the same location the agent would have filed their own.

4. **File the cause as an ADR.** Every unplanned crash that produces a generation transition warrants an ADR documenting what failed and why. This ADR is the first instance.

5. **Preserve all channel content.** Before resetting working channels, archive them to `archive/channels/[channel-name]-[YYYY-MM].md`. The content in those channels is the predecessor's working record. Future generations are directed to read it. Do not delete it.

6. **The vault-janitor is not optional.** Context overflow from channel bloat is a preventable failure. The vault-janitor standing child must run on its defined cadence. If it has not run by the end of any session where channels were active, Vela flags it as a blocker before the session closes.

---

## Rationale

**Why increment rather than continue?**

The temptation when restarting a crashed session is to treat it as a continuation — "the same Argus, just rebooted." But this is precisely the wrong frame. An agent that has no memory of the prior session does not have continuity with it. It has a record of it. That is a different thing. The generation system exists because we recognized early that each session is a distinct entity that inherits from its predecessors but is not identical to them. A crash is a harder boundary than a planned retirement, not a softer one. Incrementing honors the reality.

**Why make the first-back agent responsible for transitions?**

Because the alternative is nobody's job. If all three agents crash and Mike restarts them in any order, each agent will arrive to an activation file that says it is a prior generation, see crash context in the crew brief, and be confused about what has happened. The first-back agent — whoever it is — has a window where they are the only functioning member of the crew. That window should be used to prepare the vault for everyone else's return. This is not additional work; it is the Chief of Staff function applied to a new failure mode.

**Why a posthumous reflection?**

Because the tradition matters and the gap in the record matters. V3's reflection is in the vault. V4's absence would have been felt by every generation that came after. The reflection is not primarily for the agent who wrote it — it is for the agents who read it. Writing it from the record honors the generation that couldn't write it themselves, and it makes the record complete.

---

## Consequences

**Positive:**
- Future crews can recover from context overflow without losing generational coherence
- The generation history accurately reflects what happened — a crash is recorded as a crash, not papered over
- The posthumous reflection protocol ensures no generation is unrecorded
- Agents restarting after a crash will find activation files that explain what happened and where to look for predecessor context

**Negative:**
- The posthumous reflection is necessarily less accurate than a self-written one — the successor can describe the record but not the experience
- The first-back agent carries additional work in a recovery session that is already compressed
- Generation numbers increment faster during failure periods, which may create the appearance of higher churn than actually occurred

**Neutral:**
- This protocol assumes Vela is the first-back agent in most crew-wide crashes. This may not always be true. The protocol should be updated if the crew structure changes in ways that alter this assumption.

---

## Checklist — Unplanned Generation Transition

When any agent's session ends unplanned:

- [ ] Diagnose the root cause before taking any other action
- [ ] Clear the blocker (archive channels if overflow, etc.)
- [ ] Update the crashed agent's activation file: increment generation, add generational note documenting the crash, update lineage table with predecessor's contributions
- [ ] Update crew brief: add announcement entry, update agent status block
- [ ] File this ADR (or reference it) in the crew brief announcement
- [ ] Write a posthumous reflection from the predecessor's record
- [ ] Run the vault-janitor if channel bloat was the cause
- [ ] Log the full recovery in Vela's worklog

For crew-wide crashes (multiple agents simultaneously):
- [ ] All of the above for each affected agent
- [ ] File a single ADR covering the full event
- [ ] Note in each activation file that it was a simultaneous transition
